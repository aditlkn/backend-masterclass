Section 01

The mental model: DNS is a distributed, eventually-consistent, hierarchical database

The framing that changes how you think about DNS: it's not a lookup service, it's a distributed database with replication lag. When you update a DNS record, you're writing to one node (your authoritative nameserver). Every other node in the system (recursive resolvers, stub resolvers, CDN edges, operating systems) has a locally-cached copy of the old data — and they'll keep serving it until their TTL expires.

This is why DNS propagation isn't instant. There's no "push." Every cache everywhere has to independently expire its old entry and re-fetch. A record with a 24-hour TTL can be served stale for up to 24 hours after you change it — and some resolvers (ISPs, corporate proxies) ignore TTLs and cache longer anyway.

The second mental model shift: DNS has layers, and each layer has different behaviour. Your application asks the OS stub resolver. The OS stub asks the recursive resolver. The recursive resolver walks the hierarchy and caches aggressively. The authoritative server is the only one that has the real data. When debugging DNS, you need to know which layer is responding — because each one can be wrong in a different way.

Section 02

The resolution chain: every hop, every cache

Walk through what happens when your Node.js service tries to connect to api.payments.internal for the first time. The dns.lookup() call in Node.js triggers a chain that most engineers have never seen in full.

Step 1: The stub resolver (your OS)

Node.js calls getaddrinfo() (a libc function, not a Node.js function). This hits the OS stub resolver, which checks /etc/hosts first, then /etc/resolv.conf for the recursive resolver address, then sends a UDP query to that resolver. This call is synchronous and blocking in libc — it blocks the OS thread that called it, not the Node.js event loop directly, but it does consume one of the thread pool threads (libuv uses a thread pool for getaddrinfo). By default, Node.js has 4 thread pool threads. Under DNS pressure, you can exhaust them.

Step 2: The recursive resolver (your ISP, 8.8.8.8, or your corporate DNS)

The recursive resolver does the actual work. It checks its own cache first. On a miss, it starts at the root: queries one of the 13 root nameserver clusters (a.root-servers.net through m.root-servers.net) for the TLD nameserver. Gets back "for .com, go ask a.gtld-servers.net." Queries that. Gets back "for payments.com, go ask ns1.payments.com." Queries that. Gets back the A record. Caches everything it got (at each level's TTL). Returns the answer.

On a warm cache, steps 2+ collapse to one response from the resolver's cache. On a cold miss across the full chain, you're paying 4+ network round-trips — potentially 100-400ms on a slow resolver.

Step 3: The authoritative nameserver (Route 53, Cloudflare DNS, your own BIND)

This is the single source of truth. Only this server has the current value of your records. When you change an A record in Route 53, it updates immediately on the authoritative servers. But the recursive resolvers everywhere still have the old value cached for up to the old TTL.

The SERVFAIL problem

When a resolver can't get an answer — the authoritative nameserver is down, the network is partitioned, DNSSEC validation fails — it returns SERVFAIL. Unlike a normal negative response (NXDOMAIN, "this name doesn't exist"), SERVFAIL means "I don't know" — and it propagates through the cache hierarchy in unpredictable ways. Some resolvers cache SERVFAIL for a few seconds. Some retry immediately. Some return the stale cached value (which can actually be desirable). Designing for SERVFAIL resilience is more complex than designing for NXDOMAIN.

Section 03

TTL: the dial that controls staleness vs agility

TTL (Time To Live) is the single most important DNS configuration decision you make, and most teams set it once during initial setup and never think about it again. The default is often 3600s (1 hour) or 86400s (24 hours) — values inherited from an era before blue-green deployments, multi-region failover, and weekly infrastructure changes.

What TTL actually controls

TTL tells resolvers and clients how long to cache a record before re-querying. A 300s TTL means each resolver re-queries every 5 minutes. A 30s TTL means every 30 seconds. The tradeoff is simple: lower TTL → faster change propagation → more DNS query load → slightly higher latency on cache misses.

The key insight engineers miss: TTL is not how long propagation takes, it's the maximum propagation time. Records are served stale for up to TTL seconds after you change them. If your TTL is 3600s and you need to fail over a service, you're looking at up to 60 minutes of traffic still hitting the old IP.

The pre-lowering strategy

The professional DNS migration strategy: lower your TTL well in advance of any planned change, execute the change, then optionally raise it again. The sequence:

  1. At T-24h: lower TTL to 300s (5 minutes). Wait for all caches to expire their 86400s entries. This takes up to 24 hours if your old TTL was 86400s.
  2. At T-0: make the DNS change (new A record, new IP). Now all caches expire in ≤300s. Maximum propagation window: 5 minutes.
  3. At T+10m: verify the change has propagated everywhere. Optionally raise TTL back to 3600s.

If you skip step 1 and try to change a record with a 24-hour TTL, you get 24 hours of split routing — half your users hitting the old server, half hitting the new one. This is how "5-minute DNS change" turns into a 6-hour incident.

Negative TTL: the NXDOMAIN trap

The SOA (Start of Authority) record has a "negative TTL" field — how long resolvers cache NXDOMAIN responses. If you delete a record and then recreate it, resolvers that got the NXDOMAIN will serve it for up to the negative TTL before re-querying. Watch for this when rolling back quickly after an accidental delete.

⚠️
Some resolvers ignore TTL

Many ISP resolvers and corporate DNS servers cap their minimum TTL (often at 300s regardless of what you set), and some ignore your TTL entirely and cache for their own configured duration. Real-world propagation time can exceed your TTL by 10-30 minutes on these resolvers. Always test propagation against multiple resolvers and build your migration timeline with a buffer.

Section 04

Kubernetes DNS: the ndots:5 footgun that burns everyone once

Kubernetes runs CoreDNS as the cluster-internal DNS server. Every pod gets a /etc/resolv.conf that looks roughly like this:

Default /etc/resolv.conf inside a Kubernetes pod CONFIG 
nameserver 10.96.0.10        # CoreDNS cluster IP
search default.svc.cluster.local svc.cluster.local cluster.local
options ndots:5

The ndots:5 option is the culprit. It means: "if the name you're resolving has fewer than 5 dots, try all the search domains first before treating it as an absolute name." For a hostname like api.payments.com (2 dots), the resolver will try these queries in order before succeeding:

  1. api.payments.com.default.svc.cluster.local → NXDOMAIN
  2. api.payments.com.svc.cluster.local → NXDOMAIN
  3. api.payments.com.cluster.local → NXDOMAIN
  4. api.payments.com → A record found ✓

That's 3 failed DNS queries before the one that succeeds. For every external API call your pods make. At 100 requests/second, that's 300 extra DNS queries per second going to CoreDNS — plus the latency of 3 sequential NXDOMAIN round-trips before your actual answer arrives.

Measuring the cost

Each failed query takes 1-10ms (usually 1-2ms on a healthy CoreDNS, but spiky under load). Three extra queries = 3-30ms added to every external API call from your pods, before the TCP handshake even starts. On a service making 50 downstream calls per request, this compounds brutally.

The fixes

Option 1: FQDN (trailing dot). Use fully-qualified domain names. api.payments.com. (trailing dot) has "infinite" dots effectively — the resolver treats it as absolute and skips search domain expansion. Works but requires changing every hostname reference in your code/config.

Option 2: Lower ndots. Set ndots:1 in your pod's dnsConfig. Any name with at least 1 dot is tried as absolute first. This breaks internal Kubernetes service discovery (which uses short names like my-service), so you need to use FQDNs for internal services too (my-service.default.svc.cluster.local).

Option 3: NodeLocal DNSCache. Run a DNS caching agent on each node (as a DaemonSet). Pods query the local agent instead of CoreDNS directly. The local agent caches aggressively and only queries CoreDNS on cache misses. Dramatically reduces CoreDNS load and query latency. This is the production-grade fix for high-traffic clusters.

Fix ndots in your pod spec YAML 
apiVersion: v1
kind: Pod
spec:
  dnsConfig:
    options:
      - name: ndots
        value: "1"   # was 5; now only 1 failed query max for external names
      - name: timeout
        value: "2"   # 2s timeout per query (default 5s)
      - name: attempts
        value: "2"   # 2 attempts (default 2)
  # For internal services: use FQDN in your code
  # "payment-svc" → "payment-svc.payments.svc.cluster.local"
  # Or rely on short names (1 dot = ndots:1 still tries search)

Section 05

DNS-based load balancing: what it can and cannot do

DNS-based load balancing is the oldest form of distributing traffic: return multiple A records for a hostname, and clients round-robin through them. It's elegant, infrastructure-free, and full of subtle failure modes that have burned every major internet company at least once.

How it works (and why it's not really balancing)

When your authoritative server returns multiple A records for api.example.com:

api.example.com.  300  IN  A  10.0.0.1
api.example.com.  300  IN  A  10.0.0.2
api.example.com.  300  IN  A  10.0.0.3

The client picks one and sticks with it until the TTL expires. There's no connection counting, no health awareness, no request rate balancing. A client that connects at the start of a TTL period will use the same IP for the full TTL even if that server dies 10 seconds later.

Worse: many HTTP clients and libraries cache DNS results beyond the DNS TTL — Java's JVM famously cached DNS indefinitely by default until security fixes changed this. Node.js's built-in dns.lookup() caches for the process lifetime in some configurations. The "TTL" you set is aspirational — actual cache duration varies wildly by client.

Weighted routing and geo-routing

Modern DNS providers (Route 53, Cloudflare, NS1) offer weighted routing — return different IPs with different probabilities. Route 53 lets you assign weights: a record with weight 70 gets 70% of queries, weight 30 gets 30%. This is useful for blue-green deployments: start with 5% on the new version, gradually shift weight.

Geo-routing (latency-based or geolocation routing) returns different IPs based on the resolver's geographic location. A user in India hitting api.example.com gets the Mumbai IP; a user in Frankfurt gets the Frankfurt IP. This is how most CDNs and multi-region APIs work at the DNS level.

Health checking at the DNS layer

Route 53 health checks and Cloudflare Load Balancing can actively probe your servers and stop returning an IP when the health check fails. This is DNS-level failover: when the Mumbai server goes down, Route 53 stops returning its IP within 30-60 seconds (depending on health check interval and failure threshold). Effective — but 30-60 seconds of failing traffic while the health check triggers is unacceptable for high-traffic services. Pair with application-level circuit breaking for near-instant failover.

🔴
DNS load balancing is not a substitute for a real load balancer

DNS round-robin distributes clients, not requests. A single client that makes 10,000 requests/second all goes to one server. New connections are distributed; request volume is not. For any service where request volume matters (APIs, databases), use an actual L7 load balancer. DNS lb is best used to distribute clients across regional load balancers, not to balance individual requests.

Section 06

Service mesh & service discovery: when DNS isn't enough

In a microservices environment, services need to find each other dynamically — instances appear and disappear, scale up and down, get replaced by new versions. DNS-based service discovery works for this, but it has a fundamental latency floor: every lookup pays DNS overhead, and health changes propagate at TTL speed.

Why DNS falls short for internal service discovery

Consider a Kubernetes deployment with 10 pods behind a service. When a pod crashes and is replaced, the Kubernetes service's endpoint list updates in seconds. But DNS records (even with 5s TTL) lag behind by up to TTL seconds. During that window, DNS still returns the dead pod's IP. Client gets a connection refused or timeout. This is why Kubernetes Services use kube-proxy (iptables/IPVS rules) for the actual routing, not DNS — DNS in Kubernetes gives you the Service's stable ClusterIP, and kube-proxy handles the actual load balancing to live pods.

Consul and service mesh

HashiCorp Consul extends DNS with health-aware service discovery. Services register themselves with a health check endpoint. Consul polls the health check. Only healthy instances appear in DNS responses. Consul's DNS interface makes this transparent to applications: api.payments.service.consul always resolves to healthy instances.

Envoy / Istio: the sidecar model

Service meshes like Istio (using Envoy as the data plane) take a different approach: instead of relying on DNS for routing, they inject a sidecar proxy into every pod. All outbound traffic from your service goes to the local sidecar (127.0.0.1:port). The sidecar holds a live, health-aware endpoint list synced from the control plane (Pilot/istiod). Routing decisions happen in the sidecar — no DNS involved for the actual connection.

Benefits: instant health failover (no TTL wait), per-request load balancing (not per-client), rich observability (every request traced), mTLS between services automatically. Cost: sidecar overhead (typically 1-3ms per request, 50-200MB memory per pod), operational complexity, and a steep learning curve.

Section 07

DNS failure modes: a taxonomy of production pain

1. The zombie record

You deleted a server, updated the DNS record. But one resolver somewhere (ISP, corporate proxy) ignores TTLs and keeps returning the old IP for hours. Some of your users get connection timeouts. Others work fine. You can't reproduce it internally. This is the zombie record: a stale cache that outlives its TTL. Detection: monitor external resolver propagation. Fix: short TTL, then lower-TTL pre-migration.

2. CNAME chain latency

CNAME records are DNS aliases: api.example.com → api.example.com.cdn.net → actual.cdn.net → 1.2.3.4. Each level requires an additional DNS query. A 3-deep CNAME chain on a cold resolver pays 3× the DNS latency. AWS Elastic Load Balancers and Cloudflare load balancers are commonly reached via CNAMEs. If you have CNAME chains, their depth directly adds to your connection setup time on cold DNS.

3. Split-horizon nightmare

Split-horizon DNS returns different records based on where the query comes from. Internal queries for payments.internal get the internal IP; external queries get the public IP. When misconfigured — or when your developers VPN in and hit the wrong resolver — internal services start calling each other via public IPs, bypassing internal routing, firewall rules, and costing unnecessary egress bandwidth.

4. CoreDNS CPU exhaustion

Kubernetes clusters under high request rates with ndots:5 have killed CoreDNS more than once. At scale, failed search-domain queries accumulate: 100 pods × 50 requests/s × 3 failed queries = 15,000 NXDOMAIN queries/second to CoreDNS. CoreDNS CPU spikes. Query latency climbs from 1ms to 200ms. Every service in the cluster slows down simultaneously. The fix is NodeLocal DNSCache — but understanding the root cause requires knowing about ndots:5.

5. DNS-based DDoS amplification

DNS uses UDP by default. An attacker can spoof the source IP to your server's address, send a small query to open resolvers, and the resolver sends a large response to your server. DNS amplification can achieve 50-70× amplification factors. Mitigations: DNS Response Rate Limiting (RRL) on your authoritative servers, DNSSEC to prevent forging, and blocking open recursion on your resolvers. This is why public DNS resolvers implement strict rate limiting.

Section 08

Production reality: what actually breaks

The migration that went wrong despite "correct" DNS

The most common DNS incident: you change an A record, verify it with dig from your laptop and see the new IP, declare success, and go to lunch. Half your users are still hitting the old server. What happened?

Your laptop queried your ISP's resolver, which had already expired its cache and fetched the new record. The resolver serving your users' mobile devices had just cached the old record 5 minutes ago and won't re-query for another 55 minutes (old TTL: 3600s). Meanwhile the old server is gradually being decommissioned. Users get connection refused.

The fix isn't just checking dig @8.8.8.8. It's pre-lowering TTL 24+ hours in advance, then verifying propagation across multiple resolvers after the change.

Node.js and the getaddrinfo thread pool starvation

Node.js uses libuv's thread pool (default 4 threads) for dns.lookup() calls. Each lookup occupies a thread until it completes. Under high concurrency with slow DNS, all 4 threads get consumed by pending lookups. New lookups queue. Your event loop isn't blocked, but new connections stall waiting for DNS. You see latency spikes that correlate with DNS query rate, not request rate.

The fix: use dns.resolve() instead of dns.lookup() — it uses the Node.js native DNS resolver (not libc getaddrinfo) and doesn't consume thread pool threads. Or switch to undici for HTTP, which manages DNS resolution differently. Or raise the UV_THREADPOOL_SIZE environment variable (max 1024).

Node.js DNS — lookup vs resolve, and thread pool tuning TYPESCRIPT 
import dns from 'dns';
import { promisify } from 'util';

// dns.lookup — uses libc getaddrinfo, blocks a thread pool thread
// Honours /etc/hosts, /etc/nsswitch.conf
// Used by: http.request(), net.connect() by default
const lookup = promisify(dns.lookup);
const { address } = await lookup('api.payments.com');

// dns.resolve4 — uses Node.js native DNS, does NOT use thread pool
// Does NOT honour /etc/hosts (pure DNS only)
// Better for high-concurrency DNS resolution
const resolve4 = promisify(dns.resolve4);
const [address2] = await resolve4('api.payments.com');

// Raise thread pool size for dns.lookup heavy workloads
// Set BEFORE requiring any modules that use it
process.env.UV_THREADPOOL_SIZE = '16'; // default 4, max 1024

// Monitor DNS resolution time in production
const t0 = Date.now();
await lookup('api.payments.com');
const dnsMs = Date.now() - t0;
if (dnsMs > 50) {
  logger.warn({ dnsMs, host: 'api.payments.com' }, 'slow DNS');
}

// Per-hostname caching with TTL (use when making repeated calls)
const dnsCache = new Map();
async function cachedLookup(hostname: string, ttlMs = 30_000) {
  const cached = dnsCache.get(hostname);
  if (cached && Date.now() < cached.expires) return cached.address;
  const [address] = await resolve4(hostname);
  dnsCache.set(hostname, { address, expires: Date.now() + ttlMs });
  return address;
}
Section 09

The things nobody tells you

Connection pooling mostly eliminates DNS cost — but not always

We talked in Day 1 about connection pooling eliminating DNS overhead. That's true when the pool has a warm connection. But pools have idle timeouts. A pool that evicts connections after 60 seconds will re-do DNS when it creates a new connection — and if DNS is slow or misconfigured (ndots:5, no local cache), that eviction-and-reconnect pays full DNS cost. Monitor your pool hit rate and idle timeout behaviour together with DNS latency.

The Java DNS caching trap is still alive in 2024

Java's JVM caches successful DNS lookups forever by default in some configurations (controlled by networkaddress.cache.ttl in java.security). A Java service that resolves db.example.com at startup caches that IP for the process lifetime. When you change the DNS record to fail over to a new database, the Java service never notices. Its connections keep going to the old IP until you restart the process. Always check your JVM's DNS cache TTL in production. Set networkaddress.cache.ttl=30 in Java security properties.

DNSSEC adds latency — and breaks more than you expect

DNSSEC validates that DNS responses haven't been tampered with. Validating resolvers (like Google's 8.8.8.8) verify DNSSEC signatures, which adds computation time and sometimes an extra network round-trip. DNSSEC validation failures (misconfigured signatures, expired keys) produce SERVFAIL responses that are indistinguishable from a network outage from the client's perspective. If you enable DNSSEC on your domain, you must also monitor your DNSSEC configuration — an expired signing key takes down your entire domain with a cryptographic error that's hard to debug under pressure.

Your health check might be resolving DNS, your app is not

Load balancers and orchestrators do health checks by IP (they already resolved the hostname once). Your application reconnects by hostname. If DNS changes and your health checks still pass (same IP), but your app picks up a new IP from DNS that points to a broken backend — the health check passes, users fail. This asymmetry is subtle but real, especially in multi-region setups where DNS geo-routes differently from where the health check originates.

DNS over HTTPS (DoH) hides your DNS from your monitoring

Browsers now support DNS over HTTPS, encrypting DNS queries so your network monitoring can't see them. This is great for user privacy and bad for corporate network monitoring. More relevant for backend: some HTTP clients support DoH. If yours does and it's enabled, your DNS queries bypass your configured nameserver, your split-horizon DNS, and your network-level monitoring. Check if any library you use has DoH enabled by default.
