Production Incident  ·  DNS  ·  Kubernetes ndots

Every API call got 40ms slower after migrating to Kubernetes

Post-migration  ·  Duration: 3 weeks unnoticed  ·  Severity: P2  ·  All external API calls affected

The team had just completed a 3-month Kubernetes migration. Everything worked. No errors. But something felt slow. P99 latency on all external API calls — payment processor, SMS provider, email service — was consistently 40-60ms higher than before the migration. Not breaking. Just slow. The kind of slow that makes you wonder if you're imagining it.

The team spent two weeks investigating the wrong things: pod resource limits, network policies, node proximity to the external services. The actual cause was invisible — happening before the HTTP request even started.

Week 1 post-migration
P99 on external API calls: ~90ms. Pre-migration was ~45ms. Assumed to be "k8s networking overhead."
Week 2
Pod CPU/memory limits raised — no change. Network policy simplified — no change. Node affinity adjusted — no change.
Week 3, Day 1
Senior engineer runs tcpdump inside a pod during an API call. Sees 3 DNS queries before the successful one. All three return NXDOMAIN. Something is expanding the hostname.
Week 3, Day 1
cat /etc/resolv.conf inside the pod: options ndots:5. api.payments.com has 2 dots. Less than 5. The resolver tries api.payments.com.default.svc.cluster.local, api.payments.com.svc.cluster.local, api.payments.com.cluster.local — all NXDOMAIN — before trying api.payments.com.
Week 3, Day 2
Fix: ndots:1 in pod dnsConfig. Latency drops back to ~48ms. CoreDNS NXDOMAIN query rate drops by 75%.
Root cause

Kubernetes sets ndots:5 by default in every pod's /etc/resolv.conf. Any hostname with fewer than 5 dots gets the search domain list appended first before being tried as an absolute name. api.payments.com (2 dots) triggers 3 NXDOMAIN queries before the real query. Each NXDOMAIN costs ~3ms to CoreDNS. At 100 external API calls per second across all pods, this was ~300 wasted DNS queries per second — and 9ms of serial DNS overhead added to every single API call.

The fix — pod dnsConfig in your deployment manifestYAML
❌ Default /etc/resolv.conf inside every Kubernetes pod
nameserver 10.96.0.10   # CoreDNS ClusterIP
search default.svc.cluster.local svc.cluster.local cluster.local
options ndots:5          # ← the problem

✅ Fixed deployment spec
apiVersion: apps/v1
kind: Deployment
spec:
  template:
    spec:
      dnsConfig:
        options:
          - name: ndots
            value: "1"   # try hostname as absolute first if it has ≥1 dot
          - name: timeout
            value: "2"   # 2s per query (default 5s)
          - name: attempts
            value: "2"   # 2 retries (default 2)
      # For internal services: use FQDN
      # "payment-svc" → "payment-svc.default.svc.cluster.local"

      # Or: deploy NodeLocal DNSCache as a DaemonSet
      # Caches DNS on each node — NXDOMAIN responses cached, not repeated
What this incident teaches
▶ndots:5 is the Kubernetes default and it's wrong for external APIs. It was designed for internal service discovery (my-service resolves via search domains). External hostnames pay the NXDOMAIN tax.
▶tcpdump inside a pod is often the fastest path to "what's actually happening on the wire." App logs won't show DNS queries. The network will.
▶Latency that appeared after a platform change (not a code change) is almost always infrastructure. Look at what changed — in this case, /etc/resolv.conf.
▶Connection pooling also fixes ndots-related overhead. If you reuse connections, DNS is only queried once per pool lifetime, not per request. Pool + ndots:1 together are the full fix.
