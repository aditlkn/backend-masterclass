Deep Dive 01

Connection pool internals
& sizing correctly

A connection pool is a cache of established TCP+TLS connections. The concept is simple — the implementation details are where most teams get it wrong: wrong pool size, wrong eviction policy, wrong scheduling strategy, and no visibility into whether the pool is actually being used.

What a pool actually does

When your code calls fetch('https://api.example.com/orders'), the HTTP client checks if the pool has an available connection to api.example.com:443. If yes, it reuses it — skipping DNS, TCP handshake, and TLS handshake. If no, it opens a new connection. When the response is received, the connection goes back to the pool instead of being closed.

The pool has two critical counters: active connections (in use right now) and idle connections (established, waiting). The pool size limits are typically: maxSockets (maximum total connections per host) and maxFreeSockets (maximum idle connections to keep warm).

LIFO vs FIFO scheduling — why LIFO is almost always right

When a request arrives and multiple idle connections are available, which one gets picked? Two strategies:

  • FIFO (First In, First Out): Pick the connection that has been idle the longest. Spreads usage across all connections evenly. Sounds fair — is actually wrong. A connection that's been idle for 59 seconds is on the verge of being dropped by a NAT gateway or firewall. Using it now means it might be dead. Also, a FIFO pool under low traffic rotates through all connections, keeping all of them "warm" even if you only need one.
  • LIFO (Last In, First Out): Pick the most recently used connection. It's the warmest — most likely to still be alive, most likely to have warm TCP buffers. Under low traffic, LIFO concentrates usage on a small set of connections, letting unused ones idle out and be evicted. This is the correct default. Node.js's https.Agent uses LIFO when you set scheduling: 'lifo'.

Sizing: the maxSockets formula

Too small: requests queue waiting for a free connection, adding latency. Too large: you hold more connections than the upstream server's connection limit, causing connection refused errors or starving other clients. The formula:

maxSockets = (target_concurrency × avg_response_time_s) + headroom

Example: you make 200 requests/second to a payment API with a 50ms average response time. At any instant, you have 200 × 0.05 = 10 concurrent in-flight requests. Set maxSockets: 15 (10 + 50% headroom for spikes). Setting it to 100 wastes connections and may overwhelm the upstream. Setting it to 5 creates a queue.

Idle socket eviction and the NAT gateway problem

Idle connections must be evicted before they're killed externally. AWS NAT Gateway drops connections idle for 350 seconds. Corporate firewalls often drop at 60-300 seconds. Your pool's timeout (idle eviction) must be shorter than the shortest external timeout you're behind.

The safe default: evict idle sockets after 30 seconds. This keeps the pool warm for sustained traffic while ensuring connections are retired well before NAT kills them.

Production HTTP client — full configuration with sizing rationaleTYPESCRIPT
import https from 'https';
import { Dispatcher, Pool } from 'undici'; // preferred over axios for Node.js

// undici Pool — modern, no thread pool usage (unlike axios/node-fetch)
const paymentPool = new Pool('https://api.payments.example.com', {
  connections:      15,      // maxSockets — see sizing formula above
  pipelining:       1,       // 1 = no pipelining (safer), 0 = disable
  keepAliveTimeout: 30_000, // evict idle sockets after 30s (< NAT 350s)
  keepAliveMaxTimeout: 30_000,
  connect: {
    rejectUnauthorized: true,
    sessionTimeout:    300,   // TLS session cache TTL in seconds
  },
});

// Monitor pool health — export as metrics
setInterval(() => {
  const { connected, free, queued, size } = paymentPool.stats();
  metrics.gauge('pool.connected', connected, { host: 'payments' });
  metrics.gauge('pool.free',      free,      { host: 'payments' });
  metrics.gauge('pool.queued',   queued,    { host: 'payments' });
  // Alert if queued > 0 for > 5s — pool is undersized or upstream is slow
}, 5000);

// If using axios (legacy): configure keep-alive agent
const httpsAgent = new https.Agent({
  keepAlive:        true,
  maxSockets:       15,
  maxFreeSockets:   5,       // keep 5 idle warm connections
  timeout:          30_000, // idle socket eviction
  scheduling:       'lifo', // always LIFO — uses freshest connection
});
// Pass httpsAgent to axios.create({ httpsAgent })
Queue buildup (pool too small)
Requests wait for a free socket. Visible as P99 latency spikes that correlate with concurrency, not with upstream response time. The pool's queued stat will be non-zero.
→ Increase maxSockets. Alert on queued > 0 sustained for 5+ seconds.
ECONNRESET on reused sockets (idle too long)
The pool returns a socket that NAT or a firewall dropped while it was idle. The first write to the socket fails. Node.js emits ECONNRESET. Usually happens at low-traffic hours when connections sit idle longest.
→ Set keepAliveTimeout to 30s. Deploy tcp_keepalive_time=60 at OS level.
Pool created per-request (the incident)
HTTP client or agent instantiated inside a request handler. Every request pays full DNS + TCP + TLS setup. Pool is garbage-collected after the request. No reuse possible.
→ Move pool/agent to module level. Create once, share forever.
Verify in production
✓Pool free stat is > 0 under normal load — if always 0, pool is undersized or not reusing connections.
✓First request latency vs subsequent request latency — they should differ by DNS+TCP+TLS cost (~150ms). If all requests are slow, pooling isn't working.
✓ss -s on the server: ESTABLISHED count should be stable at roughly maxSockets × number of client instances. Climbing CLOSE_WAIT indicates pool not releasing connections.
✓Zero ECONNRESET errors during 3am low-traffic window — if they appear at night but not during the day, idle timeout is wrong.
Deep Dive 02

TLS session resumption:
eliminating the handshake cost

A full TLS 1.3 handshake costs 1 round-trip (1 RTT). On a 20ms connection to your cloud region, that's 20ms added to every new connection. TLS session resumption eliminates this — subsequent connections to the same server skip the handshake entirely. Most engineers know resumption exists; few understand the two mechanisms or their tradeoffs.

Mechanism 1: Session IDs (TLS 1.2 and earlier)

The server assigns a session ID after a full handshake and stores the session state (negotiated cipher, keys) in server memory. On reconnect, the client presents the session ID. The server looks it up, skips the handshake, and resumes. Simple — but the session cache is server-local. In a multi-instance deployment behind a load balancer, the client might reconnect to a different server instance that doesn't have the session. Cache miss = full handshake again. Session ID resumption requires sticky sessions or a shared session cache (Redis) to work reliably at scale.

Mechanism 2: Session Tickets (TLS 1.2+, primary in TLS 1.3)

The server encrypts the session state and sends it to the client as a "ticket." The client stores the ticket and presents it on reconnect. The server decrypts it and resumes — no server-side storage required. This works across all server instances as long as they share the same ticket encryption key (TLS Session Ticket Key, STEK). nginx and most TLS terminators support configuring a shared STEK across instances.

The critical operational requirement: rotate the STEK regularly (every 24 hours is standard). An exposed STEK can decrypt all past session tickets, potentially revealing session keys. Most nginx defaults don't rotate automatically — you need a cron job or a ticket key management system.

0-RTT (TLS 1.3 early data) — and why to be careful

TLS 1.3 introduced 0-RTT resumption: the client can send application data in the first packet, before the handshake completes. This eliminates the connection setup latency entirely for resumed sessions. The performance gain is real — but so is the risk.

The replay attack: 0-RTT data can be replayed by a network attacker. If the client sends a POST /payments/charge in 0-RTT early data, an attacker who captures that packet can replay it — causing a duplicate charge without the client's knowledge. This is not theoretical. The RFC explicitly prohibits using 0-RTT for non-idempotent requests.

Safe uses of 0-RTT: GET requests, read-only queries, anything idempotent. Never use it for mutations, payments, or anything with side effects unless you've implemented anti-replay protection (which is complex and server-specific).

nginx TLS configuration — session tickets, STEK rotation, 0-RTT controlNGINX
# TLS session resumption — nginx
server {
    listen 443 ssl http2;

    # Session cache (server-side, for session ID resumption)
    ssl_session_cache   shared:SSL:50m;  # 50MB shared across workers
    ssl_session_timeout 1h;              # session valid for 1 hour

    # Session tickets (stateless, client-side storage)
    ssl_session_tickets on;             # enabled by default in nginx
    ssl_session_ticket_key /etc/nginx/ticket.key;  # shared STEK
    # Rotate via cron: openssl rand 80 > /etc/nginx/ticket.key && nginx -s reload
    # Rotate every 24h. All nginx instances must use the same key file.

    # TLS 1.3 only (disables 1.2 — do this only if all clients support 1.3)
    ssl_protocols TLSv1.3;
    ssl_prefer_server_ciphers off;

    # 0-RTT: disabled by default in nginx. Enable only for GET-heavy APIs.
    # ssl_early_data on;  ← dangerous for POST/mutation endpoints
    # If you enable it, check: $ssl_early_data == "1" in POST handlers
    # and reject those requests — they may be replays.
}

# Verify session resumption with openssl
openssl s_client -connect api.example.com:443 -reconnect 2>&1 | grep "Reused"
# Output: "Reused, TLSv1.3, Cipher is TLS_AES_256_GCM_SHA384"
# If you see "New" instead of "Reused" — resumption is not working
Verify TLS resumption is working
✓openssl s_client -connect host:443 -reconnect 2>&1 | grep -E "New|Reused" — should show "Reused" on second connection.
✓Check nginx stats: ssl_session_reuses counter should be increasing. If always 0, resumption is broken.
✓All nginx instances use the same ticket key file — otherwise cross-instance resumption fails.

Deep Dive 03

HTTP/2 HOL blocking:
the problem multiplexing didn't fully solve

HTTP/2's multiplexing eliminated HTTP/1.1's head-of-line blocking at the application layer. But it reintroduced it at the TCP layer — and in some conditions, this makes HTTP/2 worse than HTTP/1.1. Understanding exactly where the blocking happens tells you when HTTP/3 is worth the operational cost.

What HTTP/1.1 HOL blocking was

HTTP/1.1 without pipelining: one request per connection at a time. To make 6 requests in parallel, browsers open 6 connections. Pipelining (sending multiple requests without waiting for responses) was specified but widely disabled because of HOL blocking: if the first request's response is slow, all subsequent pipelined responses are held up behind it — even if they're ready to send. The pipeline is a queue; the head blocks the line.

What HTTP/2 fixed (application-layer HOL)

HTTP/2 multiplexes multiple streams over a single TCP connection. Each stream is independent at the HTTP layer — stream 3's response doesn't wait for stream 1's response to finish. The server can interleave frames from different streams freely. This eliminates application-layer HOL blocking.

What HTTP/2 didn't fix (TCP-layer HOL)

TCP guarantees ordered, reliable delivery. Every byte arrives in sequence. If a TCP segment is lost in transit, TCP halts delivery of all subsequent segments until the lost segment is retransmitted and received. This is called TCP HOL blocking — and it affects all streams on the connection simultaneously.

In HTTP/1.1 with 6 parallel connections, a packet loss on connection 1 only blocks connection 1. The other 5 connections continue normally. In HTTP/2 with 1 multiplexed connection, a single packet loss blocks all streams on that connection — potentially all 100 concurrent requests. Under lossy network conditions (mobile, congested links), HTTP/2 can actually be slower than HTTP/1.1 because a single loss event stalls everything.

HTTP/3 and QUIC: the actual fix

HTTP/3 runs over QUIC instead of TCP. QUIC implements streams natively at the transport layer — a packet loss only blocks the specific stream that was carrying that packet. Other streams continue unaffected. QUIC also eliminates the TCP + TLS handshake (merged into a single 1-RTT or 0-RTT handshake), and handles connection migration (IP address changes don't break the connection — useful for mobile clients switching between WiFi and cellular).

The operational cost: QUIC runs over UDP, which many corporate firewalls block or deprioritise. HTTP/3 clients fall back to HTTP/2 when QUIC is unavailable — but this fallback adds latency. For internal service-to-service communication on reliable datacenter networks, TCP HOL blocking is rarely a real problem. For consumer-facing APIs serving mobile clients on lossy networks, HTTP/3 is a genuine win.

When HTTP/2 is actually worse

A 1% packet loss rate on a 100-stream HTTP/2 connection means roughly 1 loss event per second. Each loss stalls all 100 streams while TCP retransmits. On 6 HTTP/1.1 connections with 1% loss, each connection has a 1% independent loss rate — the other 5 continue. Measure actual packet loss on your client paths before assuming HTTP/2 is always better. On reliable internal networks: HTTP/2 wins clearly. On mobile consumer traffic with 2-5% loss: the benefit is much smaller and HTTP/3 is worth investigating.

Enabling HTTP/3 in nginx + diagnosing HOL blockingBASH
# nginx with HTTP/3 support (nginx 1.25+ or nginx-quic build)
# server {
#   listen 443 ssl http2;       # HTTP/2 for TCP clients
#   listen 443 quic reuseport;  # HTTP/3 for QUIC clients
#   add_header Alt-Svc 'h3=":443"; ma=86400'; # advertise HTTP/3
# }

# Diagnose: what protocol is actually being used?
curl -I --http2 https://api.example.com/healthz
# Look for: HTTP/2 200 in the response — confirms HTTP/2

curl -I --http3 https://api.example.com/healthz
# Requires curl built with HTTP/3 support (brew install curl --with-quic)

# Check if TCP retransmits are high (indicator of HOL blocking risk)
ss -s
# Look at "retransmitted" counter — high relative to "sent" = lossy path

# Wireshark: filter for TCP retransmissions on a connection
tcp.analysis.retransmission && ip.addr == <server_ip>

# nginx: confirm HTTP/2 stream multiplexing is working
curl -v --http2 https://api.example.com 2>&1 | grep "stream"
Verify in production
✓Confirm HTTP/2 is actually in use: curl -I --http2 https://yourapi.com — response line should start with HTTP/2.
✓Check TCP retransmit rate on server: netstat -s | grep retransmit. Above 0.1% of total segments is worth investigating.
✓For consumer mobile APIs: measure P99 vs P50 latency ratio. A ratio above 10× suggests HOL blocking on lossy paths.
✓If enabling HTTP/3: verify the Alt-Svc header is returned, and monitor QUIC vs TCP connection ratio in access logs.
