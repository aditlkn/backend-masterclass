Production Incident  ·  HTTP  ·  Connection Pooling

P99 jumped from 8ms to 4 seconds after a routine deploy

Monday 14:23 IST  ·  Duration: 2h 40m  ·  Severity: P1  ·  ~200 upstream calls/sec affected

The team had just deployed a "minor config refactor" — no logic changes, just cleaning up how the HTTP client was initialised. Within 90 seconds of the deploy, P99 latency on the payments service went from 8ms to 3.8 seconds. The service wasn't down. Requests were completing. Just slowly. Very slowly.

Error rate was zero. The upstream payment processor was healthy. CPU on all pods was normal. Memory normal. The only signal was latency — and it was uniformly bad, not spiky. Every single request was slow, not just some of them.

14:23 IST
Deploy completes. P99 latency alert fires: 3.8s (was 8ms). On-call engineer begins investigation.
14:31 IST
Upstream payment processor checked — fully healthy, their P99 is still 6ms. Problem is not downstream.
14:45 IST
curl from inside a pod to the payment processor: 3.7s. Same curl a second time: 7ms. First request slow, all subsequent fast.
14:52 IST
curl -w "dns:%{time_namelookup} tcp:%{time_connect} tls:%{time_appconnect} ttfb:%{time_starttransfer}" — TLS time: 180ms on first request, 0ms on second. DNS+TCP+TLS paid every time.
15:08 IST
Engineer reviews the config refactor diff. The old code: keepAlive: true. The new code: HTTP client created fresh on every request. No connection pool. No keep-alive. Every request: DNS → TCP → TLS → request → response → close.
16:03 IST
Fix deployed: HTTP client created once at startup, shared across requests. P99 returns to 8ms. 2h 40m of 3.8s latency resolved.
Root cause

The "config refactor" moved the HTTP client instantiation from module-level (created once, reused) to inside the request handler (created fresh per request). Without keepAlive: true and a connection pool, every outbound HTTP call paid the full setup cost: DNS lookup (~2ms) + TCP handshake (1 RTT ~20ms) + TLS 1.3 handshake (1 RTT ~20ms) + request. At 200 calls/sec to the payment processor, the connection setup overhead dominated completely. The processor itself was responding in 6ms — but getting to the processor was taking 180ms each time.

The diff that caused 2h 40m of 3.8s latencyTYPESCRIPT
import http from 'http';
import https from 'https';

// ❌ INCIDENT: client created inside the request handler
async function chargePayment(amount: number) {
  // New client on every call = no connection reuse
  const client = axios.create({ baseURL: 'https://pay.example.com' });
  return client.post('/charge', { amount });
  // Every call: DNS → TCP → TLS → request → close. ~180ms overhead.
}

// ✅ FIX: client created once at module level, shared across all calls
const keepAliveAgent = new https.Agent({
  keepAlive:            true,
  maxSockets:           50,     // pool size per host
  maxFreeSockets:       10,     // idle connections to keep warm
  timeout:              30000, // socket timeout
  scheduling:           'lifo', // reuse most-recent socket (stays warm)
});

const paymentClient = axios.create({
  baseURL:  'https://pay.example.com',
  httpsAgent: keepAliveAgent,   // reuse connections across all calls
  timeout:  30000,
});

async function chargePayment(amount: number) {
  return paymentClient.post('/charge', { amount });
  // First call: DNS → TCP → TLS → request. ~180ms.
  // All subsequent calls: request only. ~6ms.
}
What this incident teaches
▶Connection setup cost is invisible in development. You call an API once to test it and it works. You don't notice the 180ms because it's a one-time cost. Under 200 req/sec in production, every request pays that cost.
▶"First request slow, subsequent fast" is the tell. If you see this pattern — either from curl or in your latency histogram — it's connection setup. Not your handler, not your DB, not the upstream.
▶HTTP clients should be module-level singletons. Creating an axios/fetch/got/undici client inside a request handler is almost always wrong. Create once, reuse everywhere.
▶Always run curl -w timing on slowness. The breakdown of dns/tcp/tls/ttfb/total tells you exactly which phase is slow in under 10 seconds. This is always step one.
