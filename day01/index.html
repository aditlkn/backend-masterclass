Section 01

The mental model most engineers are missing

Here's a confession: for the first few years of my career I thought of HTTP as basically a function call. You send a request, you get a response. The network is the wire in between. That model is fine until the day you're debugging why your API is taking 800ms when the database query is only 20ms — and you realise you have no idea what's happening in those other 780ms.

HTTP is not a single thing. It's a layered stack of protocols, each of which adds cost, and each of which you can optimise independently. When a client makes a request to your server, here's what actually happens at minimum:

  • DNS resolution — the hostname gets resolved to an IP. This is a round-trip to a DNS resolver that can add anywhere from 0ms (cached) to 200ms (cold, remote resolver).
  • TCP connection — a 3-way handshake before a single byte of HTTP is sent. Minimum one round-trip time (RTT).
  • TLS handshake — if HTTPS (and it should always be HTTPS), add one or two more RTTs before the first encrypted byte.
  • HTTP request/response — only now does your actual application protocol run.

On a mobile device in India hitting a server in Mumbai, each RTT might be 40-80ms. That means before your server has processed a single byte, you've potentially spent 200-400ms in protocol overhead. Your 5ms handler is irrelevant.

💡
The mental model shift

Stop thinking about HTTP latency as "server processing time." Start thinking about it as: DNS + TCP + TLS + Server + Transfer. Each term is independently measurable and independently optimisable. Most backend engineers only ever optimise the "Server" term.

Section 02

The full request lifecycle, phase by phase

Let's walk through exactly what happens when a Node.js service calls fetch('https://api.payments.com/v1/charge').

Phase 1: DNS Resolution

Your OS has a stub resolver. It checks its local cache first (TTL-controlled). If not cached, it queries the recursive resolver (usually your ISP's or Google's 8.8.8.8). The recursive resolver walks the DNS tree: root → .com → payments.com → api.payments.com. Each hop is a network round-trip. A cache miss on a cold server can easily add 50-150ms before you've opened a single socket.

The critical implication: HTTP clients that re-resolve DNS on every request are burning 50-150ms per request unnecessarily. This is why connection pooling (which reuses existing TCP connections and therefore skips DNS + TCP + TLS) is so valuable — it's not just about TCP setup cost, it's about avoiding the entire pre-HTTP stack.

Phase 2: TCP Three-Way Handshake

TCP is a reliable, connection-oriented protocol. Before any data flows, client and server negotiate:

  • Client sends SYN (synchronise) with its initial sequence number
  • Server replies SYN-ACK with its own sequence number
  • Client sends ACK — connection established

This takes exactly 1 RTT. At 50ms RTT, that's 50ms before a byte of application data moves. This is why TCP connections are expensive to create and valuable to reuse.

There's also a subtlety: TCP's slow-start algorithm means a new connection starts with a congestion window of ~14KB and ramps up exponentially. If your response is 100KB, a new connection will deliver it slower than a warmed-up connection, even if RTT is the same.

Phase 3: TLS Handshake

TLS 1.2 adds 2 RTTs of handshake before any application data. TLS 1.3 reduces this to 1 RTT for a new connection and 0 RTT for session resumption. The handshake involves:

  • Negotiating cipher suites and protocol version
  • Certificate exchange and verification (including OCSP for revocation checking)
  • Key exchange (ECDHE in modern TLS)
  • Deriving session keys

On a 50ms RTT link: TLS 1.2 costs 100ms. TLS 1.3 costs 50ms. TLS 1.3 with session resumption costs 0ms. This is a dramatic difference that most teams leave on the table because they never check which TLS version their servers are negotiating.

Phase 4: HTTP Request & Response

Only here does your application protocol run. The request is serialised (method, URL, headers, optional body), sent over the TLS-encrypted TCP connection, processed by your server, and a response is sent back.

Section 03

HTTP/1.1 vs HTTP/2 vs HTTP/3 — the real differences

This isn't just version trivia. Each version was designed to solve specific, real performance problems. Understanding what problem each solved helps you know when you're actually benefiting from it.

HTTP/1.1: The good and the terrible

HTTP/1.1 introduced persistent connections (keep-alive) — a TCP connection could be reused for multiple requests. Before this, HTTP/1.0 opened a new TCP connection for every single request. That was fine in 1996 when pages had 3 images. By 2005, with 50+ assets per page, it was catastrophic.

But HTTP/1.1's fundamental flaw is head-of-line blocking. With pipelining (which barely anyone implements correctly), requests must be processed and responded to in order. If request #3 is a slow database query, requests #4, #5, and #6 are all blocked waiting for it, even if they could be served in milliseconds.

The browser workaround? Open 6 parallel TCP connections per hostname. This is why domains like img1.cdn.com, img2.cdn.com existed — domain sharding to get more parallel connections. An ugly hack that became a best practice.

HTTP/2: Multiplexing changes everything

HTTP/2's core innovation is multiplexing over a single TCP connection. Instead of text-based messages, HTTP/2 uses a binary framing layer where requests and responses are broken into frames, each tagged with a stream ID. Multiple streams can be interleaved on a single connection.

The implications:

  • No more domain sharding — one connection is now better than six because the server can prioritise streams
  • Header compression (HPACK) — HTTP headers are repetitive (same Authorization, Accept, Cookie on every request). HPACK compresses headers using a shared dictionary. On an API making many calls, this can reduce header size by 80-90%.
  • Server push — server can proactively send resources before the client asks. In practice, this feature is mostly unused because browsers got smarter about preloading.
  • Stream prioritisation — clients can signal which responses matter more. A CSS file that blocks rendering can be prioritised over a tracking pixel.

But HTTP/2 has a critical flaw: it solved HTTP-level head-of-line blocking, but TCP-level head-of-line blocking still exists. TCP is a reliable byte stream — if a single packet is lost anywhere in the stream, all streams on that connection are stalled until that packet is retransmitted. On lossy networks (mobile, WiFi), HTTP/2 can actually perform worse than HTTP/1.1 with multiple connections.

HTTP/3: QUIC changes the transport layer

HTTP/3 runs on QUIC, a transport protocol built on UDP, developed by Google and standardised by the IETF. QUIC bakes TLS 1.3 directly into the connection setup, achieving 0-RTT connection establishment for returning clients. More importantly, QUIC implements per-stream reliability — a lost packet only stalls the specific stream it belongs to, not all streams.

QUIC also handles connection migration: if you switch from WiFi to cellular, a QUIC connection can survive the IP address change (using connection IDs instead of 4-tuple IP+port identification). This is transformative for mobile apps.

The catch: QUIC uses UDP, and many corporate firewalls and middleboxes block or throttle UDP. Browsers fall back to HTTP/2 automatically when this happens. For your backend APIs over UDP-permissive networks, HTTP/3 is genuinely faster, especially for mobile clients.

Section 04

TLS: what actually happens in that handshake

Most backend engineers treat TLS as a checkbox: "yes, I have HTTPS." But the TLS handshake is a complex cryptographic negotiation that has huge performance implications, several common misconfiguration failure modes, and a history of vulnerabilities that came from protocol-level decisions.

TLS 1.3 handshake, step by step

TLS 1.3 (RFC 8446, 2018) dramatically simplified the handshake compared to TLS 1.2. Here's what actually happens:

  1. ClientHello — Client sends: supported TLS version, a random nonce, supported cipher suites, and key shares for supported elliptic curves (ECDHE). This last part is what makes TLS 1.3 faster — the client sends its key material on the first message rather than waiting for server agreement.
  2. ServerHello + {EncryptedExtensions + Certificate + CertificateVerify + Finished} — Server sends its chosen cipher, its own key share, and immediately after (now encrypted with the derived handshake key) the certificate chain, a signature, and a Finished message. This is 1 RTT from ClientHello.
  3. Client Finished + First Application Data — Client verifies the certificate, sends its Finished message, and can immediately send application data in the same flight. Effective: 1 RTT to first request byte.

Session resumption and 0-RTT

When a client has previously connected to a server, TLS 1.3 supports session resumption via a Pre-Shared Key (PSK). The server issues a session ticket after the initial handshake. On reconnect, the client includes the PSK in its ClientHello, and the server can respond with application data immediately — 0 RTT to first byte.

The security caveat: 0-RTT data is vulnerable to replay attacks. An attacker who captures a 0-RTT ClientHello can replay it to re-execute the first request. For this reason, 0-RTT should only be used for idempotent, safe requests (GET, HEAD). Never for POST requests that trigger state changes. Cloudflare, for example, will only forward 0-RTT for GET requests.

Certificate verification: the hidden cost

Verifying a certificate chain involves:

  • Checking the certificate's signature against the issuing CA's public key
  • Checking the CA's certificate against its parent (chain building)
  • Checking for revocation — by default via OCSP (Online Certificate Status Protocol), which is a synchronous network request to the CA's OCSP responder

That OCSP request is a surprise network call you didn't know you were making. If your CA's OCSP responder is slow or unreachable, your TLS handshakes get slow or fail. The solution is OCSP Stapling: your server periodically fetches its own OCSP response and includes ("staples") it in the TLS handshake, eliminating the client's need to make a separate OCSP request. Almost all production web servers support this; almost no one enables it.

nginx — enabling TLS 1.3, OCSP stapling, and session tickets NGINX CONFIG 
server {
    listen 443 ssl http2;

    # Modern TLS only — no 1.0 or 1.1
    ssl_protocols TLSv1.2 TLSv1.3;

    # TLS 1.3 cipher suites are fixed by the spec
    # For TLS 1.2 compatibility, restrict to these:
    ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384;
    ssl_prefer_server_ciphers off; # Let client pick in TLS 1.3

    # Session resumption — reuses handshake state for returning clients
    ssl_session_cache shared:SSL:10m;
    ssl_session_timeout 1d;
    ssl_session_tickets off; # Tickets have forward secrecy issues; use cache

    # OCSP stapling — eliminates client's OCSP round-trip
    ssl_stapling on;
    ssl_stapling_verify on;
    resolver 8.8.8.8 8.8.4.4 valid=300s;

    # HSTS — tells browser to always use HTTPS for 2 years
    add_header Strict-Transport-Security "max-age=63072000" always;
}

Section 05

Keep-Alive & connection pooling: the single biggest quick win

I cannot overstate how much latency you can eliminate by simply reusing HTTP connections. I've seen APIs go from 400ms average to 60ms average purely by enabling connection pooling in the HTTP client. No server changes. No database tuning. Just not throwing away connections after each request.

How keep-alive works

In HTTP/1.1, connections are persistent by default (Connection: keep-alive). After a request/response cycle, the connection is held open and returned to a pool. The next request to the same host can skip DNS + TCP + TLS and go straight to sending the HTTP request.

The TCP connection already has its congestion window warmed up. You're essentially skipping 1-3 RTTs and multiple milliseconds of cryptographic setup. For an API that makes 10 requests per user action, this compounds dramatically.

Node.js: where the default bites you

Here's the thing about Node.js's built-in fetch and older http.request: by default, connections are not pooled. Every request creates a new TCP connection. In 2024 this is still biting people who use node-fetch or raw https.request without an agent.

Node.js — connection pooling with undici (the right way) TYPESCRIPT 
import { Pool } from 'undici';

// Create one pool per upstream origin — reuse across all requests in the process
// undici is what Node.js fetch() uses internally since Node 18
const paymentsPool = new Pool('https://api.payments.internal', {
  // Max simultaneous connections to this origin
  connections: 10,

  // Keep connections alive for up to 60s after last use
  // Server must agree (its keepalive timeout must be higher)
  keepAliveTimeout: 60_000,

  // Per-phase timeouts — NOT a single "timeout"
  headersTimeout: 5_000,   // time to receive response headers
  bodyTimeout: 30_000,      // time to receive full body
  connectTimeout: 2_000,    // TCP connection timeout
});

export async function chargePayment(payload: ChargePayload) {
  const { statusCode, body } = await paymentsPool.request({
    path: '/v1/charge',
    method: 'POST',
    headers: {
      'content-type': 'application/json',
      'idempotency-key': payload.idempotencyKey,
    },
    body: JSON.stringify(payload),
  });

  if (statusCode !== 200) {
    throw new PaymentError(statusCode, await body.text());
  }
  return body.json();
}

// When your process shuts down gracefully:
process.on('SIGTERM', async () => {
  await paymentsPool.close();
});

The keepalive timeout mismatch bug

This is a production bug that hits everyone eventually. Here's the scenario:

  • Your Node.js client has a keep-alive timeout of 60 seconds — it holds connections open for 60s
  • Your upstream server (nginx, AWS ALB) has a keep-alive timeout of 30 seconds — it closes idle connections after 30s
  • A connection sits idle for 31 seconds. The server closes it. Your client doesn't know yet.
  • A new request comes in, your client grabs the "alive" connection from the pool and sends the request
  • The server responds with a TCP RST (connection reset) because the connection was already closed
  • You see an intermittent ECONNRESET error in production with no obvious cause

The fix: Always set your client's keep-alive timeout to be meaningfully shorter than the server's. If the server closes at 60s, set the client to 45s. This way, the client proactively retires connections before the server does, eliminating the race condition.

⚠️
AWS ALB keep-alive timeout is 60 seconds

If you're behind an AWS Application Load Balancer, its idle keep-alive timeout defaults to 60 seconds. If your Node.js service's pool hold connections for 60 seconds, you will see intermittent ECONNRESET under load. Set your client keep-alive to 45-50 seconds maximum. This one setting fixes a class of mysterious 5xx errors that teams spend days debugging.

Section 06

Headers that actually matter in production

There are hundreds of HTTP headers. Most of them you'll never think about. Here are the ones that directly affect reliability, performance, and security — and how they interact in ways most engineers don't realise.

Content-Type and the sneaky parsing bugs

Always send Content-Type: application/json; charset=utf-8 on JSON responses. The charset=utf-8 part matters: without it, some older HTTP clients fall back to ISO-8859-1 and misparse non-ASCII characters. On a global API serving users from India, Japan, or anywhere with non-Latin scripts, this matters.

Also: always validate the incoming Content-Type on your server. An API that accepts Content-Type: text/plain and parses it as JSON is susceptible to a class of CSRF attacks (cross-site form posts can't set Content-Type: application/json).

Cache-Control: the most important header you're probably getting wrong

Cache-Control controls caching at every layer: browser cache, reverse proxy, CDN. The directives that matter:

  • no-store — don't cache at all. Use for authenticated API responses that are user-specific and change frequently.
  • no-cache — misleadingly named. The response can be cached, but must be revalidated with the origin before use. Good for resources that change infrequently but must always be current.
  • private — can be cached by the browser, not by a shared cache (CDN/proxy). Use for personalised responses.
  • public, max-age=3600 — can be cached anywhere for 1 hour. Use for static-ish API responses (config endpoints, feature flags).
  • stale-while-revalidate=60 — the underrated one. Serve stale content for up to 60s while refreshing in the background. Nearly eliminates cache miss latency spikes.
Express.js — setting appropriate Cache-Control per response type TYPESCRIPT 
import express from 'express';

const app = express();

// Authenticated user data — never cache, user-specific
app.get('/api/v1/profile', auth, (req, res) => {
  res.set('Cache-Control', 'no-store');
  res.json(userProfile);
});

// Public, slow-changing data (product catalogue)
// CDN can cache for 5min; serve stale for 30s while refreshing
app.get('/api/v1/products', (req, res) => {
  res.set('Cache-Control', 'public, max-age=300, stale-while-revalidate=30');
  res.json(products);
});

// Static assets — cache for 1 year (content-addressed filenames)
app.use('/static', express.static('public', {
  maxAge: '1y',
  immutable: true,  // tells browser this URL will never change
}));

// Conditional requests — ETag-based revalidation
// Client sends: If-None-Match: "abc123"
// Server returns 304 Not Modified if unchanged (saves bandwidth)
app.get('/api/v1/config', (req, res) => {
  const etag = computeEtag(configData);
  if (req.headers['if-none-match'] === etag) {
    return res.status(304).end();
  }
  res.set('ETag', etag);
  res.set('Cache-Control', 'public, max-age=60');
  res.json(configData);
});

Security headers: the ones your nginx should set

These should come from your reverse proxy, not your application code. Set them once, apply everywhere:

nginx — production security headers NGINX CONFIG 
# In your nginx server {} block or http {} block

# Prevent browsers sniffing content-type
add_header X-Content-Type-Options "nosniff" always;

# Block clickjacking
add_header X-Frame-Options "SAMEORIGIN" always;

# Force HTTPS for 2 years, include subdomains
add_header Strict-Transport-Security "max-age=63072000; includeSubDomains" always;

# Content Security Policy — most impactful XSS mitigation
# Start with report-only mode to understand your content sources:
add_header Content-Security-Policy-Report-Only
  "default-src 'self'; script-src 'self'; report-uri /csp-report" always;

# Referrer policy — don't leak full URLs to third parties
add_header Referrer-Policy "strict-origin-when-cross-origin" always;

# Permissions policy — disable features you don't use
add_header Permissions-Policy "camera=(), microphone=(), geolocation=()" always;

# Correlation ID — your most valuable debugging header
# Set at the edge, propagate through every service
map $http_x_request_id $request_id_final {
    default   $http_x_request_id;
    ""        $request_id;  # generate if not provided
}
add_header X-Request-Id $request_id_final always;
proxy_set_header X-Request-Id $request_id_final;

Compression: gzip vs Brotli, and when it backfires

Compression is one of the highest-ROI optimisations in HTTP. A typical JSON API response compresses 60-80% with gzip, 70-85% with Brotli. The tradeoff is CPU time for compression — but for most APIs, this is negligible compared to bandwidth savings, especially for mobile clients.

gzip vs Brotli

gzip is universal — every HTTP client supports it. Compression levels 1-9: level 1 is fast/low ratio, level 9 is slow/high ratio. For dynamic content (API responses), level 5-6 is the sweet spot: most of the compression benefit with a fraction of the CPU cost.

Brotli (br) is Google's algorithm, supported by all modern browsers and HTTP clients since ~2017. At equivalent compression levels, Brotli achieves 15-25% better compression ratio than gzip. For static assets, pre-compress with Brotli level 11 and cache the result. For dynamic content, Brotli level 4-5 is comparable speed to gzip level 6 with better compression.

When compression makes things worse

Never compress content that's already compressed: JPEG, PNG, WebP, MP4, zip files. Trying to compress them wastes CPU and often makes them larger. The test: if Content-Type is an image, video, or archive type, skip compression.

Also: don't compress tiny responses. Compressing a 50-byte JSON response adds headers that are larger than the savings. The threshold I use: only compress responses larger than 1KB.

Node.js — compression middleware with Brotli and smart thresholds TYPESCRIPT 
import compression from 'compression';
import { createBrotliCompress, constants } from 'zlib';

// Option 1: Use nginx for compression (preferred in production)
// nginx handles this more efficiently than Node.js
// In nginx.conf:
// gzip on; gzip_min_length 1000; gzip_types application/json text/plain;
// brotli on; brotli_min_length 1000; brotli_types application/json text/plain;

// Option 2: Node.js compression (useful when nginx isn't in the stack)
app.use(compression({
  // Only compress responses larger than 1KB
  threshold: 1024,

  // Skip compression for pre-compressed types
  filter: (req, res) => {
    const type = res.getHeader('Content-Type') || '';
    if (/image|video|audio|zip|gzip|br/.test(type)) return false;
    return compression.filter(req, res);
  },

  level: 5, // gzip level 5: good ratio, reasonable CPU cost
}));

// Measuring compression effectiveness in production
app.use((req, res, next) => {
  const start = process.hrtime.bigint();
  res.on('finish', () => {
    const encoding = res.getHeader('content-encoding');
    const original = res.getHeader('x-original-size');
    const compressed = res.getHeader('content-length');
    if (encoding && original && compressed) {
      const ratio = (1 - compressed / original) * 100;
      metrics.histogram('http.compression.ratio', ratio, { encoding });
    }
  });
  next();
});

Section 08

Timeouts done right — the thing everyone gets wrong

Here is the single most common backend bug I've seen in production systems: a service has a "30 second timeout" configured somewhere, and yet requests hang for 5 minutes before failing. This happens because most engineers set one timeout and think they're covered. You're not.

There are at least 4 distinct timeouts in an HTTP request

  1. DNS timeout — how long to wait for DNS resolution. Default is often system-controlled and can be 30+ seconds on a misconfigured resolver. Set explicitly: 2-5 seconds.
  2. Connect timeout — how long to wait for the TCP handshake to complete. If a server is overloaded and not accepting connections, its TCP accept queue fills up and SYNs are dropped. Set: 2-5 seconds.
  3. TLS timeout — how long to wait for the TLS handshake. Can hang if the server is under severe CPU load. Usually covered by the connect timeout in most libraries, but worth verifying.
  4. Read/response timeout — how long to wait for the response after the request is sent. This is what most people set. But there are actually two sub-timeouts: time to first byte (server starts responding) and time to last byte (full response received). A server that streams a large response slowly can be "responding" but take minutes to finish.

Setting timeouts in undici (Node.js)

Comprehensive timeout configuration — never use a single "timeout" TYPESCRIPT 
import { Pool } from 'undici';

const pool = new Pool('https://upstream.service.internal', {
  // Time to establish TCP connection (includes DNS in undici)
  connectTimeout: 3_000,

  // Time from sending request to receiving response HEADERS
  // This catches a slow/overloaded server that hasn't started responding
  headersTimeout: 10_000,

  // Time from receiving headers to receiving the complete body
  // This catches a server that starts responding but then stalls
  bodyTimeout: 30_000,

  // Keep-alive timeout (must be shorter than server's)
  keepAliveTimeout: 45_000,
  keepAliveMaxTimeout: 45_000,
});

// Per-request timeout override for critical paths
const { statusCode, body } = await pool.request({
  path: '/v1/critical-operation',
  method: 'POST',
  headers: { 'content-type': 'application/json' },
  body: JSON.stringify(payload),

  // Override for this specific request — tight SLA
  headersTimeout: 3_000,
  bodyTimeout: 5_000,
});

// Axios equivalent (for those who use it)
const axiosClient = axios.create({
  baseURL: 'https://upstream.service.internal',
  timeout: 10_000,  // Axios only supports a single timeout :(
  // This covers time from request sent to response fully received
  // It does NOT separately control connect vs read timeout
  // For better control, use undici or node-fetch with AbortController
  httpAgent: new http.Agent({ keepAlive: true, keepAliveMsecs: 45_000 }),
  httpsAgent: new https.Agent({ keepAlive: true, keepAliveMsecs: 45_000 }),
});

Production reality: what actually bites you

The 502/504 gateway timeout that isn't your server

You're behind a load balancer. Your server is healthy. Users are seeing intermittent 502 or 504 errors. Your first instinct is to look at server logs. Server logs show nothing — the request never arrived.

What's happening: the load balancer has a request timeout (AWS ALB defaults to 60 seconds). Your server's response takes 61 seconds. ALB drops the connection and returns a 504 to the client. Your server finishes processing and writes the response to a closed socket, which silently disappears.

The fix has two parts: first, your server's timeout must be shorter than the LB's timeout, so your server fails fast and returns a proper error before the LB gives up. Second, fix the underlying slow query rather than just adjusting timeouts.

Chunked transfer encoding surprises

When a server doesn't know the total response size (streaming responses, server-sent events), it uses chunked transfer encoding: Transfer-Encoding: chunked. Instead of a fixed Content-Length, the body is sent as a series of sized chunks, terminated by a zero-length chunk.

This matters because: reverse proxies buffer chunked responses differently. nginx buffers the full response before forwarding to the client by default (proxy_buffering on). For streaming APIs (LLM completions, real-time dashboards), you need to disable this: proxy_buffering off. Otherwise your "streaming" API isn't actually streaming — it's being buffered by nginx and delivered as one big response.

The HTTP/2 reset attack (CVE-2023-44487)

In late 2023, a novel DDoS attack was disclosed: HTTP/2 Rapid Reset. Attackers open a stream and immediately send a RST_STREAM frame, cancelling it. Because the server has to process each stream establishment even if immediately cancelled, an attacker can generate enormous server load with a relatively small number of connections. Cloudflare, Google, and AWS all absorbed multi-terabit attacks using this technique.

The mitigations are in your HTTP server library — update nginx (1.25.3+), update your Node.js/Go/Python HTTP libraries. This is why staying current on HTTP server versions matters even for "stable" production systems.

Measuring what matters: TTFB vs total time

Time To First Byte (TTFB) is the time from request sent to first byte of response received. It measures server processing time + 1 RTT. This is what your backend latency metrics should focus on.

Total time includes transfer time and depends on response size. A 10MB download will have great TTFB but terrible total time if the client is on a slow connection. Don't conflate the two. Track both separately.

Instrumenting HTTP client calls with detailed timing in Node.js TYPESCRIPT 
import { Pool, interceptors } from 'undici';
import { metrics } from './telemetry';

function createInstrumentedPool(origin: string, serviceName: string) {
  const pool = new Pool(origin, {
    connections: 10,
    connectTimeout: 3_000,
    headersTimeout: 10_000,
    bodyTimeout: 30_000,
  });

  // undici diagnostics channel for precise timing
  const { DiagnosticsChannel } = await import('undici');

  DiagnosticsChannel.subscribe('undici:request:trailers', (data: any) => {
    const { request, timings } = data;
    const tags = { service: serviceName, status: request.statusCode };

    // Individual phase timings in milliseconds
    metrics.histogram('http.client.dns_ms',       timings.dns,     tags);
    metrics.histogram('http.client.connect_ms',    timings.connect,  tags);
    metrics.histogram('http.client.tls_ms',         timings.tls,      tags);
    metrics.histogram('http.client.ttfb_ms',         timings.ttfb,     tags);
    metrics.histogram('http.client.total_ms',        timings.total,    tags);
    metrics.counter('http.client.connection_reused', timings.dns === 0 ? 1 : 0, tags);
  });

  return pool;
}
Section 10

The things nobody tells you

HTTP/2 can be slower than HTTP/1.1 on your internal network

HTTP/2's advantages (multiplexing, header compression) are most valuable when: (a) there's high latency between client and server, and (b) there are many concurrent requests. On your internal Kubernetes network where pod-to-pod latency is <1ms and most service calls are sequential, HTTP/2 overhead (binary framing, header encoding) can add more cost than it saves. For internal service-to-service calls, HTTP/1.1 with persistent connections is often faster. Test with your actual traffic; don't assume HTTP/2 is always better.

Your "JSON API" sends more bytes than you think

JSON is verbose by design — human-readable keys repeated on every object in an array. A list of 100 user objects, each with {"user_id": 123, "first_name": "...", "last_name": "...", "email": "..."}, sends the key names 100 times. For high-frequency APIs (mobile apps fetching feeds), this overhead is real. Sparse field selection (?fields=id,name), cursor-based pagination with smaller page sizes, and eventually Protocol Buffers for true high-volume paths are the progression.

The Nagle's Algorithm surprise

Nagle's algorithm (enabled by default in TCP) batches small packets together to improve efficiency: it holds a small packet until either a full-sized packet can be formed or an ACK is received from the previous packet. For interactive protocols (HTTP API calls where you send a small request and wait for a response), Nagle causes a 40-200ms delay on small request bodies because the OS holds the packet waiting to batch it.

The fix: TCP_NODELAY socket option, which disables Nagle. Every production HTTP server and HTTP client should set this. Most frameworks do by default, but verify your stack actually sets it — especially if you're using a lower-level TCP library or writing a custom server.

Status codes are a contract, not a suggestion

200 OK with {"error": "not found"} in the body is a sin. It breaks every monitoring system, every load balancer health check, every client retry logic. Use 404. Use 409 for conflicts. Use 422 for validation errors. Use 503 when your service is intentionally shedding load. Semantically correct status codes are what let your infrastructure components (load balancers, CDNs, monitoring) make intelligent decisions automatically.

The OPTIONS request you forgot about

If your API serves browser clients with CORS, every non-simple cross-origin request is preceded by an OPTIONS preflight. If your API takes 50ms to respond, every browser action actually takes 100ms: 50ms for the OPTIONS and 50ms for the real request. Solution: cache preflight responses with Access-Control-Max-Age: 86400 (24 hours). Do this and browsers cache the CORS negotiation for a day, making the preflight cost a one-time payment per origin per day.

How the Performance API works

Every browser silently records timing data for every resource it loads via window.performance.getEntriesByType('resource'). Each entry has domainLookupStart, domainLookupEnd, connectStart, secureConnectionStart, connectEnd, requestStart, responseStart, responseEnd — the exact same phases curl -w measures. You can expose this in your own apps to give users (or your monitoring) real browser-side latency data. This is how Real User Monitoring (RUM) works.

When things go wrong

Your API is slow — check in this order

Most teams jump straight to profiling the server. This sequence checks the cheapest, most common causes first.

1
Run curl -w timing breakdown first
Before touching any code, run curl -w "dns:%{time_namelookup} tcp:%{time_connect} tls:%{time_appconnect} ttfb:%{time_starttransfer} total:%{time_total}" -o /dev/null -s <url>. This tells you which phase is slow in 10 seconds. If TTFB is 20ms but total is 800ms, the problem is download size, not your server code. If TLS is 200ms, you're on TLS 1.2 not 1.3. Most latency investigations skip this and go straight to profiling the wrong thing.
tool: curl -w / httpstat
2
Is DNS slow? (look at time_namelookup)
Anything above 5ms on a warm cache is a problem. Above 50ms means a cold miss — your connection pool may not be reusing connections or your DNS cache isn't warm. Fix: connection pooling eliminates DNS cost entirely for repeat calls. If DNS is consistently slow, your resolver is the issue — not your app.
threshold: >5ms = investigate
3
Is TLS slow? Check version and OCSP stapling
TLS 1.2 costs 2 RTTs. TLS 1.3 costs 1 RTT. TLS 1.3 resumed costs 0 RTT. Check with openssl s_client -connect host:443 2>&1 | grep Protocol. If TLS time is 0ms on repeated calls — session resumption is working. If it's consistently 1 RTT worth — you're paying per connection. Enable session caching in nginx and OCSP stapling to eliminate the CA round-trip.
TLS 1.2 = double the handshake cost
4
Is TTFB high? Now look at the server
TTFB = server processing + 1 RTT. If TTFB is >100ms, the problem is in your application code or database. This is the only phase where your backend code matters. Check slow query logs, N+1 patterns, missing indexes. Everything before this step was network and protocol overhead — none of which is fixed by optimising your handler.
only now: profile your application code
5
Is download time high? Look at response size
High download time with fast TTFB means the response is large. Check: is compression enabled? (Content-Encoding: gzip in the response). Are you returning fields the client doesn't need? Is the page size too large? Compression on a 100KB JSON response typically brings it to 15-20KB — a 5x bandwidth reduction with no code changes.
fix: gzip/brotli + sparse fieldsets
6
Is it only slow on the first request? Connection pooling.
If subsequent requests are fast but the first one is slow, you're paying DNS + TCP + TLS setup on every new connection. Enable connection pooling in your HTTP client (undici Pool, axios with keepAlive agent). The first request to a cold pool pays setup cost. Every subsequent request reuses the connection and pays only TTFB + download.
fix: connection pooling
