Storage & Databases · Beginner

The N+1
Query Problem

You fetch a list of N items, then make N more queries to fetch related data for each one. One query becomes N+1. At small scale it's invisible. At production scale it's the most common cause of database-induced slowdowns.

Section 01

The waiter who makes one trip per person

A table of 10 people all order dessert at the same time. The waiter walks to the first person, takes their order, walks to the kitchen, delivers it to the chef, walks back. Then walks to the second person. Takes their order. Walks to the kitchen. Delivers it. Walks back. Repeat ten times.

The kitchen is fast. The chef can prepare all ten desserts in the time it takes to prepare one. The problem is not the kitchen — it is the 10 separate trips when one trip with all 10 orders would have done the same job in a fraction of the time.

The database is the kitchen. The queries are the trips. Each individual query might take 2ms. A hundred of them take 200ms — not because the database is slow, but because you made a hundred round-trips when one would have sufficed.

The fix is not a faster kitchen. It is not a faster waiter. It is one trip with all the orders.

Section 02

What it is, precisely

The N+1 problem occurs in two steps:

  1. You execute 1 query to fetch a list of N parent records (events, users, orders, posts).
  2. For each of the N records, you execute 1 more query to fetch related data (the venue for each event, the author for each post, the items for each order).

Total queries: 1 + N. Hence "N+1." For a list of 100 events with their venues, that is 101 queries. For 1,000 it is 1,001.

The classic N+1 — what it looks like in codeTypeScript
// 1 query: fetch 100 events
const events = await db.query('SELECT * FROM events LIMIT 100');

for (const event of events.rows) {
  // N queries: one per event — this runs 100 times
  const venue = await db.query(
    'SELECT * FROM venues WHERE id = $1',
    [event.venue_id]
  );
  event.venue = venue.rows[0];
}
// Total: 101 queries for 100 events

The loop is the tell. Any await db.query() inside a for loop over database records is almost certainly an N+1.

Section 03

Why it keeps happening

It is invisible in development. You seed your dev database with 5 events. Five venue queries feel fine — 10ms total. The same code in production with 500 events makes 500 venue queries. At 2ms each that is 1,000ms — a full second just for venue lookups, before the rest of the request does anything.

ORMs make it easy to write accidentally. The most seductive N+1 is the one that looks like a clean object access:

ORM lazy loading — the hidden N+1TypeScript
// Prisma, TypeORM, Sequelize — lazy associations look clean
const events = await prisma.event.findMany({ take: 100 });

for (const event of events) {
  // This triggers a SELECT on the venues table for EVERY event
  // It looks like a property access. It is a database query.
  console.log(event.venue.name);  // N hidden queries
}

// The ORM hides the query behind a property — easy to miss in review

It compounds across layers. A GraphQL resolver that fetches a list of posts, and each post's resolver fetches the author, and each author resolver fetches their profile — that is N+1 nested inside N+1. A list of 50 posts with authors and profiles becomes 1 + 50 + 50 = 101 queries minimum.

⚠️
GraphQL is especially prone to N+1

Each GraphQL field resolver runs independently. Without a DataLoader or query batching layer, a list query that resolves nested relationships will generate an N+1 for each relationship level. This is why DataLoader was invented — Facebook built it specifically to solve this problem in their GraphQL server.

Section 05

How to detect it

pg_stat_statements — the production fingerprint

In PostgreSQL, pg_stat_statements tracks every query that runs. An N+1 has an unmistakable signature: a parameterised query with a calls count far higher than the number of requests.

Detecting N+1 in PostgreSQLSQL
-- Run during a load test or normal traffic window
SELECT
  LEFT(query, 80)            AS query_preview,
  calls,
  ROUND(mean_exec_time::numeric, 2) AS mean_ms,
  ROUND(total_exec_time::numeric)   AS total_ms
FROM pg_stat_statements
ORDER BY calls DESC
LIMIT 20;

-- N+1 signature:
-- query_preview                              calls    mean_ms
-- SELECT * FROM venues WHERE id = $1         48,000   2
-- SELECT * FROM events ORDER BY... LIMIT 50  960      12
--
-- 48,000 venue queries for 960 list requests = 50 per request
-- If your page size is 50 events: that IS the N+1

Log query counts per request

In development, log how many queries each request makes. More than 10 for a simple list endpoint is a red flag.

Query count middleware — catch N+1 in devTypeScript
// Middleware: count queries per request
app.use((req, res, next) => {
  req.queryCount = 0;
  const originalQuery = db.query.bind(db);
  db.query = (...args) => {
    req.queryCount++;
    return originalQuery(...args);
  };
  res.on('finish', () => {
    if (req.queryCount > 10) {
      console.warn(
        `[N+1 ALERT] ${req.method} ${req.path} made ${req.queryCount} queries`
      );
    }
  });
  next();
});

// Output:
// [N+1 ALERT] GET /events made 101 queries

ORM query logging

Prisma: set log: ['query'] in the client constructor. TypeORM: set logging: true. Sequelize: set logging: console.log. Scroll through the output for a single request — repeated identical queries with different IDs are N+1.

Fix 01

Fix: JOIN — one query, all the data

The most direct fix. Instead of fetching events then fetching venues separately, fetch both together in a single query. The database does the join internally — one round-trip, all the data.

JOIN — eliminates the N+1 entirelyTypeScript
// Before: 1 + N queries
const events = await db.query('SELECT * FROM events LIMIT 100');
for (const event of events.rows) {
  const venue = await db.query('SELECT * FROM venues WHERE id = $1', [event.venue_id]);
  event.venue = venue.rows[0];
}

// After: 1 query
const result = await db.query(`
  SELECT
    e.id,
    e.name,
    e.starts_at,
    e.description,
    v.id          AS venue_id,
    v.name        AS venue_name,
    v.city        AS venue_city,
    v.capacity    AS venue_capacity
  FROM events e
  JOIN venues v ON v.id = e.venue_id
  ORDER BY e.starts_at
  LIMIT 100
`);

const events = result.rows.map(row => ({
  id:          row.id,
  name:        row.name,
  startsAt:    row.starts_at,
  description: row.description,
  venue: {
    id:       row.venue_id,
    name:     row.venue_name,
    city:     row.venue_city,
    capacity: row.venue_capacity,
  },
}));

// Total: 1 query regardless of how many events

When to use it: When you always need the related data alongside the parent. When the join does not fan out (one event has one venue — not one event has many bookings, which would multiply rows).

Watch out for: Joining a one-to-many relationship without aggregation multiplies rows. An event with 50 bookings joined to the events table returns 50 rows per event. Use a subquery or aggregate instead.

Fix 02

Fix: Batch query — collect IDs, fetch once

When a JOIN is not practical (different services, different databases, or you need more control), collect all the IDs from the first query, then fetch all related records in one query using WHERE id = ANY($1).

Batch query — 2 queries instead of N+1TypeScript
// Step 1: fetch the list
const events = await db.query('SELECT * FROM events LIMIT 100');

// Step 2: collect all venue IDs (deduplicated)
const venueIds = [...new Set(events.rows.map(e => e.venue_id))];

// Step 3: ONE query to fetch all needed venues
const venues = await db.query(
  'SELECT * FROM venues WHERE id = ANY($1)',
  [venueIds]
);

// Step 4: build a lookup map
const venueMap = new Map(venues.rows.map(v => [v.id, v]));

// Step 5: attach — no more queries
const result = events.rows.map(event => ({
  ...event,
  venue: venueMap.get(event.venue_id),
}));

// Total: 2 queries regardless of N
// Works across different services, different databases
// Works when you can't control the SQL (third-party ORM, microservice)
Fix 03

Fix: DataLoader — automatic batching per request

DataLoader collects all individual .load(id) calls that happen within a single event loop tick, batches them into one database query, then distributes results back to each caller. This is the standard fix for GraphQL — it makes individual field resolvers safe even in a list context.

DataLoader pattern — batch automaticallyTypeScript
import DataLoader from 'dataloader';

// Create per-request (fresh cache each request)
function createLoaders() {
  return {
    venue: new DataLoader<string, Venue>(
      async (ids: readonly string[]) => {
        // Called ONCE with all IDs collected this tick
        const rows = await db.query(
          'SELECT * FROM venues WHERE id = ANY($1)',
          [[...ids]]
        );
        const map = new Map(rows.rows.map(v => [v.id, v]));
        // Must return results in same order as input ids
        return ids.map(id => map.get(id) ?? null);
      }
    ),
  };
}

// In a GraphQL resolver or route handler:
app.get('/events', async (req, res) => {
  const loaders = createLoaders();          // fresh per request
  const events = await db.query('SELECT * FROM events LIMIT 100');

  // Each .load() call looks individual but they're batched automatically
  const withVenues = await Promise.all(
    events.rows.map(async event => ({
      ...event,
      venue: await loaders.venue.load(event.venue_id),
    }))
  );

  res.json(withVenues);
});
// Result: 2 queries — 1 for events, 1 batched venue query
// Even if 100 resolvers each call loaders.venue.load() independently
✅
DataLoader is the correct fix for GraphQL

In a REST API, you control the query structure and can use a JOIN or batch query directly. In GraphQL, each field resolver is independent — you can't easily restructure them. DataLoader solves this by transparently batching all .load() calls that happen in the same tick, regardless of where in the resolver tree they originate. Create one DataLoader per entity type per request.

Fix 04

Fix: ORM eager loading — tell the ORM what you need

Every major ORM has a way to declare upfront which relationships to include. This generates a JOIN or a batched query rather than lazy-loading each one.

Eager loading in Prisma, TypeORM, and SequelizeTypeScript
// ── PRISMA ──────────────────────────────────────────────────────
// Bad: lazy load triggers N queries
const events = await prisma.event.findMany({ take: 100 });
for (const e of events) { console.log(e.venue.name); } // N queries

// Good: include declares the relationship upfront
const events = await prisma.event.findMany({
  take: 100,
  include: { venue: true },           // 1 extra query, batched
  // or: select specific fields
  include: {
    venue: { select: { name: true, city: true } }
  },
});

// ── TYPEORM ─────────────────────────────────────────────────────
const events = await eventRepo.find();         // lazy — N+1
const events = await eventRepo.find({
  relations: { venue: true },                  // eager — JOIN
});

// or with QueryBuilder for more control:
const events = await eventRepo
  .createQueryBuilder('event')
  .leftJoinAndSelect('event.venue', 'venue')   // explicit JOIN
  .take(100)
  .getMany();

// ── SEQUELIZE ───────────────────────────────────────────────────
const events = await Event.findAll({ limit: 100 }); // lazy
const events = await Event.findAll({
  include: [{ model: Venue }],                 // eager
  limit: 100,
});

Section 07

Prevention — stop N+1 before it reaches production

The rule of thumb

Any await inside a loop over database records is a red flag. Stop and ask: can this be done as a single query outside the loop? The answer is almost always yes.

❌ Await inside loop
for (const item of items) { // Runs N times const rel = await db.find( item.relId ); }
✅ Batch outside loop
const ids = items.map( i => i.relId ); const rels = await db.findMany(ids); // One query, then map

Enable query logging in development

Always run with query logging on in development. Every query should be visible. A healthy list endpoint makes 1–3 queries. More than 10 is always worth investigating. The cost of reviewing logs in dev is far less than debugging a production slowdown.

Add a query count assertion in tests

Test that enforces query countTypeScript
import { assertQueryCount } from './test-helpers';

test('GET /events uses at most 3 queries', async () => {
  await assertQueryCount(
    () => request(app).get('/events?limit=50'),
    { max: 3 }
  );
  // This test fails if someone introduces an N+1 later
  // Catches regressions before they reach production
});

pg_stat_statements in staging

Run your full test suite against a staging database with pg_stat_statements enabled. After the run, query for any statement with calls suspiciously higher than your request count. This catches N+1 patterns that unit tests miss because the loop only ran once in the test but runs 100 times under real load.

The database is not slow. The queries are fast. It is the hundred separate trips to the kitchen when one trip would have done. The fix is always the same: collect what you need, fetch it together, map it back. One trip.
