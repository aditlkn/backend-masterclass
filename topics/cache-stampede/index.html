Stampede

A hot cache key expires. Hundreds of requests simultaneously find the cache empty and race to recompute the same value. Every one of them hits the database. The cache was supposed to protect the database — instead its expiry caused a spike worse than steady-state load.

Section 01

The library with one copy of a popular book

A library has one copy of a popular book. It comes back from loan. Forty people are waiting for it. They all rush to the desk simultaneously — each one asks the librarian to photocopy it for them. The librarian makes forty photocopies of the same book. Forty identical copies, forty times the work, when one copy distributed to all forty would have been sufficient.

The book is the cached value. The photocopying is the database query and computation. The forty people are the concurrent requests. The correct solution: one person makes the photocopy, everyone else waits for that one copy, then shares it.

The stampede does not happen because the value is hard to compute. It happens because every requester independently decides they are the one who will compute it, all at the same moment.

Section 02

What it is

Cache stampede (also called "cache dogpile" or "thundering herd on a cache key") occurs when:

  1. A cache key that many requests depend on expires (TTL reached, explicit eviction, or cache restart).
  2. Multiple requests arrive simultaneously, all finding the key missing.
  3. Each of them independently queries the database and recomputes the value.
  4. All of them write the same result back to the cache — redundant work, extra DB load.

The key word is simultaneously. A single cache miss is normal and expected. A stampede is when the miss affects many requests at once because the key was popular and its expiry was not gracefully handled.

The naive cache pattern — vulnerable to stampedeTypeScript
// Every request runs this independently
async function getEventListing(): Promise<Event[]> {
  const cached = await redis.get('events:listing');
  if (cached) return JSON.parse(cached);

  // Key is missing — every concurrent request reaches here
  const events = await db.query('SELECT * FROM events...'); // N times
  await redis.set('events:listing', JSON.stringify(events), 'EX', 300);
  return events;
}
// When the key expires at peak traffic:
// 500 concurrent requests × 1 DB query each = 500 queries for 1 value
⚠️
Hot keys are the most vulnerable

A cache key hit by 10 requests per second produces a stampede of at most 10 concurrent misses. A key hit by 1,000 requests per second — homepage data, top-10 listing, featured products — produces 1,000 simultaneous misses when it expires. The more popular the key, the more important it is to protect it from stampede. The keys that need the most protection are exactly the ones most likely to cause a production incident when they expire.

Section 03

The cost — it spikes exactly when you're busiest

The cruel property of cache stampede: it is worst during peak traffic. At low traffic, few requests arrive during the recomputation window — the stampede is small. At peak traffic, hundreds of requests arrive in the milliseconds after the key expires. The database spike is proportional to your traffic, hitting hardest when the database is already most loaded.

A typical sequence during a high-traffic event listing page expiry:

  1. TTL expires at 14:23:05.000
  2. 14:23:05.001 — 50 requests find cache empty, all query the DB
  3. 14:23:05.200 — DB query completes (200ms). 50 write to cache, 50 return results
  4. 14:23:05.001–14:23:05.200 — 300 more requests find cache empty, also query DB
  5. 350 total DB queries for one result. DB latency spikes. Other queries slow. P99 rises.
Fix 01

Fix: Mutex lock — one recomputes, others wait

When a cache miss occurs, the first request acquires a lock before querying the database. Any other request that misses the cache while the lock is held waits for the lock to be released, then reads the freshly populated cache value. Only one database query happens.

Mutex pattern using Redis NX lockTypeScript
async function getWithMutex<T>(
  key: string,
  compute: () => Promise<T>,
  ttlSeconds: number
): Promise<T> {
  // 1. Try cache first
  const cached = await redis.get(key);
  if (cached) return JSON.parse(cached);

  const lockKey = `lock:${key}`;
  const lockTtl = 10; // lock expires in 10s (prevents deadlock)

  // 2. Try to acquire lock (NX = only set if not exists)
  const acquired = await redis.set(lockKey, '1', 'NX', 'EX', lockTtl);

  if (acquired) {
    // 3a. We got the lock — compute and populate cache
    try {
      const value = await compute();
      await redis.set(key, JSON.stringify(value), 'EX', ttlSeconds);
      return value;
    } finally {
      await redis.del(lockKey); // always release
    }
  } else {
    // 3b. Lock held by another — poll until cache is populated
    const start = Date.now();
    while (Date.now() - start < 5000) { // 5s timeout
      await sleep(50);
      const result = await redis.get(key);
      if (result) return JSON.parse(result);
    }
    // Timeout — fallback: compute anyway (avoids blocking forever)
    return compute();
  }
}

Trade-off: Waiting requests are held until recomputation completes. If computation takes 500ms, those requests have 500ms added to their latency. For very slow computations, combine with stale-while-revalidate (serve stale while refreshing).

Fix 02

Fix: Probabilistic early expiry — recompute before expiry

Instead of waiting for the TTL to expire, start recomputing the value probabilistically before it expires. Requests near the TTL boundary have a random chance of triggering a recomputation. The probability increases as the key gets closer to expiry. By the time the TTL actually hits, the value has already been refreshed.

This is the algorithm used by the XFetch paper (2015). It requires storing the computation time alongside the cached value.

Probabilistic early expiry (XFetch algorithm)TypeScript
interface CacheEntry<T> {
  value: T;
  computedAt: number;    // timestamp when value was computed
  ttlSeconds: number;    // original TTL
  delta: number;          // time taken to compute (ms)
}

async function getWithEarlyExpiry<T>(
  key: string,
  compute: () => Promise<T>,
  ttlSeconds: number,
  beta = 1   // higher beta = more aggressive early recomputation
): Promise<T> {
  const raw = await redis.get(key);

  if (raw) {
    const entry: CacheEntry<T> = JSON.parse(raw);
    const expiresAt = entry.computedAt + entry.ttlSeconds * 1000;
    const now = Date.now();

    // XFetch formula: recompute if this random check says so
    const shouldRecompute =
      now - beta * entry.delta * Math.log(Math.random()) >= expiresAt;

    if (!shouldRecompute) return entry.value; // serve from cache
  }

  // Recompute (either miss or early expiry triggered)
  const start = Date.now();
  const value = await compute();
  const delta = Date.now() - start;

  const entry: CacheEntry<T> = {
    value,
    computedAt: Date.now(),
    ttlSeconds,
    delta,
  };
  await redis.set(key, JSON.stringify(entry), 'EX', ttlSeconds + 60);
  return value;
}

// How it works:
// - Values that take longer to compute trigger earlier recomputation
// - As TTL approaches, the probability of recomputation increases
// - No lock needed — occasional early recomputations are cheap
// - The key never actually expires under traffic
Fix 03

Fix: Background refresh — never let hot keys expire

For keys that are always needed, never let them expire during traffic. A background job refreshes the value on a schedule (every 4 minutes for a 5-minute TTL), before the TTL can hit. The key is always warm. Requests always hit the cache. There is no window where the key can be missing.

Background refresh patternTypeScript
// Start background refresh on server startup
function startBackgroundRefresh() {
  // Initial population
  refreshEventListing();

  // Refresh every 4 minutes (TTL is 5 minutes)
  // Key is always warm — never expires under traffic
  setInterval(refreshEventListing, 4 * 60 * 1000);
}

async function refreshEventListing() {
  try {
    const events = await db.query('SELECT * FROM events...');
    await redis.set(
      'events:listing',
      JSON.stringify(events),
      'EX',
      300   // 5-minute TTL as safety net
    );
  } catch (err) {
    // Log but don't crash — stale cache is better than no cache
    logger.error({ err }, 'event listing refresh failed');
  }
}

// Request handler — always hits cache
app.get('/events', async (req, res) => {
  const cached = await redis.get('events:listing');
  if (!cached) {
    // Should never happen in normal operation
    // Only on first startup before background job runs
    await refreshEventListing();
    return res.json(JSON.parse(await redis.get('events:listing')!));
  }
  res.json(JSON.parse(cached));
});

Best for: Data that changes on a known schedule or can tolerate being slightly stale (event listings, leaderboards, pricing tables, featured content). Not appropriate for user-specific data or anything requiring real-time accuracy.

Fix 04

Fix: Stale-while-revalidate — serve stale, refresh async

Serve the stale cached value immediately — zero latency for the user — while triggering a background recomputation. The next request gets the fresh value. No request ever waits for recomputation. No stampede because only one background refresh runs at a time.

Stale-while-revalidate patternTypeScript
interface SWREntry<T> {
  value: T;
  refreshAfter: number;  // timestamp: serve fresh if before this
  expiresAt: number;     // hard expiry — never serve after this
}

const refreshing = new Set<string>();  // prevent concurrent refreshes

async function getStaleWhileRevalidate<T>(
  key: string,
  compute: () => Promise<T>,
  freshSeconds: number,    // serve fresh for this long
  staleSeconds: number     // serve stale for this long after fresh expires
): Promise<T> {
  const raw = await redis.get(key);
  const now = Date.now();

  if (raw) {
    const entry: SWREntry<T> = JSON.parse(raw);

    if (now < entry.refreshAfter) {
      return entry.value;  // fresh — serve directly
    }

    if (now < entry.expiresAt) {
      // Stale but not expired — serve stale, refresh in background
      if (!refreshing.has(key)) {
        refreshing.add(key);
        compute()
          .then(value => writeEntry(key, value, freshSeconds, staleSeconds))
          .catch(err => logger.error({ err, key }, 'swr refresh failed'))
          .finally(() => refreshing.delete(key));
      }
      return entry.value;  // return stale immediately
    }
  }

  // No entry or fully expired — must compute synchronously
  const value = await compute();
  await writeEntry(key, value, freshSeconds, staleSeconds);
  return value;
}

async function writeEntry<T>(key: string, value: T, fresh: number, stale: number) {
  const now = Date.now();
  const entry: SWREntry<T> = {
    value,
    refreshAfter: now + fresh * 1000,
    expiresAt:    now + (fresh + stale) * 1000,
  };
  await redis.set(key, JSON.stringify(entry), 'EX', fresh + stale);
}

// Usage: fresh for 60s, stale for 300s
const events = await getStaleWhileRevalidate(
  'events:listing', fetchEvents, 60, 300
);

In practice: combine approaches

Most production systems use a combination. Background refresh for hot, shared, slow-changing keys (homepage, listings, pricing). Stale-while-revalidate for user-specific data where serving slightly stale is acceptable. Mutex as a safety net for anything else. Probabilistic early expiry is elegant but requires storing extra metadata — worth it for keys with expensive, slow recomputation.

One copy of the book. Forty people wanting it. The mistake is letting all forty try to photocopy it simultaneously. One person makes the copy. Everyone else waits — or reads the slightly outdated edition that is already on the shelf. Either way: one trip to the photocopier.
