The Thundering
Herd Problem

A resource goes down, then comes back up. Every client that was waiting simultaneously tries to reconnect. The sudden burst overwhelms the recovering system and takes it down again — worse than the original failure.

Section 01

The stadium turnstile that jams

50,000 fans arrive at a stadium. The turnstiles jam. Everyone stops. Then the jam clears. All 50,000 people push forward simultaneously — crushing whoever is at the front, overwhelming the turnstiles, and causing them to jam again immediately.

If instead, the operator announced "fans with tickets ending in 1–3, please approach now" — then 2–4 — then 5–7 — the flow would be orderly, the turnstiles would never be overwhelmed, and everyone would get through faster than in the stampede.

The database, the cache, the API — each one is a turnstile. When it recovers from an outage and 10,000 queued requests arrive in the same millisecond, it falls over again. Not because it is weak. Because the load was perfectly synchronised.

Section 02

What it is

A thundering herd occurs when a large number of processes or threads, all waiting for the same event, are woken up simultaneously — and all try to acquire the same resource at once. Only one (or a few) can succeed. The rest fail, retry, or overwhelm the system.

The name comes from operating systems: in early Unix, when a process was waiting for a network connection, the kernel would wake all waiting processes when the connection became available. All of them would race to accept the connection. Only one could win. The rest wasted CPU cycles and had to sleep again. At scale, this wasted effort became a measurable performance problem.

In backend systems the pattern is the same: many clients waiting on the same unavailable thing, all activated at once when it becomes available.

Section 03

What triggers it

  • Service restart: A service restarts after a crash or deploy. All clients that backed off retrying simultaneously hit their retry interval and reconnect at once.
  • Database failover: The primary goes down, promotion of the replica takes 30 seconds. All 500 application server connections try to reconnect the moment the new primary is ready.
  • Rate limit reset: A third-party API rate limits you at minute boundary. You and every other client reset and fire requests simultaneously at :00 of the next minute.
  • Scheduled jobs: 1,000 cron jobs all configured to run at 00:00. They all hit the database simultaneously.
  • Fixed retry interval: Every client is configured to retry after exactly 5 seconds. After a 5-second outage, every client retries at exactly the same moment.
  • Cold start: Auto-scaling spins up 50 new instances simultaneously. They all try to warm their caches and connect to the database at once.
Section 04

Why it makes outages worse

The dangerous property of a thundering herd is that it turns a recoverable outage into a repeated one. The sequence:

  1. Database slows under heavy load.
  2. Clients start timing out. They back off and retry after 5 seconds.
  3. Database recovers — load drops, performance returns.
  4. All clients hit their 5-second retry simultaneously. 10,000 requests arrive in the same second.
  5. Database falls over again under the burst. Goes back to step 1.

The system oscillates. It never fully recovers because every recovery attempt is met with the full queued load all at once. Without jitter, this loop can continue indefinitely — the system appears to recover briefly every few seconds, then immediately falls over again.

🔴
Fixed retry intervals are the root cause

If every client uses the same fixed retry delay — whether that's 1 second, 5 seconds, or 30 seconds — they will always retry in sync. A 5-second outage synchronises all clients to the same retry moment. The fix is not a longer interval. A 30-second fixed interval just creates a bigger burst every 30 seconds. The fix is randomisation.

Fix 01

Fix: Jitter — break the synchronisation

Jitter adds randomness to retry delays. Instead of all clients retrying at exactly T+5s, they retry at T+4.1s, T+5.7s, T+3.8s, T+6.2s. The burst is spread across a window. No single moment gets the full load.

Jitter implementationsTypeScript
// ❌ Fixed interval — all clients retry at the same moment
const RETRY_DELAY = 5000;
await sleep(RETRY_DELAY);

// ✅ Full jitter — random delay within the window
function jitter(baseMs: number): number {
  return Math.random() * baseMs;   // 0ms to baseMs
}
await sleep(jitter(5000));   // anywhere from 0s to 5s

// ✅ Equal jitter — half base, half random (avoids very short retries)
function equalJitter(baseMs: number): number {
  const half = baseMs / 2;
  return half + Math.random() * half;  // baseMs/2 to baseMs
}
await sleep(equalJitter(5000));   // 2.5s to 5s

// ✅ Decorrelated jitter — AWS recommended, avoids correlation between retries
function decorrelatedJitter(prev: number, base: number, cap: number): number {
  return Math.min(cap, Math.random() * (prev * 3 - base) + base);
}
let delay = base;
for (let attempt = 0; attempt < maxRetries; attempt++) {
  await sleep(delay);
  delay = decorrelatedJitter(delay, base, cap);
}
✅
AWS recommends "full jitter" for most cases

AWS published an analysis of jitter strategies in 2015 ("Exponential Backoff and Jitter"). Their conclusion: full jitter (completely random delay within the window) minimises total client wait time across a fleet. Equal jitter is a reasonable compromise when you need a minimum wait. Decorrelated jitter is best when clients are running many concurrent retries. For most retry logic in backend services, full jitter with exponential backoff is the correct default.

Fix 02

Fix: Exponential backoff — reduce total retry volume

Jitter spreads the load across time. Exponential backoff reduces the total volume of retries. Each retry attempt waits longer than the last — the delay doubles (or grows by some multiplier) on each failure. A system under load gets progressively less traffic from retrying clients, giving it time to recover.

Exponential backoff with jitter — production patternTypeScript
interface RetryOptions {
  maxAttempts: number;
  baseDelayMs: number;
  maxDelayMs:  number;
  shouldRetry?: (err: Error) => boolean;
}

async function withRetry<T>(
  fn: () => Promise<T>,
  opts: RetryOptions
): Promise<T> {
  let lastError: Error;

  for (let attempt = 0; attempt < opts.maxAttempts; attempt++) {
    try {
      return await fn();
    } catch (err) {
      lastError = err as Error;

      // Don't retry non-transient errors
      if (opts.shouldRetry && !opts.shouldRetry(lastError)) {
        throw lastError;
      }

      if (attempt === opts.maxAttempts - 1) break;

      // Exponential backoff with full jitter
      const exponential = opts.baseDelayMs * Math.pow(2, attempt);
      const capped = Math.min(exponential, opts.maxDelayMs);
      const delay  = Math.random() * capped;   // full jitter

      await sleep(delay);
    }
  }

  throw lastError!;
}

// Usage
const result = await withRetry(
  () => db.query('SELECT ...'),
  {
    maxAttempts: 5,
    baseDelayMs: 100,    // 100ms base
    maxDelayMs:  30_000, // cap at 30s
    shouldRetry: (err) => err.code === 'ECONNRESET' || err.code === 'ETIMEDOUT',
  }
);
// Delays (approx): 50ms, 200ms, 800ms, 3200ms, give up
// Each attempt randomly between 0 and the exponential cap
Fix 03

Fix: Queue limiting — cap concurrent reconnections

Even with jitter, if 10,000 clients all recover within a 5-second window, that is still 2,000 requests per second — potentially more than the recovering system can handle. Queue limiting caps the number of requests that can be in-flight simultaneously.

Concurrency limiter — cap simultaneous requestsTypeScript
class ConcurrencyLimiter {
  private running = 0;
  private queue: (() => void)[] = [];

  constructor(private readonly limit: number) {}

  async run<T>(fn: () => Promise<T>): Promise<T> {
    if (this.running >= this.limit) {
      // Wait until a slot opens
      await new Promise<void>(resolve => this.queue.push(resolve));
    }
    this.running++;
    try {
      return await fn();
    } finally {
      this.running--;
      this.queue.shift()?.();   // release next waiter
    }
  }
}

// Cap reconnection attempts to 20 concurrent
const limiter = new ConcurrencyLimiter(20);

// All 10,000 clients call this — only 20 run at a time
await limiter.run(() => db.connect());

Fix 04

Fix: Connection pooling — absorb the burst

A connection pool holds a fixed number of open connections. New requests wait in the pool's queue rather than each opening their own connection. When a database restarts, only the pool reconnects — not every application thread simultaneously.

With a pool of 20 connections and 10,000 concurrent requests: 20 connections go to the database, 9,980 wait in the pool queue. The database sees a steady stream of 20 connections, not 10,000. The pool absorbs the thundering herd.

This is why connection poolers like PgBouncer are standard infrastructure — they sit between the application and the database and enforce a connection ceiling regardless of application load.
