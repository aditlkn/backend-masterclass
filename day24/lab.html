Distributed Monolith  ·  Shared Database  ·  Deploy Blocked

The payment service reads the bookings table. Now we cannot deploy either without breaking both.

Sprint 14 · Schema migration blocked · Two teams, one database

The booking team needs to add a group_booking_id column to the bookings table and rename booking_status to status to match the rest of the codebase. The migration is ready. But when they notify the payment team, the payment team sends back a Slack message: "We query that table directly in 6 places. If you rename that column, our refund job breaks."

Nobody told the booking team that the payment service had its own SQL queries against the bookings table. The payment service has been doing this for 18 months. There are 6 direct SQL queries scattered across the payment codebase. The two services are deployed together on every release. Neither team can ship independently. The "microservices" are a distributed monolith.

What this situation teaches
▶The service boundary is the database boundary, not the code boundary. Two processes on separate deployments sharing one database is not microservices — it is a monolith with extra network hops. Until the payment service stops querying the bookings table, there is no real boundary between them.
▶The expand-contract pattern is the safe path for schema changes. Add the new column or endpoint alongside the old one. Migrate consumers. Remove the old thing only after all consumers are confirmed off it. Big-bang migrations break things. Incremental migrations do not.
▶Once the boundary exists, the booking team has full autonomy. After payment-service uses the API and has zero direct DB queries, the booking team can rename columns, add tables, change types — without telling the payment team. The API is the contract. The database is the implementation. This is the actual value of the microservice boundary.
