Auth Bypass  ·  Token Revocation  ·  JWT Stateless

A user was banned for fraud. They are still booking tickets 40 minutes later.

Trust & Safety escalation · usr_8821 · access_token expiry: 60 minutes

The trust and safety team banned user usr_8821 at 14:00 for confirmed ticket fraud — they were reselling tickets at 10x face value. The ban was applied in the users database: UPDATE users SET banned=true WHERE id='usr_8821'. The trust team expected the user to be locked out immediately.

At 14:40, the user successfully books four more tickets. The booking service has no idea the user is banned. Their JWT is valid: correct signature, correct issuer, not expired. The banned field is in the database, not in the token. The booking service never checks the database — it trusts the token.

The root cause: access tokens were set to 60-minute expiry. No revocation mechanism. The hotel key card does not expire until checkout time.

What this incident teaches
▶Token expiry IS your revocation window. A 60-minute expiry means a 60-minute breach window after any ban, password change, or account compromise. 15 minutes is the industry standard for a reason. The shorter the expiry, the smaller the inherent breach window of stateless JWTs.
▶Revoke refresh tokens first, access tokens second. The correct hierarchy: on ban, immediately revoke the refresh token (the auth server has state over this). The user cannot get new access tokens. The existing access token expires naturally within its window. This is the right trade-off for most applications.
▶Option C (DB check on each booking) is a valid immediate fix but has a cost. Checking the banned status on every booking creation is correct for this specific operation — it adds one DB lookup per booking, which is acceptable. Checking on every API request (including reads) at 10,000 req/sec is a different story. Segment where you check: for write operations with high fraud risk, a DB check is appropriate. For reads, accept the token window.
