LAB
Production Incident  ·  REST API  ·  Idempotency

342 customers charged twice — "the retry was supposed to be safe"

Monday 10:45 IST  ·  Duration: 3.5 hours  ·  Severity: P0  ·  ₹18.4L in duplicate charges

Monday morning. A payment gateway engineer noticed the refund queue was unusually busy. Then a customer support ticket came in. Then another. By 10:45am there were 12 tickets — all the same complaint: charged twice for one order. By 11:30am the support team had found 342 such cases from the previous weekend.

The engineering team reviewed the payment service code. Everything looked correct. The charge endpoint was called once per order. Then someone checked the mobile SDK logs. The SDK had a retry policy: if a request timed out (>30s), retry once. The payment server was processing charges in 35 seconds during a weekend traffic spike — just over the 30-second client timeout. The client retried. The server processed both.

Emergency fix: Idempotency-Key header required on POST /v1/charges. Redis-based deduplication deployed. Incident closed. Refunds initiated.
Root cause

The POST /v1/charges endpoint had no idempotency protection. The mobile SDK correctly retried on timeout — this is the right behaviour for a client. But the server had no mechanism to detect that two requests with the same intent (charging the same order for the same amount) were actually the same operation. Both requests created independent charge records. The timeout was the trigger, but the missing idempotency key was the root cause — it was always a ticking clock waiting for the server to be slow.

app.post('/v1/charges', async (req, res) => {
  const charge = await db.createCharge(req.body); // creates new row, always
  res.json({ charge_id: charge.id, status: 'SUCCESS' });

  requireIdempotencyKey(),   // 400 if header missing
  idempotencyMiddleware(),    // deduplicate in Redis
  async (req, res) => {
    const charge = await db.createCharge(req.body);
    res.json({ charge_id: charge.id, status: 'SUCCESS' });
  }
);

// Mobile SDK — always send a key, scoped to the user's intent
const idempotencyKey = `charge:\${userId}:\${orderId}`; // deterministic
await api.post('/v1/charges', payload, {
  headers: { 'Idempotency-Key': idempotencyKey }
}); // Safe to retry any number of times — same key = same result
What this incident teaches
▶Client retries are correct behaviour — the server must be ready for them. The mobile SDK retrying on timeout was right. The server not being idempotent was the bug.
▶Any endpoint involving money, emails, or orders needs idempotency. If the operation has a real-world side effect that's expensive or irreversible, it must be safe to call twice.
▶Zero error rate ≠ no incident. Both requests succeeded. The monitoring was green. The damage was in the data, not the errors. Business logic bugs are invisible to infrastructure monitoring.
▶The idempotency key must be deterministic and user-generated. charge:{userId}:{orderId} — the same user placing the same order always produces the same key, even across retries and app restarts.
