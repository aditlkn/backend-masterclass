Most engineers can build a REST API. Section 01

Resource design: URLs are a user interface

The most consequential decisions in REST API design are URL structure decisions. They're hard to change (breaking change), widely seen (documentation, logs, client code), and immediately signal whether the API was designed or just evolved. The mental model shift: your URL hierarchy is the information architecture of your domain, not a reflection of your database schema.

Resources, not actions

The single most common REST mistake: naming endpoints after actions rather than resources. POST /createOrder, GET /getOrderById, POST /cancelOrder — this is RPC dressed up as REST. The HTTP methods are the verbs; your URLs should be nouns.

The correct model: POST /orders (create), GET /orders/{id} (read), PATCH /orders/{id} (update), DELETE /orders/{id} (cancel). The resource is orders. The action is implicit in the method. This matters because HTTP infrastructure — caches, proxies, monitoring — understands methods. A CDN knows GET is cacheable and POST is not. A monitoring system can automatically track 4xx rates by resource. When you use POST /getOrder, you break all of that.

Nesting: when to and when not to

Nested resources (/users/{userId}/orders/{orderId}) express ownership: this order belongs to this user. The rule: nest when the child resource only makes sense in the context of the parent, and when the parent ID is part of the child's identity. Don't nest more than 2 levels — it creates brittle URLs that break when the ownership relationship changes.

The anti-pattern: nesting for filtering. /users/{userId}/orders to get a user's orders is fine. But if you also need orders by product, by date range, by status — you'd need /products/{productId}/orders, /dates/{date}/orders, etc. The cleaner pattern: flat resource with query parameters. /orders?user_id={userId}&status=pending&after=2024-01-01.

Naming conventions that don't age badly

  • Plural nouns for collections: /orders not /order. GET /orders returns a list; GET /orders/{id} returns one. Consistent.
  • snake_case for JSON fields, kebab-case for URL segments: /payment-methods in the URL, payment_method_id in the body. URLs are case-insensitive; JSON fields are not. Be consistent — pick one and enforce it.
  • No file extensions: /orders/{id} not /orders/{id}.json. Content negotiation belongs in Accept headers.
  • No version in every endpoint: version the API, not each resource. /v1/orders means everything under /v1/ is a coherent API version, not that just orders is versioned.

Section 02

Pagination at scale: why offset pagination breaks and what to use instead

Pagination is one of the most deceptively simple API problems. Every beginner implementation uses offset pagination: GET /orders?offset=0&limit=20, ?offset=20&limit=20, and so on. It works fine in development with 1,000 rows. It breaks in production with 10 million.

The offset pagination problem

SQL's OFFSET n clause doesn't skip rows efficiently. It reads all n rows up to the offset, discards them, and returns the next batch. SELECT * FROM orders ORDER BY created_at OFFSET 1000000 LIMIT 20 reads 1,000,020 rows from disk to return 20. At page 50,000, you're scanning half your table to deliver a page of results.

The second problem: data instability. If a new order is inserted while a user is paging through results, every page after that point shifts by one. The user either sees a duplicate (the item that shifted into the previous page range) or misses an item (the one that shifted out). At high insert rates, paginated results are meaningless.

Cursor-based pagination: the right answer

Cursor pagination uses a pointer into the dataset instead of a position. The cursor encodes enough information to resume from an exact point: typically the ID or (timestamp, id) tuple of the last item seen. The query becomes: SELECT * FROM orders WHERE (created_at, id) < ($cursor_time, $cursor_id) ORDER BY created_at DESC, id DESC LIMIT 20.

This uses a composite index efficiently — the B-tree seeks directly to the cursor position and reads forward. Cost is O(page size), not O(offset). Doesn't matter if you're on page 1 or page 500,000. And because the cursor is anchored to specific rows, new inserts don't shift pages.

The tradeoff: you lose the ability to jump to an arbitrary page number ("go to page 47"). For most UIs, this is fine — infinite scroll, "load more", "next/prev" navigation don't need page numbers. For analytics exports or admin tools that need arbitrary access, offset pagination (with a limit on max offset) may be acceptable.

Cursor pagination — production implementation TYPESCRIPT 
// Cursor encodes (created_at, id) — enough to resume exactly
function encodeCursor(item: { created_at: Date; id: string }): string {
  return Buffer.from(
    JSON.stringify({ t: item.created_at.toISOString(), id: item.id })
  ).toString('base64url');
}

function decodeCursor(cursor: string): { t: string; id: string } {
  return JSON.parse(Buffer.from(cursor, 'base64url').toString());
}

async function listOrders(params: {
  cursor?: string; limit?: number; userId: string
}) {
  const limit = Math.min(params.limit ?? 20, 100); // cap at 100
  let query = db.from('orders')
    .where({ user_id: params.userId })
    .orderBy([{ column: 'created_at', order: 'desc' }, { column: 'id', order: 'desc' }])
    .limit(limit + 1); // fetch one extra to determine has_more

  if (params.cursor) {
    const { t, id } = decodeCursor(params.cursor);
    // Seek: rows where (created_at, id) < cursor
    query = query.where(
      db.raw('(created_at, id) < (?, ?)', [t, id])
    );
  }

  const rows = await query;
  const hasMore = rows.length > limit;
  const items   = rows.slice(0, limit);

  return {
    data:        items,
    has_more:    hasMore,
    next_cursor: hasMore ? encodeCursor(items[items.length - 1]) : null,
    // Expose total count only when cheap (add a count query if needed)
  };
}

Section 03

Idempotency: the difference between a retry and a disaster

Networks are unreliable. Mobile clients lose connectivity mid-request. Load balancers time out. Servers restart mid-flight. The client doesn't know if its request was received, processed, or lost. So it retries. If your API isn't idempotent, that retry creates a double charge, a duplicate order, or a second email to the user.

An operation is idempotent if performing it multiple times has the same effect as performing it once. GET, PUT, and DELETE are idempotent by definition (though your implementation might violate this). POST is not — POST /orders twice creates two orders. Making POST safe to retry requires explicit design.

The Idempotency-Key header

The pattern: the client generates a unique key (UUID v4) and sends it in the Idempotency-Key header. The server stores the key and the response. On retry, the server detects the key has been seen before and returns the stored response without re-executing. The client gets the same result. No double charge.

Implementation details that matter:

  • Key scope: keys should be scoped to a specific operation. An idempotency key for POST /charges should not be reusable for POST /refunds. Stripe enforces this — the same key on a different endpoint type returns a 422.
  • Key expiry: 24 hours is the standard. Long enough for retry storms. Short enough not to fill your Redis indefinitely.
  • Race conditions: two requests with the same key arriving simultaneously. Use Redis SET NX EX as a distributed lock — only one wins, the other waits and then gets the stored response.
  • Storage: Redis with TTL is the right backend. Don't use your primary database for idempotency key storage — it's on the critical path of every mutation.
Idempotency middleware — production-ready TYPESCRIPT 
import { Request, Response, NextFunction } from 'express';
import redis from './redis';

const IDEMPOTENCY_TTL = 86400; // 24 hours

export async function idempotencyMiddleware(
  req: Request, res: Response, next: NextFunction
) {
  const key = req.headers['idempotency-key'] as string;
  if (!key || !/^[0-9a-f-]{36}$/i.test(key)) {
    return res.status(400).json({
      error: { code: 'MISSING_IDEMPOTENCY_KEY', message: 'Valid UUID required in Idempotency-Key header' }
    });
  }

  const storeKey = `idem:\${req.path}:\${key}`; // scoped to endpoint

  // Try to acquire lock — SET NX = only set if not exists
  const lock = await redis.set(storeKey + ':lock', '1', 'NX', 'EX', 30);
  if (!lock) {
    // Another request with this key is in-flight — wait for it
    const stored = await pollForResult(storeKey, 10_000);
    if (stored) return replayStoredResponse(res, stored);
    return res.status(409).json({ error: { code: 'CONCURRENT_REQUEST' } });
  }

  // Check for existing result
  const existing = await redis.get(storeKey);
  if (existing) {
    await redis.del(storeKey + ':lock');
    return replayStoredResponse(res, JSON.parse(existing));
  }

  // Intercept response to store it
  const originalJson = res.json.bind(res);
  res.json = function(body) {
    const stored = { status: res.statusCode, body };
    redis.setex(storeKey, IDEMPOTENCY_TTL, JSON.stringify(stored));
    redis.del(storeKey + ':lock');
    return originalJson(body);
  };

  next();
}

function replayStoredResponse(res: Response, stored: { status: number; body: unknown }) {
  res.set('Idempotent-Replayed', 'true'); // signal to client it's a replay
  res.status(stored.status).json(stored.body);
}

Section 04

Error design: the API surface that gets the least attention and causes the most pain

Error responses are your API's most important feature. When things go right, any response body structure works. When things go wrong, the error response is the only signal your client has to decide what to do next — retry, show a user-facing message, alert the engineering team, or silently log and continue.

Most APIs return errors as an afterthought: {"error": "something went wrong"} or the classic {"success": false, "message": "Invalid email"}. These are useless to machines. The machine can't distinguish a retryable error from a permanent failure, a validation error from an authentication error, or a problem with field "email" from a problem with field "phone".

The anatomy of a good error response

A production error response needs:

  • error.code — a stable machine-readable string, never changes. INSUFFICIENT_FUNDS, not "Insufficient funds." Clients switch on this.
  • error.message — a human-readable description for developers, not users. Can change between versions.
  • error.details — field-level validation errors, structured. [{ field: "email", issue: "invalid_format" }]
  • error.request_id — the correlation ID for this request. The user can give this to support; support can find the server-side trace in seconds.
  • error.docs_url — a link to documentation for this error code. Stripe does this; it's the fastest path from "what does this mean" to "I know what to do."

HTTP status codes are not optional

The status code must be semantically correct. Not as a formality — because every layer of infrastructure between you and the client makes decisions based on it:

  • 400 — bad request (client error, don't retry)
  • 401 — unauthenticated (no credentials or invalid credentials)
  • 403 — unauthorised (valid credentials, insufficient permission)
  • 404 — not found (the resource doesn't exist)
  • 409 — conflict (state conflict — concurrent edit, duplicate key)
  • 422 — unprocessable entity (validation error — correct format, wrong content)
  • 429 — rate limited (retry after Retry-After seconds)
  • 500 — server error (your fault, client should retry with backoff)
  • 503 — service unavailable (retry, with Retry-After)

The worst sin: 200 OK with an error body. CDNs cache 200s. Monitoring doesn't alert on 200s. Retry logic doesn't trigger on 200s. You've broken every automatic system that depends on HTTP semantics.

Section 05

HTTP semantics: the rules most teams quietly ignore

REST is built on top of HTTP's semantics. When you follow them, you get caching, retry logic, and monitoring for free. When you violate them, you pay the cost — manually reimplementing all of that, badly.

GET: safe and idempotent

GET must be safe (no side effects) and idempotent (multiple calls = same result). Browsers prefetch GET requests. CDNs cache them. Load balancers retry them on failure. Violating this — GET /orders/confirm/{id} that actually confirms an order — can result in orders being confirmed twice by a retry, or a browser prefetch that fires an irreversible action.

POST: neither safe nor idempotent by default

POST creates new resources or triggers actions. It is neither safe nor idempotent. Browsers warn before re-submitting a form via POST. CDNs don't cache POST responses. Client libraries don't auto-retry POST. This is why the idempotency key pattern exists — you opt into idempotency explicitly when you need it.

PUT vs PATCH: the difference that matters

PUT replaces the entire resource. PATCH applies a partial update. If your users object has 20 fields and you want to update just the email, PUT requires sending all 20 fields (or you'll null out the ones you omit). PATCH only requires sending the changed fields. The rule: use PATCH for partial updates, PUT only when you genuinely intend to replace the entire resource (uploading a file, replacing config).

DELETE: idempotent, but tricky

DELETE is idempotent — deleting something twice should have the same result as deleting it once. But many implementations return 404 on the second delete (the resource is already gone). Strictly speaking, returning 404 on a second delete violates idempotency — the response changed. The correct response for an already-deleted resource is 204 No Content, same as the first delete. Stripe gets this right.

Section 06

API versioning: the decision you can't easily undo

At some point you'll need to make a breaking change. A field rename, a type change, a removed endpoint, a different authentication mechanism. The question isn't whether to version — it's how. The three main strategies each make different tradeoffs between client simplicity, server complexity, and REST purity.

URI versioning: /v1/, /v2/

The most common approach. Version is in the URL: /v1/orders, /v2/orders. Clean, visible, easy to route in a reverse proxy (location /v1/ { proxy_pass backend_v1; }). Browsers can bookmark versioned URLs. Logs clearly show which version is being called.

The downsides: it "breaks REST" purists (a resource has one URL, not two). You end up maintaining /v1/ indefinitely because someone always has clients on the old version. And you can't easily serve different versions of different resources — /v2/orders but /v1/users requires explicit routing rules.

Accept header versioning

Accept: application/vnd.example.v2+json. Theoretically the most RESTful — the resource URL is stable, the representation version is in content negotiation. The reality: most HTTP clients, logging systems, and reverse proxies don't surface the Accept header. Caches must include Vary: Accept or they'll serve the wrong version. Debugging is harder — you can't tell from a URL which version is being called.

Query parameter versioning

GET /orders?version=2. Simplest to implement. Easiest to test (just change the URL). Works without custom HTTP client configuration. The problem: query parameters are supposed to filter/sort resources, not change the API contract. And caches treat ?version=1 and ?version=2 as different resources (usually correct behaviour, but semantically odd).

The practical answer

For almost all production APIs: use URI versioning. It's not the most theoretically pure, but it's the most operationally sane. Stripe, GitHub, Twilio — all use URI versioning. The REST purity argument is less important than the reality that your ops team can route, monitor, and debug by URL.

Regardless of strategy: add versioning before you need it. An unversioned API that needs a breaking change has no escape hatch. The cost of starting with /v1/ upfront is two characters in every URL. The cost of retrofitting versioning is a multi-month migration project.

Section 07

Rate limiting: protecting your API and teaching clients to be good citizens

Rate limiting serves two purposes: protecting your infrastructure from abuse (intentional or accidental), and giving clients a clear signal to slow down. The second purpose is as important as the first — a client that gets 429 with a Retry-After header can back off gracefully. A client that gets connection timeouts because your server is overloaded has no idea what to do.

The token bucket algorithm

Token bucket is the most intuitive rate limiting algorithm. Imagine a bucket with a capacity of N tokens. The bucket refills at a constant rate (R tokens per second). Each request consumes one token. If the bucket is empty, the request is rejected.

The key property: token bucket allows short bursts. A bucket with capacity 100 that refills at 10/second allows 100 requests in one second as long as the bucket was full, then falls back to 10/second. This is correct behaviour for real clients — they have bursty patterns, and you don't want to penalise bursts if the sustained rate is within limit.

What to rate limit on

The answer is usually "multiple things simultaneously": per API key (primary), per IP address (secondary, for unauthenticated endpoints), per user ID (for authenticated endpoints where one user can have multiple API keys). Each dimension catches a different abuse pattern.

The headers clients need

A 429 without guidance is useless. Always include:

  • X-RateLimit-Limit — the total limit for the window
  • X-RateLimit-Remaining — how many requests are left
  • X-RateLimit-Reset — Unix timestamp when the window resets
  • Retry-After — seconds until the client can retry (on 429)

Return these headers on every response, not just 429s. Clients can use them to self-throttle before hitting the limit — a well-designed client watches X-RateLimit-Remaining and slows down proactively.

Section 08

Sparse fieldsets: the bandwidth win nobody implements

Your order object has 40 fields. Your mobile feed view needs 4 of them: id, status, total_amount, created_at. But your GET /orders endpoint returns all 40, including nested objects, embedded metadata, and computed fields. You're sending 10x more data than needed on every request.

Sparse fieldsets (or field selection) let the client specify exactly which fields to return: GET /orders?fields=id,status,total_amount,created_at. For a mobile client on a slow connection making this call 100 times per session, the bandwidth and parse time savings are substantial.

Implementation approaches

The simplest approach: parse the fields query parameter, validate each field name against an allowlist, then either project at the database level (SELECT id, status, total_amount, created_at FROM orders) or filter the full object in application code before serialising. Database-level projection is more efficient (less data transferred from DB), but application-level filtering is simpler to implement and still saves network bandwidth.

For nested fields, a dot-notation syntax works well: ?fields=id,status,payment.method,shipping.address.city. This is how the Google API Design Guide specifies field masks, and how many gRPC-based APIs expose it.

🚀
GraphQL vs REST field selection

GraphQL was largely invented to solve exactly this problem — clients specifying exactly what data they need. If you find yourself building increasingly complex ?fields= and ?embed= and ?expand= parameters, you've reinvented a subset of GraphQL. At that complexity level, consider whether GraphQL (or gRPC with field masks) is the right tool. For most APIs, simple ?fields= is sufficient and much simpler to maintain.

Section 09

Production reality: the API mistakes that hurt

The 200 OK lie

I've seen this in production at multiple companies: an API that returns 200 OK with {"success": false, "error": "Payment declined"}. The developer who built it thought they were being helpful by always returning 200. The consequence: the CDN caches the payment declined response and serves it to the next user. The monitoring system never fires a payment failure alert (it's watching 4xx/5xx rates). The client-side retry logic doesn't kick in. The transaction just silently disappears.

The integer ID migration

You launched with integer IDs: {"id": 12345}. You scale to 10 billion records. Integer overflow. Now you need UUIDs. Except every client has hard-coded parseInt(order.id) because it was an integer. The migration requires a versioned breaking change and touching every client. The fix is to always return IDs as strings, even if they're currently integers: {"id": "12345"}. JavaScript loses precision on integers above 2^53 anyway — string IDs are universally safer.

Leaking internals through errors

Stack traces, SQL query text, internal service names, table names, column names — all have appeared in production API error responses. Beyond being a security risk (attackers learn your schema from your errors), it creates a maintenance nightmare: now your DB schema is part of your API contract. Always sanitise error responses at the boundary. Never res.json({ error: err.stack }).

The missing Retry-After

You add rate limiting. You return 429. But you forget Retry-After. Your mobile clients hit the limit, see 429, don't know when to retry, so they retry immediately with exponential backoff. But they start the backoff from "now" — which means the first retry is in 1 second, the second in 2 seconds, etc. If your rate limit window resets in 30 seconds, the first 5 retries all fail. The client exponential-backoffs for 30 seconds doing wasted retries, then recovers. With Retry-After: 30, the client waits exactly 30 seconds and the first retry succeeds.

Section 10

The things nobody tells you

Your API is never truly versioned — it's your clients that are

When you release /v2/, you don't get to retire /v1/. There's always a client that didn't upgrade — an old mobile app still in use, a third-party integration that was set up and never updated, a B2B partner whose engineering team is "getting to it." Stripe still serves /v1/ from 2011. Plan for it: /v1/ is a commitment you're making to your clients, not a temporary measure.

HATEOAS sounds good in theory and fails in practice

HATEOAS (Hypermedia As The Engine Of Application State) is the theoretical ideal: responses embed links to related actions, clients discover the API from the responses rather than documentation. In practice, nobody has made this work at scale. Clients are built with explicit knowledge of the API. Every real production API uses documentation-driven contracts, not hypermedia discovery. Don't implement HATEOAS; document your API well instead.

Your documentation is your API

At the client layer, the documentation is the contract. If your docs say a field is always present and your code occasionally omits it, your clients break — not because your code is wrong, but because you violated the documented contract. Treat your API documentation like production code: version it, review it, test it. The best APIs (Stripe, Twilio) have documentation that's as carefully crafted as the code itself.

Null vs missing field: a breaking change you might not notice

If your API returns {"shipping_address": null} and you later change it to omit the field entirely (because "null and missing are the same thing, right?"), you've broken every client that does if (order.shipping_address !== undefined) vs if (order.shipping_address !== null). They're different. Pick one convention (prefer null for "present but empty," omit for "not applicable") and never change it.

API design decisions

Designing a new endpoint — ask in this order

These questions surface the decisions that are expensive to reverse. Answer them before writing the handler.

1
What is the resource? Name it as a noun, not an action.
If you find yourself typing a verb into the URL (/createOrder, /getUser, /cancelSubscription), stop. The noun is your resource (orders, users, subscriptions). The HTTP method is the verb. Locking in action-based URLs means every new action needs a new endpoint — you end up with 40 endpoints where 8 would do.
rule: URL = noun, method = verb
2
Will this endpoint mutate state? Add a version prefix now.
If the answer is yes — or even "maybe someday" — start the URL with /v1/. The cost is 4 characters. The cost of retrofitting versioning after clients are in production is months of migration work. You will need a breaking change eventually. Whether you have an escape hatch when that day comes depends on a decision you make today.
no version prefix = no escape hatch for breaking changes
3
Can this endpoint be called twice safely? If not, add idempotency.
Mobile clients retry on network failure. Load balancers retry on timeout. Your client SDK probably has automatic retry. If POST /v1/charges is called twice with the same intent, does the customer get charged twice? If yes, require an Idempotency-Key header and deduplicate in Redis. This decision made after launch requires a client-side migration. Made now, it's 30 lines of middleware.
any money/order/email endpoint needs this
4
Could this return more than 100 rows? Design cursor pagination now.
Offset pagination works fine at 1,000 rows and breaks at 1,000,000. The fix (cursor pagination) requires a different query structure and a different API contract — you can't switch transparently after clients depend on ?page=47. If there's any chance this list grows, use cursor pagination from day one. The cursor-based contract is a strict superset of what clients actually need from a list endpoint.
offset pagination = technical debt with a deadline
5
What's the error contract? Define the error codes now.
Decide your machine-readable error codes before you write the happy path. INSUFFICIENT_FUNDS, DUPLICATE_ORDER, RATE_LIMITED — these are part of the API contract. Clients switch on them. If you ship "error": "something went wrong" and clients start handling it with string matching, you can never change the message. Machine codes are stable; messages can evolve.
error codes = the contract clients actually depend on
6
Will mobile clients use this? Add sparse fieldset support.
If yes: your order object with 40 fields costs the same bandwidth whether a mobile feed needs 4 of them or all 40. Add ?fields=id,status,total,created_at support. At 1M requests/day, the difference between sending 4 fields vs 40 is often hundreds of MB of bandwidth per day. This is a query parameter and a filter — add it to any endpoint that returns more than 5 fields and has mobile consumers.

URI versioning from day one. Commit to supporting all published versions.

4xx
Error codes

Structured errors with machine codes, request_id, docs_url. Never 200 with error body.

CUR

Cursor pagination for all list endpoints. Offset only for bounded small datasets.

429
Rate limiting

Rate limit at gateway. Always include Retry-After and X-RateLimit-* headers.

STR

Always return IDs as strings. Avoid future integer overflow migrations.
