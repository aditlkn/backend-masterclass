Deep Dive 01

Cursor pagination:
the full SQL implementation

The concept is straightforward — use a cursor instead of an offset. The implementation has several non-obvious requirements: the cursor must encode enough to resume exactly, the index must support the seek pattern efficiently, and the sort order must be stable across pages.

Why the index matters as much as the query

Cursor pagination only beats offset pagination if the database can use an index to seek directly to the cursor position. If it can't, it does a full scan anyway — same cost as offset, different query shape. The index must match the ORDER BY columns exactly.

For ORDER BY created_at DESC, id DESC (the standard time-based cursor), you need a composite index on (created_at DESC, id DESC) or equivalently (created_at, id) with the query using the correct direction. PostgreSQL will use a (created_at, id) index for both ascending and descending queries via a backwards scan.

The sort stability problem

Sorting by created_at DESC alone is not stable — if two rows have the same created_at timestamp (common in bulk inserts), their relative order is non-deterministic. Page 1 might return row A before row B, and page 2's cursor might skip both or return one twice, depending on how the query planner resolves ties.

Fix: always include a unique column as the tiebreaker. ORDER BY created_at DESC, id DESC is stable because id is unique. The cursor must encode both values.

-- Index: must match ORDER BY exactly for index seek (not full scan)
-- Run this once: CREATE INDEX CONCURRENTLY idx_orders_cursor
-- ON orders (created_at DESC, id DESC);
-- CONCURRENTLY = no table lock, safe on production

-- The seek query: WHERE (created_at, id) < ($cursor_time, $cursor_id)
-- PostgreSQL uses the composite index to seek directly to the cursor
-- Cost: O(page_size) regardless of how far into the dataset you are

interface Cursor { t: string; id: string; }

function encodeCursor(row: { created_at: Date; id: string }): string {
  const payload: Cursor = { t: row.created_at.toISOString(), id: row.id };
  return Buffer.from(JSON.stringify(payload)).toString('base64url');
}

function decodeCursor(cursor: string): Cursor {
  try {
    return JSON.parse(Buffer.from(cursor, 'base64url').toString());
  } catch {
    throw new ApiError(400, 'INVALID_CURSOR', 'Cursor is malformed or expired');
  }
}

async function listOrders({ cursor, limit = 20, userId }: {
  cursor?: string; limit?: number; userId: string;
}) {
  const safeLimit = Math.min(Math.max(limit, 1), 100); // clamp 1-100

  let rows: Order[];

  if (!cursor) {
    // First page: no WHERE clause on cursor, just ORDER BY
    rows = await db.query(`
      SELECT * FROM orders
      WHERE user_id = $1
      ORDER BY created_at DESC, id DESC
      LIMIT $2
    `, [userId, safeLimit + 1]); // +1 to detect has_more
  } else {
    const { t, id } = decodeCursor(cursor);
    // Seek query: ROW comparison uses composite index efficiently
    rows = await db.query(`
      SELECT * FROM orders
      WHERE user_id = $1
        AND (created_at, id) < ($2, $3)
      ORDER BY created_at DESC, id DESC
      LIMIT $4
    `, [userId, t, id, safeLimit + 1]);
  }

  const hasMore = rows.length > safeLimit;
  const items   = rows.slice(0, safeLimit);

  return {
    data:        items,
    has_more:    hasMore,
    next_cursor: hasMore
      ? encodeCursor(items[items.length - 1])
      : null,
    // Don't include total_count by default — it requires a full table scan
    // Add only if explicitly requested and you have a COUNT(*) index
  };
}

Unstable sort causing duplicate/missing rows
Sorting by a non-unique column (status, date without time) means ties are broken arbitrarily. Page 1 and page 2 may overlap or have gaps at the boundary.
→ Always add a unique column (id) as the final sort key. Encode both in the cursor.
Full scan despite having the index
PostgreSQL may choose a sequential scan if the selectivity estimate is wrong (stats stale) or the page size is very large. The query plan changes based on data distribution.
→ Run EXPLAIN ANALYZE after each deploy on representative data. Run ANALYZE after bulk inserts.
Cursor from different sort order passed as next_cursor
If your API supports multiple sort orders and a client passes a cursor from a DESC query to an ASC endpoint, the seek condition is backwards and results are wrong.
→ Include sort direction in the cursor encoding. Validate on decode.
Deep Dive 02

Idempotency edge cases:
the three failure modes

The basic idempotency pattern (check Redis, return stored response if key exists) handles the simple retry case. Production has three additional failure modes that require explicit design.

Edge case 1: concurrent requests with the same key

Two requests with the same idempotency key arrive simultaneously — a race condition in the retry logic. Both check Redis, both get a miss, both start processing. Both complete and try to store their result. You've now processed the operation twice, even with idempotency keys.

Fix: use Redis SET NX EX (Set if Not Exists with expiry) as a distributed lock. Only one request wins the lock. The other waits, then reads the stored result when the lock is released. This is atomic — Redis guarantees only one client wins a SET NX.

Edge case 2: partial failure — operation succeeded but response not stored

The charge to the payment processor succeeds. Your code tries to store the result in Redis. Redis is down. The response never gets stored. The client retries with the same idempotency key. Redis is back up. Redis has no record. Your code processes it again. Duplicate charge.

Fix: store the idempotency record before calling the external service, in a "processing" state. On completion, update it to the final result. On retry, if the status is "processing," wait for completion (or return a 409 if it's been processing too long). This requires a small state machine but prevents the partial-failure duplicate.

Edge case 3: key reuse across different operation types

The same UUID used as an idempotency key for POST /charges on Monday, and then reused (accidentally or maliciously) for POST /refunds on Tuesday. If keys are not scoped to operation type, the refund sees an existing key record from the charge and returns the charge's response — the refund is silently swallowed.

Fix: scope keys to operation type. Store keys as charges:{key} and refunds:{key} in Redis. Or include the endpoint path in the Redis key. Stripe does this — the same key on a different endpoint type returns a 422 Idempotency Key Used For Different Endpoint.

Production idempotency — all three edge cases handledTYPESCRIPT
type IdemState = 'processing' | 'complete';
type IdemRecord = { state: IdemState; status?: number; body?: unknown; };

const TTL       = 86400;    // 24h — idempotency window
const LOCK_TTL  = 30;      // 30s — processing lock
const POLL_MAX  = 10_000; // 10s — wait for concurrent request

export async function idempotencyMiddleware(req: Request, res: Response, next: NextFunction) {
  const rawKey = req.headers['idempotency-key'] as string;
  if (!rawKey) return res.status(400).json({ error: { code: 'MISSING_IDEMPOTENCY_KEY' } });
  if (!/^[0-9a-f-]{36}$/i.test(rawKey)) return res.status(400).json({ error: { code: 'INVALID_IDEMPOTENCY_KEY' } });

  // Edge case 3: scope key to operation type
  const scopedKey = `idem:\${req.method}:\${req.path}:\${rawKey}`;
  const lockKey   = scopedKey + ':lock';

  // Check existing record
  const existing = await redis.get(scopedKey);
  if (existing) {
    const record: IdemRecord = JSON.parse(existing);

    if (record.state === 'processing') {
      // Edge case 1: concurrent request — poll for completion
      const result = await pollForCompletion(scopedKey, POLL_MAX);
      if (!result) return res.status(409).json({ error: { code: 'CONCURRENT_REQUEST', message: 'Request with this key is already processing' } });
      return replay(res, result);
    }

    if (record.state === 'complete') {
      return replay(res, record);
    }
  }

  // Edge case 1: acquire lock — only one request proceeds
  const locked = await redis.set(lockKey, '1', 'NX', 'EX', LOCK_TTL);
  if (!locked) {
    const result = await pollForCompletion(scopedKey, POLL_MAX);
    if (result) return replay(res, result);
    return res.status(409).json({ error: { code: 'CONCURRENT_REQUEST' } });
  }

  // Edge case 2: mark as processing BEFORE external call
  const processingRecord: IdemRecord = { state: 'processing' };
  await redis.setex(scopedKey, TTL, JSON.stringify(processingRecord));

  // Intercept response to store final result
  const origJson = res.json.bind(res);
  res.json = async function(body) {
    const complete: IdemRecord = { state: 'complete', status: res.statusCode, body };
    await redis.setex(scopedKey, TTL, JSON.stringify(complete));
    await redis.del(lockKey);
    return origJson(body);
  };

  next();
}

function replay(res: Response, record: IdemRecord) {
  res.set('Idempotent-Replayed', 'true');
  res.status(record.status!).json(record.body);
}

async function pollForCompletion(key: string, maxMs: number): Promise<IdemRecord | null> {
  const deadline = Date.now() + maxMs;
  while (Date.now() < deadline) {
    const v = await redis.get(key);
    if (v) {
      const r: IdemRecord = JSON.parse(v);
      if (r.state === 'complete') return r;
    }
    await new Promise(r => setTimeout(r, 200));
  }
  return null;
}
Deep Dive 03

Error contract versioning:
stable codes, evolving messages

Your error response structure is a contract just like your success response. Clients write code that switches on error codes, parses field-level details, and displays messages. The rules for evolving this contract safely are different from evolving success responses — violations are often silent.

What's safe to change vs what breaks clients

Safe changes (backward compatible):

  • Adding new fields to the error object (error.docs_url, error.retry_after)
  • Changing error.message text — messages are for humans, not for switch statements
  • Adding new error codes for new error conditions
  • Adding items to error.details array

Breaking changes (clients will fail):

  • Removing or renaming error.code — clients switch on this. Renaming INSUFFICIENT_FUNDS to BALANCE_TOO_LOW silently breaks every client that checks for the old code.
  • Changing the HTTP status code for an existing error — clients may have retry logic or error handling keyed to status codes
  • Changing the shape of error.details items — removing field or renaming issue breaks form validation logic
  • Moving the error wrapper — changing {"error": {...}} to {"errors": [...]} is a breaking structural change

The error code registry

Treat your error codes like an API — maintain a registry, version changes, and never delete or rename published codes. Deprecate instead: continue returning the old code but add a deprecated_code field alongside the new one, and document the migration. Give clients a migration window (1+ API versions) before removing the old code.

Error contract — stable schema, extensible, versioned codesTYPESCRIPT
// errors/codes.ts — the registry. Treat like a database migration.
// Never rename or remove. Only add new codes.
export const ERROR_CODES = {
  // Authentication
  MISSING_TOKEN:        'MISSING_TOKEN',
  INVALID_TOKEN:        'INVALID_TOKEN',
  TOKEN_EXPIRED:        'TOKEN_EXPIRED',

  // Validation
  VALIDATION_ERROR:     'VALIDATION_ERROR',
  INVALID_FIELD:        'INVALID_FIELD',

  // Business logic
  INSUFFICIENT_FUNDS:   'INSUFFICIENT_FUNDS',
  ORDER_ALREADY_PLACED: 'ORDER_ALREADY_PLACED',
  DUPLICATE_ORDER:      'DUPLICATE_ORDER',

  // Rate limiting
  RATE_LIMITED:         'RATE_LIMITED',

  // Server errors
  INTERNAL_ERROR:       'INTERNAL_ERROR',
  SERVICE_UNAVAILABLE:  'SERVICE_UNAVAILABLE',
} as const;

// The stable error envelope — adding fields is safe, removing is not
interface ApiError {
  error: {
    code:        typeof ERROR_CODES[keyof typeof ERROR_CODES];
    message:     string;           // can change
    request_id:  string;           // never remove
    details?:    FieldError[];     // can add items, can't remove fields
    docs_url?:   string;           // added in v1.2, optional
    retry_after?: number;          // added in v1.3, optional
  };
}

interface FieldError {
  field:   string;  // stable: dot-notation path to field
  issue:   string;  // stable: machine-readable issue code
  message: string;  // can change: human-readable description
  value?:  unknown; // added in v1.1: the value that failed
}

// Deprecating a code: add new code, keep old for 2+ versions
// Old: { code: "PAYMENT_DECLINED" }  ← v1 clients still use this
// New: { code: "INSUFFICIENT_FUNDS", deprecated_codes: ["PAYMENT_DECLINED"] }
// Remove PAYMENT_DECLINED in v3 after migration period
Verify your error contract
✓All error codes are in a central registry — no ad-hoc strings scattered across handlers.
✓Every error response has a request_id — test with: curl -I https://api/nonexistent | grep x-request-id
✓No 200 OK with error body — test with invalid auth: response must be 401, not 200.
✓Validation errors include field paths: error.details[0].field should be "email" not "body.email" or undefined.
✓Server errors never leak stack traces: error.message on 500s should be generic. Test with a deliberate crash in a staging endpoint.
