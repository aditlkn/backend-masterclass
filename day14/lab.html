Production Incident  ·  MongoDB  ·  Document Size Limit

Post document hit 16MB — writes failed for the most popular content

Saturday 11:30 IST  ·  Discovered via user reports  ·  Severity: P1  ·  All comments broken on viral posts

The platform launched with a clean MongoDB schema: a posts collection where each post document contained a comments array. Early in the product's life, this worked perfectly — most posts had fewer than 100 comments, documents stayed small, and the embedded approach made reading posts fast.

Eight months later, a founder posted a thread that went viral. It accumulated 24,000 comments over 48 hours. At some point overnight, the post document crossed 16,777,216 bytes — MongoDB's hard document size limit. When the 24,001st comment tried to push to the array, MongoDB returned: MongoServerError: Document failed validation: document too large. Every subsequent comment attempt failed. The post was frozen in time, unable to accept new comments.

Root cause

Comments were embedded in the post document as an unbounded array. MongoDB's 16MB document size limit is a hard ceiling — when hit, all writes to the document fail. The design was correct for typical posts (bounded comment counts) but broke under viral conditions. The fix: separate comments collection with a post_id reference, and only embed the most recent 5 comments in the post document for display performance.

❌ Incident: unbounded embedded array
// posts collection
{ _id: "post_123", title: "...", body: "...",
  comments: [  // ← unbounded array, WILL hit 16MB on viral posts
    { user: "alice", text: "...", created_at: ... },
    { user: "bob",   text: "...", created_at: ... },
    // ... 24,000 more ...
  ]
}

✅ Fix: hybrid — embed top 5, reference the rest
// posts collection — bounded document size
{ _id: "post_123", title: "...", body: "...",
  comment_count: 24001,
  top_comments: [           // ← latest 5 comments, embedded for speed
    { _id: ObjectId("..."), user: "alice", text: "Great post!", created_at: ... },
    // 4 more max
  ]
}

// comments collection — unbounded, no problem
{ _id: ObjectId("..."), post_id: "post_123",
  user_id: "usr_456", user_name: "alice",
  text: "Great post!", likes: 42,
  created_at: ISODate("...")
}

// Add a comment (Node.js)
async function addComment(postId, userId, text) {
  // 1. Insert into comments collection (always works, no size limit)
  const comment = await db.comments.insertOne({
    post_id: postId, user_id: userId, text, created_at: new Date()
  });
  // 2. Update post: increment count, update top_comments (capped at 5)
  await db.posts.updateOne({ _id: postId }, {
    $inc: { comment_count: 1 },
    $push: { top_comments: { $each: [comment], $slice: -5, $sort: { created_at: -1 } } }
  });
}

// Index on comments for efficient lookup
db.comments.createIndex({ post_id: 1, created_at: -1 })
What this incident teaches
▶Never embed unbounded arrays in MongoDB. Comments, events, messages, notifications — any array that can grow without bound will eventually hit 16MB. The question isn't "will it happen?" but "when?"
▶The hybrid pattern solves the display performance vs scalability tradeoff. Embed only the data needed for initial page load (top 5 comments). Store the full collection separately for pagination. One document read gets you the post + preview; subsequent reads paginate from the collection.
▶Design for the worst case, not the average case. The average post had 12 comments. The schema was designed for the average. The viral post had 24,000. Always ask: "What happens when this is 1000× larger than expected?"
▶Add document size monitoring. Monitor Object.bsonsize(doc) or use MongoDB Atlas's document size metrics. Alert when any document in a collection exceeds 1MB — it means you're on track to hit the limit.
