Section 01

Why NoSQL exists — and what it actually solves

NoSQL databases didn't emerge because SQL was bad. They emerged because specific classes of problems don't map well to the relational model, and trying to force them creates either schema pain or performance pain. Understanding the actual problems each NoSQL database solves prevents the most common mistake: picking a NoSQL database because it sounds modern, then spending months fighting the data model.

The four NoSQL categories

Document stores (MongoDB, CouchDB): Data naturally exists as nested objects. A blog post has tags, comments, and metadata — all logically part of the post, awkward to split across three relational tables with JOINs. Documents store the whole object together, making reads of the complete entity cheap. Poor fit for: anything requiring cross-document queries, strong consistency, or complex multi-entity transactions.

Key-value stores (Redis, DynamoDB): The simplest possible model — a key maps to a value. O(1) lookup. Extremely fast. Poor fit for: anything requiring queries other than by exact key, complex data structures that need to be partially updated.

Wide-column stores (Cassandra, HBase): Data is organised by partition key (determines which node) and clustering columns (determines sort order within the partition). Optimised for time-series, IoT, activity feeds — anything where you always query "all items for entity X, sorted by time." Poor fit for: ad-hoc queries, JOINs, anything without a clear partition key.

Graph databases (Neo4j, Neptune): Data is fundamentally relational — friends of friends, product recommendations based on purchase graph, fraud detection. Poor fit for: non-graph data where the graph model adds complexity without benefit.

⚠️
NoSQL ≠ schema-less in practice

MongoDB is called schema-free but your application code enforces a schema anyway — it expects specific fields, validates types, and breaks when documents have unexpected shapes. The schema just moved from the database to your application layer, where it's harder to enforce and easier to let drift. Use schema validation in MongoDB (JSON Schema) or a library like Zod/Joi. "No schema" means no help from the database when you violate your implicit schema.

Section 02

MongoDB Document modeling

MongoDB stores data as BSON documents (Binary JSON) in collections. The fundamental design question is always: what does your application need to read, and what shape does it need that data in? The schema should be the answer to that question — not a normalised representation of an abstract data model.

MongoDB's sweet spots

  • Variable-shape data: Products in an e-commerce store have wildly different attributes — a shirt has size/color, a laptop has RAM/storage/CPU. In SQL: EAV tables or JSON columns. In MongoDB: each product document has exactly the fields it needs.
  • Nested entities with co-read patterns: A blog post with its tags, author info, and first 5 comments. All fetched together, all written together. One document, one read.
  • Rapid iteration: Schema changes in MongoDB are just application deployments — no ALTER TABLE, no migration locks. Add a field, old documents just don't have it (handle with defaults).

MongoDB's weaknesses

Cross-collection queries: `$lookup` (MongoDB's JOIN) is expensive and doesn't use indexes effectively. If your access patterns require frequent joins across collections, PostgreSQL is likely a better fit.

Transactions: Multi-document ACID transactions exist in MongoDB 4.0+ but are slower than single-document operations. Document modeling should aim to make most operations single-document.

Aggregations at scale: Complex `$group`, `$unwind`, `$facet` pipelines on large collections can be slow. MongoDB is not an analytics database.

Section 03

MongoDB: embed vs reference

Every relationship in MongoDB comes down to one question: should the related data be embedded in the parent document, or stored in a separate collection with a reference? The answer is always determined by access patterns, not by normalisation theory.

Embed when:

  • The child data is always read with the parent (comments when displaying a post)
  • The child data belongs to exactly one parent (order line items belong to one order)
  • The child data is bounded in size (a user's last 10 notifications — not all-time history)
  • The child data is updated only through the parent (updating a post's tags)

Reference when:

  • The child data is read independently (a user's profile is read without needing their orders)
  • The child data belongs to many parents (a product referenced by many orders)
  • The child data is unbounded (all comments ever on a post — could be millions)
  • The child data is updated independently (a user's address changes without touching their orders)
🔴
The 16MB document size limit

MongoDB documents cannot exceed 16MB. If you embed unbounded arrays — all comments on a viral post, all events in a long-running session, all messages in a chat thread — you will eventually hit this limit. The document can no longer be updated, writes fail. Model unbounded lists as separate collections, not embedded arrays. A common pattern: store the first 5 comments embedded in the post document, reference the rest in a comments collection.

Section 04

Redis Data structures as a service

Redis is not a cache with some extra features. It's a data structure server that happens to be extremely fast (operations are in-memory) and optionally persistent. The distinction matters: you can model complex problems correctly using Redis's native data structures — sorted sets, streams, pub/sub — rather than bending its key-value interface to store serialised JSON for everything.

Redis data structures and their use cases

String: The simplest. A key maps to a bytes value (can be a number, JSON blob, image). Use for: cache entries, counters (INCR), session tokens, feature flags, distributed locks (SET key value NX EX 30).

Hash: A key maps to a field-value map. Use for: user session data (each field is a session attribute), object properties you need to update individually (increment login_count without fetching the whole object).

List: Ordered sequence, push/pop from either end. O(1) push/pop, O(N) access by index. Use for: message queues, activity feeds with chronological ordering, job queues (LPUSH / BRPOP).

Set: Unordered unique values. Use for: tracking unique visitors (add user IDs, count the set), tags on posts, mutual friends (intersection of two user sets), deduplication.

Sorted Set (ZSet): Unique values, each with a score. Ordered by score. Use for: leaderboards (score = points, ZRANGE by rank), rate limiting (score = timestamp), expiring data (score = expiry timestamp, ZRANGEBYSCORE to find expired).

Stream: Append-only log with consumer groups. Persistent ordered sequence. Use for: event sourcing, activity logs, replacing simple Kafka use cases, IoT data ingestion.

Pub/Sub: Fire-and-forget messaging. Publishers push to channels, subscribers receive. No persistence — if subscriber is offline, messages are lost. Use for: real-time notifications, live dashboards, cache invalidation signals.

Section 05

Redis patterns in production

Cache-aside (lazy loading)

The most common pattern. Application checks Redis first. On miss, reads from the database and writes to Redis with a TTL. Cache is populated lazily — only cached when actually requested. The cold start problem: after a deploy or Redis flush, all reads hit the database until the cache warms up.

Write-through

Every database write also writes to Redis. Cache is always warm, never cold. Cost: every write is 2 operations. Use when: read-to-write ratio is high and you can't afford cache misses.

Distributed lock with Redlock

SET lock:resource_id unique_value NX EX 30 — atomic set only if key doesn't exist, with 30-second expiry. Only one process can hold the lock. On expiry, the lock auto-releases even if the holder crashes. Critical: the lock expiry must exceed the maximum expected operation time plus clock drift.

Rate limiting with sorted sets

Sliding window rate limiting: each request adds a member to a sorted set with score = timestamp. Remove members older than the window. Count remaining members. If count exceeds limit, reject. Atomic with Lua scripts or Redis transactions.

Redis patterns — production implementationsTYPESCRIPT
// ── Cache-aside pattern ────────────────────────────────────────
async function getUser(id: string): Promise<User> {
  const cached = await redis.get(`user:\${id}`);
  if (cached) return JSON.parse(cached);
  const user = await db.query('SELECT * FROM users WHERE id=$1', [id]);
  await redis.setex(`user:\${id}`, 3600, JSON.stringify(user)); // 1hr TTL
  return user;
}

// ── Distributed lock ──────────────────────────────────────────
async function withLock<T>(key: string, ttlSec: number, fn: () => Promise<T>): Promise<T> {
  const lockVal = crypto.randomUUID();
  const acquired = await redis.set(`lock:\${key}`, lockVal, 'NX', 'EX', ttlSec);
  if (!acquired) throw new Error(`Failed to acquire lock: \${key}`);
  try { return await fn(); }
  finally {
    // Only release if we still own it (check-and-delete atomically via Lua)
    const script = `if redis.call('get',KEYS[1])==ARGV[1] then return redis.call('del',KEYS[1]) else return 0 end`;
    await redis.eval(script, 1, `lock:\${key}`, lockVal);
  }
}

// ── Sliding window rate limiter ───────────────────────────────
async function isRateLimited(userId: string, limit: number, windowMs: number): Promise<boolean> {
  const key = `ratelimit:\${userId}`;
  const now = Date.now();
  const windowStart = now - windowMs;
  const pipeline = redis.pipeline();
  pipeline.zremrangebyscore(key, 0, windowStart);  // remove old requests
  pipeline.zadd(key, now, `\${now}-\${Math.random()}`);     // add this request
  pipeline.zcard(key);                                      // count in window
  pipeline.pexpire(key, windowMs);                          // auto-expire the key
  const results = await pipeline.exec();
  const count = results[2][1] as number;
  return count > limit;
}

// ── Leaderboard with sorted set ───────────────────────────────
async function addScore(userId: string, score: number) {
  await redis.zadd('leaderboard:global', score, userId);
}
async function getTopPlayers(n: number) {
  return redis.zrevrange('leaderboard:global', 0, n - 1, 'WITHSCORES');
}
async function getUserRank(userId: string) {
  return redis.zrevrank('leaderboard:global', userId); // 0-indexed rank
}

Section 06

Cassandra Wide-column and query-driven design

Cassandra is built for one thing: massive write throughput at high availability across multiple datacenters, with predictable read latency for specific access patterns. It achieves this by being extremely opinionated about how data is stored and accessed. You don't get flexible queries — you get near-infinite write scale for the queries you designed for.

How Cassandra is different

Cassandra is not a relational database with a different storage engine. Its data model, consistency model, and operational characteristics are fundamentally different:

  • No JOINs: Ever. Each query must be answered by a single table. You denormalise aggressively, maintaining multiple copies of data optimised for different queries.
  • No secondary indexes (practically): Cassandra's secondary indexes are local — they don't work well at scale. Every query should go by partition key.
  • Eventual consistency: By default. You can tune consistency levels per-query (ONE, QUORUM, ALL) trading latency for consistency.
  • No ACID transactions across partitions: Lightweight transactions (LWT) provide compare-and-set within a single partition, but they're slow. Design around them.
  • Distributed by default: Every table is automatically sharded across nodes by partition key. No manual sharding configuration.

The partition key and clustering columns

Every Cassandra table has a partition key — determines which node stores the data. All data with the same partition key is stored together on the same node, sorted by clustering columns. This physical layout is exactly why Cassandra is fast for specific queries: a query by partition key returns all matching rows from a single node, already sorted by the clustering columns.

The query you need to run determines the primary key you must design. If you need "all events for device X, ordered by time, newest first" — your primary key is ((device_id), timestamp DESC). Design the table for the query, not the data.

Section 07

Cassandra modeling rules

Cassandra's data modeling has a set of rules that, if violated, lead to performance problems that can't be fixed without redesigning the table. These aren't guidelines — they're constraints imposed by the physical storage model.

Rule 1: Model around queries, not entities

Start with the queries you need to support. For each query, design a table that answers it efficiently. You will have multiple tables storing the same data differently. This is correct — it's not duplication, it's optimisation.

Rule 2: Partition key determines co-location

All rows with the same partition key are stored on the same node. This means: queries that filter by partition key are fast (single node). Queries that scan across partitions are slow (full cluster scan, ALLOW FILTERING). Never use ALLOW FILTERING in production — it's a cluster-wide scan.

Rule 3: Avoid hot partitions

If all your writes go to one partition (e.g. partition key = today's date and you write one event per millisecond), one node gets all the traffic — a Cassandra hot partition. Mitigate by adding a bucket suffix: partition_key = date + '_' + (user_id % 10) distributes load across 10 partitions.

Rule 4: Limit partition size

Cassandra partitions have a soft limit of ~100MB and a practical limit in terms of performance around 10MB. An unbounded partition (all events for a device with no date range in the key) grows forever and becomes slow to read and compact. Always include a time bucket in your partition key for time-series data.

Cassandra table design — IoT event storageCQL
-- ❌ Bad: unbounded partition
CREATE TABLE device_events_bad (
  device_id   UUID,
  event_time  TIMESTAMP,
  event_type  TEXT,
  payload     TEXT,
  PRIMARY KEY (device_id, event_time)
) WITH CLUSTERING ORDER BY (event_time DESC);
-- Problem: one device active for years → partition grows unboundedly
-- A device that generates 1 event/second → 86,400 rows/day → 3.1M rows/year
-- Single partition, single node, hot spot

-- ✅ Good: bucketed partition by month
CREATE TABLE device_events (
  device_id   UUID,
  month       TEXT,          -- '2024-06'
  event_time  TIMESTAMP,
  event_type  TEXT,
  payload     MAP<TEXT, TEXT>,
  PRIMARY KEY ((device_id, month), event_time)
) WITH CLUSTERING ORDER BY (event_time DESC)
  AND default_time_to_live = 7776000;  -- 90 days TTL, auto-deletes old data

-- Queries that work:
SELECT * FROM device_events
WHERE device_id = ? AND month = '2024-06'
ORDER BY event_time DESC LIMIT 100;
-- ✓ Hits exactly one partition. Fast.

-- Queries that need multiple reads (but still designed):
-- "Last 30 days" = 1 query per month, merge in application
-- months = ['2024-06', '2024-05'] → 2 queries, merge results

-- ❌ ALLOW FILTERING (never in production)
SELECT * FROM device_events WHERE event_type = 'temperature' ALLOW FILTERING;
-- Scans ALL partitions on ALL nodes. Full cluster scan. Will time out.

-- ✅ Create a separate table for the query pattern you need
CREATE TABLE device_events_by_type (
  device_id   UUID,
  event_type  TEXT,
  month       TEXT,
  event_time  TIMESTAMP,
  payload     MAP<TEXT, TEXT>,
  PRIMARY KEY ((device_id, event_type, month), event_time)
) WITH CLUSTERING ORDER BY (event_time DESC);

Section 09

Production reality

PostgreSQL can do 80% of what you think you need MongoDB for

JSONB columns in PostgreSQL support full GIN-indexed queries on nested JSON, array operations, and key-exists checks. If you're considering MongoDB for its schema flexibility or nested document support, benchmark PostgreSQL with JSONB first. The operational simplicity of one database is usually worth more than the data-model fit of two.

Redis is not durable by default

Redis's default configuration is no persistence — a restart loses all data. Enable AOF (Append Only File) persistence or RDB snapshots for any data you can't afford to lose. Even with persistence, Redis is not a primary database for user data — use it as the accelerator layer in front of a persistent database.

Cassandra's tombstone problem

When you delete data in Cassandra, it writes a tombstone — a deletion marker. Tombstones accumulate until compaction clears them. If you do frequent deletes (expired sessions, time-bounded data with application-level cleanup), tombstone accumulation can cause severe read latency. Use TTL (default_time_to_live) instead of application-level deletes — Cassandra handles TTL-based deletion during compaction efficiently.

MongoDB's write concern and journaling

By default, MongoDB writes are acknowledged only after writing to the journal (write concern w:1, j:true on 4.0+). But in a replica set, w:1 only waits for the primary — a primary failure immediately after acknowledgment can lose the write if the primary's journal wasn't replicated. For financial data: use w:"majority" which waits for a quorum of replicas to acknowledge.

Section 10

The things nobody tells you

MongoDB ObjectId contains a timestamp

The first 4 bytes of a MongoDB ObjectId are a Unix timestamp. new ObjectId().getTimestamp() gives you the creation time. This means: you often don't need a separate created_at field (though having one is still useful for clarity), and you can sort by _id to get chronological order — the default index is already there.

Redis KEYS command is O(N) and will block production

KEYS pattern* scans all keys in Redis. For a Redis instance with 10 million keys, this blocks the server for seconds. Never use KEYS in production. Use SCAN instead — it iterates in batches without blocking. SCAN 0 MATCH pattern* COUNT 100 returns up to 100 matching keys per call without blocking other clients.

Cassandra LWTs (lightweight transactions) are slow

Cassandra Lightweight Transactions (INSERT IF NOT EXISTS, UPDATE IF condition) use the Paxos consensus protocol. They're approximately 4× slower than regular writes and reduce throughput significantly. Use them only for true uniqueness constraints (user email, username). If you're using LWTs frequently, reconsider whether Cassandra is the right database for your consistency requirements.

MongoDB's aggregation pipeline replaced MapReduce

MongoDB's old MapReduce API is deprecated. The aggregation pipeline ($match → $group → $project → $sort → $limit) is the current standard. But the pipeline can be slow on large collections without proper stage ordering. Always put $match first (uses indexes), then $sort (uses indexes before grouping), then $group, then $project. Moving a $match after a $group means the group runs on all documents before filtering.

Decision sequence

Choosing a NoSQL database — work through this

Most teams should exhaust PostgreSQL before reaching for NoSQL. Start here.

1
Can PostgreSQL handle this? (It usually can)
PostgreSQL supports JSONB for nested data, arrays, full-text search, and time-series via partitioning. It has ACID transactions, JOINs, and decades of operational experience behind it. The question isn't whether PostgreSQL is theoretically capable — it's whether the specific access pattern is better served by a specialised database. If you're not certain, start with PostgreSQL.
default answer: PostgreSQL first
2
Is the bottleneck sub-millisecond access or cache-layer speed?
Database query returning in 5ms but you need 0.1ms? Redis. Session storage, rate limiting, leaderboards, caches, distributed locks — all Redis. Use it as the fast layer in front of your persistent database, not as a replacement for it. Size your Redis memory budget carefully: Redis is in-memory, cost grows with data size.
speed layer → Redis
3
Is the data highly variable in shape with co-read nested structures?
Products with wildly different attributes, CMS content types, user-generated structures. If each "item" has a different set of fields and you always read the whole item together, MongoDB fits. Key test: do you need to JOIN this data with other collections frequently? If yes, the document model will fight you — stay with PostgreSQL or denormalise heavily.
variable nested data → MongoDB
4
Is this time-series, IoT, or high-volume append-only at massive scale?
Millions of events per second, multi-datacenter active-active writes, predictable latency at any scale. If you're writing 100K+ events/second with no relational requirements and you need multi-region HA, Cassandra. If you're writing 10K events/second to a single region, PostgreSQL with partitioning or TimescaleDB handles it. Cassandra's operational complexity is only worth it at genuine scale.
massive time-series scale → Cassandra
5
Define your access patterns BEFORE choosing the schema
Write down every query the application needs to run. Frequency, latency requirement, data shape. For MongoDB: embed vs reference is answered by this list. For Cassandra: every table primary key is answered by this list. If you design the schema first and derive the queries later, you will fight the data model.
schema follows queries, not the other way around
6
Validate with a prototype before committing
Spend one day building the key access patterns against your chosen database with realistic data volumes. Run EXPLAIN on MongoDB. Run nodetool tablestats on Cassandra. Read latency with cache warm vs cold. A one-day prototype will surface problems that take months to find in production. Changing database choice after launch is a multi-month project.
prototype with real data before committing the architecture

Section 12

NoSQL data modeling checklist

QRY
Queries first

Write down every query before designing the schema. For Cassandra: one table per query pattern. For MongoDB: embed based on co-read patterns.

16M
Bound arrays

Never embed unbounded arrays in MongoDB documents. Comments, events, messages — separate collection. The 16MB limit is a production incident waiting to happen.

TTL
Expire everything

Every Redis key that isn't permanent data should have a TTL. Memory fills, OOM kills. Set EXPIRE on every cache key. Set maxmemory-policy: allkeys-lru as a safety net.

AOF
Redis persistence

Enable AOF persistence for any Redis data you care about. Default Redis loses all data on restart. appendonly yes in redis.conf.

HOT
Partition size

Monitor Cassandra partition sizes monthly. >10MB per partition → redesign with time buckets. ALLOW FILTERING in any query is a red flag — create a dedicated table.

TMB
Use TTL not DELETE

In Cassandra, prefer TTL over explicit deletes to avoid tombstone accumulation. Set default_time_to_live on tables with time-bounded data.
