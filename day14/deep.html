Deep Dive 01

MongoDB aggregation pipeline:
stage ordering and performance

The aggregation pipeline processes documents through a sequence of stages. Each stage transforms the document stream. Stage ordering dramatically affects performance — a misplaced $match can cause the entire collection to be loaded before any filtering occurs.

The golden rule: $match and $sort early

MongoDB can only use indexes in the first $match and $sort stages of a pipeline. Once any other stage runs, indexes are no longer available. Always put $match first, then $sort (if sorting on an indexed field), before any $group, $lookup, or $project stages.

MongoDB aggregation — e-commerce order analyticsJAVASCRIPT
// ❌ Bad: $match AFTER $group (full collection scan before filtering)
db.orders.aggregate([
  { $group: { _id: "$user_id", total: { $sum: "$amount" } } },
  { $match: { total: { $gt: 50000 } } }  // ← too late! already grouped everything
])
// Groups ALL orders first, then filters. Processes entire collection.

// ✅ Good: $match first (uses index), then $group
db.orders.aggregate([
  { $match: {
    created_at: { $gte: ISODate("2024-01-01") },  // ← uses index
    status: "completed"                              // ← filters early
  }},
  { $group: {
    _id: "$user_id",
    total_spent: { $sum: "$amount" },
    order_count: { $sum: 1 },
    avg_order:   { $avg: "$amount" },
    last_order:  { $max: "$created_at" }
  }},
  { $match: { total_spent: { $gt: 50000 } } },  // second $match is fine — on grouped output
  { $sort: { total_spent: -1 } },
  { $limit: 100 },
  { $lookup: {
    from: "users",
    localField: "_id",
    foreignField: "_id",
    as: "user"
  }},
  { $unwind: "$user" },
  { $project: {
    name:        "$user.name",
    email:       "$user.email",
    total_spent: 1,
    order_count: 1,
    avg_order:   { $round: ["$avg_order", 2] },
    last_order:  1,
    tier: { $switch: { branches: [
      { case: { $gte: ["$total_spent", 500000] }, then: "platinum" },
      { case: { $gte: ["$total_spent", 100000] }, then: "gold" }
    ], default: "silver" }}
  }}
])

// Explain the pipeline (check if index is used)
db.orders.aggregate(pipeline, { explain: true })
// Look for: "IXSCAN" (good) vs "COLLSCAN" (bad) in first $match
// Look for: "nReturned" much smaller than "totalKeysExamined" → bad selectivity

// $facet: run multiple aggregations in one pass (for search results with facets)
db.products.aggregate([
  { $match: { category: "electronics", in_stock: true } },
  { $facet: {
    results: [
      { $sort: { relevance: -1 } },
      { $limit: 20 },
      { $project: { name: 1, price: 1, image: 1 } }
    ],
    price_ranges: [
      { $bucket: {
        groupBy: "$price",
        boundaries: [0, 10000, 25000, 50000, 100000],
        default: "100000+",
        output: { count: { $sum: 1 } }
      }}
    ],
    brand_counts: [
      { $group: { _id: "$brand", count: { $sum: 1 } } },
      { $sort: { count: -1 } },
      { $limit: 10 }
    ],
    total: [{ $count: "count" }]
  }}
])
// Returns results + facets in ONE query — much more efficient than separate queries
Verify MongoDB aggregation performance
✓First $match uses an index: db.collection.aggregate(pipeline, {explain:true}) should show IXSCAN not COLLSCAN.
✓$lookup is after $limit: never $lookup on the full collection — filter/limit first, then enrich with $lookup.
✓$project removes unnecessary fields early: large documents flowing through the pipeline waste memory — project to only needed fields before expensive stages.
Deep Dive 02

Redis Streams:
reliable message delivery with consumer groups

Redis Streams add what Pub/Sub lacks: persistence, consumer groups, and acknowledgment. Unlike LPUSH/BRPOP queues where a message is consumed and gone, Streams keep messages and allow multiple consumer groups to independently read the same stream — each at their own position.

Redis Streams — producer, consumer group, dead letter handlingTYPESCRIPT
// ── Producer: add events to stream ────────────────────────────
async function publishEvent(event: { type: string; payload: Record<string, string> }) {
  const id = await redis.xadd(
    'events:orders',        // stream key
    'MAXLEN', '~', 1000000, // keep ~1M messages (trims lazily)
    '*',                    // auto-generate ID (timestamp-sequence)
    'type', event.type,
    'payload', JSON.stringify(event.payload),
    'published_at', Date.now().toString()
  );
  return id; // e.g. "1700000000000-0"
}

// ── Create consumer groups (once on startup) ──────────────────
async function initGroups() {
  for (const group of ['email-service', 'analytics-service', 'inventory-service']) {
    try {
      await redis.xgroup('CREATE', 'events:orders', group, '$', 'MKSTREAM');
      // '$' = only new messages from now on
      // '0' = all messages from the beginning
    } catch (e) {
      if (!e.message.includes('BUSYGROUP')) throw e; // ignore "already exists"
    }
  }
}

// ── Consumer: read + process + acknowledge ────────────────────
async function runConsumer(group: string, consumerId: string) {
  while (true) {
    // XREADGROUP: read up to 10 new messages, block for 5s if none
    const messages = await redis.xreadgroup(
      'GROUP', group, consumerId,
      'COUNT', 10,
      'BLOCK', 5000,
      'STREAMS', 'events:orders', '>'
      // '>' = new messages not yet delivered to this group
    );

    if (!messages) continue; // timeout, no messages

    for (const [id, fields] of messages[0][1]) {
      try {
        await processMessage(fields);
        // ACK: removes from Pending Entry List (PEL)
        await redis.xack('events:orders', group, id);
      } catch (err) {
        // Message stays in PEL — will be redelivered
        logger.error({ err, id }, 'Failed to process message');
        // After N retries → move to dead letter stream
        await handleFailedMessage(group, id, err);
      }
    }
  }
}

// ── Reclaim abandoned messages (consumer crashed without ACK) ──
async function reclaimStuckMessages(group: string, consumerId: string) {
  const STUCK_AFTER_MS = 60000; // 1 minute
  const MAX_RETRIES = 3;

  // XAUTOCLAIM: find messages idle > 60s, reassign to this consumer
  const [, claimed] = await redis.xautoclaim(
    'events:orders', group, consumerId,
    STUCK_AFTER_MS, '0-0', 'COUNT', 10
  );

  for (const [id, fields] of claimed) {
    const info = await redis.xpending('events:orders', group, 'IDLE', 0, id, id, 1);
    const deliveryCount = info[0][3];

    if (deliveryCount >= MAX_RETRIES) {
      // Dead letter: move to dead-letter stream + ACK original
      await redis.xadd('events:orders:dead', '*', ...fields, 'original_id', id, 'group', group);
      await redis.xack('events:orders', group, id);
      logger.error({ id, group }, 'Message moved to dead letter queue');
    } else {
      await processMessage(fields).then(() => redis.xack('events:orders', group, id));
    }
  }
}
Deep Dive 03

Cassandra denormalisation:
one domain, multiple tables

In Cassandra, normalisation is the enemy. You maintain multiple tables storing the same data, each optimised for a different query. The application is responsible for keeping them in sync. This is the fundamental trade Cassandra makes: query performance in exchange for write complexity and data duplication.

Domain: e-commerce orders — three query patterns, three tables

Cassandra denormalisation — one domain, three tablesCQL + TYPESCRIPT
-- Domain: Orders
-- Query 1: "Get order by ID" (for order detail page)
-- Query 2: "Get all orders for a user, newest first" (for order history)
-- Query 3: "Get all orders with status=pending, sorted by created_at" (for ops dashboard)

-- Table 1: by order ID (lookup by order)
CREATE TABLE orders_by_id (
  order_id     UUID PRIMARY KEY,
  user_id      UUID,
  status       TEXT,
  total        DECIMAL,
  items        LIST<FROZEN<order_item>>,
  shipping     FROZEN<address>,
  created_at   TIMESTAMP,
  updated_at   TIMESTAMP
);

-- Table 2: by user (user's order history)
CREATE TABLE orders_by_user (
  user_id      UUID,
  created_at   TIMESTAMP,
  order_id     UUID,
  status       TEXT,
  total        DECIMAL,
  item_summary TEXT,           -- denormalised: "MacBook Pro, USB Hub"
  PRIMARY KEY (user_id, created_at, order_id)
) WITH CLUSTERING ORDER BY (created_at DESC, order_id ASC);
-- Query: WHERE user_id = ? ORDER BY created_at DESC LIMIT 20

-- Table 3: by status (ops dashboard)
CREATE TABLE orders_by_status (
  status       TEXT,
  bucket       TEXT,           -- 'pending:2024-06' (daily bucket)
  created_at   TIMESTAMP,
  order_id     UUID,
  user_id      UUID,
  total        DECIMAL,
  PRIMARY KEY ((status, bucket), created_at, order_id)
) WITH CLUSTERING ORDER BY (created_at DESC, order_id ASC);
-- Query: WHERE status = 'pending' AND bucket = 'pending:2024-06'

-- ── Dual-write: application maintains all tables ──────────────
async function createOrder(order: Order): Promise<void> {
  const bucket = `\${order.status}:\${order.created_at.toISOString().slice(0,7)}`;
  const batch = [
    // Write to all three tables atomically (batch within single partition is lightweight)
    await cassandra.execute(`
      INSERT INTO orders_by_id (order_id, user_id, status, total, items, created_at)
      VALUES (?, ?, ?, ?, ?, ?)`,
      [order.id, order.userId, order.status, order.total, order.items, order.createdAt]
    ),
    await cassandra.execute(`
      INSERT INTO orders_by_user (user_id, created_at, order_id, status, total, item_summary)
      VALUES (?, ?, ?, ?, ?, ?)`,
      [order.userId, order.createdAt, order.id, order.status, order.total, summarizeItems(order.items)]
    ),
    await cassandra.execute(`
      INSERT INTO orders_by_status (status, bucket, created_at, order_id, user_id, total)
      VALUES (?, ?, ?, ?, ?, ?)`,
      [order.status, bucket, order.createdAt, order.id, order.userId, order.total]
    ),
  ];
  await Promise.all(batch);
  // Note: these are 3 separate writes — not a single transaction.
  // If one fails: inconsistency. Handle with idempotent retries.
}

-- ── Status change: update all tables that store status ────────
async function updateOrderStatus(orderId: UUID, newStatus: string) {
  // 1. Fetch current order from orders_by_id
  const order = await getOrderById(orderId);
  const oldBucket = `\${order.status}:\${order.createdAt.toISOString().slice(0,7)}`;
  const newBucket = `\${newStatus}:\${order.createdAt.toISOString().slice(0,7)}`;

  await Promise.all([
    // Update orders_by_id
    cassandra.execute('UPDATE orders_by_id SET status=? WHERE order_id=?', [newStatus, orderId]),
    // Update orders_by_user
    cassandra.execute('UPDATE orders_by_user SET status=? WHERE user_id=? AND created_at=? AND order_id=?',
      [newStatus, order.userId, order.createdAt, orderId]),
    // Remove from old status bucket, add to new (Cassandra: delete + insert)
    cassandra.execute('DELETE FROM orders_by_status WHERE status=? AND bucket=? AND created_at=? AND order_id=?',
      [order.status, oldBucket, order.createdAt, orderId]),
    cassandra.execute('INSERT INTO orders_by_status (status, bucket, ...) VALUES (?,...)',
      [newStatus, newBucket, order.createdAt, orderId, order.userId, order.total]),
  ]);
}
// This is why Cassandra modeling is complex: status change touches 3 tables.
// Design to minimise these multi-table writes in your access patterns.
Verify Cassandra data model health
✓No ALLOW FILTERING in any production query: grep -r "ALLOW FILTERING" ./src — any match is a red flag.
✓Partition sizes in bounds: nodetool tablehistograms keyspace.table — check the partition size histogram. P99 should be under 10MB.
✓TTL is set for time-bounded tables: SELECT default_time_to_live FROM system_schema.tables WHERE keyspace_name=?
✓Tombstone counts are low: nodetool cfstats keyspace.table | grep Tombstone — high tombstone count means too many deletes; switch to TTL.
