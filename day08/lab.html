Production Incident  ·  PostgreSQL  ·  Statistics & Query Planner

A query that ran in 12ms for 8 months suddenly took 47 seconds

Tuesday 09:14 IST  ·  Duration: 6h  ·  Severity: P1  ·  Orders list page timing out for all users

The orders listing page had been fast since launch. A query joining orders and users, filtered by status and date range, returning paginated results. P99 was consistently 12ms. No code changes were deployed on Monday. On Tuesday morning, the page started timing out. Engineers checked the server — healthy. Checked the database — healthy. The query was the same. The index was there. The index existed and the query was timing out anyway.

The team spent three hours verifying the index existed, rewriting the query, and checking network latency. The actual problem took 30 seconds to fix once it was found. The cause: a data migration the previous weekend had loaded 8 million new orders into the table. The table grew from 2 million to 10 million rows. PostgreSQL's statistics — its internal estimates of data distribution — were still based on the 2 million row snapshot. The planner thought the query would return 200 rows. It actually returned 45,000. It chose an index scan when a sequential scan would have been much faster, then switched to a sequential scan internally after seeing the actual volume — but in the worst possible way.

Root cause

The data migration loaded 8M rows without running ANALYZE afterward. PostgreSQL's statistics still reflected the pre-migration data distribution (2M rows, ~2% status='pending'). After migration, ~18% of rows were status='pending' (the new orders). The planner estimated 200 matching rows and chose an index scan + nested loop join. The actual result was 45,000 rows — the planner had made the wrong strategy choice based on wrong statistics. The index existed. The query was correct. The statistics were wrong.

The checklist that would have prevented 6 hours of downtimeSQL
❌ The migration script (no ANALYZE)
INSERT INTO orders SELECT * FROM orders_staging;  -- 8M rows loaded
-- No ANALYZE. Statistics still reflect 2M rows.
-- Planner now has wrong estimates for every query on this table.

✅ Fixed migration script
INSERT INTO orders SELECT * FROM orders_staging;
ANALYZE orders;  -- always run after bulk operations
-- Now planner has accurate row counts and value distributions

-- Better: set autovacuum to analyze more aggressively on high-write tables
ALTER TABLE orders SET (
  autovacuum_analyze_scale_factor = 0.01,  -- analyze after 1% rows change (default: 20%)
  autovacuum_analyze_threshold    = 1000,  -- minimum rows before analyzing
);

-- Alert if stats are stale (add to monitoring)
SELECT relname, last_analyze, last_autoanalyze,
       EXTRACT(EPOCH FROM now() - GREATEST(last_analyze, last_autoanalyze))/3600 AS hours_since_analyze
FROM pg_stat_user_tables
WHERE GREATEST(last_analyze, last_autoanalyze) < now() - interval '2 hours'
ORDER BY hours_since_analyze DESC;
-- Alert if any high-write table hasn't been analyzed in > 2 hours
What this incident teaches
▶Indexes don't help if the planner makes the wrong strategy choice. The index existed. The query was correct. The planner chose wrong because its statistics said the query would return 200 rows when it actually returned 45,000.
▶Always run ANALYZE after bulk data operations. Any INSERT/UPDATE/DELETE that changes more than 20% of a table (the default threshold) should be followed by an explicit ANALYZE. Don't rely on autovacuum timing after migrations.
▶EXPLAIN ANALYZE's "rows=" estimate vs "(actual rows=)" divergence is the fastest diagnostic. A 200× difference in estimate vs actual tells you immediately that statistics are wrong. This takes 30 seconds to check and prevents hours of wrong-direction debugging.
▶Monitor statistics freshness as an operational metric. A query that was fast for 8 months and is now slow is almost always statistics — not code, not the index, not the server. Build an alert for tables where autovacuum hasn't run recently.
