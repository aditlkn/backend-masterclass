Deep Dive 01

Covering index implementation:
from 0 heap fetches to verified

A covering index is worth nothing until you've verified it with EXPLAIN ANALYZE and confirmed "Heap Fetches: 0". Here's the complete workflow — from designing the index to confirming it's actually working in production.

Step 1: Identify the query and what it needs

Start with a specific query. Not "queries on the orders table" — one specific query with its exact SELECT columns, WHERE conditions, and ORDER BY. The covering index is designed for this query, not generically.

Designing a covering index — step by step with verificationSQL
-- TARGET QUERY: mobile orders list, called ~500 times/second
SELECT id, status, total_amount_paise, created_at, payment_method
FROM orders
WHERE user_id = $1
ORDER BY created_at DESC
LIMIT 20;

-- Step 1: Run EXPLAIN before any index changes
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT id, status, total_amount_paise, created_at, payment_method
FROM orders WHERE user_id = 'usr_123'
ORDER BY created_at DESC LIMIT 20;

-- Likely output (no index):
-- Seq Scan on orders (cost=0..485000 rows=20 width=60)
--   Filter: (user_id = 'usr_123')
--   Rows Removed by Filter: 9999980
-- Planning Time: 0.2ms / Execution Time: 4200ms ← full table scan

-- Step 2: Create the index. CONCURRENTLY = no table lock.
-- Key columns: user_id (equality), created_at (order + filter)
-- INCLUDE columns: everything else the query needs
CREATE INDEX CONCURRENTLY idx_orders_user_covering
ON orders (user_id, created_at DESC)
INCLUDE (id, status, total_amount_paise, payment_method);
-- This takes: seconds to minutes depending on table size
-- Monitor progress:
-- SELECT phase, blocks_done, blocks_total FROM pg_stat_progress_create_index;

-- Step 3: Verify the plan changed to Index Only Scan
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT id, status, total_amount_paise, created_at, payment_method
FROM orders WHERE user_id = 'usr_123'
ORDER BY created_at DESC LIMIT 20;

-- Expected output:
-- Index Only Scan Backward using idx_orders_user_covering on orders
--   (cost=0.4..1.2 rows=20 width=60)
--   (actual time=0.08..0.12 rows=20 loops=1)
--   Index Cond: (user_id = 'usr_123')
--   Heap Fetches: 0       ← THE NUMBER WE WANT
--   Buffers: shared hit=4
-- Execution Time: 0.18ms  ← from 4200ms to 0.18ms

-- Step 4: If Heap Fetches is NOT 0, run VACUUM first
-- Heap Fetches > 0 means the visibility map isn't populated
-- VACUUM populates the visibility map so index-only scans can confirm
-- row visibility without touching the heap:
VACUUM orders;  -- or wait for autovacuum to run
-- Then re-run EXPLAIN — Heap Fetches should drop to 0

-- Step 5: Size check — covering index trades size for speed
SELECT
  indexrelname,
  pg_size_pretty(pg_relation_size(indexrelid)) AS index_size
FROM pg_stat_user_indexes
WHERE relname = 'orders'
ORDER BY pg_relation_size(indexrelid) DESC;
-- Covering indexes are larger than key-only indexes
-- The INCLUDE columns add data to each leaf page
-- Acceptable trade: larger index, faster queries, no heap I/O

fill_factor: leaving room for updates

By default, PostgreSQL fills index pages to 100% during bulk loads. When rows are updated and new index entries are needed, there's no room — PostgreSQL must allocate a new page. This causes index fragmentation and bloat over time for high-update tables.

Setting fillfactor=70 during index creation leaves 30% of each page empty, reserving it for future insertions. The result: fewer page splits, less fragmentation, better long-term performance on high-write tables. The downside: the initial index is 30% larger. Worth it for indexes on high-update columns.

fill_factor and other index storage parametersSQL
-- Default fillfactor=90 (leaves 10% for updates)
-- For high-update tables, reduce to 70-80%
CREATE INDEX CONCURRENTLY idx_orders_status ON orders (status, created_at DESC)
WITH (fillfactor = 70);

-- Check current fillfactor of an existing index
SELECT relname, reloptions
FROM pg_class
WHERE relname = 'idx_orders_status';

-- Change fillfactor of existing index (requires REINDEX)
ALTER INDEX idx_orders_status SET (fillfactor = 70);
REINDEX INDEX CONCURRENTLY idx_orders_status;

-- For append-only tables (logs, events, immutable data):
-- fillfactor=100 is fine — no updates means no fragmentation
CREATE INDEX ON events (created_at DESC)
WITH (fillfactor = 100);
Verify your covering index is working
✓EXPLAIN shows "Index Only Scan" not "Index Scan" — the heap is not being touched.
✓"Heap Fetches: 0" in the EXPLAIN output — if non-zero, run VACUUM tablename and recheck.
✓All SELECT columns are either in the index key or the INCLUDE clause — missing one forces heap access.
✓No invalid index exists after CONCURRENTLY build: SELECT indexrelname FROM pg_stat_user_indexes JOIN pg_index USING (indexrelid) WHERE NOT indisvalid
Deep Dive 02

VACUUM internals:
what actually happens when it runs

VACUUM is one of the most important background processes in PostgreSQL, but most engineers think of it as "the thing that cleans up dead rows." Understanding what it actually does — which data structures it touches, in which order, and what each step enables — explains why it matters beyond just freeing disk space.

What VACUUM does, in order

  1. Scan heap pages for dead tuples. VACUUM reads through heap pages and identifies tuples whose xmax (the transaction ID that deleted them) is old enough to be visible to all current transactions. These are safe to reclaim.
  2. Remove dead index entries. For each dead heap tuple, VACUUM finds and removes the corresponding entries in every index on the table. This is why VACUUM must touch all indexes — dead index entries pointing to dead heap tuples cause incorrect results if left in place.
  3. Mark dead heap space as reusable. Dead tuples aren't immediately overwritten — their space is marked as free in the Free Space Map (FSM). Future INSERTs can reuse this space without extending the table file.
  4. Update the Visibility Map. For heap pages where all tuples are visible to all transactions, VACUUM sets a bit in the Visibility Map. This is what enables Index Only Scans — when the visibility map bit is set for a page, a query using an index-only scan doesn't need to check the heap to confirm row visibility.
  5. Update statistics (only ANALYZE does this). VACUUM alone does not update the planner statistics. Run VACUUM ANALYZE or ANALYZE separately to update both.
  6. Freeze old transaction IDs (VACUUM FREEZE). PostgreSQL uses 32-bit transaction IDs, which wrap around after ~2 billion transactions. VACUUM FREEZE marks old tuples as "frozen" — permanently visible to all future transactions — preventing transaction ID wraparound. If this doesn't happen, PostgreSQL will refuse writes to protect data integrity (the "transaction ID wraparound" catastrophe).

VACUUM vs VACUUM FULL

VACUUM (regular): marks dead space as reusable but doesn't return it to the OS. The table file stays the same size on disk. Fast, non-blocking (allows concurrent reads and writes).

VACUUM FULL: rewrites the entire table to a new file, reclaiming all dead space and returning it to the OS. The table file shrinks. Slow, requires an exclusive lock on the table — blocks all reads and writes for the duration. Only use VACUUM FULL when you've deleted a large fraction of a table and need to reclaim disk space. For normal maintenance, regular VACUUM is correct.

VACUUM commands and monitoring in productionSQL
-- Standard VACUUM (non-blocking)
VACUUM orders;

-- VACUUM + update statistics in one pass
VACUUM ANALYZE orders;

-- VACUUM FULL: blocks all reads and writes. Use rarely.
-- VACUUM FULL orders;  ← don't run on live production without a maintenance window

-- Monitor an in-progress VACUUM
SELECT
  schemaname,
  relname,
  phase,
  heap_blks_scanned,
  heap_blks_total,
  ROUND(heap_blks_scanned::numeric / NULLIF(heap_blks_total,0) * 100, 1) AS pct_done,
  dead_tuple_count,
  index_vacuum_count
FROM pg_stat_progress_vacuum;

-- Check visibility map coverage (higher = better index-only scan support)
SELECT
  relname,
  n_live_tup,
  n_dead_tup,
  ROUND(n_dead_tup::numeric / NULLIF(n_live_tup,0) * 100, 1) AS dead_pct,
  last_vacuum,
  last_autovacuum,
  last_analyze,
  last_autoanalyze
FROM pg_stat_user_tables
WHERE relname = 'orders';

-- Check transaction ID age (alert if approaching 1 billion)
SELECT
  datname,
  age(datfrozenxid) AS xid_age,
  current_setting('autovacuum_freeze_max_age')::int AS freeze_max
FROM pg_database
ORDER BY age(datfrozenxid) DESC;
-- Alert if xid_age > 1.5 billion — approaching wraparound danger zone
Deep Dive 03

Autovacuum tuning:
the complete parameter table

Autovacuum runs VACUUM and ANALYZE automatically in the background. Its default settings were designed for moderate workloads. High-write production tables routinely need per-table tuning — the defaults let bloat accumulate faster than autovacuum can clean it.

How autovacuum decides when to run

Autovacuum triggers a VACUUM on a table when dead tuples exceed: autovacuum_vacuum_threshold + (autovacuum_vacuum_scale_factor × n_live_tup)

With defaults: 50 + (0.20 × n_live_tup). For a 10M row table, autovacuum waits until 2,000,050 dead tuples accumulate before running. That's 20% of your live rows sitting as dead bloat before any cleanup. For a table with 100 UPDATE/s, that's 5.5 hours of accumulation.

ANALYZE triggers when: autovacuum_analyze_threshold + (autovacuum_analyze_scale_factor × n_live_tup). Default: 50 + (0.10 × n_live_tup). For 10M rows: wait until 1M row changes. That's the statistics gap that caused our Day 8 incident.

Per-table autovacuum tuning — production configurationSQL
-- High-write table (orders, events, logs): aggressive autovacuum
ALTER TABLE orders SET (
  autovacuum_vacuum_scale_factor    = 0.01,   -- vacuum when 1% dead (10K rows on 1M table)
  autovacuum_vacuum_threshold       = 100,
  autovacuum_analyze_scale_factor   = 0.005,  -- analyze when 0.5% changed
  autovacuum_analyze_threshold      = 100,
  autovacuum_vacuum_cost_delay      = 2,      -- default (don't throttle aggressively)
  autovacuum_vacuum_cost_limit      = 400     -- 2× default = faster vacuuming
);

-- Read-heavy, rarely updated table (product catalog, users): relaxed
ALTER TABLE products SET (
  autovacuum_vacuum_scale_factor  = 0.10,   -- default is fine
  autovacuum_analyze_scale_factor = 0.05
);

-- Append-only table (audit logs, immutable events): almost never needs vacuum
ALTER TABLE audit_logs SET (
  autovacuum_vacuum_scale_factor = 0.5,    -- only vacuum when 50% dead (near-never)
  autovacuum_analyze_scale_factor = 0.05  -- but analyze regularly for new data distribution
);

-- View current per-table settings
SELECT relname, reloptions
FROM pg_class
WHERE reloptions IS NOT NULL
  AND relkind = 'r'
ORDER BY relname;

-- Monitor whether autovacuum is keeping up
SELECT
  relname,
  n_dead_tup,
  n_live_tup,
  ROUND(n_dead_tup::numeric / NULLIF(n_live_tup,0) * 100, 2) AS dead_pct,
  last_autovacuum,
  last_autoanalyze,
  autovacuum_count
FROM pg_stat_user_tables
WHERE n_dead_tup > 1000
ORDER BY dead_pct DESC;
-- If dead_pct keeps climbing despite autovacuum running:
-- 1. Long-running transactions blocking vacuum (check pg_stat_activity)
-- 2. autovacuum_cost_limit too low (vacuum too throttled)
-- 3. autovacuum_max_workers exhausted (too many tables needing vacuum simultaneously)
⚠️
The long transaction bloat trap

A transaction that stays open for a long time (common in batch jobs, manual psql sessions left open, or ORM connection pools with BEGIN but no COMMIT) prevents VACUUM from cleaning up any rows modified after that transaction started. Even with perfect autovacuum settings, bloat will accumulate unboundedly if there are long-running transactions. Monitor SELECT pid, now() - xact_start AS duration, query FROM pg_stat_activity WHERE xact_start IS NOT NULL ORDER BY duration DESC — kill anything open for more than your acceptable bloat window.

Verify autovacuum is working correctly
✓Dead tuple percentage stays below 5% on high-write tables: SELECT relname, ROUND(n_dead_tup::numeric/NULLIF(n_live_tup,0)*100,1) FROM pg_stat_user_tables ORDER BY 2 DESC LIMIT 10
✓Autovacuum ran recently on all high-write tables: last_autovacuum should be within last 30 minutes for tables with 100+ updates/second.
✓No long-running transactions blocking vacuum: SELECT pid, now()-xact_start AS age FROM pg_stat_activity WHERE xact_start < now()-interval '10 minutes'
✓Transaction ID age is safe: SELECT datname, age(datfrozenxid) FROM pg_database ORDER BY 2 DESC — alert if age exceeds 1.5 billion.
✓Statistics freshness: last_autoanalyze within last 2 hours on tables that receive bulk imports or have rapidly changing data distributions.
