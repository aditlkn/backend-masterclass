Section 01

B-tree internals: why the data structure matters for query performance

PostgreSQL's default index type is a B-tree (Balanced tree). Understanding its structure explains why some queries are fast, why some predicates can't use an index efficiently, and why column order in a composite index isn't arbitrary.

A B-tree index is a tree where each node holds multiple keys, sorted in order. The tree is "balanced" — every leaf node is at the same depth. This guarantees that any search takes O(log N) comparisons regardless of which value you're looking for. For a table with 100 million rows and a B-tree index, finding a specific value requires at most log₂(100,000,000) ≈ 27 comparisons — compared to 100 million for a sequential scan.

The three node types

Root node: The entry point. Holds the highest-level partition of key ranges. Every search starts here. In PostgreSQL, a B-tree index always has exactly one root.

Internal nodes (branch pages): Navigate the tree. Each internal node holds N keys and N+1 child pointers. The keys are "separator values" — values k in the node mean "left child has values < k, right child has values ≥ k." They're not data; they're routing information.

Leaf nodes (leaf pages): Where the actual index data lives. Each leaf entry contains the indexed value and a pointer (called a TID — Tuple ID) to the actual row in the heap. Leaf nodes are linked in a doubly-linked list, which is why range scans are efficient: once the B-tree finds the start of a range, it follows the leaf chain forward without returning to the root.

What operations the B-tree makes fast

  • Equality: WHERE email = 'a@b.com' — traverse from root to leaf, O(log N)
  • Range scans: WHERE amount BETWEEN 100 AND 500 — find the start, follow leaf chain
  • Ordering: ORDER BY created_at — leaf pages are already in sorted order
  • Prefix matching: WHERE name LIKE 'Adnan%' — works on the first column of a composite index

What the B-tree cannot do

  • Suffix matching: WHERE name LIKE '%kumar' — the tree is sorted by prefix, not suffix. Full scan required. Use pg_trgm GIN index for this.
  • Function on indexed column: WHERE LOWER(email) = 'a@b.com' — the index is on email values, not LOWER(email) values. Create a functional index: CREATE INDEX ON users (LOWER(email)).
  • Non-equality on non-leading column: WHERE status = 'active' AND created_at > now() - interval '7 days' — if the index is (status, created_at) this works. If the index is (created_at, status) and your predicate is on status with a range on created_at, the planner may not use it efficiently.

Section 02

Index seek vs sequential scan: when the planner switches strategies

Here's something counterintuitive: the query planner sometimes ignores your index even when one exists. This isn't a bug. It's correct behaviour — and understanding when it happens prevents a lot of confused debugging.

The selectivity threshold

The query planner estimates how many rows will match your predicate — called selectivity. If a predicate matches 1% of rows, an index seek is fast: find the leaf entries, follow 1% of TID pointers to the heap, done. If a predicate matches 80% of rows, an index seek is actually slower than a sequential scan: you visit the B-tree leaf pages, then for each TID you jump to a random page in the heap. 80% of a table's rows means 80% of heap pages, accessed in random order — potentially more I/O than just reading the heap sequentially.

The threshold varies, but roughly: below 5-10% row selectivity, an index seek is faster. Above 10-20%, a sequential scan is often faster. The exact threshold depends on table size, page layout, memory availability (if the table fits in shared_buffers, random access is cheap), and hardware (SSDs vs spinning disks).

Statistics and stale estimates

The planner uses statistics to estimate selectivity. PostgreSQL collects these via ANALYZE — a scan of a random sample of the table to build histograms of value distributions. If statistics are stale (table has grown significantly since the last ANALYZE), the planner's estimates are wrong, and it may choose the wrong strategy.

Common symptom: a query was fast for months, then suddenly became slow after a bulk insert. The data distribution changed but statistics haven't been updated. Fix: ANALYZE <table>. Permanent fix: ensure autovacuum is running and its analyze threshold is appropriate for your table's update rate.

Understanding the planner's decision — check with EXPLAINSQL
-- See what the planner would do (no execution)
EXPLAIN
  SELECT * FROM orders WHERE status = 'pending';

-- Actually execute and show real timing + row counts
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
  SELECT * FROM orders WHERE status = 'pending';

-- Compare planner estimate vs actual rows
-- If "rows=100" estimated but "rows=50000" actual → stale statistics
-- Fix: ANALYZE orders;

-- Force the planner to use (or avoid) an index for testing

SET enable_indexscan = off; -- force seq scan
-- Always reset after testing:
RESET enable_seqscan;
RESET enable_indexscan;

-- Check current statistics for a column

FROM pg_stats
WHERE tablename = 'orders' AND attname = 'status';
-- correlation: 1.0 = heap rows in same order as index (ideal for range scans)
-- correlation: 0.0 = random order (index seek pays random I/O per row)

-- When was the table last analyzed?

FROM pg_stat_user_tables WHERE relname = 'orders';

Section 03

Composite indexes: column order is everything

A composite index on (user_id, status, created_at) is not the same as an index on (status, user_id, created_at) or (created_at, user_id, status). The column order determines which queries can use the index efficiently — and getting it wrong means adding an index that helps some queries while silently not helping others.

The left-prefix rule

A composite index can only be used efficiently if your query predicates start from the leftmost column and include consecutive columns. An index on (a, b, c) can be efficiently used by:

  • WHERE a = ? — uses first column only
  • WHERE a = ? AND b = ? — uses first two columns
  • WHERE a = ? AND b = ? AND c = ? — uses all three
  • WHERE a = ? AND b BETWEEN ? AND ? — uses first column for equality, second for range

What cannot efficiently use the index:

  • WHERE b = ? — skips the first column, can't use the index
  • WHERE a = ? AND c = ? — skips the middle column; index used only on a
  • WHERE b = ? AND c = ? — doesn't start from the left

The range column must be last

Equality predicates can be in any order (the planner reorders them). But a range predicate (BETWEEN, >, <, LIKE 'foo%') stops the index from being useful for any columns to its right. If your index is (status, created_at, user_id) and your query is WHERE status = 'active' AND created_at > '2024-01-01' AND user_id = 'usr_123' — the index is used for status and created_at but not for user_id, because the range on created_at stops further filtering in the index.

The rule: put equality columns first, range column last.

Choosing column order: cardinality vs query patterns

Two competing principles:

  • Put high-cardinality columns first (more unique values = better filtering)
  • Put columns used in equality predicates before range predicates

The second principle wins. Query pattern determines order. If you always query WHERE user_id = ? AND created_at > ?, the index should be (user_id, created_at) regardless of the relative cardinality of these columns.

Section 04

Covering indexes: eliminating the heap fetch

A standard index lookup has two phases: (1) traverse the B-tree to find matching TIDs, (2) fetch the actual rows from the heap (the table data) using those TIDs. Step 2 — the "heap fetch" — is often the expensive part, especially when rows are scattered across many heap pages.

A covering index (or index-only scan) eliminates the heap fetch entirely. If the index contains all the columns your query needs, PostgreSQL can return the result directly from the index without touching the heap at all. This can make a query 2-10× faster by avoiding random I/O.

How to make an index covering

Two approaches in PostgreSQL:

Include all needed columns in the index key: CREATE INDEX ON orders (user_id, status) INCLUDE (total_amount, created_at). The INCLUDE clause adds columns to the leaf pages without making them part of the B-tree sort key. This means: user_id and status are usable for index seeks; total_amount and created_at are available at the leaf for queries that need them but aren't part of the search.

Add all needed columns to the index key: CREATE INDEX ON orders (user_id, status, total_amount, created_at). More flexible (all columns can be used in predicates) but the index is larger and writes pay more.

The visibility map requirement

Index-only scans require PostgreSQL's visibility map to confirm that a row is visible to the current transaction. If a page isn't in the visibility map (because it has recently been modified or VACUUM hasn't run), PostgreSQL falls back to fetching the heap page even with a covering index. Run VACUUM (or ensure autovacuum is running) to populate the visibility map and fully realise the covering index benefit.

Creating and verifying a covering indexSQL
-- Query we want to optimise (mobile orders list)
SELECT id, status, total_amount_paise, created_at
FROM orders
WHERE user_id = 'usr_123'
ORDER BY created_at DESC
LIMIT 20;

-- Regular index: heap fetch required for total_amount_paise
CREATE INDEX idx_orders_user ON orders (user_id, created_at DESC);

-- Covering index: all 4 columns available from the index alone
CREATE INDEX idx_orders_user_covering ON orders
  (user_id, created_at DESC)
  INCLUDE (id, status, total_amount_paise);
-- Note: INCLUDE columns don't affect B-tree sort order
-- They're only available at the leaf pages, not for filtering/ordering

-- Verify: EXPLAIN shows "Index Only Scan" vs "Index Scan"
EXPLAIN (ANALYZE, BUFFERS)
SELECT id, status, total_amount_paise, created_at
FROM orders WHERE user_id = 'usr_123'
ORDER BY created_at DESC LIMIT 20;
-- Look for: "Index Only Scan" in the plan
-- Bad:       "Heap Fetches: 1000" → visibility map not up to date
-- Good:      "Heap Fetches: 0"   → true index-only scan
-- Fix high heap fetches: VACUUM orders;

-- Check index size vs table size
SELECT
  relname AS index_name,
  pg_size_pretty(pg_relation_size(indexrelid)) AS index_size,
  pg_size_pretty(pg_relation_size(indrelid))   AS table_size
FROM pg_stat_user_indexes
WHERE relname LIKE 'idx_orders%';

Section 05

Reading EXPLAIN ANALYZE: the skill that diagnoses everything

EXPLAIN ANALYZE is the most powerful diagnostic tool in PostgreSQL. Everything else in this lesson is context for reading it correctly. Most engineers look at the output, see a wall of text, find the number that seems large, and guess. The right approach is systematic.

Reading the plan tree

The output is an indented tree representing the query execution plan. Innermost nodes execute first. Outer nodes consume their output. Each node shows: node type, estimated cost, estimated rows, actual time, actual rows, loops.

The five things to look for

  1. Seq Scan on a large table: Missing index, or the planner chose not to use the existing one (selectivity too low or stale statistics).
  2. Rows estimate vs actual rows divergence: If estimated rows are 100 but actual rows are 50,000 — the planner is working with wrong statistics. Run ANALYZE.
  3. High "Heap Fetches" on an index scan: Your covering index isn't actually covering — the heap is still being accessed. Columns are missing from the index, or VACUUM hasn't run.
  4. Nested Loop with large outer side: A join where the outer table returns many rows, each triggering an inner lookup. Can be fine (if inner is indexed), can be catastrophic (if inner is a seq scan).
  5. Sort on non-indexed column: Appears as a "Sort" node in the plan. Either add an index to eliminate the sort, or accept the cost if it's small.

Section 06

Index bloat: the invisible performance problem

PostgreSQL uses MVCC (Multi-Version Concurrency Control) — instead of updating a row in place, it writes a new version of the row and marks the old version as dead. Dead tuple versions accumulate until VACUUM cleans them up. The same thing happens to indexes: dead index entries accumulate and the index grows, becoming slower to traverse.

Index bloat is insidious because it doesn't cause errors or obvious failures. It causes gradual performance degradation — the index scans that used to take 5ms now take 25ms because the B-tree has more pages to traverse and more dead entries to skip. The table and index sizes on disk grow even though the live data hasn't changed.

What causes bloat

  • High update rate: Every UPDATE creates a new row version and a new index entry. The old entry becomes dead and must be vacuumed.
  • High delete rate: Deleted rows leave dead index entries.
  • Autovacuum not keeping up: If your update/delete rate exceeds autovacuum's capacity to clean up, dead tuples accumulate faster than they're removed.
  • Long-running transactions: PostgreSQL can't vacuum dead tuples that might be visible to long-running transactions. A transaction open for 1 hour prevents VACUUM from cleaning up anything modified in that hour.

Detecting and fixing bloat

Rebuilding a bloated index is the cure: REINDEX INDEX CONCURRENTLY idx_name (PostgreSQL 12+). This rebuilds the index without locking reads or writes. On a 10GB bloated index, a rebuild might bring it down to 2GB and halve its scan time.

Detecting index bloat and rebuilding without downtimeSQL
-- Check for table bloat (dead tuples)
SELECT
  relname,
  n_live_tup,
  n_dead_tup,
  ROUND(n_dead_tup::numeric / NULLIF(n_live_tup + n_dead_tup, 0) * 100, 1) AS dead_pct,
  last_vacuum,
  last_autovacuum
FROM pg_stat_user_tables
ORDER BY dead_pct DESC NULLS LAST;
-- Alert if dead_pct > 20% — autovacuum may not be keeping up

-- Check index size vs expected size (pgstattuple extension)
CREATE EXTENSION IF NOT EXISTS pgstattuple;

SELECT
  indexrelname,
  pg_size_pretty(pg_relation_size(indexrelid)) AS index_size,
  pgstatindex(indexrelid) AS stats
FROM pg_stat_user_indexes
WHERE schemaname = 'public'
ORDER BY pg_relation_size(indexrelid) DESC;
-- stats.leaf_fragmentation > 20% → consider REINDEX
-- stats.avg_leaf_density < 50% → significant bloat

-- Rebuild index without downtime (PG 12+)
REINDEX INDEX CONCURRENTLY idx_orders_status_created;
-- Takes: minutes to hours depending on index size
-- No lock on reads or writes during rebuild
-- Old index stays active until new one is ready

-- Tune autovacuum for high-write tables
ALTER TABLE orders SET (
  autovacuum_vacuum_scale_factor    = 0.01,  -- vacuum when 1% of rows are dead (default 20%)
  autovacuum_analyze_scale_factor   = 0.005, -- analyze when 0.5% changed
  autovacuum_vacuum_cost_delay      = 2      -- faster vacuuming (default 2ms)
);

Section 07

Partial indexes: index only the rows that matter

A regular index on status indexes every row in the table. If 95% of your orders have status 'completed' and only 5% are 'pending', and you only ever query pending orders, you're maintaining an index on 100% of rows to satisfy queries on 5% of them. That's wasteful in index size and write overhead.

A partial index adds a WHERE clause — it only indexes rows that match the condition. CREATE INDEX ON orders (created_at) WHERE status = 'pending' creates an index that only contains pending orders. It's 95% smaller, faster to scan, and costs less to maintain on writes (only updates to pending orders need to update the index).

When partial indexes shine

  • Soft deletes: WHERE deleted_at IS NULL — index only live rows, not the historical deleted ones
  • Queue-like tables: WHERE processed = false — only index unprocessed jobs, which is a tiny fraction of the table
  • Status-filtered queries: WHERE status IN ('pending', 'processing') — index only active states, not historical completed ones
  • Non-null columns: WHERE column IS NOT NULL — exclude the nulls from the index (useful when a column is mostly null)

The constraint: the query must match the partial index condition

The query planner will only use a partial index if it can prove the query's WHERE clause is at least as restrictive as the index's WHERE clause. A partial index WHERE status = 'pending' will be used by queries that have WHERE status = 'pending' in them — but not by queries for all statuses. This is a feature: partial indexes are purpose-built for specific query patterns.

Partial index patterns — size and performance comparisonSQL
-- Full index (basline)
CREATE INDEX idx_orders_created_full ON orders (created_at DESC);
-- Size: indexes all 10M rows → ~500MB

-- Partial index: only pending orders (5% of table → ~25MB)
CREATE INDEX idx_orders_pending ON orders (created_at DESC)
WHERE status = 'pending';
-- 20× smaller, faster to scan, cheaper to maintain

-- Planner will use this for:
SELECT * FROM orders
WHERE status = 'pending' AND created_at > now() - interval '7 days'
ORDER BY created_at DESC;
-- ✓ Query has "status = 'pending'" → matches index condition

-- Planner will NOT use this for:
SELECT * FROM orders WHERE created_at > now() - interval '7 days';
-- ✗ No status filter → can't use partial index (might miss non-pending rows)

-- Soft delete pattern
CREATE INDEX idx_users_active ON users (email)
WHERE deleted_at IS NULL;
-- All queries looking up active users by email use this small index
-- Historical deleted users don't bloat it

-- Unique constraint on active rows only
CREATE UNIQUE INDEX idx_users_active_email ON users (email)
WHERE deleted_at IS NULL;
-- Enforces uniqueness only for active users
-- Deleted users can have duplicate emails (e.g. if they re-register)

-- Compare sizes
SELECT indexrelname, pg_size_pretty(pg_relation_size(indexrelid))
FROM pg_stat_user_indexes WHERE tablename = 'orders';

Section 08

Write amplification: the dark side of indexes

Every index on a table is a data structure that must be kept in sync with the table. Each INSERT, UPDATE, or DELETE must update every affected index, not just the table. A table with 10 indexes means every write touches 11 structures (the heap + 10 indexes). This is write amplification — and it's the reason you shouldn't add indexes indiscriminately.

The cost compounds in two ways:

  • CPU cost: Each index update involves a B-tree insertion or deletion, which may require page splits or merges. The more indexes, the more CPU per write.
  • I/O cost: Each index is stored on disk. Writing to 10 indexes means writing to 10 different areas of the disk (or SSD), potentially triggering 10 separate I/O operations per write.

The rule: indexes are a read-write trade-off

Every index you add makes reads faster and writes slower. For read-heavy workloads (OLAP, reporting, dashboards), this trade-off strongly favours indexes. For write-heavy workloads (event ingestion, logging, audit trails), over-indexing can cause write throughput to collapse.

Finding and removing unused indexes

PostgreSQL tracks index usage statistics. Any index with zero (or near-zero) scans is pure write overhead. Drop it — reads won't notice, writes will get faster.

Find unused indexes and measure write amplificationSQL
-- Find indexes that have never been used (since last stats reset)
SELECT
  schemaname || '.' || relname AS table,
  indexrelname AS index,
  pg_size_pretty(pg_relation_size(indexrelid)) AS size,
  idx_scan AS scans_since_reset
FROM pg_stat_user_indexes
WHERE idx_scan = 0
  AND schemaname NOT IN ('pg_catalog')
ORDER BY pg_relation_size(indexrelid) DESC;
-- These indexes are pure write overhead. Consider dropping.

-- Duplicate indexes (same columns, different names)
SELECT
  indrelid::regclass AS table,
  array_agg(indexrelid::regclass) AS indexes,
  array_agg(indkey) AS columns
FROM pg_index
GROUP BY indrelid, indkey
HAVING count(*) > 1;

-- Write amplification per table
SELECT
  relname,
  n_tup_ins + n_tup_upd + n_tup_del AS total_writes,
  (SELECT count(*) FROM pg_index WHERE indrelid = relid) AS index_count,
  (n_tup_ins + n_tup_upd + n_tup_del) *
    (SELECT count(*) FROM pg_index WHERE indrelid = relid) AS amplified_writes
FROM pg_stat_user_tables
ORDER BY amplified_writes DESC;

-- Monitor: index writes vs table writes ratio
SELECT
  t.relname,
  t.n_tup_ins,
  sum(i.idx_tup_fetch) AS index_reads,
  sum(i.idx_tup_read)  AS index_entries_scanned
FROM pg_stat_user_tables t
JOIN pg_stat_user_indexes i ON t.relid = i.relid
GROUP BY t.relname, t.n_tup_ins
ORDER BY t.n_tup_ins DESC;

Section 09

Production reality: the index problems that actually page you

The query that worked for months and then didn't

A query was fast for 8 months. No code changes. No schema changes. On a Monday it became slow. The table had grown from 2 million to 20 million rows. The index still exists. But the planner's statistics were last collected when the table was 2M rows — the distribution of values in the index columns has changed. The planner is now making wrong estimates and choosing sequential scans over index scans. Fix: ANALYZE the table. Set up monitoring for tables where autovacuum/autoanalyze hasn't run recently.

The migration that locked the table

CREATE INDEX (without CONCURRENTLY) acquires a ShareLock on the table — it blocks all writes for the duration of the index build. On a 10GB table, that's minutes of write downtime. Always use CREATE INDEX CONCURRENTLY on production tables. The downside: it can't be run inside a transaction, it takes 2-3× longer, and it can fail if there are long-running transactions. A failed concurrent index build leaves an invalid index behind — check for and drop invalid indexes: SELECT * FROM pg_indexes WHERE tablename = '...' AND NOT indisvalid.

The functional index nobody knew they needed

A query like WHERE LOWER(email) = LOWER($1) is a case-insensitive lookup. There's an index on email. But the planner can't use it — the index is on email values, not LOWER(email) values. These are different. The query does a full table scan. The fix is a functional index: CREATE INDEX ON users (LOWER(email)). The query must also be written as WHERE LOWER(email) = LOWER($1) — or better, store emails lowercase and use citext extension.

The index that made writes 10× slower

A team indexed every column of their events table (12 columns, 12 indexes). Write throughput dropped from 50,000 events/second to 5,000. Each event insert touched 13 B-tree structures (heap + 12 indexes). Solution: drop unused indexes, replace with composite indexes that cover multiple query patterns with fewer index structures.

Section 10

The things nobody tells you

The planner ignores your index if work_mem is too low

Sort and hash operations in PostgreSQL use work_mem of memory before spilling to disk. If work_mem is too small, the planner may prefer a sequential scan + in-memory sort over an index scan + heap fetch, because it estimates the sort will fit in memory but the random I/O of an index scan won't. This is highly workload-dependent. Running EXPLAIN ANALYZE and looking for "external merge Disk" is the tell.

Null values are in the index, and that matters

PostgreSQL includes null values in B-tree indexes by default. This is different from some other databases (MySQL excludes nulls from unique indexes). If you're creating a unique index and want to allow multiple null values (nulls are never equal to each other in SQL), that already works in PostgreSQL — no special syntax needed. But be aware: a query like WHERE col IS NULL can use an index.

CREATE INDEX CONCURRENTLY can fail silently

If CREATE INDEX CONCURRENTLY

fails midway (due to a long-running transaction, a unique constraint violation, or a server restart), it leaves an invalid index. This index has zero rows and will never be used — but it still takes up space and still costs write overhead on every write because PostgreSQL tries to maintain it. Check for invalid indexes regularly: SELECT indexrelname FROM pg_stat_user_indexes JOIN pg_index USING (indexrelid) WHERE NOT indisvalid. Drop and recreate them.

Index scans are not always sequential I/O

A common misconception: "sequential scan is slow, index scan is fast." It's more nuanced. An index scan can cause random I/O — the TIDs from the index point to rows scattered across the heap, and reading them means jumping around on disk. For spinning disks (and to a lesser extent SSDs), random I/O is expensive. This is why the planner sometimes chooses a sequential scan over an index scan for larger result sets — a sequential scan is predictable, linear I/O even if it reads more data.

Query is slow despite having an index — check in this order

If Index Scan is slow: check heap fetches and bloat
If EXPLAIN shows an Index Scan but it's slow, look at "Buffers: shared read=X" — high read count means the index pages aren't cached. Also look for "Heap Fetches: X" on an Index Only Scan — non-zero means the covering index isn't working (VACUUM needed). Finally, check index bloat: SELECT pgstatindex('idx_name') and look at leaf_fragmentation. Above 30% → rebuild with REINDEX INDEX CONCURRENTLY.
bloated index = slow even when used. Run REINDEX CONCURRENTLY.
5
Check composite index column order and left-prefix rule
If you have an index but it's not being used, check whether your query respects the left-prefix rule. Your index might be (user_id, status, created_at) but your query filters only on status and created_at — the leading column is missing. Either rewrite the query to include user_id, or create a new index that starts with the columns your query actually uses.
missing the leftmost column = index not used for that query
6
Check for function or implicit cast on the indexed column
If your query has WHERE LOWER(email) = ? and your index is on email — the index can't be used. Any function, cast, or expression on the indexed column invalidates the index for that predicate. Fix: create a functional index matching the expression: CREATE INDEX ON users (LOWER(email)). Or restructure the query to not apply a function to the indexed column.
function on indexed column = index skipped entirely
Section 12

Production indexing checklist

EXP
EXPLAIN first

Always run EXPLAIN ANALYZE before and after adding an index. Verify the plan changed.

CON
Concurrent builds

Always CREATE INDEX CONCURRENTLY on production. Never lock a live table.

CVR
Covering indexes

Use INCLUDE for high-frequency read queries. Verify "Heap Fetches: 0" in EXPLAIN.

ANA
Stats freshness

Alert if autovacuum/autoanalyze hasn't run on high-write tables in >1 hour.

UNU
Unused indexes

Monthly audit: drop indexes with idx_scan = 0. Pure write overhead.

BLT
Bloat monitoring

Check n_dead_tup / n_live_tup monthly. REINDEX CONCURRENTLY when bloated.
