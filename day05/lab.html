LAB
Production Incident  ·  Load Balancer  ·  Slow-Start

Scaling up during a traffic spike made the outage worse

Sale event  ·  Duration: 22 minutes  ·  Severity: P0  ·  New server crashed 4 minutes after joining pool

A major sale event drove 8× normal traffic. Three servers were handling it — struggling, but holding. The on-call engineer made the right call: scale up, add three more servers. The load balancer immediately distributed 50% of traffic to the new servers. Four minutes later, two of the three new servers crashed. The outage got worse.

The new servers were cold. No warm JVM heap. No hot OS page cache. No established database connection pool. The connection pool was still going through its initial handshake burst with the database. At exactly the moment the new servers needed the most resources to warm up, they received full production traffic.

T+0
Traffic spike hits. 3 existing servers at 78% CPU. P99 climbing. On-call decides to scale up.
T+2min
3 new servers added to load balancer pool. LB immediately routes 50% of all traffic to them (round-robin).
T+4min
New Server B and C crash. DB connection pool not warmed up — connection burst on startup saturated DB max_connections. OOM on cold heap. Now down to 1 original + 1 new server. Worse than before scaling.
T+8min
Team scrambles. Re-adds servers with a manual workaround: temporarily reduces LB weights on new servers to 1 while old servers stay at 10. Bleeding stops.
T+22min
New servers warmed up. Weights gradually raised. Traffic distributed evenly. Incident resolved. Post-mortem: add slow-start to all LB configs.
Root cause

No slow-start configured on the load balancer. New servers joining the pool received 33% of traffic immediately — the same share as servers that had been running for hours. Cold JVM, cold page cache, cold DB connection pool, and 33% of peak sale traffic simultaneously is a recipe for OOM. The fix is slow-start: new servers ramp from 0% to their full share over 30-60 seconds, giving them time to warm up before receiving full traffic.

upstream api_backend {
    least_conn;
    server 10.0.0.1:3000 slow_start=30s;
    server 10.0.0.2:3000 slow_start=30s;
    server 10.0.0.3:3000 slow_start=30s;  # ramps from 0% → 33% over 30s
}

# AWS ALB: set "Slow Start Duration" in Target Group attributes
aws elbv2 modify-target-group-attributes \
  --target-group-arn arn:aws:... \
  --attributes Key=slow_start.duration_seconds,Value=60
What this incident teaches
▶Scaling during a crisis can make the crisis worse. Cold servers + full traffic = crash. The time to configure slow-start is before the spike, not during.
▶Every server is cold when it joins a pool. No warm caches, no warm DB pool, no warm JVM heap. 100% traffic on a cold server is a bet that it can handle it immediately. That bet often loses under spike conditions.
▶Slow-start also applies to servers recovering from health check failures. A server that was sick and just recovered is also cold. Re-adding it without slow-start floods a still-fragile server.
▶Test your scale-up procedure before you need it. Autoscaling policies that work in normal conditions fail under spike conditions if slow-start isn't configured.
