Section 01

The lie: "balanced" doesn't mean what you think

Here is a true story from a production system I worked on. Four servers. Identical hardware, identical code, identical configuration. The load balancer was set to round-robin — the default. One server was consistently at 85% CPU. The other three were at 20%. The load balancer reported everything as healthy. On-call engineer spent two hours investigating a "broken server" before finding the root cause: one server was handling all the slow requests.

Round-robin distributes connections evenly, not work evenly. If Server 1 gets a 2ms health check and Server 3 gets a 12-second report export on the same turn, they both count as "one request" in the round-robin. Server 3 is now busy with a 12-second job while the next 5 requests from round-robin keep landing on it. By the time that export finishes, Server 3 has a queue of 5 waiting requests while Server 1 has handled 20 quick ones and is idle.

This is the fundamental problem: load balancers distribute without knowing cost. The algorithm you choose is your approximation of "cost" — and each algorithm makes different assumptions about what requests cost.

Section 02

The algorithms: what each one actually does

Round-robin

The simplest and most common default. Request 1 → Server 1, Request 2 → Server 2, Request 3 → Server 3, Request 4 → Server 1, and so on. It knows nothing about server state. It doesn't know if Server 2 is handling a 30-second database migration. It doesn't care.

When it's appropriate: When all requests have roughly the same cost and all servers are identical. Stateless, fast API endpoints (health checks, simple reads, low-compute operations) with homogeneous server hardware. The moment requests have variable cost or servers have different capacities, round-robin starts creating the exact kind of hot-spot from the opening story.

Least connections

Route to the server with the fewest active connections. Better than round-robin because it's aware of server state. If Server 3 has 50 active connections and Server 1 has 5, new requests go to Server 1.

The limitation: connection count ≠ actual load. Server 1's 5 connections might all be long-running streaming responses sending large payloads. Server 3's 50 connections might all be sub-millisecond health check pings. Least connections sees 5 vs 50 and routes to Server 1 — the wrong choice by actual CPU/memory load. But it's still much better than round-robin in most real-world scenarios with variable-duration requests.

Weighted round-robin / weighted least-connections

Assign relative weights to servers. A server with weight 3 gets 3× the traffic of a server with weight 1. Solves the heterogeneous hardware problem — if one server has 8 cores and another has 2, weight them 4:1. The problem: weights are static. You set them once. Your traffic patterns change. Your server farm changes. Nobody ever goes back and re-tunes the weights. Three months later the 2-core server is gone, replaced by a 16-core one, still weighted at 1.

Least response time

Route to the server that's currently responding fastest. Actually tracks real server performance rather than proxies like connection count. HAProxy calls this "leastconn" in HTTP mode (which combines response time awareness). This is the closest thing to "aware of actual load" without adding a full service mesh. Most production setups should graduate to this over round-robin — and most never do.

IP hash / consistent hashing

Hash the client's IP (or some other key) to a server. The same client always hits the same server. Solves the session stickiness problem without a session store — useful for legacy stateful applications. The catastrophic downside: a corporate office with 10,000 employees behind a single NAT address sends all 10,000 to one server. Also, when a server is removed from the pool, consistent hashing reshuffles only 1/N of traffic (regular hash remaps everything), but IP hash still means some clients get a different server on every change. Prefer consistent hashing over IP hash if you need this pattern.

Section 03

P2C: Power of Two Choices — the algorithm that actually works

P2C (Power of Two Choices) is the algorithm that Nginx, Envoy, and modern load balancers use under the hood. It's not the default you'd reach for, but it's the one that performs best under real load. The idea is elegant and comes from a 1994 paper on balls and bins.

The algorithm

Instead of picking the globally least-loaded server (which requires a global view and adds latency for coordination), P2C does this:

  1. Pick two random servers from the pool
  2. Compare their current connection count (or load metric)
  3. Route to the less loaded of the two

That's it. No global coordination. No complex state. Just two random picks and a comparison.

Why this works so well

The mathematical result is surprising. Pure random routing leads to the maximum load being O(log N / log log N) above average — roughly logarithmic imbalance. Round-robin achieves O(1) imbalance but with no load awareness. P2C achieves O(log log N) imbalance — exponentially better than random, and load-aware unlike round-robin. This was called "the power of two choices" because choosing the better of two random options gives you almost all the benefit of finding the globally optimal choice, at almost none of the coordination cost.

In practice: under variable-cost workloads, P2C dramatically reduces the probability that any single server becomes a hot spot. The heavy request that lands on a server increases its connection count, making it less likely to be selected in the next P2C comparison. The load self-corrects without any explicit tracking.

nginx: enabling P2C (least_conn is P2C in nginx upstream) NGINX 
# nginx uses P2C internally when least_conn is set with multiple servers
upstream api_backend {
    least_conn;  # P2C under the hood for multiple upstreams

    server 10.0.0.1:3000 weight=1;
    server 10.0.0.2:3000 weight=1;
    server 10.0.0.3:3000 weight=1;
    server 10.0.0.4:3000 weight=1;

    keepalive 32;  # connection pool per worker to each upstream
}

# HAProxy: explicit P2C with leastconn balance algorithm
backend api_servers
    balance leastconn
    option httpchk GET /healthz
    server s1 10.0.0.1:3000 check inter 2s rise 2 fall 3
    server s2 10.0.0.2:3000 check inter 2s rise 2 fall 3

# Envoy: uses P2C by default in LEAST_REQUEST policy
# lb_policy: LEAST_REQUEST  (default choice_count: 2 = P2C)
# Can increase choice_count for better distribution at cost of overhead
💡
The mental model

Think of P2C like choosing a checkout lane at a supermarket. You don't survey all 20 lanes (expensive). You glance at two random ones nearby and pick the shorter one. That simple heuristic gets you surprisingly close to optimal lane selection — the mathematical guarantee of P2C in a nutshell.

Section 04

L4 vs L7: what each layer sees and what it can do

There are two fundamentally different kinds of load balancers, and choosing the wrong one for your use case is expensive to undo. The distinction is which layer of the network stack the balancer operates at.

L4: Transport layer (TCP/UDP)

An L4 load balancer sees: source IP, destination IP, source port, destination port. That's it. It never reads the payload. It doesn't know if it's routing HTTP or PostgreSQL or a custom binary protocol. It makes routing decisions based purely on connection metadata.

What it can do: TCP port-based routing, connection-level load balancing, SSL passthrough (it doesn't terminate TLS — the backend server does). Very fast — it operates at kernel level (DPDK, eBPF) and can handle millions of connections per second with microsecond latency.

What it cannot do: Route based on URL path, HTTP headers, cookies, or request content. Can't do TLS termination (inspection). Can't return health check responses itself. Can't apply application-level rate limiting.

When to use L4: Non-HTTP protocols (databases, message queues, gaming servers), when you need absolute maximum throughput and minimum latency, as the outer layer in an L4 → L7 topology (L4 distributes connections to a cluster of L7 proxies, which do intelligent HTTP routing).

L7: Application layer (HTTP)

An L7 load balancer terminates the client connection, reads the full HTTP request (method, URL, headers, cookies, body if needed), makes a routing decision, and opens a new connection to the backend. It acts as a full HTTP proxy.

What it can do: Route based on URL path (/api/ → backend-cluster, /static/ → CDN-origin), route based on headers (A/B test routing, canary deployments by header), terminate TLS (decrypt and re-encrypt, or decrypt and send plaintext internally), modify requests and responses (add headers, rewrite URLs), perform health checks that understand HTTP (check for 200 OK, not just TCP open), apply rate limiting per user/token/path.

The cost: Each request pays the overhead of two TCP connections (client→LB and LB→backend), two TLS handshakes if HTTPS end-to-end, and the CPU cost of HTTP parsing. For most web traffic, this overhead is negligible. For ultra-high-frequency small requests (HFT, telemetry), it can matter.

🚀
The right topology for production

For most production web APIs: L7 load balancer (nginx, HAProxy, AWS ALB) directly in front of your backends. The L7 overhead is worth it for TLS termination, URL routing, and health checks that understand HTTP. Only add an L4 layer in front if you're handling non-HTTP protocols or need to distribute load across multiple L7 clusters.

Section 05

Health checks: the thing between "server is down" and "users know about it"

Health checks are how load balancers decide which servers to route to. Get them wrong and you get one of two failure modes: routing traffic to broken servers (wrong positive), or removing healthy servers from the pool (wrong negative). Both cause user-facing errors. The difference is which ones are your fault and which ones are the server's fault.

Active vs passive health checks

Active (probing): The load balancer periodically sends a test request to each server — either a TCP connection attempt or an HTTP request to a dedicated health endpoint. If the server fails N consecutive probes, it's removed from the pool. The critical parameter: probe interval. 10-second intervals mean 10 seconds of failed traffic before the server is removed. 1-second intervals mean better detection but more overhead.

Passive (observing real traffic): The load balancer watches the responses of actual requests. If a server starts returning 5xx errors or timing out on real requests, it's flagged. No polling overhead. Faster detection than active checks (you're watching real traffic, not waiting for a probe). But: if a server has zero traffic (e.g., at off-peak hours), passive checks won't detect it going down. And you're relying on real users seeing errors to trigger the detection.

Production setup: both. Active probes for baseline availability. Passive observation for "you're alive but returning errors." Each catches what the other misses.

What your health endpoint should check

This is the most common health check mistake: the /healthz endpoint returns 200 OK even when the server can't serve traffic. What it must check:

  • Database connectivity: SELECT 1 against the primary DB. If the DB is unreachable, the server can't serve reads or writes.
  • Critical dependency availability: If your service needs Redis for session management and Redis is down, the service is effectively down. Check Redis ping.
  • Not just "process is alive": A Node.js process can be alive but stuck in a blocking operation, consuming 100% CPU with no available event loop capacity. A shallow health check won't catch this.

The endpoint should return fast (within 1-2 seconds). If a dependency is slow to respond, the health check becomes a slow probe, which throws off your detection timing.

Rise and fall thresholds

Don't remove a server on the first failed probe — one probe failure might be a transient network blip. Configure a fall threshold (number of consecutive failures before removal, typically 3) and a rise threshold (consecutive successes before re-adding to pool, typically 2). This prevents thrashing — a flaky server repeatedly added and removed.

Health check endpoint — what to actually check TYPESCRIPT 
import { Router } from 'express';

const router = Router();

// /ping — lightweight liveness check (LB tcp probe, k8s liveness)
// Should NEVER check dependencies — just "is the process alive"
router.get('/ping', (req, res) => res.json({ status: 'ok' }));

// /healthz — deep readiness check (LB http probe, k8s readiness)
// Should check all critical dependencies
router.get('/healthz', async (req, res) => {
  const checks: Record<string, boolean> = {};
  let healthy = true;

  // Database — most important
  try {
    await db.raw('SELECT 1');
    checks.database = true;
  } catch {
    checks.database = false; healthy = false;
  }

  // Redis — if your app needs it for core functionality
  try {
    await redis.ping();
    checks.redis = true;
  } catch {
    checks.redis = false; healthy = false;
  }

  const status = healthy ? 200 : 503;
  res.status(status).json({ status: healthy ? 'ok' : 'degraded', checks });
  // LB uses HTTP status: 200 = keep in pool, 503 = remove
});

Section 06

Slow-start: the problem with adding a server to a busy pool

You've had a traffic spike. You scale up — add two new servers to the load balancer pool. The load balancer immediately routes 25% of traffic to each new server (you now have 4 total). But those new servers are cold: no warm JVM, no hot OS file cache, no warm connection pools to the database. Every request they handle takes 3× longer than a warm server. They queue up requests faster than they can process them. One of them crashes under the sudden load. You've made the traffic spike worse by scaling.

This is the cold-start problem, and the solution is slow-start: when a new server joins the pool, give it a gradually increasing fraction of traffic over a configurable ramp-up period. Start at 5-10% of its eventual share, increase linearly over 30-60 seconds. By the time it's receiving full traffic, it's warm and performing like the other servers.

Slow-start in nginx

nginx's upstream module has a slow_start parameter for weighted servers. When a server re-enters the pool (either newly added or recovered from a health check failure), its weight ramps from 0 to its configured weight over the slow-start duration. Requests are distributed proportionally to weight, so zero weight means no traffic initially.

nginx slow-start configuration NGINX 
upstream api_backend {
    least_conn;

    # slow_start: ramp new/recovered servers over 30s
    # Only available in nginx Plus or with 3rd-party modules for OSS
    server 10.0.0.1:3000 slow_start=30s;
    server 10.0.0.2:3000 slow_start=30s;
    server 10.0.0.3:3000 slow_start=30s;  # newly added

    keepalive 32;
}

# HAProxy equivalent: weight ramp with agent-check
# Or: add server with low initial weight and raise it via API

Section 07

Connection draining: removing a server without breaking requests

Removing a server from the pool without connection draining is like pulling the plug on a phone mid-conversation. All in-flight requests immediately fail. Users see errors. The right process is graceful: stop routing new requests to the server, let existing requests complete, then remove it.

This is connection draining (AWS calls it "deregistration delay"). The sequence:

  1. Signal the server for removal (deployment, scaling down, maintenance)
  2. Load balancer stops routing new requests to this server
  3. Existing connections are held open until they complete or a maximum drain timeout expires
  4. After drain timeout, any remaining connections are forcibly closed
  5. Server is removed from the pool

Configuring the drain timeout

The drain timeout should be longer than your maximum expected request duration. For typical REST APIs with a 30-second timeout, a 60-second drain timeout is safe. For services with long-running streaming responses or websocket connections, you may need 5+ minutes.

AWS ALB's default deregistration delay is 300 seconds (5 minutes). For fast-deploying services with short requests, this is overkill and slows deployments. Set it to 30-60 seconds for API services. For websocket or streaming services, keep it at 300s or higher.

The graceful shutdown sequence in your app

Connection draining happens at the load balancer level, but your application also needs to handle graceful shutdown. When the process receives SIGTERM:

  1. Stop accepting new connections (server.close())
  2. Set readiness probe to return 503 (triggers LB to drain)
  3. Wait for in-flight requests to complete
  4. Close database connections cleanly
  5. Exit with code 0
Graceful shutdown — Node.js TYPESCRIPT 
let isShuttingDown = false;

// Readiness probe: 503 during shutdown triggers LB draining
app.get('/healthz', (req, res) => {
  if (isShuttingDown) return res.status(503).json({ status: 'shutting_down' });
  res.json({ status: 'ok' });
});

async function gracefulShutdown(signal: string) {
  console.log(`Received \${signal}, starting graceful shutdown`);
  isShuttingDown = true;

  // Give LB time to detect 503 and stop routing (one health check interval)
  await new Promise(r => setTimeout(r, 5_000));

  // Stop accepting new connections
  await new Promise<void>((resolve, reject) => {
    server.close(err => err ? reject(err) : resolve());
  });

  // Close DB pool
  await db.destroy();
  await redis.quit();

  console.log('Graceful shutdown complete');
  process.exit(0);
}

process.on('SIGTERM', () => gracefulShutdown('SIGTERM'));
process.on('SIGINT',  () => gracefulShutdown('SIGINT'));

// Set timeout to force-exit if graceful shutdown hangs
setTimeout(() => { console.log('Forced shutdown'); process.exit(1); }, 60_000).unref();

Section 08

Session affinity: the sticky session trap

Sticky sessions (session affinity) route all requests from a given client to the same backend server. The load balancer uses a cookie or IP to identify the client and maintain this mapping. For legacy applications that store session state in process memory, sticky sessions are often the only option for keeping login sessions alive.

Why sticky sessions are a problem

Single point of failure: When the server a user is stuck to goes down, that user's session is lost. They're logged out. They see an error. The load balancer eventually routes them to a healthy server, but their session state is gone.

Defeats load balancing: Sticky sessions mean some servers are stuck with "heavy users" while others handle "light users." You can't redistribute load during a traffic spike without breaking sessions. The whole point of having multiple servers is undermined.

Complicates deployments: You can't remove a server from the pool without losing the sessions of all users stuck to it. This forces you to either wait for sessions to expire (could be hours) or accept that some users get logged out during every deployment.

The right answer: stateless + external session store

Move session state out of the application process and into a shared external store — Redis. The application becomes stateless: any server can handle any request because the session data is always retrievable from Redis. No stickiness needed. Servers can be added, removed, and replaced freely. Sessions survive server restarts.

⚠️
JWT is not a session store replacement

Moving to JWT-based auth doesn't eliminate the need for a session store — it just changes what you store. You still need server-side token revocation (blocklist), which requires Redis or a database. A JWT without server-side validation means a stolen token stays valid until expiry, which is often hours. If you need instant logout (user reports account compromise), you need a revocation store. Use short-lived JWTs (15 minutes) with server-tracked refresh tokens.

Section 09

Production reality: the LB problems that actually page you

The 502 that isn't your server's fault

Users see 502 Bad Gateway. You check your application logs — the request never arrived. You check the load balancer — it shows the backend as healthy. What happened?

The load balancer sent the request to a backend server over a keep-alive connection. Between the last successful request and this one, the backend server restarted or the keep-alive timed out. The server sent a TCP RST on the "alive" connection. The load balancer didn't detect this before sending the request. The backend closes the connection mid-request. The LB returns 502.

This is the keep-alive timeout mismatch from Day 1, appearing as a load balancer problem. Fix: the LB's upstream keep-alive timeout must be shorter than the backend server's keep-alive timeout. Set your app server's keep-alive to 75s and the LB's upstream pool keep-alive to 60s.

The thundering herd on server recovery

Server goes down. Load balancer removes it. 5 minutes later, health checks pass, server is re-added to pool. Load balancer routes 25% of traffic to it immediately. Server has cold caches, cold connection pools. It starts responding slowly. Health check still passes (it's alive, just slow). Round-robin keeps sending it traffic. The slow responses back up. Eventually the server is overwhelmed and the health check fails. It's removed again. 30 seconds later it's re-added. Repeat.

This is why slow-start matters. It's also why your fall threshold shouldn't be too high — if you need 10 consecutive failures to remove a server, a slow-but-alive server stays in the pool hurting users for a long time.

Health check as DDoS

You have 10 load balancer instances (typical in a large deployment) each sending health checks every 2 seconds to 50 backend servers. That's 10 × 50 / 2 = 250 health check requests per second to your backends. For backends that have expensive health checks (checking DB connectivity, Redis, downstream services), this is 250 DB queries per second that your application never directly caused. Keep health checks cheap: SELECT 1, not a full business logic validation.

Section 10

The things nobody tells you

The load balancer is also a single point of failure

Your application is highly available across 10 servers. Your load balancer is a single instance. Every request goes through it. When it goes down, everything goes down. Production load balancers must run in active-active or active-passive pairs with a shared VIP (virtual IP) and a fast failover mechanism (Keepalived, Pacemaker, or managed solutions like AWS ALB which is inherently multi-AZ). Never run a single load balancer in production.

The load balancer adds measurable latency

An L7 load balancer typically adds 0.5-2ms per request (TCP connection overhead, HTTP parsing, connection pool lookup). For an API with a 5ms handler, this is 10-40% overhead. For most services, this is negligible. For ultra-low-latency services (real-time bidding, HFT, gaming), this matters. In those cases, use L4 LB or client-side load balancing (each client has the backend list and picks directly).

Monitoring the LB itself

Most teams monitor their application servers. Fewer monitor the load balancer itself. What to track: connection count per backend, 5xx rate per backend (not just total), request distribution variance (are all backends getting similar load?), active connection count at the LB (are you approaching limits?), queue depth (requests waiting for a backend connection). If your LB is nginx, expose the stub_status module. HAProxy has a built-in stats page. AWS ALB exposes all of this via CloudWatch.

The request distribution lie in your metrics

Your load balancer metrics show even distribution across backends — 25% each. Your application metrics show one backend doing 60% of the work. How? The LB distributes connections evenly, but one backend is handling all the slow requests (database-heavy, large responses). Connection count is even; request cost is not. Always correlate LB distribution metrics with application-level metrics (CPU, response time per backend) to detect this mismatch.

Design decisions

Configuring a load balancer — decide in this order

Each step narrows your options. Answer them top to bottom when designing or reviewing any load-balanced system.

1
What protocol are you balancing?
HTTP/HTTPS → use L7 (nginx, HAProxy, ALB). Raw TCP (database proxies, game servers, message queues) → use L4. Mixed (HTTP APIs + WebSocket + gRPC) → L7 with protocol detection. This single decision determines which balancer you even consider. Choosing L4 for an HTTP API locks you out of URL routing, TLS termination, and HTTP-aware health checks.
HTTP = L7. Everything else starts at L4.
2
Are your requests uniform cost or variable cost?
Uniform (health checks, simple reads, fast writes at similar latency) → round-robin is fine. Variable (some requests hit the DB hard, some are cache hits, some trigger async jobs) → use least-conn or P2C. Round-robin on variable-cost workloads creates the exact hotspot from the opening story: one server drowning while others idle. This is the most impactful algorithm decision you'll make.
variable cost + round-robin = hidden hotspots
3
Configure health checks — active + passive, and what they actually check
Active probes for "is the server alive." Passive observation of real traffic for "is the server healthy." Your /healthz endpoint must check DB connectivity and Redis — not just return 200. Set fall=3 (remove after 3 consecutive failures) and rise=2 (restore after 2 passes). Separate /ping for liveness (never checks deps) from /healthz for readiness (always checks deps).
200 OK without checking deps = the LB is fooled
4
Enable slow-start for new and recovering servers
Every new server is cold: no warm JVM, no hot OS cache, no warmed DB connection pool. Sending it 25% of traffic immediately is how you crash servers you just added to fix a traffic spike. Set slow_start=30s in nginx, deregistration delay in ALB, or weight ramp in HAProxy. 30-60 seconds of gradual ramp-up prevents the thundering herd on server recovery.
no slow-start = new servers crash on join
5
Set connection draining timeout longer than your longest request
When a server is removed (deployment, scale-in, crash recovery), in-flight requests need time to complete. AWS ALB default is 300s — usually too long for API services, causing slow deploys. For REST APIs with 30s timeouts, set drain to 60s. For streaming/websocket services, 300s is appropriate. Also implement graceful shutdown in your app: SIGTERM → 503 → stop accepting → drain → close DB → exit.
drain < max request time = requests killed mid-flight
6
Monitor the LB itself — it can become the bottleneck
Track: connection count per backend (is distribution even?), 5xx rate per backend (which server is broken?), accept queue depth (are connections being dropped?), and LB-added latency. If all backends look healthy but users see errors, the problem is often at the LB level — connection limits, TLS overhead, or the balancer itself being underprovisioned. Never run a single LB instance in production.
alert: 5xx per backend + queue depth + distribution skew
