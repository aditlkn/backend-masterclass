Production Incident  ·  Dead Letter Queue  ·  Schema Mismatch

10,000 confirmation emails never sent. Users started calling on day 4.

Friday deploy v2.1.4 → Monday morning customer complaints · Severity: P1 · 10,847 affected bookings

Friday at 17:00, a deploy went out that changed the confirmation email template structure. The old producer sent { userEmail, bookingId, eventName } in the job payload. The new worker expected { userId, bookingId } and fetched everything else from the database. The mismatch wasn't caught in staging because staging had no queue backlog.

For jobs enqueued before the deploy, the worker received userEmail in the payload but tried to fetch the user by userId — which wasn't there. Every pre-existing job in the queue failed with TypeError: Cannot read property 'id' of undefined. All 10,847 of them moved to the Dead Letter Queue.

There was no DLQ depth alert. The queue dashboard showed "email queue healthy" because the queue depth was zero — all jobs had been processed. They'd just been processed into the DLQ. Monday morning, customers started calling: "I booked a ticket Friday but never got a confirmation email." The support team couldn't find the issue for 3 hours because they were looking at the API, not the job queue.

Root cause + three-part fix

1: The schema mismatch. Worker should handle both old and new payload shapes with a fallback. 2: No DLQ alert. An alert at DLQ depth > 10 would have fired within 5 minutes of the deploy. 3: No backward compatibility gate. Job schema changes must be backward-compatible — old jobs in the queue must still process correctly under the new worker. Never make breaking changes to job payload schemas.

The fix: backward-compatible schema handling

email worker — before (broken) vs after (backward-compatible)TYPESCRIPT
❌ Before v2.1.4 — expects only new schema
const worker = new Worker("email", async (job) => {
  const { userId, bookingId } = job.data;  // NEW schema only
  const user = await db.findUser(userId);  // userId may be undefined!
  const booking = await db.findBooking(bookingId);
  await sendEmail(user.email, booking);    // crashes if user is null

✅ After — handles both old and new schema
const worker = new Worker("email", async (job) => {
  const { userId, userEmail, bookingId } = job.data;

  // Support OLD schema (userEmail in payload) and NEW schema (userId → fetch)
  let email: string;
  if (userEmail) {
    email = userEmail;                     // old schema: use directly
  } else if (userId) {
    const user = await db.findUser(userId); // new schema: fetch by ID
    if (!user) throw new UnrecoverableError(`User ${userId} not found`);
    email = user.email;
  } else {
    throw new UnrecoverableError("Job has neither userId nor userEmail");
  }

  const booking = await db.findBooking(bookingId);
  if (!booking) throw new UnrecoverableError(`Booking ${bookingId} not found`);

  await sendConfirmationEmail(email, booking);
});
What this incident teaches
▶DLQ depth must be monitored and alerted. The queue depth showed zero (correctly — all jobs were "processed"). The DLQ depth showed 10,847 (incorrectly treated as non-critical). Alert at DLQ depth > 10. This would have fired at 17:05 Friday, not Monday morning.
▶Job schema changes must always be backward-compatible. Jobs enqueued before a deploy will be processed by the new worker. The new worker must handle old job payloads. Rule: add fields, never remove or rename existing fields. When you want to change the shape, add new fields alongside old ones, drain the queue, then remove old fields.
▶Staging needs a queue backlog test. The bug was invisible in staging because staging had no pre-existing queued jobs. Before shipping a worker change: seed staging with jobs using the old schema, deploy the new worker, verify old jobs process correctly. This would have caught this exact bug in 2 minutes.
▶Recovery: retry all DLQ jobs after deploying the fix. Once the backward-compatible worker was deployed, all 10,847 jobs in the DLQ were retried. They all succeeded. Emails went out 72 hours late, but they went out. The DLQ is not a trash can — it's a holding area for problems that can be fixed.
