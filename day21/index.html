Section 01

Why async: the work the user shouldn't wait for

Every API response has two parts: the work that must complete before you can respond, and the work that merely needs to happen eventually. Most teams never explicitly separate these. The result: users wait for email sends, PDF generation, webhook deliveries, and analytics events — all of which have nothing to do with confirming their transaction.

The rule is simple: if it doesn't change the response, it doesn't belong in the request path. An email confirmation doesn't change "your ticket is booked." A PDF doesn't need to exist before you say "payment successful." A webhook to a third-party doesn't affect what you tell the user.

What belongs in the async queue

  • Notifications: Email, SMS, push notifications, Slack messages. Inherently fire-and-forget from the user's perspective.
  • Document generation: PDFs, invoices, receipts. These take 100-500ms and can be generated after the fact, linked from the confirmation page.
  • Downstream webhooks: Notifying partners, CRMs, analytics platforms. If Salesforce is slow, that's not the user's problem.
  • Image and media processing: Resizing, transcoding, watermarking. Never synchronous.
  • Bulk operations: Importing a CSV of 10,000 records, sending a newsletter to 50,000 users.
  • Cache warming: Precomputing expensive views after data changes.
  • Audit and compliance events: Write to your audit log asynchronously — the response shouldn't wait for this.

What must stay synchronous

Payment authorisation (you need the result to confirm the transaction), inventory reservation (two users cannot book the same seat — this is a synchronous write with locking, Day 9), authentication and authorisation (obviously), and anything that changes the data the response needs to report. The decision tree: if you need the result to formulate the response, it's synchronous. If you don't, it's a candidate for async.

💡
Async also improves resilience

A synchronous email send means your checkout fails if SendGrid is down. An async queue means the email just gets delayed — the checkout succeeds, the email goes out when the provider recovers. Decoupling with a queue makes the primary user flow resilient to failures in non-critical downstream systems. Day 18's circuit breaker pattern and async queues are complementary: use circuit breakers for synchronous calls you must make, and async queues for calls you can defer.

Section 02

Queue anatomy: producers, queues, workers, jobs

A job queue system has four components. Understanding each one and its responsibilities is necessary before the failure modes make sense.

Producer: The code that creates jobs. Usually your API handler, after completing the synchronous work. The producer serialises the job payload and pushes it onto the queue. The producer should not know or care who processes the job or when — it just needs to know the job was successfully enqueued.

Queue: The persistent store for jobs waiting to be processed. In Redis-backed queues (BullMQ, Bull), the queue is a set of Redis data structures. Jobs in the queue survive application restarts. The queue is responsible for durability, ordering, and state transitions.

Worker: The process that pulls jobs from the queue and executes them. Workers are horizontally scalable — you run more workers when you need more throughput. Workers are independently deployable — you can update worker code without touching the API. Workers should be idempotent where possible.

Job: A unit of work with a payload, metadata (created at, attempts, status), and a lifecycle (waiting → active → completed | failed → dead letter).

Section 03

Job patterns: fire-and-forget, fan-out, chaining, scheduling

Fire-and-forget

The simplest pattern: enqueue a job, return the response, never look at the result. Appropriate for notifications — you don't need to confirm the email sent before continuing. The downside: you have no feedback loop. If the job fails, who knows? The answer: the dead letter queue and your alerting on DLQ depth.

Fan-out

One event triggers multiple independent jobs. A ticket purchase creates: a confirmation email job, a PDF generation job, a webhook delivery job, and an analytics event job — all in parallel. Each runs independently; failure in one doesn't block the others. The API handler enqueues all four and returns immediately.

Job chaining

Jobs that must run in sequence, where each step's output feeds the next. Generating a PDF then emailing it: you can't email what hasn't been generated yet. BullMQ supports chains with job.add({ ... }, { parent: previousJob }). The chain fails atomically if any step fails — no partial completions.

Scheduled / cron jobs

Jobs that run on a schedule: daily report emails, weekly digest, hourly cache warm-up. BullMQ's repeat option uses cron syntax. The critical constraint: ensure exactly-once scheduling across multiple worker pods — if all workers have the scheduler enabled, you'll get duplicate cron jobs on deployment. Use a single designated scheduler or leader election.

BullMQ — producer patternsTYPESCRIPT
import { Queue, QueueEvents } from 'bullmq';
import { redis } from './redis';

const emailQueue    = new Queue('email',    { connection: redis });
const pdfQueue      = new Queue('pdf',      { connection: redis });
const webhookQueue  = new Queue('webhook',  { connection: redis });
const analyticsQueue = new Queue('analytics', { connection: redis });

// Pattern 1: Fan-out — one purchase creates 4 independent jobs
async function onBookingConfirmed(booking: Booking) {
  await Promise.all([
    emailQueue.add('booking-confirmation', {
      to:        booking.userEmail,
      bookingId: booking.id,
      eventName: booking.eventName,
      seats:     booking.seats,
    }),
    pdfQueue.add('ticket-pdf', {
      bookingId: booking.id,
      template:  'ticket-v2',
    }),
    webhookQueue.add('partner-notify', {
      partnerId: booking.partnerId,
      event:     'booking.confirmed',
      payload:   booking,
    }, {
      attempts: 5,              // more retries for webhooks
      backoff: { type: 'exponential', delay: 2000 },
    }),
    analyticsQueue.add('track-purchase', {
      event:     'purchase_completed',
      userId:    booking.userId,
      amountCents: booking.amountCents,
    }, {
      priority: 10,    // low priority — analytics can wait
    }),
  ]);
}

// Pattern 2: Delayed job — retry payment in 30 minutes
await paymentQueue.add('retry-payment', { orderId }, {
  delay: 30 * 60 * 1000,  // 30 minutes in ms
});

// Pattern 3: Scheduled cron — daily digest at 08:00 IST
await emailQueue.add('daily-digest', {}, {
  repeat: { cron: '0 2 * * *' },  // 02:00 UTC = 07:30 IST
  jobId: 'daily-digest',         // fixed ID prevents duplicates
});

Section 04

Reliability: at-least-once delivery and idempotency

Job queues guarantee at-least-once delivery, not exactly-once. A worker that crashes mid-job, a network partition during the acknowledgment, or a Redis failover can all cause a job to be redelivered and processed again. Your workers must be prepared for this.

Why at-least-once, not exactly-once

Exactly-once delivery across a distributed system requires a two-phase commit between the queue and the worker's side effects — atomically dequeuing the job and completing the work. For most job types, this is either impossible (sending an email is not a transaction) or prohibitively complex. At-least-once is the realistic guarantee: the job will be processed, but may be processed more than once.

Making workers idempotent

An idempotent worker produces the same outcome whether it runs once or ten times. For each job type, identify the side effects and design around duplicates:

  • Email sends: Check if the email was already sent for this job ID before sending. Store sent job IDs in Redis with a TTL. SET email:sent:{jobId} 1 EX 86400 NX — if SET returns nil, it was already sent.
  • Database writes: Use INSERT ... ON CONFLICT DO NOTHING or upserts. Never use raw INSERT in a worker that might run twice.
  • Webhook deliveries: Include an idempotency key in the webhook payload. Recipients should deduplicate by key. Include the job ID as the idempotency key.
  • PDF generation: Check if the PDF already exists in storage before generating. Store by a deterministic key (booking ID + template version).
⚠️
The lock-before-process pattern

For jobs where idempotency is hard to achieve (external API calls with no dedup support), use a distributed lock as a gate: SET lock:{jobId} 1 NX EX 300. If the SET succeeds, process the job. If it fails, another worker is processing it — skip or wait. Set the lock TTL to longer than your maximum job duration. This converts at-least-once into approximately-once, with a small risk of double-processing if the lock expires during a long job.

Section 05

Retry with exponential backoff: not all failures are equal

When a job fails, the right response depends on why it failed. A transient network error — retry soon. A rate limit from an external API — retry after the Retry-After header says to. A business logic error (invalid data) — don't retry, move to DLQ immediately. A timeout — retry with a longer timeout. Treating all failures the same is how you hammer a recovering service with retries until it can't recover.

BullMQ's backoff strategies

  • fixed: Wait exactly N milliseconds between retries. Simple but can create thundering herds when many jobs fail simultaneously.
  • exponential: Wait 2^attempt × baseDelay. 1st retry: 1s, 2nd: 2s, 3rd: 4s, 4th: 8s. Spreads retries over time, gives services a chance to recover.
  • Custom: Implement your own backoff logic — read the error type, inspect the Retry-After header, apply different delays for different failure modes.
BullMQ worker with typed retry logicTYPESCRIPT
import { Worker, Job } from 'bullmq';

const worker = new Worker('email', async (job: Job) => {
  const { to, bookingId, eventName } = job.data;

  // Check idempotency before doing any work
  const alreadySent = await redis.set(
    `email:sent:\${job.id}`, '1', 'EX', 86400, 'NX'
  );
  if (!alreadySent) {
    logger.info({ jobId: job.id }, 'email.already_sent.skipping');
    return { skipped: true };
  }

  try {
    await sendgrid.send({ to, subject: `Your tickets for \${eventName}`, ... });
    logger.info({ jobId: job.id, to, bookingId }, 'email.sent');
  } catch (err) {
    // Clear the idempotency key on failure so next attempt can try again
    await redis.del(`email:sent:\${job.id}`);

    // Don't retry on non-retryable errors
    if (err.code === 'INVALID_EMAIL') {
      throw new UnrecoverableError('Invalid email address'); // → DLQ, no retry
    }

    // Rate limited — throw with retry delay hint
    if (err.status === 429) {
      const retryAfter = parseInt(err.headers['retry-after'] || '60') * 1000;
      job.update({ ...job.data, _retryDelay: retryAfter });
      throw err;  // BullMQ will use the configured backoff
    }

    throw err;  // Other errors: use configured backoff
  }
}, {
  connection: redis,
  concurrency: 5,       // process 5 jobs simultaneously per worker
  limiter: { max: 100, duration: 1000 },  // max 100 jobs/sec across all workers
});

// Queue config with retry policy
const emailQueue = new Queue('email', {
  connection: redis,
  defaultJobOptions: {
    attempts:  4,
    backoff: { type: 'exponential', delay: 2000 },
    // Attempts: T+0s (initial), T+2s, T+6s, T+14s, then DLQ
    removeOnComplete: { count: 1000 },  // keep last 1000 completed
    removeOnFail: false,              // keep failed jobs for inspection
  },
});

Section 06

The Dead Letter Queue: where failed jobs go to wait

A job that exhausts all retry attempts doesn't disappear — it moves to the Dead Letter Queue (DLQ). The DLQ is a holding area for jobs that need human attention: they couldn't be processed automatically, but the data shouldn't be lost. The DLQ is not a trash can; it's a queue of problems waiting to be diagnosed.

What the DLQ tells you

Depth and growth rate of the DLQ are critical metrics. A stable DLQ of 5 jobs might be acceptable — permanent failures like bad email addresses. A growing DLQ means a systematic problem: an API is broken, a schema changed, or your processing logic has a bug that affects a whole class of jobs.

DLQ operations

  • Inspect: Read the job payload and the failure reason. Most failures have obvious causes when you look at the error message.
  • Retry: After fixing the underlying problem, move jobs from DLQ back to the main queue. BullMQ: job.retry().
  • Discard: Some jobs are permanently unprocessable (bad data, user deleted). Remove them after logging the event.
  • Replay: For a class of jobs affected by a bug you've now fixed, replay all DLQ jobs through the fixed worker.
🔴
Alert on DLQ depth — don't let it grow silently

The DLQ is invisible without monitoring. A worker bug that moves 10,000 jobs to the DLQ over a weekend looks fine in your API metrics (requests are still succeeding) but means 10,000 confirmation emails were never sent, 10,000 PDFs were never generated, and 10,000 webhook deliveries to your partners never happened. Alert when DLQ depth exceeds a threshold (e.g. 10 jobs). Set it up before you need it.

Section 07

Worker scaling: concurrency, throughput, and resource limits

Worker scaling has two axes: concurrency per worker process (how many jobs each worker handles simultaneously) and number of worker instances (how many worker processes are running). Getting the right combination matters — too little and jobs pile up, too much and workers exhaust their resources or overwhelm downstream services.

Concurrency per worker

For I/O-bound jobs (HTTP calls, database queries, email sends), you can run many concurrent jobs per worker because each job spends most of its time waiting. BullMQ's concurrency: 20 means the worker pulls up to 20 jobs from the queue simultaneously and runs them concurrently. For CPU-bound jobs (image processing, PDF generation), concurrency should match your CPU count — more than that and you're just context-switching.

Number of worker instances

Scale the number of worker instances based on queue depth and job processing rate. Kubernetes horizontal pod autoscaler can scale workers based on a custom metric (KEDA supports scaling based on queue depth directly). The rule: target a queue depth that means jobs wait no longer than your acceptable delay. If jobs should process within 30 seconds and each worker handles 10 jobs/minute, and you have 100 jobs queued, you need 10 workers online.

Rate limiting workers

Workers can overwhelm downstream services. If your email queue processes 5,000 jobs and each fires a SendGrid API call, you might hit their rate limits. BullMQ's limiter option caps the global processing rate across all workers sharing the same Redis connection: { max: 100, duration: 1000 } = 100 jobs per second total, regardless of how many workers are running.

Section 08

Job priorities and queue separation

Not all jobs are equal. A confirmation email for a just-completed £500 booking is more urgent than a weekly digest. A webhook delivery to a paying partner is more urgent than an analytics event. Two strategies for handling priority: numeric priority within a queue, and separate queues with separate workers.

Numeric priority (within one queue)

BullMQ supports job priorities where lower numbers run first. Jobs with priority: 1 are processed before priority: 10. Simple and effective when you have a single logical job type with different urgency levels. Limitation: a flood of high-priority jobs can completely starve low-priority jobs.

Separate queues (recommended for distinct job types)

Separate queues with separate workers provide true isolation. The critical queue (confirmation emails) always has workers dedicated to it. A burst of analytics jobs can't delay confirmation emails because they're on different queues with different worker pools. The cost: more infrastructure to manage. Worth it for jobs with hard latency requirements.

✅
The queue topology for most applications

Three queues covers 90% of use cases: critical (confirmation emails, payment retries, partner webhooks — 5 workers, processed first), default (PDF generation, internal notifications — 3 workers), bulk (newsletter sends, CSV exports, analytics — 1-2 workers, can be slow). This prevents bulk operations from ever blocking critical user-facing jobs, and is easy to scale independently.

Section 09

Production reality

Redis as queue backend: durability concerns

BullMQ uses Redis. Redis in its default configuration is not durable — a restart without AOF persistence loses all queued jobs. For production job queues, enable Redis AOF (Append-Only File) with appendonly yes and appendfsync everysec. This persists every write to disk every second, limiting loss to at most 1 second of jobs on an unclean shutdown. For zero job loss, use appendfsync always (slower) or use a managed Redis with AOF enabled (ElastiCache, Upstash, Redis Cloud).

BullMQ vs alternatives

BullMQ (Node.js/Redis) is the right choice for most teams starting out. For polyglot systems where workers exist in multiple languages, consider Temporal (workflow orchestration with durable execution) or a message queue like RabbitMQ or Amazon SQS that any language can consume. The tradeoff: BullMQ has the richest feature set for Node.js; SQS is operationally simpler but less feature-rich; Temporal is the most powerful but also the most complex.

Job timeout: don't let workers hang forever

A worker that picks up a job must complete it or fail it within some time limit. Without a timeout, a stuck worker holds the job in "active" state indefinitely, preventing retries. BullMQ's lockDuration option defines how long a worker can hold a job before it's considered abandoned and re-queued. Set it to slightly longer than your longest legitimate job duration. A job that legitimately takes 10 minutes should have lockDuration: 700000 (70s longer than max) — not the default 30 seconds.

Section 10

The things nobody tells you

Job payloads should be small references, not full objects

It's tempting to put the entire user object, the entire order, and all related data into a job payload. Don't. Store the IDs and fetch fresh data in the worker. A payload with stale data causes subtle bugs: the email is generated with the address the user had at booking time, not at email-send time if they changed it. Fetch fresh data in the worker using the IDs in the payload. Smaller payloads also mean faster Redis operations and less memory.

Worker crashes during job processing leave jobs "stuck in active"

If a worker process crashes (out of memory, OOM kill, forced deployment) while processing a job, the job stays in "active" state. BullMQ's lockDuration handles this — after the lock expires, the job is considered abandoned and moved back to waiting for retry. But if your lockDuration is too long (hours), stuck jobs can sit in active for a long time before being retried. Monitor the count of jobs in "active" state — if it exceeds the number of running workers, something is stuck.

Deployment causes job processing gaps

When you deploy a new worker version, the old workers stop and new ones start. Jobs in the queue during this gap wait until new workers are up. If your deployment takes 60 seconds and you have 10 jobs arriving per second, 600 jobs queue up. Make your workers start processing quickly after boot, and use rolling deployments to keep at least some workers running at all times.

Scheduled jobs and multiple worker pods don't mix without care

If you run cron jobs from your worker process and you have 5 worker pods, each pod independently schedules the cron job — you get 5× the intended executions. Fix: use a fixed jobId with BullMQ's repeat (duplicate job IDs are silently ignored), or use a dedicated scheduler pod, or use a leader election lock before scheduling.

Decision sequence

Should this work be async?

Work through this before adding any new async job.

1
Does the response need the result of this work?
If yes — it must be synchronous. Auth validation, inventory reservation, payment authorisation: the response depends on the outcome. If no — it's a candidate for async. Email sends, PDF generation, webhooks, analytics: the response "order confirmed" doesn't depend on whether the email was sent yet.
response needs result = synchronous, always
2
Does the user need immediate confirmation of this specific action?
Sending a newsletter: the admin submits, you say "sending in progress." That's async. Sending a password reset: the user waits for "check your email" — they expect it immediately. If the user is staring at a spinner waiting for this specific thing, keep it synchronous or make the async very fast. If they're okay with "we'll take care of it," make it async.
user is watching = either sync or fast async
3
Is the operation idempotent, or can you make it idempotent?
Non-idempotent operations in async queues are risky — at-least-once delivery means duplicates happen. Sending an email twice is annoying. Charging a card twice is catastrophic. For non-idempotent operations, add idempotency guards (dedup keys, upserts, existence checks) before moving to async. If you can't add idempotency, consider keeping it synchronous.
non-idempotent + async = guaranteed duplicate bugs
4
Which queue tier does this belong in?
Critical (confirmation emails, payment retries, partner webhooks): must-process-fast, high retry count, dedicated workers. Default (PDF generation, internal alerts): should process within minutes. Bulk (newsletter, CSV export, analytics): can process over hours. Mixing bulk and critical jobs in one queue means a newsletter send can delay a booking confirmation. Separate them.
separate queues = separate failure domains
5
What happens if this job fails permanently?
Design the DLQ handling before shipping the job type. Who gets alerted? What's the remediation? For confirmation emails: alert on-call, manually resend from DLQ. For analytics events: might be acceptable to discard. For partner webhooks: alert the partner, require manual intervention. The answer determines retry count, DLQ alert threshold, and runbook.
no DLQ plan = silent failures users eventually notice
6
Set up DLQ monitoring and queue depth alerts before shipping
Alert when DLQ depth > 10, when queue depth > 1000 (backlog building), when job failure rate > 5%. These metrics are invisible without explicit instrumentation — BullMQ doesn't alert on your behalf. BullMQ Board or your APM tool can expose these. Set them up before the first production incident, not during it.
unmonitored queues = problems you find out about from users

Async job queue checklist

IDP
Idempotent workers

Every worker handles duplicate delivery safely. Dedup keys, upserts, existence checks — at-least-once delivery is guaranteed.

DLQ
DLQ monitored

Alert fires when DLQ depth > 10. Runbook exists for each job type. DLQ is not a trash can.

REF
IDs not objects

Job payloads contain IDs and minimal context. Workers fetch fresh data at processing time. No stale payload bugs.

AOF
Redis durability

Redis AOF enabled with appendfsync everysec. Queue jobs survive a Redis restart. Tested with a manual restart.

TMO
Job timeouts set

lockDuration set to longer than max legitimate job duration. Stuck jobs don't stay "active" forever.

SEP
Queues separated

Critical, default, and bulk jobs on separate queues with separate worker pools. Bulk can't delay critical.
