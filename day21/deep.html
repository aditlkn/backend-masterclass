Deep Dive 01

Complete BullMQ three-tier setup:
typed queues, workers, error handling

A production BullMQ setup has more moving parts than the basic examples suggest: typed job payloads, the UnrecoverableError pattern for DLQ routing, graceful shutdown to avoid mid-job crashes on deploy, and Redis configured for durability. This is the full setup.

queues.ts — typed queue definitions, three-tier architectureTYPESCRIPT
import { Queue, QueueEvents, UnrecoverableError } from 'bullmq';
import Redis from 'ioredis';

// Redis connection — shared across all queues
export const redis = new Redis({
  host: process.env.REDIS_HOST || 'localhost',
  port: 6379,
  maxRetriesPerRequest: null,  // required by BullMQ
  enableReadyCheck: false,       // required by BullMQ
});

// Typed job payload interfaces
interface BookingConfirmationJob {
  bookingId: string;
  userId:    string;
  // Deliberately minimal: worker fetches fresh data using these IDs
}

interface TicketPDFJob {
  bookingId:       string;
  templateVersion: string;
}

interface PartnerWebhookJob {
  partnerId:       string;
  event:           string;
  idempotencyKey:  string;
  payload:         Record<string, unknown>;
}

interface NewsletterJob {
  campaignId: string;
  userIds:    string[];  // batch of 100 users per job
}

// CRITICAL queue — confirmation emails, payment retries, SLA webhooks
export const criticalQueue = new Queue<BookingConfirmationJob | PartnerWebhookJob>(
  'critical',
  {
    connection: redis,
    defaultJobOptions: {
      attempts:  5,
      backoff: { type: 'exponential', delay: 1000 },
      removeOnComplete: { count: 5000, age: 3600 },  // keep 1h or 5000
      removeOnFail: false,  // keep all failed — inspect them
    },
  }
);

// DEFAULT queue — PDFs, internal notifications, cache invalidation
export const defaultQueue = new Queue<TicketPDFJob>(
  'default',
  {
    connection: redis,
    defaultJobOptions: {
      attempts: 3,
      backoff: { type: 'exponential', delay: 2000 },
      removeOnComplete: { count: 1000 },
      removeOnFail: false,
    },
  }
);

// BULK queue — newsletters, analytics, CSV exports
export const bulkQueue = new Queue<NewsletterJob>(
  'bulk',
  {
    connection: redis,
    defaultJobOptions: {
      attempts: 2,
      backoff: { type: 'fixed', delay: 30000 },    // 30s between retries
      removeOnComplete: { count: 100 },
      removeOnFail: { count: 500 },  // keep 500 for inspection, then drop
    },
  }
);

// ── Email worker with full error handling and graceful shutdown ──
import { Worker } from 'bullmq';

const emailWorker = new Worker(
  'critical',
  async (job) => {
    if (job.name !== 'booking-confirmation') return;  // only handle this type
    const { userId, bookingId } = job.data as BookingConfirmationJob;

    // Idempotency: claim before any work
    const claimed = await redis.set(`email:sent:\${job.id}`, '1', 'EX', 86400, 'NX');
    if (!claimed) return { skipped: true };

    // Fetch fresh data (not from stale payload)
    const [user, booking] = await Promise.all([
      db.findUser(userId),
      db.findBooking(bookingId),
    ]);

    // UnrecoverableError → goes to DLQ immediately, no retries
    if (!user)    throw new UnrecoverableError(`User \${userId} not found`);
    if (!booking) throw new UnrecoverableError(`Booking \${bookingId} not found`);

    try {
      await sendgrid.send({ to: user.email, ...buildEmailTemplate(booking) });
      logger.info({ jobId: job.id, bookingId, userId }, 'email.confirmation.sent');
    } catch (err: any) {
      await redis.del(`email:sent:\${job.id}`);  // release idempotency key on failure
      if (err.code === 'INVALID_EMAIL_ADDRESS') {
        throw new UnrecoverableError(`Invalid email: \${user.email}`);
      }
      throw err;  // retryable: network, rate limit, etc.
    }
  },
  {
    connection: redis,
    concurrency: 20,         // 20 concurrent email sends per worker
    lockDuration: 60_000,    // 60s lock — email send is fast
    limiter: { max: 200, duration: 1000 },  // 200 emails/sec global
  }
);

// Graceful shutdown — finish in-flight jobs before exiting
async function gracefulShutdown() {
  logger.info('SIGTERM received. Draining worker...');
  await emailWorker.close();  // finish active jobs, don't pick new ones
  await redis.quit();
  process.exit(0);
}
process.on('SIGTERM', gracefulShutdown);
process.on('SIGINT', gracefulShutdown);
Verify BullMQ setup
✓Jobs survive Redis restart: enable AOF (appendonly yes in redis.conf), add a job, restart Redis, verify job is still in the queue.
✓Idempotency works: manually run the same job twice (using job.retry()); second run should return { skipped: true }.
✓UnrecoverableError goes straight to DLQ: trigger an invalid email error; verify job appears in failed state without reaching max attempts.
✓Graceful shutdown: send SIGTERM to worker process during active job; job should complete before process exits (not killed mid-send).
Deep Dive 02

KEDA autoscaling:
scale workers on queue depth

KEDA (Kubernetes Event-Driven Autoscaling) extends Kubernetes HPA to scale pods based on external metrics — including Redis list length and BullMQ queue depth. This means your worker deployment automatically scales from 1 pod at idle to 20 pods during a ticket rush, and back down when the queue drains.

keda-scaledobject.yaml — autoscale email workers on queue depthYAML
# Install KEDA first: helm install keda kedacore/keda

apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: critical-worker-scaler
  namespace: production
spec:
  scaleTargetRef:
    name: critical-worker          # name of the Deployment to scale
  minReplicaCount: 1               # always at least 1 worker running
  maxReplicaCount: 20              # max 20 worker pods
  pollingInterval: 10              # check queue depth every 10 seconds
  cooldownPeriod: 60               # wait 60s before scaling down
  triggers:
    - type: redis
      metadata:
        address: redis.production.svc.cluster.local:6379
        listName: bull:critical:wait  # BullMQ waiting list key
        listLength: "50"             # scale up when >50 jobs waiting
        # Each pod handles 50 jobs (concurrency:20 × ~2.5s average = 50 jobs/pod/s)

---

# Separate ScaledObject for bulk workers
apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: bulk-worker-scaler
  namespace: production
spec:
  scaleTargetRef:
    name: bulk-worker
  minReplicaCount: 0               # can scale to zero! (no idle bulk workers)
  maxReplicaCount: 5
  pollingInterval: 30              # check every 30s (bulk is less time-sensitive)
  cooldownPeriod: 300              # wait 5 min before scaling down
  triggers:
    - type: redis
      metadata:
        address: redis.production.svc.cluster.local:6379
        listName: bull:bulk:wait
        listLength: "100"          # higher threshold — bulk can queue longer

---

# Worker Deployment (what KEDA scales)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: critical-worker
  namespace: production
spec:
  replicas: 1                      # KEDA overrides this
  selector:
    matchLabels:
      app: critical-worker
  template:
    metadata:
      labels:
        app: critical-worker
    spec:
      terminationGracePeriodSeconds: 120  # allow 2min for job to finish
      containers:
        - name: worker
          image: your-registry/workers:latest
          command: ["node", "dist/workers/critical.js"]
          env:
            - name: REDIS_HOST
              valueFrom:
                secretKeyRef: { name: redis-secret, key: host }
          resources:
            requests: { memory: "256Mi", cpu: "100m" }
            limits:   { memory: "512Mi", cpu: "500m" }
Deep Dive 03

Queue monitoring:
BullMQ Board + custom metrics

BullMQ Board is the built-in dashboard for queue visibility. Beyond that, exposing queue metrics to Prometheus and building Grafana dashboards gives you alertable, historical queue health data.

queue-metrics.ts — expose BullMQ metrics to PrometheusTYPESCRIPT
import { Queue } from 'bullmq';
import { Registry, Gauge } from 'prom-client';

const registry = new Registry();

// Gauge per queue × state
const queueDepthGauge = new Gauge({
  name:    'bullmq_queue_depth',
  help:    'Number of jobs in each state per queue',
  labelNames: ['queue', 'state'],
  registers: [registry],
});

const queues = [criticalQueue, defaultQueue, bulkQueue];
const queueNames = ['critical', 'default', 'bulk'];

async function collectMetrics() {
  for (const [i, queue] of queues.entries()) {
    const counts = await queue.getJobCounts();
    const name = queueNames[i];
    for (const [state, count] of Object.entries(counts)) {
      queueDepthGauge.set({ queue: name, state }, count);
    }
  }
}

// Collect every 15 seconds
setInterval(collectMetrics, 15_000);

// Prometheus scrape endpoint
app.get('/metrics', async (req, res) => {
  res.set('Content-Type', registry.contentType);
  res.end(await registry.metrics());
});

## ── Grafana dashboard queries (PromQL) ──────────────────────────

## Panel 1: DLQ depth over time (alert threshold: 10)
## bullmq_queue_depth{state="failed"}

## Panel 2: Queue throughput (jobs/min)
## rate(bullmq_queue_depth{state="completed"}[1m]) * 60

## Panel 3: Active workers (vs jobs waiting)
## bullmq_queue_depth{state="active"}
## bullmq_queue_depth{state="waiting"}

## Panel 4: Queue drain time estimate (waiting / throughput)
## bullmq_queue_depth{state="waiting"}
##   /
## rate(bullmq_queue_depth{state="completed"}[5m]) / 60

## ── Grafana alert rules ─────────────────────────────────────────

## Alert 1: DLQ growing (critical — page on-call)
## WHEN bullmq_queue_depth{state="failed"} > 10 FOR 5m
## MESSAGE: "{{$labels.queue}} queue has {{$value}} failed jobs"

## Alert 2: Queue backlog (warning — check within 30 min)
## WHEN bullmq_queue_depth{state="waiting",queue="critical"} > 1000 FOR 10m

## Alert 3: No workers processing (critical — workers may be down)
## WHEN bullmq_queue_depth{state="waiting",queue="critical"} > 10
##   AND bullmq_queue_depth{state="active",queue="critical"} == 0
##   FOR 2m
