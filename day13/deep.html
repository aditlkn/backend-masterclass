Deep Dive 01

WireGuard VPN:
developer access to private subnets

WireGuard is the fastest, simplest VPN protocol available. Installing it on a small EC2 instance in your public subnet gives developers SSH and database access to private resources without exposing anything publicly.

WireGuard server setup — EC2 in public subnetBASH
# ── On the WireGuard EC2 (Ubuntu 22.04, public subnet) ─────────

# 1. Install WireGuard
apt update && apt install -y wireguard

# 2. Generate server keypair
wg genkey | tee /etc/wireguard/server_private.key | wg pubkey > /etc/wireguard/server_public.key
chmod 600 /etc/wireguard/server_private.key
cat /etc/wireguard/server_public.key    # share this with clients

# 3. Enable IP forwarding (required to route traffic)
echo 'net.ipv4.ip_forward=1' >> /etc/sysctl.conf
sysctl -p

# 4. Create server config (/etc/wireguard/wg0.conf)
cat > /etc/wireguard/wg0.conf << 'EOF'
[Interface]
Address    = 10.100.0.1/24          # VPN subnet (different from VPC CIDR)
ListenPort = 51820
PrivateKey = <server_private_key>

# Forward VPN traffic into the VPC
PostUp   = iptables -A FORWARD -i wg0 -j ACCEPT; \
           iptables -A FORWARD -o wg0 -j ACCEPT; \
           iptables -t nat -A POSTROUTING -o eth0 -j MASQUERADE
PostDown = iptables -D FORWARD -i wg0 -j ACCEPT; \
           iptables -D FORWARD -o wg0 -j ACCEPT; \
           iptables -t nat -D POSTROUTING -o eth0 -j MASQUERADE

# Add each developer as a [Peer] block
[Peer]
# Developer: Adnan
PublicKey  = <adnan_public_key>
AllowedIPs = 10.100.0.2/32           # Adnan gets this VPN IP

[Peer]
# Developer: Backend team
PublicKey  = <team_public_key>
AllowedIPs = 10.100.0.3/32
EOF

# 5. Start WireGuard
systemctl enable wg-quick@wg0
systemctl start wg-quick@wg0
wg show                               # verify interface is up

# 6. Security group for WireGuard EC2
# Inbound: UDP 51820 from 0.0.0.0/0 (VPN handshake)
# Inbound: SSH 22 from your office IP only (for administration)
# Outbound: all to 10.0.0.0/8 (the VPC) — for routing VPN traffic

# 7. VPC route table: add route so VPN clients can reach private subnets
# In private subnet route table: 10.100.0.0/24 → wg-instance-id
# (Add a route pointing to the WireGuard EC2 instance)
WireGuard client config — developer laptop (split tunneling)INI
# /etc/wireguard/wg0.conf (on developer laptop)
# Or use a GUI client: Wireguard app on macOS/Windows

[Interface]
PrivateKey = <developer_private_key>
Address    = 10.100.0.2/32           # this developer's VPN IP
DNS        = 10.0.0.2                # VPC DNS resolver (for .internal names)

[Peer]
PublicKey  = <server_public_key>
Endpoint   = 54.239.28.85:51820      # WireGuard EC2 public IP:port

# SPLIT TUNNELING: only route VPC traffic through VPN
# YouTube, GitHub, Slack go directly (not through VPN)
AllowedIPs = 10.0.0.0/8,            # all RFC 1918 10.x.x.x (your VPC)
             10.100.0.0/24           # VPN subnet itself

# FULL TUNNEL (all traffic through VPN — simpler but slower):
# AllowedIPs = 0.0.0.0/0

PersistentKeepalive = 25             # send keepalive every 25s (keeps NAT table alive)

# After connecting, developer can:
# ssh ec2-user@10.0.10.50          # SSH to private EC2 directly
# psql -h 10.0.20.100 -U app mydb  # connect to RDS private IP
# curl http://10.0.10.80:8080/health  # hit internal services directly
# dig payment-svc.prod.internal     # resolves via VPC DNS
Verify WireGuard is working
✓Server: wg show shows the interface and connected peers with latest handshake timestamps.
✓Client: ping 10.0.10.1 reaches the VPC gateway (proves routing works).
✓Split tunneling: curl ifconfig.me returns your actual ISP IP (not the VPN server's), confirming internet traffic doesn't go through VPN.
✓Internal DNS: dig payment-svc.prod.internal resolves to a private IP when connected (uses VPC DNS at 10.0.0.2).
Deep Dive 02

Kong declarative config:
full kong.yaml with deck

The Kong Admin API is great for exploration. For production, you want declarative config stored in git — kong.yaml managed by deck (the declarative config tool). Changes go through code review, diffs are previewed before applying, and your entire gateway config is version-controlled.

kong.yaml — complete production config for a microservices gatewayYAML
# kong.yaml — managed by deck, stored in git
_format_version: "3.0"
_transform: true

## ── Upstreams (backend server pools with health checks) ──────────
upstreams:
  - name: payment-upstream
    algorithm: round-robin
    healthchecks:
      active:
        http_path: /health
        interval: 10
        healthy:
          successes: 2
          http_statuses: [200]
        unhealthy:
          http_failures: 3
          timeouts: 3
          interval: 5
    targets:
      - target: 10.0.10.50:8080
        weight: 100
      - target: 10.0.10.51:8080
        weight: 100

  - name: order-upstream
    algorithm: least-connections    # prefer less-loaded backends
    healthchecks:
      active:
        http_path: /health
        interval: 10
        healthy: { successes: 2, http_statuses: [200] }
        unhealthy: { http_failures: 3, timeouts: 3 }
    targets:
      - target: 10.0.10.60:8080
        weight: 100

## ── Services (logical backends) ─────────────────────────────────
services:
  - name: payment-service
    host: payment-upstream
    protocol: http
    port: 80
    connect_timeout: 5000    # 5s to establish connection
    write_timeout: 30000     # 30s to send full request
    read_timeout: 30000      # 30s to receive full response
    retries: 2               # retry failed requests twice
    routes:
      - name: payment-route
        paths: [/api/v1/payments]
        methods: [GET, POST]
        strip_path: false
        protocols: [https]
        plugins:
          - name: jwt
            config:
              key_claim_name: kid
              secret_is_base64: false
          - name: rate-limiting
            config:
              minute: 1000
              hour: 10000
              policy: redis
              redis_host: redis.prod.internal
              redis_port: 6379
          - name: correlation-id
            config:
              header_name: X-Request-ID
              generator: uuid

  - name: order-service
    host: order-upstream
    protocol: http
    port: 80
    connect_timeout: 5000
    write_timeout: 60000     # orders can take longer to process
    read_timeout: 60000
    routes:
      - name: order-route
        paths: [/api/v1/orders]
        methods: [GET, POST, PUT]
        strip_path: false
        plugins:
          - name: jwt
          - name: rate-limiting
            config:
              minute: 500
              policy: redis
              redis_host: redis.prod.internal

## ── Global plugins (apply to all services) ───────────────────────
plugins:
  - name: prometheus
    config:
      per_consumer: true
      status_code_metrics: true
      latency_metrics: true

  - name: request-termination
    enabled: false           # disabled by default, enable for maintenance mode
    config:
      status_code: 503
      message: "Service temporarily unavailable for maintenance"

## ── Consumers (identified callers with custom limits) ────────────
consumers:
  - username: internal-order-svc
    jwt_secrets:
      - key: order-svc-key
        secret: ${ORDER_SVC_JWT_SECRET}    # deck reads from env
    plugins:
      - name: rate-limiting
        config:
          minute: 50000      # internal services get much higher limits
          policy: redis
          redis_host: redis.prod.internal

  - username: mobile-app-ios
    jwt_secrets:
      - key: mobile-ios-key
        secret: ${MOBILE_IOS_JWT_SECRET}
    plugins:
      - name: rate-limiting
        config:
          minute: 200        # per mobile client limit
          policy: redis
          redis_host: redis.prod.internal
deck workflow — diff, sync, backupBASH
# Install deck
brew install deck                     # macOS
# or: curl -sL https://github.com/Kong/deck/releases/latest/.../deck_linux_amd64.tar.gz | tar -xz

# Preview changes before applying (always do this first)
deck diff --kong-addr http://kong-admin:8001 --state kong.yaml
# Output shows: + added, ~ modified, - removed objects
# Review carefully — accidental deletions are possible

# Apply changes
deck sync --kong-addr http://kong-admin:8001 --state kong.yaml

# Export current Kong state to file (for initial migration)
deck dump --kong-addr http://kong-admin:8001 --output-file kong-current.yaml

# Validate yaml syntax before pushing to git
deck validate --state kong.yaml

# CI/CD pipeline (GitHub Actions)
# on: pull_request
#   - deck validate --state kong.yaml
#   - deck diff --kong-addr $KONG_STAGING --state kong.yaml
# on: push to main
#   - deck sync --kong-addr $KONG_PROD --state kong.yaml

# Secrets: never put secrets in kong.yaml directly
# deck reads from environment: ${ORDER_SVC_JWT_SECRET}
# Set env vars in your CI/CD secrets store
Deep Dive 03

AWS VPC in Terraform:
the complete production setup

This is the Terraform for a production-ready VPC: two AZs, public and private subnets, NAT gateway, S3 VPC endpoint, and security groups using reference-based rules (not CIDR ranges). Copy-paste with your CIDR choices.

vpc.tf — complete production VPCTERRAFORM
# vpc.tf
variable "vpc_cidr" { default = "10.3.0.0/16" }   # prod VPC range
variable "env"      { default = "prod" }

## ── VPC ───────────────────────────────────────────────────────────
resource "aws_vpc" "main" {
  cidr_block           = var.vpc_cidr
  enable_dns_support   = true    # needed for VPC endpoints and RDS hostnames
  enable_dns_hostnames = true    # gives EC2 instances internal DNS hostnames
  tags = { Name = "${var.env}-vpc" }
}

## ── Internet Gateway ───────────────────────────────────────────────
resource "aws_internet_gateway" "igw" {
  vpc_id = aws_vpc.main.id
  tags   = { Name = "${var.env}-igw" }
}

## ── Public Subnets (one per AZ) ────────────────────────────────────
resource "aws_subnet" "public" {
  for_each          = { "a" = "10.3.1.0/24", "b" = "10.3.2.0/24" }
  vpc_id            = aws_vpc.main.id
  cidr_block        = each.value
  availability_zone = "us-east-1${each.key}"
  map_public_ip_on_launch = false  # never auto-assign public IPs
  tags = { Name = "${var.env}-public-${each.key}", Tier = "public" }
}

## ── Private Subnets ────────────────────────────────────────────────
resource "aws_subnet" "private" {
  for_each          = { "a" = "10.3.10.0/24", "b" = "10.3.20.0/24" }
  vpc_id            = aws_vpc.main.id
  cidr_block        = each.value
  availability_zone = "us-east-1${each.key}"
  tags = { Name = "${var.env}-private-${each.key}", Tier = "private" }
}

## ── NAT Gateway (one per AZ for HA) ───────────────────────────────
resource "aws_eip" "nat" {
  for_each = aws_subnet.public
  domain   = "vpc"
}

resource "aws_nat_gateway" "ngw" {
  for_each      = aws_subnet.public
  subnet_id     = each.value.id
  allocation_id = aws_eip.nat[each.key].id
  tags          = { Name = "${var.env}-nat-${each.key}" }
}

## ── Route Tables ────────────────────────────────────────────────────
# Public: 0.0.0.0/0 → IGW
resource "aws_route_table" "public" {
  vpc_id = aws_vpc.main.id
  route {
    cidr_block = "0.0.0.0/0"
    gateway_id = aws_internet_gateway.igw.id
  }
  tags = { Name = "${var.env}-public-rt" }
}

resource "aws_route_table_association" "public" {
  for_each       = aws_subnet.public
  subnet_id      = each.value.id
  route_table_id = aws_route_table.public.id
}

# Private: 0.0.0.0/0 → NAT Gateway (per AZ)
resource "aws_route_table" "private" {
  for_each = aws_subnet.private
  vpc_id   = aws_vpc.main.id
  route {
    cidr_block     = "0.0.0.0/0"
    nat_gateway_id = aws_nat_gateway.ngw[each.key].id
  }
  tags = { Name = "${var.env}-private-rt-${each.key}" }
}

resource "aws_route_table_association" "private" {
  for_each       = aws_subnet.private
  subnet_id      = each.value.id
  route_table_id = aws_route_table.private[each.key].id
}

## ── S3 Gateway Endpoint (free — eliminates NAT for S3) ─────────────
resource "aws_vpc_endpoint" "s3" {
  vpc_id            = aws_vpc.main.id
  service_name      = "com.amazonaws.us-east-1.s3"
  vpc_endpoint_type = "Gateway"
  route_table_ids   = concat(
    [aws_route_table.public.id],
    [for rt in aws_route_table.private : rt.id]
  )
  tags = { Name = "${var.env}-s3-endpoint" }
}

## ── Security Groups ─────────────────────────────────────────────────
# ALB security group: internet → ALB
resource "aws_security_group" "alb" {
  name   = "${var.env}-alb-sg"
  vpc_id = aws_vpc.main.id
  ingress {
    from_port   = 443; to_port = 443; protocol = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }
  ingress {
    from_port   = 80; to_port = 80; protocol = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }
  egress {
    from_port = 0; to_port = 0; protocol = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }
}

# App security group: ALB → app (referenced by SG, not CIDR!)
resource "aws_security_group" "app" {
  name   = "${var.env}-app-sg"
  vpc_id = aws_vpc.main.id
  ingress {
    from_port       = 3000; to_port = 3000; protocol = "tcp"
    security_groups = [aws_security_group.alb.id]  # only from ALB SG
  }
  egress {
    from_port = 0; to_port = 0; protocol = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }
}

# RDS security group: app → RDS only
resource "aws_security_group" "rds" {
  name   = "${var.env}-rds-sg"
  vpc_id = aws_vpc.main.id
  ingress {
    from_port       = 5432; to_port = 5432; protocol = "tcp"
    security_groups = [aws_security_group.app.id]  # only from app SG
  }
  # No egress needed for RDS — RDS doesn't initiate outbound connections
}
Verify VPC setup is correct
✓Public subnets route to IGW: aws ec2 describe-route-tables --filters "Name=tag:Tier,Values=public" — should show 0.0.0.0/0 → igw-xxx.
✓Private subnets route to NAT GW: same command for private — should show 0.0.0.0/0 → nat-xxx (not igw).
✓S3 endpoint active: aws ec2 describe-vpc-endpoints --filters "Name=service-name,Values=*s3*" — state should be "available".
✓RDS only reachable from app SG: attempt connection from a host NOT in the app SG — should timeout (connection refused at network level).
✓No overlapping CIDRs with other VPCs: aws ec2 describe-vpcs --query "Vpcs[].CidrBlock" — all VPCs should have unique non-overlapping ranges.
