Section 01

Private vs public IP addresses: why the distinction exists

IPv4 has approximately 4.3 billion addresses. The internet has far more than 4.3 billion devices. This was known to be a problem in the early 1990s, and the fix — before IPv6 was ready — was to designate certain IP ranges as "private": usable by anyone on their internal network, but not routable on the public internet.

The three private ranges (RFC 1918)

These IP ranges are reserved for private use. Every router on the public internet is configured to drop packets destined for these addresses — they simply don't route:

  • 10.0.0.0/8 — 16,777,216 addresses. The most common in cloud environments. Your AWS VPC is probably in this range.
  • 172.16.0.0/12 — 1,048,576 addresses. Docker's default bridge network uses 172.17.0.0/16 from this range.
  • 192.168.0.0/16 — 65,536 addresses. Your home router almost certainly assigns addresses here.

The implication: thousands of companies can all use 10.0.0.1 internally without conflict, because these addresses never appear on the public internet. Your laptop at 192.168.1.100 and your company's VPN endpoint at 192.168.1.100 don't collide — they're in separate networks.

Public IP addresses

Everything outside the RFC 1918 ranges is "public" — globally unique, routable on the internet. When you buy a domain and point it at your server, you're pointing it at a public IP. Your server itself almost never has a public IP directly assigned to its network interface — instead, a load balancer or NAT gateway has the public IP and forwards traffic to the server's private IP.

Special addresses to know

  • 127.0.0.1 — loopback (localhost). Traffic never leaves the machine.
  • 169.254.x.x — link-local. AWS uses 169.254.169.254 as the instance metadata endpoint — reachable from any EC2 instance without routing through anything.
  • 0.0.0.0 — "any address" — used in routing tables and server bind configs to mean "all interfaces."

Section 02

Subnets and CIDR: slicing up an IP range

CIDR (Classless Inter-Domain Routing) notation describes a range of IP addresses. The format is IP/prefix-length where the prefix length tells you how many bits are fixed (the network part) and how many are variable (the host part). 10.0.0.0/24 means the first 24 bits are fixed (10.0.0) and the last 8 bits vary — giving you 256 addresses (10.0.0.0 through 10.0.0.255).

The math

Available host addresses = 2^(32 - prefix). Subtract 2 for the network address (first) and broadcast address (last):

  • /32 — exactly 1 address. Used for a specific host in routing rules.
  • /30 — 4 addresses, 2 usable. Point-to-point links.
  • /24 — 256 addresses, 254 usable. A typical small subnet.
  • /16 — 65,536 addresses. A VPC or large network segment.
  • /8 — 16,777,216 addresses. The entire 10.x.x.x range.

Subnets in a VPC

A VPC starts with a CIDR block (e.g. 10.0.0.0/16 — 65,536 addresses). You carve it into subnets. A typical two-tier setup: public subnets for load balancers and NAT gateways (10.0.1.0/24, 10.0.2.0/24) and private subnets for application servers and databases (10.0.10.0/24, 10.0.20.0/24). Each subnet lives in one Availability Zone — this is why you need multiple subnets for high availability.

⚠️
Overlapping CIDRs break VPC peering

If two VPCs have overlapping CIDR blocks (both use 10.0.0.0/16), you cannot peer them — the routing is ambiguous. Plan your VPC CIDR blocks before creating them. A common pattern: dev uses 10.1.0.0/16, staging uses 10.2.0.0/16, prod uses 10.3.0.0/16. This lets you peer them freely for cross-environment tools.

Section 03

NAT: how private IPs reach the internet

Network Address Translation solves the fundamental problem: a server with a private IP (10.0.1.50) wants to call an external API (api.stripe.com). The internet can't route a response back to 10.0.1.50 — that address is meaningless outside your network. NAT is the mechanism that makes this work.

How NAT works (SNAT — Source NAT)

When a packet leaves your private subnet toward the internet, the NAT gateway intercepts it. It replaces the source IP (10.0.1.50) with its own public IP (54.239.28.85) and assigns a port number to track the connection. When the response arrives (src: api.stripe.com, dst: 54.239.28.85:41234), the NAT gateway looks up its translation table, replaces the destination with the original private IP and port, and forwards it back to the server.

The server at 10.0.1.50 sees the response as if it came directly. The internet sees all outbound traffic from your entire private subnet appearing to come from a single public IP — the NAT gateway's IP.

DNAT — Destination NAT (how inbound traffic works)

The reverse: a user hits your load balancer's public IP. The load balancer (or a DNAT rule) replaces the destination IP with the private IP of a backend server. The packet arrives at the backend with the load balancer's private IP as the source — which is why your application logs show internal IPs unless you explicitly forward the original client IP via X-Forwarded-For.

The AWS NAT Gateway timeout problem

AWS NAT Gateway is a managed NAT service. It maintains a connection tracking table — for each active connection through NAT, it remembers the port mapping. If a connection is idle for 350 seconds (about 6 minutes), AWS NAT Gateway drops the entry. From the client's perspective, the connection appears alive. The next packet through that connection gets a TCP RST. This is the exact cause of the Day 2 lab incident — database connections idle through a NAT Gateway get silently killed. Fix: TCP keepalives at 60s (Day 2), or use PrivateLink/VPC peering so DB traffic never goes through NAT.

Section 04

VPC architecture: the canonical layout

A VPC (Virtual Private Cloud) is a logically isolated network within AWS. You define its IP range (CIDR block), create subnets within it, and control traffic with route tables, security groups, and network ACLs. Understanding the standard architecture explains why your infrastructure is set up the way it is.

The canonical two-tier layout

Public subnets have a route to an Internet Gateway (IGW) — resources here can receive inbound traffic from the internet and initiate outbound traffic directly. This is where your load balancers and NAT gateways live.

Private subnets have no route to the IGW — resources here cannot receive unsolicited inbound traffic from the internet. For outbound internet access (calling external APIs, pulling packages), they route through the NAT gateway in the public subnet. This is where your application servers and databases live.

Security groups vs Network ACLs

Security groups are stateful firewalls attached to individual resources (EC2 instances, RDS, load balancers). "Stateful" means: if you allow outbound traffic to port 443, the response is automatically allowed back in, even without an explicit inbound rule. Security groups operate at the instance level. This is your primary traffic control tool.

Network ACLs (NACLs) are stateless firewalls attached to subnets. Stateless means you must explicitly allow both directions. They're coarser — they apply to everything in the subnet. Most teams rely on security groups and rarely touch NACLs.

The traffic path in — step by step

User request → DNS resolves to load balancer's public IP → Internet Gateway passes traffic to the load balancer (in public subnet) → security group on LB allows port 443 → LB forwards to app server's private IP (in private subnet) → security group on app server allows only traffic from the LB's security group → app server processes, calls database using database's private IP → RDS security group allows only traffic from the app server's security group.

Section 05

The four traffic paths every backend engineer encounters

Every network call your system makes falls into one of four patterns. Understanding which pattern applies determines the latency, cost, security posture, and failure modes of that call.

Path 1: Public ingress (internet → your service)

User → DNS → public IP → IGW → Load Balancer (public subnet) → App Server (private subnet). This is the only path that should ever accept unsolicited traffic from the internet. Everything else should be private.

Characteristics: Goes through IGW, LB, two security group hops. TLS terminates at the LB. App server never sees the public internet directly. Latency: whatever the user's internet + your region + LB overhead.

Path 2: Internal service-to-service (same VPC)

Service A (10.0.10.50) → Service B (10.0.10.80). Direct private IP routing, entirely within the VPC. Never touches NAT, never leaves AWS's network, never hits the internet.

Characteristics: Fastest possible path. Free (no data transfer costs within AZ, $0.01/GB cross-AZ). Sub-millisecond within the same AZ. Uses internal DNS — payment-svc.internal resolves to 10.0.10.80. This is the correct path for all microservice calls.

Path 3: Cross-VPC (VPC peering or Transit Gateway)

Service in VPC-A → Service in VPC-B. Requires VPC peering or Transit Gateway. Once peered, traffic flows via private IPs between VPCs — still within AWS, never on the public internet.

Characteristics: Requires explicit routing setup. CIDRs must not overlap. Same latency profile as internal once peered. Use case: separate VPCs for prod/staging, or different teams owning different VPCs. Transit Gateway is more scalable than peering for hub-and-spoke topologies (many VPCs).

Path 4: External API call (your service → internet)

Service (private subnet) → NAT Gateway (public subnet) → Internet Gateway → Public internet → api.stripe.com. Traffic leaves AWS's network and traverses the public internet.

Characteristics: Slowest. Costs money (NAT Gateway data processing charges). Subject to NAT timeout issues. 350s idle timeout kills long-lived connections. Mitigate: connection pooling, keep-alives, or use AWS PrivateLink for AWS-hosted services.

Section 06

Internal service-to-service communication

When service A calls service B inside your VPC, the traffic path and discovery mechanism depend on how you've set things up. There are three common patterns, each with different characteristics:

Direct private IP

Hardcode the private IP of the destination service. Simple, fast, zero overhead. The problem: private IPs can change if an instance is replaced or the service scales. Almost never the right answer for services — use DNS instead.

Internal DNS (Route 53 private hosted zone or service discovery)

Service B registers as payment-svc.prod.internal. Service A calls that DNS name. The DNS resolver (Route 53 Resolver in AWS, running at 169.254.169.253) returns the current private IP(s). When instances are replaced, DNS updates. This is the standard pattern.

In ECS/Kubernetes: service discovery is automatic. ECS Service Connect or AWS Cloud Map registers task IPs. Kubernetes gives every Service a stable internal DNS name: payment-svc.default.svc.cluster.local.

Internal load balancer

An internal (non-internet-facing) ALB sits in front of the service. Internal ALB has a private IP, not reachable from the internet. Service A calls the ALB's DNS name, ALB load balances across instances of service B. This adds an extra hop but provides: health checking, load balancing, TLS termination, and connection pooling at the network level.

Service discovery patterns — DNS-based internal routingBASH
# What your service sees internally

# Public DNS (external): api.yourcompany.com → 54.239.28.85 (load balancer public IP)
$ dig api.yourcompany.com
54.239.28.85

# Internal service DNS: payment-svc.prod.internal → 10.0.10.80 (direct private IP)
$ dig payment-svc.prod.internal
10.0.10.80

# AWS Cloud Map: auto-registers ECS task IPs
$ dig payment-svc.prod.local
10.0.10.80    # instance 1
10.0.11.95    # instance 2 (cross-AZ)

# Kubernetes internal DNS
$ curl http://payment-svc.payments-namespace.svc.cluster.local:8080/health
# Resolves to ClusterIP (virtual IP), kube-proxy routes to actual pod

# AWS RDS: always use the endpoint DNS, never hardcode IP
# mydb.cluster-xyz.us-east-1.rds.amazonaws.com → 10.0.20.100
# If instance is replaced/failover, DNS updates — hardcoded IP would break

# Internal ALB DNS (private, not internet-facing)
# internal-payment-alb-1234.us-east-1.elb.amazonaws.com → 10.0.1.25

# Never call internal services via their public-facing hostname
# Bad:  fetch('https://api.yourcompany.com/payment/charge')  ← goes out to internet and back
# Good: fetch('http://payment-svc.prod.internal:8080/charge') ← stays internal

Section 07

VPN: private network over public internet

A VPN (Virtual Private Network) creates an encrypted tunnel over the public internet, allowing devices or networks to communicate as if they were directly connected on the same private network. It doesn't change the underlying routing — packets still travel over the public internet — but they're encrypted and encapsulated so the contents are private.

Two types you'll encounter as a backend engineer

Site-to-site VPN: Connects two entire networks — e.g. your on-premises office network to your AWS VPC. Once established, all resources on both sides can reach each other via private IPs, as if the networks were merged. AWS Site-to-Site VPN is the standard tool for this. Used when you need to access AWS resources from your own data centre, or extend your corporate network into the cloud.

Client VPN: Connects individual devices (developer laptops, remote workers) to a private network. When your developers need to SSH into EC2 instances in a private subnet, they connect via VPN first. The VPN gives their laptop a private IP in your VPC's range, and from there they can reach private resources directly.

How WireGuard works (the modern VPN protocol)

WireGuard is the modern, fast, simpler alternative to OpenVPN and IPsec. It runs in the Linux kernel, uses state-of-the-art cryptography (Curve25519, ChaCha20, Poly1305), and has ~4000 lines of code vs ~400,000 for OpenVPN. Each peer has a public/private key pair. To add a peer, you add their public key to your config. Packets are encrypted with the recipient's public key and decrypted with their private key. No certificates, no CA — just keys.

Split tunneling

By default, a VPN routes all traffic through the tunnel — including your Netflix stream. Split tunneling routes only specific traffic through the VPN (e.g. your company's 10.0.0.0/8 subnets) and sends everything else directly. Better for performance, but means company-bound traffic is the only traffic encrypted through the tunnel. Most corporate VPNs for developers use split tunneling.

💡
VPN vs VPC Peering vs PrivateLink

These three solve different problems. VPN: encrypted tunnel over public internet, used for connecting external networks (office, developer laptops) to your VPC. VPC Peering: direct private routing between two VPCs within AWS, no internet involved — fastest, but requires non-overlapping CIDRs and has no transitive routing. AWS PrivateLink: exposes a specific service (not a whole VPC) to other VPCs or AWS accounts, with a private endpoint in the consumer's VPC — used for SaaS products to offer private connectivity to customers without exposing the whole network.

Section 08

Kong Gateway: the public ingress layer in practice

Kong sits at the boundary between public traffic and your internal services — exactly at the transition from public subnet to private subnet in the VPC architecture above. Understanding Kong's object model explains how it routes incoming requests to the right internal service.

The Kong object model

Everything in Kong is built from five objects:

  • Service: Represents an upstream service — your internal API. Points to the internal DNS name or private IP: host: payment-svc.prod.internal, port: 8080. Kong forwards matched requests to this address.
  • Route: Defines what incoming traffic matches a Service. Match rules: host, path, method, headers. paths: [/api/payment] → forward to Payment Service. A Service can have many Routes.
  • Plugin: Middleware attached to a Service, Route, or globally. Rate limiting, authentication, request transformation, logging. Plugins run in a defined phase order (access, header_filter, body_filter, log).
  • Upstream: Load balancing config for a Service's backend pool. Defines health checks, load balancing algorithm, targets. One Service maps to one Upstream.
  • Target: An individual backend server in an Upstream pool. 10.0.10.50:3000, 10.0.10.51:3000. Targets have weights for weighted round-robin.

The request lifecycle

Request arrives at Kong → Route matching (which Service?) → Plugin chain runs (auth, rate limiting, transforms) → Kong forwards to Upstream → picks a Target via load balancing → response returns through Plugin chain (response transforms, logging) → back to client.

Kong object model — complete setup via Admin APIBASH
# Kong Admin API runs at :8001 (internal only — never expose to internet)
# Kong Proxy runs at :8000/:8443 (public — this is what users hit)

## 1. Create an Upstream (backend pool with health checks)
curl -X POST http://kong:8001/upstreams \
  -d name=payment-upstream \
  -d healthchecks.active.http_path=/health \
  -d healthchecks.active.interval=10 \
  -d healthchecks.active.healthy.successes=2 \
  -d healthchecks.active.unhealthy.http_failures=3

## 2. Add Targets to the Upstream
curl -X POST http://kong:8001/upstreams/payment-upstream/targets \
  -d target=10.0.10.50:8080 \
  -d weight=100

curl -X POST http://kong:8001/upstreams/payment-upstream/targets \
  -d target=10.0.10.51:8080 \
  -d weight=100

## 3. Create a Service pointing to the Upstream
curl -X POST http://kong:8001/services \
  -d name=payment-service \
  -d host=payment-upstream \   # references upstream by name
  -d protocol=http \
  -d port=80 \
  -d connect_timeout=5000 \
  -d write_timeout=30000 \
  -d read_timeout=30000

## 4. Create a Route matching incoming traffic to the Service
curl -X POST http://kong:8001/services/payment-service/routes \
  -d name=payment-route \
  -d paths[]=/api/v1/payments \
  -d methods[]=GET \
  -d methods[]=POST \
  -d strip_path=false       # keep /api/v1/payments in the upstream request

## 5. Attach Plugins to the Route
# Rate limiting
curl -X POST http://kong:8001/routes/payment-route/plugins \
  -d name=rate-limiting \
  -d config.minute=1000 \
  -d config.hour=10000 \
  -d config.policy=redis \
  -d config.redis_host=redis.prod.internal

# JWT authentication
curl -X POST http://kong:8001/routes/payment-route/plugins \
  -d name=jwt \
  -d config.secret_is_base64=false \
  -d config.key_claim_name=kid

# Request ID injection (trace correlation)
curl -X POST http://kong:8001/services/payment-service/plugins \
  -d name=correlation-id \
  -d config.header_name=X-Request-ID \
  -d config.generator=uuid

# deck diff --config kong.yaml  # preview changes

Section 09

Production reality

Your services calling each other via public hostname is a common mistake

A new service gets deployed. The developer tests it by calling https://api.yourcompany.com/payment/charge — the public URL. It works! They leave it. That call now: goes out through NAT Gateway, across the internet, back into your load balancer, through a security group, to the service. It costs money, adds latency, and goes through your internet bandwidth cap. The correct call: http://payment-svc.prod.internal:8080/charge. Add an internal DNS check to your code review checklist.

Cross-AZ traffic has a cost

AWS charges $0.01/GB for traffic crossing Availability Zones. An app server in AZ-1a calling a database in AZ-1b pays this for every byte. At high throughput this adds up. Mitigation: deploy your service and its primary data source in the same AZ. Use multi-AZ for failover, but optimise for same-AZ in normal operation.

Security group rules are not documentation

Security groups accumulate rules over time. An engineer opens port 5432 to an IP range "temporarily" and forgets. A year later, nobody knows why the rule exists or whether removing it would break something. Use Infrastructure as Code (Terraform, CDK) for security groups and review them in code review. The code is the documentation.

The VPC CIDR you can't change

AWS lets you add secondary CIDR blocks to a VPC, but you cannot change the primary CIDR. If you started with 10.0.0.0/16 and later need to peer with another VPC that also uses 10.0.0.0/16 — you're stuck. Plan CIDR blocks before creating VPCs. One team, one CIDR range, no overlaps.

Section 10

The things nobody tells you

169.254.169.254 is the AWS metadata endpoint and you should memorise it

Any EC2 instance can reach http://169.254.169.254/latest/meta-data/ without any routing setup. This returns instance metadata: instance ID, IAM role credentials, AZ, public hostname, everything. It's also a security target — SSRF attacks that can reach this endpoint can steal IAM credentials. IMDSv2 (token-based access) mitigates this. Always enforce IMDSv2 in your launch templates.

0.0.0.0/0 in a route table means "everything else goes here"

Route tables work by longest-prefix-match. 10.0.0.0/16 → local takes priority over 0.0.0.0/0 → IGW for local traffic. The default route (0.0.0.0/0) only handles traffic that doesn't match any more specific route. Public subnets have 0.0.0.0/0 → IGW. Private subnets have 0.0.0.0/0 → NAT Gateway. This is the fundamental difference between public and private subnets — one line in the route table.

Kong's Admin API must never be publicly accessible

Kong runs two ports: :8000/:8443 (proxy — the one users hit) and :8001/:8444 (admin API — the one you configure Kong with). If :8001 is exposed to the internet, anyone can add routes, plugins, and backend targets to your API gateway. Lock the admin API to a security group that only allows access from your bastion host or VPN IP range.

Split DNS: different answers inside vs outside the VPC

Route 53 private hosted zones let you return different DNS answers inside vs outside your VPC. api.yourcompany.com resolves to the load balancer's public IP from the internet, but to the load balancer's private IP (or the internal ALB directly) from within the VPC. This eliminates unnecessary NAT traversal and reduces latency for internal callers without any application code changes.

Decision sequence

Designing how service A calls service B

Work through this every time you write a URL in application code or design a new integration.

1
Is service B in the same VPC?
If yes: use its internal DNS name and private IP. Never use the public hostname. The internal path is faster, free, and doesn't leave your network. Check whether B has an internal load balancer — use that DNS name rather than hardcoding IPs.
same VPC = internal DNS, private IP, no NAT, no cost
2
Is service B in a different VPC (same company)?
Establish VPC peering or use a Transit Gateway if you have many VPCs. Once peered, add routes in both route tables and use private IPs — same as same-VPC. Verify CIDRs don't overlap before starting. For one-to-one VPC connections: peering. For five or more VPCs talking to each other: Transit Gateway.
different VPC = peering or TGW, then private IPs
3
Is it an AWS service (S3, DynamoDB, SQS)?
Use a VPC Endpoint instead of going through NAT. Gateway endpoints (S3, DynamoDB) are free and keep traffic in AWS's network. Interface endpoints (most other services) cost ~$0.01/hour but eliminate NAT costs and latency. Check your NAT Gateway data transfer bill — most of it is usually S3 traffic that should be using a gateway endpoint.
AWS services = VPC endpoint, not NAT
4
Is it a third-party API (Stripe, Twilio, GitHub)?
Traffic goes through NAT Gateway → internet. This is unavoidable for true external APIs. Mitigate: connection pooling (don't establish new connections per request), DNS caching (don't re-resolve on every call), and TCP keepalives (avoid NAT timeout at 350s). Set keepalive in your HTTP client to 60s.
external API = NAT → internet. Enable keepalives, pool connections.
5
Is it on-premises or a different cloud?
VPN tunnel (encrypted, over internet, ~10ms overhead) or AWS Direct Connect (dedicated physical line, expensive, ~1ms, no internet). VPN is the right default for most companies. Direct Connect only makes sense if you have very high bandwidth requirements (>1 Gbps sustained) or strict compliance requirements that prohibit internet-transited data.
on-prem = Site-to-Site VPN or Direct Connect
6
Does traffic pass through Kong? Check what plugins run.
If the call goes through Kong (whether internal or external), verify: which plugins are attached to the route? Rate limits that were appropriate for external clients may be too restrictive for internal service-to-service calls. Add a Consumer in Kong for each internal service with higher rate limits, and authenticate service-to-service calls using JWT or a shared secret so Kong can identify the caller.
Kong rate limits apply to internal callers too — configure consumers

Section 12

Network architecture checklist

INT
Internal DNS

Services in the same VPC must call each other by internal DNS, not public hostname. Code review: no api.yourcompany.com in service code.

EP
VPC Endpoints

S3 and DynamoDB get gateway endpoints (free). Check your NAT bill — most of it is usually avoidable AWS-to-AWS traffic.

SG
Security Groups

Managed in Terraform, not console. Reference security groups by ID not CIDR. RDS allows only app-server SG, not 0.0.0.0/0.

NAT
NAT keepalives

TCP keepalive ≤ 60s for connections through NAT Gateway. AWS drops idle connections at 350s. This catches 80% of ECONNRESET incidents.

CIDR
CIDR planning

One team = one CIDR range. Dev/staging/prod don't overlap. Document the plan before creating any VPC — you can't change the primary CIDR later.

KON
Kong admin locked

Port 8001 (Kong admin) accessible only from VPN or bastion. Declarative config (deck) in git. Internal services have their own Consumers with higher rate limits.
