Production Incident  ·  Network Routing  ·  NAT Hairpin

Internal service calling its own public URL — 40ms became 400ms under load

Thursday 16:00 IST  ·  Discovered during load test  ·  Severity: P2  ·  order-svc latency 10× higher than expected

The platform had three microservices: order-svc, payment-svc, and notification-svc. The developer who built order-svc had originally written it as a monolith and tested it against the staging API endpoint. When it was split into a microservice, the API calls stayed pointing at the public hostname: https://api.staging.company.com/payment/charge.

In development this was fine — the call took 40-60ms, well within acceptable limits. During the load test before the production launch, the team noticed order-svc latency climbing above 400ms under sustained load. They added more instances of order-svc. Latency still climbed. They scaled payment-svc. Still climbing. They added more NAT gateways. Better, but still slow.

The root cause took 90 minutes to find: every call from order-svc to payment-svc was exiting through the NAT gateway to the public internet, hitting the load balancer's public IP, and routing back into payment-svc. Under load, the NAT gateway and load balancer became the shared bottleneck. The fix was a single URL change.

During development
order-svc hardcoded to call https://api.staging.company.com/payment/charge. Works fine in dev — 40ms call time, nobody notices the path.
Load test, 500 concurrent users
order-svc latency: 400ms+. Expected: <50ms. Team scales order-svc pods from 3 → 12. No improvement.
+30 min
Scales payment-svc from 3 → 6 pods. Latency drops slightly to 300ms. Team adds second NAT gateway. Still 280ms.
+60 min
Engineer traces actual network path with curl -v --trace-time. Sees DNS resolves to the public ALB IP. Realises traffic is going out to internet and back.
+90 min
Fix: change URL to http://payment-svc.prod.internal:8080/charge. Latency: 0.8ms. Load test passes. Root cause: internal service calling its own public hostname.
Root cause

Internal services calling each other via public hostnames causes traffic to exit through NAT Gateway, traverse the public internet, hit the load balancer, and return to the private subnet. This "NAT hairpin" adds 50-400ms of latency (internet RTT) and adds NAT Gateway and ALB as shared bottlenecks. The fix is always the same: use the internal DNS name of the destination service. The fact that it worked at low load masked the problem until the load test.

The fix — and how to catch it in code reviewTYPESCRIPT + BASH
❌ The incident: calling internal service via public URL
const PAYMENT_URL = process.env.PAYMENT_URL || 'https://api.company.com';
// ↑ Goes out to internet, comes back in. 40ms → 400ms under load.

✅ Fix 1: Internal DNS (simplest, best)
const PAYMENT_URL = process.env.PAYMENT_URL || 'http://payment-svc.prod.internal:8080';
// 0.8ms. Never leaves AWS. Free. The correct default.

✅ Fix 2: Split DNS (api.company.com resolves differently inside vs outside VPC)
# Route 53 private hosted zone: api.company.com → 10.0.1.25 (internal ALB IP)
# Public Route 53:               api.company.com → 54.239.100.10 (public ALB IP)
# Same URL, different answer depending on where you resolve from.
# Benefit: no code change needed, works for all services immediately.

# How to detect NAT hairpin in production:
# Any request where source and destination are both in your VPC
# but the packet passes through NAT GW is a hairpin.

# Check NAT Gateway traffic volume:
aws cloudwatch get-metric-statistics \
  --namespace AWS/NatGateway \
  --metric-name BytesOutToDestination \
  --period 3600 --statistics Sum
# If this is high while your external API calls are known/small:
# Internal services are leaking through NAT.

# Code review checklist: grep for internal-service-calling-public-URL
grep -r "api.company.com\|api.staging.company.com" ./src --include="*.ts" --include="*.js"
# Any match in a backend service should be changed to internal DNS
# UNLESS it's a test file or SDK that needs the public endpoint
What this incident teaches
▶40ms at low load becomes 400ms under load because the bottleneck isn't the service — it's the shared NAT gateway and load balancer. A single NAT Gateway becomes a funnel for all outbound traffic from all private subnets.
▶Internal services must use internal DNS names. Add a grep check to CI: if a backend service contains api.yourcompany.com in its service-to-service call URLs, fail the build. This catches hairpin routing before it ships.
▶Split DNS is the zero-code-change fix. Configure Route 53 private hosted zones so api.company.com returns the internal ALB IP from inside the VPC. All existing code works correctly without any URL changes.
▶Load tests reveal routing bugs that low-traffic testing misses. The hairpin was invisible in development (40ms is acceptable). It only became a problem at scale when the NAT Gateway saturated. Always load test before launch.
