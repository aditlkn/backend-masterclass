SSRF · AWS Credentials · Active Exploitation

A webhook URL field is being used to fetch AWS IAM credentials from the metadata service.

Security incident · booking-service · webhook endpoint · EC2 deployment

Your security team's anomaly detection fires at 03:00: unusual S3 API calls from an IAM role (booking-service-ec2-role) are reading files from the customer data bucket. The calls are coming from outside your normal EC2 fleet IPs.

Investigation shows that your booking service has a webhook registration feature: organisers can register a URL that gets called when a booking is confirmed. The endpoint accepts any URL, fetches it during a test call, and logs the response. An attacker has been registering http://169.254.169.254/latest/meta-data/iam/security-credentials/booking-service-ec2-role as their webhook URL. Your server fetches this, returns the credentials in the webhook test response — which goes back to the attacker. They now have IAM credentials with S3 read access.

Step 1: Understand the vulnerable code

// Current webhook registration endpoint app.post('/webhooks/register', authenticate, async (req, res) => { const { url, eventTypes } = req.body; // Store the webhook await db.createWebhook({ userId: req.user.id, url, eventTypes }); // Test call to confirm the endpoint is reachable try { const testResponse = await fetch(url, { method: 'POST', headers: { 'Content-Type': 'application/json' }, body: JSON.stringify({ type: 'webhook.test', timestamp: Date.now() }), }); const body = await testResponse.text(); res.json({ registered: true, testStatus: testResponse.status, testBody: body }); // ^^^^^^^^ // returns the response body to caller // = returns AWS credentials to attacker } catch (err) { res.json({ registered: true, testStatus: 'unreachable' }); } }); // Attacker registers: // { url: "http://169.254.169.254/latest/meta-data/iam/security-credentials/booking-service-ec2-role" } // Server fetches the metadata URL, testBody contains: // { "AccessKeyId": "ASIA...", "SecretAccessKey": "...", "Token": "...", "Expiration": "..." } // Attacker reads testBody from the response. What this incident teaches
▶Credential rotation is more urgent than vulnerability patching. Once credentials are exfiltrated, the attacker has them regardless of what you do to the application. Patch the vulnerability, but revoke the credentials first. In AWS: rotate the role's access keys, or if using instance metadata, the credentials expire within hours — but the attacker may have already used them or moved laterally.
▶The response body is as dangerous as the request. SSRF works because your server makes the request and returns the result. A URL validator that blocks the fetch is necessary. But also: never return the body of a fetch to a user-supplied URL back to the user. The combination of fetch + return body is the complete exploit chain.
▶Least-privilege IAM is SSRF's best mitigation. If the booking-service-ec2-role had only the permissions it actually needed (write to a specific SQS queue, read from a specific S3 prefix), the exfiltrated credentials would have done the attacker much less good. The credential exposure was serious. The blast radius was determined entirely by the IAM role's permissions.
▶IMDSv2 would have required an extra step — not prevented this. IMDSv2 requires a PUT request to get a session token before a GET can retrieve credentials. Some HTTP clients follow this flow, making SSRF against IMDSv2 still possible. Enable IMDSv2 on all EC2 instances, but do not treat it as a substitute for URL validation.
