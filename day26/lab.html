Projection Bug  ·  booking_summary_view  ·  3 weeks of wrong data

The booking count dashboard has been wrong for three weeks. The finance team just noticed.

Post-incident review · booking_count_projection · group bookings miscounted

The weekly revenue dashboard shows confirmed bookings per day. For the past three weeks it has been undercounting by roughly 15%. The finance team noticed because actual ticket sales reported by the venue were higher than the dashboard numbers.

Root cause: a bug in the booking_count_projection. When the GroupBookingConfirmed event type was introduced three weeks ago, the projection handler only handles BookingConfirmed. Group bookings — which can confirm 2-20 seats in a single event — were never counted. The projection was silently dropping 15% of confirmations.

In a traditional state-store system, those three weeks of wrong counts would require a manual reconciliation script, probably against payment records, and there would be uncertainty about whether the corrected numbers are accurate. In your event-sourced system, every GroupBookingConfirmed event is sitting in the event store. The fix is to rebuild the projection.

Step 4: Verifying the rebuild

-- After rebuild completes, verify against the event stream directly: SELECT date_trunc('day', (payload->>'confirmedAt')::timestamptz) AS day, SUM(CASE WHEN event_type = 'BookingConfirmed' THEN 1 WHEN event_type = 'GroupBookingConfirmed' THEN (payload->>'seatCount')::int ELSE 0 END) AS expected_count FROM events WHERE event_type IN ('BookingConfirmed', 'GroupBookingConfirmed') AND created_at >= NOW() - INTERVAL '30 days' GROUP BY 1 ORDER BY 1; -- Compare with the rebuilt read model: SELECT date, count FROM booking_counts WHERE date >= NOW() - INTERVAL '30 days' ORDER BY date; -- If they match: rebuild is correct. -- If they do not: the handler still has a bug.
What this scenario teaches
▶The event stream is the source of truth — the read model is disposable. Three weeks of wrong data in a state-store system is a data integrity problem that may be unrecoverable. In an event-sourced system it is a projection bug that is fixable by rebuilding. The events were always correct. Only the projection was wrong.
▶Silent event drops are the most dangerous projection bug. The handler did not crash. It did not error. It processed every event it knew about and silently skipped the ones it did not. Always log unknown event types. An event your projection does not handle should at minimum emit a warning metric so you know it exists.
▶Verify the rebuild against the event stream, not against another read model. The event stream is authoritative. The SQL query against the events table gives you the ground truth. If the rebuilt read model matches it, you are correct. If it matches a different read model (which might also have a bug), you have proven nothing.
▶New event types require updating all affected projections before the event is published. The coordination failure here was that GroupBookingConfirmed was introduced in the write side without auditing which projections needed to handle it. Add a projection audit to your event type introduction checklist: which projections consume this stream? Do any of them need to handle the new event?
