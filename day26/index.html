Section 01

The bank ledger — why accountants never delete anything

A bank does not store your balance. It stores every transaction that ever happened to your account. Deposit ₹10,000 on Jan 3. Withdrawal ₹3,000 on Jan 7. Transfer in ₹5,000 on Jan 12. Withdrawal ₹2,800 on Jan 15. Your balance — ₹9,200 — is derived by summing those events. The bank did not update a number. It appended a record.

This is not a performance optimisation. It is an architectural choice with profound consequences. Because the ledger is append-only and complete, the bank can reconstruct your balance as of any date in history. It can produce an audit trail for any regulator without a separate audit table. It can answer questions that were not anticipated when the system was designed — "what was the average daily balance in Q3?" — by replaying the events through a new calculation.

The teller at the counter writes to the ledger. The summary on your phone reads from a different view. When you deposit money, the teller records a MoneyDeposited event. That event flows to several projections: the balance display on your phone, the monthly statement generator, the overdraft monitor, the fraud detection system. Each projection maintains its own read-optimised view. The teller does not think about any of them. They just record the event.

What the teller writes and what you read are separated on purpose. The write model is simple and strict: valid events, in order, immutable once written. The read models are multiple, optimised for their specific query, and rebuilt from the events whenever needed. This separation — Command Query Responsibility Segregation — means each side can evolve independently. Add a new read model without touching the write side. Change the query shape of an existing projection without migrating any data.

Where the ledger analogy breaks. A bank ledger is sequential and centralised. Distributed event-sourced systems deal with concurrency, out-of-order events, and eventual consistency across aggregates. And unlike a ledger where the accountant understands every entry, event-sourced systems accumulate thousands of event types over years — schema evolution becomes a significant engineering problem. The ledger gives you the intuition. The engineering gives you the discipline to make it work.

Section 02

What does event sourcing actually look like in code?

An aggregate is the unit of consistency in event sourcing. A Booking aggregate starts as an empty shell. Commands are applied to it — CreateBooking, ConfirmPayment, CancelBooking. Each command, if valid, produces one or more events. The events are appended to the aggregate's event stream. The current state of the aggregate is never stored directly — it is always derived by replaying the events.

The three components

Commands: Intents to change state. ConfirmBooking { bookingId, userId, seatId }. A command may be rejected if the aggregate is in the wrong state (you cannot confirm a booking that is already cancelled). Commands produce events, not state changes.

Events: Facts that have happened. BookingConfirmed { bookingId, userId, seatId, confirmedAt }. Events are immutable. They are in the past tense. They cannot be rejected — once written, they are true. The question "did this event happen?" always has the same answer.

Apply functions: Pure functions that take the current state and an event and return the new state. apply(state, BookingConfirmed) => { ...state, status: 'confirmed' }. No side effects. No I/O. Just state transitions.

Booking aggregate — event sourcing patternTYPESCRIPT
// Events — immutable facts in the past tense
type BookingEvent =
  | { type: 'BookingCreated';   bookingId: string; userId: string; seatId: string; createdAt: string }
  | { type: 'PaymentConfirmed'; bookingId: string; amountCents: number; paidAt: string }
  | { type: 'BookingConfirmed'; bookingId: string; confirmedAt: string }
  | { type: 'BookingCancelled'; bookingId: string; reason: string; cancelledAt: string };

// State — derived, never stored
interface BookingState {
  status: 'pending' | 'confirmed' | 'cancelled';
  userId: string;
  seatId: string;
  amountCents?: number;
}

// Apply — pure function, no side effects
function apply(state: Partial<BookingState>, event: BookingEvent): Partial<BookingState> {
  switch (event.type) {
    case 'BookingCreated':
      return { status: 'pending', userId: event.userId, seatId: event.seatId };
    case 'PaymentConfirmed':
      return { ...state, amountCents: event.amountCents };
    case 'BookingConfirmed':
      return { ...state, status: 'confirmed' };
    case 'BookingCancelled':
      return { ...state, status: 'cancelled' };
    default: return state;
  }
}

// Reconstruct current state by replaying events
function reconstruct(events: BookingEvent[]): Partial<BookingState> {
  return events.reduce(apply, {});
}

// Handle a command
async function confirmBooking(bookingId: string) {
  const events = await eventStore.load(bookingId);
  const state  = reconstruct(events);

  if (state.status !== 'pending') throw new Error('Cannot confirm: not pending');

  const newEvent: BookingEvent = {
    type: 'BookingConfirmed',
    bookingId,
    confirmedAt: new Date().toISOString(),
  };

  await eventStore.append(bookingId, newEvent, events.length); // optimistic locking
}
Section 03

What does the event store actually need to guarantee?

An event store is an append-only log partitioned by aggregate ID. For a given aggregate (a booking, a user account, an order), events must be stored in exact order with no gaps. The event store makes three guarantees that a regular database does not:

  • Ordered append within an aggregate: Event 5 for booking-123 always comes after event 4. No concurrent writers can insert between them.
  • Optimistic concurrency control: When appending, the writer specifies the expected version (number of events already seen). If someone else has appended since the writer loaded, the append is rejected. The writer must reload and retry. This prevents lost updates without locks.
  • Global ordering for projections: Events from all aggregates are also accessible in global sequence order, so projections can subscribe to "all events" in the order they were written.

Snapshots — skipping the replay on long streams

A booking aggregate might have 8 events over its lifetime. A financial account might have 80,000. Replaying 80,000 events on every command is not feasible. Snapshots solve this: periodically serialise the current state and store it alongside the event stream. On load, find the most recent snapshot and replay only the events since it. The snapshot is a performance optimisation — the event stream remains the source of truth and can always be used to verify or rebuild the snapshot.

⚠️
EventStoreDB vs rolling your own

EventStoreDB is purpose-built for this: it implements ordered append, optimistic concurrency, global event subscriptions, and projections natively. Building on PostgreSQL (using a table with an aggregate_id + version composite key and a unique constraint to enforce optimistic locking) is a common and valid approach for teams that do not want another operational dependency. Kafka is not an event store — it does not support per-aggregate ordering with optimistic concurrency control. Do not use Kafka as your primary event store.

Section 04

Projections — building read models from events

A projection subscribes to the event stream and builds a read-optimised view. It is a pure function from events to a read model. The same events can feed multiple projections, each optimised for a different query pattern. A booking confirmation email needs different data than a revenue report, which needs different data than a fraud detection signal. Three projections, one event stream.

Catch-up subscriptions vs live subscriptions

A catch-up subscription starts from a specific position in the event stream (typically position 0 for a new projection) and processes events from there to the current head. Useful for rebuilding a projection from scratch, building a new projection against historical data, or recovering a projection after downtime.

A live subscription receives new events as they are written. Used by projections that need to stay current in near-real-time. Production projections typically do both: catch-up from a checkpoint to the current head, then switch to live. The checkpoint is stored so the projection can resume from where it left off after a restart.

Projection rebuild — the killer feature

You discover a bug in your booking confirmation projection — it was calculating the seat label incorrectly for group bookings for the past three months. In a traditional system, you write a migration, pray it works, and accept that the data from the past three months is wrong. In an event-sourced system, you fix the projection code, delete the read model, and replay all events from the beginning. The read model is rebuilt correctly from the immutable event history. The past three months are fixed.

✅
Projections are disposable — event streams are not

This is the most important mental shift in event sourcing. Your read models are derived, temporary, and rebuildable. They can be wrong (rebuilt). They can be deleted (recreated). They can be added (new projection against historical data). Your event streams are permanent, immutable, and the only source of truth. Treat them accordingly: the event store gets your highest durability guarantees, your most careful schema design, and your most conservative change policy.

Section 05

CQRS — why reads and writes deserve different models

Command Query Responsibility Segregation is a simpler idea than it sounds: the model you use to write data does not have to be the same model you use to read it. In a traditional CRUD system, you write a booking record and you read back that same booking record. In CQRS, you issue a CreateBooking command that modifies the write-side aggregate, and a separate read model — built by a projection — answers your queries.

CQRS does not require event sourcing. You can have CQRS with a regular database: write to a normalised table, maintain denormalised read tables updated by triggers or application code. But event sourcing and CQRS are natural companions because the event stream is the perfect mechanism for keeping the read models in sync with the write model.

Why separate them?

Write models and read models have fundamentally different requirements. Write models need consistency guarantees: one aggregate, one transaction, no conflicts. Read models need query flexibility: joins across multiple aggregates, pagination, search, sorting. Trying to serve both from the same schema produces a compromise that serves neither well. CQRS lets each side be optimal for its own purpose.

The consistency trade-off

The read model is eventually consistent with the write model. There is a lag — typically milliseconds, sometimes seconds — between when a command is processed and when the projection updates the read model. A user who creates a booking and immediately refreshes their booking list may not see it yet. This is not a bug in the implementation; it is an inherent property of the pattern. Product and engineering must align on which operations require strong consistency and handle them explicitly (return the created resource directly from the command, rather than reading it back from the projection).

Section 06

Process managers and sagas — coordinating across aggregates

Event sourcing is excellent within a single aggregate. The hard problem is coordination across aggregates: booking a seat (inventory aggregate) and charging a card (payment aggregate) as part of one business operation. Each aggregate has its own event stream. There is no distributed transaction across them.

A process manager (sometimes called a saga) is a long-running workflow that reacts to events and issues commands. It maintains its own state — also event-sourced — representing where the workflow is. When BookingInitiated fires, the process manager issues HoldSeat to inventory. When SeatHeld fires, it issues ChargeCard to payment. When ChargeSucceeded fires, it issues ConfirmBooking. When ChargeFailed fires, it issues ReleaseSeat.

The process manager does not call services directly — it publishes commands to a command bus. Each service handles its own commands and publishes events. The process manager reacts to those events. No service knows about any other service. The process manager knows about the workflow but not the implementation of any step.

💡
Process manager vs choreography — when to use each

Choreography (Day 24's saga pattern) has services reacting directly to each other's events. Simple and decoupled, but hard to trace when the workflow spans many steps. Process managers centralise the workflow logic — one place to see what happens next — at the cost of an additional component. For workflows with 3+ steps, compensating transactions, or timeout handling, a process manager makes the flow explicit and debuggable. For simple 2-step workflows, choreography is usually sufficient.

Section 07

When event sourcing is the wrong answer

Event sourcing adds significant complexity. It is the right choice for specific problems and the wrong choice for most systems. Be honest about which category you are in.

The complexity it adds

  • Schema evolution is hard. Events are immutable. When the shape of an event needs to change, you cannot migrate existing events. You must handle multiple versions of the same event type in your apply functions forever. This accumulates over years.
  • Eventual consistency is pervasive. Every read is potentially stale. The entire team — engineering and product — must internalise this. Features that require strong consistency (show the user the thing they just created) require explicit workarounds.
  • Querying is harder. You cannot ad-hoc query the event store like a SQL database. Every query shape you need requires a projection. Projections take time to build and maintain. Complex analytical queries require careful projection design.
  • Testing is more complex. Unit testing aggregates is clean (given these events, apply this command, expect these new events). Integration testing projections, process managers, and the overall system is significantly more involved.

Good fits for event sourcing

  • Financial systems where audit trail is a compliance requirement
  • Systems where "what happened?" is as important as "what is the current state?"
  • Domains with complex business rules where state transitions need clear modelling
  • Systems that need to rebuild read models as business requirements evolve

Bad fits

  • CRUD-heavy systems with simple state (user profiles, product catalogs)
  • Systems where the team is not already comfortable with eventual consistency
  • Prototypes or early-stage products where the domain model is not yet stable
  • Systems where queries are complex joins that cannot be pre-materialised

Section 08

Production reality

EventStoreDB vs PostgreSQL as event store

EventStoreDB is purpose-built: it has native projections, subscriptions, and cluster support. The operational cost is a new database to manage. PostgreSQL with a well-designed events table (aggregate_id, version, event_type, payload, created_at, with a unique constraint on aggregate_id + version) works for moderate scale and teams that already run Postgres. At high event rates (>10K events/sec), PostgreSQL write throughput becomes a limiting factor and EventStoreDB or Kafka with compacted topics becomes necessary.

Event schema versioning

Events accumulate. An event type defined in 2023 will still be in your event stream in 2028. When the shape needs to change, you have three options: upcasting (transform old events to new shape on read — no data migration), versioned event types (BookingConfirmedV2 alongside BookingConfirmed), or copy-and-transform (write new events for all existing aggregates in the new shape — expensive but clean). Upcasting is the most common approach. Build it into your event store client from day one.

Projection lag monitoring

The gap between the latest event written and the projection's current position is your projection lag. Monitor it. Alert when it grows unexpectedly (projection is falling behind, which means read models are increasingly stale). A healthy projection typically lags by <1 second. A lagging projection may indicate the projection handler is slow, the read store is under load, or a poison message is causing repeated failures.

Section 09

The things nobody tells you

The first event is the hardest to name

Naming events correctly requires understanding the domain deeply. BookingCreated vs BookingInitiated vs SeatRequested — these are not stylistic choices. They reflect your understanding of what actually happened in the business domain. Getting event names wrong means your event stream describes a system that does not match reality, which confuses everyone who reads it later. Invest time in event naming. Use past tense. Use domain language. Write them down before writing code.

Snapshots are a performance feature, not a correctness feature

Teams often introduce snapshots to "fix" a performance problem and then start treating snapshots as the source of truth. They are not. The event stream is the source of truth. Snapshots can be deleted and rebuilt. If there is ever a discrepancy between a snapshot and the event replay, the event replay is correct. Do not let application code treat snapshots as authoritative without verifying they match the events.

You will need to delete data for GDPR

"Events are immutable" is an engineering principle, not a legal defence. When a user requests erasure under GDPR, you must handle it. Standard approaches: crypto-shredding (encrypt personal data in events with a per-user key, delete the key to make the data unreadable), separate PII store (reference user IDs in events, not names and emails — look them up from a separate store that can be deleted), or event compaction (replace sensitive events with sanitised versions). Design for this before you have an erasure request.

The read model is not a cache

Teams sometimes implement CQRS read models as a cache in front of the write database — "if the cache misses, fall back to the database." This defeats the purpose entirely. The read model is not a cache. It is a different representation of the data, optimised for different access patterns. It has its own schema, its own storage, its own lifecycle. If the read model is down, the correct response is to return 503, not to fall back to the write database. The separation is the point.

Decision sequence

Adopting event sourcing

Work through this before writing a single aggregate.

1
Do you actually need audit history or temporal queries?
If the business does not need to answer "what was the state at time T?" or "what happened between event A and event B?", event sourcing adds complexity for no benefit. A regular database with an audit log table (append-only, write trigger) achieves 80% of the benefits with 20% of the complexity for most systems.
if you do not need the history, do not pay the price
2
Model your events in the domain language before writing code
List every meaningful thing that can happen to your key aggregates. Use domain expert language. Past tense. Specific. BookingConfirmed, not BookingUpdated. SeatHeld, not InventoryChanged. Wrong event names cannot be fixed later without migrating your entire event history. This is the most important step and should take days, not hours.
wrong event names = wrong mental model forever
3
Choose your event store before writing aggregates
EventStoreDB if you want native projections and subscriptions. PostgreSQL events table if you already run Postgres and are at moderate scale. Either requires optimistic concurrency control — a unique constraint on (aggregate_id, version) that rejects concurrent writes. Build upcasting support from day one — you will need it.
EventStoreDB for new projects; Postgres for teams already on it
4
Build projections for your known query patterns
Identify every read pattern your application needs. Each one is a projection. Build them as separate components that subscribe to the event stream and maintain their own read store. Do not try to serve all queries from the event store directly — it is not designed for arbitrary queries. The projection is the bridge from events to queryable state.
each query pattern = one projection = one optimised read store
5
Design for GDPR erasure before you have user data
Decide your erasure strategy before the first user signs up: crypto-shredding (encrypt PII with per-user key, delete key), separate PII store (events contain user IDs only, not names/emails), or both. Retrofitting erasure into an event-sourced system with years of PII in immutable events is one of the most painful engineering problems you can create for yourself.
immutable events + unplanned GDPR = very bad time
6
Monitor projection lag from day one
The gap between the latest event and the projection head is your staleness window. Expose it as a metric. Alert when it exceeds your SLO. A projection that is 10 seconds behind is usually fine. A projection that is 10 minutes behind and growing means your read models are dangerously stale and something is wrong in your projection handler.
projection lag = the SLO for your read models

Event sourcing production checklist

EVT
Event names

Past tense. Domain language. Reviewed with a domain expert. Written down before code is written.

OCC
Optimistic locking

Unique constraint on (aggregate_id, version). Concurrent appends rejected. Writers must reload and retry.

UPC
Upcasting

Event schema versioning strategy defined. Upcaster registered for every event type that has changed shape.

LAG
Projection lag

Projection head position monitored. Lag exposed as Prometheus metric. Alert fires when lag grows.

GDPR
Erasure strategy

Crypto-shredding or PII separation designed before first user data written. Tested with a simulated erasure request.

RBL
Rebuild tested

Projection rebuild from zero tested in staging. Time to rebuild measured. Acceptable SLA defined.

The bank does not store your balance. It stores what happened to your money. The balance is just the most recent answer to the question "what do all the events add up to?" Ask a different question — average daily balance, largest single transaction, spending by category — and the same ledger answers it without a schema migration.

The next time you reach for UPDATE — ask: does what happened here matter as much as where I ended up?
