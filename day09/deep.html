Deep Dive 01

SELECT FOR UPDATE variants:
the full syntax reference

SELECT FOR UPDATE is not a single feature — it has four variations that each lock rows differently. Using the wrong one can cause unnecessary blocking or miss the protection you need.

FOR UPDATE

Acquires an exclusive row-level lock on every row returned by the SELECT. Other transactions that try to UPDATE, DELETE, or SELECT FOR UPDATE the same rows will block until your transaction commits or rolls back.

Use when: You intend to update the rows you read, and you need to prevent any other transaction from modifying them between your read and your write. The classic read-then-update pattern.

FOR SHARE

Acquires a shared row-level lock. Multiple transactions can hold FOR SHARE locks on the same row simultaneously. But a transaction trying to UPDATE or SELECT FOR UPDATE on those rows will block.

Use when: You're reading data that another transaction must not modify while you're using it, but you don't intend to modify it yourself. Example: reading a parent row to validate a foreign key before inserting a child row.

FOR NO KEY UPDATE

Weaker than FOR UPDATE — it doesn't block transactions that only need a FOR SHARE lock. Specifically allows other transactions to insert rows that reference this row as a foreign key, while still blocking UPDATE on the row's key columns.

Use when: You're updating non-key columns and don't want to block FK-referencing inserts. Useful in tables with high insert rates of child rows.

FOR KEY SHARE

Weakest lock — only blocks FOR UPDATE. Allows FOR SHARE and FOR NO KEY UPDATE concurrently. Automatically used by foreign key checks.

SKIP LOCKED — the job queue primitive

SELECT ... FOR UPDATE SKIP LOCKED attempts to lock each returned row. Any row already locked by another transaction is silently skipped rather than causing a wait. The query returns only the rows it successfully locked.

This is exactly what a job queue needs: multiple workers can each call the same SELECT and each get a different row without blocking each other. Without SKIP LOCKED, the second worker would block waiting for the first worker to release its lock.

Complete SELECT FOR UPDATE reference — with real examplesSQL
-- FOR UPDATE: exclusive row lock. Block all others.
SELECT balance, version
FROM wallets
WHERE user_id = $1
FOR UPDATE;
-- Use for: balance updates, inventory decrement, seat booking
-- Other transactions: blocked on any attempt to lock/update these rows

-- FOR SHARE: shared read lock. Allow other reads, block writes.
SELECT id, name FROM users WHERE id = $1
FOR SHARE;
-- Use for: FK validation, reading a parent before inserting child
-- Other transactions: can also SELECT FOR SHARE; blocked on UPDATE/FOR UPDATE

-- NOWAIT: fail immediately instead of waiting if lock can't be acquired
SELECT balance FROM wallets WHERE user_id = $1
FOR UPDATE NOWAIT;
-- If row is already locked: ERROR: could not obtain lock on row in relation "wallets"
-- Use when: you want to handle lock contention in app code rather than queuing

-- SKIP LOCKED: skip already-locked rows, return only lockable ones
-- The job queue pattern:
SELECT id, payload, attempts
FROM job_queue
WHERE status = 'pending'
  AND next_retry_at <= now()
ORDER BY priority DESC, created_at ASC
LIMIT 1
FOR UPDATE SKIP LOCKED;
-- Returns: the highest-priority pending job that no other worker has locked
-- Multiple workers run this simultaneously — each gets a different job
-- Zero blocking between workers. High throughput job processing.

-- Complete job queue worker loop
async function processNextJob() {
  await db.query('BEGIN');
  const { rows } = await db.query(`
    SELECT id, payload FROM job_queue
    WHERE status = 'pending' AND next_retry_at <= now()
    ORDER BY priority DESC, created_at
    LIMIT 1 FOR UPDATE SKIP LOCKED
  `);

  if (rows.length === 0) {
    await db.query('ROLLBACK');
    return null; // no available jobs
  }

  const job = rows[0];
  await db.query('UPDATE job_queue SET status=$1, locked_by=$2 WHERE id=$3',
    ['processing', workerId, job.id]);
  await db.query('COMMIT'); // releases FOR UPDATE lock — other workers can now see job as 'processing'

  try {
    await executeJob(job.payload);
    await db.query('UPDATE job_queue SET status=$1 WHERE id=$2', ['done', job.id]);
  } catch (err) {
    await db.query('UPDATE job_queue SET status=$1, attempts=attempts+1, next_retry_at=$2 WHERE id=$3',
      ['pending', new Date(Date.now() + 30000), job.id]);
  }
}

Deep Dive 02

Advisory locks:
application-level distributed mutex

Advisory locks are arbitrary application-defined locks that PostgreSQL manages but doesn't tie to any specific table row or schema object. They're purely for application coordination — a way for your application code to say "I'm doing X, nobody else should do X at the same time." They don't protect data directly; they coordinate behaviour.

When row locks aren't enough

Row locks protect specific rows. But some coordination requirements don't map to specific rows:

  • "Only one cron job should run the nightly reconciliation at a time"
  • "Only one process should be generating the daily report"
  • "Only one worker should be processing webhooks for merchant_id=123"
  • "Prevent two deployments from running simultaneously"

For these cases, you need a named mutex — advisory locks provide exactly this, using your existing PostgreSQL connection without any additional infrastructure.

Session vs transaction advisory locks

Session-level: Held until explicitly released or the session disconnects. Independent of transaction state — not released by ROLLBACK or COMMIT. Use for long-running processes where the lock should outlast individual transactions.

Transaction-level: Automatically released when the transaction commits or rolls back. Use for per-request coordination where the lock should live exactly as long as the transaction.

Advisory locks — session level, transaction level, and try variantsSQL
-- Session-level advisory lock (held until released or disconnect)
-- Lock key: any int8 (or two int4) that you define
SELECT pg_advisory_lock(12345);         -- blocks if already held
SELECT pg_advisory_lock(12345, 67890);  -- two-int4 variant (more keyspace)

-- Try to acquire: returns true if got it, false if already held
SELECT pg_try_advisory_lock(12345);
-- Non-blocking. Use when you want to skip if someone else has it.

-- Release session lock
SELECT pg_advisory_unlock(12345);
SELECT pg_advisory_unlock_all(); -- release all session locks held by this connection

-- Transaction-level (released automatically on COMMIT/ROLLBACK)
BEGIN;
  SELECT pg_advisory_xact_lock(12345);
  -- ... do work ...
COMMIT; -- lock automatically released here

-- Pattern: distributed cron job mutex (Node.js)
async function runNightlyReconciliation() {
  const LOCK_KEY = 9876543210; // arbitrary app-defined int64
  const { rows } = await db.query('SELECT pg_try_advisory_lock($1)', [LOCK_KEY]);
  const gotLock = rows[0].pg_try_advisory_lock;

  if (!gotLock) {
    logger.info('Reconciliation already running on another instance — skipping');
    return;
  }

  try {
    await runReconciliation(); // only one instance runs this
  } finally {
    await db.query('SELECT pg_advisory_unlock($1)', [LOCK_KEY]);
  }
}

-- Pattern: per-merchant lock (prevent concurrent webhook processing)
async function processWebhook(merchantId: number, event: WebhookEvent) {
  await db.query('BEGIN');
  // Transaction-level: released on commit/rollback, even if we forget to unlock
  await db.query('SELECT pg_advisory_xact_lock($1)', [merchantId]);
  // All webhook processing for this merchant is now serialized
  await applyWebhookEvent(merchantId, event);
  await db.query('COMMIT');
}

-- View currently held advisory locks
SELECT pid, granted, classid, objid, objsubid, mode
FROM pg_locks
WHERE locktype = 'advisory'
ORDER BY pid;
⚠️
Session advisory locks survive transaction rollback

If you acquire a session-level advisory lock and your transaction rolls back, the lock is NOT released. You must call pg_advisory_unlock() explicitly in a finally block. This is different from row locks, which are always released on transaction end. Forgetting to release session locks causes them to stay until the database connection closes — which with connection pooling might be never, blocking all other instances indefinitely.

Verify advisory lock usage
✓Always use try/finally with session-level advisory locks — the unlock must happen even if the operation throws.
✓Prefer pg_advisory_xact_lock over session locks when the lock should live exactly as long as the transaction — automatic release prevents leaks.
✓Monitor stuck advisory locks: SELECT pid, objid, now()-query_start AS age FROM pg_locks JOIN pg_stat_activity USING (pid) WHERE locktype='advisory'
✓Use pg_try_advisory_lock for "skip if busy" semantics (cron jobs), pg_advisory_lock for "wait until available" (per-resource serialization).
Deep Dive 03

Savepoints & partial rollback:
rollback within a transaction

A savepoint creates a named rollback point within an open transaction. If a subsequent operation fails, you can roll back to the savepoint — undoing only the work since that point — without aborting the entire transaction. This enables "try something, if it fails keep going" patterns within a single database transaction.

The problem savepoints solve

Without savepoints: if any operation in a transaction fails, the entire transaction is aborted. You can't selectively undo part of it. For workflows that need to "try to apply each item in a batch, skip failures, commit the successes," you'd need multiple separate transactions — each with its own overhead and commit cost.

With savepoints: you can create a checkpoint before a risky operation, attempt it, and if it fails roll back to the checkpoint while keeping the rest of the transaction intact.

Savepoints in practice: batch processing with partial failure

Savepoints — batch import with per-item failure handlingSQL + TYPESCRIPT
-- SQL syntax
SAVEPOINT sp1;                  -- create savepoint named "sp1"
ROLLBACK TO SAVEPOINT sp1;      -- undo everything since sp1, keep sp1 in place
RELEASE SAVEPOINT sp1;          -- remove savepoint (optional, frees resources)

-- Pattern: batch import, skip invalid rows, commit valid ones
async function importOrders(rows: OrderRow[]): Promise<{ ok: number; failed: number }> {
  await db.query('BEGIN');
  let ok = 0, failed = 0;

  for (const row of rows) {
    // Create savepoint before each row
    await db.query('SAVEPOINT row_insert');

    try {
      await db.query(
        'INSERT INTO orders (id, user_id, amount) VALUES ($1, $2, $3)',
        [row.id, row.userId, row.amount]
      );
      // Release savepoint — we don't need to roll back to here anymore
      await db.query('RELEASE SAVEPOINT row_insert');
      ok++;
    } catch (err) {
      // Row failed (constraint violation, duplicate key, etc.)
      // Roll back to savepoint — only this row's INSERT is undone
      await db.query('ROLLBACK TO SAVEPOINT row_insert');
      await db.query('RELEASE SAVEPOINT row_insert');
      logger.warn({ row, err: err.message }, 'Row skipped during import');
      failed++;
    }
  }

  // Commit all successfully inserted rows in one transaction
  await db.query('COMMIT');
  return { ok, failed };
}

-- Why not use separate transactions per row?
-- Each COMMIT has overhead: WAL flush, lock release, snapshot cleanup
-- Savepoints + single COMMIT: much faster for large batches
-- 10,000 rows: ~100× faster with single COMMIT vs per-row transactions

ORMs and savepoints: what pg, prisma, and sequelize do

Most ORMs implement nested transactions using savepoints transparently. When you call transaction() inside another transaction, the ORM creates a savepoint instead of a real nested transaction (which PostgreSQL doesn't support). The inner "rollback" becomes ROLLBACK TO SAVEPOINT, and the inner "commit" becomes RELEASE SAVEPOINT.

This means: ORM-level nested transactions behave like savepoints, not separate atomic units. If the outer transaction rolls back, the inner "committed" transaction also rolls back — because it was just a savepoint release, not a true commit.

Savepoints in Prisma — nested transactions map to savepointsTYPESCRIPT
// Prisma's $transaction() creates a savepoint when called inside another $transaction
await prisma.$transaction(async (tx) => {
  await tx.order.create({ data: orderData });

  // Nested transaction = SAVEPOINT under the hood
  try {
    await prisma.$transaction(async (innerTx) => {
      await innerTx.inventory.update({ ... });
      await innerTx.shipment.create({ ... });
    });
  } catch (err) {
    // Inner transaction failed → ROLLBACK TO SAVEPOINT
    // Outer transaction continues: order.create() is still in place
    logger.warn('Shipment creation failed, order still recorded');
  }

  // Order create is still intact even if inner tx failed
  await tx.notification.create({ data: notifData });
}); // COMMIT — order + notification committed, inventory/shipment rolled back
Verify savepoint usage
✓After ROLLBACK TO SAVEPOINT, the savepoint still exists — you can retry and either release or roll back to it again. No need to re-create it.
✓RELEASE SAVEPOINT is optional but good practice — it frees the savepoint's state and any sub-transaction memory. In large batches, always release after a successful insert.
✓Savepoints add per-savepoint overhead. For 10,000-row batches, create one savepoint per row. Don't nest savepoints 10 levels deep — PostgreSQL handles it but it adds complexity.
✓PostgreSQL's error recovery: after any error in a transaction (even outside a savepoint), the transaction is in an aborted state. The only valid commands are ROLLBACK and ROLLBACK TO SAVEPOINT. Any other query returns "ERROR: current transaction is aborted."
