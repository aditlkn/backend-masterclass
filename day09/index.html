Section 01

ACID: what the letters actually mean in practice

ACID is not a binary feature a database has or doesn't have. It's four independent guarantees, each of which can be violated in specific ways, each of which costs something to maintain. Understanding what each guarantee means — and what breaks when it fails — is what lets you design correctly.

Atomicity: all or nothing

A transaction's operations are treated as a single unit. Either all of them commit, or none of them do. There's no state where some operations within a transaction have taken effect and others haven't — at least not from the perspective of any other transaction.

What breaks without it: a payment transaction that debits an account and credits another. If the debit commits but the credit fails, money disappears. Atomicity prevents this — if the credit fails, the debit is rolled back.

How PostgreSQL implements it: Write-Ahead Logging (WAL). Before any data page is modified, the change is written to the WAL. On crash recovery, PostgreSQL replays committed WAL entries and rolls back uncommitted ones. The WAL is the atomicity mechanism.

Consistency: valid states only

A transaction brings the database from one valid state to another valid state. "Valid" is defined by your constraints: foreign keys, check constraints, unique constraints, not-null constraints. If a transaction would violate any of these, it's rejected entirely.

Consistency is the only ACID property that isn't purely a database responsibility — your application also bears responsibility for it. The database enforces the constraints you define; it can't enforce business rules you haven't expressed as constraints.

What breaks without it: a transfer that results in a negative balance, if you haven't defined a check constraint preventing it. The database maintains its internal consistency (foreign keys, etc.) but can't enforce rules it doesn't know about.

Isolation: concurrent transactions don't interfere

This is the most complex ACID property, with a spectrum of guarantees and a direct tradeoff against performance. Full isolation means concurrent transactions execute as if they were serial — one after another — even if they actually run in parallel. But full isolation is expensive. SQL defines four isolation levels that offer progressively weaker guarantees in exchange for better concurrency.

Durability: committed data survives crashes

Once a transaction commits, its effects survive — even if the database process crashes immediately after. This is guaranteed by the WAL: the commit is not acknowledged until the WAL record is flushed to disk (or, with synchronous_commit=off, at least to the OS buffer). On restart, PostgreSQL replays the WAL and the committed transaction is present.

⚠️
synchronous_commit = off breaks durability

Setting synchronous_commit = off allows PostgreSQL to acknowledge commits before WAL is flushed to disk. Writes are faster, but there's a window (typically ~200ms) where a crash can lose committed transactions. This is sometimes acceptable for high-throughput non-critical writes but must never be used for financial or audit data. It's a durability tradeoff, not a free performance win.

Section 02

Isolation levels: the four read phenomena

SQL defines four isolation levels. Each level prevents specific "read phenomena" — ways that concurrent transactions can produce incorrect reads. The higher the isolation level, the fewer phenomena are possible, and the more the database must do to prevent them (more locking, more conflict detection, more retries).

The three read phenomena

Dirty read: Transaction A reads a value written by Transaction B, which hasn't committed yet. If B rolls back, A has read data that never existed. Only READ UNCOMMITTED allows this — and PostgreSQL doesn't actually implement READ UNCOMMITTED (it silently upgrades it to READ COMMITTED).

Non-repeatable read: Transaction A reads a row, Transaction B updates and commits it, Transaction A reads the same row again and gets a different value. The same row gives different results within the same transaction. READ COMMITTED allows this.

Phantom read: Transaction A runs a query returning a set of rows. Transaction B inserts a new row matching that query's criteria and commits. Transaction A runs the same query again and gets additional rows that weren't there before. Phantoms appear "out of nowhere." REPEATABLE READ prevents non-repeatable reads but in some databases still allows phantoms. PostgreSQL's REPEATABLE READ prevents both through MVCC snapshots.

The lost update problem — the one that actually pages you

Two transactions both read a value, both compute a new value based on the old one, and both write the new value. The second write overwrites the first without knowing about it. One update is silently lost. This is the Day 9 lab incident — two concurrent refund operations both reading the same balance, both writing their result, one overwriting the other.

This happens at READ COMMITTED — the default isolation level. READ COMMITTED only takes a new snapshot at each statement, not across the whole transaction. By the time the second write executes, the first has committed — but the second transaction already read the old value and is computing from it.

Fixes: SELECT FOR UPDATE (explicit row lock), UPDATE ... WHERE balance = $old_balance (optimistic lock), or SERIALIZABLE isolation (automatic conflict detection).

Section 03

Locking: how isolation is actually enforced

PostgreSQL uses two mechanisms to enforce isolation: MVCC (Multi-Version Concurrency Control) for reads, and locks for writes. Understanding which operations acquire which locks — and which locks conflict with each other — explains why some concurrent operations are fast and others block each other.

The PostgreSQL lock hierarchy

PostgreSQL has 8 table-level lock modes, ranging from weakest (ACCESS SHARE, acquired by SELECT) to strongest (ACCESS EXCLUSIVE, acquired by DROP TABLE, TRUNCATE, VACUUM FULL). The key question for any two operations running concurrently: do their lock modes conflict?

  • ACCESS SHARE — acquired by SELECT. Conflicts only with ACCESS EXCLUSIVE. Multiple SELECTs never block each other.
  • ROW SHARE — acquired by SELECT FOR UPDATE/SHARE. Allows concurrent reads and most writes.
  • ROW EXCLUSIVE — acquired by INSERT, UPDATE, DELETE. Conflicts with SHARE, SHARE ROW EXCLUSIVE, EXCLUSIVE, ACCESS EXCLUSIVE.
  • SHARE UPDATE EXCLUSIVE — acquired by VACUUM, ANALYZE, CREATE INDEX CONCURRENTLY. Allows reads and writes, blocks other schema changes.
  • SHARE — acquired by CREATE INDEX (non-concurrent). Blocks all writes but allows reads.
  • ACCESS EXCLUSIVE — acquired by ALTER TABLE, DROP TABLE, TRUNCATE. Blocks everything — reads and writes. The most dangerous lock to hold in production.

Row-level locks

In addition to table locks, PostgreSQL acquires row-level locks when rows are modified. UPDATE and DELETE lock the specific rows they touch. This is why two transactions updating different rows don't block each other, even on the same table.

SELECT FOR UPDATE explicitly acquires row-level locks for rows returned by the SELECT, preventing other transactions from modifying those rows until your transaction completes. This is the correct way to implement "read-then-update" operations where you need to prevent lost updates.

Lock monitoring in productionSQL
-- See all current locks
SELECT
  pid,
  locktype,
  relation::regclass AS table,
  mode,
  granted,
  pg_blocking_pids(pid) AS blocked_by
FROM pg_locks
WHERE relation IS NOT NULL
ORDER BY granted, locktype;

-- Find blocking queries (the ones causing waits)
SELECT
  blocking.pid                  AS blocking_pid,
  blocking.query                AS blocking_query,
  blocked.pid                   AS blocked_pid,
  blocked.query                 AS blocked_query,
  now() - blocked.query_start   AS wait_duration
FROM pg_stat_activity blocked
JOIN pg_stat_activity blocking
  ON blocking.pid = ANY(pg_blocking_pids(blocked.pid))
WHERE blocked.wait_event_type = 'Lock'
ORDER BY wait_duration DESC;

-- Kill a blocking query (use pid from above)
SELECT pg_cancel_backend(blocking_pid);   -- graceful: cancel current query
SELECT pg_terminate_backend(blocking_pid); -- hard: kill the connection

-- See lock wait timeout (queries that waited too long)
SHOW lock_timeout;
-- Set per-session: SET lock_timeout = '5s';
-- Any lock wait exceeding this raises ERROR: canceling statement due to lock timeout

Section 04

Deadlocks: when two transactions wait for each other

A deadlock occurs when Transaction A holds a lock that Transaction B needs, and Transaction B holds a lock that Transaction A needs. Neither can proceed. Neither will release their lock. They wait forever — until PostgreSQL's deadlock detector breaks the cycle by aborting one of the transactions.

The classic deadlock pattern

Transaction A: UPDATE accounts SET balance = balance - 100 WHERE id = 1 then UPDATE accounts SET balance = balance + 100 WHERE id = 2

Transaction B (running concurrently): UPDATE accounts SET balance = balance - 50 WHERE id = 2 then UPDATE accounts SET balance = balance + 50 WHERE id = 1

A locks row 1, B locks row 2. A tries to lock row 2 — waits. B tries to lock row 1 — waits. Deadlock. PostgreSQL detects the cycle and aborts one transaction (the one with the most recently acquired locks), which releases its lock and allows the other to proceed. The aborted transaction gets an error: ERROR: deadlock detected.

The fix: consistent lock ordering

Deadlocks between transactions that touch the same rows are almost always preventable by ensuring all transactions acquire locks in the same order. If both A and B always lock row 1 before row 2 (i.e., always sort the rows by ID and process in ascending order), A will acquire both locks before B starts — no deadlock possible.

In application code: when a transaction needs to update multiple rows, sort them by primary key and update in that order. This single rule eliminates the vast majority of deadlocks.

💡
Deadlocks are always application bugs

PostgreSQL can detect and resolve deadlocks — but a deadlock means your application has a design problem. The transaction that gets aborted must retry, causing latency and complexity. The correct fix is consistent lock ordering or restructuring transactions to not hold locks across multiple operations. Relying on deadlock detection to "handle" deadlocks in normal operation is not a strategy.

Section 05

MVCC: how PostgreSQL avoids read-write conflicts

Multi-Version Concurrency Control is the mechanism that allows PostgreSQL to serve reads without blocking writes, and vice versa. Instead of locking rows when they're read, PostgreSQL keeps multiple versions of each row and shows each transaction the version that was current at the start of that transaction's snapshot.

How row versions work

Every row in PostgreSQL has two hidden system columns: xmin (the transaction ID that created this version) and xmax (the transaction ID that deleted or superseded this version, or 0 if the row is current). When a row is updated, PostgreSQL doesn't modify it in place — it inserts a new version with the new data and a new xmin, and sets the old version's xmax to the current transaction ID.

When a transaction reads a row, it only sees versions where xmin <= snapshot_xid (version existed when snapshot was taken) and xmax = 0 OR xmax > snapshot_xid (version not yet deleted when snapshot was taken). This is the MVCC visibility check — done entirely without acquiring any locks on the rows being read.

Snapshot timing by isolation level

Under READ COMMITTED, a new snapshot is taken at the start of each SQL statement. This means a long-running transaction can see data committed by other transactions between its statements — which is why non-repeatable reads are possible at this level.

Under REPEATABLE READ and SERIALIZABLE, the snapshot is taken once at the start of the transaction and held for the entire transaction. Every read within the transaction sees the same consistent view of the database as of that point in time.

Observing MVCC snapshots directly in PostgreSQLSQL
-- See the hidden system columns on any table
SELECT xmin, xmax, id, balance FROM accounts WHERE id = 1;
-- xmin: transaction that inserted this row version
-- xmax: transaction that deleted/updated it (0 = current version)

-- See the current transaction's snapshot
SELECT txid_current();           -- current transaction ID
SELECT txid_current_snapshot();  -- current snapshot: xmin:xmax:xip_list
-- Format: "100:105:101,103"
-- Means: txid < 100 are visible, txid >= 105 are not,
-- and 101,103 (in-progress) are not visible

-- Demonstrate MVCC: read your own writes
BEGIN;
  INSERT INTO accounts (id, balance) VALUES (99, 1000);
  SELECT xmin, xmax, id, balance FROM accounts WHERE id = 99;
  -- xmin = current txid (your own insert), xmax = 0
  -- Visible to you (same transaction), invisible to others
ROLLBACK;

-- Dead tuple count (the MVCC cost)
SELECT n_live_tup, n_dead_tup FROM pg_stat_user_tables WHERE relname = 'accounts';
-- High n_dead_tup means old MVCC row versions accumulating
-- VACUUM is needed to clean these up

Section 06

Two-phase commit: atomicity across multiple databases

Two-phase commit (2PC) extends atomicity across multiple independent databases or services. If a transaction needs to update Database A and Database B, and we need either both to commit or neither to commit, 2PC provides the protocol.

Phase 1: Prepare

The coordinator sends a "prepare" message to all participants. Each participant executes the transaction up to the point of commit, writes the intent to its own WAL, and replies "ready" or "abort." Crucially, a participant that responds "ready" is committing to commit if asked — it cannot change its mind afterward. If any participant responds "abort," the coordinator aborts the entire transaction.

Phase 2: Commit or Abort

If all participants responded "ready," the coordinator writes the commit decision to its own WAL and then sends "commit" to all participants. Each participant commits and releases its locks. If any participant aborted, the coordinator sends "abort" to all.

The availability problem with 2PC

If the coordinator crashes after Phase 1 but before Phase 2, participants are stuck — they've prepared (locked resources) but don't know whether to commit or abort. They hold their locks waiting for the coordinator to recover. This is 2PC's fundamental weakness: it blocks under coordinator failure. This is one of the primary reasons microservices architectures prefer sagas over distributed transactions.

PostgreSQL supports 2PC via PREPARE TRANSACTION / COMMIT PREPARED. Prepared transactions that are never resolved (due to coordinator failure) appear in pg_prepared_xacts and hold locks indefinitely until manually resolved.

PostgreSQL two-phase commit — explicit syntaxSQL
-- Phase 1: prepare on each participant database
BEGIN;
  UPDATE inventory SET stock = stock - 1 WHERE product_id = $1;
PREPARE TRANSACTION 'order-txn-abc123';
-- Transaction is now prepared: resources locked, not yet committed
-- This state persists across crashes

-- Phase 2a: if all participants prepared successfully
COMMIT PREPARED 'order-txn-abc123';

-- Phase 2b: if any participant failed
ROLLBACK PREPARED 'order-txn-abc123';

-- Check for stuck prepared transactions (holding locks!)
SELECT gid, prepared, owner, database, transaction
FROM pg_prepared_xacts;
-- Any row here is a prepared transaction not yet resolved
-- Manually resolve stuck ones:
-- ROLLBACK PREPARED 'order-txn-abc123';  -- if uncertain, rollback is safer

-- Alert: prepared transactions older than 5 minutes are a problem
SELECT gid, now() - prepared AS age FROM pg_prepared_xacts
WHERE prepared < now() - interval '5 minutes';

Section 07

Sagas: distributed transactions without distributed locking

A saga is a sequence of local transactions where each step publishes an event or message that triggers the next step. If any step fails, the saga executes compensating transactions to undo the completed steps. Sagas trade the all-or-nothing guarantee of ACID for eventual consistency — the system will end up in a consistent state, but may pass through temporarily inconsistent states.

Choreography vs orchestration

Choreography: Each service reacts to events from the previous step. The Order Service places an order, publishes an "OrderPlaced" event. The Inventory Service listens and reserves stock, publishes "StockReserved." The Payment Service listens and charges the customer. No central coordinator — the saga emerges from the interactions. Simpler to implement, harder to observe and debug.

Orchestration: A central Saga Orchestrator sends commands to each service and waits for responses. The orchestrator drives the entire saga: "Reserve stock, then charge payment, then confirm order." Easier to observe and add compensation logic, but the orchestrator becomes a central point of failure and complexity.

Compensating transactions

The saga's "rollback" is a compensating transaction — a new forward transaction that undoes the effect of a completed step. Unlike a database rollback, compensating transactions execute new business logic. If a payment was charged and the inventory reservation then fails, the compensating transaction is a refund — a real financial operation that must be idempotent, logged, and auditable.

Compensating transactions can fail. Retry policies, dead-letter queues, and human escalation paths are required for each compensation step. This is the operational complexity sagas add compared to local ACID transactions.

Saga orchestrator — order placement with compensationTYPESCRIPT
type SagaStep = {
  execute:     () => Promise<void>;
  compensate:  () => Promise<void>;
  name:        string;
};

async function placeOrderSaga(order: Order): Promise<void> {
  const steps: SagaStep[] = [
    {
      name: 'ReserveInventory',
      execute:    () => inventory.reserve(order.items),
      compensate: () => inventory.release(order.items), // undo reservation
    },
    {
      name: 'ChargePayment',
      execute:    () => payment.charge(order.amount, order.userId),
      compensate: () => payment.refund(order.amount, order.userId), // real refund
    },
    {
      name: 'CreateOrder',
      execute:    () => orders.create(order),
      compensate: () => orders.cancel(order.id),
    },
    {
      name: 'SendConfirmation',
      execute:    () => notifications.sendOrderConfirmed(order),
      compensate: () => notifications.sendOrderCancelled(order), // can't un-send email!
    },
  ];

  const completed: SagaStep[] = [];

  for (const step of steps) {
    try {
      await step.execute();
      completed.push(step);
      logger.info({ saga: 'PlaceOrder', step: step.name }, 'step completed');
    } catch (err) {
      logger.error({ err, step: step.name }, 'step failed — running compensations');

      // Compensate in reverse order
      for (const done of [...completed].reverse()) {
        try {
          await done.compensate();
          logger.info({ step: done.name }, 'compensation succeeded');
        } catch (compErr) {
          // Compensation failed — needs human intervention
          logger.error({ compErr, step: done.name }, 'COMPENSATION FAILED — manual intervention required');
          await alerting.page(`Saga compensation failed for \${done.name}`);
        }
      }
      throw err;
    }
  }
}
// Every execute() and compensate() must be idempotent
// Use idempotency keys so retries don't cause double charges/reservations

Section 08

Production reality

READ COMMITTED is almost always correct for APIs

PostgreSQL's default isolation level is READ COMMITTED, and it's the right choice for the vast majority of API endpoints. It allows maximum concurrency while preventing dirty reads. For endpoints that don't do read-then-update logic, it's both safe and fast. The cases where you need a higher isolation level are specific and identifiable.

SELECT FOR UPDATE for read-then-update

Any time your transaction reads a value and then updates based on that value, you need to prevent another transaction from changing it between your read and write. The correct fix is SELECT ... FOR UPDATE — it acquires a row-level lock at read time, preventing other transactions from updating or locking those rows until your transaction commits. This eliminates lost updates without requiring SERIALIZABLE isolation.

SERIALIZABLE for complex invariants

When your business logic has complex invariants that can't be expressed as a single SELECT FOR UPDATE — for example, "no two doctors can be on-call at the same time" — SERIALIZABLE is the right level. PostgreSQL's Serializable Snapshot Isolation (SSI) detects write-skew anomalies that can violate such invariants, and aborts one of the conflicting transactions. The application must retry aborted transactions.

Long transactions are dangerous

A transaction that stays open for minutes (or hours) causes problems at every layer: it holds row locks (blocking other writes), prevents VACUUM from cleaning dead tuples (causing bloat), and holds an old snapshot (increasing MVCC overhead). Set statement_timeout and idle_in_transaction_session_timeout to catch and kill runaway transactions before they cause cascading problems.

Section 09

The things nobody tells you

TRUNCATE is transactional in PostgreSQL

Unlike in MySQL, TRUNCATE in PostgreSQL is fully transactional. You can BEGIN; TRUNCATE orders; ROLLBACK; and the table will be restored. This is because PostgreSQL implements TRUNCATE by dropping and recreating the underlying storage files, and records this in the WAL. MySQL's TRUNCATE is a DDL operation that implicitly commits — a critical difference if you're coming from MySQL.

SERIALIZABLE doesn't mean "slow"

SSI in PostgreSQL doesn't use locking — it uses conflict detection. Reads don't block writes. Writes don't block reads. The cost of SERIALIZABLE is: tracking read/write sets for conflict detection (memory overhead), and potentially aborting and retrying conflicting transactions (application complexity). For workloads with low conflict rates, SERIALIZABLE throughput approaches READ COMMITTED. The cost only materialises when actual conflicts occur.

FOR UPDATE SKIP LOCKED is a queue primitive

SELECT ... FOR UPDATE SKIP LOCKED selects rows and locks them, skipping any row that is already locked by another transaction. This is exactly what a job queue needs: multiple workers can each grab the next available job without stepping on each other. Used by every serious PostgreSQL-based job queue (pgboss, pg-boss, que, etc.).

Savepoints let you partially rollback

Within a transaction, SAVEPOINT creates a rollback point. If a subsequent operation fails, you can ROLLBACK TO SAVEPOINT sp1 to undo only the work since the savepoint, while keeping the transaction open and the earlier work intact. This is how you implement "try to do X, if it fails keep going" within a single transaction — without closing and reopening the transaction.

Decision sequence

Data anomaly in production — diagnose in this order

Something went wrong with concurrent writes. Work through this before touching anything.

1
Is it a lost update? Check if two transactions both read then wrote the same row
Pull the application logs for the affected rows. Did two concurrent requests both read the old value and write a new one? Lost updates show up as "final value doesn't reflect all the writes that happened." Fix: add SELECT ... FOR UPDATE to the read, or use optimistic locking (WHERE version = $old_version).
lost update = READ COMMITTED + read-then-update without locks
2
Is it a deadlock? Check PostgreSQL logs for ERROR: deadlock detected
PostgreSQL logs deadlocks with full detail — which transactions were involved, which locks they held, which they were waiting for. The log entry is unmistakable. Fix the underlying cause: ensure all code paths acquire locks on the same tables/rows in the same order. Sort by primary key before multi-row operations.
deadlock = lock order inconsistency across code paths
3
Is something blocked? Check pg_stat_activity for waiting queries
Run: SELECT pid, wait_event_type, wait_event, query, now()-query_start AS duration FROM pg_stat_activity WHERE wait_event_type = 'Lock' ORDER BY duration DESC. Waiting queries usually mean a long-running transaction is holding a lock. Find it with pg_blocking_pids() and decide whether to wait or kill it with pg_cancel_backend().
lock wait = find the blocker with pg_blocking_pids()
4
Is it a saga compensation failure? Check the saga log / dead letter queue
In distributed systems, a failed saga mid-way can leave data in a partially updated state. Check your saga execution log (or event store) for sagas that started but didn't reach a terminal state (completed or fully compensated). Each incomplete saga represents a business inconsistency that needs manual or automated compensation.
incomplete saga = compensation log + manual intervention may be required
5
Is it write-skew? Upgrade to SERIALIZABLE
Write-skew is when two transactions each read overlapping data, each decide it's OK to write, and both write — but the combined result violates a business invariant that neither could detect individually. The "on-call doctor" problem. If your anomaly can't be fixed with SELECT FOR UPDATE (because the invariant spans multiple rows), use SERIALIZABLE isolation and implement retry logic for serialization failures (ERROR: could not serialize access due to concurrent update).
write-skew = SERIALIZABLE + retry on serialization failure
6
Is it durability? Check synchronous_commit setting and crash timing
If transactions are apparently committed but data disappears after a crash, check SHOW synchronous_commit. If it's off or local, there's a window where acknowledged commits can be lost on crash. For financial data: synchronous_commit = on is non-negotiable. For non-critical high-throughput writes: off may be acceptable if you understand the loss window.
missing committed data = check synchronous_commit setting immediately

Section 11

Transaction design checklist

SFU
Read-then-write

Any read-then-update pattern needs SELECT FOR UPDATE or optimistic locking. Never read and write in separate transactions.

ORD
Lock ordering

Multi-row operations always sort by primary key. Consistent lock order eliminates deadlocks.

IDL
Idle timeout

Set idle_in_transaction_session_timeout. Open transactions block VACUUM and hold locks indefinitely.

IDM
Saga compensation

Every saga step needs an idempotent compensating transaction. Compensation failures need alerting.

SER
Serializable retry

SERIALIZABLE transactions must retry on serialization failure (error code 40001). Implement with exponential backoff.

DUR
Durability config

synchronous_commit=on for financial/audit data. Document any tables where it's off and accept the loss window.
