Production Incident  ·  PostgreSQL  ·  Lost Update

₹2.3L in incorrect balances — two concurrent refunds, one silent overwrite

Saturday 02:15 IST  ·  Duration: discovered Monday morning  ·  Severity: P0  ·  847 affected accounts

A fintech platform processed a batch of refunds late Friday night. The refund service was horizontally scaled — three worker pods each pulling jobs from a queue. Two pods happened to pick up refund jobs for the same user account at the same time. Both workers ran the same code: read the current balance, add the refund amount, write the new balance.

The bug was invisible in testing because tests never ran two refunds simultaneously. It was invisible in code review because the logic was correct — for a single transaction. The problem was what happened when two correct transactions ran at the same time on the same row.

Friday 02:14:33 IST

Monday 09:30 IST

Monday 11:45 IST

Root cause

Two concurrent transactions both read balance=₹5,200, both computed their new value independently, and both wrote — the second write silently overwrote the first. This is the classic lost update problem at READ COMMITTED isolation. Neither transaction did anything wrong in isolation. The bug only exists in the gap between the read and the write, which the database had no protection against because no row lock was held during that gap.

Three fixes — same problem, different approachesTYPESCRIPT + SQL
// ❌ INCIDENT: read-then-write without any lock
async function refund_broken(userId: string, amount: number) {
  const { balance } = await db.queryOne('SELECT balance FROM wallets WHERE id=$1', [userId]);
  // ← gap: another transaction can read+write between here and the UPDATE
  await db.query('UPDATE wallets SET balance=$1 WHERE id=$2', [balance + amount, userId]);
}

// ✅ FIX 1: SELECT FOR UPDATE — exclusive row lock at read time
async function refund_sfu(userId: string, amount: number) {
  await db.query('BEGIN');
  const { balance } = await db.queryOne(
    'SELECT balance FROM wallets WHERE id=$1 FOR UPDATE', [userId]
  ); // ← Worker B will WAIT here until Worker A commits
  await db.query('UPDATE wallets SET balance=$1 WHERE id=$2', [balance + amount, userId]);
  await db.query('COMMIT');
}

// ✅ FIX 2: Optimistic lock — detect concurrent modification, retry
async function refund_optimistic(userId: string, amount: number) {
  const { balance, version } = await db.queryOne(
    'SELECT balance, version FROM wallets WHERE id=$1', [userId]
  );
  const result = await db.query(
    'UPDATE wallets SET balance=$1, version=version+1 WHERE id=$2 AND version=$3',
    [balance + amount, userId, version]
  );
  if (result.rowCount === 0) throw new Error('Concurrent modification — retry');
  // ← if version changed, UPDATE matches 0 rows → detect and retry
}

// ✅ FIX 3: Atomic UPDATE — no read at all
async function refund_atomic(userId: string, amount: number) {
  await db.query(
    'UPDATE wallets SET balance = balance + $1 WHERE id = $2',
    [amount, userId]
  ); // ← database handles the read-increment-write atomically
   // No race condition possible. Simplest fix when you don't need the old value.
}
What this incident teaches
▶READ COMMITTED doesn't protect read-then-update. Both transactions were individually correct. The bug only exists in the concurrent execution. READ COMMITTED is the right default for most things — but read-then-update is not most things.
▶The atomic UPDATE is the simplest correct fix when you don't need the old value. UPDATE wallets SET balance = balance + $1 is atomic. The database reads and writes in a single operation with no gap. Use this whenever you just need to increment/decrement.
▶SELECT FOR UPDATE is correct when you need the old value in application logic. If you need to compute something based on the current balance (e.g. check it's above zero, apply a cap), read it with FOR UPDATE so no other transaction can modify it between your read and write.
▶Optimistic locking works best when conflicts are rare. It avoids holding locks but requires retry logic. Under high conflict (like batch refunds to the same account), it causes many retries. SELECT FOR UPDATE is more predictable for known-contention scenarios.
