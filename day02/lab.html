ECONNRESET at 3am — no deploy, no traffic change

Wednesday 03:12 IST  ·  Duration: sporadic, recurring  ·  Severity: P2  ·  DB connection pool errors

Nothing changed. No deploy. No traffic spike. No configuration update. At 3:12am, the payments service started throwing ECONNRESET errors on database connections. The connection pool was showing connections as "alive" — but when they were actually used, they were dead. The service was using connections that the network had already silently discarded.

It happened again at 3:58am. And at 4:44am. Exactly 46 minutes apart. This regularity was the clue — though the team didn't spot it until the next morning. Something was killing connections after exactly 350 seconds of inactivity.

03:12 IST
First wave of ECONNRESET errors from DB pool. ~12 errors in 30 seconds, then clears as pool reconnects.
03:58 IST
Second wave. Same pattern. Pool shows connections as valid — but they fail on first use. Engineers wake up.
04:12 IST
DB checked — healthy, no issues. Network team checked — no packet drops. Postgres logs show connections closed from client side.
04:44 IST
Third wave. Engineer notes: exactly 46 minutes between events. 46 × 60 = 2760 seconds. No obvious significance.
09:20 IST
Senior engineer reviews. Checks AWS documentation. NAT Gateway drops idle connections after 350 seconds. The DB pool's idle connections exceed 350s between 3am-5am (low traffic). NAT drops them silently. Pool thinks they're alive.
11:45 IST
Fix: tcp_keepalive_time=60 on all EC2 instances. DB pool validation query added. Errors stop permanently.
Root cause

AWS NAT Gateway silently drops TCP connections idle for more than 350 seconds. The Linux default TCP keepalive is 7200 seconds (2 hours) — meaning the OS never sent keepalive probes before NAT dropped the connection. The DB connection pool had no way to know — from its perspective the connections were established and valid. The 46-minute pattern: at 3am traffic drops, connections go idle, NAT kills them at 350s, first requests fail, pool reconnects, connections go idle again, repeat.

The sysctl settings that prevented this foreverBASH
# Check current values (run on your EC2 instance)
sysctl net.ipv4.tcp_keepalive_time
sysctl net.ipv4.tcp_keepalive_intvl
sysctl net.ipv4.tcp_keepalive_probes

❌ Linux defaults — breaks on AWS NAT Gateway
net.ipv4.tcp_keepalive_time   = 7200   # 2 hours before first probe
net.ipv4.tcp_keepalive_intvl  = 75     # 75s between probes
net.ipv4.tcp_keepalive_probes = 9      # 9 failed probes before ECONNRESET
# NAT kills at 350s. First keepalive probe at 7200s. Too late.

✅ Fixed values — set in /etc/sysctl.d/99-keepalive.conf
net.ipv4.tcp_keepalive_time   = 60    # first probe after 60s idle
net.ipv4.tcp_keepalive_intvl  = 10    # probe every 10s if no response
net.ipv4.tcp_keepalive_probes = 3     # 3 failed probes = dead
# Dead connection detected in 60 + (10×3) = 90s. Well before NAT's 350s.
# Apply without reboot:
sysctl -p /etc/sysctl.d/99-keepalive.conf

# Also: add pool-level validation (defense in depth)
# pg (node-postgres):
const pool = new Pool({
  connectionString: process.env.DATABASE_URL,
  idleTimeoutMillis:    30_000,  // evict idle connections after 30s
  connectionTimeoutMillis: 5_000,
  keepAlive:        true,        // socket-level keepalive
  keepAliveInitialDelayMillis: 10_000,
});
What this incident teaches
▶AWS NAT Gateway has a 350-second idle connection timeout. It's documented, but nobody reads it until an incident. Any service behind a NAT that holds idle DB/Redis connections beyond 350s will see this.
▶The Linux default 7200s keepalive is wrong for cloud environments. It predates cloud NAT gateways. Always set tcp_keepalive_time to 60s on any cloud VM.
▶A regular pattern in errors always means a timer. 46 minutes. 350 seconds. When your errors arrive on a clock, you're looking for a timeout value — not a load issue.
▶Connection pools need both OS-level keepalive and pool-level idle timeout. OS keepalive keeps the socket alive. Pool idle timeout evicts connections that haven't been used, forcing fresh reconnects.
