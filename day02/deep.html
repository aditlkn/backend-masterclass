Deep Dive 01

Socket buffer sizing
& throughput limits

TCP's throughput is bounded by the receive buffer size divided by the round-trip time. This is the Bandwidth-Delay Product — and if your buffers are too small, you're leaving throughput on the table regardless of how fast your network or CPU is.

The Bandwidth-Delay Product (BDP)

TCP uses a sliding window: the sender can have at most window_size bytes in-flight before it must wait for an acknowledgment. The maximum throughput is therefore:

max_throughput = window_size / RTT

If your receive buffer is 87KB (Linux default) and your RTT to a client is 50ms: 87KB / 0.05s = 1.74 MB/s. On a 1Gbps network you're capped at 1.74 MB/s per connection — not because of bandwidth, but because of buffer size. This is the BDP problem, and it dominates large file transfers and bulk data streaming even on high-bandwidth networks.

For intra-datacenter connections with 0.1ms RTT, the default 87KB buffer gives 870 MB/s — more than enough. The BDP problem only manifests for connections with high RTT (cross-region, client-facing).

Linux TCP auto-tuning

Modern Linux (2.6.17+) auto-tunes socket buffers within configurable limits. The kernel starts with a small buffer and grows it based on observed RTT and available memory. The critical parameters are the maximum values — the ceiling the auto-tuner can grow to. The defaults (4MB receive max, 4MB send max) are too low for high-throughput cross-region scenarios.

Socket buffer tuning — calculate BDP and set correct limitsBASH
# Step 1: measure your actual RTT to key destinations
ping -c 20 api.partner.com | tail -1
# rtt min/avg/max/mdev = 18.2/19.4/21.1/0.8 ms

# Step 2: calculate BDP for your target throughput
# Target: 100 MB/s on a 20ms RTT connection
# BDP = bandwidth × RTT = 100 MB/s × 0.020s = 2 MB per connection
# Set max buffer to 4× BDP for headroom: 8MB

# Step 3: set buffer limits
# /etc/sysctl.d/99-tcp-buffers.conf

# Receive buffer: min / default / max
net.ipv4.tcp_rmem = 4096 87380 8388608   # max: 8MB

# Send buffer: min / default / max
net.ipv4.tcp_wmem = 4096 65536 8388608   # max: 8MB

# Global socket memory limit (all sockets combined)
net.core.rmem_max = 8388608
net.core.wmem_max = 8388608

# Enable auto-tuning (should already be on)
net.ipv4.tcp_moderate_rcvbuf = 1

# Apply without reboot
sysctl -p /etc/sysctl.d/99-tcp-buffers.conf

# Verify: watch buffer growth on an active connection
ss -itm dst <remote_ip>
# Look for: rcv_space — this grows as auto-tuning kicks in
# If rcv_space hits your max and throughput plateaus: increase max

⚠️
Docker and Kubernetes don't inherit host sysctl

Container workloads don't automatically use host sysctl values for network settings. In Kubernetes, set pod-level sysctl via securityContext.sysctls for safe sysctls (net.ipv4.tcp_keepalive_time etc.), or configure them at the node level via DaemonSet for unsafe ones (net.core.somaxconn). Always verify inside a running pod with kubectl exec -it <pod> -- sysctl net.core.somaxconn.

Deep Dive 03

TIME_WAIT at scale:
port exhaustion math

TIME_WAIT is the state a TCP socket enters after closing an outbound connection. It lasts for 2× MSL (Maximum Segment Lifetime) — 60-120 seconds on most systems. For services that make many short-lived outbound connections (HTTP without keep-alive, database connections that close frequently), TIME_WAIT accumulates and eventually exhausts the ephemeral port range.

When TIME_WAIT becomes a problem

The ephemeral port range is the pool of local ports available for outbound connections. Default on Linux: ports 32768-60999 = 28,231 ports. Each TIME_WAIT socket holds one port for up to 60 seconds. If you're closing connections faster than ports are freed, you run out.

The math: if you close 500 connections per second and each TIME_WAIT lasts 60 seconds, you accumulate 500 × 60 = 30,000 TIME_WAIT sockets. The default port range holds ~28,000. At 500 closures/second you will exhaust ports — every new outbound connection attempt returns EADDRNOTAVAIL: Cannot assign requested address.

The solutions in order of preference

  1. Fix the root cause: use connection pooling. Keep-alive connections don't cycle through TIME_WAIT because they're not closed between requests. A pool of 20 persistent connections handles 10,000 requests without any TIME_WAIT accumulation. This is always the right first fix.
  2. Expand the port range: net.ipv4.ip_local_port_range = 1024 65535 gives ~64,000 ports instead of ~28,000. Doubles your capacity with no downsides.
  3. Enable tcp_tw_reuse: Allows the kernel to reuse a TIME_WAIT socket for a new outbound connection to the same destination if it's safe to do so (the existing socket's data has been flushed). Safe for outbound connections. Never enable tcp_tw_recycle (removed in Linux 4.12, was dangerous).
  4. Reduce fin_timeout: net.ipv4.tcp_fin_timeout = 15 reduces TIME_WAIT duration from 60s to 15s, freeing ports 4× faster.
🔴
Never use tcp_tw_recycle

net.ipv4.tcp_tw_recycle was removed in Linux 4.12 because it caused connection failures for clients behind NAT (which is most of your users). It used per-destination timestamp tracking that broke when multiple clients shared an IP. If you see it in legacy configs, remove it. It's not available in modern kernels, and if it were, it would break NAT'd traffic.

Diagnosing and monitoring TIME_WAIT accumulationBASH
# Current TIME_WAIT socket count
ss -s | grep TIME-WAIT
# or: cat /proc/net/sockstat | grep TCP

# Watch it in real time
watch -n1 'ss -s | grep TIME-WAIT'

# List all TIME_WAIT sockets with remote addresses
ss -tan state time-wait | head -20

# Check current ephemeral port range
cat /proc/sys/net/ipv4/ip_local_port_range
# → 32768   60999  (default: 28K ports)

# Calculate TIME_WAIT risk for your connection rate
# connections_per_second × fin_timeout = TIME_WAIT accumulation
# e.g.: 200/s × 60s = 12,000 TIME_WAIT sockets (fine with 28K port range)
# e.g.: 600/s × 60s = 36,000 TIME_WAIT sockets (exceeds 28K — port exhaustion!)

# Alert threshold: TIME_WAIT count > 80% of port range
# 80% of 28231 = 22584 — set alert here, not at exhaustion

# If you hit EADDRNOTAVAIL — confirm it's port exhaustion
dmesg | grep "connect: Cannot assign requested address"
# If this appears: you're exhausted. Apply fixes immediately.
Verify in production
✓Monitor TIME_WAIT count continuously: ss -s | grep TIME-WAIT. Alert at 80% of port range.
✓Port range is expanded: sysctl net.ipv4.ip_local_port_range should show 1024 65535 not the default.
✓Connection pool is actually working: TIME_WAIT count should be low (hundreds, not thousands) if keep-alive is enabled.
✓No EADDRNOTAVAIL in application logs or dmesg. If present, fix immediately — connections are silently failing.
✓tcp_tw_recycle is NOT set: sysctl net.ipv4.tcp_tw_recycle should return "no such file" on modern kernels.
