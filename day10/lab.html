Production Incident  ·  PostgreSQL  ·  Replication Slot

Orphaned replication slot filled the disk — primary crashed at 3am

Tuesday 03:47 IST  ·  Duration: 2h 15m  ·  Severity: P0  ·  Full database outage

Three weeks earlier, the team had set up a Debezium CDC connector to stream database changes to Kafka. The connector had a dedicated PostgreSQL replication slot: debezium_connector. The Debezium setup worked well for a week — then the Kafka cluster ran out of disk, the connector failed, and the team paused the connector to fix Kafka. They fixed Kafka. They forgot to restart Debezium. They definitely forgot about the replication slot.

For three weeks, debezium_connector sat in pg_replication_slots with active = false. The primary dutifully retained every WAL segment generated since the slot last consumed data — because the slot might reconnect and need it. Nobody was watching. The WAL directory grew from 2GB to 380GB. At 3:47am, the data volume filled completely. PostgreSQL crashed.

Root cause

An inactive replication slot (active = false) prevents PostgreSQL from recycling WAL segments. The slot holds a restart_lsn — the WAL position it last acknowledged — and the primary retains all WAL from that position forward, indefinitely, until the slot either consumes it or is dropped. Three weeks of WAL accumulation at 17GB/day filled a 400GB volume. The fix: drop the slot. The prevention: alert on retained WAL bytes per slot.

The monitoring query that would have prevented thisSQL
❌ No monitoring: discovered at 3am when disk was full

✅ Add to your monitoring dashboard (run every 5 minutes)
SELECT
  slot_name,
  active,
  pg_size_pretty(
    pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn)
  ) AS retained_wal,
  pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn) AS retained_bytes,
  now() - (CASE WHEN active THEN now() ELSE pg_postmaster_start_time() END) AS inactive_for
FROM pg_replication_slots
ORDER BY retained_bytes DESC;

-- Alert condition (Prometheus / Datadog / CloudWatch):
SELECT slot_name, retained_bytes
FROM pg_replication_slots
WHERE pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn) > 5368709120;  -- 5GB
-- Page on-call immediately. Don't wait until disk is 90% full.

-- Fix: drop the orphaned slot
SELECT pg_drop_replication_slot('debezium_connector');
-- Instantly frees all WAL retained by the slot
-- Debezium will need to do a full resync when restarted
What this incident teaches
▶Replication slots are not automatically cleaned up. When a consumer goes away — Debezium fails, a replica is decommissioned, a subscriber is paused — the slot stays and WAL accumulates. There is no automatic expiry.
▶Alert on slot WAL retained bytes, not just disk usage. By the time disk alerts fire, you might have minutes before the crash. Alert on retained_bytes > 5GB to catch it days early.
▶Operational runbook: always drop slots when decommissioning consumers. Any time you pause a Debezium connector, remove a replica, or stop a logical subscriber permanently — drop its replication slot immediately. Make it a checklist item.
▶wal_keep_size is safer than slots for read replicas. For physical standby replicas, consider using wal_keep_size instead of replication slots. If the replica falls too far behind, it can't reconnect (acceptable — re-initialise). But at least the primary's disk doesn't fill.
