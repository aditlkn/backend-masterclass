Deep Dive 01

Full Patroni configuration:
every parameter explained

Patroni is the most widely used PostgreSQL HA framework. Its YAML configuration is dense and its parameters interact in non-obvious ways. Here's a complete production configuration with every setting annotated.

Complete patroni.yml — production-ready, fully annotatedYAML
# /etc/patroni/patroni.yml
# Run on EVERY node in the cluster (primary + all replicas)
# Each node has its own file with its own name/address

scope: my-postgres-cluster     # cluster name — must be same on all nodes
namespace: /db/                # etcd key prefix for this cluster's data
name: postgres-node-1          # THIS node's name — unique per node

# ── Distributed Configuration Store (DCS) ──────────────────────
etcd3:                         # or: etcd, consul, zookeeper
  hosts:
    - etcd1.internal:2379
    - etcd2.internal:2379
    - etcd3.internal:2379      # 3-node etcd for quorum. Never use 1 node.
  ttl: 30                      # leader key TTL in seconds
                               # if primary misses heartbeat for 30s, key expires
  loop_wait: 10                # heartbeat interval (must be < ttl)
  retry_timeout: 10            # timeout for DCS operations

# ── REST API (for patronictl and monitoring) ────────────────────
restapi:
  listen: 0.0.0.0:8008         # Patroni REST API port
  connect_address: 10.0.0.1:8008  # THIS node's address for the REST API
  authentication:
    username: patroni
    password: ${PATRONI_API_PASSWORD}

# ── Bootstrap (first-time cluster initialization) ──────────────
bootstrap:
  dcs:                         # DCS-stored cluster config (shared, not per-node)
    ttl: 30
    loop_wait: 10
    retry_timeout: 10
    maximum_lag_on_failover: 1048576  # 1MB: don't promote replica lagging more than this
    postgresql:
      use_pg_rewind: true      # use pg_rewind to repair old primary on rejoin
      use_slots: true          # use replication slots (ensures no WAL loss)
      parameters:
        wal_level: replica
        hot_standby: "on"
        wal_keep_size: 1024    # 1GB fallback WAL retention
        max_wal_senders: 10
        max_replication_slots: 10
        wal_log_hints: "on"    # required for pg_rewind
        synchronous_commit: "on"
        synchronous_standby_names: "ANY 1 (postgres-node-2, postgres-node-3)"

  initdb:                      # only used for first-time cluster init
    - encoding: UTF8
    - data-checksums           # required for pg_rewind. Always enable.

  pg_hba:                      # access control — applied to all nodes
    - host replication replicator 10.0.0.0/24 scram-sha-256
    - host all all 10.0.0.0/24 scram-sha-256

# ── PostgreSQL settings (per-node) ─────────────────────────────
postgresql:
  listen: 0.0.0.0:5432
  connect_address: 10.0.0.1:5432  # THIS node's address for replication
  data_dir: /var/lib/postgresql/data
  bin_dir: /usr/lib/postgresql/16/bin
  pgpass: /tmp/pgpass            # credentials file for replication

  authentication:
    replication:
      username: replicator
      password: ${REPLICATION_PASSWORD}
    superuser:
      username: postgres
      password: ${POSTGRES_PASSWORD}

  parameters:
    max_connections: 200
    shared_buffers: 4GB
    work_mem: 16MB
    log_min_duration_statement: 1000  # log queries > 1s
    log_connections: "on"
    log_disconnections: "on"

  callbacks:                   # shell scripts called on role changes
    on_start:    /etc/patroni/callbacks/on_start.sh
    on_stop:     /etc/patroni/callbacks/on_stop.sh
    on_restart:  /etc/patroni/callbacks/on_restart.sh
    on_role_change: /etc/patroni/callbacks/on_role_change.sh
    # on_role_change gets called with "master" or "replica" argument
    # Use to: update DNS, send alerts, reconfigure connection pooler

# ── Tags (for read traffic routing) ────────────────────────────
tags:
  nofailover: false            # set true to exclude from failover candidates
  noloadbalance: false         # set true for analytics replica (don't route user traffic)
  clonefrom: false             # prefer this node as source for new replica
  nosync: false                # exclude from synchronous standby list

Key Patroni commands

patronictl — daily operationsBASH
# View cluster state
patronictl -c /etc/patroni/patroni.yml list
# + Cluster: my-postgres-cluster (7234567890) --+----+-----------+
# | Member          | Host        | Role    | State   | TL | Lag in MB |
# +-----------------+-------------+---------+---------+----+-----------+
# | postgres-node-1 | 10.0.0.1:5432 | Leader | running |  1 |           |
# | postgres-node-2 | 10.0.0.2:5432 | Replica| running |  1 |         0 |
# | postgres-node-3 | 10.0.0.3:5432 | Replica| running |  1 |         0 |

# Planned switchover (primary → replica, no downtime)
patronictl -c /etc/patroni/patroni.yml switchover \
  --master postgres-node-1 \
  --candidate postgres-node-2 \
  --scheduled now

# Reinitialise a replica from scratch
patronictl -c /etc/patroni/patroni.yml reinit my-postgres-cluster postgres-node-3

# Edit cluster DCS configuration (changes apply to all nodes)
patronictl -c /etc/patroni/patroni.yml edit-config

# Pause automatic failover (for maintenance)
patronictl -c /etc/patroni/patroni.yml pause --wait
# Resume:
patronictl -c /etc/patroni/patroni.yml resume --wait

# Restart PostgreSQL on a node (Patroni-managed)
patronictl -c /etc/patroni/patroni.yml restart my-postgres-cluster postgres-node-1
Verify Patroni is working correctly
✓patronictl list shows all nodes as "running" with "Lag in MB: 0" on replicas. Any positive lag persisting for minutes needs investigation.
✓REST API health check returns 200 for leader, 503 for replicas (by design): curl -s http://10.0.0.1:8008/leader
✓Test failover quarterly: patronictl switchover and verify traffic routes correctly, then patronictl switchover back.
✓data-checksums enabled: psql -c "SHOW data_checksums" should return "on" — required for pg_rewind.
Deep Dive 02

pg_rewind: repairing a failed primary
without a full backup

After a failover, the old primary has WAL that diverges from the new primary's timeline. pg_rewind efficiently repairs this by finding the divergence point and syncing only the changed files. For a 1TB database, this takes minutes instead of hours (pg_basebackup time).

When pg_rewind is needed

After any unplanned failover: the old primary ran ahead of the new timeline and accumulated some writes that aren't on the new primary. It cannot simply be added back as a replica — PostgreSQL will refuse to follow a timeline it has diverged from. pg_rewind identifies the last common WAL position, rolls back the old primary to that point, and prepares it to stream from the new primary.

Prerequisites

  • wal_log_hints = on in postgresql.conf, or data checksums enabled. Without one of these, pg_rewind cannot determine which blocks have changed.
  • A replication-capable user with REPLICATION privilege (or superuser).
  • The new primary must still have WAL from the divergence point (not yet recycled).
pg_rewind — complete procedure after failoverBASH
# Situation: postgres-node-1 was primary, crashed.
# postgres-node-2 was promoted. postgres-node-1 is now back online.
# postgres-node-1 has diverged from postgres-node-2's timeline.

# Step 1: Stop PostgreSQL on the old primary (node-1)
pg_ctl stop -D /var/lib/postgresql/data -m fast
# Must be stopped. pg_rewind cannot run on a live server.

# Step 2: Run pg_rewind (pointing at new primary as source)
pg_rewind \
  --target-pgdata=/var/lib/postgresql/data \
  --source-server="host=10.0.0.2 port=5432 user=replicator dbname=postgres" \
  --progress \
  --no-sync
# --progress: shows which files are being synced
# --no-sync: skip fsync (faster for initial rewind; do a sync yourself)
# --dry-run: preview what would be changed without doing it

# What pg_rewind does internally:
# 1. Reads WAL on target (old primary) to find divergence LSN
# 2. Connects to source (new primary) to get block-level diff
# 3. Copies only the blocks that differ (much less than full backup)
# 4. Copies WAL from source from divergence point forward

# Step 3: Configure as replica of new primary
cat >> /var/lib/postgresql/data/postgresql.conf << 'EOF'
primary_conninfo = 'host=10.0.0.2 port=5432 user=replicator'
recovery_target_timeline = 'latest'
EOF

# Create standby.signal to enter recovery mode
touch /var/lib/postgresql/data/standby.signal

# Step 4: Start PostgreSQL — it will stream from new primary
pg_ctl start -D /var/lib/postgresql/data
# Check it's streaming:
psql -c "SELECT status, sender_host FROM pg_stat_wal_receiver"
# → streaming | 10.0.0.2

# Step 5: Verify replication lag on new primary
psql -h 10.0.0.2 -c "SELECT application_name, replay_lag FROM pg_stat_replication"
# lag should reach 0 after catch-up

# ── If pg_rewind fails: fallback to pg_basebackup ──────────────
# This happens when new primary has recycled WAL from divergence point
rm -rf /var/lib/postgresql/data/*
pg_basebackup \
  -h 10.0.0.2 -U replicator \
  -D /var/lib/postgresql/data \
  -P -Xs -R
# -R: write recovery config automatically (standby.signal + primary_conninfo)
# Slower (copies entire DB) but always works
🔴
pg_rewind cannot recover from timeline gap without WAL archives

pg_rewind needs WAL from the divergence point through the promotion on the new primary. If the new primary has already recycled those WAL segments (and you have no WAL archive), pg_rewind fails. This is why wal_keep_size and/or WAL archiving are important — they give pg_rewind enough WAL to work with. If pg_rewind fails, pg_basebackup is your fallback.

Deep Dive 03

Logical replication pub/sub
& CDC with wal2json

Logical replication uses a publication/subscription model to replicate specific tables at the row level. This section covers the full setup — publisher to subscriber — plus the wal2json plugin for Change Data Capture, which streams change events to external consumers (Kafka, Elasticsearch, etc.).

Built-in logical replication (PostgreSQL → PostgreSQL)

Publication and subscription setup — full procedureSQL
-- ── PUBLISHER (source database) ────────────────────────────────
-- Requires: wal_level = logical (higher than replica)
ALTER SYSTEM SET wal_level = logical;
SELECT pg_reload_conf();

-- Create a publication for specific tables
CREATE PUBLICATION orders_pub
FOR TABLE orders, order_items, payments;
-- Replicates: INSERT, UPDATE, DELETE on these tables
-- Does NOT replicate: DDL (ALTER TABLE, CREATE INDEX, etc.)

-- Or: publish all tables in the database
CREATE PUBLICATION all_tables_pub FOR ALL TABLES;

-- Or: publish with row filtering (PG 15+)
CREATE PUBLICATION active_orders_pub
FOR TABLE orders (id, user_id, status, amount)
WHERE (status != 'archived');
-- Only replicate non-archived orders, and only these columns

-- View publications
SELECT * FROM pg_publication;
SELECT * FROM pg_publication_tables;

-- Create replication user for subscriber to connect as
CREATE USER logical_repl WITH REPLICATION PASSWORD 'password';
GRANT SELECT ON ALL TABLES IN SCHEMA public TO logical_repl;

-- ── SUBSCRIBER (destination database) ──────────────────────────
-- Target tables must exist with matching schema
-- (schema not replicated — must create tables manually)

CREATE SUBSCRIPTION orders_sub
CONNECTION 'host=source-db.internal dbname=app user=logical_repl password=...'
PUBLICATION orders_pub;
-- PostgreSQL automatically:
-- 1. Takes initial snapshot (copies existing data)
-- 2. Switches to streaming new changes

-- Monitor subscription lag
SELECT
  subname,
  received_lsn,
  latest_end_lsn,
  pg_wal_lsn_diff(latest_end_lsn, received_lsn) AS lag_bytes
FROM pg_stat_subscription;

-- Operations on subscriptions
ALTER SUBSCRIPTION orders_sub DISABLE;    -- pause
ALTER SUBSCRIPTION orders_sub ENABLE;     -- resume
ALTER SUBSCRIPTION orders_sub REFRESH PUBLICATION;  -- pick up new tables added to publication
DROP SUBSCRIPTION orders_sub;            -- removes slot on publisher too

-- !! Schema changes must be applied manually to subscriber
-- If you ALTER TABLE orders ADD COLUMN on publisher:
-- 1. Apply same ALTER TABLE on subscriber
-- 2. ALTER SUBSCRIPTION orders_sub REFRESH PUBLICATION

CDC with wal2json — streaming to external systems

wal2json is a PostgreSQL logical decoding plugin that converts WAL records to JSON change events. It's the foundation for Debezium, which wraps it with Kafka integration, schema registry support, and offset management.

wal2json — direct CDC without DebeziumBASH + SQL
# ── Setup (requires wal2json extension installed) ───────────────
# wal_level = logical required

# Create a logical replication slot with wal2json output
psql -c "SELECT pg_create_logical_replication_slot('cdc_slot', 'wal2json')"

# Peek at changes (non-destructive — doesn't advance the slot)
psql -c "SELECT * FROM pg_logical_slot_peek_changes('cdc_slot', NULL, NULL,
  'pretty-print', '1',
  'include-timestamp', '1',
  'include-lsn', '1')"

# Sample wal2json output:
# {
#   "lsn": "0/3A2F1C0",
#   "timestamp": "2024-01-15 03:14:22.847392+05:30",
#   "change": [{
#     "kind":      "update",
#     "schema":    "public",
#     "table":     "orders",
#     "columnnames": ["id","status","updated_at"],
#     "columnvalues": [123, "shipped", "2024-01-15T03:14:22"],
#     "oldkeys": {"keynames": ["id"], "keyvalues": [123]}
#   }]
# }

# Consume and advance the slot (destructive — marks WAL as consumed)
psql -c "SELECT * FROM pg_logical_slot_get_changes('cdc_slot', NULL, NULL)"

# ── Node.js CDC consumer (streams changes to Kafka) ─────────────
import { Client } from 'pg';
import { Kafka } from 'kafkajs';

const pg    = new Client({ connectionString: process.env.DATABASE_URL });
const kafka = new Kafka({ brokers: ['kafka:9092'] });
const producer = kafka.producer();

async function streamChanges() {
  await pg.connect();
  await producer.connect();

  while (true) {
    const { rows } = await pg.query(`
      SELECT lsn, data FROM pg_logical_slot_get_changes(
        'cdc_slot', NULL, 100,  -- batch 100 changes at a time
        'include-timestamp', '1'
      )`);

    for (const row of rows) {
      const change = JSON.parse(row.data);
      await producer.send({
        topic: `db.changes.\${change.change[0]?.table || 'unknown'}`,
        messages: [{ key: row.lsn, value: row.data }],
      });
    }

    if (rows.length === 0) await new Promise(r => setTimeout(r, 100));
  }
}
// Always: monitor slot retained bytes, drop slot on consumer decommission
Verify logical replication health
✓Subscription lag is near-zero: SELECT subname, latest_end_lsn - received_lsn AS lag FROM pg_stat_subscription
✓All expected tables are included: SELECT * FROM pg_publication_tables WHERE pubname = 'orders_pub'
✓CDC slot retained bytes alerting: SELECT slot_name, pg_size_pretty(pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn)) FROM pg_replication_slots WHERE slot_type='logical'
✓Schema changes handled: after any DDL on publisher, verify schema matches on subscriber and refresh subscription if new tables added.
✓Replica identity set for UPDATE/DELETE tracking: ALTER TABLE orders REPLICA IDENTITY FULL — without this, UPDATEs may not include old column values in CDC events.
