Section 01

The Write-Ahead Log: what replication actually ships

Replication doesn't copy SQL statements. It doesn't copy rows. It copies the Write-Ahead Log — the low-level record of every change made to every data page in the database. Understanding what the WAL is explains why replication works the way it does, why replicas have lag, and what breaks when the WAL can't be consumed fast enough.

What the WAL contains

Every time a transaction modifies data in PostgreSQL, before changing the actual data page, it writes a WAL record describing the change. A WAL record says something like: "At offset 847 on page 3 of file base/16384/2619, change bytes 120-136 from [old bytes] to [new bytes]." This is a physical, byte-level description of the change — not a SQL statement, not a row value, but the actual bytes that changed on a data page.

The WAL is organised as a sequence of 16MB segment files, each identified by a log sequence number (LSN). The LSN is a monotonically increasing 64-bit integer that identifies a specific position in the WAL stream. The primary's current LSN tells you how far ahead the primary is relative to any replica.

WAL and durability

The WAL is what makes transactions durable. A transaction isn't considered committed until its WAL record is flushed to disk. On crash recovery, PostgreSQL reads the WAL from the last checkpoint forward, replays all committed WAL records, and rolls back any uncommitted ones. The data pages on disk may be behind the WAL (dirty pages that weren't flushed before the crash) — but the WAL is complete and can bring the data pages up to date.

WAL and replication

Streaming replication works by shipping the WAL from the primary to replicas in real time. The replica applies WAL records to its own data pages, keeping them in sync with the primary. The replica is always some number of WAL bytes behind the primary — this is replication lag, measured either in bytes or in time.

Inspecting the WAL and replication positionSQL
-- Current WAL position on primary
SELECT pg_current_wal_lsn();
-- → 0/3A2F1B8  (LSN: file 0, byte offset 0x3A2F1B8 = 61,017,016)

-- Size of WAL generated since a point in time
SELECT pg_wal_lsn_diff(pg_current_wal_lsn(), '0/3A2F1B8') AS bytes_generated;

-- Replication lag per replica (run on primary)
SELECT
  application_name,
  client_addr,
  state,
  sent_lsn,
  write_lsn,
  flush_lsn,
  replay_lsn,
  pg_wal_lsn_diff(pg_current_wal_lsn(), replay_lsn) AS lag_bytes,
  write_lag,     -- time to write WAL on replica
  flush_lag,     -- time to flush WAL to replica disk
  replay_lag     -- time to replay WAL on replica (the full lag)
FROM pg_stat_replication
ORDER BY lag_bytes DESC;

-- WAL segment files on disk
SELECT * FROM pg_ls_waldir() ORDER BY modification DESC LIMIT 10;
-- Each file is 16MB. Many files = primary is running ahead of replicas

-- Check WAL archiving status
SELECT archived_count, failed_count, last_archived_wal, last_failed_wal
FROM pg_stat_archiver;

Section 02

Streaming replication: how it works end to end

PostgreSQL streaming replication is built on a persistent TCP connection between the primary and each replica. The replica connects to the primary using a replication protocol (not the standard SQL protocol) and starts receiving WAL records as they're generated.

The walsender / walreceiver pair

On the primary, a walsender process is spawned for each connected replica. The walsender reads WAL records as they're written and sends them over the TCP connection to the replica. On the replica, a walreceiver process receives the WAL records and writes them to the replica's WAL directory. A separate startup process on the replica then applies (replays) the WAL records to the data files.

This separation between receiving WAL and applying WAL is important: a replica might have received WAL up to LSN X, but only applied WAL up to LSN X-5MB. The difference represents WAL that's been written to the replica's disk but not yet replayed to its data pages. This means the replica has the data durably but hasn't made it queryable yet.

Replication slots

Without replication slots, the primary can recycle WAL segments once it thinks all replicas have consumed them. If a replica falls behind — due to network issues, replica overload, or being taken offline for maintenance — the primary might delete WAL that the replica still needs. The replica then can't reconnect and must be re-initialised from a full base backup.

Replication slots prevent this: the primary retains WAL until the slot's consumer has confirmed it's been applied. The downside: if the consumer goes away permanently (a replica is decommissioned but the slot isn't dropped), WAL accumulates indefinitely, filling the disk. Always drop replication slots for replicas you're permanently removing.

Setting up streaming replication — primary and replica configBASH
# ── PRIMARY: postgresql.conf ──────────────────────────────
wal_level = replica         # minimum for streaming replication
max_wal_senders = 5         # max concurrent replicas
wal_keep_size = 1GB         # keep at least 1GB of WAL for late replicas
                            # (alternative: use replication slots)

# ── PRIMARY: pg_hba.conf — allow replica to connect ───────
# TYPE  DATABASE    USER         ADDRESS         METHOD
host    replication replicator   10.0.0.2/32     scram-sha-256

# ── PRIMARY: create replication user ──────────────────────
CREATE USER replicator WITH REPLICATION PASSWORD 'strong-password';

# ── REPLICA: take base backup from primary ─────────────────
pg_basebackup \
  --host=10.0.0.1 \
  --username=replicator \
  --pgdata=/var/lib/postgresql/data \
  --wal-method=stream \
  --checkpoint=fast \
  --progress \
  --verbose

# ── REPLICA: postgresql.conf ──────────────────────────────
# Create standby.signal file (PG 12+) to put replica in standby mode
touch /var/lib/postgresql/data/standby.signal

# primary_conninfo in postgresql.conf or recovery.conf (PG 11-)
primary_conninfo = 'host=10.0.0.1 port=5432 user=replicator password=...'
recovery_target_timeline = 'latest'

# ── REPLICA: monitor replication status (run on replica) ──
SELECT status, received_lsn, last_msg_send_time, last_msg_receipt_time
FROM pg_stat_wal_receiver;

# ── Create a replication slot (primary) ───────────────────
SELECT pg_create_physical_replication_slot('replica1_slot');

# ── Monitor all replication slots ─────────────────────────
SELECT slot_name, active, pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn) AS retained_bytes
FROM pg_replication_slots;
# Alert if retained_bytes > 10GB — slot consumer is too far behind
# Drop orphaned slots: SELECT pg_drop_replication_slot('slot_name');

Section 03

Replication lag: causes, measurement, and safe replica reads

Replication lag is the delay between a write being committed on the primary and that write being visible on a replica. It's unavoidable — even with synchronous replication, there's at least the network round-trip time. The question is: how much lag do you have, what's causing it, and what does your application need to do about it?

What causes lag

  • Network latency: WAL must travel from primary to replica. A cross-region replica has baseline lag equal to the RTT. A 100ms RTT adds at least 100ms of lag to every transaction.
  • Replica I/O bottleneck: The replica must write WAL to its own disk and apply it to data pages. If the replica's disk is slow (or being used for read queries that generate lots of I/O), it falls behind.
  • CPU-intensive WAL replay: Some operations generate WAL that's expensive to replay — large index builds, bulk updates that touch many pages. The replica's WAL applier can fall behind during these bursts.
  • Long-running queries on the replica blocking WAL apply: By default, PostgreSQL does not cancel replica queries when WAL replay needs to modify the same rows. This is max_standby_streaming_delay — the replica will wait up to this long for conflicting queries to finish before applying the WAL record that conflicts with them.

Reading from replicas safely

A read replica may be milliseconds to seconds behind the primary. For most reads this is acceptable — reading yesterday's orders from a replica that's 200ms behind is fine. But some reads are unsafe on lagging replicas:

  • Read-your-own-writes: User submits a form, you redirect to a page that reads the data they just submitted. If the page read goes to a lagging replica, the user sees a blank page or stale data. Fix: route reads to the primary immediately after a write, or implement session-level read consistency.
  • Causal consistency: Write A must be visible before write B can be processed. If B's dependent read goes to a lagging replica that hasn't applied A yet, B sees inconsistent data.
  • Counters and aggregates: SELECT COUNT(*) on a replica that's lagging will under-count. For exact counts after recent writes, go to the primary.

Section 04

Read replicas: scaling reads and the routing problem

A read replica is a replica that serves SELECT queries. The primary handles all writes; replicas absorb some fraction of the read traffic. This is the most common use of replication in production — not primarily for high availability, but for read scalability.

What to route where

Route to primary: Any read immediately after a write from the same user session. Any read that requires counting recently-inserted rows exactly. Any read used as part of a write decision (SELECT FOR UPDATE, exists check before insert). Any read where 100ms stale data would cause a user-visible problem.

Route to replicas: Analytics and reporting queries (often the biggest load). Historical data reads. Search queries. Any read where the user won't notice 1-2 second staleness. Background job queries.

Session consistency: tracking your own writes

The standard pattern for "read your own writes" is to track the LSN of the last write in the user's session, and when routing a read to a replica, only send it to a replica whose replay_lsn is at or ahead of that write's LSN. If no replica is caught up yet, either wait or fall back to the primary.

PostgreSQL provides pg_last_wal_replay_lsn() on replicas to check their current apply position, and you can query this before routing. Many connection poolers (PgBouncer) and proxy layers (pgpool-II, RDS Proxy) implement this automatically.

Read replica routing in application codeTYPESCRIPT
import { Pool } from 'pg';

const primaryPool = new Pool({ host: 'primary.db.internal', ... });
const replicaPools = [
  new Pool({ host: 'replica1.db.internal', ... }),
  new Pool({ host: 'replica2.db.internal', ... }),
];

// Track last write LSN per user session (stored in JWT or session)
async function routedRead(sql: string, params: any[], sessionLsn?: string) {
  if (!sessionLsn) {
    // No recent write — any replica is fine
    const pool = replicaPools[Math.floor(Math.random() * replicaPools.length)];
    return pool.query(sql, params);
  }

  // Find a replica that has caught up to the user's last write
  for (const pool of replicaPools) {
    const { rows } = await pool.query('SELECT pg_last_wal_replay_lsn() >= $1::pg_lsn AS ready', [sessionLsn]);
    if (rows[0].ready) {
      return pool.query(sql, params);
    }
  }

  // No replica is caught up — fall back to primary
  logger.debug({ sessionLsn }, 'All replicas behind session LSN, reading from primary');
  return primaryPool.query(sql, params);
}

// After any write, capture the LSN and store in session
async function writeWithLsn(sql: string, params: any[]) {
  const client = await primaryPool.connect();
  try {
    await client.query(sql, params);
    const { rows } = await client.query('SELECT pg_current_wal_lsn()::text AS lsn');
    return rows[0].lsn; // store this in user's session cookie/JWT
  } finally {
    client.release();
  }
}

Section 05

Synchronous vs asynchronous replication

Asynchronous replication (the default) means the primary acknowledges a transaction commit immediately, without waiting for the replica to confirm receipt. The write is durable on the primary (WAL flushed to primary disk), but the replica might not have it yet. If the primary crashes immediately after the commit acknowledgment, the replica might not have the last few transactions.

Synchronous replication means the primary waits for at least one replica to confirm it has received and written the WAL before acknowledging the commit to the application. This eliminates the data loss window — any committed transaction is on at least two servers before the commit is acknowledged. The cost: every write pays an extra round-trip (to the replica and back).

synchronous_commit levels

PostgreSQL offers granular control over the synchronous commitment level:

  • off — WAL written to OS buffer only. Fastest. ~200ms loss window on crash.
  • local — WAL flushed to primary disk. No replica wait. This is the safe async default.
  • remote_write — Wait for replica to write WAL to its OS buffer (not flushed to disk). Faster than flush, but replica could lose data on its own crash.
  • remote_apply — Wait for replica to apply WAL to its data pages. Strongest consistency — replica is queryable for this data immediately after commit.
  • on (default sync) — Wait for replica to flush WAL to disk. Between remote_write and remote_apply.
⚠️
Synchronous replication blocks if the replica goes down

If synchronous_standby_names names a specific replica and that replica becomes unreachable, all writes on the primary block until it reconnects or you change the configuration. This is the availability tradeoff of synchronous replication — you get durability but lose write availability if the sync replica goes away. Use ANY N (replica_list) syntax to require acknowledgment from any N of a list, so you can lose one without blocking writes.

Section 06

Failover: promoting a replica to primary

When the primary fails, the goal is to restore write capability as quickly as possible with minimum data loss. The process of making a replica the new primary is called failover (or promotion). How it's done — manually, semi-automatically, or fully automatically — determines your Recovery Time Objective (RTO).

Manual failover

An operator identifies the primary is down, selects a replica (typically the one with the least lag), and promotes it. On the replica: pg_ctl promote or SELECT pg_promote(). The replica exits standby mode, begins accepting writes, and generates a new timeline. Other replicas must be reconfigured to follow the new primary.

Automated failover with Patroni

Patroni is the most widely-used HA framework for PostgreSQL. It uses a distributed consensus store (etcd, Consul, or ZooKeeper) to manage leader election. When the primary fails, Patroni's agents on each replica detect the failure, run an election, and the winner promotes itself. The DCS (Distributed Configuration Store) prevents split-brain — only one node can believe it's the primary at any time.

The split-brain problem

Split-brain occurs when two nodes both believe they're the primary. In a network partition, the old primary might still be running and serving writes while a replica has been promoted. Both accept writes. When the partition heals, you have two diverged histories that must be reconciled — some writes on both sides will be lost. Patroni prevents this using fencing: before promoting a replica, it ensures the old primary can no longer accept writes (by revoking its DCS lease).

Manual failover — PostgreSQL commandsBASH
# ── Step 1: Verify primary is truly down (not a network blip) ──
pg_isready -h 10.0.0.1 -p 5432
# If not responding after 3 attempts with 5s timeout: proceed

# ── Step 2: Pick the replica with least replication lag ────────
# On each replica, run:
psql -c "SELECT pg_last_wal_replay_lsn();"
# Pick the replica with the highest LSN (least lag)

# ── Step 3: Promote the chosen replica ─────────────────────────
# Method A: pg_ctl
pg_ctl promote -D /var/lib/postgresql/data

# Method B: SQL (PG 12+)
SELECT pg_promote(wait := true, wait_seconds := 60);

# ── Step 4: Verify promotion ───────────────────────────────────
psql -c "SELECT pg_is_in_recovery();"
# Should return: f (false) — no longer in recovery mode

psql -c "SELECT timeline_id FROM pg_control_checkpoint();"
# Timeline should have incremented (e.g. 1 → 2)

# ── Step 5: Update connection strings ──────────────────────────
# Update primary_conninfo on remaining replicas to point to new primary
# Reload PostgreSQL on each replica
# Update application connection strings / DNS / load balancer

# ── Step 6: Handle the old primary when it comes back ──────────
# The old primary has diverged from the new timeline
# It CANNOT simply rejoin as a replica (timeline mismatch)
# Options:
# A: pg_rewind — fast, repairs the old primary to follow new primary
pg_rewind --target-pgdata=/var/lib/postgresql/data \
          --source-server="host=10.0.0.2 user=replicator"
# B: Re-initialize from pg_basebackup (slower but always works)

Section 07

Logical replication: selective, version-crossing replication

Physical replication copies the raw WAL bytes — it replicates the entire cluster at the byte level. Logical replication decodes the WAL into logical changes (INSERT/UPDATE/DELETE operations on specific tables) and replicates those. This enables use cases physical replication can't handle.

What logical replication enables

  • Selective table replication: Replicate only specific tables, not the entire database cluster.
  • Cross-version replication: Replicate between different major PostgreSQL versions (e.g. PG 14 → PG 16 during an upgrade).
  • Cross-platform replication: Replicate to a different architecture or even to a non-PostgreSQL database (via logical decoding plugins like pgoutput, wal2json, Debezium).
  • Multi-master setup: Two databases each accepting writes and replicating to each other (with careful conflict management).
  • Live major version upgrades: Stand up a new PG version, replicate from old to new, cut over with minimal downtime.

Publication / subscription model

Logical replication uses a pub/sub model. The publisher creates a publication — a named set of tables to replicate. The subscriber creates a subscription pointing to the publisher and specifying which publication to follow. The subscriber then receives INSERT/UPDATE/DELETE events for all tables in the publication.

Section 08

Production reality

Replica slots accumulate WAL silently

A replication slot for a replica that was taken offline, decommissioned but not dropped, or simply falling behind will accumulate WAL indefinitely. The primary's WAL directory grows unboundedly until the disk fills and the primary crashes. Monitor pg_replication_slots and alert when any slot's retained bytes exceed a threshold (10GB is a reasonable starting point for most workloads).

max_standby_streaming_delay causes replica query failures

When a long-running query on a replica conflicts with WAL replay (the WAL wants to modify a row the query is reading), PostgreSQL waits max_standby_streaming_delay (default: 30 seconds) and then cancels the conflicting query. The query fails with ERROR: canceling statement due to conflict with recovery. Applications must retry these queries. Increase the delay if your analytical queries take longer than 30 seconds.

pg_rewind is the fast path back after a failed primary

When a failed primary comes back online, it has a different WAL timeline than the new primary. It cannot simply rejoin as a replica — it will refuse to follow the new timeline because its WAL diverges. pg_rewind efficiently repairs this: it identifies the divergence point, rewinds the old primary's WAL to that point, and sets it up to follow the new primary. Without pg_rewind, you must do a full pg_basebackup — which takes much longer for large databases.

Replication doesn't protect against logical errors

If you DROP TABLE orders on the primary, the WAL for that DROP is immediately replicated to all replicas. All replicas drop the table too, within seconds. Replication protects against hardware failure, not human error. Point-in-Time Recovery (PITR) from WAL archives is the protection against logical errors — and you need to test your PITR process before you need it.

Section 09

The things nobody tells you

Replica lag during VACUUM is normal and expected

When VACUUM runs on the primary, it generates a burst of WAL (updating the visibility map, removing dead tuples from index pages). Replicas fall behind during this burst and catch up afterward. Don't alarm on temporary lag spikes during VACUUM — alarm on sustained lag that doesn't recover.

A replica that's never queried still prevents VACUUM

PostgreSQL's MVCC visibility rules on replicas mean that a long-running transaction on the replica prevents VACUUM from cleaning up dead tuples on the primary — even if the replica is only used for failover and never receives queries. The parameter hot_standby_feedback = on tells the primary about the replica's oldest transaction, allowing the primary to delay VACUUM accordingly (or cause it to defer cleanup, adding bloat). Setting hot_standby_feedback = off prevents this but causes replica queries to fail more often due to recovery conflicts.

Timeline IDs are crucial for PITR

Every time a replica is promoted, the timeline ID increments. A WAL archive contains WAL from multiple timelines. When doing PITR to a specific time, you must specify which timeline you want to recover to — otherwise PostgreSQL defaults to the earliest timeline that includes your target time, which might not be what you want.

Decision sequence

Primary is down — what to do in the next 5 minutes

Stay calm. Work through this in order. Rushing step 5 without completing step 2 is how you get split-brain.

1
Confirm the primary is actually down — not a network blip
Run pg_isready -h primary -p 5432 three times over 30 seconds. Check if your monitoring shows the primary process crashed (OOM kill, disk full, OS crash) vs network connectivity issues. Premature failover when the primary is actually fine leads to split-brain. Wait 30-60 seconds unless you have clear evidence of process death.
premature promotion = split-brain risk. Confirm before acting.
2
If using Patroni / automated HA — let it handle it
Patroni, repmgr, and similar tools handle leader election through a distributed consensus store. Don't manually promote while automated failover is running — you'll create a second primary. Check Patroni's status: patronictl -c /etc/patroni/config.yml list. If automated failover has already elected a new leader, update your application's connection string and you're done.
automation handles this: verify it worked, update connection strings
3
Pick the replica with the highest replay LSN
On each replica: SELECT pg_last_wal_replay_lsn(). The replica with the highest LSN has the most data — promoting it minimises data loss. If replicas are equal (synchronous replication), pick the one in the same AZ/region as your application for lower latency.
highest LSN = least data loss. Check before promoting.
4
Fence the old primary before promoting
If there's any chance the old primary could come back online and start accepting writes, fence it first: revoke its network access, stop its process, or use STONITH (Shoot The Other Node In The Head). This prevents split-brain. Only after fencing is confirmed: promote the replica with pg_ctl promote or SELECT pg_promote().
fence before promote. No exceptions.
5
Update application connection strings and reconfigure remaining replicas
Update DNS / load balancer / connection pool to point writes at the new primary. Update primary_conninfo on remaining replicas to follow the new primary and reload (pg_ctl reload). Monitor that replicas are streaming from the new primary: SELECT * FROM pg_stat_replication on the new primary.
verify replicas are following new primary before closing the incident
6
Repair the old primary with pg_rewind, don't re-initialise
When the old primary comes back online, run pg_rewind to efficiently align it with the new primary's timeline. This is much faster than pg_basebackup for large databases — it only copies the pages that diverged. Then start the old primary as a new replica of the current primary. Drop any orphaned replication slots on the new primary from the old topology.
pg_rewind = minutes vs pg_basebackup = hours for large DBs

Section 11

Replication checklist

SLT
Slot monitoring

Alert if any replication slot retains >10GB of WAL. Drop slots for decommissioned replicas immediately.

LAG
Lag alerting

Monitor replay_lag on pg_stat_replication. Alert at 30s lag for sync-critical replicas, 5min for async read replicas.

RWD
Read routing

Track session LSN after writes. Route subsequent reads to replicas only when caught up, else fall back to primary.

FOV
Failover tested

Test your failover procedure quarterly. Time it. Verify pg_rewind works on your actual data size. Don't find out at 3am.

PIR
PITR tested

Point-in-time recovery is your protection against DROP TABLE. Test it. Verify your WAL archives are complete and restorable.

FNC
Fence first

Always fence the old primary before promoting a replica. Split-brain from simultaneous primaries is worse than extended downtime.
