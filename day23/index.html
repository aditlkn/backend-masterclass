Section 01

The postal sorting centre — and why your services need one

A letter arrives at a sorting centre. The sender dropped it in a postbox without knowing or caring who would deliver it, when, or by which route. The sorting centre reads the address, assigns it to a bin — Mumbai South, bin 3 — and a delivery van picks it up when it is ready. The sender has already moved on. The recipient collects it at their own pace. Neither knows the other exists.

That is Kafka. The sender is a producer. The sorting centre is a Kafka broker. The address bins are partitions. The delivery vans are consumers. The letter is an event. And crucially — unlike a real letter, this one is never destroyed after delivery. It stays in the bin. Any number of vans can read it. A new van that starts tomorrow can read letters from last week.

Multiple sorting bins for the same destination. If Mumbai South gets ten thousand letters a day, one bin and one van cannot keep up. So you add bins — Mumbai South bin 1, bin 2, bin 3 — and assign a separate van to each. Letters are distributed across bins by a rule (typically the first letter of the street name, or a hash of the recipient address). Each van works its bin independently. This is partitioning, and it is why Kafka scales horizontally: more partitions, more parallelism, more throughput.

Multiple postal services, same letters. Royal Mail reads the letters. FedEx reads the same letters. DHL reads them too. Each maintains its own bookmark — "I have delivered up to letter 847" — and works at its own pace without interfering with the others. This is a consumer group. Each service is a separate group with its own offset tracking. One set of letters, multiple independent readers.

A tracking number for every letter. Every letter in the sorting centre has a position number — its place in the bin, from 0 upward. This is the offset. When a van completes a delivery, it marks its position: "I have read up to offset 1,247 in bin 3." If the van breaks down and a replacement takes over, it resumes from 1,247. No letter is skipped. No letter is lost. And if you discover a routing error happened between offset 800 and 1,100, you can rewind the replacement van to 800 and re-deliver those letters.

Where the analogy breaks. A real letter takes one route and arrives once. Kafka events can be consumed by every consumer group that subscribes — it is more like a notice board than a letter box. Real postal sorting centres discard letters after delivery. Kafka retains them for a configurable period (7 days by default), which is what makes replay possible. And a real sorting centre cannot process a million letters per second on commodity hardware. Kafka can.

Section 02

Topics and partitions — how Kafka organises events

A topic is a named stream of events. Think of it as a category of letters — all booking confirmations go in the booking.confirmed topic, all payment events go in payment.completed. Producers write to topics. Consumers read from topics. Topics are the contract between them.

Each topic is divided into partitions — the ordered, immutable, append-only logs that are the actual storage unit. A topic with 6 partitions stores events across 6 independent logs. Events within a partition are strictly ordered. Events across partitions have no ordering guarantee.

How many partitions?

Partition count determines your maximum parallelism: you can have at most as many active consumers as partitions. A topic with 6 partitions can be consumed by at most 6 consumers simultaneously (within a single consumer group). More partitions = more parallelism = more throughput, but also more overhead: each partition is a file on disk, and Kafka brokers manage leader election per partition.

A reasonable starting point: estimate your target throughput (events/sec), divide by what a single consumer can handle, and round up to the next power of 2. Start with 6 or 12 for most production topics. You can increase partition count but you cannot decrease it — partition reduction requires recreating the topic.

Partition keys — controlling where events land

When a producer sends an event, Kafka hashes the partition key to determine which partition it lands in. Events with the same key always go to the same partition — which means they are always in order relative to each other. This is the mechanism for ordering guarantees: if you need all events for a given booking to be processed in order, use the booking ID as the partition key. All booking-123 events land in partition 2. Consumer 2 processes them in order.

💡
Partition keys are a ordering contract, not just a routing rule

Choosing a partition key is one of the most consequential decisions in a Kafka design. A key that is too coarse (e.g. event type) concentrates all events in one partition and destroys parallelism. A key that is too fine (e.g. a random UUID) distributes evenly but loses ordering. The right key is the entity whose events must be ordered relative to each other — typically a user ID, order ID, or session ID. Get this wrong and you will either have ordering bugs or a throughput bottleneck that requires a full topic rebuild to fix.

Section 03

Producers — writing events reliably

A Kafka producer batches events in memory before sending them to the broker. This batching is invisible to your code but critical to throughput: instead of one network call per event, the producer accumulates events for up to linger.ms milliseconds (default: 0) and sends them together. At high volume, batching reduces network overhead by orders of magnitude.

The three producer acknowledgement modes

acks=0 (fire and forget): The producer does not wait for any acknowledgement. Events may be lost if the broker fails between receiving and writing. Highest throughput, no durability. Use only for metrics or analytics where occasional loss is acceptable.

acks=1 (leader acknowledgement): The leader broker acknowledges after writing to its own log. If the leader fails before followers replicate, events are lost. The default in older Kafka versions. A trap: it feels safe but has a real loss window.

acks=all / acks=-1 (full replication): The leader waits for all in-sync replicas to acknowledge before responding. Combined with min.insync.replicas=2, this guarantees no data loss even if a broker fails immediately after acknowledgement. Use this for any event that matters. The latency cost is one extra network round-trip to a replica.

Producer config — durable event publishingTYPESCRIPT
import { Kafka, CompressionTypes } from 'kafkajs';

const kafka = new Kafka({
  clientId: 'checkout-service',
  brokers:  ['kafka-1:9092', 'kafka-2:9092', 'kafka-3:9092'],
});

const producer = kafka.producer({
  allowAutoTopicCreation: false,    // never auto-create in production
  idempotent: true,                  // exactly-once producer semantics
  transactionalId: 'checkout-producer-1',
});

await producer.connect();

await producer.send({
  topic: 'booking.confirmed',
  compression: CompressionTypes.GZIP,
  messages: [{
    key:   booking.id,            // partition key — all events for this booking stay ordered
    value: JSON.stringify({
      bookingId:   booking.id,
      userId:      booking.userId,
      eventId:     booking.eventId,
      amountCents: booking.amountCents,
      confirmedAt: new Date().toISOString(),
    }),
    headers: {
      'correlation-id': requestId,
      'schema-version': '2',
    },
  }],
});

// Producer config for acks=all + idempotency
// Set in broker config: min.insync.replicas=2
// With replication.factor=3, this tolerates 1 broker failure
// without any data loss.

Section 04

Consumer groups — parallelism without duplication

A consumer group is a set of consumers that cooperate to process a topic. Kafka assigns each partition to exactly one consumer in the group at a time. With 6 partitions and 3 consumers, each consumer handles 2 partitions. With 6 partitions and 6 consumers, each handles 1. With 6 partitions and 7 consumers, one consumer is idle — you cannot have more active consumers than partitions.

Each consumer group maintains its own offset tracking. The email-service group and the analytics-service group both read the same booking.confirmed topic independently. One being slow does not affect the other. One resetting its offset to replay events does not affect the other. This is the mechanism that lets a single topic feed multiple independent downstream systems.

Rebalancing — the expensive moment

When a consumer joins or leaves a group, Kafka triggers a rebalance: it reassigns partitions among the remaining consumers. During a rebalance, consumption stops entirely for that group — no events are processed until the rebalance completes. In the worst case (many consumers, slow session timeouts), a rebalance can pause processing for tens of seconds.

Causes of unwanted rebalances: consumers that take longer than max.poll.interval.ms to process a batch (default: 5 minutes), consumers that miss heartbeats due to GC pauses, and rolling deploys that cycle through consumer instances. Mitigation: use static group membership (group.instance.id), which gives each consumer a stable identity so Kafka waits for it to rejoin rather than triggering an immediate rebalance.

Section 05

Offsets and replay — the superpower nobody talks about

Every event in every partition has an offset — a monotonically increasing integer starting from 0. When a consumer processes an event, it commits its offset back to Kafka: "I have successfully processed up to offset 1,247 in partition 3." If the consumer crashes and restarts, it resumes from 1,248.

This seems straightforward until you realise the implication: you can replay any event stream from any point in time, as long as it is within the retention window. Deployed a bug that corrupted downstream data between 14:00 and 17:00? Reset the consumer offset to 14:00, replay those events through the fixed consumer, rebuild the corrupted state. No database restore. No backfill scripts. Just offset reset and replay.

Offset commit strategies

Auto-commit (the trap): Kafka commits the offset automatically every auto.commit.interval.ms (default: 5 seconds). If your consumer crashes after committing but before finishing processing, those events are skipped — offset advanced, work not done. At-least-once processing requires manual offset commit.

Manual commit after processing: Call consumer.commitOffsets() only after successfully processing the event. If the consumer crashes between processing and committing, the event will be redelivered — this is at-least-once semantics. Your processing logic must be idempotent.

Transactional (exactly-once): Write the event processing result and the offset commit in a single atomic transaction. Requires Kafka Streams or a transaction-aware consumer. Necessary only when idempotent processing is not achievable.

⚠️
Disable auto-commit in production consumers

Auto-commit is the default and it is wrong for any consumer that does meaningful work. It commits offsets on a timer, independent of whether processing succeeded. A consumer that crashes after auto-commit but before writing to the database will skip those events permanently. Set enable.auto.commit=false and commit manually after successful processing. The extra 5 lines of code prevent a class of data-loss bugs that are nearly impossible to debug after the fact.

Section 06

Delivery guarantees — at-most-once, at-least-once, exactly-once

GuaranteeEvents lost?Duplicates?HowUse when
At-most-onceYes (on failure)NoCommit before processingMetrics, analytics — loss acceptable
At-least-onceNoYes (on retry)Commit after processing, idempotent handlerMost business events — make handler idempotent
Exactly-onceNoNoKafka transactions (EOS)Financial transactions where idempotency is impossible

In practice, at-least-once with idempotent consumers is the right answer for almost every use case. Exactly-once in Kafka is real and production-ready, but it adds complexity: transactions span producers and consumers, require Kafka Streams or careful manual coordination, and have meaningful throughput overhead. The simpler path — make your consumer idempotent (check if you already processed this event ID before doing the work) — achieves the same outcome without the complexity.

Making consumers idempotent

Before processing any event, check a deduplication store: SET processed:{eventId} 1 NX EX 86400. If the key already exists, the event was already processed — skip it. If not, process and set the key. This pattern, combined with manual offset commit, gives you effectively exactly-once processing without Kafka transactions.

Section 07

When Kafka is the wrong answer

Kafka is optimised for high-throughput, persistent, replayable event streams. It is not optimised for everything. Reaching for Kafka as the default messaging solution is a common over-engineering mistake.

Use caseKafkaBetter choiceWhy
Task queue (jobs)Possible but awkwardBullMQ, SQSKafka has no native job acknowledgement or DLQ
Request/reply RPCWrong toolgRPC, HTTPKafka is async; RPC is synchronous by definition
Low volume (<1K/sec)OverkillRedis Streams, SQSKafka operational overhead not justified at low volume
High-throughput event logRight tool—Kafka designed exactly for this
Fan-out to many servicesRight tool—Consumer groups make fan-out zero-cost for producers
Event replay / auditRight tool—Retention + offset reset is Kafka's unique capability

The threshold question: if you have fewer than ~5,000 events per second and do not need replay, Redis Streams or Amazon SQS will serve you with far less operational overhead. Kafka requires ZooKeeper (or KRaft in newer versions), multiple brokers for replication, careful partition planning, and schema management. That cost is worth it at scale. At 500 events per day, it is not.

Section 08

Production reality

Schema Registry — the contract between producers and consumers

When a producer changes the shape of an event — adds a field, renames one, changes a type — every consumer breaks. Schema Registry (Confluent or Apicurio) solves this by storing event schemas and enforcing compatibility rules. Producers register a schema version before publishing. Consumers validate incoming events against the schema. Breaking changes are caught at publish time, not at runtime in production. Use Avro or Protobuf with Schema Registry for any topic that more than one team consumes.

Consumer lag — the metric that tells you everything

Consumer lag is the difference between the latest offset in a partition and the consumer group's committed offset. A lag of zero means the consumer is keeping up. A lag of 10,000 and growing means the consumer cannot keep up with the producer. Monitor lag continuously. Alert when it grows beyond a threshold. Do not alert on absolute lag value — some lag is normal. Alert on growing lag, which indicates a consumer that is falling behind.

KRaft mode — Kafka without ZooKeeper

Kafka 3.3+ supports KRaft (Kafka Raft) mode, which removes the ZooKeeper dependency. ZooKeeper was a separate system that needed its own cluster, its own monitoring, and its own operational expertise. KRaft integrates metadata management into Kafka itself. For new deployments, use KRaft. For existing ZooKeeper-based clusters, plan migration — ZooKeeper support will be removed in a future version.

Section 09

The things nobody tells you

Increasing partition count is a one-way door

You can add partitions to a topic but never remove them. Adding partitions breaks the partition-key-to-partition mapping — events that used to go to partition 2 may now go to partition 5, destroying ordering guarantees for events produced before and after the change. The practical approach: over-provision partitions at topic creation time. Starting with 12 instead of 6 costs almost nothing and avoids the painful topic recreation process later.

Consumer group rebalancing is your biggest latency spike

A rolling deploy of 12 consumer instances will trigger 12 rebalances — one for each instance cycling out and back in. Each rebalance pauses all consumption for that group. At 30 seconds per rebalance in the worst case, a 12-instance rolling deploy can cause 6 minutes of processing gaps. Use static group membership (group.instance.id) to give each consumer a stable identity — Kafka waits for the consumer to rejoin rather than immediately rebalancing.

Log compaction is not the same as log retention

Kafka has two cleanup policies: delete (remove events older than the retention period) and compact (keep only the latest event per partition key). Compacted topics are used for "current state" views — only the most recent user profile event, the most recent inventory count. Delete topics are used for event logs where history matters. Using the wrong policy is how you accidentally discard order history that accounting needs, or accidentally keep millions of outdated events that should have been discarded.

Kafka is not a database

Retention gives Kafka some database-like properties, but it is not a database. There is no random access by key. There is no indexing beyond partition key. There is no query language. If you need to ask "what is the current state of booking ord_123?", you need a database that is kept in sync by a consumer. Kafka is the event log. A database is the materialised view of that log. Both are necessary.

Decision sequence

Designing a Kafka event pipeline

Work through this before writing any producer or consumer code.

1
Do you actually need Kafka?
If throughput < 5K events/sec and you do not need replay: use Redis Streams or SQS. If you need a job queue with retries and DLQ: use BullMQ (Day 21). Kafka is for high-throughput event logs that multiple services consume independently, with replay capability. If that is not your use case, the operational overhead is not worth it.
most teams reach for Kafka 3 years too early
2
Design your topic schema before writing any code
What is the event name? What fields does it carry? What is the partition key? Who produces it? Who consumes it? Write this down as a schema (Avro or Protobuf) and register it in Schema Registry. Changing an event schema after consumers are in production is painful. Getting it right before is free.
partition key is the hardest decision and the most permanent
3
Choose partition count with room to grow
Formula: target_throughput / consumer_throughput = minimum partitions. Round up to a power of 2. Double it. You cannot reduce partitions, so over-provision. A topic with 12 partitions handling 100 events/sec is fine. A topic with 3 partitions at 10,000 events/sec is a bottleneck you cannot fix without recreating the topic.
default to 12 partitions for any production topic
4
Set acks=all and min.insync.replicas=2
For any event that matters: acks=all on the producer, min.insync.replicas=2 on the topic, replication.factor=3 on the cluster. This combination guarantees no data loss even if a broker fails immediately after acknowledgement. The latency cost is one extra round-trip to a replica. Worth it for every business event.
acks=1 feels safe and is not
5
Disable auto-commit, make consumers idempotent
Set enable.auto.commit=false. Commit offsets manually after successful processing. Make your consumer idempotent — check a Redis deduplication key before processing. This gives you at-least-once delivery with effectively-exactly-once behaviour, without Kafka transactions.
auto-commit is a silent data-loss mechanism
6
Monitor consumer lag from day one
Alert when consumer lag grows (not just when it is large). A lag of 50,000 that is stable is fine. A lag of 1,000 that grows by 100 per minute will exhaust retention in days. Use kafka-consumer-groups.sh --describe or expose lag as a Prometheus metric. No Kafka deployment is production-ready without lag monitoring.
growing lag = consumer falling behind = eventual data loss

Section 11

Kafka production checklist

KEY
Partition key

Chosen based on ordering requirements. Entity whose events must be ordered relative to each other. Documented and stable.

ACK
Durability

acks=all, min.insync.replicas=2, replication.factor=3. Verified with broker config. Tested with broker failure.

IDM
Idempotent consumer

Auto-commit disabled. Manual offset commit after processing. Deduplication check on event ID before any side effects.

LAG
Lag monitored

Consumer lag exposed as Prometheus metric. Alert fires when lag grows. Dashboard visible to whole team.

SCH
Schema registry

Avro or Protobuf schemas registered. Backward compatibility enforced. Breaking changes caught at publish time.

RET
Retention policy

Delete vs compact chosen deliberately. Retention period set based on replay needs. Tested with offset reset.

The postal sorting centre does not know what is in the letters. It does not know who will read them or when. It just makes sure they get to the right bin, stay there long enough for collection, and hands each postal service its own independent copy. Ten teams can read the same event. One slow reader does not slow the others. A letter posted last Tuesday can be re-read today.

The next time a system needs to notify ten other systems about something that happened — ask: does each one need its own letter, or should they all read from the same bin?
