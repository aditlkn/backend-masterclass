Section 01

The architect — constraints before materials

An architect does not start with materials. They start with constraints. How many people will use this building? What do they need to do in it? What is the budget and the site? What regulations apply? What is the expected lifespan? Only when the constraints are understood does the architect begin sketching the structure — and even then, the first sketches are not blueprints. They are explorations of the space of possible solutions, each making different trade-offs.

A hospital and a hotel might have the same number of rooms and similar infrastructure — plumbing, HVAC, electrical, lifts. But a hospital requires redundant power systems, specific ventilation for sterile environments, different egress requirements. An architect who treats them as the same problem because they look similar on the outside will produce a building that works for neither. The constraints drive everything.

System design works identically. A ticketing system and a social media feed might both need to handle millions of users and store large amounts of data. But a ticketing system requires strong consistency (two people cannot buy the last seat), while a social media feed is fine with eventual consistency (seeing a post 500ms late matters less than serving it fast). An engineer who designs both systems the same way will produce one that is over-engineered and expensive, and one that causes lost revenue through double-bookings.

The architect sketches before specifying. Back-of-envelope estimation is the architect's first sketch — a rough calculation to understand whether the proposed structure is even feasible. Can a single server handle this load? How much storage does this actually require? What does the network cost? These calculations take 10 minutes. They prevent spending 3 months building the wrong thing.

Principal engineering is architectural judgment. Any engineer can learn to build a particular type of building correctly. A principal engineer knows which type of building the client actually needs, and can explain clearly why the alternative they proposed would have failed. That judgment — about constraints, trade-offs, and context — is what separates an engineer who knows many tools from one who knows when to use each one.

Section 02

Back-of-envelope estimation — the 10-minute sanity check

Back-of-envelope estimation is not about precision. It is about order of magnitude. A calculation that is off by 3x is useful. A calculation that is off by 1,000x is a crisis. The goal is to know whether you need one server or one thousand, whether your storage fits in a database or requires a data warehouse, whether your read volume justifies a cache.

The estimation template

Five questions, in order:

  1. DAU / MAU: How many daily active users? Peak concurrent?
  2. Read/write ratio: Is this read-heavy (10:1 or 100:1) or write-heavy?
  3. Requests per second: Peak DAU × actions per day ÷ 86,400.
  4. Storage per year: Writes per second × bytes per write × seconds per year.
  5. Bandwidth: Requests per second × response size.
📋
Worked estimation: a ticket booking system

DAU: 1M. Peak: 10× average (concert on-sale). Read/write: 50:1 (browse events vs book). Reads: 1M × 20 page views ÷ 86,400 ≈ 230 reads/sec avg, 2,300/sec peak. Writes: 230 ÷ 50 = 4.6 writes/sec avg, 46/sec peak. Storage: 46 writes/sec × 1KB/write × 31M sec/year ≈ 1.4TB/year. Bandwidth: 2,300/sec × 50KB response = 115MB/s. Conclusion: reads are manageable on one PostgreSQL primary + read replicas. Writes are low enough for a single primary. Storage is 1.4TB/year — plan for 5TB in year one including overhead. Peak concert on-sale (10×) needs caching — 23,000 reads/sec would overwhelm a database.

Section 03

CAP theorem — the trade-off you are always making

CAP theorem states that a distributed system can only guarantee two of three properties: Consistency (every read sees the most recent write), Availability (every request receives a response), and Partition tolerance (the system continues operating when network messages are lost). In practice, network partitions will happen — so the real choice is between consistency and availability when a partition occurs.

The choice is not abstract. It is concrete and consequential. A bank account balance must be consistent — you cannot read a stale balance before approving a transaction. A social media like count can be eventually consistent — seeing 9,842 likes instead of 9,843 for 500ms is acceptable. A ticketing system seat reservation must be consistent — two people cannot buy the same seat. A product review count can lag by seconds. The system designer chooses consistency or availability per operation, not per system.

PACELC — the more useful model

PACELC extends CAP: even when there is no Partition, distributed systems must choose between Latency and Consistency. Strong consistency requires waiting for all replicas to acknowledge a write before confirming it — which adds latency. Eventual consistency allows acknowledging the write at one node and propagating asynchronously — lower latency, but reads may see stale data. Most real systems choose: strong consistency for financial operations and inventory, eventual consistency for social counts, content feeds, and analytics.

Section 04

The system design canvas — a framework for any problem

The canonical system design interview format exists for a reason: it mirrors how experienced engineers actually approach new systems. The steps are not bureaucratic — each one makes the next one easier. You cannot choose a database until you know the query patterns. You cannot design the API until you know the data model. You cannot plan for scale until you have done the estimation.

Step 1: Clarify requirements (5 minutes)

Before drawing anything: what does this system actually need to do? What does it explicitly not need to do? What are the scale targets? What are the consistency requirements? What is the latency SLO? A system designed without these answers will be wrong in ways that cannot be fixed by adding components later. In an interview, ask the questions. In real work, write the requirements document before the architecture diagram.

Step 2: Estimate scale (5 minutes)

Run the five-question estimation. Write down the numbers. Now you know: does this need sharding? Does this need a CDN? Does this need a dedicated cache layer? Does the write volume exceed what a single PostgreSQL primary can handle? These numbers constrain the design. Without them, every choice is arbitrary.

Step 3: Define the API (5 minutes)

What does the client call? What does it return? The API is the contract between your system and everything that uses it. Define it before the internals. The API forces you to think about what the system produces, not how it produces it. Getting the API right is more important than getting the implementation right — the implementation can change, the API is a commitment.

Step 4: Design the data model (10 minutes)

What are the entities? What are the relationships? What are the access patterns? The data model drives every other decision. Choose the wrong data model and you will fight your database forever. Design for your query patterns, not for normalisation theory — a denormalised table that answers your queries in one scan is better than a normalised schema that requires three joins.

Step 5: Design the high-level architecture (10 minutes)

Now — and only now — draw the boxes. API gateway, services, databases, caches, message queues. Each box represents a constraint: a database is synchronous, a message queue is asynchronous, a cache is eventually consistent. The lines between boxes represent coupling — minimise it. Start with the simplest architecture that satisfies the requirements and estimation. Add complexity only when you can name the specific requirement it satisfies.

Step 6: Deep-dive the hard parts (15 minutes)

Every system has one or two genuinely hard problems. For a ticketing system it is seat reservation under concurrent load. For a social media feed it is fan-out at scale. For a URL shortener it is nothing — it is a simple system and should be designed simply. Identify the hard parts and solve them specifically. Do not design a sophisticated distributed system for a problem that does not need one.

Section 05

The trade-off vocabulary — how principal engineers talk about choices

Every architectural decision is a trade-off. The mark of a principal engineer is not that they always make the right trade-off — it is that they can name the trade-off clearly and explain why this context favours this side of it. "I chose eventual consistency here because the write latency of strong consistency would violate the 50ms SLO, and the business impact of a user seeing stale event metadata for 500ms is negligible" is a principal-level answer. "I chose Redis because it's fast" is not.

The trade-offs that appear in every system

Latency vs consistency: Strong consistency requires waiting for all nodes. Eventual consistency allows faster writes. Choose strong consistency when incorrect data causes business harm (inventory, payments). Choose eventual consistency when staleness is acceptable (social counts, search indexes).

Throughput vs latency: Batching increases throughput by amortising fixed costs over more work. Batching increases latency for individual operations. Stream processing gives low latency, batch processing gives high throughput. Choose based on the SLO — is this operation latency-sensitive or throughput-sensitive?

Storage cost vs query performance: Denormalised tables are faster to query but cost more storage and create update anomalies. Normalised tables are cheaper to store and update, but require joins. OLTP systems lean normalised. OLAP and read-heavy services lean denormalised (materialised views, CQRS projections).

Operational complexity vs capability: A managed service (RDS, DynamoDB, ElastiCache) is operationally simpler but less configurable. Self-managed infrastructure is more configurable but more work to run. For most teams, managed services are correct for databases and infrastructure. For very specific performance requirements, self-managed may be justified.

Coupling vs coordination: Synchronous calls between services are simple but create coupling — if the downstream is slow, the upstream is slow. Async messaging decouples services — the publisher does not wait for consumers — but adds complexity (ordering, idempotency, dead letters). Choose synchronous calls for operations that need a response. Choose async for operations that can be fire-and-forget or that need to fan out to multiple consumers.

Section 06

Worked example: design a seat reservation system

This is the hard problem from the ticketing system estimation. Thousands of users simultaneously trying to book the last few seats for a popular concert. The naive implementation — check if seat is available, then book it — fails under concurrency. Between the check and the book, another user may have taken the seat.

Why this is hard

At peak, 10,000 concurrent users are trying to book 200 remaining seats. The correct answer must be: exactly one user gets each seat. No double-bookings. No seat lost (successfully booked by one system but not reflected). No phantom bookings (seat appears taken but no one bought it). Under concurrent load, this requires a concurrency control mechanism.

Three approaches, three trade-offs

Optimistic locking: Read the seat with a version number. Update only if the version matches. If two users try simultaneously, one succeeds, one gets a conflict error and must retry. Low lock contention, good for moderate concurrency. At extreme concurrency (10,000 users, 200 seats), retry rates become very high — most requests fail and retry repeatedly.

Database SELECT FOR UPDATE: Lock the row during the transaction. Only one transaction can hold the lock at a time. Simple, correct, but creates a bottleneck at the database row level. At high concurrency, queue depth grows and P99 latency climbs. Appropriate for moderate scale; may not handle extreme concert on-sale events.

Redis atomic reservation: Use Redis SETNX (set if not exists) with a TTL — seat is atomically marked as held for 10 minutes, or the SETNX fails (seat is taken). Redis processes 100K ops/sec on a single thread — far higher throughput than PostgreSQL row locking. The hold is persisted to PostgreSQL asynchronously after payment. Risk: if the async write fails, the hold exists in Redis but not in PostgreSQL — requires reconciliation.

🎪
The correct answer for most systems

For a ticketing system at typical scale (thousands of concurrent users, not millions), SELECT FOR UPDATE on PostgreSQL with a short transaction timeout is correct, simple, and safe. Redis atomic reservation adds complexity and a reconciliation requirement. It is justified only when PostgreSQL cannot sustain the write throughput — which for most concerts it can. Design for your actual scale, not for the scale of Ticketmaster. The system that sells 10,000 tickets does not need the architecture of a system that sells 10 million.

Section 07

Principal thinking — the judgment layer

Technical knowledge is necessary but not sufficient for principal-level engineering. What distinguishes a principal engineer from a senior engineer with the same technical knowledge is judgment — the ability to identify which trade-offs matter for this specific context, communicate them clearly, and make a defensible decision.

The five questions a principal engineer asks first

  1. What is the actual business constraint? "We need low latency" is not a constraint. "We need P99 checkout completion under 2 seconds, as measured by the user's browser, or we lose 15% of revenue" is a constraint. The specificity changes the design.
  2. What failure mode is unacceptable? Losing a booking is worse than a user seeing an error. A double charge is worse than a failed charge. Identifying the worst acceptable failure mode drives consistency decisions — the system must prevent what is unacceptable, even at the cost of availability.
  3. What is the simplest thing that could possibly work? This is not a question about being lazy. It is a question about operational cost. Every piece of infrastructure you add must be operated, monitored, upgraded, and debugged. A system with one database and one service tier has one failure mode. A system with five services, two queues, three caches, and a CDC pipeline has dozens. Each component must earn its place.
  4. What will be hard to change later? Data models are hard to change once they have data in them. API contracts are hard to change once clients depend on them. Database choices are hard to change once there are terabytes of data. Everything else is relatively easy to change. Invest most of your design time on the things that are hard to change.
  5. What does this look like in two years? The system that is correct for today's scale is often wrong for 10× scale. Principal engineers design for the next 2-3× of growth without over-engineering for 100×. "We might need this someday" is not a justification for adding complexity now. "We will need this within 18 months based on current growth" is.
🌟
The most important skill is saying what you are not building

A principal engineer who proposes a simple system with one database and explains precisely why the more sophisticated alternatives are not justified for this context is demonstrating better judgment than one who proposes a distributed system with event sourcing, CQRS, and a Kafka pipeline because they learned those patterns recently. The hardest thing in system design is not building the complex system. It is recognising when the simple system is correct — and defending that position when others suggest adding complexity.

Section 09

What comes next

Thirty-one days of concepts is not mastery. Mastery comes from the third time you debug a production incident involving exactly the pattern you read about here — and you know immediately what it is, where to look, and what fix will hold. The course gives you the vocabulary. Production gives you the intuition.

The three practices that build judgment

Read postmortems. Every major outage that has been written up publicly is a compressed lesson in what actually fails in production — not what textbooks predict will fail. AWS, Google, Stripe, Cloudflare, and GitHub all publish incident reports. Read them. Notice which concepts from this course appear. Notice which failure modes were foreseeable and which were not.

Design systems before you build them. Before your next feature, write a one-page design document: requirements, data model, API shape, consistency trade-offs, how you will know it is working. Even if nobody reads it, the act of writing forces the questions you would otherwise skip. The systems that cause the most production pain are the ones that were never designed — they were just coded.

Explain what you build. Teaching requires understanding at a level that using does not. When you implement something — a caching layer, a message queue, a new database — explain it to someone else in terms they can understand without the code. If you cannot explain why the queue is there, you do not yet understand why the queue is there. The explanation will surface the gaps.
