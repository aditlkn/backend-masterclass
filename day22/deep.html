Deep Dive 01

Prometheus recording rules:
efficient multi-window SLI computation

Computing error rates over 28-day windows at query time is expensive — Prometheus has to scan millions of samples on every dashboard load. Recording rules pre-aggregate metrics on a schedule, making dashboard queries fast regardless of time window length.

slo-recording-rules.yaml — pre-aggregate SLI across multiple windowsYAML
groups:
  - name: slo_checkout_api
    interval: 30s   # recompute every 30s
    rules:
      # Error rate over multiple windows — used by burn rate alerts
      - record: job:http_requests_total:error_rate5m
        expr: |
          rate(http_requests_total{job="checkout-api",status=~"5.."}[5m])
          /
          rate(http_requests_total{job="checkout-api"}[5m])

      - record: job:http_requests_total:error_rate30m
        expr: |
          rate(http_requests_total{job="checkout-api",status=~"5.."}[30m])
          /
          rate(http_requests_total{job="checkout-api"}[30m])

      - record: job:http_requests_total:error_rate1h
        expr: |
          rate(http_requests_total{job="checkout-api",status=~"5.."}[1h])
          /
          rate(http_requests_total{job="checkout-api"}[1h])

      - record: job:http_requests_total:error_rate6h
        expr: |
          rate(http_requests_total{job="checkout-api",status=~"5.."}[6h])
          /
          rate(http_requests_total{job="checkout-api"}[6h])

      - record: job:http_requests_total:error_rate1d
        expr: |
          rate(http_requests_total{job="checkout-api",status=~"5.."}[1d])
          /
          rate(http_requests_total{job="checkout-api"}[1d])

      - record: job:http_requests_total:error_rate28d
        expr: |
          rate(http_requests_total{job="checkout-api",status=~"5.."}[28d])
          /
          rate(http_requests_total{job="checkout-api"}[28d])

      # Error budget consumed — fraction of 28d budget used so far
      # SLO = 99.9%, so error_budget = 0.001
      - record: job:slo_error_budget:remaining
        expr: |
          1 - (
            job:http_requests_total:error_rate28d
            / 0.001  # error budget = 1 - SLO = 0.001
          )

      # Burn rate — how fast budget is being consumed right now
      - record: job:slo_burn_rate:1h
        expr: job:http_requests_total:error_rate1h / 0.001

      - record: job:slo_burn_rate:6h
        expr: job:http_requests_total:error_rate6h / 0.001

      # Latency SLI — fraction of requests under 500ms
      - record: job:http_request_duration:p95_5m
        expr: |
          histogram_quantile(0.95,
            rate(http_request_duration_seconds_bucket{
              job="checkout-api"
            }[5m])
          )

  # Burn rate alert rules (reference recording rules — fast to evaluate)
  - name: slo_checkout_alerts
    rules:
      - alert: CheckoutAPIFastBurn
        expr: |
          job:slo_burn_rate:1h > 14.4
          AND
          job:http_requests_total:error_rate5m / 0.001 > 14.4
        for: 2m
        labels: { severity: critical, slo: checkout_api }
        annotations:
          summary: "Checkout API: error budget burning at >14.4× rate"
          description: "Current 1h burn: {{ $value | humanize }}×. Budget exhausted in <48h."

      - alert: CheckoutAPISlowBurn
        expr: |
          job:slo_burn_rate:6h > 6
          AND
          job:http_requests_total:error_rate30m / 0.001 > 6
        for: 15m
        labels: { severity: warning, slo: checkout_api }
        annotations:
          summary: "Checkout API: error budget burning at >6× rate"
Deep Dive 02

Sloth: SLO configuration as code

Sloth generates all the recording rules and alerts above from a simple YAML definition. Instead of maintaining complex Prometheus expressions by hand, you define your SLO once and Sloth generates the complete Prometheus configuration.

checkout-api-slo.yaml — Sloth SLO definitionYAML
version: "prometheus/v1"
service: "checkout-api"
labels:
  team: "platform"
  env: "production"

slos:
  - name: "requests-success-rate"
    objective: 99.9   # 99.9% SLO
    description: "99.9% of checkout API requests must succeed"
    sli:
      events:
        error_query: |
          sum(rate(http_requests_total{
            job="checkout-api",
            status=~"5.."
          }[{{.window}}]))
        total_query: |
          sum(rate(http_requests_total{
            job="checkout-api"
          }[{{.window}}]))
    alerting:
      name: CheckoutAPISuccessRate
      annotations:
        summary: "Checkout API error budget burning"
        runbook: "https://runbooks.internal/checkout-slo"
      page_alert:
        labels: { severity: critical }
      ticket_alert:
        labels: { severity: warning }

  - name: "request-latency"
    objective: 95    # 95% of requests must complete within 500ms
    description: "95% of checkout requests complete within 500ms"
    sli:
      events:
        error_query: |
          sum(rate(http_request_duration_seconds_bucket{
            job="checkout-api",
            le="0.5"   # 500ms
          }[{{.window}}]))
        total_query: |
          sum(rate(http_request_duration_seconds_count{
            job="checkout-api"
          }[{{.window}}]))
    alerting:
      name: CheckoutAPILatency
      annotations:
        summary: "Checkout API latency SLO burning"

# Generate Prometheus rules:
# sloth generate -i checkout-api-slo.yaml > checkout-api-rules.yaml
# kubectl apply -f checkout-api-rules.yaml

Deep Dive 03

Grafana dashboard:
the five panels every SLO dashboard needs

Grafana panel queries — PromQL for each SLO panelPROMQL
## Panel 1: Error budget remaining (gauge, 0-100%)
## Green >50%, orange 10-50%, red <10%
job:slo_error_budget:remaining * 100

## Panel 2: Current burn rate (stat panel)
## Threshold lines at 1× (SLO pace), 6× (slow burn), 14.4× (fast burn)
job:slo_burn_rate:1h

## Panel 3: Error budget consumption over time (time series)
## Shows budget draining visually across the 28-day window
1 - job:slo_error_budget:remaining

## Panel 4: SLI over time — actual error rate (time series)
## Red zone = above SLO error budget (0.1%)
job:http_requests_total:error_rate5m * 100

## Panel 5: Latency P95 vs threshold (time series)
## Threshold line at 500ms
job:http_request_duration:p95_5m * 1000  # convert to ms

## Stat panel: Budget remaining in minutes
job:slo_error_budget:remaining * 28 * 24 * 60  # minutes in window

## Stat panel: Requests in window (denominator)
increase(http_requests_total{job="checkout-api"}[28d])
