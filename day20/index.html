Section 01

Why unstructured logs fail at scale

A log line like Payment failed for user 12345: card declined is readable. It's human-friendly. It's also nearly useless at scale. You can't ask "how many payments failed in the last hour?" without parsing strings. You can't filter by payment method without regex. You can't aggregate by user plan tier without joining to another system. You can grep, and that's about it.

Structured logging treats each log entry as a document with typed fields. Instead of a string, you emit an object: {"level":"error","event":"payment.failed","user_id":12345,"amount_cents":4999,"reason":"card_declined","payment_method":"visa"}. Now you can query, filter, aggregate, and alert on any field — without parsing strings at query time.

The cost of unstructured logs at 10K req/s

At 10,000 requests per second, each emitting 5 log lines, you have 50,000 log lines per second. To answer "how many 5xx errors in the last 5 minutes from mobile clients?", an unstructured log system needs to parse 15 million log lines. A structured log system needs to query an index. The difference is seconds versus milliseconds, and it's why teams with high-traffic services move to structured logging.

💡
Logs are a search problem, not a storage problem

Teams often focus on how to store logs cheaply. The real constraint is how to find the right log line when you're debugging at 3am. A log that's cheap to store but expensive to search is worse than a log that costs more but lets you find the error in 30 seconds. Design your log structure for the queries you'll run under pressure.

Section 02

Anatomy of a well-structured log entry

Every log entry should be a self-contained document. Someone reading it cold — no context, 3am, incident in progress — should be able to understand what happened, where, and why without needing to cross-reference other systems.

Required fields on every log entry

  • timestamp: ISO 8601 with millisecond precision. "2024-01-15T14:32:01.234Z". Always UTC, never local time. Log aggregators sort by timestamp; local time zones cause ordering chaos across distributed services.
  • level: The severity. error, warn, info, debug. Lowercase string. This is the primary filter in every log query.
  • message: A short, human-readable description of what happened. Static string where possible — the variable data goes in fields. "payment.charge.failed" not "Payment failed for user 12345".
  • service: The name of the service that emitted this log. Indispensable in a distributed system where logs from 12 services are aggregated into one stream.
  • traceId / spanId: From Day 19. The correlation keys that connect this log line to its trace.

The message field: static strings are searchable, interpolated strings are not

This is the most common structural mistake. "Payment failed for user 12345 with card ending 4567" is a unique string — it will never appear twice. You can't aggregate it, count it, or alert on it as a category. "payment.charge.failed" appears every time a payment fails — you can count it, trend it, alert on it.

The variable data — user ID, card, amount, error — goes in fields, not the message. The message names the event. The fields describe the instance.

Section 03

Log levels: what each means and when to use it

Log levels are a filtering mechanism. Every log aggregator lets you filter by level — but only if your team uses levels consistently. A codebase where everything is info or everything is error has no effective filtering.

LevelMeaningWho caresExampleProduction volume
fatalProcess cannot continue. Will crash or restart.On-call immediatelyDatabase connection pool exhausted on startupShould never happen
errorOperation failed. User or system impacted.On-call within minutesPayment charge failed, unhandled exceptionLow — alert if spikes
warnUnexpected but handled. May indicate a growing problem.Review in business hoursRetry attempt #2, rate limit approaching, deprecated API calledLow-medium

debugDetailed diagnostic. Disabled in production.Developer during local workCache key computed, function entered, variable valueHigh — never in prod

The error vs warn distinction

error means something broke that a user or system can observe. A payment failed. An API returned 500. An exception propagated to the user. These warrant alerting. warn means something unexpected happened but was handled — a retry succeeded, a fallback was used, a deprecated codepath was triggered. These warrant monitoring trends but not immediate alerts.

⚠️
Error-level inflation kills alerting

When everything is logged at error, alerts fire constantly and get ignored. When engineers add logger.error() for things that are actually warnings, the signal-to-noise ratio collapses. Fix: treat error-level as "this should alert someone right now." If you'd be annoyed to get paged for it at 2am, it's a warn, not an error.

Section 04

Context and fields: what to log with every request

A log line is only useful in context. "Payment failed" is noise. "Payment failed for user usr_123, plan=pro, amount=₹4,999, method=visa, reason=insufficient_funds, attempt=2" is actionable. The fields you attach to a log entry determine whether you can answer the question "who was affected and why" without making a second query.

The logger context pattern

Rather than repeating context on every log call, create a child logger with the request context bound to it. Every subsequent log call on that child logger includes the bound fields automatically — user ID, order ID, trace ID, request ID — without the caller having to remember to include them.

Child logger pattern — bind context once, log cleanly everywhereTYPESCRIPT
import pino from 'pino';

const baseLogger = pino({
  level: process.env.LOG_LEVEL || 'info',
  base: {
    service: 'checkout-service',
    version: process.env.APP_VERSION,
    env:     process.env.NODE_ENV,
  },
  timestamp: pino.stdTimeFunctions.isoTime,
  formatters: {
    level: (label) => ({ level: label }),  // string not number
  },
});

// Request middleware — create child logger with request context
app.use((req, res, next) => {
  req.log = baseLogger.child({
    requestId: req.headers['x-request-id'] || generateId(),
    method:    req.method,
    path:      req.route?.path || req.path,  // route pattern, not full URL
    userAgent: req.headers['user-agent'],
    ip:        req.ip,
  });
  next();
});

// Auth middleware — enrich with user context
app.use(async (req, res, next) => {
  const user = await verifyToken(req.headers.authorization);
  req.user = user;
  req.log = req.log.child({   // layer more context onto the existing child
    userId:   user.id,
    userPlan: user.plan,
    orgId:    user.orgId,
  });
  next();
});

// In your route handler — clean call sites, rich output
app.post('/checkout', async (req, res) => {
  req.log.info({ orderId }, 'checkout.started');

  try {
    const result = await processPayment(req.body);
    req.log.info({
      orderId:       result.orderId,
      amountCents:   result.amount,
      paymentMethod: result.method,
      durationMs:    result.duration,
    }, 'checkout.completed');
    res.json(result);
  } catch (err) {
    req.log.error({
      err,            // pino serialises Error.message + stack automatically
      orderId,
      amountCents: req.body.amount,
    }, 'checkout.failed');
    res.status(500).json({ error: 'checkout_failed' });
  }
});

// Output for req.log.info({orderId}, 'checkout.started'):
// {
//   "level": "info",
//   "time": "2024-01-15T14:32:01.234Z",
//   "service": "checkout-service",

Section 05

Log sampling: not every log line needs to survive

At 10,000 requests per second with 5 log lines per request, you generate 50,000 log lines per second — 4.3 billion per day. Storing all of them is expensive. More importantly, the vast majority are identical successful requests that you'll never look at. Log sampling reduces volume while preserving visibility where it matters.

What to never sample away

  • Errors and warnings: Always keep every error and warning log line. These are the minority and the most important.
  • Audit trail events: User login, permission changes, data exports, payment transactions. These often have compliance requirements.
  • Slow operations: Any request over your P99 threshold is worth keeping — it represents a user who had a bad experience.

What's safe to sample

High-volume happy-path logs: successful health checks (these are the noisiest and most useless logs in most systems), routine request logs for healthy endpoints with normal latency, debug logs that were accidentally left on. Sampling 1-10% of successful request logs while keeping 100% of errors dramatically reduces volume without hiding problems.

Dynamic log sampling — based on level and request characteristicsTYPESCRIPT
// Sampling middleware — reduces high-volume healthy-path logs
function shouldLog(level: string, req: Request, res: Response): boolean {
  // Always log errors and warnings
  if (['error', 'warn', 'fatal'].includes(level)) return true;

  // Always log slow requests (> 500ms)
  const duration = Date.now() - req.startTime;
  if (duration > 500) return true;

  // Always log non-2xx responses
  if (res.statusCode >= 300) return true;

  // Never log health check endpoints (extremely noisy)
  if (req.path === '/healthz' || req.path === '/readyz') return false;

  // Sample 10% of healthy info logs
  return Math.random() < 0.10;
}

// Response logging with sampling
app.use((req, res, next) => {
  const startTime = Date.now();
  res.on('finish', () => {
    const duration = Date.now() - startTime;
    const level = res.statusCode >= 500 ? 'error'
                : res.statusCode >= 400 ? 'warn'
                : 'info';
    if (!shouldLog(level, req, res)) return;  // skip
    req.log[level]({
      statusCode: res.statusCode,
      durationMs: duration,
      bytes:      res.get('content-length'),
    }, 'http.request.completed');
  });
  next();
});

Section 06

Log aggregation: where logs go and how to ship them

In a distributed system, logs from 20 service instances need to flow into a single queryable store. Log aggregation is the pipeline that collects, routes, and indexes them. The common patterns:

The collection pipeline

Application → stdout → container runtime → log shipper → aggregator → storage. Services write to stdout (never to files — containers are ephemeral). The container runtime (Docker, Kubernetes) captures stdout. A log shipper (Fluent Bit, Promtail, Fluentd) picks up the container logs and forwards them. The aggregator (Loki, Elasticsearch, Datadog) indexes and stores them.

Why stdout: Writing to files inside a container ties logs to that container's lifecycle. When the container dies, logs die with it. Stdout is captured by the runtime, persisted independently, and visible via kubectl logs even after the pod restarts.

✅
Grafana Loki is the cost-efficient default for most teams

Loki doesn't index log content — it indexes labels (service name, level, environment). Log content is stored compressed and scanned at query time with LogQL. This makes it dramatically cheaper than Elasticsearch (no per-field indexing), but slower for full-text search across unspecified fields. For structured logs where you know which fields you'll query, Loki's label-based model is extremely efficient. Start with Loki; move to Elasticsearch only if you need full-text search across arbitrary fields.

Section 07

Querying logs: LogQL patterns for real incidents

Grafana Loki uses LogQL — a query language modelled on PromQL. Understanding a handful of patterns covers 90% of incident investigation queries. The two stages: label selectors (which log streams to look at) and filter/parse expressions (what to do with those lines).

LogQL patterns — the queries you'll actually runLOGQL
## Basic: find errors in checkout service
{service="checkout-service"} |= "error"

## Better: parse JSON and filter by structured field
{service="checkout-service"}
  | json
  | level="error"
  | line_format "{{.time}} {{.msg}} user={{.userId}} order={{.orderId}}"

## Count errors per minute (for alerting)
count_over_time(
  {service="checkout-service"} | json | level="error" [1m]
)

## Rate of payment failures (metric from logs)
rate(
  {service="checkout-service"} | json | msg="payment.charge.failed" [5m]
)

## Find all logs for a specific trace
{service=~"checkout-service|payment-service|inventory-service"}
  | json
  | traceId="4bf92f3577b34da6a3ce929d0e0e4736"

## Slow requests: duration > 2000ms
{service="checkout-service"}
  | json
  | durationMs > 2000
  | line_format "{{.time}} {{.durationMs}}ms {{.path}} user={{.userId}}"

## Payment failures by reason in last hour
sum by (reason) (
  count_over_time(
    {service="checkout-service"}
    | json
    | msg="payment.charge.failed" [1h]
  )
)

## Error rate percentage
sum(rate({service="checkout-service"} | json | level="error" [5m]))
  /
sum(rate({service="checkout-service"} | json [5m]))
  * 100

Section 08

Log volume and cost control

Log ingestion costs money — Datadog, Elasticsearch, and CloudWatch all charge per GB ingested and stored. At high traffic, uncontrolled logging becomes a significant cost. The levers for control:

  • Log level in production: Set production log level to info. Debug logs should never reach production. A service accidentally left at debug level can generate 10-50× normal log volume.
  • Health check suppression: Kubernetes liveness and readiness probes hit health check endpoints every 5-10 seconds. That's 6-12 log lines per minute per pod that you will never look at. Never log health check requests at info level. Return 200 silently or filter them out in your log shipper before they reach storage.
  • Retention tiers: Keep 7 days of logs at full resolution, 30 days of error-only logs, 90 days of nothing (or cheap cold storage). Most debugging happens within 24 hours. 90-day log retention at full volume is usually compliance theatre, not operational necessity.
  • Compression: JSON logs compress extremely well (5-10× ratio with gzip). Most log shippers compress before shipping. If yours doesn't, enable it.
🔴
Never log request/response bodies in production

Logging full HTTP request and response bodies seems useful during development. In production it's a compliance disaster and a cost explosion. Request bodies contain passwords, tokens, PII, and payment data. Response bodies can be megabytes. Log the metadata — status code, duration, content-length — never the body. If you need request body logging for debugging, do it for specific endpoints for specific time windows, then turn it off.

Section 09

Production reality

Pino is faster than Winston — by a lot

Pino (Node.js) benchmarks at ~5× faster than Winston for JSON logging. At high throughput, logging is measurable overhead — Pino achieves this by doing minimal work in the hot path (serialisation happens asynchronously in a worker thread). If you're on Node.js and performance matters, use Pino. If you're on an early-stage service where configuration flexibility matters more than throughput, Winston is fine.

Correlation IDs are not the same as trace IDs

Many teams have X-Request-ID headers for correlating logs within a single service. Trace IDs (from Day 19) correlate across services. You need both: the request ID for "find all logs from this specific HTTP request to this service" and the trace ID for "find all logs from this request across every service it touched." Don't conflate them.

Structured logging in multiple languages

If your system has services in Python (structlog), Java (Logback + logstash-logback-encoder), and Go (zap or zerolog), the challenge is consistent field names. A log with userId in one service and user_id in another breaks cross-service log queries. Document and enforce a field naming convention — it's one of those boring things that matters enormously during incidents.

Log-based alerts: set them up before you need them

In Grafana, you can create alerts directly from LogQL queries — "alert if error rate exceeds X per minute." Set up alerts for level=error rate spikes before your first incident, not during it. The alert setup takes 10 minutes and will save hours during the first real outage.

Section 10

The things nobody tells you

Logging inside tight loops will destroy performance

A logger.debug() inside a loop processing 10,000 items generates 10,000 log calls even if the log level is set to info. Most loggers still pay the cost of evaluating arguments even when the level is suppressed. Check if your logger supports level checking before argument construction (Pino does this with logger.isLevelEnabled('debug')) or wrap loops in level checks manually.

Log shipping has backpressure problems

When your log aggregator is slow or unavailable, the log buffer in your shipper fills up. Most shippers then either drop logs (data loss) or block application threads waiting for the buffer to drain (performance impact). Know which your shipper does and configure appropriately. Fluent Bit defaults to dropping on overflow; Promtail queues on disk up to a configurable limit. Neither failure mode is obvious without testing it.

Sensitive data in logs is a compliance and security problem

It's surprisingly easy to accidentally log a full request body containing a password, log an error message that contains a credit card number, or log a JWT token in an authorization header. Establish a review checklist for new log statements. Use a sanitisation function for objects before logging: strip known sensitive fields, truncate long strings, redact patterns that look like card numbers or tokens.

Production log levels need a runtime toggle

The scenario: you're debugging an intermittent production issue and need debug logs. Deploying a new build to change the log level takes 10 minutes and may not reproduce the issue. The solution: support runtime log level changes via an admin endpoint or environment variable reload, so you can temporarily raise the log level on one pod for 5 minutes and then restore it — without a deploy.

Decision sequence

Setting up structured logging for a service

Do these in order before shipping to production.

1
Choose a logger with native JSON output
Node.js: Pino (performance) or Winston (flexibility). Python: structlog. Java: Logback + logstash-logback-encoder. Go: zap or zerolog. The logger must emit JSON natively — not formatted strings that you manually parse. If your current logger uses string templates ("Payment failed for %s"), replace it.
Pino for Node.js performance · structlog for Python
2
Define base fields: service, version, env on every log
Configure your logger's base metadata: service, version, env. These go on every single log line automatically without any call-site changes. Without them, every log query starts with "which service is this from?" — a question you should never have to ask.
base fields save minutes per incident investigation
3
Create a request-scoped child logger with traceId
In request middleware: create a child logger with requestId, method, path, and traceId. Attach it to req.log. Every subsequent log call in the request handler uses req.log — the request context is always present, never forgotten. Add user context after auth middleware resolves.
child logger = context once, everywhere
4
Agree on message naming convention: event.action not sentences
Messages should be dot-separated event identifiers: payment.charge.started, payment.charge.completed, payment.charge.failed. Not sentences. Not interpolated strings. This convention lets you search for all payment events with a prefix match and makes log counts meaningful. Document it and enforce it in code review.
interpolated messages = uncountable, unsearchable noise
5
Suppress health check logs and set level to info in production
Filter out /healthz and /readyz request logs — they're noise that inflates volume with zero debugging value. Set production log level to info. Verify no service is accidentally running at debug in production (debug level can 10× log volume). Check log aggregator cost per day after this step — the difference is usually dramatic.
health check logs are the most expensive useless logs you have
6
Set up a log-based alert for error rate before shipping
In Grafana/Datadog/CloudWatch: create an alert that fires when level=error log rate exceeds your baseline by 2× for 5 minutes. This alert is free to create and will fire before your metrics-based alerts in many incident scenarios — logs are closer to the application than infrastructure metrics. Do it now, not during the first incident.

Section 12

Structured logging checklist

JSN
JSON output

Every log line is valid JSON. No string formatting. Fields are typed: numbers as numbers, booleans as booleans.

CTX
Request context

Every log line has: service, requestId, traceId, userId. Set once in middleware, not repeated at call sites.

MSG
Static messages

Messages are event identifiers: payment.charge.failed, not "Payment failed for user 123." Variable data in fields.

LVL
Correct levels

error = user impacted, alert now. warn = handled but unexpected. info = normal ops. debug = never in prod.

HLT
No health logs

/healthz and /readyz requests are not logged. They generate thousands of useless lines per pod per day.

ALT
Log-based alert

Alert fires when error log rate spikes. Set up before the first incident, not during it.
