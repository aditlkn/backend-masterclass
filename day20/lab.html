Side-by-side comparison  ·  Same incident  ·  Different logs

3 hours of grepping vs 90 seconds of querying. Same production bug.

Payment failure rate spike · Friday 18:00 IST · Two teams, two log setups

Two different teams. Same incident: payment failure rate spiked from 2% to 18% at 18:00 on a Friday. Team A has unstructured logs. Team B has structured logs. The bug — a misconfigured currency conversion causing payments over ₹5,000 to fail on international cards — took one team 3 hours to find and one team 90 seconds.

Team A's logs have lines like Payment processing failed for customer: card declined. They can grep for "failed" and count lines, but they can't tell: which cards? What amount threshold? Which payment method? They start adding debug logging, deploying, waiting for the bug to recur. Three hours later they find it by adding console.log statements.

Team B runs one LogQL query: group payment failures by paymentMethod and amountCents range. Result arrives in 800ms. International Visa and Mastercard cards, exclusively on transactions above ₹5,000. They look at the currency conversion code deployed at 17:55. Fix deployed at 18:32. Total time: 32 minutes.

LogQL query
{service="checkout-service"} | json | msg="payment.charge.failed" | paymentMethod=~"visa|mastercard" | amountCents > 500000 — if this returns only the failures, ₹5000 (500000 paise) is the threshold

The fix — currency conversion bug revealed by logsTYPESCRIPT
// Bug: deployed at 17:55 in currency-utils.ts
// The INR to "payment processor units" conversion was wrong
// Payment processors expect amounts in the LOWEST denomination
// INR: 1 rupee = 100 paise. ₹5000 = 500000 paise
// The new code was dividing by 100 instead of keeping as-is

// Before (correct):
function convertToProcessorUnits(amountInRupees: number): number {
  return amountInRupees * 100;  // ₹50 → 5000 paise ✓
}

// After deploy at 17:55 (broken):
function convertToProcessorUnits(amountInRupees: number): number {
  return amountInRupees / 100;  // ₹5000 → 50 (too small) — card declines
  // ₹50 → 0.5 — rounds to 0 — card declines
  // Small amounts still worked because even divided by 100, they were
  // above the minimum transaction threshold (₹1000 = 10 units — passes)
  // Large amounts (₹5000 = 50 units) — below minimum — declines
}

// The structured log that revealed it:
// { "msg": "payment.charge.failed", "amountCents": 500000,
//   "processorAmount": 50, "reason": "below_minimum_transaction" }
// ← "processorAmount: 50" is the smoking gun. Should be 500000.

// Fix (deployed at 18:32):
function convertToProcessorUnits(amountInRupees: number): number {
  return Math.round(amountInRupees * 100);  // ₹5000 → 500000 ✓
}
What this incident teaches about structured logging
▶The fields you log determine the questions you can answer. Team A couldn't answer "which cards are failing at which amounts" because that information wasn't in their logs. By the time they added the logging, 2 hours had passed. Team B could answer it in one query because the fields were already there.
▶Log the derived values as well as the inputs. The smoking gun was processorAmount: 50 — the value after conversion, not the input amount. Logging only amountCents: 500000 would have shown the right input. Logging the converted value revealed the bug in the conversion function.
▶Deploy events should be in your log stream. The query that found the deploy.completed event at 17:55 immediately correlated the failure spike with a deployment. Without that, the debugging timeline gets longer. Log deployments as structured events: version, component, who triggered it.
▶Numeric fields enable range queries. amountCents > 500000 was the query that found the threshold. This only works because amountCents was logged as an integer, not as a formatted string like "₹5,000". Always log monetary amounts as integers (lowest denomination). Always log durations as integers (milliseconds).
