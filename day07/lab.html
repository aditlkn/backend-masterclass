LAB
Production Incident  ·  gRPC  ·  Load Balancing

One backend at 100% CPU, three completely idle — the LB dashboard showed "healthy"

Post gRPC migration  ·  Duration: 2 weeks unnoticed  ·  Severity: P2  ·  Backend instance B at constant 100% CPU

The team had spent three months migrating three internal services from REST to gRPC. Cleaner contracts, deadline propagation, faster serialisation — all the right reasons. They deployed behind the same HAProxy instance that had been load-balancing their REST traffic for two years. HAProxy showed all backends healthy. Dashboards were green.

Two weeks after the migration, a routine capacity review found something strange: one of the four backend instances was at 98-100% CPU at all times. The other three were at 8-12%. Not a traffic routing issue — HAProxy's stats page showed roughly equal connection counts. But CPU told a completely different story.

Week 2 post-migration
Capacity review: Backend B at 98% CPU constant. Backends A, C, D at 8-12%. HAProxy shows 4 backends, all healthy, connections ~25% each.
Investigation Day 1
Backend B has 4 active HTTP/2 connections. A, C, D have 1 each. HAProxy is distributing TCP connections evenly — but all gRPC calls from client pods are going through those 4 connections on Backend B.
Investigation Day 1
Root cause found: HAProxy was configured as an L4 TCP proxy. It distributed the initial HTTP/2 connections evenly. But each gRPC client maintains a single persistent HTTP/2 connection — all RPCs from that client multiplex over one connection. Client pods that connected to Backend B send all their gRPC calls there.
Investigation Day 2
Fix: Switch HAProxy to HTTP/2 mode (L7). HAProxy terminates HTTP/2 connections and routes individual gRPC streams to backends. CPU evens out to ~25% each within 60 seconds.
Root cause

L4 TCP load balancers distribute connections, not requests. With REST over HTTP/1.1, connections are short-lived — each request is essentially a connection. Distribution is effectively per-request. With gRPC over HTTP/2, one connection multiplexes all RPCs indefinitely. A client that connects to Backend B sends every single gRPC call to Backend B for the lifetime of that connection. The load balancer's connection-count stats looked balanced. The actual RPC workload was not.

What this incident teaches
▶L4 LB + gRPC = not load balanced. L4 distributes connections. gRPC reuses connections. The result is that all RPCs from any given client stick to one backend indefinitely.
▶Connection count metrics lie about gRPC load distribution. HAProxy showed 25% connections to each backend — technically true. But 80% of RPCs were going to Backend B. Check CPU, not connection count.
▶When you change the protocol, revisit your load balancer config. HAProxy L4 was perfectly correct for REST/HTTP1.1. It became wrong the moment gRPC/HTTP2 was introduced. Protocol migration requires infrastructure review.
▶Three solutions: L7 LB, Envoy sidecar, or client-side LB. All solve the problem. L7 LB is simplest operationally. Envoy gives you more (mTLS, circuit breaking, observability). Client-side is fastest (no proxy hop).
