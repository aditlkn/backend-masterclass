Deep Dive 01

Protobuf encoding
at the byte level

Understanding the exact byte encoding matters for two reasons: it explains why field numbers are sacred (they're in the wire format, names aren't), and it helps you make informed decisions about field types and number assignments in performance-critical messages.

Varint: how small numbers use fewer bytes

The most common Protobuf wire type is varint — variable-length integer encoding. The lower 7 bits of each byte carry data. The highest bit is a "continuation bit" — 1 means another byte follows, 0 means this is the last byte. This means:

  • Values 0-127: 1 byte
  • Values 128-16383: 2 bytes
  • Values 16384-2097151: 3 bytes
  • Values up to 2^28: 4 bytes
  • Values up to 2^35: 5 bytes

For a payment amount in paise (INR cents), ₹500 = 50000 paise. 50000 in binary is 1100001101010000 — 16 bits. As a varint: 3 bytes. As a JSON number: 5 characters = 5 bytes. Not huge, but at 100,000 messages/second with 20 numeric fields, the difference compounds.

The tag byte: field number + wire type in 1-2 bytes

Every field in a Protobuf message starts with a tag: tag = (field_number << 3) | wire_type. Wire types: 0 = varint, 1 = 64-bit, 2 = length-delimited, 5 = 32-bit. For field numbers 1-15, the tag fits in one byte. For field numbers 16-2047, the tag needs two bytes. This is why high-frequency fields should have low field numbers.

🔴
Never use int32/int64 for negative numbers

Proto3's int32 and int64 encode negative numbers as 10-byte varints (because they sign-extend to 64 bits). A single -1 encoded as int64 is 10 bytes. Use sint32 or sint64 for any field that can be negative — they use zigzag encoding and produce compact varints for negative values too.

Deep Dive 02

Deadline propagation:
the full implementation

Setting a deadline is one line. Propagating it correctly through a multi-service call chain, handling cancellation, and aborting work when the deadline fires requires a consistent pattern applied at every service boundary.

The deadline is an absolute timestamp, not a duration

This is the most important implementation detail. The client sets a deadline as an absolute wall-clock time (e.g., Date.now() + 5000). This timestamp travels in the grpc-timeout header as a duration from "now" — but each service in the chain converts it back to an absolute timestamp. When Service A forwards to Service B, it calculates the remaining time: original_deadline - now. Service B receives the remaining duration, not the original 5 seconds.

This propagation is automatic in gRPC — you don't manually pass the deadline. When you create a child call with { deadline: parentCall.getDeadline() }, the framework handles the math. But you must explicitly pass the parent's deadline — the framework doesn't infer it.

Complete deadline propagation — client, gateway, and downstream serviceTYPESCRIPT
import * as grpc from '@grpc/grpc-js';

// ── CLIENT: always set a deadline ──────────────────────────────────
async function placeOrder(request: OrderRequest): Promise<OrderResponse> {
  const deadline = new Date(Date.now() + 5000); // absolute: 5s from now
  return new Promise((resolve, reject) => {
    orderClient.PlaceOrder(request, { deadline }, (err, response) => {
      if (err) {
        if (err.code === grpc.status.DEADLINE_EXCEEDED) {
          // Log remaining time for debugging
          logger.warn({ deadline: deadline.toISOString(), overrunMs: Date.now() - deadline.getTime() }, 'order deadline exceeded');
        }
        return reject(err);
      }
      resolve(response!);
    });
  });
}

// ── SERVICE A (Order Service): propagate deadline to downstream ──────
class OrderService {
  async PlaceOrder(call: grpc.ServerUnaryCall<OrderRequest, OrderResponse>, cb: grpc.sendUnaryData<OrderResponse>) {
    // Get the deadline from the incoming call
    const deadline = call.getDeadline(); // absolute Date object
    const remainingMs = deadline instanceof Date ? deadline.getTime() - Date.now() : Infinity;

    // Fail fast if not enough time left
    if (remainingMs < 200) {
      logger.warn({ remainingMs }, 'insufficient deadline — failing fast');
      return cb({ code: grpc.status.DEADLINE_EXCEEDED, message: 'Insufficient time remaining' });
    }

    // Listen for cancellation (client gave up or deadline fired upstream)
    let cancelled = false;
    call.on('cancelled', () => {
      cancelled = true;
      logger.info('Call cancelled by client — aborting downstream work');
    });

    try {
      // Call Payment Service with THE SAME deadline
      // NOT a new deadline — pass the parent deadline through
      const charge = await new Promise((resolve, reject) => {
        paymentClient.Charge(
          { amount: call.request.amount_paise, user_id: call.request.user_id },
          { deadline }, // ← same deadline, not Date.now() + 5000
          (err, res) => err ? reject(err) : resolve(res!)
        );
      });

      if (cancelled) return; // don't send response if client is gone

      // Database write — also check if cancelled before committing
      const order = await db.createOrder({ ..., charge_id: charge.id });
      if (cancelled) return; // client gave up while we wrote — that's ok, idempotency handles it

      cb(null, { order_id: order.id, status: 'CREATED' });
    } catch (err: any) {
      const code = err.code === grpc.status.DEADLINE_EXCEEDED
        ? grpc.status.DEADLINE_EXCEEDED // propagate upstream deadline
        : grpc.status.INTERNAL;
      cb({ code, message: err.message });
    }
  }
}

// ── PATTERN: deadline budget allocation ───────────────────────────
// If you need to call multiple services, allocate the deadline across them
// Not hard-coded splits — proportional to what each service typically needs
function allocateDeadline(parentDeadline: Date, fractionForThisCall: number): Date {
  const remaining = parentDeadline.getTime() - Date.now();
  const allocated = remaining * fractionForThisCall;
  return new Date(Date.now() + allocated);
}

Deep Dive 03

buf.build CI pipeline:
lint, breaking changes, generate, push

buf's value is proportional to how early in the development cycle it catches problems. A breaking change detected in CI before merge is free. The same change detected after deployment costs hours of incident investigation and a migration project. Here's the complete pipeline.

buf.yaml + buf.gen.yaml — project configurationYAML
# buf.yaml — in your proto directory root
version: v1
name: buf.build/yourorg/payments   # buf registry module name
lint:
  use:
    - DEFAULT                      # all standard lint rules
  except:
    - PACKAGE_VERSION_SUFFIX       # if you use versioned packages already
  ignore_only:
    RPC_REQUEST_RESPONSE_UNIQUE:   # allow shared request/response types
      - payments/v1/charge.proto
breaking:
  use:
    - FILE                         # file-level compatibility (recommended)
    # Alternatives:
    # - PACKAGE  (less strict, allows adding fields across the package)
    # - WIRE     (strictest, only wire-format compatible changes)
deps:
  - buf.build/googleapis/googleapis  # if using google.api annotations

---
# buf.gen.yaml — code generation config
version: v1
managed:
  enabled: true
  go_package_prefix:
    default: github.com/yourorg/payments/gen/go
plugins:
  # Go: generate both proto and gRPC code
  - plugin: buf.build/protocolbuffers/go
    out: gen/go
    opt: paths=source_relative

  - plugin: buf.build/grpc/go
    out: gen/go
    opt:
      - paths=source_relative
      - require_unimplemented_servers=false

  # TypeScript: using connect-es for modern TS gRPC
  - plugin: buf.build/bufbuild/es
    out: gen/ts
    opt: target=ts

  - plugin: buf.build/bufbuild/connect-es
    out: gen/ts
    opt: target=ts

  # Python (if needed)
  - plugin: buf.build/protocolbuffers/python
    out: gen/python
GitHub Actions CI pipeline — lint, breaking change, generate, pushYAML
# .github/workflows/proto.yml
name: Proto CI

on:
  push:
    branches: [main]
    paths: ['proto/**']
  pull_request:
    paths: ['proto/**']

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: bufbuild/buf-setup-action@v1
        with:
          version: '1.28.1'          # pin version for reproducibility
          buf_api_token: ${{ secrets.BUF_API_TOKEN }}

      # Lint: naming, style, package structure
      - name: Lint proto files
        run: buf lint proto/
        # Fails on: wrong case (PascalCase for messages, snake_case for fields)
        # Fails on: missing package declaration
        # Fails on: service methods without request/response suffix

      # Breaking change detection: compare against main branch
      - name: Check for breaking changes
        run: |
          buf breaking proto/ --against '.git#branch=main,subdir=proto'
        if: github.event_name == 'pull_request'
        # Fails on: deleted fields, changed field numbers, changed types
        # Fails on: removed RPCs, changed streaming patterns
        # Safe: added fields, added RPCs, renamed fields (same number)

  generate:
    needs: lint
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'  # only generate on merge to main
    steps:
      - uses: actions/checkout@v4
      - uses: bufbuild/buf-setup-action@v1
        with:
          buf_api_token: ${{ secrets.BUF_API_TOKEN }}

      # Generate code in all configured languages
      - name: Generate code
        run: buf generate proto/

      # Commit generated code back to repo
      - name: Commit generated code
        uses: stefanzweifel/git-auto-commit-action@v5
        with:
          commit_message: 'chore: regenerate proto code [skip ci]'
          file_pattern: 'gen/**'

  push-registry:
    needs: generate
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'
    steps:
      - uses: actions/checkout@v4
      - uses: bufbuild/buf-setup-action@v1
        with:
          buf_api_token: ${{ secrets.BUF_API_TOKEN }}

      # Push schema to buf.build registry for sharing
      - name: Push to buf registry
        run: buf push proto/ --tag ${{ github.sha }}
        # Other teams can depend on: buf.build/yourorg/payments
        # Version pinning: buf.build/yourorg/payments:<commit-sha>
