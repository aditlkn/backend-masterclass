Section 01

Why gRPC exists: the problems REST doesn't solve well

REST is excellent for what it was designed for: resource-oriented APIs consumed by browsers and third-party developers. It's built on HTTP/1.1 semantics that are universally understood and toolable. But REST has structural limitations that become painful at scale in a microservices architecture.

The interface definition problem. REST has no standard schema definition. OpenAPI (Swagger) is widely used but optional and frequently out of sync with actual server behaviour. A REST client learns the API shape from documentation, not from a machine-readable contract that both client and server are compiled against. When you change a field name in a REST API, nothing stops you from shipping the server and forgetting to update a client. You find out when production breaks.

The streaming problem. HTTP/1.1 wasn't designed for long-lived bidirectional streams. WebSockets are a hack layered on top. Server-sent events are one-directional. Building a real-time bidirectional protocol on REST requires significant custom engineering. gRPC's streaming is first-class and works the same way for unary, server-streaming, client-streaming, and bidirectional use cases.

The deadline problem. In REST, if a client sets a 5-second timeout, that information dies at the first service boundary. Service A knows the client wants a response in 5 seconds, but when it calls Service B, Service B doesn't know there are only 3 seconds left. gRPC propagates deadlines across every service call in the chain — Service B and Service C all know the original deadline and can fail fast rather than doing work that will be discarded.

The performance problem. JSON is text-based and verbose. Field names are repeated in every object. Numbers are serialised as characters. A typical microservice message in JSON is 3-10x larger than its Protobuf equivalent, and JSON parsing is significantly more CPU-intensive than Protobuf decoding. For high-frequency internal service calls at thousands per second, this compounds.

gRPC solves all four. It's built on HTTP/2 (multiplexed streams, binary framing), uses Protocol Buffers (schema-enforced, binary encoded), propagates deadlines as first-class header metadata, and generates client and server code in any language from the same schema definition.

Section 02

Protocol Buffers: how binary encoding actually works

Protobuf is not just "JSON but binary." The encoding is fundamentally different in ways that matter for both size and forward/backward compatibility. Understanding the encoding helps you understand why certain schema changes are safe and others are breaking.

The key insight: field names are not encoded

In JSON, field names are strings in the payload: {"user_id": 123, "email": "a@b.com"}. Every byte of "user_id" is transmitted every time. In Protobuf, field names don't exist in the wire format. Instead, each field has a field number — a small integer defined in the .proto schema. The wire format encodes the field number, a wire type (varint, length-delimited, etc.), and the value. The decoder looks up the field name from the compiled schema.

This is why Protobuf is compact. A field named payment_method_type (21 bytes as a JSON key) becomes field number 3 (1-2 bytes in Protobuf, depending on the number). A message with 20 fields saves the key name overhead on every field for every message. At thousands of messages per second, this is significant CPU and bandwidth.

Wire types

Protobuf has 6 wire types that determine how many bytes follow:

  • Varint (0): Variable-length integer encoding. Small numbers use fewer bytes. The number 1 encodes in 1 byte; the number 300 in 2 bytes; 2^28 in 5 bytes. int32, int64, bool, enum.
  • 64-bit (1): Fixed 8 bytes. fixed64, double.
  • Length-delimited (2): A varint length followed by that many bytes. Used for string, bytes, nested messages, and packed repeated fields.
  • 32-bit (5): Fixed 4 bytes. fixed32, float.

The tag byte: field number + wire type in one

Each field in the wire format starts with a tag — a varint encoding both the field number and wire type: tag = (field_number << 3) | wire_type. Field 1, wire type 0 (varint) = tag value 8 = 1 byte. Field 1, wire type 2 (length-delimited) = tag value 10 = 1 byte. This is why small field numbers are important — they keep tags to 1 byte.

💡
Why field numbers are sacred

The wire format has no field names — just field numbers. If you renumber field 3 to field 5, old clients sending field 3 will have their data silently ignored by new servers expecting field 5. This is not an error — it's how Protobuf achieves forward compatibility. Unknown fields are preserved but ignored. Changing field numbers is the single most dangerous schema change you can make.

Proto3 message definition and what the wire encoding looks like PROTO 
// payments.proto
syntax = "proto3";
package payments.v1;

message ChargeRequest {
  string  idempotency_key = 1;   // field 1, wire type 2 (string)
  int64   amount_paise    = 2;   // field 2, wire type 0 (varint)
  string  user_id         = 3;   // field 3, wire type 2
  string  currency        = 4;   // field 4, wire type 2
  PaymentMethod method    = 5;   // field 5, wire type 2 (nested message)
}

// JSON equivalent: 94 bytes
// {"idempotency_key":"idem_abc123","amount_paise":50000,"user_id":"usr_789","currency":"INR"}

// Protobuf encoding: ~45 bytes (approximately, no method field)
// 0a 0a 69 64 65 6d 5f 61 62 63 31 32 33   ← field 1: "idem_abc123"
// 10 d0 86 03                               ← field 2: 50000 (varint)
// 1a 07 75 73 72 5f 37 38 39               ← field 3: "usr_789"
// 22 03 49 4e 52                           ← field 4: "INR"
// Total: ~48% smaller than JSON

// Service definition
service PaymentService {
  // Unary: one request, one response
  rpc Charge (ChargeRequest) returns (ChargeResponse);

  // Server streaming: one request, stream of responses
  rpc StreamTransactions (TransactionFilter) returns (stream Transaction);

  // Client streaming: stream of requests, one response
  rpc BatchCharge (stream ChargeRequest) returns (BatchChargeResponse);

  // Bidirectional: both sides stream simultaneously
  rpc RealtimePaymentFeed (stream FeedRequest) returns (stream PaymentEvent);
}

Section 03

The 4 streaming patterns: when to use each

gRPC's streaming is one of its most powerful features and the one most teams underuse. Most gRPC adopters only use unary RPCs — which is fine, but leaves the streaming advantage on the table for the use cases where it matters most.

Unary: request-response (what you already know)

One request, one response. Functionally identical to a REST API call, but with binary encoding, schema enforcement, and deadline propagation. The right choice for the majority of service calls: CRUD operations, authentication checks, any operation where you need a single answer.

Server streaming: one request, many responses

Client sends one request; server streams back a sequence of responses on the same HTTP/2 stream until done. The server controls when the stream ends. Use cases: exporting large datasets (stream rows instead of buffering everything in memory), live dashboards (stream updates as data changes), log tailing, real-time analytics feeds. The key advantage over REST polling: the server pushes data; the client doesn't need to poll. Lower latency, lower overhead.

Client streaming: many requests, one response

Client streams a sequence of requests; server accumulates them and sends one final response. Use cases: file upload (stream chunks, server returns a single completion response), batch operations where you want to stream input rather than send one giant payload, telemetry ingestion (stream events, server returns an ack with count). Rare in practice — most batch patterns are better served by a message queue. But for synchronous bulk ingestion with backpressure, it's elegant.

Bidirectional streaming: both sides stream simultaneously

Both client and server send and receive independently on the same connection. Neither side needs to wait for the other. Use cases: real-time chat, collaborative editing, trading/order book updates, game state synchronisation, voice/video control channels. This is the pattern that previously required WebSockets or custom TCP protocols — gRPC makes it a first-class RPC pattern with all the schema and deadline benefits.

All 4 streaming patterns in Node.js TYPESCRIPT 
import * as grpc from '@grpc/grpc-js';

// 1. UNARY — one request, one response
const deadline = new Date(Date.now() + 5000); // always set deadlines
client.Charge(request, { deadline }, (err, response) => {
  if (err) throw err;
  console.log(response.charge_id);
});

// 2. SERVER STREAMING — stream large dataset
const stream = client.StreamTransactions(
  { user_id: 'usr_123', from_date: '2024-01-01' },
  { deadline: new Date(Date.now() + 30000) }
);
stream.on('data', (transaction) => {
  processTransaction(transaction); // process as data arrives
});
stream.on('end',   () => console.log('stream complete'));
stream.on('error', (err) => console.error(err));

// 3. CLIENT STREAMING — batch ingest
const batchCall = client.BatchCharge(
  { deadline: new Date(Date.now() + 60000) },
  (err, result) => {
    console.log(`Processed \${result.charged_count} charges`);
  }
);
for (const charge of charges) {
  batchCall.write(charge);  // stream requests to server
}
batchCall.end();             // signal end of client stream

// 4. BIDIRECTIONAL STREAMING — realtime feed
const feed = client.RealtimePaymentFeed({ deadline: ... });
feed.on('data', (event) => renderUpdate(event));
feed.write({ filter: { currency: 'INR', min_amount: 1000 } });
// Later: update filter without reconnecting
feed.write({ filter: { currency: 'INR', min_amount: 5000 } });

Section 04

Deadline propagation: the feature that prevents cascading failures

Deadlines are the single most important gRPC feature that has no equivalent in REST. They're so fundamental that the gRPC framework enforces them at the protocol level — not as an application concern but as a network-layer concern.

How gRPC deadlines work

When a gRPC call is made with a deadline, the remaining time is encoded in the request metadata as a grpc-timeout header. Every service in the call chain receives this header and knows exactly how much time is left from the original client's perspective. Crucially: the deadline doesn't reset at each service boundary. If the client set 5 seconds and Service A took 2 seconds to process before calling Service B, Service B receives a 3-second deadline — not 5 seconds.

This means Service B can check its deadline before starting expensive work. If the deadline is 200ms and Service B knows its DB query typically takes 500ms, it can immediately return a DEADLINE_EXCEEDED error rather than starting work that will be abandoned when the deadline fires anyway. This is deadline-aware processing, and it's the key to preventing the cascade where Service A's slow response causes Service B to do useless work.

The REST equivalent doesn't exist

In REST, a client timeout is a client-side concept only. You can set a 5-second timeout on your HTTP client. When you call Service A, Service A doesn't know about your timeout. When Service A calls Service B, Service B definitely doesn't know. You might add a custom X-Request-Timeout header and have every service read and propagate it — but this is manual, inconsistent, and not enforced by the framework.

Context cancellation propagation

When a client cancels a gRPC call (because its deadline expired or the user navigated away), the cancellation propagates through the entire call chain. Service A's context is cancelled, which cancels Service B's context, which cancels the database query. Work is abandoned at every level simultaneously rather than continuing to consume resources for a response nobody will receive.

Deadline propagation — always set, always check TYPESCRIPT 
// WRONG: No deadline — call hangs forever if downstream is slow
client.Charge(request, (err, response) => { /* ... */ });

// RIGHT: Absolute deadline (wall clock time, not duration)
const deadline = new Date(Date.now() + 5000); // 5s from now
client.Charge(request, { deadline }, (err, response) => {
  if (err?.code === grpc.status.DEADLINE_EXCEEDED) {
    // Timed out — log with remaining time info for debugging
    logger.warn({ deadline: deadline.toISOString() }, 'charge deadline exceeded');
  }
});

// Server-side: check deadline BEFORE expensive work
async function Charge(call: ServerUnaryCall, callback: sendUnaryData) {
  // Check if client already gave up before we start
  const deadline = call.getDeadline();
  const remainingMs = deadline - Date.now();

  if (remainingMs < 100) {
    // Less than 100ms left — not enough time to do anything useful
    return callback({ code: grpc.status.DEADLINE_EXCEEDED });
  }

  // Listen for cancellation — stop DB work if client cancelled
  let cancelled = false;
  call.on('cancelled', () => { cancelled = true; });

  try {
    const charge = await db.createCharge(call.request);
    if (cancelled) return; // client gave up while we were working
    callback(null, { charge_id: charge.id, status: 'SUCCESS' });
  } catch (err) {
    callback({ code: grpc.status.INTERNAL, message: err.message });
  }
}

Section 05

Schema evolution: the rules that make or break compatibility

Protobuf's binary encoding enables backward and forward compatibility — but only if you follow the rules. The encoding's lack of field names means that compatible changes are genuinely compatible across versions, and breaking changes are genuinely breaking in ways that can be subtle and silent.

Safe changes (backward and forward compatible)

  • Adding a new field with a new field number: Old clients ignore unknown fields. New servers return default values for unset fields. Both directions work.
  • Deleting a field: The field number should be reserved so it's never reused. Old clients sending deleted fields have their data ignored. New clients receive default values. Compatible, but use reserved to prevent accidental field number reuse.
  • Renaming a field: The field name doesn't exist in the wire format — only the number does. You can rename field 3 from user_id to customer_id and the wire format is identical. This is one of Protobuf's most useful properties.
  • Changing a singular field to repeated: Old messages with one value become a one-element list. Compatible.

Breaking changes — never do these in production

  • Changing a field number: If field 3 was user_id and you renumber it to field 5, old clients send field 3 which new servers don't recognise. The data is silently dropped. This is the most dangerous change.
  • Changing a field's wire type: Changing int32 to string changes the wire type from varint to length-delimited. The decoder will misinterpret the bytes. Silent data corruption.
  • Reusing a field number: If you delete field 3 and then add a new field also numbered 3 with a different type, old messages with the old field 3 will be decoded as the new field 3. Data corruption.

The reserved keyword

Proto3 has a reserved statement that prevents field number and name reuse: reserved 3, 5; reserved "old_field_name";. The compiler will error if any future field tries to use these numbers or names. Always reserve deleted field numbers and names.

Safe schema evolution with reserved fields PROTO 
// Version 1 of the message
message ChargeRequest {
  string idempotency_key = 1;
  int64  amount_paise    = 2;
  string user_id         = 3; // will be deleted in v2
  string currency        = 4;
}

// Version 2: user_id moved to metadata, new customer_reference added
message ChargeRequest {
  string idempotency_key     = 1;
  int64  amount_paise         = 2;
  // Field 3 deleted — MUST be reserved or someone will reuse it
  string currency             = 4;
  string customer_reference   = 5; // new field — safe to add
  PaymentMethod method        = 6; // new nested message — safe

  // Prevent field number 3 and name "user_id" from being reused
  reserved 3;
  reserved "user_id";
}

// buf.build breaking change detection catches this automatically:
// buf breaking --against .git#branch=main
// ERROR: payments/v1/charge.proto:4:3:
#   Field "3" on message "ChargeRequest" changed type from
#   "TYPE_STRING" to "TYPE_INT64" -- this is a breaking change

Section 06

Tooling: buf, grpcurl, and the gRPC ecosystem

One of REST's genuine advantages is toolability — curl works, browsers work, Postman works. gRPC's binary protocol makes ad-hoc testing harder. The ecosystem has caught up significantly, but you need to know which tools to reach for.

buf: the modern Protobuf toolchain

buf is the most important tool in the gRPC ecosystem. It replaces the raw protoc compiler with a tool that handles dependency management, linting, breaking change detection, and a registry (buf.build) for sharing schemas.

  • buf lint — enforces style rules (field names, service names, package structure). Catches naming inconsistencies before code review.
  • buf breaking --against .git#branch=main — detects breaking changes in your Protobuf schemas against a previous version. Runs in CI. Blocks PRs that would break client compatibility.
  • buf generate — generates client and server code from .proto files. Replace the complex protoc invocations with a simple config file.
  • buf push — publish schemas to buf.build for team sharing and dependency management.

grpcurl: curl for gRPC

grpcurl lets you call gRPC services from the command line. Requires either server reflection enabled (the server advertises its schemas) or a .proto file. Equivalent to curl for REST debugging.

Evans: interactive gRPC client

Evans is an interactive REPL for gRPC — like a gRPC equivalent of psql or redis-cli. Tab-completion of service and method names, interactive input of request messages. Excellent for exploration and debugging.

Load balancing in gRPC: why it's harder than HTTP and how to solve it


  
gRPC's use of HTTP/2 creates a load balancing challenge that most teams don't anticipate until they hit it in production. With HTTP/1.1 REST APIs, each request is a new TCP connection (or a connection from a pool). Load balancers distribute connections, which distributes requests. With gRPC over HTTP/2, a single long-lived connection multiplexes all requests. If a client maintains one connection to a server, all its requests go to that server regardless of the load balancer sitting in between.


  
Concretely: you have 5 gRPC backend instances behind a load balancer. A client establishes one HTTP/2 connection. The load balancer routes that connection to Instance 2. All subsequent gRPC calls from that client go to Instance 2. Instances 1, 3, 4, and 5 see nothing from this client. This is the opposite of load balancing.


  
Solution 1: L7 load balancing (recommended)

  
Use an L7-aware load balancer that understands HTTP/2 — Envoy, nginx (1.13.10+), or a service mesh. These terminate the HTTP/2 connection and distribute individual gRPC frames (not just TCP connections) across backends. Each request gets independently routed, even though the client has one connection. This is the simplest solution for most architectures.


  
Solution 2: Client-side load balancing

  
The gRPC client library knows about all backend instances (from DNS or a service registry) and distributes requests across them directly. No load balancer in the path — the client implements the balancing algorithm. gRPC's built-in round_robin and pick_first load balancers, and Envoy's LEAST_REQUEST, work at this layer. Requires the client to know backend addresses — typically via DNS with multiple A records or a service discovery system like Consul.


  
Solution 3: Proxy with gRPC awareness

  
AWS ALB supports gRPC natively (since 2020). GCP Cloud Load Balancing supports gRPC. These are the easiest solutions for cloud-hosted gRPC services — they handle HTTP/2 termination and per-request routing automatically.

Section 08

  
gRPC vs REST: when to use which


  
This is not a debate with a winner. They're tools with different tradeoffs. The right choice depends on who your clients are, what your communication patterns are, and what your team is willing to operate.


  
Use REST when:

  
    
    
  • External/public API: Browsers, mobile apps, third-party developers, any client you don't control. REST is universally understood. gRPC requires generated client libraries or transcoding. External developers expect REST.
    • 
    
  • Simple CRUD operations: Creating, reading, updating, deleting resources with no streaming. REST's resource model is elegant and well understood. gRPC is overkill.
    • 
    
  • Human-readable debugging matters: REST+JSON can be inspected with curl, browser DevTools, Postman. No special tooling required. gRPC requires grpcurl or Evans.
    • 
    
  • The team is small or REST-experienced: gRPC introduces proto files, code generation, a new mental model for streaming, and unfamiliar error codes. This overhead is real and shouldn't be underestimated.
    • 
  


  
Use gRPC when:

  
    
    
  • Internal service-to-service: You control both client and server. Generated clients are an advantage, not a burden. Binary encoding and deadline propagation have clear value. This is the primary use case.
    • 
    
  • Streaming is required: Real-time updates, large data exports, bidirectional communication. gRPC streaming is vastly cleaner than WebSockets or polling.
    • 
    
  • High-frequency, low-latency internal calls: Hundreds of thousands of small messages per second between services. The encoding efficiency and HTTP/2 multiplexing make a measurable difference.
    • 
    
  • Polyglot services: Protobuf generates idiomatic clients in Go, Rust, Python, Java, TypeScript simultaneously from one schema. REST has OpenAPI generators but they're inconsistent across languages.
    • 
    
  • Schema enforcement is critical: Proto files are the contract. Breaking changes are caught by tooling (buf) before deployment. REST contract enforcement requires discipline; gRPC enforces it structurally.
    •

Section 09

  
Production reality: gRPC surprises


  
gRPC-Web: the browser gap

  
Standard gRPC requires HTTP/2 trailers — a feature browsers don't support via the Fetch API. You cannot call gRPC directly from a browser. The solutions: gRPC-Web (a modified protocol that works with browsers, requires an Envoy proxy to translate), or just use REST for the browser layer and gRPC for internal services. Most production architectures do the latter: REST gateway → gRPC internal services.


  
Error codes don't map to HTTP status codes cleanly

  
gRPC has 16 status codes: OK, CANCELLED, UNKNOWN, INVALID_ARGUMENT, NOT_FOUND, ALREADY_EXISTS, PERMISSION_DENIED, UNAUTHENTICATED, RESOURCE_EXHAUSTED, and others. When exposing gRPC services via an HTTP transcoding layer (grpc-gateway, Envoy HTTP transcoding), mapping these to HTTP status codes requires explicit configuration. INVALID_ARGUMENT → 400, NOT_FOUND → 404, UNAUTHENTICATED → 401 — but RESOURCE_EXHAUSTED might mean 429 (rate limited) or 503 (overloaded) depending on context.


  
Proto files are your API contract — treat them like production code

  
Teams that treat .proto files as implementation details rather than API contracts end up with exactly the problems Protobuf was designed to prevent. Run buf breaking in CI. Review schema changes as carefully as code changes. Version your packages (payments.v1, payments.v2) rather than making breaking changes in place. The schema is the contract that every client in every language is compiled against.


  
The health check protocol

  
gRPC has a standard health check protocol: grpc.health.v1.Health. Implement it. Kubernetes, Envoy, and AWS ALB all know how to use it for readiness and liveness probes. Using a custom /healthz HTTP endpoint alongside a gRPC service means running two servers or adding an HTTP bridge. The gRPC health protocol is one service, works natively with your gRPC server, and is understood by the entire gRPC ecosystem.







  
Section 10

  
The things nobody tells you


  
Proto3 has no required fields — this is intentional and you need to validate manually

  
Proto2 had required fields. Proto3 removed them. Every field in proto3 is optional. If a client sends a message with no amount_paise, the server receives 0 (the default for int64). You won't get an error — you'll process a charge of ₹0. This means proto3 validation is entirely your responsibility. Use a validation library (protoc-gen-validate or manual checks) to enforce business rules that proto3 won't.


  
Default values are indistinguishable from missing values in proto3

  
In proto3, if a field has its default value (0 for numbers, "" for strings, false for bool), it's not encoded on the wire. You cannot tell if a field was explicitly set to 0 or simply not included. If your business logic cares about "user explicitly set amount to 0" vs "user didn't set amount" — you need a wrapper type (google.protobuf.Int64Value) or a boolean "has_amount" field. Proto2 had hasField() to check presence; proto3 does not for scalar fields (though proto3 optional in newer versions adds this back).


  
Multiplexing means one slow RPC starves others — sometimes

  
HTTP/2 multiplexing allows many gRPC calls to share one connection. But if one call is downloading a large server-streaming response at full speed, it can consume the connection's bandwidth and delay other calls sharing the same connection. This is TCP-level head-of-line blocking reappearing. The mitigation: separate channels for bulk streaming and latency-sensitive unary calls, or use gRPC's flow control to limit stream bandwidth.


  
Your Protobuf messages are logs, not just wire formats

  
In many systems, gRPC request and response messages get logged for debugging and audit. Sensitive fields (payment card numbers, PII, secrets) encoded in proto messages end up in your log infrastructure. Add field-level annotations to mark sensitive fields, and configure your logging middleware to redact them. The same care you apply to JSON logging applies to Protobuf logging.







  
Decision framework

  
Adopting gRPC — decide in this order

  
These questions determine whether gRPC is the right tool and how to deploy it safely. Answer them before writing the first proto file.

  


    

      

      
1

      

        
Who are your clients? External/public → REST. Internal only → gRPC is a fit.

        
If browsers or third-party developers will call this API, REST is almost always the right choice. The tooling gap (no curl, no browser DevTools) is a real burden for external clients. If the callers are your own backend services where you control the generated client, gRPC's contract enforcement and deadline propagation pay off immediately. The most common pattern: REST at the edge, gRPC internally.
          
external clients = REST gateway + gRPC backend
        

      

    


    

      

      
2

      

        
Do you need streaming or high-frequency small messages? These are gRPC's strongest cases.

        
Real-time feeds, bidirectional communication, large dataset export, telemetry ingestion — these are where gRPC's streaming patterns genuinely replace solutions that would otherwise require WebSockets or polling hacks. If your communication is simple request-response CRUD with no streaming requirements and moderate volume, REST is simpler and the gRPC overhead isn't justified.
          
streaming need = gRPC's biggest advantage
        

      

    


    

      

      
3

      

        
Set up buf.build before writing any proto files. Lint and breaking change detection from day one.

        
The most expensive gRPC decision to fix retroactively is a broken schema. buf lint in CI catches naming convention violations. buf breaking catches field number changes, type changes, and removed fields before they reach production. Add both to your CI pipeline before the first service goes live. Adding them after you have 10 services with schema debt is painful.
          
no buf breaking in CI = silent schema corruption risk
        

      

    


    

      

      
4

      

        
Always set deadlines. Every call. No exceptions.

        
A gRPC call without a deadline is a resource leak waiting to happen. If the downstream service hangs, your call hangs indefinitely, holding a goroutine/thread/connection. At scale, a few hung calls can exhaust your connection pool. The gRPC framework makes deadline propagation trivial — use it. The rule: every gRPC call gets a deadline, every server checks the deadline before starting expensive work.
          
no deadline = unbounded resource hold
        

      

    


    

      

      
5

      

        
Implement grpc.health.v1.Health for every service. Use it for k8s probes.

        
The standard gRPC health protocol is one import and a few lines. It gives you readiness and liveness probes that work natively with Kubernetes, Envoy, AWS ALB, and every gRPC-aware tool. Skipping it and using a separate HTTP health endpoint means running a second server or adding HTTP bridging code. The standard protocol exists — use it.
          
standard protocol = zero config with k8s + Envoy
        

      

    


    

      
6

      

        
Use an L7-aware load balancer. Don't put gRPC behind a plain TCP load balancer.

        
A TCP load balancer distributes connections — with gRPC's long-lived HTTP/2 connections, this means one client sticks to one backend indefinitely. Use an HTTP/2-aware L7 balancer (Envoy, nginx 1.13.10+, AWS ALB, GCP Cloud LB) that distributes individual gRPC calls, not just connections. Or use client-side load balancing with DNS-based discovery. Discovering this in production after launch means a maintenance window to restructure your load balancing.
          
TCP LB + gRPC = all traffic to one backend
