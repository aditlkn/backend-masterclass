Deep Dive 01

Flyway zero-downtime:
migration file patterns

Flyway's job is tracking which migrations have run. Your job is making those migrations safe. These patterns encode zero-downtime discipline directly into your migration files so every developer writing a migration gets the safety defaults automatically.

Flyway migration patterns — the right way to write every migrationSQL
-- ── Migration file naming ─────────────────────────────────────────
-- V{version}__{description}.sql
-- V1__create_users_table.sql
-- V2__add_email_index.sql
-- V3__expand_add_display_name.sql    ← expand phase
-- V4__contract_drop_user_name.sql    ← contract phase (separate deploy)

-- ── Every migration starts with this header ──────────────────────
-- V5__add_metadata_column.sql

-- SAFETY: fail fast if can't acquire lock (don't block traffic)
SET lock_timeout = '3s';
-- SAFETY: don't limit the migration itself (it may run long for large tables)
SET statement_timeout = '0';
-- SAFETY: see exactly what the migration is doing if it goes long
SET client_min_messages = 'DEBUG';

-- ── ADD COLUMN (instant on PG11+) ────────────────────────────────
ALTER TABLE orders ADD COLUMN IF NOT EXISTS metadata JSONB;
-- IF NOT EXISTS: migration is idempotent (safe to retry if something fails mid-way)

-- ── CREATE INDEX (always CONCURRENTLY) ───────────────────────────
-- Flyway runs inside a transaction by default.
-- CONCURRENTLY cannot run inside a transaction!
-- Solution: mark the migration as non-transactional

-- flyway.conf (or per-migration annotation):
-- Add a special marker for non-transactional migration files:
-- V6__add_metadata_index.sql — starts with a no-tx marker

-- The no-transaction migration file:
-- (Flyway detects this when it sees the file doesn't start with BEGIN)
-- Must be the ONLY statement in the file for CONCURRENTLY migrations

CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_orders_metadata_type
ON orders((metadata->>

'type'))
WHERE metadata IS NOT NULL;

-- ── Expanding a column rename (V7) ───────────────────────────────
-- V7__expand_add_display_name.sql
SET lock_timeout = '3s';
ALTER TABLE users ADD COLUMN IF NOT EXISTS display_name TEXT;
-- Note: NO backfill here. Schema change only.
-- Backfill runs separately as a background job (not in migration files).

-- ── Contracting (V8) — run ONLY after backfill is complete ───────
-- V8__contract_drop_user_name.sql
SET lock_timeout = '3s';
ALTER TABLE users DROP COLUMN IF EXISTS user_name;

-- ── Flyway repair: fix a failed migration ────────────────────────
-- If a migration fails halfway, Flyway marks it as FAILED in flyway_schema_history.
-- Fix the migration file, then:
-- flyway repair  ← resets the failed migration's checksum so it can run again
-- flyway migrate ← reruns the fixed migration

-- ── Flyway baseline: start tracking an existing database ─────────
-- If you add Flyway to an existing production database:
-- flyway baseline --baselineVersion=5  ← marks existing schema as version 5
-- Future migrations run from V6 onwards. V1-V5 are skipped (already applied).

-- ── flyway.conf ──────────────────────────────────────────────────
-- url=jdbc:postgresql://primary.db.internal:5432/myapp
-- user=${FLYWAY_USER}
-- password=${FLYWAY_PASSWORD}
-- locations=filesystem:./migrations
-- validateOnMigrate=true     ← catches edited migration files
-- outOfOrderAllowed=false    ← prevents running V6 before V5
-- lockRetryCount=10          ← retry acquiring Flyway lock 10 times
-- connectRetries=5           ← retry DB connection on startup
Verify Flyway migration health
✓flyway info shows all migrations with Pending/Success/Failed status — no Failed rows.
✓Every migration file starts with SET lock_timeout = '3s'; — enforce via a CI lint check.
✓Index migrations are in separate files marked for non-transactional execution — CONCURRENTLY cannot run inside a transaction.
✓No migration file contains UPDATE ... SET on a large table without a WHERE clause limiting rows — data migrations run as background jobs, not in migration files.
Deep Dive 02

gh-ost production runbook:
pre-flight to cleanup

A gh-ost migration is not just a command you run. It's an operation that runs for hours against a live database. This runbook walks every step: pre-flight checks, execution with monitoring, the final swap, and what to do if things go wrong.

gh-ost runbook — complete production migration procedureBASH
## ── STEP 1: Pre-flight checks ─────────────────────────────────────

# 1a. Check table size
mysql -e "SELECT table_rows, data_length/1024/1024 AS data_mb,
  index_length/1024/1024 AS index_mb
  FROM information_schema.tables
  WHERE table_schema='myapp' AND table_name='orders';"

# 1b. Check current replica lag (should be near 0 before starting)
mysql -h replica-1 -e "SHOW SLAVE STATUS\G" | grep Seconds_Behind_Master

# 1c. Check for long-running transactions
mysql -e "SELECT trx_id, trx_started, trx_mysql_thread_id, trx_query
  FROM information_schema.innodb_trx
  WHERE trx_started < NOW() - INTERVAL 60 SECOND;"

# 1d. Confirm binary logging is enabled
mysql -e "SHOW VARIABLES LIKE 'log_bin'; SHOW VARIABLES LIKE 'binlog_format';"
# Must be: log_bin=ON, binlog_format=ROW

# 1e. Verify gh-ost user has required permissions
mysql -e "SHOW GRANTS FOR 'gh_ost_user'@'%';"

## ── STEP 2: Dry run ───────────────────────────────────────────────
gh-ost \
  --host=primary.db.internal --port=3306 \
  --user=gh_ost_user --password="${DB_PASSWORD}" \
  --database=myapp --table=orders \
  --alter="ADD COLUMN metadata JSON DEFAULT NULL, ADD INDEX idx_orders_metadata_type ((metadata->>'$.type'))" \
  --dry-run 2>&1 | tee /tmp/ghost-dryrun.log

# Verify dry-run succeeded before proceeding
grep -i "error\|panic" /tmp/ghost-dryrun.log && echo "DRY RUN FAILED" || echo "DRY RUN OK"

## ── STEP 3: Execute ───────────────────────────────────────────────
# Start in a tmux/screen session — migration runs for hours
tmux new-session -s ghost-migration

gh-ost \
  --host=primary.db.internal --port=3306 \
  --user=gh_ost_user --password="${DB_PASSWORD}" \
  --database=myapp --table=orders \
  --alter="ADD COLUMN metadata JSON DEFAULT NULL" \
  --assume-rbr \                      # assume ROW-based binary logging
  --chunk-size=1000 \                 # rows per chunk (start conservative)
  --max-load="Threads_running=25" \   # pause if DB is loaded
  --critical-load="Threads_running=50" \  # abort if critically loaded
  --max-lag-millis=1500 \             # pause if replica lag > 1.5s
  --throttle-control-replicas="replica-1.db.internal:3306" \
  --ok-to-drop-table \                # allow dropping old table post-swap
  --initially-drop-ghost-table \      # clean up from any previous failed run
  --execute \
  2>&1 | tee /tmp/ghost-migration-$(date +%Y%m%d-%H%M).log

## ── STEP 4: Monitor (in a second terminal) ───────────────────────
# Progress via socket
watch -n 5 'echo "status" | nc -U /tmp/gh-ost.orders.sock'

# Replica lag
watch -n 10 'mysql -h replica-1 -e "SHOW SLAVE STATUS\G" | grep Seconds_Behind_Master'

# DB load
watch -n 5 'mysql -e "SHOW GLOBAL STATUS LIKE \"Threads_running\";"'

# If load spikes: throttle gh-ost without stopping it
echo "throttle" | nc -U /tmp/gh-ost.orders.sock
echo "unthrottle" | nc -U /tmp/gh-ost.orders.sock

## ── STEP 5: Verify post-swap ──────────────────────────────────────
mysql -e "DESCRIBE myapp.orders;"            # new column exists
mysql -e "SHOW INDEX FROM myapp.orders;"     # new index exists
mysql -e "SELECT COUNT(*) FROM myapp.orders;"  # row count matches

# Old table preserved as _orders_del until you drop it
mysql -e "SHOW TABLES LIKE '%orders%';"      # should show _orders_del
mysql -e "SELECT COUNT(*) FROM myapp._orders_del;"  # matches original count
mysql -e "DROP TABLE myapp._orders_del;"    # ONLY after verifying everything is correct

## ── STEP 6: Rollback (if needed) ──────────────────────────────────
# If swap completed but something is wrong:
mysql -e "RENAME TABLE myapp.orders TO myapp._orders_new, myapp._orders_del TO myapp.orders;"
# This swaps back — old table becomes orders, new table gets the _new suffix
# Drop the new table when safe: DROP TABLE myapp._orders_new;
Deep Dive 03

Alembic expand-contract:
column rename across three migrations

Alembic's autogenerate feature detects schema differences and generates migration scripts. But for expand-contract, you write the migrations manually — autogenerate would generate a single RENAME COLUMN which is exactly what you're trying to avoid. Here's the three-migration sequence for a safe column rename.

Alembic expand-contract — column rename across 3 migrationsPYTHON
# ── Migration 1: Expand — add new column ─────────────────────────
# alembic/versions/a1b2c3_expand_add_display_name.py

def upgrade():
    # Instant on PG11+: column with nullable, no default
    op.execute("SET lock_timeout = '3s'")
    op.add_column('users', sa.Column('display_name', sa.Text(), nullable=True))
    # DO NOT backfill here. This migration runs at deploy time.
    # Backfill runs as a separate background job.

def downgrade():
    op.execute("SET lock_timeout = '3s'")
    op.drop_column('users', 'display_name')

# ── Application code change (deployed with migration 1) ───────────
# Write to BOTH columns
def create_user(user_name: str, email: str) -> User:
    user = User(
        user_name=user_name,
        display_name=user_name,   # ← write to both
        email=email,
    )
    db.add(user)
    db.commit()
    return user

# ── Backfill script (runs as separate background job) ─────────────
# scripts/backfill_display_name.py
def backfill_display_name(batch_size=1000, sleep_ms=50):
    while True:
        result = db.execute(text("""
            UPDATE users
            SET display_name = user_name
            WHERE display_name IS NULL
              AND id IN (
                SELECT id FROM users
                WHERE display_name IS NULL
                LIMIT :batch_size
              )
        """), {'batch_size': batch_size})
        if result.rowcount == 0:
            break  # done
        db.commit()
        lag = check_replica_lag()
        if lag > 1000:  # ms
            time.sleep(5)  # let replicas catch up
        else:
            time.sleep(sleep_ms / 1000)

# Verify backfill before proceeding to migration 2:
# SELECT COUNT(*) FROM users WHERE display_name IS NULL;
# Must be 0 before proceeding.

# ── Migration 2: Flip reads (separate deploy, after backfill) ────
# This is an application code change only — no SQL migration needed.
# Change all SELECTs from user_name → display_name.

# models.py: add display_name to the model (if not already done)
class User(Base):
    __tablename__ = 'users'
    id           = Column(Integer, primary_key=True)
    user_name    = Column(Text)     # keep for now — still write to it
    display_name = Column(Text)     # now the primary read column
    email        = Column(Text)

# ── Migration 3: Contract — drop old column ──────────────────────
# alembic/versions/d4e5f6_contract_drop_user_name.py
# Run ONLY after:
# 1. All pods are on the code that reads from display_name
# 2. Application no longer writes to user_name

def upgrade():
    op.execute("SET lock_timeout = '3s'")
    op.drop_column('users', 'user_name')

def downgrade():
    # Cannot restore data — this downgrade just adds the column back (empty)
    op.execute("SET lock_timeout = '3s'")
    op.add_column('users', sa.Column('user_name', sa.Text(), nullable=True))
    # Note: downgrade leaves user_name NULL for all rows.
    # If you truly need to roll back here, restore from backup.

# ── Alembic pitfall: autogenerate generates RENAME not expand-contract ──
# NEVER run alembic revision --autogenerate for column renames.
# Autogenerate detects: model has display_name, DB has user_name → generates RENAME.
# That breaks all old code immediately.
# For expand-contract: always write migrations manually, never autogenerate.
Verify expand-contract is complete before running migration 3
✓Backfill is complete: SELECT COUNT(*) FROM users WHERE display_name IS NULL returns 0.
✓No code references old column: grep -r "user_name" ./app --include="*.py" returns no results (or only migration files).
✓All pods running new code: check deploy status, all instances are on the version that reads display_name.
✓No writes to old column in logs: search application logs for any INSERT/UPDATE referencing user_name — should be empty.
