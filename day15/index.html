Section 01

Why production migrations go wrong

Schema migrations are one of the most dangerous operations in backend engineering. Not because the SQL is complex — it usually isn't. Because of what happens at the storage engine level when you run that SQL against a table with 200 million rows and 1,000 concurrent transactions.

The naive migration timeline

You run ALTER TABLE orders ADD COLUMN metadata JSONB. PostgreSQL acquires an AccessExclusiveLock on the table. This lock blocks every other operation — reads, writes, any other DDL. It waits for all existing transactions to finish. While it waits, new transactions queue up behind the lock. Your application's connection pool fills with waiting queries. Response times spike. Health checks fail. Load balancer removes the instances. Your site is down.

How long does it stay down? Until the ALTER TABLE finishes. For a 200M row table with a column default value, PostgreSQL must rewrite every row. At 50MB/s disk throughput, that's tens of minutes to hours.

The three ways migrations kill production

  • Table lock contention: The migration acquires an exclusive lock and blocks all other traffic while it waits for long-running transactions to finish.
  • Replication lag: A large DDL operation or bulk data change generates enormous WAL. Replicas fall behind. Read traffic routed to replicas gets stale. If the primary fails during the migration, promotion takes forever while replicas catch up.
  • Long-running transactions blocking the lock: Your migration is waiting to acquire the lock. Meanwhile, new short queries queue behind it. Nothing else can run. This is the queue-of-death pattern — a slow migration waiting for a long transaction creates a thundering herd of blocked queries.
🔴
lock_timeout is not optional in production

Always run migrations with SET lock_timeout = '3s'. Without it, your migration waits indefinitely for the lock, blocking all traffic behind it for as long as it takes. With lock_timeout=3s, if the lock isn't acquired in 3 seconds, the migration fails with an error — traffic continues normally, and you retry at a better time. A failed migration is infinitely better than a live outage.

PostgreSQL lock modes: what blocks what

PostgreSQL has eight lock modes, from the weakest (AccessShare — a regular SELECT) to the strongest (AccessExclusive — DDL operations). Understanding which operations require which locks tells you exactly what will be blocked during a migration.

The lock wait queue problem

When your ALTER TABLE is waiting for an AccessExclusiveLock, it sits in the wait queue. Every subsequent statement — including fast SELECTs — queues behind it. The ALTER is waiting for existing long transactions; new short queries are waiting for the ALTER. This is the queue-of-death: one slow migration can back up hundreds of queries.

Detecting this in production: SELECT pid, wait_event_type, wait_event, query FROM pg_stat_activity WHERE state = 'active' ORDER BY wait_event_type NULLS LAST. If you see many rows with wait_event_type = 'Lock', something is holding a lock that others are waiting for.

Section 03

What's actually safe on a live table

Not all schema changes are equal. Some operations are instant and safe at any scale. Others require careful orchestration. Knowing the difference lets you skip unnecessary ceremony for safe operations and apply the full expand-contract pattern only when actually needed.

Instant in PostgreSQL (no table rewrite)

  • ADD COLUMN col TYPE DEFAULT constant — since PostgreSQL 11, adding a column with a constant default is instant. PostgreSQL stores the default in metadata, not in each row. Reading a row without the column returns the default. This is the single most important change in PostgreSQL 11.
  • ADD COLUMN col TYPE (nullable, no default) — also instant. Rows without the column return NULL.
  • DROP COLUMN col — instant. PostgreSQL marks the column as dropped but doesn't rewrite rows immediately. Space is reclaimed by VACUUM later.
  • CREATE INDEX CONCURRENTLY — doesn't block reads or writes. Takes longer than a regular index build but doesn't lock the table.
  • ALTER TABLE ... ADD CONSTRAINT ... NOT VALID + VALIDATE CONSTRAINT — two-step constraint addition that avoids a full table lock on the second step.

Always requires a table rewrite (expensive)

  • ALTER COLUMN TYPE to an incompatible type — rewrites every row to convert the value
  • ADD COLUMN col NOT NULL without a default in PostgreSQL <11 — must write the default into every existing row
  • SET NOT NULL on a column that has NULLs — must verify every row (full scan)
✅
PostgreSQL 11+ changed the game for column addition

Before PG 11: ALTER TABLE orders ADD COLUMN metadata JSONB DEFAULT '{}'::jsonb rewrote every row — potentially hours for a large table. After PG 11: the same command is instant. The default is stored in the system catalog; individual rows are rewritten lazily (by VACUUM or the next UPDATE). If you're on PG 11+, adding a nullable column or a column with a simple constant default is now safe to run without gh-ost or expand-contract. Verify your PostgreSQL version before designing an elaborate migration ceremony for something that's now instant.

Section 04

Expand-contract: the pattern for incompatible changes

Expand-contract (also called parallel change or blue-green schema migration) is the pattern for schema changes that can't be done atomically without downtime. The core insight: decompose an incompatible change into a sequence of backward-compatible steps, each deployed separately, so the database and application are never in an inconsistent state.

When you need expand-contract

Any change where the new schema is incompatible with the old application code, or where the old schema is incompatible with the new application code. Renaming a column, changing a column type, splitting one column into two, moving data between tables — all require expand-contract.

The three phases

Phase 1 — Expand: Add new columns/tables alongside the old ones. Write code that writes to both old and new. Deploy this version. Now both old and new schema exist simultaneously.

Phase 2 — Migrate: Backfill the new columns with data from the old columns. Verify correctness. Once backfill is complete and verified, flip reads to the new columns while keeping writes to both.

Phase 3 — Contract: Once all application code reads from the new schema only, remove writes to the old columns, then drop the old columns. Only do this after the new schema is fully deployed and validated.

The key discipline: each phase must be deployed and stabilised before moving to the next. Trying to combine phases is how expand-contract fails.

Section 05

gh-ost: online schema change for MySQL

gh-ost (GitHub's Online Schema Changer) solves the table-rewrite problem for MySQL. Instead of running ALTER TABLE directly, gh-ost creates a ghost table with the new schema, streams all existing rows from the original table to the ghost table using MySQL's binary log, applies all concurrent writes to the ghost table, and then swaps the tables atomically. Production traffic continues throughout — reads and writes hit the original table until the final atomic swap.

How gh-ost works

  1. Creates a _orders_gho ghost table with the new schema
  2. Applies the schema change to the ghost table (instant, it's empty)
  3. Connects to the MySQL binary log as a replica
  4. Copies existing rows in batches from the original to the ghost table
  5. Applies all ongoing changes (binlog events) to the ghost table as they happen
  6. When the row copy is complete and binlog position is caught up, performs an atomic table swap
  7. Drops the original table (now renamed to _orders_del)

gh-ost vs pt-online-schema-change

pt-online-schema-change (Percona Toolkit) uses triggers to capture changes during migration. Triggers have overhead on every write to the original table during migration. gh-ost uses the binary log instead, which has much lower overhead and doesn't require any locks on the original table. For production use, gh-ost is generally preferred.

For PostgreSQL, the equivalent is pglogical or a manual expand-contract. PostgreSQL's CREATE INDEX CONCURRENTLY handles the most common case (adding indexes), and PG 11+'s instant column addition handles much of the rest.

gh-ost — running an online schema changeBASH
# gh-ost: online ALTER TABLE for MySQL — no table lock on production

# ── Dry run first (always) ──────────────────────────────────────
gh-ost \
  --host=primary.db.internal \
  --port=3306 \
  --user=gh_ost_user \
  --password="${DB_PASSWORD}" \
  --database=myapp \
  --table=orders \
  --alter="ADD COLUMN metadata JSON DEFAULT NULL, ADD INDEX idx_status_created (status, created_at)" \
  --dry-run              # validate without actually doing anything

# ── Production run ──────────────────────────────────────────────
gh-ost \
  --host=primary.db.internal \
  --port=3306 \
  --user=gh_ost_user \
  --password="${DB_PASSWORD}" \
  --database=myapp \
  --table=orders \
  --alter="ADD COLUMN metadata JSON DEFAULT NULL" \
  --chunk-size=1000 \          # rows per batch copy (tune for your load)
  --max-load="Threads_running=25" \  # pause if DB is under load
  --critical-load="Threads_running=50" \  # abort if DB is critically loaded
  --max-lag-millis=1500 \      # pause if replica lag > 1.5s
  --throttle-control-replicas="replica-1,replica-2" \
  --ok-to-drop-table \         # drop old table after swap (else renamed to _del)
  --execute                    # actually run (without this: dry-run)

# ── Throttle/unthrottle interactively ────────────────────────────
# gh-ost creates a socket file for live control during migration
echo throttle | nc -U /tmp/gh-ost.orders.sock   # pause migration
echo unthrottle | nc -U /tmp/gh-ost.orders.sock  # resume migration
echo status | nc -U /tmp/gh-ost.orders.sock       # progress report

# ── gh-ost progress output ───────────────────────────────────────
# Copy: 23.4M/200M rows (11.7%), time: 2h, ETA: 15h
# Lag: 0ms, Throttled: no, chunk-size: 1000
# Binlog events applied: 1.2M (all concurrent writes captured)

# ── Required MySQL permissions for gh-ost user ──────────────────
GRANT SELECT, INSERT, UPDATE, DELETE, CREATE, DROP, ALTER ON myapp.* TO 'gh_ost_user'@'%';
GRANT REPLICATION SLAVE, REPLICATION CLIENT ON *.* TO 'gh_ost_user'@'%';

Section 06

Backfilling data: the right way

After adding a new column, you need to populate it with data derived from existing columns. This is the backfill. Done wrong, a backfill is as dangerous as the migration itself — a single UPDATE orders SET display_name = user_name holds a lock and generates enormous replication lag on 50M rows.

The batched backfill pattern

Never update all rows in a single transaction. Update in small batches with a sleep between batches to give replication a chance to catch up and the database breathing room. The pattern: UPDATE ... WHERE id BETWEEN batch_start AND batch_end AND new_col IS NULL LIMIT 1000.

Controlling replication lag

Monitor replica lag during the backfill. If lag exceeds your threshold (typically 1-5 seconds), pause the backfill until replicas catch up. In production: build this into the backfill script. Tools like pt-online-schema-change and gh-ost do this automatically; for custom backfills, you must do it manually.

Batched backfill — production-safe patternPYTHON
import psycopg2, time, logging

def backfill_display_name(conn, batch_size=1000, sleep_ms=50, max_lag_ms=1000):
    """Backfill display_name from user_name in safe batches."""
    cursor = conn.cursor()
    total_updated = 0

    # Get min/max ID to iterate over
    cursor.execute("SELECT MIN(id), MAX(id) FROM users WHERE display_name IS NULL")
    min_id, max_id = cursor.fetchone()
    if not min_id:
        logging.info("Nothing to backfill")
        return

    batch_start = min_id
    while batch_start <= max_id:
        batch_end = batch_start + batch_size * 10  # ID range per batch

        # Check replication lag before each batch
        lag = get_replica_lag_ms(conn)
        if lag > max_lag_ms:
            logging.warning(f"Replica lag {lag}ms > {max_lag_ms}ms, pausing...")
            time.sleep(5)
            continue

        cursor.execute("""
            UPDATE users
            SET display_name = user_name
            WHERE id >= %s AND id < %s
              AND display_name IS NULL
        """, (batch_start, batch_end))
        conn.commit()

        rows = cursor.rowcount
        total_updated += rows
        logging.info(f"Updated {rows} rows (IDs {batch_start}–{batch_end}), total: {total_updated}")

        batch_start = batch_end
        time.sleep(sleep_ms / 1000)  # breathe between batches

    logging.info(f"Backfill complete: {total_updated} rows updated")

def get_replica_lag_ms(conn):
    cursor = conn.cursor()
    cursor.execute("""
        SELECT EXTRACT(EPOCH FROM (now() - pg_last_xact_replay_timestamp())) * 1000
        FROM pg_stat_replication
        ORDER BY replay_lag DESC LIMIT 1
    """)
    row = cursor.fetchone()
    return row[0] if row else 0

Section 07

Index migrations: the safest path

Adding indexes is the most common migration task and the one that trips people up most often. The naive CREATE INDEX takes a Share lock — blocking all writes for the duration of the index build. For a large table, that's many minutes. The answer is almost always CREATE INDEX CONCURRENTLY.

CREATE INDEX CONCURRENTLY

CONCURRENTLY builds the index without holding a lock that blocks writes. Instead it does two table scans, using a weakened lock that only conflicts with other DDL. The index build takes longer (roughly 2× as long as a normal build) and uses more CPU, but production traffic continues normally throughout.

Caveats: if the CONCURRENTLY build fails (e.g. due to a lock timeout or constraint violation), it leaves an invalid index behind. You must drop the invalid index before retrying. Check: SELECT indexname, indisvalid FROM pg_indexes JOIN pg_index ON pg_index.indexrelid = (SELECT oid FROM pg_class WHERE relname = indexname) WHERE tablename = 'orders' AND NOT indisvalid.

Dropping indexes safely

DROP INDEX CONCURRENTLY also exists and avoids taking an exclusive lock. Use it when dropping large indexes in production.

The unique index pattern

Adding a UNIQUE constraint requires verifying uniqueness across all existing rows — potentially slow on large tables. The safe pattern: first CREATE UNIQUE INDEX CONCURRENTLY (builds the index, checks uniqueness without blocking writes), then ALTER TABLE orders ADD CONSTRAINT ... USING INDEX (promotes the index to a constraint — instant, the work is already done).

Index migration patterns — safe operationsSQL
-- ❌ Blocks ALL writes for duration of index build
CREATE INDEX idx_orders_user_status ON orders(user_id, status);

-- ✅ No lock on writes — production-safe
CREATE INDEX CONCURRENTLY idx_orders_user_status ON orders(user_id, status);

-- ✅ Drop without blocking writes
DROP INDEX CONCURRENTLY idx_orders_old;

-- ✅ Adding a UNIQUE constraint safely (2-step)
-- Step 1: build the index concurrently (slow, but no write lock)
CREATE UNIQUE INDEX CONCURRENTLY idx_users_email_uniq ON users(email);

-- Step 2: promote to constraint (instant — index already built)
ALTER TABLE users ADD CONSTRAINT users_email_unique USING INDEX idx_users_email_uniq;

-- ✅ Check for invalid indexes after a failed CONCURRENTLY build
SELECT i.relname AS index_name, t.relname AS table_name, ix.indisvalid
FROM pg_class t
JOIN pg_index ix ON t.oid = ix.indrelid
JOIN pg_class i ON i.oid = ix.indexrelid
WHERE NOT ix.indisvalid
  AND t.relkind = 'r';
-- Any row here: DROP INDEX CONCURRENTLY that_index_name and retry

-- ✅ Partial index (only indexes rows that match a condition)
-- Much smaller than a full index, often just as effective
CREATE INDEX CONCURRENTLY idx_orders_pending
ON orders(created_at)
WHERE status = 'pending';
-- Only indexes the ~5% of rows that are pending — tiny + fast

-- ✅ SET lock_timeout before any DDL in production
SET lock_timeout = '3s';
ALTER TABLE orders ADD COLUMN metadata JSONB;
-- If can't acquire lock in 3s → fails with error, traffic unaffected

Section 08

Migration tools: Flyway, Liquibase, and custom scripts

Schema migration tools solve the problem of tracking which migrations have run, in what order, and ensuring they run exactly once across all environments. Without them, you're managing a pile of SQL scripts and hoping nothing runs twice.

Flyway

Flyway uses version-numbered SQL files (V1__create_users.sql, V2__add_email_index.sql). It tracks which versions have run in a flyway_schema_history table. Simple, opinionated, SQL-first. Good for teams that want to stay close to SQL. Doesn't support reversible migrations well.

Liquibase

Liquibase uses changelog files (XML, YAML, JSON, or SQL). More powerful than Flyway — supports rollbacks, conditional migrations, and database-agnostic change sets. More complex to set up and reason about. Good for enterprise environments that need audit trails and rollback capabilities.

Alembic (SQLAlchemy)

Python-native migration tool. Generates migration scripts from model diffs (alembic revision --autogenerate). Supports both upgrade and downgrade paths. Standard in Python/SQLAlchemy stacks.

The zero-downtime rule

Whatever tool you use, the migrations themselves must follow zero-downtime principles. A migration tool that runs ALTER TABLE without CONCURRENTLY or lock_timeout is still going to take your site down — the tool is just orchestration, not safety.

💡
Separate schema migrations from data migrations

Schema migrations (adding/removing columns, indexes, constraints) should run at deploy time via your migration tool. Data migrations (backfilling values, transforming data) should run as background jobs, separately, with batching and lag monitoring. Mixing them means your deployment step can take hours for a large table backfill — failing your health checks and rolling back the deploy. Schema changes: fast, in migration files. Data changes: async background workers.

Section 09

Production reality

Test migrations on a production-size database

A migration that takes 0.3 seconds on a 10,000 row dev database takes 5 hours on a 200 million row production database. Every migration should be tested against a recent production database dump (anonymised) before being run in production. If you don't have this, you're flying blind on the most dangerous operation you do.

The long transaction problem

Your migration is waiting to acquire an AccessExclusiveLock. Some long-running transaction is holding conflicting locks. Your migration waits. Your migration is also in the lock queue, blocking all subsequent traffic. How long is the oldest transaction? Check: SELECT pid, now() - pg_stat_activity.xact_start AS duration, query FROM pg_stat_activity WHERE state != 'idle' ORDER BY duration DESC NULLS LAST. If you see transactions running for minutes, your migration will wait for all of them to finish. Consider terminating the long-running ones before running the migration: SELECT pg_terminate_backend(pid).

Replication lag is your canary

Monitor replication lag during any migration. A spike in lag means the primary is generating more WAL than replicas can apply — your migration is generating too much write traffic. Pause, let replicas catch up, reduce batch size, add more sleep between batches. If a replica can't keep up with your migration, it also can't keep up with a failover during your migration.

Keep a rollback plan

For expand-contract, rollback is straightforward — the old schema still exists until the contract phase. For gh-ost, the original table is preserved as _orders_del until you explicitly drop it. For PostgreSQL migrations, have a down migration ready and tested. For irreversible operations (data transformations where the original data is overwritten), take a snapshot before the migration.

Section 10

The things nobody tells you

Adding a NOT NULL column without a default is a table rewrite in PostgreSQL <11

Everyone knows about this on MySQL. Fewer know it on PostgreSQL. If you're on PostgreSQL 10 or earlier, ALTER TABLE orders ADD COLUMN status TEXT NOT NULL DEFAULT 'pending' rewrites every row to set the default value. The safe pattern on older PostgreSQL: add the column as nullable first, backfill the values, then add the NOT NULL constraint using ADD CONSTRAINT ... NOT NULL NOT VALID followed by VALIDATE CONSTRAINT.

VALIDATE CONSTRAINT uses ShareUpdateExclusiveLock

When you ALTER TABLE orders ADD CONSTRAINT fk_user_id FOREIGN KEY (user_id) REFERENCES users(id) NOT VALID, the constraint is added instantly but only enforced on new rows. Later, ALTER TABLE orders VALIDATE CONSTRAINT fk_user_id scans existing rows to enforce the constraint — but uses a ShareUpdateExclusiveLock instead of AccessExclusive. This means reads and writes continue normally during validation. The constraint validation is the expensive part, and it doesn't block traffic.

Schema changes during rolling deploys need two compatibility windows

If you deploy application code in a rolling fashion (pods replaced one by one), at some point you have old code and new code running simultaneously against the same database. A schema change must be backward-compatible with the old code (so old pods can still function) AND forward-compatible with the new code (so new pods can use the new schema). This often means the expand phase must be deployed separately, before any code that uses the new schema.

Flyway checksums catch unintended changes to migration files

Flyway stores a checksum of each migration file. If you edit a migration file that has already run (even to fix a comment), Flyway refuses to run. This is a feature — it prevents silent schema divergence between environments. The solution: never edit a migration that's been applied to any environment. Create a new migration instead. This discipline is non-negotiable in multi-environment setups.

Decision sequence

Planning a schema migration — work through this

Every production migration needs a plan. Start here, skip nothing.

1
Is this actually necessary?
Can you achieve the same outcome without a schema change? A new column that's always NULL until the application populates it can often be added easily. A renamed column should be asked: does renaming add real clarity, or does it create a migration ceremony for no user-visible benefit? Migrations have risk — only run them when the benefit clearly outweighs the risk.
cost-justify before planning
2
Is it an instant operation?
Check: adding a nullable column → instant in all PG versions. Adding a column with constant default → instant in PG 11+. Dropping a column → instant. Creating an index with CONCURRENTLY → no write lock. If it's instant: add SET lock_timeout = '3s' and run it. You don't need expand-contract or gh-ost for instant operations.
instant = lock_timeout + run
3
What is the table size and write rate?
Test the migration on a copy of production data. Time it. If it takes more than 1 second, it will cause noticeable latency. If it takes more than 30 seconds, plan for a maintenance window or use expand-contract. Check write rate: a table that receives 10,000 writes/second generates enormous WAL during a migration — replicas will fall behind even if the migration itself is "safe."
test on production-size data first
4
Does the new schema break existing application code?
If yes: you need expand-contract. The new schema must be added while old code can still run, and old schema must remain while new code is deployed. Map out the phases explicitly: which deploy enables the new schema? Which deploy stops writing to the old schema? Which deploy drops the old schema? Write down the order before starting.
backward incompatible = expand-contract required
5
Set lock_timeout and statement_timeout on every migration
SET lock_timeout = '3s'; SET statement_timeout = '0'; — every migration should start with lock_timeout. If the lock isn't acquired in 3 seconds, the migration fails cleanly and traffic continues. Then remove statement_timeout on the migration itself (it may run long, and that's expected). But keep lock_timeout. Always.
no lock_timeout = migration can take down production
6
Separate schema change from data migration
Schema change (adding the column) runs at deploy time via migration tool — fast, usually safe with the above precautions. Data migration (backfilling values into the new column) runs as an async background job — batched, monitored for replication lag, never in the same transaction as the schema change. Mixing them makes your deploy dangerous and slow.
schema change ≠ data change. separate them.

Section 12

Migration safety checklist

LT
Lock timeout

Every migration starts with SET lock_timeout = '3s'. Without this a stuck migration blocks all traffic indefinitely.

CIC
CONCURRENTLY

All index operations use CREATE/DROP INDEX CONCURRENTLY. No exceptions in production.

EC
Expand-contract

Any incompatible change uses expand-contract. Never rename a column in a single migration that also updates application code.

SEP
Separate schema/data

Schema changes in migration files (fast). Data backfills as async background jobs (batched, monitored). Never both in the same migration.

LAG
Watch replica lag

Monitor replication lag during any migration or backfill. If lag exceeds 1s, pause and let replicas catch up before continuing.

TST
Test on prod size

Run every migration against a production-size database snapshot before production. The timing difference between 10K and 200M rows is not linear.
