Production Incident  ·  PostgreSQL  ·  Schema Migration

ALTER TABLE held a lock for 4 hours — the long transaction nobody noticed

Monday 11:00 IST  ·  Duration: 4h 12m  ·  Severity: P0  ·  Complete site outage

The team had scheduled a "quick schema change" for Monday morning: add a metadata JSONB column to the orders table (180 million rows). The engineer had tested it on staging — it took 4 seconds. They ran it without lock_timeout. At 11:00 they ran the migration.

At 11:01, alerts started firing. Not from the migration itself — from the application. ERROR: could not obtain lock on relation "orders". Then connection pool exhaustion. Then health checks failing. Load balancer marking instances as unhealthy.

The migration was waiting. An analytics job had been running since 10:45 — a long-running SELECT that generated a massive report once per hour. It held an AccessShareLock on orders. The ALTER TABLE was waiting for it. And every subsequent query — every order creation, every order status check — was queuing behind the ALTER TABLE. Three queries waiting in a queue became three hundred. The connection pool was full. The application was down.

10:45 IST
Analytics job starts: SELECT * FROM orders WHERE ... GROUP BY ... — expected runtime ~20 minutes. Holds AccessShareLock.
11:00 IST
Engineer runs: ALTER TABLE orders ADD COLUMN metadata JSONB DEFAULT '{}'::jsonb (PostgreSQL 10, no lock_timeout). Migration tries to acquire AccessExclusiveLock — waits for analytics job.
11:00 IST
New queries (SELECT, INSERT, UPDATE) arrive — they queue behind the ALTER TABLE. Not behind the analytics job. Behind the waiting migration. Queue of death begins.
11:02 IST
Connection pool (50 connections) fills with waiting queries. Application returns 503. P0 declared.
11:05 IST
Team finds migration in pg_stat_activity: state='idle in transaction', wait_event='relation'. Kills the analytics job (pg_terminate_backend). Migration acquires lock.
11:05 IST
Migration starts rewriting 180M rows (PostgreSQL 10 — before PG11's instant column add). Lock held for the entire rewrite.
15:12 IST
Rewrite completes (4h 7m). Lock released. Backlog of queued queries drains. Site recovers. 4h 12m total outage.
Root causes (two failures, not one)

1: No lock_timeout — the migration waited indefinitely, blocking all subsequent traffic. With SET lock_timeout = '3s', the migration would have failed in 3 seconds with no impact, and retried at a better time. 2: PostgreSQL 10 rewriting all rows — on PG 11+, adding a column with a constant default is instant (metadata only, no row rewrite). The team was running PG 10. After the incident they upgraded to PG 11, making the same migration instant the next time.

The fix — always use lock_timeout + check for long transactions firstSQL
❌ How it was run (incident)
ALTER TABLE orders ADD COLUMN metadata JSONB DEFAULT '{}'::jsonb;
-- No lock_timeout → waits indefinitely → blocks all traffic

✅ Fix 1: lock_timeout — fails fast instead of blocking
SET lock_timeout = '3s';
ALTER TABLE orders ADD COLUMN metadata JSONB DEFAULT '{}'::jsonb;
-- If can't lock in 3s → ERROR (traffic unaffected) → retry later

✅ Fix 2: Check for long transactions BEFORE running migration
SELECT pid, now() - xact_start AS duration, left(query, 80) AS query
FROM pg_stat_activity
WHERE state != 'idle' AND xact_start IS NOT NULL
ORDER BY duration DESC;
-- Any transaction > 1 minute? Investigate before running migration.
-- The analytics job would have appeared here. Kill it or wait for it to finish.

✅ Fix 3: Upgrade to PostgreSQL 11+ (makes column add instant)
-- PG 11+: constant default stored in catalog, no row rewrite
-- This 4h migration becomes ~50ms
SELECT version();  -- verify you're on PG 11+

Pre-migration runbook (run BEFORE every migration):
-- 1. Check long transactions
SELECT pid, now() - xact_start AS duration FROM pg_stat_activity
WHERE state != 'idle' AND xact_start < now() - interval '1 minute';

-- 2. Set timeouts
SET lock_timeout = '3s';
SET statement_timeout = '0';  -- migration itself may run long

-- 3. Run migration
-- 4. Verify
SELECT column_name, data_type FROM information_schema.columns
WHERE table_name = 'orders' ORDER BY ordinal_position;
What this incident teaches
▶lock_timeout is mandatory on every migration. Without it, a migration waiting for a lock blocks all subsequent traffic indefinitely. Three seconds is the right default — it gives the lock a fair chance without causing visible degradation.
▶Check for long transactions before running any migration. Anything holding a lock longer than a minute will block your migration, which will block everything else. Check pg_stat_activity, terminate the long transaction if safe to do so, or reschedule the migration.
▶Staging timing is misleading. The migration took 4 seconds in staging (small table). It took 4 hours in production (180M rows). Always time migrations against a production-size data snapshot, not a dev database.
▶Upgrading PostgreSQL changed the economics. The same migration that caused a 4-hour outage on PG10 became a 50ms operation on PG11. Schema changes are a strong argument for staying current on major PostgreSQL versions.
