Production Incident  ·  PgBouncer  ·  Transaction Mode

"ERROR: prepared statement does not exist" — 3 minutes after PgBouncer deploy

Thursday 14:22 IST  ·  Duration: 47 minutes  ·  Severity: P1  ·  100% error rate on all write endpoints

The infrastructure team had spent two weeks planning the PgBouncer migration. The connection count on the primary had been climbing — 380 connections at peak, dangerously close to max_connections = 500. PgBouncer in transaction mode would bring that down to ~30. The plan was solid. The execution was solid. The single missing detail was one line in the Django settings.

At 14:22, PgBouncer was deployed. DNS was updated. The first few requests succeeded. Then the errors started. ProgrammingError: ERROR: prepared statement "s1" does not exist. Not all requests — roughly every second or third. Growing quickly. By 14:25, all write endpoints were failing. Read endpoints still worked. Rollback initiated.

Senior engineer checks Django docs. Django uses server-side prepared statements by default for performance. psycopg2 sends PREPARE s1; EXECUTE s1; for parameterised queries. In transaction mode, the PREPARE lands on one server connection, the EXECUTE on a different one.
15:09 IST
Fix: DISABLE_SERVER_SIDE_CURSORS = True added to Django settings. Redeployed. Errors stop. PgBouncer running correctly. Connection count drops to 28.
Root cause

Django's psycopg2 driver uses server-side prepared statements by default. When psycopg2 sees a parameterised query, it sends PREPARE s1 AS SELECT... on the first execution, then EXECUTE s1($1, $2). In PgBouncer's transaction mode, each transaction may be assigned a different server connection from the pool. The PREPARE executed on connection A, but EXECUTE was sent to connection B — which had no record of s1. The fix is one line: DISABLE_SERVER_SIDE_CURSORS = True in Django settings, which makes psycopg2 inline parameters instead of using prepared statements.

The one-line fix for each major frameworkPYTHON / JS / RUBY
## Django (Python) — DISABLE_SERVER_SIDE_CURSORS
❌ Default: prepared statements enabled
DATABASES = {
    'default': {
        'ENGINE': 'django.db.backends.postgresql',
        'NAME': 'myapp',
        # DISABLE_SERVER_SIDE_CURSORS not set = prepared statements ON
    }
}

✅ Fixed
DATABASES = {
    'default': {
        'ENGINE': 'django.db.backends.postgresql',
        'NAME': 'myapp',
        'DISABLE_SERVER_SIDE_CURSORS': True,  ← one line
    }
}

## Node.js / pg driver — statement_timeout in connection URL
❌
const pool = new Pool({ connectionString: 'postgresql://...' });
✅
const pool = new Pool({
  connectionString: 'postgresql://...?options=-c%20jit%3Doff',
  statement_timeout: 30000,
  prepared: false,  // postgres.js / node-postgres: disable prepared statements
});

## Ruby on Rails — prepared_statements: false
❌ config/database.yml default
production:
  adapter: postgresql
  database: myapp
  # prepared_statements defaults to true
✅
production:
  adapter: postgresql
  database: myapp
  prepared_statements: false  ← one line

## Prisma — pgBouncer mode in connection URL
DATABASE_URL="postgresql://...?pgbouncer=true&connection_limit=1"
# pgbouncer=true disables prepared statements automatically
What this incident teaches
▶ORMs use server-side prepared statements silently. You don't write PREPARE anywhere. The driver does it automatically for performance. You only discover it when transaction mode breaks it.
▶Test PgBouncer in staging with the exact same ORM and settings as production. This error is 100% reproducible. If you'd run one write query in staging before deploying to production, you'd have caught it.
▶Read endpoints work but write endpoints fail. This is the exact symptom — reads are simple parameterised SELECTs that happen to work, writes use prepared statements that break. Use this to diagnose similar incidents quickly.
▶PgBouncer 1.21+ supports prepared statements in transaction mode. It tracks them per client connection and rewrites the protocol automatically. If you can't modify your ORM config or need prepared statements for performance, upgrade PgBouncer instead.
