Section 01

Why PostgreSQL connections are expensive

Unlike MySQL's thread-per-connection model, PostgreSQL uses a process-per-connection model. Every client connection spawns a new OS process via fork(). This process stays alive for the duration of the connection, consuming RAM, OS file descriptors, and shared memory segments regardless of whether it's actively serving a query or sitting idle.

The memory math

Each PostgreSQL backend process uses approximately:

  • Base RSS: 5-10MB per connection (shared libraries, stack, per-process memory)
  • work_mem: Up to work_mem per sort/hash operation within a query. With work_mem = 16MB and a complex query with 3 sort nodes, that's 48MB per active connection
  • Shared buffers: Shared across all connections (not per-connection), but each connection holds locks on buffer pages it's using

At 500 connections with work_mem = 16MB each running a moderately complex query: 500 × (8MB base + 32MB work_mem) = 20GB of RAM consumed by PostgreSQL processes alone. For a server with 32GB RAM and PostgreSQL itself needing shared_buffers, this leaves very little room.

The hidden connection establishment cost

Opening a new connection isn't free at query time either. Each new connection requires: fork() system call, authentication handshake (password verification, SSL handshake), session parameter initialisation (search_path, time zone, etc.), and shared memory attachment. This adds 5-30ms to the first query on any new connection — invisible in testing, painful at high connection churn rates.

The max_connections ceiling

PostgreSQL's max_connections defaults to 100. Setting it to 1000 doesn't scale linearly — above about 200 concurrent connections, PostgreSQL's lock management overhead (which uses shared memory structures that require locking) begins to degrade throughput even when queries are fast. The actual ceiling varies by workload, but the practical limit for OLTP is typically 200-400 connections before performance degrades.

Measuring connection overhead in productionSQL
-- Current connection count and state breakdown
SELECT state, count(*) FROM pg_stat_activity GROUP BY state ORDER BY count DESC;
-- active: running a query right now
-- idle: connected but doing nothing (the waste)
-- idle in transaction: open transaction, not running query (dangerous)
-- idle in transaction (aborted): error in transaction, not cleaned up

-- Memory per connection (approximate)
SELECT pid, rss/1024 AS rss_mb, query
FROM pg_stat_activity
JOIN (
  SELECT pid, CAST(split_part(split_part(pg_read_file('/proc/' || pid || '/status'),
    E'VmRSS:\t', 2), E'\n', 1) AS integer) AS rss
  FROM pg_stat_activity WHERE pid != pg_backend_pid()
) mem USING (pid)
ORDER BY rss_mb DESC LIMIT 10;

-- How long have idle connections been open?
SELECT pid, usename, application_name, state,
  now() - state_change AS time_in_state,
  now() - backend_start AS conn_age
FROM pg_stat_activity
WHERE state = 'idle'
ORDER BY time_in_state DESC;
-- Connections idle for hours are almost certainly from a misconfigured pool

Section 02

What PgBouncer does and how it works

PgBouncer sits between your application and PostgreSQL. Applications connect to PgBouncer (which appears to be a normal PostgreSQL server). PgBouncer maintains a small pool of real connections to PostgreSQL. When an application wants to run a query, PgBouncer assigns it one of the pooled connections for the duration of the transaction (or query), then returns the connection to the pool.

The fundamental multiplexing

1,000 application connections → PgBouncer → 50 PostgreSQL connections. The 1,000 application connections are "virtual" — they're TCP connections to PgBouncer, which is a single lightweight process. PgBouncer can handle thousands of client connections with very low overhead (it's single-threaded event-driven, not process-per-connection). The 50 real PostgreSQL connections are what the database actually sees.

At any moment, only the connections actively executing a transaction are using a real PostgreSQL connection. If your 1,000 application connections are 95% idle (waiting for the next request), only 50 are ever in use simultaneously — which fits in 50 real connections with room to spare.

The authentication layer

PgBouncer handles authentication from clients using its own user database (userlist.txt or HBA-style config). It connects to PostgreSQL with a fixed set of credentials. This means client passwords are verified by PgBouncer, not PostgreSQL directly. PgBouncer supports scram-sha-256, md5, and pass-through authentication (where it forwards the client's credentials to PostgreSQL unchanged).

The operational implication: changing a user's password in PostgreSQL requires also updating PgBouncer's userlist.txt. If they fall out of sync, PgBouncer will still authenticate the old password.

Section 03

Pooling modes: session, transaction, statement

This is the most important decision you make with PgBouncer. The mode determines when a server connection is returned to the pool. Getting it wrong either negates the pooling benefit or breaks your application.

Session mode

A server connection is assigned to a client for the entire duration of the client connection. The client gets the same server connection from connect to disconnect. This is functionally identical to connecting directly to PostgreSQL — the only benefit is that PgBouncer manages the connection lifecycle and can reconnect if the server restarts. No multiplexing benefit.

When to use: when you can't use transaction mode (see below), as a migration step toward transaction mode, or when you need per-session state to persist (SET commands, prepared statements, advisory locks).

Transaction mode (the goal)

A server connection is assigned to a client only for the duration of a single transaction. Between transactions, the connection returns to the pool. This is the mode that enables 1000→50 multiplexing. A client that runs one query per second and each query takes 10ms is using a server connection for 10ms out of every 1000ms — the other 990ms the connection is available for other clients.

The catch: server-side state doesn't persist between transactions. SET search_path, prepared statements, advisory locks, and LISTEN/NOTIFY subscriptions are tied to a server connection and don't survive being returned to the pool. Applications that use these features need modification to work with transaction mode.

Statement mode

A server connection is assigned for the duration of a single SQL statement. Even more aggressive than transaction mode. The consequence: multi-statement transactions are impossible — each statement gets a different server connection, so BEGIN ... COMMIT wraps three statements that execute on three different connections. Effectively no transactions. Only use for simple, single-statement, non-transactional workloads (read-heavy analytics, simple lookups).

Prepared statements break in transaction mode

ORMs like Django, Rails, and many PostgreSQL drivers use server-side prepared statements by default for performance. In transaction mode, a PREPARE on one connection is unavailable on a different connection — the next query in the "same session" may land on a different server connection. This causes prepared statement "s1" does not exist errors. Fix: disable server-side prepared statements in your ORM/driver, or use PgBouncer's max_prepared_statements feature (PgBouncer 1.21+ tracks and rewrites prepared statement traffic). The safest immediate fix: set prepared_statements=false in your connection URL.

Section 04

Pool sizing: the math that actually matters

The most common mistake with PgBouncer is setting pool_size too large. The goal is not to maximise pool_size — it's to have just enough server connections to keep all app threads busy without starving PostgreSQL of resources.

The formula

Start with PostgreSQL's practical ceiling. For most OLTP workloads: optimal_server_conns ≈ 2 × CPU_cores + 1. For a 16-core database server: ~33 connections. This sounds low, but if queries are CPU-bound, adding more connections beyond this doesn't increase throughput — it just adds scheduling overhead.

For I/O-bound workloads (lots of disk reads): you can go higher, maybe 4 × CPU_cores, because connections spend time waiting for I/O and can be interleaved. But even here, diminishing returns set in quickly above ~100 connections.

Total pool_size = connections_per_database × number_of_databases. PgBouncer's pool_size is per database-user combination, so with 3 application databases and pool_size=25: 75 total server connections.

The client_side math

How many client connections can PgBouncer handle? Client connections are cheap in PgBouncer (single-threaded event loop, no fork). max_client_conn can be set to 5,000-10,000 without stress. The question is whether your application's connection overhead at the OS level (TCP sockets) is acceptable.

With 10 application servers each running 50 threads: 500 potential simultaneous queries. With pool_size=50 and 500ms average query time: a queue builds if more than 50 queries are in-flight simultaneously. The average in-flight count = 500 threads × 50ms active query / 1000ms = 25 connections needed on average. Pool_size=50 gives 2× headroom. This is correct sizing.

PgBouncer configuration: the full reference

pgbouncer.ini — production configuration with annotationsINI
# /etc/pgbouncer/pgbouncer.ini

[databases]
# Format: alias = host=... dbname=... port=... user=... password=...
myapp = host=primary.db.internal dbname=myapp port=5432
myapp_ro = host=replica.db.internal dbname=myapp port=5432
# Clients connect to: psql -h pgbouncer_host -d myapp
# * wildcard routes any DB name to primary: * = host=primary.db.internal

[pgbouncer]
## ── Network ───────────────────────────────────────────────────
listen_addr = 0.0.0.0
listen_port = 5432              # same port as PostgreSQL — apps need no change
unix_socket_dir = /var/run/postgresql

## ── Authentication ────────────────────────────────────────────
auth_type = scram-sha-256       # or: md5, trust, hba
auth_file = /etc/pgbouncer/userlist.txt
# userlist.txt format: "username" "scram-sha-256$..." (hashed password)
# Generate hash: psql -c "SELECT concat('\"', usename, '\" \"', passwd, '\"') FROM pg_shadow WHERE usename='myapp'"
auth_query = SELECT usename, passwd FROM pg_shadow WHERE usename=$1
# auth_query: PgBouncer queries PostgreSQL to validate passwords at login
# Eliminates the need to maintain userlist.txt manually

## ── Pooling ───────────────────────────────────────────────────
pool_mode = transaction         # session | transaction | statement
max_client_conn = 5000          # max simultaneous client connections
default_pool_size = 25          # server connections per (db, user) pair
min_pool_size = 5               # keep at least 5 connections warm
reserve_pool_size = 5           # extra connections for traffic bursts
reserve_pool_timeout = 5.0      # wait 5s before using reserve pool
max_db_connections = 100        # global cap across all pools for one db
max_user_connections = 50       # global cap per user across all dbs

## ── Timeouts ──────────────────────────────────────────────────
server_connect_timeout = 15     # timeout connecting to PostgreSQL
server_login_retry = 15         # retry interval after failed server connect
client_login_timeout = 60       # client must authenticate within 60s
query_timeout = 0               # 0 = disabled. Set in app, not here.
query_wait_timeout = 120        # cancel if client waits >120s for a connection
idle_transaction_timeout = 30   # close connections idle in transaction >30s
# idle_transaction_timeout prevents "idle in transaction" accumulation

server_idle_timeout = 600       # recycle idle server connections after 10min
server_lifetime = 3600          # recycle server connections after 1 hour max
# server_lifetime prevents issues with very long-lived connections

## ── TLS (terminating at PgBouncer, re-encrypting to PostgreSQL) 
client_tls_sslmode = require
client_tls_cert_file = /etc/pgbouncer/server.crt
client_tls_key_file  = /etc/pgbouncer/server.key
server_tls_sslmode = require    # PgBouncer→PostgreSQL also encrypted

## ── Operations ────────────────────────────────────────────────
logfile  = /var/log/pgbouncer/pgbouncer.log
pidfile  = /var/run/pgbouncer/pgbouncer.pid
log_connections = 0             # 0=quiet (high traffic), 1=log all connects
log_disconnections = 0
log_pooler_errors = 1           # always log pool errors
stats_period = 60               # emit stats every 60s to log

admin_users = pgbouncer_admin   # can connect to pgbouncer virtual DB for stats
stats_users = monitoring        # read-only stats access

Section 06

PgCat and alternatives: beyond PgBouncer

PgBouncer is single-threaded. At very high connection rates, it can become a CPU bottleneck — not because pooling is slow, but because a single thread handles all client communication. PgCat is the multi-threaded successor that addresses this.

PgCat (Supabase's pooler)

PgCat is written in Rust, multi-threaded, and includes features PgBouncer lacks: native load balancing across primary and replicas, query-based routing (read queries automatically to replicas), prepared statement support in transaction mode (via query rewriting), and health checks with automatic replica failover. For high-throughput deployments (>50,000 connections/second), PgCat is worth evaluating.

pgpool-II

pgpool-II is an older, more feature-rich alternative that combines connection pooling, query routing, and even query caching. It's more complex to operate than PgBouncer and has its own quirks. Most teams that use it are using it for its query routing capabilities, not just pooling.

Application-side pools (pg, knex, Sequelize)

Every Node.js PostgreSQL driver has a built-in connection pool. These pools are per-process — in a single-process Node.js app, this is fine. In a multi-process deployment (PM2 clusters, Kubernetes pods), each process has its own pool, multiplying connection count. 10 Kubernetes pods × 20 connections per pool = 200 server connections from one service. Without PgBouncer, scaling to 50 pods means 1,000 connections.

Section 07

Application-side pools: sizing and monitoring

Even with PgBouncer in front of PostgreSQL, your application still needs its own connection pool to PgBouncer. This is the pool from your application process to the pooler. The sizing rules are different: PgBouncer connections are cheap (they're connections to a fast proxy, not to PostgreSQL itself), so you can size these more aggressively.

The correct sizing formula

app_pool_size = max_concurrent_requests × avg_db_time_fraction

If your Node.js service handles 1,000 concurrent requests and each request spends 5% of its time waiting for database responses: you need at most 50 simultaneous PgBouncer connections. A pool of 50 with a small buffer (60-70) is correct. Setting the pool to 500 "just in case" means 500 connections to PgBouncer, which wastes OS resources and confuses monitoring.

What to monitor in the application pool

  • Pool size (active + idle): Should be stable. Growing pool = load increase or connection leak.
  • Pending (queued) requests: Requests waiting for a free connection. Non-zero sustained = pool undersized or DB slow.
  • Connection creation rate: High rate = pool is cycling (not reusing). Usually means pool size is too small or connections are being closed too aggressively.
  • Checkout time P99: Time to get a connection from the pool. Should be <1ms. If it's >10ms: pool is undersized or DB is slow.

Section 08

Production reality

The pgbouncer admin console

PgBouncer exposes a virtual database called pgbouncer where you can run SQL-like commands to inspect pool state. Connect with: psql -h pgbouncer_host -U pgbouncer_admin -d pgbouncer. The most useful commands:

PgBouncer admin console — monitoring commandsSQL
-- Connect to admin console
-- psql -h pgbouncer -U pgbouncer_admin -d pgbouncer -p 5432

-- Pool statistics (most useful command)
SHOW POOLS;
-- database | user | cl_active | cl_waiting | sv_active | sv_idle | sv_used | sv_tested | sv_login | maxwait
-- cl_active:  client connections currently assigned a server connection
-- cl_waiting: clients waiting for a free server connection ← alert if > 0 sustained
-- sv_active:  server connections currently in use
-- sv_idle:    server connections idle in pool
-- maxwait:    longest wait time in seconds ← alert if > 1s

-- Throughput and latency statistics
SHOW STATS;
-- total_query_count, total_query_time, avg_query_time per database
-- Use to calculate actual QPS and average query duration through the pooler

SHOW STATS_AVERAGES;  -- rolling averages, easier to read

-- Client connections
SHOW CLIENTS;
-- Shows each client connection, its state, and how long it's been waiting

-- Server connections (to PostgreSQL)
SHOW SERVERS;
-- Shows each server connection, its state, and the client it's serving

-- Configuration (can be changed without restart)
SHOW CONFIG;
SET default_pool_size = 30;   -- live config change, takes effect immediately
RELOAD;                        -- reload pgbouncer.ini without full restart

-- Graceful maintenance
PAUSE myapp;    -- pause one database (waits for active txn to finish)
RESUME myapp;   -- resume
SUSPEND;        -- suspend ALL databases (for maintenance)
RESUME;

-- Kill a stuck connection
KILL myapp;     -- kill all connections to a database immediately

The "idle in transaction" killer

Connections stuck in "idle in transaction" hold row locks and prevent VACUUM. PgBouncer's idle_transaction_timeout setting kills these automatically. Set it to 30 seconds. Any legitimate transaction that's open for 30 seconds without making progress is either a bug or should be redesigned.

PgBouncer behind a load balancer

In production, run multiple PgBouncer instances for redundancy and horizontal scaling. Put them behind a TCP load balancer (HAProxy, AWS NLB). Applications connect to the load balancer — if one PgBouncer fails, connections are rerouted to the others. Use PgBouncer's health endpoint: a TCP connection that succeeds means PgBouncer is up and can route to PostgreSQL.

Section 09

The things nobody tells you

PgBouncer doesn't show up in pg_stat_activity with the real client info

When queries come through PgBouncer, pg_stat_activity shows PgBouncer's server-side connection as the client — not the application connection that sent the query. All queries from a pool of 25 connections appear to come from "pgbouncer" with the same IP. This makes it much harder to identify which application process is responsible for a slow or misbehaving query. Workaround: use application_name in the connection string (PgBouncer passes this through), or use query comments with the caller's identity (/* caller=api:order-service */).

The "too many clients" error is from PgBouncer's max_client_conn

When you see sorry, too many clients already, this error often comes from PgBouncer (when max_client_conn is hit), not PostgreSQL. The error message is the same. Check PgBouncer's log and admin console first before assuming it's PostgreSQL's max_connections.

Transaction mode and SET commands

SET search_path = myschema executed in transaction mode is lost when the server connection returns to the pool and is reassigned. The next query from that "session" may run on a connection with a different search_path. Fix: either use session mode (loses multiplexing benefit) or set the search_path at the database level (ALTER DATABASE myapp SET search_path = myschema), which persists across all connections.

Scram-sha-256 auth requires PgBouncer 1.14+

Older PgBouncer versions only support md5. If your PostgreSQL is configured to require scram-sha-256 (which it should be), you need PgBouncer 1.14 or newer for pass-through auth, or 1.21+ for native scram-sha-256 support. Check your version: pgbouncer --version.

Decision sequence

Configuring connection pooling — decide in this order

Each step eliminates a configuration mistake before you make it.

1
Does your app use prepared statements, advisory locks, or LISTEN/NOTIFY?
If yes: you need to either disable these features, use session mode (no multiplexing benefit), or use PgBouncer 1.21+ which handles prepared statements in transaction mode. Check your ORM: Django defaults to prepared statements (DISABLE_SERVER_SIDE_CURSORS = True to disable), Rails uses them, TypeORM uses them. Identify before choosing pool_mode.
prepared statements + transaction mode = broken queries
2
Calculate your required server connections (not "as many as possible")
Formula: pool_size = (CPU_cores × 2) + effective_spindle_count. For a 16-core server with SSD (treat as 1 spindle): pool_size = 33. For I/O-heavy workloads: CPU_cores × 4. This determines how many server connections PgBouncer needs. Set default_pool_size to this value, not to "what max_connections allows."
bigger pool_size ≠ more throughput above the CPU ceiling
3
Set idle_transaction_timeout — never skip this
Default is 0 (disabled). A connection stuck in "idle in transaction" holds row locks, blocks VACUUM, and prevents pool reuse. Set idle_transaction_timeout = 30. Any transaction open for 30 seconds without making progress is a bug. This setting kills it and forces the client to retry — which surfaces the bug rather than hiding it.
idle_transaction_timeout = 0 is a production hazard
4
Verify with SHOW POOLS — not just config inspection
After deploying, connect to the PgBouncer admin console and run SHOW POOLS. Check: cl_waiting should be 0 (clients not waiting). sv_idle should be >0 (connections available). maxwait should be <0.1s. If cl_waiting > 0 sustained: either pool_size is too small or queries are too slow. Don't assume config = reality.
SHOW POOLS is your ground truth
5
Reduce application pool sizes to match PgBouncer reality
Your application's pool (pg, knex, SQLAlchemy) connects to PgBouncer. With PgBouncer in transaction mode and pool_size=25, there's no point having each application pod with a pool of 50 — most of those connections will sit idle waiting for PgBouncer to assign them a server connection. Reduce application pool size to max_client_conn / number_of_pods. This reduces OS socket overhead and makes monitoring cleaner.
application pool → PgBouncer → PostgreSQL. Size each layer correctly.
6
Add alerting on cl_waiting and maxwait
Query SHOW STATS from your monitoring system every 30 seconds. Alert on cl_waiting > 0 sustained for more than 5 seconds (pool undersized or DB degraded), and maxwait > 1 (clients waiting >1 second for a connection — severe degradation). These two metrics are your early warning system for pooling problems before they become user-visible.
cl_waiting > 0 = pool emergency. page immediately.

Section 11

Connection pooling checklist

TXN
Pool mode

Use transaction mode. Disable server-side prepared statements in ORM. Verify no SET commands persisting across transactions.

SZE
Pool sizing

pool_size = 2×CPU+1 for OLTP. Not "max_connections ÷ apps". Verify with SHOW POOLS, not just config.

IDT
Idle timeout

Set idle_transaction_timeout=30. Kills "idle in transaction" connections that hold locks and prevent VACUUM.

MON
Monitoring

Alert on cl_waiting>0 and maxwait>1s from SHOW STATS. Export via pgbouncer_exporter to Prometheus.

HA
High availability

Run 2+ PgBouncer instances behind HAProxy/NLB. A single PgBouncer is a single point of failure in the connection path.

VER
Version

PgBouncer 1.21+ for prepared statement support in transaction mode. 1.14+ for scram-sha-256. Check: pgbouncer --version.
