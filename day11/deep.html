Deep Dive 01

PgBouncer HA:
dual instance with HAProxy

A single PgBouncer instance is a single point of failure. If PgBouncer crashes, all database connections from the application fail — even if PostgreSQL is healthy. Running two PgBouncer instances behind HAProxy provides redundancy: if one fails, HAProxy routes all traffic to the other within seconds.

Architecture

Two PgBouncer instances, each with the same configuration, both pointing at the same PostgreSQL primary. HAProxy listens on port 5432 (the standard PostgreSQL port — applications connect here as if talking to PostgreSQL directly). HAProxy health-checks each PgBouncer instance via TCP and routes connections to healthy instances.

When one PgBouncer fails: HAProxy marks it down and routes all new connections to the surviving instance. Existing connections through the failed PgBouncer are reset (the application will see connection errors and retry). Recovery: restart the failed PgBouncer, HAProxy detects it's healthy, and begins routing to it again.

HAProxy config for dual-PgBouncer HACONFIG
# /etc/haproxy/haproxy.cfg

global
    log stdout format raw local0 info
    maxconn 10000
    tune.ssl.default-dh-param 2048

defaults
    log     global
    mode    tcp                    # L4 TCP passthrough — don't inspect PostgreSQL protocol
    timeout connect  5s
    timeout client   10m           # idle client connections can stay open for 10 min
    timeout server   10m
    timeout tunnel   1h            # long-lived DB connections
    option  tcplog
    option  tcp-check              # TCP health check

# ── Frontend: what applications connect to ─────────────────────
frontend postgres_frontend
    bind *:5432
    default_backend pgbouncer_pool

# ── Backend: PgBouncer instances ───────────────────────────────
backend pgbouncer_pool
    balance leastconn              # distribute to instance with fewest connections
    option tcp-check
    
    # Health check: TCP connect to PgBouncer port
    # PgBouncer answers TCP connections even when paused/draining
    # For deeper check, use pgbouncer's admin console via check-send/check-expect
    tcp-check connect port 5432
    
    server pgbouncer-1 10.0.1.1:5432 check inter 5s rise 2 fall 3
    server pgbouncer-2 10.0.1.2:5432 check inter 5s rise 2 fall 3
    #
    # inter 5s:  health check every 5 seconds
    # rise 2:    need 2 consecutive successes to mark UP after failure
    # fall 3:    3 consecutive failures to mark DOWN
    # Failover time: 3 failures × 5s = 15 seconds max to detect failure

# ── Stats page (internal only) ─────────────────────────────────
frontend stats
    bind *:8404
    stats enable
    stats uri /stats
    stats refresh 10s
    stats auth admin:${HAPROXY_STATS_PASSWORD}

# ── Admin socket (for haproxy CLI) ─────────────────────────────
# Allows: echo "show servers state" | socat stdio /var/run/haproxy.sock
# global: stats socket /var/run/haproxy.sock mode 660 level admin
Systemd service — ensure PgBouncer restarts automaticallySYSTEMD
# /etc/systemd/system/pgbouncer.service
[Unit]
Description=PgBouncer connection pooler
After=network.target
Wants=network.target

[Service]
Type=forking
User=postgres
ExecStart=/usr/sbin/pgbouncer -d /etc/pgbouncer/pgbouncer.ini
ExecReload=/bin/kill -HUP $MAINPID
PIDFile=/var/run/pgbouncer/pgbouncer.pid
Restart=always                  # restart if crashed
RestartSec=5                    # wait 5s before restarting
StartLimitIntervalSec=60
StartLimitBurst=5               # don't restart more than 5 times per minute

[Install]
WantedBy=multi-user-target

# Apply:
systemctl daemon-reload
systemctl enable pgbouncer
systemctl start pgbouncer
systemctl status pgbouncer
Verify HA setup
✓Both PgBouncer instances show in HAProxy stats: curl http://haproxy:8404/stats | grep pgbouncer — both should show UP.
✓Failover test: stop one PgBouncer, verify connections route to the other within 15 seconds. Restart it, verify it rejoins.
✓Both instances have identical config: diff /etc/pgbouncer/pgbouncer.ini node1: node2: — only listen_addr should differ.
✓Systemd restarts on crash: kill -9 $(pgrep pgbouncer) — verify it restarts within 5 seconds.
Deep Dive 02

Prometheus metrics:
what to alert on and why

pgbouncer_exporter scrapes PgBouncer's SHOW STATS, SHOW POOLS, and SHOW CLIENTS output and exports them as Prometheus metrics. Here are the metrics that matter, the alert rules that catch real problems, and the Grafana queries to build a useful dashboard.

pgbouncer_exporter setup + key Prometheus alert rulesYAML
# docker-compose.yml — run pgbouncer_exporter as a sidecar
services:
  pgbouncer:
    image: pgbouncer/pgbouncer:1.21.0
    # ...

  pgbouncer-exporter:
    image: prometheuscommunity/pgbouncer-exporter:v0.8.0
    environment:
      DATA_SOURCE_NAME: "postgresql://pgbouncer_admin:password@pgbouncer:5432/pgbouncer?sslmode=disable"
    ports:
      - "9127:9127"          # Prometheus scrapes :9127/metrics
    depends_on:
      - pgbouncer

---
# prometheus.yml scrape config
scrape_configs:
  - job_name: pgbouncer
    static_configs:
      - targets: ['pgbouncer-1:9127', 'pgbouncer-2:9127']
    relabel_configs:
      - source_labels: [__address__]
        target_label: instance

---
# prometheus-alerts.yml — the rules that matter
groups:
  - name: pgbouncer
    rules:

    # CRITICAL: clients waiting for a server connection
    - alert: PgBouncerClientsWaiting
      expr: pgbouncer_pools_client_waiting_connections > 0
      for: 10s                 # wait 10s to avoid flapping on brief spikes
      labels:
        severity: critical
      annotations:
        summary: "PgBouncer clients waiting for connections"
        description: "{{ $value }} clients are queued waiting for a server connection in pool {{ $labels.database }}/{{ $labels.user }}. Pool may be undersized or DB is slow."

    # WARNING: maximum wait time too high
    - alert: PgBouncerMaxWaitHigh
      expr: pgbouncer_pools_max_wait_seconds > 1
      for: 30s
      labels:
        severity: warning
      annotations:
        description: "Clients waiting up to {{ $value }}s for a connection. SLA impact likely."

    # WARNING: pool near capacity
    - alert: PgBouncerPoolNearCapacity
      expr: >
        pgbouncer_pools_server_active_connections /
        (pgbouncer_pools_server_active_connections + pgbouncer_pools_server_idle_connections) > 0.9
      for: 2m
      labels:
        severity: warning
      annotations:
        description: "Pool {{ $labels.database }} is {{ $value | humanizePercentage }} full. Consider increasing pool_size."

    # CRITICAL: PgBouncer down
    - alert: PgBouncerDown
      expr: up{job="pgbouncer"} == 0
      for: 30s
      labels:
        severity: critical

    # WARNING: high avg query time (potential slow queries)
    - alert: PgBouncerHighQueryTime
      expr: >
        rate(pgbouncer_stats_total_query_duration_microseconds[5m]) /
        rate(pgbouncer_stats_total_query_count[5m]) / 1e6 > 1
      for: 5m
      labels:
        severity: warning
      annotations:
        description: "Average query time through PgBouncer is {{ $value | humanizeDuration }}. Check for slow queries."

---
# Key Grafana dashboard queries

# QPS through PgBouncer
rate(pgbouncer_stats_total_query_count[1m])

# Active server connections (how full is the pool?)
pgbouncer_pools_server_active_connections

# Client wait count (the critical metric)
pgbouncer_pools_client_waiting_connections

# Average query latency through PgBouncer (microseconds)

rate(pgbouncer_stats_total_query_count[5m])

# Connection reuse rate (how often is a free connection found vs new one created)
rate(pgbouncer_stats_total_server_assignment_count[5m]) -

Deep Dive 03

Transaction mode migration:
zero-downtime playbook

Migrating from direct connections (or PgBouncer session mode) to transaction mode is the highest-value change you can make to your database architecture. Done wrong, it takes down production in minutes (the prepared statement incident). Done right, it's invisible to users and drops server connection count by 90%. Here's the exact sequence.

Step 1: Audit your codebase for transaction-mode incompatible features

Before touching any infrastructure, identify everything that will break. The list to check:

  • PREPARE / EXECUTE — explicit prepared statements in SQL files
  • ORM-level prepared statements — Django, Rails, TypeORM all use them by default
  • SET search_path, SET work_mem, or any SET in session scope
  • LISTEN / NOTIFY — subscriptions tied to a connection
  • pg_advisory_lock() — session-level advisory locks (transaction-level works)
  • DECLARE cursor FOR SELECT ... — server-side cursors
  • CREATE TEMP TABLE — temp tables are connection-scoped
Complete migration playbook — 6 steps to transaction modeBASH
## ── STEP 1: Audit (run before any config change) ──────────────

# Find all SET commands in your codebase
grep -r "SET search_path\|SET work_mem\|SET session\|pg_advisory_lock\|LISTEN\|PREPARE\|TEMP TABLE" \
  ./app --include="*.py" --include="*.rb" --include="*.ts" --include="*.js"

# Check if ORM uses prepared statements
# Django: check if DISABLE_SERVER_SIDE_CURSORS is absent (default = prepared statements on)
# Rails: check config/database.yml for prepared_statements: false
# node-postgres: check if options.prepare is set

# Enable pg_stat_statements to find actual prepared statement usage
psql -c "SELECT calls, query FROM pg_stat_statements WHERE query LIKE '%PREPARE%' ORDER BY calls DESC LIMIT 20"

## ── STEP 2: Fix ORM prepared statements ───────────────────────

# Django
DATABASES['default']['DISABLE_SERVER_SIDE_CURSORS'] = True

# Rails
production: { prepared_statements: false }

# TypeORM
extra: { statement_timeout: 30000 }  # and set pgbouncer=true in URL

# Prisma
DATABASE_URL="...?pgbouncer=true"

# node-postgres (pg)
new Pool({ ... })  # pg doesn't use prepared statements by default ✓

## ── STEP 3: Fix SET commands ───────────────────────────────────

# Option A: move SET to database/role defaults (persists across connections)
psql -c "ALTER DATABASE myapp SET search_path = myschema, public"
psql -c "ALTER ROLE myapp_user SET work_mem = '64MB'"
# These settings apply to every new connection automatically

# Option B: move SET into the transaction (only needs this for the duration)
BEGIN;
SET LOCAL work_mem = '256MB';  -- SET LOCAL: only lives for this transaction
... expensive query ...
COMMIT;  -- SET LOCAL is released here
# SET LOCAL is safe in transaction mode -- it's scoped to the transaction

## ── STEP 4: Deploy PgBouncer in SESSION mode first ────────────

# Start with session mode — same as direct connection, no multiplexing
# but validates that applications can connect through the pooler
pool_mode = session

## ── STEP 5: Shadow traffic test in staging ────────────────────

# Change to transaction mode in staging
pool_mode = transaction

# Run your full integration test suite against staging
# Monitor for "prepared statement does not exist" errors
# Run load test: 100 concurrent users for 5 minutes
# Verify: error rate = 0, latency normal

## ── STEP 6: Canary deploy to production ───────────────────────

# Deploy ONE pod pointing at PgBouncer (transaction mode)
# Route 5% of production traffic to this pod
# Monitor for 10 minutes:
#   - Application error rate (should be 0)
#   - SHOW POOLS cl_waiting (should be 0)
#   - pg_stat_activity connection count (should start dropping)

# If all clear: roll out to 25%, then 50%, then 100%
# Each increment: monitor for 5 minutes before proceeding

# Post-rollout: verify connection count dropped as expected
psql -c "SELECT count(*) FROM pg_stat_activity WHERE application_name = 'myapp'"
# Should be ~pool_size, not number_of_pods × connections_per_pod
💡
LISTEN/NOTIFY needs a dedicated long-lived connection

Applications that use LISTEN for real-time notifications (webhooks, pub/sub) cannot go through PgBouncer's transaction mode — LISTEN requires a persistent connection to receive notifications. The solution: create a separate connection pool (or single connection) specifically for LISTEN/NOTIFY, bypassing PgBouncer for that purpose. Everything else goes through PgBouncer. This keeps 99% of your connections in the efficient transaction mode pool while maintaining LISTEN functionality on a single dedicated connection per subscriber.

Verify transaction mode migration is successful
✓Zero "prepared statement does not exist" errors in application logs after rollout.
✓pg_stat_activity connection count dropped to approximately pool_size × number_of_pgbouncer_instances.
✓SHOW POOLS shows cl_waiting = 0 under normal load — if non-zero, pool_size needs adjustment.
✓All SET commands either moved to ALTER DATABASE/ROLE defaults or converted to SET LOCAL within transactions.
✓Application response time P99 unchanged or improved after migration (pooling should reduce latency slightly by eliminating connection establishment overhead).
