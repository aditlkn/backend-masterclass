Section 01

Why rate limiting exists — and what it actually protects

Rate limiting enforces a maximum number of requests a client can make in a given time window. But "client" is a design choice: you can rate limit by IP address, by user account, by API key, by endpoint, by geographic region, or by any combination. Each granularity protects against a different class of problem.

The things rate limiting protects against

  • Accidental abuse: A developer's integration test running against production with no sleep between requests. A misconfigured cron job firing 100× instead of once. These are the most common cause of traffic spikes, and they're not malicious — but they're just as damaging.
  • Brute force attacks: Systematically trying passwords, API keys, or voucher codes. Without rate limiting on the auth endpoint, an attacker can try millions of combinations. With it, 5 failed attempts per minute makes brute force impractical.
  • Scraping and data harvesting: Competitors or bots downloading your entire product catalog or user data. Rate limiting makes this slow enough to be economically unviable.
  • Cost amplification: If your API triggers expensive downstream operations (LLM inference, external API calls, database joins), an unrestricted caller can run up costs that hurt you financially.
  • Resource starvation: One high-volume client consuming all database connections, CPU, or network bandwidth — starving other users.

Rate limiting vs throttling

Rate limiting hard-stops requests that exceed a threshold — the client gets a 429 Too Many Requests error. Throttling slows them down — requests are queued or delayed rather than rejected. Rate limiting is more common in APIs (clients can retry); throttling is more common in internal systems where queuing is acceptable.

💡
Rate limiting is not the same as load shedding

Rate limiting enforces per-client quotas. Load shedding drops requests when the system is overwhelmed — regardless of who sent them. You need both: rate limiting protects fair access, load shedding protects system stability. A well-designed system has rate limiting at the API gateway (per client) and load shedding at the application layer (global system health). Day 18 covers circuit breakers and load shedding.

Section 03

Token bucket: the right default for most APIs

The token bucket algorithm models a bucket that fills with tokens at a constant rate (the refill rate) up to a maximum capacity (the burst size). Each request consumes one or more tokens. If enough tokens are available, the request proceeds. If not, it's rejected with 429.

The bucket metaphor captures the key insight: unused capacity accumulates. A client that makes no requests for 10 seconds "saves up" tokens, allowing a burst when they do need it. This mirrors legitimate usage patterns — a user might be idle for 30 seconds and then click rapidly through several pages. A flat per-second limit would reject their burst; a token bucket accommodates it.

The two parameters

Capacity (burst size): The maximum number of tokens in the bucket. This is the maximum burst a client can ever send. A capacity of 100 with a refill rate of 10/second means a client who has been idle can send 100 requests instantly, but then must slow to 10/second.

Refill rate: How many tokens are added per second. This is the sustained rate limit. Even with a full bucket, once depleted, the client can only proceed at this rate.

Implementation: store tokens and last refill time

The elegant Redis implementation stores just two values per client: the current token count and the timestamp of the last refill calculation. On each request: calculate how many tokens have accumulated since the last check (elapsed * refill_rate), cap at capacity, subtract one for the current request, store back. The entire operation runs in a Lua script to ensure atomicity.

Section 04

Sliding window: precision without the boundary exploit

Fixed window rate limiting has a well-known vulnerability: a client can send 2× the limit by sending the full limit just before a window resets and immediately again just after. At 11:59:59 and 12:00:01, they've sent 200 requests that appear to be in different 60-second windows.

Sliding window eliminates this. Instead of a fixed reset time, the window always ends at "now" and started at "now minus window_size." Every request is considered in the context of the most recent window_size seconds, regardless of wall clock boundaries.

Two approaches

Sliding window log: Store a timestamp for every request in a sorted set. To check the limit, remove all entries older than the window, count what remains. Exact but memory-intensive — storing a timestamp per request means O(requests) memory per client.

Sliding window counter (hybrid): Combines two fixed windows with interpolation. Store counters for the current and previous window. The estimated count = prev_count × (1 - elapsed/window_size) + curr_count. Uses O(1) memory and is accurate to within about 0.1% in practice. This is what Cloudflare, Kong, and most production rate limiters use.

Sliding window counter — Redis implementationLUA + TYPESCRIPT
-- Redis Lua script: sliding window counter (atomic)
-- KEYS[1]: current window key (e.g. "rl:user:123:1700000060")
-- KEYS[2]: previous window key (e.g. "rl:user:123:1700000000")
-- ARGV[1]: window size in seconds
-- ARGV[2]: limit
-- ARGV[3]: current timestamp (seconds)

local curr_count = tonumber(redis.call('GET', KEYS[1])) or 0
local prev_count = tonumber(redis.call('GET', KEYS[2])) or 0
local window = tonumber(ARGV[1])
local limit = tonumber(ARGV[2])
local now = tonumber(ARGV[3])

-- How far we are into the current window (0.0 to 1.0)
local window_start = math.floor(now / window) * window
local elapsed_frac = (now - window_start) / window

-- Weighted estimate: prev window contributes less as we move through current
local estimated = prev_count * (1 - elapsed_frac) + curr_count

if estimated >= limit then
  return {0, math.ceil(estimated), limit}  -- denied: {allowed, count, limit}
end

-- Increment current window counter
redis.call('INCR', KEYS[1])
redis.call('EXPIRE', KEYS[1], window * 2)  -- keep for 2 windows
return {1, math.ceil(estimated) + 1, limit}  -- allowed

Section 05

Fixed window: simple, fast, and exploitable

Fixed window is the simplest implementation: INCR key; EXPIRE key window_seconds. A counter per client per time window. When the counter exceeds the limit, reject. When the window resets (key expires), the counter starts fresh.

The implementation is two Redis commands — cheaper than any other algorithm. For internal rate limits where clients are trusted and the boundary exploit isn't a concern, it's perfectly adequate. Internal service quotas, background job throttling, admin-initiated operations — fixed window is fine.

The boundary exploit matters for public-facing APIs where clients might deliberately time their requests. A sophisticated client can always send 2× your limit in a short window by straddling the reset boundary. For anything where this matters — authentication endpoints, payment processing, high-value operations — use sliding window or token bucket.

Fixed window — Redis implementation (2 operations)TYPESCRIPT
async function fixedWindowLimit(
  key: string, limit: number, windowSec: number
): Promise<{ allowed: boolean; count: number; resetAt: number }> {
  // Window key changes every windowSec seconds
  const windowKey = Math.floor(Date.now() / 1000 / windowSec);
  const redisKey = `rl:\${key}:\${windowKey}`;

  const pipeline = redis.pipeline();
  pipeline.incr(redisKey);
  pipeline.expire(redisKey, windowSec + 1);  // +1 buffer
  const [[, count]] = await pipeline.exec();

  const resetAt = (windowKey + 1) * windowSec;

  return {
    allowed: (count as number) <= limit,
    count: count as number,
    resetAt,
  };
}

// Usage: middleware
async function rateLimitMiddleware(req, res, next) {
  const key = req.user?.id || req.ip;
  const result = await fixedWindowLimit(`api:\${key}`, 100, 60);

  res.set({
    'X-RateLimit-Limit':     100,
    'X-RateLimit-Remaining': Math.max(0, 100 - result.count),
    'X-RateLimit-Reset':     result.resetAt,
  });

  if (!result.allowed) {
    res.set('Retry-After', result.resetAt - Math.floor(Date.now() / 1000));
    return res.status(429).json({ error: 'Too Many Requests', retryAfter: result.resetAt });
  }
  next();
}

Section 06

Redis implementation: why Lua scripts are essential

Rate limiting requires atomic read-modify-write operations. Without atomicity, the check-then-increment sequence has a race condition: two concurrent requests both read the current count, both see it's below the limit, both increment — and you've allowed one extra request. At high concurrency this becomes a significant problem.

Redis Lua scripts solve this. Scripts execute atomically — no other Redis command runs while the script is executing. The entire check-and-increment is one atomic operation. Additionally, Lua scripts that use only pure Redis commands (no external I/O) are also deterministic, making them safe to replicate to replica nodes.

The token bucket Lua script

The token bucket implementation needs to be especially careful about atomicity because it involves time-based calculation. The timestamp must be read inside the script (not passed from the client) to prevent clock skew between application servers from causing inconsistent behavior.

Token bucket — atomic Redis Lua scriptLUA
-- Token bucket: atomic implementation
-- KEYS[1]: bucket key (e.g. "tb:user:123")
-- ARGV[1]: capacity (max tokens)
-- ARGV[2]: refill_rate (tokens per second)
-- ARGV[3]: tokens_requested (usually 1, more for weighted requests)
-- Returns: {1, remaining} if allowed, {0, 0} if denied

local key = KEYS[1]
local capacity = tonumber(ARGV[1])
local refill_rate = tonumber(ARGV[2])
local requested = tonumber(ARGV[3])
local now = tonumber(redis.call('TIME')[1])  -- server time (not client!)

-- Get current state: {tokens, last_refill_time}
local bucket = redis.call('HMGET', key, 'tokens', 'last_refill')
local tokens = tonumber(bucket[1]) or capacity  -- start full
local last_refill = tonumber(bucket[2]) or now

-- Calculate tokens to add based on elapsed time
local elapsed = math.max(0, now - last_refill)
local new_tokens = math.min(capacity, tokens + elapsed * refill_rate)

if new_tokens < requested then
  -- Not enough tokens — update state but deny
  redis.call('HMSET', key, 'tokens', new_tokens, 'last_refill', now)
  redis.call('EXPIRE', key, math.ceil(capacity / refill_rate) + 1)
  return {0, math.floor(new_tokens)}
end

-- Allow: deduct tokens
local remaining = new_tokens - requested
redis.call('HMSET', key, 'tokens', remaining, 'last_refill', now)
redis.call('EXPIRE', key, math.ceil(capacity / refill_rate) + 1)
return {1, math.floor(remaining)}
⚠️
Use redis.call('TIME') not client-side timestamps

If you pass the current timestamp from your application server as a Lua script argument, clock skew between multiple application pods (even a few milliseconds) can cause rate limit drift. One pod's time being 100ms ahead of another means it calculates a slightly different token count. Always use redis.call('TIME') inside the Lua script to get the Redis server's clock — the single authoritative time source for all rate limit calculations.

Section 07

Distributed rate limiting: the multi-node problem

A single Redis instance handles distributed rate limiting correctly — all application pods read and write the same counter. The problem arises when you need Redis itself to be distributed: Redis Cluster or multiple Redis instances for availability. Now a request might hit a counter on node A while another request hits the counter on node B, and the counts aren't shared.

Single Redis (the right default)

Most systems should use a single Redis primary (with a replica for failover) for rate limiting. Redis can handle hundreds of thousands of rate limit operations per second on a single instance. The complexity of distributed rate limiting is only justified when your rate limiting Redis instance becomes a throughput bottleneck — which takes enormous traffic to reach.

Redis Cluster rate limiting

In Redis Cluster, each key is owned by a specific node. Rate limit keys that use the same hash slot (by using hash tags: {user:123}:rl) will always land on the same node, giving you correct counts. The counter for a specific user is always on the same node. This is actually fine — per-user rate limiting works correctly in Redis Cluster as long as you use hash tags consistently.

Eventual consistency rate limiting

For very high-throughput scenarios where even a single Redis is a bottleneck (think >500K rate-limit checks/second), approximate rate limiting with local counters works: each application pod maintains a local counter and periodically syncs with a central store. The result: slightly more permissive than exact limiting (because local counts lag behind global reality), but within a small factor. Used by Cloudflare and other edge systems at extreme scale.

✅
Use a dedicated Redis instance for rate limiting

Rate limiting generates very high write traffic — every request writes to Redis. If you share a Redis instance between your cache and your rate limiter, rate limit writes can evict cache entries (if memory is tight) and cache complexity can interfere with rate limit precision. Run a separate, small Redis instance for rate limiting. At typical API traffic, 512MB Redis handles millions of rate limit keys with room to spare.

Section 08

Response headers: what to tell the client

Good rate limit responses tell clients exactly what's happening so they can adapt. A client that receives a 429 with no information has to guess when to retry. A client that receives Retry-After: 23 and X-RateLimit-Remaining: 0 knows exactly what to do.

Rate limit headers — the standard setHTTP
# Successful response (within limit)
HTTP/1.1 200 OK
X-RateLimit-Limit: 1000          # requests allowed per window
X-RateLimit-Remaining: 842       # requests remaining in this window
X-RateLimit-Reset: 1700000060    # Unix timestamp when window resets
X-RateLimit-Policy: 1000;w=60   # IETF draft: limit;window in seconds

# Rate limited response
HTTP/1.1 429 Too Many Requests
X-RateLimit-Limit: 1000
X-RateLimit-Remaining: 0
X-RateLimit-Reset: 1700000060
Retry-After: 23                  # seconds until they can retry
Content-Type: application/json

{
  "error": "rate_limit_exceeded",
  "message": "Too many requests. Retry after 23 seconds.",
  "limit": 1000,
  "window": "60s",
  "retry_after": 23
}

# Token bucket response (remaining = tokens left, not requests)
X-RateLimit-Limit: 100           # bucket capacity
X-RateLimit-Remaining: 7        # tokens remaining
X-RateLimit-Reset: 1700000060   # when bucket will be refilled to capacity

# Multiple rate limit policies (IETF draft standard)
RateLimit: "burst";r=10;t=1, "sustained";r=1000;t=3600
# Two limits: 10 per second burst, 1000 per hour sustained

The Retry-After header is critical

Without Retry-After, a client receiving a 429 may immediately retry — causing a retry storm that makes the rate limiting problem worse. Always include Retry-After in 429 responses. The value should be the number of seconds until the client can retry, not a timestamp. Most HTTP clients understand Retry-After and will automatically honour it if they implement retry logic.

Section 09

Where to rate limit: gateway vs application vs database

Rate limiting can live at multiple layers. Each layer has different capabilities, different granularity, and different failure modes.

API Gateway (Kong, AWS API Gateway, Nginx)

The earliest possible enforcement point. Requests are rejected before they ever touch application code. The overhead is minimal — no database connections, no business logic executed. Works well for IP-based and API-key-based limits. Limitation: doesn't have access to application-level context (user plan tier, account status, past behaviour).

Application layer (your code)

Has full context. Can rate limit based on user subscription tier, account health, resource consumption, or any business rule. More expensive (request reaches application) but the only place where complex business logic can inform the limit. This is where you'd implement "free tier: 100 requests/day, paid tier: 10,000 requests/day."

Database layer

Almost never the right place for rate limiting. By the time a query reaches the database, all the expensive work is done. The only exception: preventing very expensive queries from running too frequently (e.g. a report generation endpoint that takes 10 seconds of database time — limiting it to once per hour at the application layer before the query runs).

The right answer: multiple layers

Gateway enforces coarse global limits (IP blocks, API key validity). Application enforces fine-grained business limits (tier-based, endpoint-specific, user behaviour). The gateway protects against floods; the application enforces the product's business rules.

Section 10

Production reality

Rate limiting by IP is less useful than you think

IP-based rate limiting is easily bypassed: attackers use distributed botnets, corporate users share NAT IPs (100 employees appear as one IP), mobile users roam across IPs. Rate limiting by authenticated user ID is far more effective for most threats. Use IP-based limiting only as a first line of defence against unauthenticated endpoints — login pages, registration, password reset.

Implement exponential backoff on the client side

A 429 response from your server should cause the client to back off exponentially: wait 1s, then 2s, then 4s, then 8s. Without this, clients receiving 429s may immediately retry — creating a retry storm that makes rate limiting worse. Publish your retry policy in your API documentation. SDK clients should implement backoff automatically.

Rate limit headers for gradual degradation

Don't wait until a client hits the limit to inform them. Send X-RateLimit-Remaining on every response. A well-written client can self-throttle as they approach the limit — slowing down when they see 50 requests remaining, rather than hammering until they hit 0 and start getting 429s. This makes the entire system more stable.

Monitor 429 rates, not just 5xx rates

A spike in 429s might indicate a legitimate client hitting limits (misconfiguration, increased load) or an attack. Both warrant investigation. Alert on 429 rate per API key — if a specific key suddenly generates 10× its normal 429s, it warrants investigation. Most APM tools track 5xx but not 429 — instrument it explicitly.

Section 11

The things nobody tells you

The INCR before EXPIRE race condition

A naive fixed-window implementation does INCR key; EXPIRE key 60 as two separate commands. If the server crashes between INCR and EXPIRE, the key has no TTL and will live forever — consuming memory and potentially blocking a client permanently. Fix: use a Lua script that sets both atomically, or use SET key 0 NX EX 60 to initialise with TTL before incrementing. The pipeline approach helps but doesn't fully guarantee atomicity across a crash.

Rate limiting doesn't protect against slow requests

A client sending 1 request per minute that each take 60 seconds (holding database connections, CPU, etc.) is not throttled by rate limiting. Rate limiting counts requests, not resource consumption. For this, you need request timeouts and connection limits — different tools. Rate limiting + timeouts + circuit breakers work together; each covers a different failure mode.

Your internal services need rate limits too

Most teams rate limit their public API but skip their internal services. An internal service calling another with a bug can generate millions of requests per second internally. Without rate limits between services, a bug in service A can take down service B. Implement rate limits on all service-to-service calls — smaller limits than public APIs, but limits nonetheless.

The 429 response body matters for developer experience

A 429 with body {"error": "rate_limit_exceeded", "limit": 1000, "window": "60s", "retry_after": 23, "docs": "https://api.example.com/docs/rate-limits"} is a good developer experience. A 429 with body "Too many requests" is not. Developers hitting your rate limits need to understand what limit they hit and how to stay within it. This information reduces support tickets and improves integrations.

Decision sequence

Designing rate limiting for a new API endpoint

Work through this for every endpoint that gets a rate limit.

1
What are you protecting against?
Brute force (auth endpoints) → limit by IP + user, tight limit, long window. Scraping → limit by user/key, moderate limit. Cost amplification → limit by user with weighted costs (expensive ops cost more tokens). Accidental overuse → generous limit with clear error messages. The threat model determines the algorithm, granularity, and limit values.
threat model first, then choose algorithm
2
What's the legitimate peak usage of your best customer?
Set your limit at 3-5× the 99th percentile of legitimate usage. This ensures no well-behaved client ever hits the limit, while blocking clear abusers. If your heaviest legitimate user sends 200 requests/minute, set the limit at 1000/minute. Monitor the gap between legitimate heavy users and your limit over time — if it shrinks, either limits or behaviour has changed.
limit too close to legitimate usage → false positives
3
Does your traffic have legitimate bursts?
If yes: token bucket. Users clicking rapidly, batch imports, webhook deliveries — all have legitimate burst patterns. Token bucket accommodates short bursts while enforcing a sustained rate. If traffic is smooth and steady (background jobs, polling): fixed window or sliding window counter is simpler and sufficient.
bursty traffic = token bucket
4
Is this a security-sensitive endpoint?
Login, password reset, OTP validation, API key creation — these need sliding window or token bucket to prevent the boundary exploit. Fixed window allows 2× the limit by straddling the reset. For auth endpoints, also add: IP-based limiting in addition to user-based, increasing delays on repeated failures (not just 429), and account lockout after sustained failures.
auth endpoints: sliding window, not fixed
5
Always include Retry-After and remaining headers
Every 429 response must include Retry-After (seconds until retry is safe) and the full rate limit header set. Every successful response should include X-RateLimit-Remaining. This gives clients the information to self-regulate and reduces retry storms. Add a link to your rate limit documentation in the 429 body.
no Retry-After = retry storm on every 429
6
Monitor 429 rate per client and set alerts
Track 429s per API key/user over time. A client suddenly generating 10× their normal 429 rate is either: a bug in their integration, increased load, or an attack in progress. Alert at 5× baseline 429 rate for any given client. Also monitor the aggregate 429 rate — a spike affecting many clients simultaneously suggests your limits are too tight or you're under a distributed attack.
unmonitored 429s = silent failures and missed attacks

Rate limiting checklist

LUA
Atomic scripts

All rate limit checks use Lua scripts. Never check-then-increment as separate commands — race condition allows extra requests through.

RTY
Retry-After

Every 429 includes Retry-After (seconds). Without it, clients retry immediately, creating a storm that worsens the problem.

SLI
Auth = sliding window

Login, password reset, OTP — all use sliding window, not fixed. Fixed window allows 2× the limit by straddling the reset boundary.

CLK
Server-side time

Token bucket uses redis.call('TIME') for timestamps, not client-side time. Clock skew across app pods causes drift.

MON
Monitor 429s

Alert on 429 rate per client. A sudden spike = bug, load change, or attack. Most teams only monitor 5xx — 429s go unnoticed.

INT
Internal limits too

Rate limit service-to-service calls, not just public APIs. A buggy internal service can DDoS another internal service.
