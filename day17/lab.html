Production Incident  ·  Auth Endpoint  ·  No Rate Limiting

No rate limit on login — 4.2 million attempts in one night

Tuesday night → Wednesday morning  ·  Discovered at 08:15 IST  ·  Severity: P0  ·  847 accounts compromised

The platform had a standard login endpoint: POST /api/auth/login. It accepted an email and password, checked bcrypt, returned a token. There was no rate limiting on it. The team knew they needed to add one — it was on the backlog. It never got prioritised.

Tuesday night, an attacker ran a credential stuffing attack using a list of 50 million email/password pairs leaked from another service. Their tooling sent 50 requests per second from rotating IPs. By Wednesday morning, the attack log showed 4.2 million login attempts. 847 accounts were successfully authenticated — users who had reused passwords from the breached service. The attacker now had 847 valid sessions. They started extracting data.

Tuesday 22:00 IST
Attack begins. 50 req/s from rotating IPs. Nginx logs show elevated 200 responses on /api/auth/login. No alerts configured for this.
Tuesday 23:30 IST
500K attempts. 200 successful. Auth service CPU elevated from bcrypt computation. Still no alerts trigger.
Wednesday 04:00 IST
2.1M attempts. 423 compromised accounts. Attacker's tooling begins data extraction from compromised sessions.
Wednesday 08:15 IST
On-call engineer notices unusual data export patterns. Investigates. Finds auth logs. 4.2M attempts. 847 compromised accounts. Incident declared.
Wednesday 09:30 IST
Sliding window rate limit deployed: 5 failures per IP per 15 minutes, 10 total per user per hour. All 847 compromised sessions invalidated. Affected users notified and forced to reset passwords.
Root cause + three-layer fix

No rate limiting on the authentication endpoint allowed 4.2 million credential stuffing attempts overnight. The fix added three layers: sliding window rate limit by IP (5 failures/15 min), sliding window by user (10 failures/hour), and an alert on 429 rate per endpoint. The backlog item that would have prevented this was estimated at "2 hours of work." The incident response took 3 days.

The fix — three-layer auth rate limitingTYPESCRIPT + LUA
❌ Incident: no rate limiting on auth endpoint
app.post('/api/auth/login', async (req, res) => {
  const { email, password } = req.body;
  const user = await db.findByEmail(email);
  if (!user || !await bcrypt.compare(password, user.hash))
    return res.status(401).json({ error: 'Invalid credentials' });
  // No rate limit. 4.2M attempts overnight.

✅ Fix: three layers of protection
app.post('/api/auth/login',
  rateLimitByIP({ limit: 5, window: 900 }),      // 5 per 15 min per IP
  async (req, res) => {
  const { email, password } = req.body;

  // Layer 2: rate limit by target user (before bcrypt — cheap check first)
  const userRL = await slidingWindow(`auth:user:${email}`, 10, 3600);
  if (!userRL.allowed) {
    return res.status(429).json({
      error: 'Too many attempts for this account',
      retry_after: userRL.resetAt - Date.now() / 1000
    });
  }

  const user = await db.findByEmail(email);
  const valid = user && await bcrypt.compare(password, user.hash);

  if (!valid) {
    // Layer 3: track consecutive failures → lockout
    const failures = await redis.incr(`auth:failures:${email}`);
    await redis.expire(`auth:failures:${email}`, 3600);
    if (failures >= 10) {
      await redis.setex(`auth:locked:${email}`, 3600, '1');
      // Alert security team on repeated lockout
      await alerting.notify('auth_lockout', { email, failures });
    }
    return res.status(401).json({ error: 'Invalid credentials' });
  }

  await redis.del(`auth:failures:${email}`);  // reset on success
  const token = jwt.sign({ userId: user.id }, JWT_SECRET);
  return res.json({ token });
});
What this incident teaches
▶Auth endpoints are the highest-value targets and need the strictest rate limits. 5 attempts per IP per 15 minutes is the right starting point. Most legitimate users never trigger it; attackers are stopped after 5 tries.
▶Rate limit before bcrypt, not after. bcrypt is intentionally slow (~100ms). 50 req/s × 100ms = 5 CPU cores just for bcrypt. Check rate limits first — it's a fast Redis operation. If rate limited, you never touch bcrypt.
▶Use sliding window for auth, not fixed window. Fixed window allows the boundary exploit — 10 attempts right before midnight, 10 more right after. Sliding window enforces: never more than 5 attempts in any 15-minute window.
▶Alert on elevated 401 and 429 rates. The attack ran for 10 hours before discovery. An alert on "more than 100 401s per minute on /api/auth/login" would have caught it within minutes. Instrument every auth failure explicitly.
