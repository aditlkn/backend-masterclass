You do not ship code.

that code works.

A deployment pipeline is not about automation. It is about moving risk left — catching failures in CI before they reach staging, in staging before they reach production, in canary before they reach all users. The goal is not to deploy faster. It is to deploy safely at whatever speed your pipeline allows.

Section 01

The theatre production — rehearsal before the show

You do not open a play on the first day of rehearsal. The script goes through readings, blocking sessions, costume fittings, technical rehearsals, dress rehearsals. Each stage exists to catch a different category of problem: does the script make sense? do the actors know their lines? does the lighting work with the costumes? does the whole thing run at the right pace? By opening night, you have found and fixed every problem you could possibly find before an audience sees it.

That is a CI pipeline. Every time code is pushed, the pipeline runs the equivalent of a dress rehearsal: compile, test, lint, security scan, build the container image. Problems found here cost a developer 10 minutes. Problems that reach production cost an on-call engineer 3am and cost users their trust.

Even so, you do not open to a full house on opening night. A new production starts with a soft opening — a preview for a smaller invited audience. You watch the reactions, listen for problems, make adjustments. Only when the preview goes well do you open to the full house. This is canary deployment: route 5% of traffic to the new version, watch the error rates and latency, promote to 100% if healthy, rollback in 30 seconds if not.

The understudy allows a scene to be swapped mid-performance. If a principal actor is sick, the understudy steps in without stopping the show. The audience barely notices. Feature flags are the understudy system for software: a feature can be enabled or disabled at runtime, per user segment, without a deploy. A bad feature can be turned off in seconds. An experiment can be limited to 10% of users. A migration can be done gradually with a flag controlling which path is taken.

GitOps is the stage manager with the authoritative script. The stage manager has the master copy of every cue, every blocking note, every prop placement. If anything on stage does not match the script, the stage manager corrects it. GitOps does the same: Git is the single source of truth for what should be running. If the cluster state diverges from what Git says — a pod was manually scaled, a config was edited in place — the GitOps controller corrects it automatically.

🎪
Where the theatre analogy breaks

A play is the same every night — the goal is repeatability. Software deployments aim for repeatability too, but the production environment changes constantly: traffic patterns shift, dependencies update, data grows. A production that worked fine in rehearsal can fail in ways that could never have been anticipated. This is why observability (Day 19, 20) matters as much as the pipeline — you need to see what is happening after the show opens, not just confirm it passed the dress rehearsal.

Section 02

What should every CI pipeline actually do?

A CI pipeline is not just "run the tests." It is a sequential gate process where each gate catches a different category of failure. Gates run fast-first: the 10-second lint check runs before the 5-minute integration test suite. The first failure stops the pipeline — no point running expensive tests after the build fails.

The four gates, in order

Gate 1 — Static analysis (seconds): Linting, type checking, dependency vulnerability scan. No runtime, no containers. Catches syntax errors, type mismatches, known vulnerable packages. Every engineer should run this locally before pushing.

Gate 2 — Unit tests (30 seconds to 2 minutes): Pure functions, mocked dependencies. Fast. Catches logic errors in isolation. The fastest feedback loop in the pipeline.

Gate 3 — Integration tests (2-10 minutes): Real database, real cache, real message queue — but not real downstream services. Catches interface mismatches, migration issues, connection configuration problems. Spin up dependencies with Docker Compose or testcontainers.

Gate 4 — Build and push (1-5 minutes): Build the container image, push to the registry, run a final smoke test against the built image. The output of this gate is the artifact that gets deployed — the same image, without rebuilding, across all environments.

✅
Build once, deploy everywhere

The container image built in CI is the exact same image deployed to staging, then production. No rebuilding. No "works in staging but not production" caused by environment-specific build behaviour. Configuration differences between environments come from ConfigMaps and Secrets — not from different build processes. If you are building the image differently per environment, you are testing something you are not deploying.

.github/workflows/ci.yml — GitHub Actions pipelineYAML
name: CI
on: [push, pull_request]

jobs:
  ci:
    runs-on: ubuntu-latest
    steps:
    # Gate 1: Static analysis
    - uses: actions/checkout@v4
    - uses: actions/setup-node@v4
      with: { node-version: '20' }
    - run: npm ci
    - run: npm run lint
    - run: npm run typecheck
    - uses: snyk/actions/node@master  # dependency vulnerability scan
      env: { SNYK_TOKEN: ${{ secrets.SNYK_TOKEN }} }

    # Gate 2: Unit tests
    - run: npm run test:unit -- --coverage
    - uses: actions/upload-artifact@v4
      with: { name: coverage, path: coverage/ }

    # Gate 3: Integration tests
    - name: Start test dependencies
      run: docker compose -f docker-compose.test.yml up -d
    - run: npm run test:integration
    - run: docker compose -f docker-compose.test.yml down

    # Gate 4: Build and push
    - uses: docker/login-action@v3
      with:
        registry: ghcr.io
        username: ${{ github.actor }}
        password: ${{ secrets.GITHUB_TOKEN }}
    - uses: docker/build-push-action@v5
      with:
        push: ${{ github.ref == 'refs/heads/main' }}
        tags: ghcr.io/${{ github.repository }}:${{ github.sha }}
        cache-from: type=gha
        cache-to: type=gha,mode=max

Section 03

Which deployment strategy fits which risk profile?

Every deployment strategy is a trade-off between deployment risk and deployment cost. The higher the risk of the change, the more expensive the strategy you can justify. A one-line bug fix does not warrant a full blue-green deployment. A payment processing rewrite does.

Rolling deployment — the sensible default

Replace pods one at a time, waiting for each new pod to pass its readiness probe before removing the old one. Simple, requires no extra infrastructure, works with Kubernetes out of the box (Day 27's maxUnavailable: 0). The limitation: if the new version has a bug, some users saw it before you rolled back. Rollback means another rolling deploy of the old version — not instant.

Blue-green — instant switch, instant rollback

Run two identical production environments: blue (current) and green (new). Deploy to green while blue serves all traffic. Run smoke tests against green. Switch the load balancer from blue to green in seconds. If something is wrong, switch back in seconds. Blue stays live for the rollback window. Cost: you are running two full production environments simultaneously during the switch — roughly 2x infrastructure cost for the deployment window.

Canary — gradual exposure with automatic rollback

Route a small percentage of traffic (5%, 10%) to the new version while the majority stays on the old version. Watch error rates, latency, and business metrics. Promote if healthy, rollback if not. The canary catches real-world failures that tests miss — edge cases in production data, unexpected traffic patterns, third-party integrations that behave differently under real load. Requires a traffic splitting mechanism (Istio, nginx ingress weight annotations, or an API gateway).

Section 04

Feature flags — decoupling deploy from release

A feature flag is a conditional in code that controls whether a feature is active. if (flags.isEnabled('new-checkout-flow', userId)) { ... }. The code is deployed to production but the feature is off. You turn it on for 1% of users, watch the metrics, increase to 10%, then 100%. If something breaks, you turn it off in the dashboard — no deploy, no rollback, no 3am incident.

Feature flags decouple two things that should have always been separate: deploying code (putting it on the server) and releasing a feature (making it available to users). Trunk-based development becomes practical — everyone commits to main, long-lived branches disappear, merge conflicts shrink. Code that is not ready for users is behind a flag, not in a branch.

The four types of feature flags

Release flags: Enable a new feature gradually. Turn on for internal users first, then 1%, 10%, 100%. Clean up (remove the flag and the old code path) once fully released. These are temporary by design.

Experiment flags: A/B test. 50% of users see version A, 50% see version B. Measure conversion, engagement, error rate. The winner becomes the default. The loser is removed. Requires analytics integration to know which variant the user saw.

Operational flags: Kill switches. If a feature is causing problems, turn it off. If a dependency is down, switch to a fallback. These may live permanently in the codebase as circuit breakers for specific features.

Permission flags: Entitlements. Premium users see this feature. Beta users get early access. Admin users see the debug panel. These are usually based on user attributes, not percentages.

⚠️
Flag debt is real and it accumulates fast

Every feature flag is a branch in your code. Ten flags means 2^10 = 1,024 possible combinations of active/inactive — most of which you will never test. Flags that are never cleaned up accumulate as dead code paths that maintain engineers must understand but never remove. Set a cleanup date when you create a flag. Release flags should be removed within 2-4 weeks of full release. Treat flag debt the same as technical debt.

Section 05

GitOps — Git as the source of truth for cluster state

GitOps is the practice of using Git as the single source of truth for infrastructure and application configuration. The cluster state is defined in YAML files in a Git repository. A GitOps controller (Argo CD, Flux) watches the repository and ensures the cluster matches what Git says. If someone manually edits a deployment in the cluster, the controller detects the drift and corrects it back to the Git state.

Why this matters

Without GitOps, the true state of production is "what someone last deployed with kubectl" — and that state lives in someone's terminal history. With GitOps, the true state of production is "the current HEAD of the production branch in the infrastructure repo" — and that state is audited, reviewed, and reversible with git revert.

The GitOps workflow

  1. CI builds and pushes a new container image (booking-service:abc1234)
  2. CI opens a PR updating the image tag in the Kubernetes manifests repo
  3. A human reviews and merges the PR (or automation does it for lower environments)
  4. Argo CD detects the manifest change and syncs the cluster to the new state
  5. Argo CD reports success or failure back to the PR/commit

Pull vs push deployment

Traditional CD pushes changes to the cluster: the pipeline runs kubectl apply. GitOps uses pull: the cluster controller pulls changes from Git. Pull is more secure — the cluster does not need to give CI/CD credentials access to kubectl apply. The controller in the cluster pulls from Git and applies changes itself, with tightly scoped RBAC for exactly what it can modify.

Section 06

Rollback — making it fast enough to actually use

Rollback is only useful if it is fast enough to be the first response to a production incident, not a last resort. If rollback takes 15 minutes, you will spend those 15 minutes debugging instead, making things worse. If rollback takes 30 seconds, it is the first thing you do — stop the bleeding, then investigate.

What determines rollback speed?

Strategy matters: Blue-green rollback is a load balancer switch — 30 seconds. Rolling deploy rollback is a rolling update of the old image — 5-10 minutes. Canary rollback is removing the canary route — seconds. Choose your deployment strategy based partly on how fast you need to be able to roll back.

The image must already exist: Rollback requires the old image to be in the registry. Keep at least 5 previous image versions. Container registries support image retention policies — configure them before you need a rollback.

Database migrations complicate rollback: If a deploy includes a database migration, rolling back the application without rolling back the migration may leave the old application incompatible with the new schema. Always write forward-compatible migrations: additive only, never remove or rename columns until all application versions that depended on them are gone.

🔴
Untested rollback is not rollback

Rollback procedures that have never been exercised will fail at the worst possible moment. Test your rollback in staging monthly. Treat a failed rollback test as a P1 incident. The rollback procedure is part of the deployment — if you cannot roll back, you have not finished deploying safely.

Section 07

Production reality

DORA metrics — measuring pipeline health

The DORA (DevOps Research and Assessment) four metrics measure delivery performance: Deployment frequency (how often you deploy to production — elite teams deploy multiple times per day), Lead time for changes (how long from commit to production — elite: less than 1 hour), Change failure rate (what percentage of deployments cause production incidents — elite: below 5%), Mean time to restore (how long to recover from incidents — elite: less than 1 hour). These four metrics are the closest thing to an objective measure of engineering delivery performance.

Trunk-based development enables the pipeline

Long-lived feature branches make CI meaningless — you are testing code that will not ship for weeks, not code in the state it will be deployed. Trunk-based development (everyone commits to main, no branches longer than a day or two) means CI is always running against the actual next deployment candidate. Feature flags handle incomplete features. The pipeline gates handle quality.

Environment parity — staging must mirror production

If staging uses a different database version, different infrastructure size, or different configuration than production, bugs will appear in production that did not appear in staging. Close the gap: same Kubernetes version, same database version, same secrets management approach. The difference should only be scale (3 replicas in production vs 1 in staging) and data (anonymised production data in staging, not synthetic data).

Section 08

The things nobody tells you

The pipeline is only as useful as the tests in it

A pipeline with 5,000 tests that take 45 minutes to run will be bypassed. Engineers will push to production directly to avoid waiting. A pipeline with 200 meaningful tests that run in 3 minutes will be trusted and used. Invest in making the pipeline fast — parallel test execution, test caching, testcontainers instead of full environment spin-up. Speed is not just a convenience; it determines whether the pipeline is actually used.

Canary requires metrics before it works

Canary deployment without observability is theatre. You route 5% of traffic to the new version — and then what? If you do not have error rate and latency metrics broken out by deployment version, you cannot tell whether the canary is healthy. Implement version-tagged metrics before implementing canary deployment. The deployment strategy and the observability strategy are inseparable.

Feature flags in tests are a maintenance problem

When your code has feature flags, your tests need to cover both paths — flag on and flag off. As flags accumulate, test complexity grows quadratically. Tests that rely on specific flag states break silently when flags are cleaned up. Use a feature flag testing strategy: explicitly set flag state in tests, do not rely on default values, and remove flag-specific tests when the flag is cleaned up.

Deploy frequency is a lagging indicator

Teams celebrate "we deploy 10 times a day" as if frequency is the goal. It is not — it is a result of having a safe, fast pipeline and small, independent changes. Teams that focus directly on increasing deploy frequency often achieve it by deploying more change per deploy, which increases risk. Focus on the enabling conditions: fast CI, good test coverage, feature flags, canary deployment, fast rollback. Deploy frequency follows.

Decision sequence

Building a deployment pipeline

In order. Do not skip to step 4 before steps 1-3 are solid.

1
Fast, reliable unit and integration tests first
Everything else depends on having tests that actually catch problems. A pipeline without meaningful tests is just a slow deployment script. Target: unit tests under 2 minutes, integration tests under 8 minutes, total pipeline under 15 minutes. If it is slower, engineers will bypass it.
slow or flaky tests = pipeline is bypassed
2
Build once, tag with git SHA, deploy everywhere
The container image built from a specific commit is the artifact. It gets the git SHA as its tag. The same image is promoted from staging to production — never rebuilt. Configuration changes (environment variables, secrets) come from the cluster, not the image. This ensures what you tested is what you deployed.
same image across all environments = no build-time surprises
3
Make rollback a first-class operation
Document the rollback procedure before the first production deploy. Practice it in staging. Measure how long it takes. If it takes more than 5 minutes, redesign the deployment strategy. Keep the previous 5 image versions in the registry. Write database migrations that are backward-compatible so the old application can run against the new schema.
untested rollback = no rollback at 3am
4
Add canary deployment for high-risk changes
Not every change needs a canary. A one-line bug fix in a well-tested function does not. A new payment processing flow does. Define what "high risk" means for your system — schema migrations, new external integrations, changes to core business logic — and require canary deployment for those. For everything else, rolling deploy is sufficient.
canary for high-risk, rolling for everything else
5
Add feature flags for user-facing feature releases
Start with a simple in-house flag system if needed (database table with flag name and enabled percentage). Migrate to a managed service (LaunchDarkly, Unleash, GrowthBook) when you need targeting rules, analytics, and multiple environments. Set a cleanup date when you create each flag. Remove it within 30 days of full release.
no cleanup date = permanent technical debt
6
GitOps when you have multiple environments and teams
GitOps pays dividends at scale — multiple environments, multiple teams, auditability requirements. For a single-team startup with one environment, the operational overhead of Argo CD is not justified. Add GitOps when you have the problem it solves: drift between what is deployed and what should be deployed, unclear deployment history, manual kubectl changes that do not match any PR.
GitOps when drift becomes a problem, not before

You do not open a play without a dress rehearsal. You do not open to a full house without a preview. You keep the understudy ready. The stage manager has the authoritative script. Every part of this process exists to move the risk of failure to a moment when it costs less — before users see it, before the on-call engineer wakes up, before the bad review lands.

The next time a deploy goes wrong — ask: which rehearsal would have caught this?
