Canary Incident  ·  booking-service v2.4.1  ·  8% error rate on canary

The canary is causing errors. The engineer disabled the wrong feature flag. Errors continue.

Saturday 14:00 · canary at 10% traffic · booking-service v2.4.1

A canary deployment of booking-service v2.4.1 was started at 13:00, routing 10% of traffic to the new version. At 14:00 an alert fires: error rate on the canary is 8%, vs 0.2% on stable v2.4.0.

The on-call engineer logs into the feature flag dashboard and disables new-payment-validation — a flag that was added in v2.4.1. Error rate stays at 8%. The engineer disables a second flag, enhanced-seat-selection. Error rate stays at 8%. The engineer is now not sure which flag to disable, and the canary is still causing errors for 10% of users.

Work through the correct diagnosis and actions below.

What this incident teaches
▶Feature flags only protect features that are behind flags. Core code path changes — like changing how events are fetched — affect all requests regardless of flag state. Before a canary goes out, identify which changes are in core paths and which are gated. Core path changes carry more risk than flagged features.
▶Canary rollback is not rollback of the stable deployment. The engineer instinctively wanted to "roll everything back" — but stable v2.4.0 was working fine throughout. The correct action was to remove the canary only, leaving stable untouched. Touching stable when it is healthy adds risk during an incident.
▶Read the error before touching anything. The engineer disabled two flags before reading the error message. The error was unambiguous: Cannot read properties of undefined (reading 'venueId') at line 127, in the core booking path. One minute of log reading would have saved the two flag-disabling attempts and made the diagnosis immediate.
▶Canary saved 90% of users. Despite the confusion during the incident, the canary pattern worked: only 10% of traffic was on the buggy version. 90% of bookings during those 60 minutes completed successfully on v2.4.0. Without a canary, all users would have seen 8% errors from the moment of deploy.
