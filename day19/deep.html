Deep Dive 01

Complete OTel Collector config:
tail sampling, scrubbing, multi-export

The OTel Collector is the architectural hub of your tracing stack. It receives spans from all services, applies sampling logic, scrubs PII, and exports to one or more backends. This config covers a production setup: tail-based sampling that keeps all errors and slow traces, PII scrubbing for GDPR compliance, and export to both Tempo (storage) and Jaeger Query (UI).

otel-collector-config.yaml — production setupYAML
# otel-collector-config.yaml

receivers:
  otlp:
    protocols:
      grpc:
        endpoint: 0.0.0.0:4317
      http:
        endpoint: 0.0.0.0:4318
        cors:
          allowed_origins: ["https://app.example.com"]

processors:
  # 1. Enrich spans with resource attributes from environment
  resourcedetection:
    detectors: [env, system, docker]
    timeout: 5s

  # 2. PII scrubbing — run BEFORE sampling so scrubbing is universal
  transform:
    error_mode: ignore
    trace_statements:
      - context: span
        statements:
          - delete_key(attributes, "user.email")
          - delete_key(attributes, "user.phone")
          - set(attributes["payment.card_last4"],
              Substring(attributes["payment.card_number"], 12, 4))
          - delete_key(attributes, "payment.card_number")
          - replace_pattern(attributes["db.statement"],
              "password\\s*=\\s*'[^']+'", "password='[REDACTED]'")

  # 3. Tail sampling — decide AFTER trace completes
  tail_sampling:
    decision_wait: 15s        # wait 15s for trace to complete
    num_traces: 50000         # buffer up to 50K in-flight traces
    expected_new_traces_per_sec: 100
    policies:
      # ALWAYS keep: error traces
      - name: keep-errors
        type: status_code
        status_code:
          status_codes: [ERROR]

      # ALWAYS keep: slow traces (>1 second)
      - name: keep-slow
        type: latency
        latency:
          threshold_ms: 1000

      # ALWAYS keep: specific user for debugging
      - name: keep-debug-users
        type: string_attribute
        string_attribute:
          key: user.id
          values: ["usr_debug_001"]
          enabled_regex_matching: false

      # Keep 10% of everything else
      - name: keep-10pct
        type: probabilistic
        probabilistic:
          sampling_percentage: 10

  # 4. Batch before exporting
  batch:
    timeout: 1s
    send_batch_size: 1024
    send_batch_max_size: 2048

exporters:
  # Grafana Tempo — primary storage (S3-backed)
  otlp/tempo:
    endpoint: grafana-tempo:4317
    tls:
      insecure: true

  # Jaeger — for Query UI (Jaeger reads from Tempo via plugin)
  # Or export directly to Jaeger Collector:
  jaeger:
    endpoint: jaeger-collector:14250
    tls:
      insecure: true

  # Prometheus — metrics derived from spans
  # Creates: trace_span_duration_milliseconds histogram per service+operation
  prometheus:
    endpoint: "0.0.0.0:8889"
    namespace: otel
    send_timestamps: true
    metric_expiration: 180m

extensions:
  health_check:
    endpoint: 0.0.0.0:13133
  pprof:
    endpoint: 0.0.0.0:1777
  zpages:
    endpoint: 0.0.0.0:55679  # debug UI at /debug/tracez

service:
  extensions: [health_check, pprof, zpages]
  pipelines:
    traces:
      receivers: [otlp]
      processors: [resourcedetection, transform, tail_sampling, batch]
      exporters: [otlp/tempo, jaeger]
  telemetry:
    logs:
      level: warn  # Collector's own logs
    metrics:
      address: 0.0.0.0:8888  # Collector self-metrics (spans processed, dropped, etc.)
Deep Dive 02

Docker Compose:
runnable self-hosted tracing stack

A complete local and production-ready tracing stack: OTel Collector receives spans, Grafana Tempo stores them in a local volume (swap for S3 in production), Jaeger Query provides the search and waterfall UI, and Grafana provides dashboards. Copy and run.

docker-compose.tracing.yaml — complete runnable stackYAML
version: "3.8"

services:
  # ── OTel Collector ──────────────────────────────────────────────
  otel-collector:
    image: otel/opentelemetry-collector-contrib:0.96.0
    command: ["--config=/etc/otel/config.yaml"]
    volumes:
      - ./otel-collector-config.yaml:/etc/otel/config.yaml
    ports:
      - "4317:4317"    # OTLP gRPC (from services)
      - "4318:4318"    # OTLP HTTP (from services)
      - "8888:8888"    # Collector metrics (scrape with Prometheus)
      - "8889:8889"    # Span-derived metrics
      - "13133:13133"  # Health check
      - "55679:55679"  # zPages debug UI
    depends_on:
      - grafana-tempo

  # ── Grafana Tempo (trace storage) ────────────────────────────────
  grafana-tempo:
    image: grafana/tempo:2.4.0
    command: ["-config.file=/etc/tempo.yaml"]
    volumes:
      - ./tempo-config.yaml:/etc/tempo.yaml
      - tempo-data:/var/tempo
    ports:
      - "4317"   # OTLP gRPC receiver (from Collector)
      - "3200"   # Tempo Query API

  # ── Jaeger (trace UI + search) ───────────────────────────────────
  jaeger-query:
    image: jaegertracing/jaeger-query:1.54
    environment:
      - SPAN_STORAGE_TYPE=grpc-plugin
      - GRPC_STORAGE_SERVER=grafana-tempo:7777  # Tempo's Jaeger plugin port
      - LOG_LEVEL=warn
    ports:
      - "16686:16686"  # Jaeger UI → http://localhost:16686
    depends_on:
      - grafana-tempo

  # ── Grafana (dashboards + trace search) ─────────────────────────
  grafana:
    image: grafana/grafana:10.3.0
    environment:
      - GF_AUTH_ANONYMOUS_ENABLED=true
      - GF_AUTH_ANONYMOUS_ORG_ROLE=Admin
    volumes:
      - ./grafana/provisioning:/etc/grafana/provisioning
      - grafana-data:/var/lib/grafana
    ports:
      - "3000:3000"    # Grafana UI → http://localhost:3000
    depends_on:
      - grafana-tempo

volumes:
  tempo-data:
  grafana-data:

# ── tempo-config.yaml (minimal) ──────────────────────────────────
# server:
#   http_listen_port: 3200
# distributor:
#   receivers:
#     otlp:
#       protocols: { grpc: {}, http: {} }
# storage:
#   trace:
#     backend: local  # swap for s3 in production
#     local:
#       path: /var/tempo
#
# Run: docker compose -f docker-compose.tracing.yaml up
# UI:  http://localhost:16686  (Jaeger)
#      http://localhost:3000   (Grafana)
Verify the stack is working
✓Collector healthy: curl http://localhost:13133/health/status — should return {"status":"Server available"}
✓Spans arriving: check http://localhost:55679/debug/tracez — shows spans being received and processed
✓Traces in Jaeger: open http://localhost:16686, select your service, click "Find Traces" — traces should appear within 15 seconds of the tail sampling window
✓Propagation confirmed: in Jaeger, click a trace and verify it shows spans from multiple services in the same trace tree, not as separate disconnected traces
Deep Dive 03

Trace-log correlation with Winston:
every log line gets a trace ID

The bridge between traces and logs is the trace ID in every log line. When you find a slow trace in Jaeger, you want to jump directly to the logs for that trace — filtered by trace ID, not by timestamp guessing. This Winston configuration emits the active trace ID and span ID in every log message automatically.

Winston trace-log correlation — automatic trace ID injectionTYPESCRIPT
import winston from 'winston';
import { trace, context } from '@opentelemetry/api';

// Winston format that injects active trace context into every log
const traceContextFormat = winston.format((info) => {
  const span = trace.getActiveSpan();
  if (span) {
    const ctx = span.spanContext();
    info.traceId = ctx.traceId;  // 32-char hex string
    info.spanId  = ctx.spanId;   // 16-char hex string
    info.traceFlags = ctx.traceFlags;  // 1 = sampled, 0 = not
  }
  return info;
})();

export const logger = winston.createLogger({
  level: process.env.LOG_LEVEL || 'info',
  format: winston.format.combine(
    traceContextFormat,         // inject trace context first
    winston.format.timestamp(),
    winston.format.errors({ stack: true }),
    winston.format.json(),     // structured JSON output
  ),
  defaultMeta: {
    service: process.env.SERVICE_NAME || 'unknown-service',
    version: process.env.APP_VERSION,
    env:     process.env.NODE_ENV,
  },
  transports: [
    new winston.transports.Console({
      silent: process.env.NODE_ENV === 'test',
    }),
  ],
});

// Usage — trace ID is injected automatically
logger.info('Processing checkout', { orderId, userId });
// Output:
// {
//   "level": "info",
//   "message": "Processing checkout",
//   "orderId": "ord_456",
//   "userId": "usr_123",
//   "traceId": "4bf92f3577b34da6a3ce929d0e0e4736",  ← trace correlation
//   "spanId": "00f067aa0ba902b7",
//   "service": "checkout-service",
//   "timestamp": "2024-01-15T14:32:01.234Z"
// }

// In Datadog / Grafana Loki / Elasticsearch:
// Filter logs by: traceId = "4bf92f3577b34da6a3ce929d0e0e4736"
// → See ALL log lines for that specific request across all services

// Error logging — stack trace also gets trace ID
try {
  await processPayment(orderId);
} catch (err) {
  logger.error('Payment failed', {
    err,          // winston.format.errors() extracts stack trace
    orderId,
    attemptCount: 3,
  });
  // {
  //   "level": "error",
  //   "message": "Payment failed",
  //   "traceId": "4bf92f3577b34da6a3ce929d0e0e4736",  ← same trace
  //   "err": { "message": "Timeout", "stack": "..." }
  // }
}
Verify trace-log correlation is working
✓Every log line has traceId: make a request, check the log output — traceId field should be present and non-null in every line generated during that request.
✓Same traceId across services: make a request that calls two services, compare their log outputs — the traceId should be identical in both.
✓Logs outside a request have no traceId: a background job or startup log should have traceId: undefined — not a stale ID from a previous request.
✓In your log aggregator (Loki, Datadog, Elasticsearch): filter by a traceId from Jaeger — you should see all log lines from that request across all services.
