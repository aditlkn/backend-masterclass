Production Incident  ·  Checkout Latency  ·  No Tracing

Checkout taking 4 seconds. Nobody knows why. For three days.

Ongoing · Discovered Monday 09:00 · Impact: 18% checkout abandonment increase · Revenue loss: ~₹2.4L/day

The checkout endpoint started taking 4 seconds to respond — up from a normal 340ms. Users were abandoning at the loading screen. Metrics showed the P99 spike clearly. Error rate was 0% — no exceptions thrown, no 5xx responses. Everything was technically "succeeding," just very slowly.

Without distributed tracing, the team spent three days guessing. They suspected the payment gateway (it had issues last month). They scaled up the checkout service pods. They restarted the database. None of it helped. The actual cause was two bugs compounding each other: a Redis cache key bug causing cache misses on every event seat lookup, combined with an N+1 query pattern on product pricing that nobody had noticed at lower traffic.

What tracing revealed in 4 minutes

One trace lookup showed the waterfall immediately: inventory.getSeatPricing took 3.6 seconds, containing 24 sequential db.SELECT pricing WHERE seat_id=? spans (one per seat in the booking) and a redis.GET returning a cache miss every time. The cache key was using event_id but the cache was populated with eventId — a JavaScript camelCase bug. Fix: 20 minutes. Three days of guessing would have been 4 minutes with tracing.

What the trace told us — annotated

Finding 1: inventory.getSeatPricing = 3612ms (94% of total time)
The span attributes show seat.count: 24 — this booking has 24 seats (a group booking). The span contains 24 child DB spans, one per seat. That's the N+1.
Finding 2: redis.GET on every pricing call — always a miss
Every redis.GET span has attribute cache.hit: false. The key being looked up: pricing:event_id:evt_789. The key in Redis: pricing:eventId:evt_789. One byte difference. Cache miss every time. At normal traffic (2-seat bookings) this was 2 DB queries — slow but tolerable. At 24 seats, 24 DB queries × 150ms each = 3.6 seconds.
The two bugs — and the fixesTYPESCRIPT
❌ Bug 1: cache key mismatch (event_id vs eventId)
async function getSeatPricing(seatId: string, eventId: string) {
  const cacheKey = `pricing:event_id:${eventId}`;  // ← snake_case
  const cached = await redis.get(cacheKey);
  if (cached) return JSON.parse(cached);
  // Cache was SET with: `pricing:eventId:${eventId}` (camelCase)
  // So every GET is a miss. DB hit on every seat.

❌ Bug 2: N+1 — fetching pricing per seat in a loop
async function getPricingForSeats(seatIds: string[], eventId: string) {
  const pricings = [];
  for (const seatId of seatIds) {          // ← loop!
    const p = await getSeatPricing(seatId, eventId);  // ← await each
    pricings.push(p);
  }
  return pricings;
}
// 24 seats × 150ms each = 3600ms

✅ Fix 1: consistent cache key
async function getSeatPricing(seatId: string, eventId: string) {
  const cacheKey = `pricing:event:${eventId}`;  // ← consistent key
  const cached = await redis.get(cacheKey);
  if (cached) return JSON.parse(cached);
  const pricing = await db.query(
    'SELECT * FROM pricing WHERE event_id = $1', [eventId]
  );
  await redis.setex(cacheKey, 300, JSON.stringify(pricing));
  return pricing;

✅ Fix 2: batch the DB query, run in parallel
async function getPricingForSeats(seatIds: string[], eventId: string) {
  // Cache check first (all seats share the same event pricing)
  const cacheKey = `pricing:event:${eventId}`;
  const cached = await redis.get(cacheKey);
  if (cached) return JSON.parse(cached);  // ← 1 cache check for all 24 seats

  // One query for all seats
  const pricing = await db.query(
    'SELECT * FROM pricing WHERE event_id = $1 AND seat_id = ANY($2)',
    [eventId, seatIds]
  );
  await redis.setex(cacheKey, 300, JSON.stringify(pricing));
  return pricing;
}
// Result: 1 cache check + (on miss) 1 DB query = ~5ms total

What this incident teaches about tracing
▶Tracing turns a 3-day investigation into 4 minutes. The metric showed something was slow. The trace showed exactly which span, which child spans, and which attribute (cache.hit: false) in one view. No logs grep, no SSH into servers, no guessing.
▶Add cache.hit as a span attribute on every Redis/cache call. This single attribute — true or false — is how you spot cache misses in a trace without reading logs. When you see 24 cache misses in a row, the waterfall makes the bug unmistakable.
▶N+1 bugs are invisible under normal load and catastrophic at scale. At 2 seats per booking, this was 2 DB queries — slow but survivable. At 24 seats (a group booking), 24 DB queries × 150ms = 3.6 seconds. The trace made the pattern obvious: 24 identical sequential spans with the same operation name.
▶Business context on spans makes traces searchable. The trace had user.id, order.id, seat.count: 24, and event.id. When a user reports "my booking is slow," you search for their user ID and see their exact trace — not just logs from their service. This is what makes tracing a team-wide tool, not just an engineering one.
