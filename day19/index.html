Section 01

Why logs and metrics aren't enough

In a monolith, debugging a slow request is straightforward: grep the logs for the request ID, read them in order, find the slow query. In a distributed system with 12 services, a single user request fans out across them — the API gateway calls the auth service, which calls the user service, which calls the session store; meanwhile the gateway also calls the product service, which queries the database, which hits a replica. Logs for all of these exist, scattered across 12 different log streams with no shared structure and no way to connect them.

Metrics tell you "the checkout endpoint's P99 latency spiked to 4 seconds at 14:32." They don't tell you whether those 4 seconds were in your service, in the payment gateway, in a database query, or in a Redis call. You know something is wrong. You don't know where.

Distributed tracing solves this by attaching a trace ID to every request at the entry point and propagating it through every service call, every database query, every cache lookup. When you look up that trace ID in your tracing backend, you see the complete journey — every hop, every duration, every attribute — assembled into a single unified view.

The three pillars of observability

  • Logs: What happened. Unstructured events with context. Cheap to produce, expensive to query at scale.
  • Metrics: How the system is behaving over time. Aggregated numbers: request rates, error rates, latency percentiles. Cheap to store, fast to query, but lossy — they can't tell you about an individual request.
  • Traces: The path of a specific request through the system. High-cardinality context about individual operations. Expensive to store, but irreplaceable for root cause analysis across service boundaries.
💡
Traces connect logs and metrics

The trace ID becomes a universal correlation key. A metric spike → find traces from that period → click into a trace → see which span was slow → jump to logs for that service filtered by trace ID → see exactly what the service was doing. Traces are the connective tissue between all three pillars.

Section 02

Trace anatomy: traces, spans, and attributes

A trace represents the entire journey of one request. It has a unique trace ID (128-bit, 32-char hex). Every operation creates a span — a named, timed operation with start time, end time, and attributes. Spans form a tree: the root span is created at the entry point; child spans are created by downstream calls with the calling span as their parent.

The anatomy of a span

  • Trace ID: Shared by all spans in the trace. The correlation key.
  • Span ID: Unique for this operation (16-char hex).
  • Parent span ID: Absent for the root span.
  • Operation name: checkout.payment, db.query, redis.get.
  • Start / end time: High-precision timestamps.
  • Status: OK, ERROR, or UNSET.
  • Attributes: Key-value pairs. http.method=POST, db.statement=SELECT..., user.id=123.
  • Events: Timestamped logs within the span. "Cache miss at T+234ms", "Retry #2 at T+891ms".
⚠️
Span names must be low-cardinality

Span names are used for aggregation — "how long does the payment span typically take?" If you put high-cardinality values in span names (GET /orders/847291 instead of GET /orders/{id}), you get millions of unique span types and lose aggregation entirely. Keep names to route patterns or operation types. Put specific IDs in span attributes.

Section 03

Context propagation: crossing service boundaries

The trace ID must travel from service to service — across HTTP headers, gRPC metadata, and message queue payloads. The W3C TraceContext standard defines how:

traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01

Format: version-traceID-parentSpanID-flags. The flags byte controls sampling: 01 = sampled, 00 = not sampled. The parent span ID is the calling service's span — child services use this to establish the parent-child relationship.

🔴
Missing propagation = broken traces

If Service A doesn't inject traceparent when calling Service B, Service B creates a new root span with no parent. You get two disconnected traces instead of one. The most common cause: manually setting headers and accidentally overwriting OTel's injected traceparent. Always merge headers, never replace them.

Section 04

Sampling: you can't keep every trace

At 10,000 requests per second, 100% sampling generates terabytes per day. Sampling decides which traces to keep.

Head-based sampling

Decision at the root span. If sampled, the traceparent flags byte (01) tells all downstream services to keep their spans. Simple, low overhead. Downside: you don't know at the start whether this trace will be interesting — you might discard an error that happens later.

Tail-based sampling

Collect all spans, then decide after the trace completes based on its properties — was there an error? Was it slow? Did it touch a critical path? Implemented in the OTel Collector's tail sampling processor. More intelligent but requires buffering spans in memory until the trace completes.

Smart sampling in practice

Start with 100% in development, 10% head-based in production plus always-keep for errors and slow requests. This captures what matters while reducing storage by 80-90%.

✅
Sample in the Collector, not the SDK

Configure the OTel SDK at 100% and do sampling in the OTel Collector. This lets you change sampling policy without deploying application code, and enables tail-based sampling — impossible at the SDK level because the SDK doesn't see the completed trace.

Section 05

OpenTelemetry: the standard instrumentation API

OpenTelemetry (OTel) is the CNCF standard for telemetry. Instrument once with OTel, route to any backend — Jaeger, Tempo, Honeycomb, Datadog — without changing application code. The key components:

  • API: The interface your code calls. tracer.startSpan(), span.setAttribute(), span.end().
  • SDK: Implements the API. Handles sampling, batching, export.
  • Auto-instrumentation: Automatically creates spans for HTTP, database drivers, Redis, Kafka — without any code changes.
  • OTel Collector: A standalone process between services and backends. Handles sampling, PII scrubbing, fan-out to multiple backends.
OpenTelemetry — Node.js setup and manual spansTYPESCRIPT
// instrumentation.ts — loaded BEFORE application code
import { NodeSDK } from '@opentelemetry/sdk-node';
import { OTLPTraceExporter } from '@opentelemetry/exporter-trace-otlp-http';
import { Resource } from '@opentelemetry/resources';
import { SemanticResourceAttributes as SRA } from '@opentelemetry/semantic-conventions';
import { getNodeAutoInstrumentations } from '@opentelemetry/auto-instrumentations-node';
import { ParentBasedSampler, TraceIdRatioBasedSampler } from '@opentelemetry/sdk-trace-base';

const sdk = new NodeSDK({
  resource: new Resource({
    [SRA.SERVICE_NAME]:    'checkout-service',
    [SRA.SERVICE_VERSION]: process.env.APP_VERSION,
    [SRA.DEPLOYMENT_ENVIRONMENT]: process.env.NODE_ENV,
  }),
  traceExporter: new OTLPTraceExporter({
    url: 'http://otel-collector:4318/v1/traces',
  }),
  sampler: new ParentBasedSampler({
    root: new TraceIdRatioBasedSampler(1.0), // 100% — sample in Collector
  }),
  instrumentations: [getNodeAutoInstrumentations()],
  // Auto-instruments: http, express, pg, redis, mongodb, kafka, grpc
});
sdk.start();

// Manual span for custom business logic
import { trace, SpanStatusCode } from '@opentelemetry/api';
const tracer = trace.getTracer('checkout-service');

async function processCheckout(orderId: string, userId: string) {
  return tracer.startActiveSpan('checkout.process', async (span) => {
    span.setAttributes({ 'order.id': orderId, 'user.id': userId });
    try {
      const result = await chargePayment(orderId);
      span.setStatus({ code: SpanStatusCode.OK });
      return result;
    } catch (err) {
      span.setStatus({ code: SpanStatusCode.ERROR, message: (err as Error).message });
      span.recordException(err as Error);
      throw err;
    } finally {
      span.end();  // always end the span
    }
  });
}

Section 07

Span attributes: what to put on every span

OTel defines semantic conventions — standard attribute names for common operations. Auto-instrumentation populates most of these. Add business context manually on the root span.

Semantic conventions + business context attributesTYPESCRIPT
// HTTP server span (auto-instrumented)
{ 'http.method': 'POST', 'http.route': '/checkout',  // LOW cardinality!
  'http.status_code': 200, 'net.peer.ip': '1.2.3.4' }

// Database span (auto-instrumented by pg driver)
{ 'db.system': 'postgresql', 'db.name': 'ticketdb',
  'db.operation': 'SELECT', 'db.sql.table': 'orders',
  'db.statement': 'SELECT * FROM orders WHERE id=$1' }

// Business context (add manually to root span)
span.setAttributes({
  'user.id':           userId,
  'user.plan':         'pro',
  'order.id':          orderId,
  'order.value_cents': 15000,
  'event.id':          eventId,
  'seat.tier':         'premium',
});

// Events (structured logs within a span)
span.addEvent('cache.miss', { 'cache.key': `event:\${eventId}:seats` });
span.addEvent('retry.attempt', { 'retry.count': 2, 'retry.delay_ms': 750 });
Section 08

Reading traces: patterns that reveal problems

A trace waterfall shows each span as a horizontal bar. Child spans are indented. Total trace duration is the root span's width. Four patterns appear repeatedly in production traces:

  • N+1 queries: Many identical spans in sequence, one per list item. Fix: batch into one query.
  • Unnecessary sequential calls: Independent calls starting one after another. Fix: Promise.all() — duration drops from sum to max.
  • Connection pool wait: Large gap before first child span. The root is doing nothing. Cause: waiting for a DB connection. Fix: increase pool size or reduce connection hold time.
  • Fat root, thin children: Root span takes 900ms, children total 80ms. 820ms unaccounted. Cause: uninstrumented CPU work (JSON parsing, serialisation). Fix: add manual spans around suspected code.

Section 09

Production reality

Auto-instrumentation gets you 80% of the way

OTel auto-instrumentation libraries create spans for HTTP, database queries, Redis, gRPC, and Kafka automatically. For most services this alone is useful. The remaining 20%: business context attributes on the root span, and manual spans for complex business logic.

The trace ID belongs in every log line

Emit the trace ID and span ID in every log line. Now logs and traces are connected: find a trace → jump to logs filtered by trace ID. Find an error log → jump to its trace. Without this connection, logs and traces are separate islands.

15-day retention is usually enough

Traces are primarily useful for debugging in the immediate aftermath of an incident. 15 days covers most post-incident investigations. Use metrics and logs for longer-term analysis.

Section 10

The things nobody tells you

Clock skew breaks trace ordering

A child span on Service B can appear to start before the parent span on Service A if B's clock is ahead. This causes negative durations in the parent-child gap — confusing but not a code bug. It's clock drift between servers. NTP synchronisation helps but doesn't fully eliminate it.

Async work needs explicit context serialisation

When you enqueue a job and a worker processes it later, the trace context must travel with the job payload. Most queue libraries don't propagate trace context automatically. The worker's spans should be children of the enqueuing span — but without explicit serialisation/deserialisation in the job payload, the worker starts a fresh disconnected root trace.

Tracing async work is harder than synchronous calls

Always serialise the trace context into job payloads: { ...payload, _traceContext: propagator.inject({}) }. In the worker: extract and restore the context before creating any spans.

Decision sequence

Adding tracing to a service

Do these in order. Each step builds on the previous.

1
Choose a backend first
Self-hosted on budget: Grafana Tempo + Jaeger UI. Managed, need deep queries: Honeycomb. Already on Datadog: Datadog APM. The backend doesn't affect instrumentation code — you're using OTel either way.
Tempo for cost · Honeycomb for query power
2
Deploy the OTel Collector before instrumenting
Collector is the receiver. Services send to Collector; Collector sends to backend. Having it running before you instrument means traces start flowing immediately. Without it, SDK export errors silently drop spans.
Collector first, then instrument services
3
Enable auto-instrumentation — don't write manual spans yet
Install OTel SDK with auto-instrumentation. Verify traces appear in your backend. Start at 100% sampling to confirm everything works before tuning. getNodeAutoInstrumentations() gives you HTTP, DB, Redis, Kafka spans instantly.
auto-instrument first, verify, then add custom spans
4
Add business context to root spans
Enrich root spans: user.id, org.id, plan tier, order IDs, entity IDs. This makes traces searchable by business dimension. Without business context, traces are technical artifacts nobody outside engineering can use.
business context makes traces useful to everyone
5
Add trace ID to every log line
Connect traces to logs: emit traceId and spanId in every log message. Now a trace links to logs, and logs link to traces. The two systems become one unified debugging tool.
traces without log correlation are half as useful
6
Configure smart sampling in the Collector
Once confirmed working: always keep errors and traces over 1s, sample 10% of everything else. Reduces storage ~90% while keeping all interesting traces. Revisit thresholds quarterly as traffic grows.
don't sample too aggressively early — you'll miss things

Section 12

Distributed tracing checklist

PRP
Propagation works

traceparent header is injected on every outbound call. Verify child service spans appear in the parent's trace.

BIZ
Business context

Root span has user.id, order.id, entity IDs. Makes traces searchable by business dimension, not just technical.

LOG
Trace ID in logs

Every log line emits traceId + spanId. Jump from trace to logs and back. Without this, they're disconnected.

ERR
Errors always kept

Sampling never discards error traces. Always-sample-on-error in Collector. You need these when debugging.

LOW
Low-cardinality names

Span names: route patterns (/orders/{id}), not values. IDs go in attributes. Never in names.

COL
Collector deployed

OTel Collector between services and backend. Enables backend switching, tail sampling, PII scrubbing without deploys.
