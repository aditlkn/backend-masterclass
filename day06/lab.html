Production Incident  ·  API Gateway  ·  Auth

8% of requests returning 401 on valid tokens

Thursday 02:47 IST  ·  Duration: 4h 12m  ·  Severity: P1  ·  ~40,000 affected sessions

It started with a Slack alert at 2:47am: auth error rate on the payments API had spiked from 0.02% to 8.3%. On-call engineer woke up, checked the auth service — healthy. Checked the gateway — healthy. Checked the upstream payment service — healthy. All dashboards green. Errors still coming in.

The errors were intermittent in a maddening way: the same user could make a request and get 200, then immediately retry and get 401. The token hadn't changed. The user hadn't done anything. Whether a request succeeded or failed depended entirely on which gateway instance handled it. The team didn't know that yet.

02:47 IST
PagerDuty fires. Auth error rate: 8.3%. On-call engineer begins investigation.
02:55 IST
Auth service checked — fully healthy. All token issuance working. Engineers suspect DB issue.
03:20 IST
DB checked, slow query log clear. Engineers add debug logging to auth service. Still seeing 401s. One hour lost.
04:10 IST
Second engineer joins. Notices that 401 errors all have the same kid in the JWT header: key-2024-02. Tokens with key-2024-01 are succeeding.
04:22 IST
Check JWKS endpoint — returns both keys correctly. But gateway node 3 is returning a JWKS response with only key-2024-01. Node 3 has a stale cache.
04:35 IST
Root cause found: Node 3's Redis connection had silently failed at 02:31 — 16 minutes before key rotation at 02:47. Without Redis, the background JWKS refresh had been silently failing. Node 3 never picked up the new key.
04:59 IST
Node 3 restarted. JWKS re-fetched fresh. 401 errors drop to zero. Incident resolved. 4h 12m of 8% auth failures.
Root cause

Two compounding failures: (1) The JWKS cache refresh depended on Redis for distributed locking. When Redis failed, the refresh silently stopped rather than falling back to a direct HTTP fetch. (2) There was no alerting on JWKS refresh failures — the errors were logged but nobody was watching that log stream. The key rotation that happened 16 minutes later exposed the stale cache on node 3, but the underlying bug had existed for months.

What you're actually controlling

Each toggle maps to a specific line or block in a production JWKS implementation. This is the exact code that was missing or broken during the incident.

Toggle: "Stale fallback on Redis fail" — what it controls TYPESCRIPT 
async refresh(): Promise<void> {
  try {
    const res = await fetch(jwksUri);
    const keys = await res.json();
    this.cache = new Map(keys.map((k: JWK) => [k.kid, jwkToPem(k)]));
    this.staleCopy = this.cache;  // always update stale copy on success
  } catch (err) {
    // ❌ INCIDENT VERSION — just throws, no fallback
    throw err;  // Redis failing caused this to propagate up silently

    // ✅ FIXED VERSION — serve stale keys if refresh fails
    if (this.staleCopy.size > 0) {
      this.cache = this.staleCopy;
      logger.error({ err }, '⚠ JWKS refresh failed — serving STALE keys. Alert!');
    } else {
      throw err; // first-ever fetch failed, nothing to fall back to
    }
  }
}
Toggle: "kid-miss refetch" — what it controls TYPESCRIPT 
async getKey(kid: string): Promise<string> {
  // Check cache first
  if (this.cache.has(kid)) return this.cache.get(kid)!;

  // ❌ INCIDENT VERSION — kid not found, immediately fail
  throw new Error(`Unknown kid: \${kid}`); // → 401, no retry


  // ✅ FIXED VERSION — kid miss could mean key rotation happened
  // Fetch fresh JWKS once and retry before giving up
  logger.info({ kid }, 'kid not in cache — refetching (rotation?)');
  await this.refresh();
  const pem = this.cache.get(kid);
  if (!pem) throw new Error(`Unknown kid after refetch: \${kid}`); // truly invalid
  return pem;
}
Toggle: "key-2024-02 in JWKS" — the rotation overlap window TYPESCRIPT 
// Auth provider's JWKS endpoint — what it should return during rotation

// ❌ WRONG: remove old key the moment new key is added
keys: [{ kid: 'key-2024-02', ... }]  // old key gone — breaks in-flight tokens


// ✅ CORRECT: both keys coexist during the overlap window
keys: [
  { kid: 'key-2024-01', ... },  // old key — still valid for existing tokens
  { kid: 'key-2024-02', ... },  // new key — used for new tokens from now
]
// Remove key-2024-01 only after ALL tokens signed with it have expired
// (= token TTL after switching to new signing key, typically 15min-24h)
What this incident teaches
▶ Silent failure is the most dangerous failure. Redis going down logged an error nobody saw. JWKS refresh silently stopped. No alert fired. The bug existed for 16 minutes before the key rotation made it visible.
▶ Stale data beats no data for auth infrastructure. A gateway serving 16-minute-old JWKS keys would have been completely fine. A gateway serving no keys for 4 hours broke 40,000 sessions.
▶ The kid-miss dance exists precisely for key rotation. Without it, any rotation where tokens reach a gateway node before the cache refreshes causes a 401 storm. The one-retry-on-miss pattern handles rotation transparently.
▶ Intermittent auth failures always mean split state. If some requests succeed and some fail with the same token, something has diverged between your instances — JWKS cache, feature flags, session state. Go look for the node that's different.
▶ Alert on JWKS refresh failures, not just auth failures. By the time auth failures alert, users are already affected. Alerting on "JWKS refresh failed" catches the problem before any user sees it.
