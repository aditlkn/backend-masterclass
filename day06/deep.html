Deep Dive 01

JWKS caching & key rotation

The concept is simple: validate JWTs using public keys fetched from a JWKS endpoint. The implementation has five non-obvious problems that will all bite you in production if you don't design for them upfront.

How JWKS validation actually works

A JWT has three parts. The header contains a kid (Key ID) field — a string identifying which signing key was used. The gateway needs the corresponding public key to verify the signature. That public key lives at your auth provider's JWKS endpoint: https://auth.example.com/.well-known/jwks.json.

The JWKS response is a JSON object containing an array of keys. Each key has a kid, a kty (key type, e.g. RSA), and the key material (n and e for RSA public keys). To validate a token, you find the key whose kid matches the token's header kid, reconstruct the public key from the key material, and verify the signature.

JWKS response structureJSON
{
  "keys": [
    {
      "kid": "key-2024-01",       // key identifier — must match JWT header
      "kty": "RSA",             // key type
      "alg": "RS256",           // algorithm
      "use": "sig",             // purpose: signature verification
      "n":   "0vx7agoebGcQ...",  // RSA modulus (base64url)
      "e":   "AQAB"             // RSA public exponent
    },
    {
      "kid": "key-2024-02",       // ← new key added during rotation
      "kty": "RSA",
      "alg": "RS256",
      "use": "sig",
      "n":   "1abc9xgoebGcP...",
      "e":   "AQAB"
    }
  ]
}

The five problems you need to design for

Problem 1 — Fetching on every request

Without caching, every JWT validation is an HTTP call to your auth provider. At 5,000 requests/second, that's 5,000 JWKS fetches per second — instant DDoS of your own auth service. The gateway must cache the full JWKS response in memory. Standard TTL: 5 minutes, with a background refresh before expiry so the cache never goes cold.

Problem 2 — The kid miss: when a token's key isn't in cache

This is the one that breaks naive implementations. Your auth provider rotates keys. It adds key-2024-02 to the JWKS endpoint. Immediately it starts issuing tokens signed with the new key. Your gateway's cache still has only the old key. A request arrives with a token whose kid is key-2024-02. The gateway doesn't find it in cache. What do you do?

Wrong answer: Return 401. You've just broken all logins for up to 5 minutes (your cache TTL) until the cache refreshes.

Right answer: On a kid miss, fetch the JWKS endpoint once, update the cache, retry validation. If the key still isn't there after the fresh fetch, return 401. This is the kid miss dance — it handles rotation transparently while still rejecting genuinely invalid tokens.

Problem 3 — The rotation overlap window

When a key is rotated, there's an overlap window where both the old key and the new key must be valid. Users who authenticated before the rotation have tokens signed with the old key — those tokens can be valid for hours. The auth provider must keep the old key in the JWKS response until all tokens signed with it have expired. Your gateway must cache and support all keys in the JWKS response, not just the newest one. This is why the JWKS is an array — multiple keys coexist during rotation.

Problem 4 — The JWKS endpoint going down

Your auth service goes down. Your gateway's JWKS cache expires. Now every JWT validation fails. Every API call returns 401. Your entire product is broken, even though the user's token is perfectly valid. The fix: serve stale JWKS on fetch failure. Keep the last successful response in cache indefinitely as a fallback. Log a loud alert on stale-serve. Return 401 only if you've never successfully fetched the JWKS at all.

Problem 5 — The thundering herd on cache expiry

At the moment your in-memory JWKS cache expires, all concurrent requests try to fetch the JWKS simultaneously. One network call becomes hundreds. The fix: use a mutex/lock around the refresh. Only one goroutine/thread fetches — others wait. Or proactively refresh in the background before expiry (background refresh pattern).

Production JWKS cache — kid miss dance, stale fallback, mutex refreshTYPESCRIPT
import { createPublicKey, createVerify } from 'crypto';
import jwkToPem from 'jwk-to-pem';

interface JWK { kid: string; kty: string; n: string; e: string; alg: string; }

class JWKSCache {
  private keys      = new Map<string, string>(); // kid → PEM
  private fetchedAt = 0;
  private staleFallback = new Map<string, string>();
  private fetching  = false;
  private waiters: Array<() => void> = [];
  private TTL_MS    = 5 * 60 * 1000; // 5 minutes

  constructor(private jwksUri: string) {
    // Background refresh: fetch 30s before expiry, prevents cold cache
    setInterval(() => {
      if (Date.now() - this.fetchedAt > (this.TTL_MS - 30_000)) {
        this.refresh().catch(err =>
          logger.warn({ err }, 'JWKS background refresh failed')
        );
      }
    }, 10_000);
  }

  private async refresh(): Promise<void> {
    // Mutex: only one fetch at a time
    if (this.fetching) {
      await new Promise<void>(r => this.waiters.push(r));
      return;
    }
    this.fetching = true;
    try {
      const res  = await fetch(this.jwksUri, { signal: AbortSignal.timeout(5000) });
      const json = await res.json();
      const fresh = new Map<string, string>();
      for (const key of json.keys as JWK[]) {
        fresh.set(key.kid, jwkToPem(key)); // convert JWK → PEM once
      }
      this.keys = fresh;
      this.staleFallback = fresh;  // always update stale copy on success
      this.fetchedAt = Date.now();
      logger.info({ keyCount: fresh.size }, 'JWKS refreshed');
    } catch (err) {
      // JWKS endpoint down — serve stale rather than breaking auth
      if (this.staleFallback.size > 0) {
        this.keys = this.staleFallback;
        logger.error({ err }, 'JWKS fetch failed — serving stale keys. ALERT!');
      } else {
        throw err; // first-ever fetch failed — cannot validate anything
      }
    } finally {
      this.fetching = false;
      this.waiters.forEach(r => r());
      this.waiters = [];
    }
  }

  async getKey(kid: string): Promise<string> {
    // Cold or expired cache: refresh first
    if (this.keys.size === 0 || Date.now() - this.fetchedAt > this.TTL_MS) {
      await this.refresh();
    }
    // kid found in cache — fast path
    if (this.keys.has(kid)) return this.keys.get(kid)!;

    // kid NOT in cache — could be a new key after rotation
    // Fetch once and retry. If still missing: the token is invalid.
    logger.info({ kid }, 'kid not in cache — fetching JWKS (key rotation?)');
    await this.refresh();
    const pem = this.keys.get(kid);
    if (!pem) throw new Error(`Unknown kid: \${kid}`); // → 401
    return pem;
  }
}

// Gateway middleware using the cache
const jwksCache = new JWKSCache('https://auth.example.com/.well-known/jwks.json');

export async function jwtMiddleware(req: Request, res: Response, next: NextFunction) {
  const token = req.headers.authorization?.slice(7); // strip "Bearer "
  if (!token) return res.status(401).json({ error: { code: 'MISSING_TOKEN' } });

  try {
    // 1. Decode header without verifying (to get kid)
    const headerB64 = token.split('.')[0];
    const header   = JSON.parse(Buffer.from(headerB64, 'base64url').toString());
    if (!header.kid) throw new Error('No kid in JWT header');

    // 2. Get public key (cached, with rotation handling)
    const pem = await jwksCache.getKey(header.kid);

    // 3. Verify signature + claims
    const payload = verifyJWT(token, pem, {
      algorithms: ['RS256'],
      issuer:     'https://auth.example.com',
      audience:   'api.example.com',
    });

    // 4. Strip token, inject identity headers downstream
    delete req.headers.authorization;
    req.headers['x-user-id']    = payload.sub;
    req.headers['x-user-role']  = payload.role;
    req.headers['x-tenant-id']  = payload.tenant_id;
    req.headers['x-token-exp']  = String(payload.exp); // useful for audit logs
    next();
  } catch (err: any) {
    const code = err.message.includes('expired') ? 'TOKEN_EXPIRED'
                : err.message.includes('kid')     ? 'INVALID_TOKEN'
                : 'INVALID_TOKEN';
    res.status(401).json({ error: { code, request_id: req.id } });
  }
}

Key rotation procedure

The correct key rotation sequence — every step matters. Skipping step 2 will break users.

  1. Generate the new key pair at your auth provider.
  2. Publish the new public key to JWKS — add it to the array alongside the old key. Do NOT remove the old key yet. Both keys are now in the JWKS.
  3. Wait for gateway JWKS caches to refresh — at least one full TTL (5 minutes). Every gateway instance now knows about the new key.
  4. Switch the auth provider to sign new tokens with the new key. Tokens issued from this point forward use the new key. Old tokens (still valid) use the old key. Both validate correctly because both keys are still in JWKS.
  5. Wait for all old tokens to expire. Your token TTL determines this — if access tokens expire in 15 minutes, wait 15 minutes.
  6. Remove the old key from JWKS. Only now is it safe — no valid tokens are signed with it anymore.
🔴
The rotation mistake that breaks prod

The most common mistake: rotating the key and removing the old one simultaneously. All in-flight tokens (valid for another 14 minutes) immediately start failing. Users get 401. Every active session is broken. The fix is always step 2 + waiting — the overlap window is not optional.

Verify in production
✓JWKS cache hit rate metric — should be >99.9%. Any miss indicates a rotation event or misconfiguration.
✓kid_miss_count metric — alert if >0 outside of known rotation windows.
✓JWKS fetch latency P99 — should be <200ms. Spikes indicate auth service degradation.
✓Stale-serve alert — if your gateway serves stale JWKS, you need to know immediately. The auth service is down.
✓Both old and new keys in JWKS during rotation — verify with curl https://auth.example.com/.well-known/jwks.json | jq '.keys | length'

Kong plugin phases & rate limiting internals

Kong's plugin system has 7 execution phases. Most engineers only think about what a plugin does — not which phase it runs in, which determines what data it has access to and what it can modify. Getting phase order wrong means your auth runs after your rate limiter, your logs don't include user IDs, or your circuit breaker fires before you've set the correlation ID.

The 7 plugin phases

Every Kong plugin can implement one or more phases. They run in this order on every request/response cycle:

REQUEST
init_worker
rewrite
access
header_filter
body_filter
log
RESPONSE
header_filter
body_filter
log
  • init_worker: Runs once when the worker process starts. Used to initialise shared state, timers, background jobs. Not per-request.
  • rewrite: Runs before routing. No upstream has been selected yet. Used to rewrite URLs, set variables, route based on headers before Kong decides which service to call.
  • access: The main request phase. Upstream has been selected. This is where auth, rate limiting, and request transformation happen. Most plugins live here. Can terminate the request (return 401, 429, etc.) before it reaches the upstream.
  • header_filter: Runs after the upstream responds. Can read and modify response headers. Cannot modify the body. Used to add X-Request-Id to responses, remove internal headers, add CORS headers.
  • body_filter: Can read and modify the response body. Called potentially multiple times for streaming responses (once per chunk). Expensive — avoid unless necessary.
  • log: Runs after the response is sent to the client. Cannot modify anything. Used for logging, metrics emission, audit trail writing. Has access to the full request/response including timing data.
💡
The phase ordering implication

Auth must run in access phase before rate limiting — otherwise unauthenticated requests consume rate limit quota for legitimate users. Correlation ID generation must happen in rewrite or early access so all subsequent plugins (including log) can reference it. Response logging must happen in log phase — never header_filter, where the response hasn't been fully sent yet.

Kong rate limiting: fixed window vs sliding window

The built-in rate-limiting plugin uses a fixed window counter. It divides time into 1-minute (or 1-hour) buckets and counts requests per bucket. This has a known vulnerability: a client can make 100 requests at 11:59:59 and 100 more at 12:00:01 — 200 requests in 2 seconds, with both batches within their respective windows.

The rate-limiting-advanced plugin (Kong Enterprise or community port) uses a sliding window algorithm, which prevents this burst. It maintains a rolling window that always looks back exactly N seconds from now, regardless of clock boundaries.

Rate limiting across Kong nodes: local vs Redis

Kong typically runs as multiple instances (horizontal scaling). Each instance has its own memory. If you use the local strategy for rate limiting, each instance counts independently. A client with 4 connections (one per Kong instance) gets 4× the configured limit. You must use the redis strategy to share state across nodes.

Kong rate-limiting-advanced — sliding window, Redis, consumer-level + service-levelYAML
# Global rate limit: applies to all traffic through the gateway
plugins:
  - name: rate-limiting-advanced
    config:
      limit:             [1000]        # 1000 requests...
      window_size:       [60]          # ...per 60 seconds (sliding window)
      window_type:       sliding       # sliding (not fixed)
      identifier:        consumer      # key by: consumer | ip | header
      sync_rate:         -1            # -1 = sync every request (accurate but slower)
                                       # 0.25 = sync every 250ms (faster, slight over-count)
      strategy:          redis         # shared state across all Kong nodes
      redis:
        host:            redis.internal
        port:            6379
        password:        ${REDIS_PASSWORD}
        database:        0
        timeout:         2000          # 2s timeout — fail open if Redis is down
      hide_client_headers: false       # expose X-RateLimit-* headers
      error_code:        429
      error_message:     Rate limit exceeded. See Retry-After header.

---
# Consumer-level override: premium plan gets 10x limit
# Applied to a specific consumer, overrides the global plugin
plugins:
  - name: rate-limiting-advanced
    consumer: premium-api-key-consumer
    config:
      limit:       [10000]
      window_size: [60]
      window_type: sliding

---

# /v1/reports has a separate, stricter limit regardless of consumer plan

# What happens when Redis is down?
# sync_rate: -1 → Kong fails CLOSED (allows traffic, logs error)
# This is the right default — a Redis outage shouldn't break your API
# Kong will log: "failed to get rate-limiting counter, allowing request"
Redis timeout causing 429 storms
If Redis is slow and Kong's Redis timeout is too low, every rate limit check times out. Kong can be configured to fail open (allow) or fail closed (reject). Default is fail open — correct. If you set it to fail closed, a Redis blip takes down your entire API.
→ Set redis.timeout: 2000ms. Never fail closed on Redis timeout.
sync_rate causing inaccurate limits at scale
With sync_rate: 0.25, each Kong node flushes its local counter to Redis every 250ms. Between syncs, the node uses its local count. Under high traffic, you can overshoot the limit by (local requests per 250ms × number of nodes). At 10,000 req/s across 4 nodes, you could overshoot by ~625 requests per sync interval.
→ Use sync_rate: -1 for payment/sensitive endpoints. Accept 0.25 for general API limits.
Consumer not identified — all traffic bucketed to IP
If auth fails (or is skipped for a route) and identifier is set to consumer, Kong falls back to IP limiting. All unauthenticated traffic from a shared IP (corporate NAT, mobile carrier) hits a single bucket.
→ Ensure auth plugin runs before rate limiting. Add explicit IP fallback limits for unauthenticated routes.
Verify Kong rate limiting in production
✓Every response has X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset headers.
✓429 responses include Retry-After header. Clients should wait, not immediately retry.
✓Redis key count scales linearly with active consumers — redis-cli KEYS "kong_rla:*" | wc -l
✓Plugin order in Kong Admin API: curl kong:8001/plugins?size=100 | jq '[.data[] | {name,priority}] | sort_by(.priority) | reverse'
✓Test boundary burst: send 100 requests at 11:59:55 and 100 at 12:00:05. With sliding window, second batch should be rate limited. With fixed window, it won't be.
Deep Dive 03

Cost-based rate limiting:
request units, not request counts

Counting requests treats a health check ping and a PDF report generation as identical. Stripe's API doesn't. Every API call costs "request units" — cheap endpoints cost 1, expensive endpoints cost 10 or 25. The customer's plan gives them a budget of units per minute, not a count of requests. This section covers how to implement it without adding per-request overhead.

The problem with counting requests

Consider three endpoints on the same API:

  • GET /health — 0ms, reads nothing, costs essentially zero
  • GET /orders/{id} — 5ms, one DB read
  • POST /reports/generate — 2,000ms, full table scan, PDF rendering, S3 upload

A customer with a 1,000 requests/minute limit can call GET /orders 1,000 times and barely stress your infrastructure. Or they can call POST /reports/generate 1,000 times and bring your DB to its knees. Same count, completely different infrastructure impact. Request counting doesn't capture this.

Designing the cost table

Assign costs during API design, not retroactively. The cost should approximate actual infrastructure cost — CPU, DB queries, external service calls, memory. It doesn't need to be precise; it needs to create correct relative ordering.

Cost table — config-driven, not hardcoded in handlersTYPESCRIPT
// costs.config.ts — owned by platform team, reviewed like code
export const ENDPOINT_COSTS: Record<string, number> = {
  // Pattern: "METHOD:path-pattern" → cost in request units

  // Reads — cheap
  'GET:/v1/orders/:id':           1,
  'GET:/v1/users/:id':            1,
  'GET:/v1/products':             1,
  'GET:/healthz':                  0,  // free — never counts against limit

  // Writes — medium
  'POST:/v1/orders':              5,   // creates order + inventory check + event
  'PATCH:/v1/orders/:id':         3,
  'POST:/v1/payments/charge':    10,  // calls payment processor

  // Expensive operations — high cost
  'POST:/v1/reports/generate':   25,  // full table scan + render + S3
  'POST:/v1/exports/csv':        20,  // large dataset export
  'POST:/v1/bulk/import':        15,  // per-row cost handled separately

  // Search — varies; use a base cost + result-count multiplier
  'GET:/v1/search':               3,

  // Default: unknown endpoints cost 1
  'DEFAULT':                       1,
};

// Per-plan budgets (request units per minute)
export const PLAN_BUDGETS: Record<string, number> = {
  free:       100,   // ~100 cheap reads, or 4 report generations
  starter:    1000,
  pro:        5000,
  enterprise: 50000,
};

function getCost(method: string, path: string): number {
  // Normalise path params: /v1/orders/123 → /v1/orders/:id
  const normPath = path.replace(/\/[0-9a-f-]{8,}/g, '/:id');
  const key = `\${method}:\${normPath}`;
  return ENDPOINT_COSTS[key] ?? ENDPOINT_COSTS['DEFAULT'];
}

The Redis Lua script: atomic cost deduction

The key constraint: the check-then-deduct must be atomic. If two requests arrive simultaneously, both check the budget, both see "budget available," both proceed, and you've allowed 2× the cost. A Redis Lua script runs atomically — it's the only correct way to implement this.

Cost-based rate limiter — atomic Lua, sliding window, per-plan budgetsTYPESCRIPT
// Sliding window cost deductor — replaces simple count increment
const COST_LIMIT_SCRIPT = `
local key      = KEYS[1]           -- e.g. "rl:cost:usr_123"
local cost     = tonumber(ARGV[1]) -- cost of this request (e.g. 25)
local budget   = tonumber(ARGV[2]) -- plan budget per window (e.g. 5000)
local window   = tonumber(ARGV[3]) -- window in seconds (e.g. 60)
local now      = tonumber(ARGV[4]) -- current Unix ms timestamp
local windowMs = window * 1000

-- Remove entries older than the window
redis.call('ZREMRANGEBYSCORE', key, 0, now - windowMs)

-- Sum the cost scores of all entries in window
local entries = redis.call('ZRANGEBYSCORE', key, now - windowMs, '+inf', 'WITHSCORES')
local used = 0
for i = 2, #entries, 2 do
  used = used + tonumber(entries[i])
end

-- Check if this request would exceed budget
if used + cost > budget then
  return {0, used, budget}  -- denied: {allowed=0, used, budget}
end

-- Add this request's cost to the sorted set
-- member = unique ID (now + random), score = cost
redis.call('ZADD', key, cost, now .. math.random())
redis.call('EXPIRE', key, window + 1)
return {1, used + cost, budget}  -- allowed: {allowed=1, new_used, budget}
`;

export async function costRateLimitMiddleware(
  req: Request, res: Response, next: NextFunction
) {
  const userId  = req.headers['x-user-id'] as string;
  const plan    = req.headers['x-user-plan'] as string || 'free';
  const cost    = getCost(req.method, req.path);
  const budget  = PLAN_BUDGETS[plan] ?? PLAN_BUDGETS.free;

  // Free endpoints skip rate limiting entirely
  if (cost === 0) return next();

  const [allowed, used, limit] = await redis.eval(
    COST_LIMIT_SCRIPT,
    1,
    `rl:cost:\${userId}`,
    cost, budget, 60, Date.now()
  ) as [number, number, number];

  // Always set headers — clients use these to self-throttle
  res.set({
    'X-RateLimit-Limit':          String(limit),
    'X-RateLimit-Remaining-Units': String(Math.max(0, limit - used)),
    'X-RateLimit-Cost':           String(cost),  // tells client what this call cost
    'X-RateLimit-Reset':          String(Math.floor(Date.now()/1000) + 60),
  });

  if (!allowed) {
    const retryAfter = 60; // worst case: full window
    res.set('Retry-After', String(retryAfter));
    return res.status(429).json({
      error: {
        code:    'RATE_LIMITED',
        message: `Request costs \${cost} units. Budget exhausted (\${used}/\${limit} used). Retry after \${retryAfter}s.`,
        details: { cost, used, limit, plan },
        request_id: req.id,
      }
    });
  }
  next();
}

Handling fanout: when one request triggers many downstream calls

Some endpoints fan out to multiple services. A POST /v1/dashboard/refresh might call the Orders service, the Analytics service, and the Notification service. The cost at the gateway reflects the single inbound request, but the actual infrastructure cost is 3× that.

Two approaches:

  • Gateway-only cost: Assign a higher fixed cost to fanout endpoints (e.g. 15 instead of 1). Simple, but requires keeping the cost table updated as the implementation changes.
  • Downstream cost reporting: Each downstream service reports its cost in a response header (X-Request-Units-Consumed: 5). The gateway aggregates and deducts from the budget after the response. More accurate but adds latency and requires downstream cooperation.

In practice: start with gateway-only fixed costs. Move to downstream reporting for the 2-3 most expensive endpoints only when you have evidence the fixed costs are significantly inaccurate.

⚠️
The per-row cost problem

Bulk import endpoints (POST /v1/bulk/import) have variable cost depending on payload size. A 10-row import and a 10,000-row import have the same fixed cost in the table. The solution: check the payload size at the gateway and calculate cost dynamically. cost = base_cost + Math.ceil(row_count / 100). Requires reading the request body at the gateway — adds latency. Apply only to bulk endpoints where the cost variance is significant.

Verify cost-based limiting in production
✓Every response includes X-RateLimit-Cost — clients can see what each call costs them.
✓Monitor cost distribution per endpoint — spikes in average cost indicate new expensive patterns.
✓Redis sorted set sum is O(N) — if any consumer has thousands of entries in the window, ZRANGEBYSCORE gets slow. Cap at realistic request volumes.
✓Test the exact Lua atomicity: send two POST /reports/generate (cost 25 each) simultaneously to a user with 30 units remaining. Only one should succeed.
✓Cost table is in version control, reviewed on every change. A wrong cost assignment is a business-impacting bug.
