Section 01

What belongs at the gateway — and what doesn't

The most important gateway design decision isn't which product to use — it's what responsibilities to give it. Get this wrong and you end up with one of two failure modes: an anemic gateway that does nothing (every service reimplements auth, rate limiting, and logging independently, inconsistently), or a fat gateway that accumulates business logic until changing it requires a full deployment pipeline and approval from five teams.

The mental model: the gateway owns cross-cutting concerns, not business logic. Cross-cutting means: the same thing applies to every request regardless of which service handles it. Business logic means: it depends on what the request is doing or who the user is in a domain-specific way.

Belongs at the gateway

  • Authentication: Verify that the token/API key is valid and not expired. Extract identity. Forward it downstream as trusted headers (X-User-Id, X-User-Role). Services trust the gateway-injected identity — they don't re-verify tokens.
  • Rate limiting: Per API key, per IP, per endpoint. The gateway sees all traffic; services see only their slice. Rate limiting in a service is ineffective — a single service doesn't know about the 10 other services the same client is hammering.
  • TLS termination: Decrypt HTTPS at the gateway. Forward plaintext (or re-encrypt with an internal cert) to services. Services don't need TLS complexity; the gateway handles certificate management centrally.
  • Request/response logging: Log every request with method, path, status, latency, and correlation ID at the gateway level. One place. Consistent schema. No service-specific logging gaps.
  • Correlation ID generation: Generate a unique request ID at ingress. Forward it via X-Request-Id header to every downstream service. Without this, distributed debugging is log archaeology.
  • Circuit breaking to upstreams: If a backend service is failing, the gateway should stop routing to it and return a fast error rather than letting client connections queue.

Does NOT belong at the gateway

  • Authorisation decisions: "Can this user edit this specific order?" requires domain knowledge — the order's owner, the user's role, the tenant's policy. That's business logic. The gateway can tell you who the user is; whether they're allowed is a service-level decision.
  • Request body transformation with business rules: Transforming a field name is fine. Calculating a discount, applying a pricing rule, or enriching data from a database query is not. The gateway becomes a silent business logic owner nobody notices until it breaks.
  • Orchestration: Calling service A, then service B, then combining the responses — that's an aggregation layer (or BFF), not a gateway. A gateway routes; it doesn't compose.

Section 02

The BFF pattern: one gateway per client, not one gateway for all

BFF (Backend for Frontend) is one of the most impactful architectural patterns for multi-client systems, and one of the least implemented until the team is already suffering from its absence.

The problem it solves: a mobile app, a web dashboard, and a third-party integration all calling the same generic API. The mobile app makes 6 API calls to render one screen (user profile, orders, notifications, recommendations, settings, activity feed). Each call returns far more data than the mobile client needs, because the API was designed to be general. The web dashboard makes 2 calls but needs deeper data. The third-party integration needs a completely different shape.

The generic API ends up trying to serve all three, satisfying none of them well: too much data for mobile (bandwidth waste), not enough flexibility for the web dashboard, wrong shape for third parties. Versioning becomes a nightmare because a change for the mobile client potentially breaks the others.

The BFF solution

Create a separate gateway layer per client type: a Mobile BFF, a Web BFF, and a Partner API gateway. Each BFF is owned by the team that builds the corresponding client. The Mobile BFF knows exactly what the mobile app needs — it calls 3 backend services, aggregates the data, strips fields the app doesn't use, and returns a single response. One network round-trip instead of six. The response is exactly the shape the mobile app expects, with no irrelevant fields.

Each BFF independently evolves with its client. The Mobile BFF can change its response shape without touching the Web BFF. The backend services remain generic — they don't need to know which client is calling.

⚠️
BFF is not a microservice proliferation excuse

A BFF per feature, per team, or per minor client variation becomes its own maintenance burden. Start with one BFF per meaningfully different client type (mobile vs web vs external partners). That's typically 2-3 BFFs. If you find yourself creating a BFF for "mobile iOS" vs "mobile Android" when both need the same data — you've gone too far.

Section 03

Auth at the gateway: validate once, trust downstream

Authentication is the single clearest case for gateway-level handling. The alternative — every service independently validates every token — is slow, inconsistent, and creates a surface area for vulnerabilities. If Service A has a bug in its token validation that Service B correctly fixes, you have split authentication behaviour depending on which service a request hits.

The pattern

The gateway receives the request with an Authorization: Bearer <token> header. It validates the JWT against a JWKS (JSON Web Key Set) endpoint — cryptographic signature verification, expiry check, issuer check. If validation fails, the gateway returns 401 Unauthorized immediately, and the request never reaches a backend service.

If validation passes, the gateway removes the Authorization header and injects trusted identity headers: X-User-Id: usr_123, X-User-Role: admin, X-Tenant-Id: tenant_abc. Backend services read these headers directly — no JWT library, no crypto, no network call to an auth service. They trust the gateway.

JWKS caching: the performance detail that matters

JWT validation requires fetching the public key from the JWKS endpoint to verify the signature. If you fetch the JWKS on every request, you're making an HTTP call to your auth service for every API call — which adds latency and makes your auth service a bottleneck. Cache the JWKS. Refresh the cache on a schedule (every 5 minutes) and on key ID mismatch (when a token has a kid not in cache, fetch once and retry). This is how Kong, nginx-lua, and AWS API Gateway all handle it.

The authorisation boundary

Authentication (who are you?) belongs at the gateway. Authorisation (can you do this?) mostly belongs in the service. The gateway can do coarse-grained authorisation — "only users with role admin can reach /v1/admin/*" — because that's a routing rule, not a business rule. Fine-grained authorisation — "can user 456 edit order 789?" — requires the service to check ownership in its database.

# Kong declarative config (deck format)
plugins:
  - name: jwt

      key_claim_name: kid          # JWT header field with key ID

      maximum_expiration: 86400    # reject tokens valid > 24h
      secret_is_base64: false

          - X-User-Role:$(jwt.claims.role)

Section 04

Rate limiting at the gateway: multi-dimensional and coordinated

Rate limiting in a service is broken by design. If you have 5 instances of a service and limit to 100 requests per minute per instance, an attacker with 5 connections can make 500 requests per minute — one per instance. The limit only works if there's a single enforcer that sees all traffic. That's the gateway.

Multi-dimensional limiting

Production rate limiting is never just one dimension. You need at least three, enforced simultaneously:

  • Per API key: The primary limit for authenticated clients. "Your plan allows 1,000 requests/minute." This prevents a single customer from monopolising infrastructure.
  • Per IP address: Essential for unauthenticated endpoints (login, token refresh). A single IP making 1,000 login attempts per minute is a credential stuffing attack. IP rate limiting catches this even before authentication runs.
  • Per endpoint: Some endpoints are more expensive than others. POST /reports/generate hitting the database hard should have a much lower limit than GET /users/{id}. Endpoint-level limits prevent a single expensive endpoint from exhausting capacity even within a customer's global limit.

Distributed rate limiting with Redis

The gateway runs as multiple instances (for HA). Each instance must share rate limit state — otherwise each instance has its own counter and a customer with 3 gateway connections gets 3× their limit. Redis with an atomic Lua script is the standard solution: a single Lua script atomically checks and increments the counter, returns the current count and the TTL, and the gateway either passes or rejects the request based on the result.

Multi-dimensional rate limiting middleware — Node.js / Express gateway TYPESCRIPT 
import redis from './redis';

const LIMITS = {
  apiKey:   { window: 60,   max: 1000 },  // 1000 req/min per key
  ip:       { window: 60,   max: 200  },  // 200 req/min per IP
  endpoint: { window: 60,   max: 50   },  // expensive endpoints
};

// Atomic sliding window in Redis Lua
const SLIDING_WINDOW = `
local key    = KEYS[1]
local window = tonumber(ARGV[1])
local limit  = tonumber(ARGV[2])
local now    = tonumber(ARGV[3])
redis.call('ZREMRANGEBYSCORE', key, 0, now - window * 1000)
local count = redis.call('ZCARD', key)
if count >= limit then return {0, count, limit} end
redis.call('ZADD', key, now, now .. math.random())
redis.call('EXPIRE', key, window)
return {1, count + 1, limit}
`;

async function checkLimit(key: string, window: number, max: number) {
  const [allowed, count, limit] = await redis.eval(
    SLIDING_WINDOW, 1, key, window, max, Date.now()
  ) as [number, number, number];
  return { allowed: allowed === 1, count, limit };
}

export async function rateLimitMiddleware(req: Request, res: Response, next: NextFunction) {
  const apiKey   = req.headers['x-api-key'] as string;
  const ip       = req.ip;
  const endpoint = req.method + ':' + req.path;

  // Check all dimensions in parallel
  const [keyResult, ipResult] = await Promise.all([
    apiKey ? checkLimit(`rl:key:\${apiKey}`, LIMITS.apiKey.window, LIMITS.apiKey.max) : null,
    checkLimit(`rl:ip:\${ip}`, LIMITS.ip.window, LIMITS.ip.max),
  ]);

  const blocked = [keyResult, ipResult].find(r => r && !r.allowed);
  if (blocked) {
    res.set({
      'X-RateLimit-Limit':     String(blocked.limit),
      'X-RateLimit-Remaining':  '0',
      'Retry-After':            '60',
    });
    return res.status(429).json({
      error: { code: 'RATE_LIMITED', message: 'Too many requests', request_id: req.id }
    });
  }

  // Set informational headers on every response
  if (keyResult) {
    res.set({
      'X-RateLimit-Limit':    String(keyResult.limit),
      'X-RateLimit-Remaining': String(keyResult.limit - keyResult.count),
    });
  }
  next();
}

Section 05

Request routing: the gateway's primary job

Routing is what the gateway is fundamentally for — taking an incoming request and deciding which upstream service should handle it. Everything else (auth, rate limiting, logging) is middleware around this core function.

Path-based routing

The simplest and most common pattern. Route based on the URL path prefix:

  • /v1/orders/* → Order Service
  • /v1/users/* → User Service
  • /v1/payments/* → Payment Service
  • /v1/admin/* → Admin Service (with additional auth check)
  • /static/* → CDN origin / S3

Path-based routing is easy to reason about, easy to debug (just look at the URL), and easy to configure. It's also easy to get wrong — long prefix lists with overlapping patterns (does /v1/orders/refunds match /v1/orders/* or /v1/refunds/*?) are a common source of production routing bugs.

Header-based routing

Route based on request headers. The most powerful use case: canary deployments. Route 5% of traffic to the new version by header, cookie, or percentage:

  • X-Canary: true header → new version
  • 5% by random hash of user ID → new version
  • User in beta cohort (from JWT claim) → new version

This is how Stripe, Netflix, and Google deploy: new code is running in production for a controlled slice of traffic before it reaches everyone. The gateway is the enforcer of the canary split.

Timeout hierarchy

The gateway's upstream timeout must be shorter than the client's timeout. If a client times out at 30 seconds and the gateway times out at 30 seconds, the client gives up at the same time the gateway does — no opportunity to return a meaningful error to the client. Set the gateway timeout to 25 seconds — the gateway can return a 504 Gateway Timeout to the client 5 seconds before the client's own timeout fires, giving the client useful information and time to retry or display an error.

nginx API gateway — path routing + canary + timeout hierarchy NGINX 
# Upstream definitions with connection pooling
upstream orders_v1   { least_conn; server orders-v1:3000;   keepalive 32; }
upstream orders_v2   { least_conn; server orders-v2:3000;   keepalive 32; }
upstream users_svc   { least_conn; server users:3001;       keepalive 32; }
upstream payments    { least_conn; server payments:3002;    keepalive 32; }

server {
    listen 443 ssl http2;

    # Correlation ID: generate if not provided by client
    set $req_id $http_x_request_id;
    if ($req_id = "") { set $req_id $request_id; }
    add_header X-Request-Id $req_id always;

    # Canary routing: 5% to v2 by cookie/header, rest to v1
    split_clients "${remote_addr}${request_uri}" $orders_upstream {
        5%   orders_v2;
        *    orders_v1;
    }

    # Path-based routing
    location /v1/orders/ {
        proxy_pass http://$orders_upstream;
        proxy_read_timeout 25s;      # shorter than client's 30s
        proxy_connect_timeout 3s;
        proxy_set_header X-Request-Id  $req_id;
        proxy_set_header X-User-Id     $http_x_user_id;     # injected by auth
        proxy_set_header X-Forwarded-For $remote_addr;
        proxy_hide_header Authorization;                     # strip raw token
    }

    location /v1/users/ {
        proxy_pass http://users_svc;
        proxy_read_timeout 25s;
        proxy_set_header X-Request-Id $req_id;
    }

    location /v1/payments/ {
        # Extra: payments get additional auth header check
        if ($http_x_user_role != "admin") {
            return 403 '{"error":{"code":"FORBIDDEN"}}';
        }
        proxy_pass http://payments;
        proxy_read_timeout 25s;
    }

    # Fallback: unknown routes
    location / {
        return 404 '{"error":{"code":"NOT_FOUND","message":"Unknown API path"}}';
    }
}

Section 06

  
Circuit breaking at the gateway: fail fast before you cascade


  
Without circuit breaking, a slow or failing backend service causes a cascade: clients retry, the gateway queues requests, queued requests pile up, the gateway exhausts its connection pool, other routes start being affected because the gateway is starved for resources. One broken service takes down the whole gateway.


  
A circuit breaker at the gateway level prevents this. When an upstream service's error rate or response time crosses a threshold, the circuit "opens" — the gateway immediately returns an error response for that route without connecting to the broken service. The connection pool is freed. Other routes continue working normally.


  
The three states

  
    
    
  • Closed (normal): Requests flow through. The circuit breaker tracks error rate and response time. Below threshold → stays closed.
    • 
    
  • Open (failing fast): Error rate exceeded threshold. All requests immediately return an error — no connection to the backend attempted. After a configured timeout (e.g., 30 seconds), transitions to half-open.
    • 
    
  • Half-open (probing): Allows a small number of test requests through. If they succeed, the circuit closes. If they fail, the circuit reopens for another timeout period.
    • 
  


  
Fallback strategies

  
An open circuit needs a fallback response. Options in order of preference:

  
    
    
  1. Cached response: Return the last successful response from Redis cache. Stale data is better than an error for most read operations.
    2. 
    
  2. Degraded response: Return a partial response that omits the data from the failing service. A product page without reviews is better than a 500.
    3. 
    
  3. Queue and process asynchronously: For write operations that can be deferred, accept the request and queue it for when the service recovers.
    4. 
    
  4. Fast error with Retry-After: Return a 503 with a Retry-After header telling the client when to retry. At least the client can make an informed decision.
    5. 
  


  

    

      Circuit breaker at the gateway layer — Node.js
      TYPESCRIPT
    

    
import { circuitBreaker, handleAll, ConsecutiveBreaker, ExponentialBackoff, retry, wrap } from 'cockatiel';

// One circuit breaker per upstream service
const ordersCB = circuitBreaker(handleAll, {
  halfOpenAfter: 15_000,              // probe after 15s
  breaker: new ConsecutiveBreaker(5),  // open after 5 consecutive failures
});

// Log every state transition — these are incidents
ordersCB.onStateChange(state => {
  logger.error({ state, service: 'orders' }, 'Circuit breaker state change');
  metrics.gauge('gateway.circuit.state', state === 'open' ? 1 : 0, { service: 'orders' });
});

async function proxyToOrders(req: Request, res: Response) {
  try {
    const result = await ordersCB.execute(async () => {
      return await ordersPool.request({
        path: req.path, method: req.method,
        headers: { 'x-request-id': req.id, 'x-user-id': req.userId },
        body: req.body, headersTimeout: 5_000, bodyTimeout: 25_000,
      });
    });
    res.status(result.statusCode).json(await result.body.json());
  } catch (err: any) {
    if (err.type === 'BrokenCircuitError') {
      // Circuit is open — serve stale cache or degraded response
      const cached = await redis.get(`cache:orders:\${req.params.id}`);
      if (cached) {
        res.set('X-Served-From-Cache', 'true');
        return res.json(JSON.parse(cached));
      }
      res.set('Retry-After', '15');
      return res.status(503).json({
        error: { code: 'SERVICE_UNAVAILABLE', message: 'Orders service temporarily unavailable', request_id: req.id }
      });
    }
    throw err;
  }
}

Section 07

  
Correlation IDs: the debugging feature worth 10 observability tools


  
Here is the scenario that plays out in every microservices system without correlation IDs. A user reports an error at 2:47pm. You search the Order Service logs — 50,000 log lines during that minute. You find 12 errors. Which one is theirs? You search the Payment Service — 80,000 lines. The error you're looking for called both services, but you have no way to know which Payment Service log line corresponds to which Order Service request. You spend two hours manually correlating timestamps and user IDs across logs from four services.


  
With correlation IDs: the gateway generates a UUID at request ingress (req_a1b2c3d4). Every service logs it. Every downstream call carries it. The user gives you their request ID from the error response. You search one field across all logs. You find every log line from every service involved in that exact request in under 30 seconds.


  
The implementation

  
The gateway generates an ID if the client didn't provide one (X-Request-Id). It forwards the ID to all upstream services via header. Services include the ID in every log line. Services forward the ID when they make downstream calls. The ID is returned in the response headers so clients (and users) can report it.


  
What to propagate beyond the correlation ID

  
The correlation ID is the minimum. In a well-instrumented system you also propagate:

  
    
    
  • Trace context: W3C traceparent header for distributed tracing (OpenTelemetry). The trace ID is a superset of the correlation ID — it includes span relationships between services.
    • 
    
  • User context: X-User-Id, X-Tenant-Id — injected by the gateway after auth. Services don't need to parse tokens; they read these headers.
    • 
    
  • Baggage: OpenTelemetry Baggage header can carry arbitrary key-value pairs (user tier, feature flags, A/B cohort) through the entire call graph without each service needing to know how to derive them.
    • 
  


  

    

      Correlation ID propagation — gateway + downstream service pattern
      TYPESCRIPT
    

    
// Gateway: generate and propagate
import { randomUUID } from 'crypto';

app.use((req, res, next) => {
  // Accept client-provided ID or generate one
  req.id = (req.headers['x-request-id'] as string) || `req_\${randomUUID().replace(/-/g,'').slice(0,16)}`;

  // Return in response so users/clients can report it
  res.set('X-Request-Id', req.id);

  // Forward to all upstream calls
  res.locals.upstreamHeaders = {
    'x-request-id': req.id,
    'x-user-id':    req.userId,     // set by auth middleware
    'x-tenant-id':  req.tenantId,
    'traceparent':  req.headers['traceparent'] || generateTraceparent(req.id),
  };
  next();
});

// Every downstream service: bind to request context
app.use((req, res, next) => {
  // Create child logger with request ID bound
  req.log = logger.child({
    request_id: req.headers['x-request-id'],
    user_id:    req.headers['x-user-id'],
    tenant_id:  req.headers['x-tenant-id'],
  });

  // Every log line in this request handler now includes request_id automatically
  req.log.info({ method: req.method, path: req.path }, 'request received');
  next();
});

// When making calls to OTHER services: propagate the ID
async function callInventoryService(orderId: string, requestId: string) {
  return inventoryPool.request({
    path: `/v1/inventory/\${orderId}`,
    headers: { 'x-request-id': requestId }, // same ID, forms the trace chain
  });
}

Section 08

  
Gateway vs service mesh: north-south vs east-west


  
This is a comparison that confuses most engineers because both tools involve proxies routing traffic between services. The distinction is simple once you understand which traffic each handles:


  
    
    
  • API Gateway: handles north-south traffic — traffic coming in from outside your system (browsers, mobile apps, third-party clients) going to your internal services. The entry point.
    • 
    
  • Service mesh: handles east-west traffic — service-to-service calls within your system. The internal network fabric.
    • 
  


  
They solve different problems. A gateway solves "how do external clients reach internal services securely and efficiently?" A mesh solves "how do internal services communicate with each other reliably, with observability and mutual TLS?"


  
When you need each

  
You always need a gateway. Even a single-service architecture needs something handling TLS termination, rate limiting, and auth at the edge.

  
You need a service mesh when: you have many services calling each other (east-west traffic is significant), you need mTLS between all internal services (compliance, zero-trust), you need per-request observability on internal traffic (latency, error rates per service-to-service call), or you need fine-grained traffic control between internal services (canary deployments of internal APIs).

  
The mistake: trying to use a gateway to manage east-west traffic. Routing all internal service calls through the gateway means every internal call goes out-of-process to the gateway and back — adding latency and creating a bottleneck. East-west traffic should stay on the internal network, handled by the mesh's sidecar proxies (Envoy in Istio, Linkerd-proxy in Linkerd).

Section 09

  
Production reality: gateway failure modes


  
The gateway as a single point of failure

  
Every request goes through the gateway. When it goes down, everything goes down. This is obvious — but the subtler version is the gateway as a performance bottleneck. A misconfigured gateway with too few worker processes, a memory leak in a plugin, or a Redis connection pool exhausted by rate limiting queries will degrade all traffic, not just the affected routes.

  
Run gateway instances in active-active pairs behind a network load balancer. Monitor gateway CPU, memory, and connection counts as aggressively as you monitor your application services.


  
Plugin order matters and is almost never documented

  
In Kong, nginx-lua, and most gateway plugin systems, middleware runs in a defined order. Auth before rate limiting, or rate limiting before auth? If you rate limit before auth, unauthenticated requests consume rate limit quota. If you auth before rate limiting, every request (including those that will be rejected for bad tokens) hits your JWT validation. The right order: auth first (fast reject of invalid tokens), then rate limiting per authenticated identity.


  
The timeout misconfiguration that causes 200 cascades

  
Client sets a 30s timeout. Gateway sets a 30s proxy timeout. Backend service takes 35 seconds. The gateway closes the connection to the backend at 30s and returns a 504 to the client. But the backend service doesn't know the gateway closed the connection — it keeps processing the request for another 5 seconds, then writes the response to a closed socket. The response is discarded. The work was done. The user saw an error. This is why the gateway's timeout must be 5+ seconds shorter than the client's timeout — and why your backend service must handle request cancellation (req.on('close') in Node.js) to abort work when the client is gone.


  
Fat gateway drift

  
Six months after launch: "let's just put this pricing logic in the gateway, it has the product context already." A year later: the gateway validates tokens, rate limits, routes, applies discounts, checks feature flags, enriches requests with user data, applies business rules to responses. It's now the most complex service you have, owned by no team, understood by no one. Any change requires days of testing because nobody knows what else depends on it.

  
The preventive measure: document what the gateway is responsible for on day one. Any PR that adds logic beyond cross-cutting concerns requires explicit sign-off from the team lead. Boring to write — useful when the drift starts.







  
Section 10

  
The things nobody tells you


  
Your gateway timeout is part of your SLA

  
If your gateway has a 30-second timeout and your 99th percentile backend response is 28 seconds, you have almost no margin. A traffic spike that pushes P99 to 32 seconds immediately causes gateway timeouts for 1% of requests — which you see as 504 errors in your gateway metrics, not as application errors. Track gateway-added latency separately from backend latency, and set your gateway timeout to be meaningfully shorter than your SLA (not just 1 second shorter).


  
Validate JWTs at the gateway, but cache the result

  
JWKS endpoint fetches are a network call. For a high-traffic gateway, doing one per request is a lot of load on your auth provider. Cache the JWKS (it changes rarely), and cache the validated claims per token (using the token's expiry as the cache TTL). A valid JWT for user 123 with role "admin" that expires in 900 seconds should only be validated once. Cache the result for min(900s, some_max). Subsequent requests with the same token hit the cache, not the JWKS endpoint.


  
The gateway sees your most sensitive data

  
Every request, every token, every API key, every user ID passes through the gateway. It's an extremely attractive target — not to attack, but to exfiltrate data from. Gateway access logs must never include full Authorization headers or API keys. Log the key prefix (first 8 characters) for debugging, not the full key. Apply the same data handling standards to gateway logs as you apply to your most sensitive database.


  
Rate limit by cost, not just count

  
Treating a GET /user and a POST /reports/generate as equally costing "1 request" against a rate limit is incorrect. The report generation hits the database, takes 2 seconds, and consumes 10× the resources of a user lookup. A sophisticated rate limit assigns costs: the cheap endpoint costs 1 token, the expensive endpoint costs 10. This is what Stripe's API does — their rate limit is measured in "request units," not requests.







  
Design decisions

  
Designing a gateway — decide in this order

  
These decisions compound. Getting #1 wrong makes #2-6 harder. Answer them before choosing a product.

  


    

      

      
1

      

        
What does this gateway own? Write it down before touching a config file.

        
List exactly which concerns belong here: auth, rate limiting, routing, TLS, logging, circuit breaking, correlation IDs. Write what doesn't belong just as explicitly. This list is your contract. The first time someone asks to add business logic to the gateway, you have something to point at. Without this list, fat gateway drift is inevitable — it's not a discipline problem, it's an entropy problem.
          
undocumented scope = business logic accumulation over time
        

      

    


    

      

      
2

      

        
Do different client types need different response shapes? If yes, consider BFF.

        
If your mobile app and web dashboard both hit the same generic API and mobile consistently makes 4-6 calls to render one screen — you need a BFF. One gateway per client type (mobile, web, partners). Each owned by the team shipping that client. If all your clients need the same data shape, a single generic gateway is fine. Don't add BFF complexity for the sake of it.
          
signal: mobile making 5+ calls per user action
        

      

    


    

      

      
3

      

        
Set up auth at the gateway — validate once, forward identity.

        
The gateway validates JWTs against a cached JWKS. On success: strip the Authorization header, inject X-User-Id, X-User-Role, X-Tenant-Id. Services trust these headers unconditionally. Fine-grained authorisation ("can this user edit this resource?") stays in the service. This division is hard to change later — decide it clearly now.
          
gateway = who you are. service = what you can do.
        

      

    


    

      

      
4

      

        
Configure rate limiting across all three dimensions before launch.

        
Per API key (your primary limit — tied to a plan), per IP (catches unauthenticated abuse — credential stuffing on login), per endpoint (expensive endpoints need separate, lower limits). All three running simultaneously in Redis. Add cost-based weighting for expensive endpoints from day one. Starting with a single flat limit and adding dimensions later means a flag day for your clients.
          
single-dimension limiting leaves attack vectors open
        

      

    


    

      

      
5

      

        
Set the gateway timeout shorter than every client's timeout.

        
Map all your clients and their configured timeouts. Set the gateway proxy timeout to the minimum client timeout minus 5 seconds. This ensures the gateway can always return a meaningful 504 before the client gives up. A gateway that times out at the same time as the client gives the client nothing useful. Also: the gateway timeout must be shorter than the upstream backend's own timeout, or the backend might win the race and the gateway returns a 200 on a request that didn't fully process.
          
gateway timeout = client timeout = no useful error signal
        

      

    


    

      
6

      

        
Generate correlation IDs at ingress. Make them visible in every error response.

        
Generate at the gateway edge. Log at every service. Forward through every downstream call. Return in every error response body (request_id: "req_abc123"). The first time a user reports an error and gives you that ID and you find the full trace in 30 seconds rather than 2 hours — that's the payoff. The first time a user gives you an error and you have nothing to look up with — that's the cost of skipping it.
          
30-second trace vs 2-hour log archaeology
