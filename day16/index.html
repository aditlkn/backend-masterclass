Section 01

Why caching — the numbers that motivate it

Cache performance is a function of two things: hit rate and the cost difference between a hit and a miss. A 90% cache hit rate with a 10ms miss cost and 0.2ms hit cost means: for 100 requests, 90 take 0.2ms and 10 take 10ms. Average latency: 1.18ms instead of 10ms. An 8.5× improvement from one architectural decision.

But the real motivation isn't latency — it's throughput. PostgreSQL with good indexes might handle 5,000 queries per second before CPU saturates. Redis handles 100,000+ operations per second on a single instance. If your database is the bottleneck at peak traffic, a cache in front of it can absorb 80% of reads and extend that ceiling by 5×. You defer buying more database capacity, defer sharding, defer read replicas — all from one Redis cluster.

The cache hierarchy in a typical backend

  • In-process memory (L1): Fastest possible. Microseconds. Zero network. The application's own heap. Limited by server memory and lost on restart. Used for config, feature flags, precomputed constants, very hot small datasets.
  • Distributed cache (L2): Redis or Memcached. Sub-millisecond. Survives restarts. Shared across all application instances. The standard cache layer for most backends.
  • CDN edge cache (L3): Geographically distributed. Caches at the network edge near the user. For static assets and cacheable API responses. Eliminates round-trips to your origin entirely.
  • Database query cache: Usually not a separate layer — PostgreSQL doesn't have a query result cache (it was removed). MySQL had one but it was removed in 8.0. The database buffer pool caches data pages in memory — different from a result cache.
⚠️
Cache is not a fix for bad database queries

A query that takes 2 seconds because it's doing a full table scan should be fixed with an index, not cached. Caching a slow query hides the problem until the cache expires, at which point your site takes a 2-second hit on every cache miss. Fix the underlying query first, then cache if throughput requires it. Cache is for queries that are already fast but need to run less often — not for slow queries that you're embarrassed about.

Section 02

The four cache patterns — and when each breaks

Cache-aside (lazy loading)

The application manages the cache explicitly. On a read: check the cache first. Hit → return immediately. Miss → read from database, write to cache with a TTL, return result. On a write: write to the database. The cache is left to expire naturally (or explicitly invalidated).

Cache-aside is the most common pattern and the most appropriate for most use cases. The cache only contains what has been requested — no wasted memory for data that's never accessed. The downside: the first request after a cache miss (or after the cache is flushed) goes to the database. In a cold start scenario — new deployment, Redis restart — every request is a cache miss, potentially overwhelming the database.

Read-through

The cache sits in front of the database and is responsible for loading its own data. When the application requests a key that isn't in the cache, the cache fetches it from the database, stores it, and returns it. The application always talks to the cache, never directly to the database for reads. Libraries like Spring's @Cacheable implement this pattern.

Similar to cache-aside but the cache layer is responsible for the DB read on a miss — the application code is simpler but you need a cache that supports this (not all do).

Write-through

Every write to the database also writes to the cache synchronously. The cache is always up to date. No stale reads. The downside: every write is 2 operations, and the cache holds data that may never be read again (you wrote it, but nobody reads that specific product page). Memory is wasted on data with low read frequency.

Write-behind (write-back)

Writes go to the cache first and asynchronously propagate to the database. Extremely fast writes — the application never waits for the database on a write. The risk: if the cache crashes before data is flushed to the database, you lose data. Used for: high-write scenarios where some data loss is acceptable (analytics counters, leaderboard scores), or with a durable write-ahead log to prevent data loss.

💡
Most applications need cache-aside + explicit invalidation

Write-through and write-behind add complexity that's only justified at extreme write rates. For most applications, cache-aside with a TTL is sufficient: reads are fast, stale data expires naturally, and the implementation is simple enough to reason about. Add explicit invalidation on writes (delete the cache key when the underlying data changes) to reduce staleness without the complexity of write-through.

Section 03

Cache stampede: the thundering herd

A cache stampede (thundering herd) happens when many requests simultaneously miss the cache for the same key — often because the TTL just expired — and all of them race to recompute the value and repopulate the cache. If that computation involves a database query, you've just fired N simultaneous database queries for the same data at exactly the same moment. For popular cache keys, N can be in the hundreds.

Three solutions

Mutex lock: When a cache miss occurs, acquire a distributed lock before going to the database. Only the process that acquires the lock goes to the database; others wait and then re-check the cache. When the lock holder returns from the database and populates the cache, the waiting processes find the value in the cache. Downside: latency for the waiting requests; lock holder failure can block everyone.

Probabilistic early recomputation (PER): Instead of waiting for the TTL to expire, start recomputing the value slightly before it expires. The decision to recompute is probabilistic — with higher probability as the expiry approaches. The value is still served fresh while the background recomputation runs. This eliminates the synchronisation problem entirely — there's no moment when the key is expired for everyone at once.

Stale-while-revalidate: When the TTL expires, serve the stale cached value immediately while triggering an async background refresh. Users always get a fast response; the cache is updated asynchronously. Works well when serving slightly stale data is acceptable. Used by CDNs, HTTP caches, and can be implemented in application code.

Cache stampede prevention — mutex lock patternTYPESCRIPT
async function getWithStampedeProtection<T>(
  key: string,
  fetch: () => Promise<T>,
  ttlSec: number
): Promise<T> {
  // 1. Try cache first
  const cached = await redis.get(key);
  if (cached) return JSON.parse(cached);

  // 2. Cache miss — acquire a lock before hitting DB
  const lockKey = `lock:\${key}`;
  const lockVal = crypto.randomUUID();
  const acquired = await redis.set(lockKey, lockVal, 'NX', 'EX', 10);

  if (acquired) {
    // 3. We hold the lock — fetch from DB and populate cache
    try {
      const value = await fetch();
      await redis.setex(key, ttlSec, JSON.stringify(value));
      return value;
    } finally {
      // Release lock (only if we still own it)
      const script = `if redis.call('get',KEYS[1])==ARGV[1] then return redis.call('del',KEYS[1]) else return 0 end`;
      await redis.eval(script, 1, lockKey, lockVal);
    }
  } else {
    // 4. Another process is fetching — wait and retry from cache
    await sleep(50);  // wait 50ms, then re-check cache
    const retried = await redis.get(key);
    if (retried) return JSON.parse(retried);
    // Lock holder might have failed — fall through to DB as backup
    return fetch();
  }
}

// Probabilistic early recomputation (XFetch algorithm)
async function getWithEarlyRecompute<T>(
  key: string, fetch: () => Promise<T>, ttlSec: number, beta = 1
): Promise<T> {
  const [value, expiry] = await redis.hmget(`meta:\${key}`, 'value', 'expiry');
  if (value) {
    const remainingTtl = parseFloat(expiry) - Date.now() / 1000;
    // Recompute early with increasing probability as TTL decreases
    // -beta * delta * log(random) < remaining_ttl → recompute
    const shouldRecompute = -beta * Math.log(Math.random()) > remainingTtl;
    if (!shouldRecompute) return JSON.parse(value);
  }
  // Recompute (either miss or early expiry triggered)
  const start = Date.now();
  const fresh = await fetch();
  const delta = (Date.now() - start) / 1000; // fetch time in seconds
  await redis.hset(`meta:\${key}`, 'value', JSON.stringify(fresh), 'expiry', Date.now()/1000 + ttlSec);
  await redis.expire(`meta:\${key}`, ttlSec + Math.ceil(delta * beta));
  return fresh;
}

Section 04

Cache invalidation: the hard problem

"There are only two hard problems in computer science: cache invalidation and naming things." The joke is accurate. Getting data into a cache is easy. Ensuring that cached data is removed or updated exactly when the underlying data changes — without being too aggressive (invalidating too often, defeating the purpose of caching) or too conservative (serving stale data for too long) — is where caching gets genuinely complex.

TTL-based expiry (passive invalidation)

Set a time-to-live on every cache entry. After the TTL, the entry expires and the next request goes to the database. Simple to implement. The downside: stale data for up to TTL seconds after an update. Acceptable when: the data changes infrequently relative to the TTL, or when slightly stale data doesn't cause problems (product descriptions, user profiles).

TTL tuning is a tradeoff: short TTL → more database load, fresher data. Long TTL → less database load, staler data. Most systems settle on 5-15 minutes for relatively stable data and 1-60 seconds for data that changes frequently.

Active invalidation (event-driven)

When data changes, explicitly delete or update the cache entry. This ensures consistency — the cache entry is invalid immediately after the update, not after the TTL. The downside: you need to know all the cache keys that reference the changed data — which can be complex for data that appears in multiple cache contexts.

The "delete on write" pattern is safer than "update on write": delete the cache key when the underlying data changes, let the next read repopulate it. Updating the cache on write requires that the write and cache-update are atomic (they're not by default), and a write-update failure can leave a stale entry in cache indefinitely.

Cache versioning

Embed a version number in the cache key. When you deploy a new version of the application that changes the data format, increment the version. Old cache keys (product:v1:123) become unreachable — they're not deleted, they just expire via TTL. New keys (product:v2:123) are populated by the new application code. This is effective for deployments that change the shape of cached objects.

Section 05

Eviction policies: what happens when cache is full

Redis has a fixed maximum memory (maxmemory). When the memory limit is reached and a new key needs to be written, Redis must evict an existing key. The eviction policy determines which key is removed. Choosing the wrong policy causes either data loss or cache thrashing.

PolicyWhat it evictsBest forRisk
noevictionNothing — returns errorPrimary data stores (not caches)Writes fail when full
allkeys-lruLeast recently used (any key)General-purpose cacheLow — keeps hot data
volatile-lruLeast recently used (TTL keys only)Mix of persistent + cached dataKeys without TTL are never evicted
allkeys-lfuLeast frequently used (any key)Skewed access patterns (Zipf)Low — keeps frequently accessed data
volatile-lfuLeast frequently used (TTL keys only)Selective caching with mixed dataAs above
allkeys-randomRandom keyUniform access patterns (rare)May evict hot data
volatile-ttlKey with soonest expiryWhen TTL reflects data importancePredictable but not always optimal

For a pure cache (all keys have TTLs, no persistent data): use allkeys-lru or allkeys-lfu. LRU evicts keys not accessed recently; LFU evicts keys accessed infrequently. LFU is generally better for skewed access patterns (some keys are much hotter than others) because a key accessed once and then never again won't accidentally stay alive just because it was accessed recently.

For a Redis instance used as both a cache and a data store: use volatile-lru — only keys with TTLs are eligible for eviction. Your persistent data (session stores, job queues) is never evicted. But this means you must set TTLs on all cache keys — a cache key without a TTL will never be evicted, potentially filling memory permanently.

🔴
noeviction on a cache causes write failures in production

The default Redis eviction policy is noeviction — when memory is full, writes return an error. If your cache is configured as noeviction and memory fills up, every SET operation fails. Application code that doesn't handle this error (most don't) will crash. Always configure maxmemory and a sensible eviction policy when deploying Redis as a cache.

Section 06

CDN caching: serving from the edge

A CDN (Content Delivery Network) distributes your content across geographically distributed edge nodes. When a user in Mumbai requests a resource, they hit a Mumbai edge node — not your origin server in us-east-1. If the edge node has the resource cached, it responds immediately without touching your infrastructure at all.

What should be cached at the CDN

  • Static assets: JavaScript bundles, CSS, images, fonts. These are immutable for a given deploy — use content-addressable URLs (/static/app.a3b4c5d6.js) with very long TTLs (1 year). The hash in the filename guarantees uniqueness — a new deploy produces new filenames, old files expire naturally.
  • Cacheable API responses: Product listings, public blog posts, search results for common queries. Not user-specific data, not authenticated endpoints.
  • HTML pages (for SSG/ISR): Statically generated pages or incrementally regenerated pages can be served from CDN. Next.js ISR is a common implementation — pages are generated at the edge and revalidated on a schedule.

Cache bypass for dynamic content

CDNs pass through requests they can't cache. Authenticated requests (with cookies or Authorization headers) typically bypass the cache by default in most CDN configurations — this is correct behaviour, as user-specific data must not be served from a shared cache. Configure your CDN explicitly to bypass the cache for authenticated requests: if Authorization or Cookie header is present, forward to origin without caching.

🔴
Caching authenticated responses at the CDN is a privacy catastrophe

If a CDN caches a response that includes user-specific data (user profile, shopping cart, private messages) and then serves that cached response to the next user who requests the same URL, User B sees User A's data. This has happened in production at multiple companies. Ensure your CDN is configured to never cache responses with Set-Cookie, Authorization, or any user-specific headers. Add Cache-Control: private, no-store to all authenticated API responses.

Section 07

HTTP cache headers: the full specification

HTTP caching is defined by headers. The browser cache, CDN caches, and proxy caches all respect these headers. Getting them right eliminates unnecessary round-trips; getting them wrong either causes stale content or prevents caching entirely.

Cache-Control directives

  • max-age=N — resource is fresh for N seconds. Browsers and CDNs cache without revalidating.
  • s-maxage=N — like max-age but for shared caches (CDNs) only. Allows different TTLs for browser vs CDN.
  • no-cache — must revalidate with origin before using cached copy. Not "don't cache" — just "always check."
  • no-store — don't cache at all. Sensitive data only.
  • private — only browser can cache, not shared caches (CDNs). Use for authenticated responses.
  • public — shared caches can cache this. Explicit permission for CDN caching.
  • stale-while-revalidate=N — serve stale for up to N seconds while revalidating in background. Excellent for API responses.
  • immutable — resource will never change. Browser should never revalidate. Use with content-addressable URLs.

ETag and Last-Modified (conditional requests)

Conditional requests let the client ask "has this changed since I last fetched it?" The server either returns 304 Not Modified (saving bandwidth) or the full response. ETag is a hash of the response content. If-None-Match: "etag_value" sends the stored ETag; if it matches the current ETag, the server returns 304. Last-Modified and If-Modified-Since do the same with timestamps.

Section 08

Multi-layer caching: coordinating L1, L2, CDN

Production systems often have multiple cache layers. Each layer has different characteristics: in-process L1 is fastest but limited and local; distributed L2 (Redis) is shared and durable but adds a network hop; CDN eliminates origin round-trips for eligible content. The layers interact in ways that create subtle bugs if not designed carefully.

L1 + L2 (in-process + Redis)

Check in-process cache first (nanoseconds, no network). On miss, check Redis (0.2ms, one network hop). On miss, hit the database (5ms+, one network hop). Write to both L1 and L2 on a database fetch. The challenge: L1 is per-instance. If you have 20 application pods, each has its own L1. When data changes, you invalidate Redis (L2) — but all 20 L1 caches still have the old data. You need either very short L1 TTLs (10-60 seconds, accepted staleness) or a pub/sub invalidation mechanism (invalidation event sent to all pods).

TTL coordination across layers

TTLs must be consistent across layers, or you create inconsistency windows. If L1 TTL is 5 minutes and L2 (Redis) TTL is 1 minute: after Redis expires the key, L1 still serves the cached (potentially stale) value for up to 4 more minutes. Rule: L1 TTL should always be ≤ L2 TTL. L2 TTL should always be ≤ CDN max-age.

Section 09

Production reality

Cache warm-up after a deploy

After a deployment that restarts all application pods, the L1 caches are empty. If you also flush Redis (common practice to avoid serving old data format after a breaking change), the L2 cache is also empty. All traffic hits the database simultaneously. This is the cold start problem, and for high-traffic applications it can take down the database before the cache warms up. Mitigations: stagger pod restarts so some pods still have warm caches; pre-warm the cache with a scheduled job that reads popular keys before traffic arrives; use cache versioning instead of flushing (old keys expire via TTL, new keys are populated lazily).

The thundering herd on TTL alignment

If you set the same TTL for all keys, they all expire at the same time (if they were all populated at the same time, e.g. at cache warm-up). All those keys expire simultaneously, and all the corresponding requests go to the database at once. Add jitter: TTL = base_ttl + random(0, max_jitter). A base TTL of 300 seconds with ±60 seconds of jitter means keys expire spread over a 2-minute window instead of all at once.

Monitor cache hit rate, not just Redis health

Redis being "up" doesn't tell you whether your cache is working. A cache with a 20% hit rate is barely helping. Monitor hit rate via Redis INFO stats: keyspace_hits / (keyspace_hits + keyspace_misses). Alert when hit rate drops below your baseline — it indicates either a cache flush, a traffic pattern change, or a bug in cache population.

Redis memory pressure before OOM

Watch Redis memory usage. When Redis approaches its maxmemory limit, eviction starts. Eviction is extra work for Redis and can cause latency spikes. Alert when memory usage exceeds 80% of maxmemory and take action before Redis starts evicting aggressively.

Section 10

The things nobody tells you

Caching null values is intentional

A cache miss that returns NULL from the database is still a valid lookup — it means the entity doesn't exist. If you don't cache the NULL result, every subsequent request for that non-existent entity goes to the database. Cache the NULL: redis.setex('user:99999', 60, 'null'). On a read, check for the literal string 'null' and return null to the caller. This prevents "cache stampede for non-existent entities" — a common source of database load in user-facing APIs where users can request arbitrary IDs.

JSON serialisation is your biggest cache overhead

For every cache read, you fetch a string from Redis and JSON.parse() it. For every cache write, you JSON.stringify() an object and write it. For a complex nested object with 50 fields, JSON parsing can take 0.5-2ms — not negligible compared to the 0.2ms Redis round-trip. Consider: MessagePack (compact binary format), keeping cached objects small (only cache what you display), or using Redis Hash instead of String (update individual fields without serialising the whole object).

The Cache-Control: no-cache misconception

Cache-Control: no-cache does NOT mean "don't cache." It means "cache it, but always revalidate with the origin before using it." If the server returns 304 Not Modified, the cached copy is used. If you want truly no caching: use Cache-Control: no-store. This is one of the most common HTTP caching bugs — developers use no-cache thinking they're disabling caching and are surprised when responses are still cached.

Redis keyspace notifications for cache invalidation

Redis can publish events when keys expire or are deleted — via keyspace notifications. Enable with CONFIG SET notify-keyspace-events KEA. Subscribe to __keyevent@0__:expired to be notified when any key expires. This allows reactive cache invalidation: when key A expires (the product cache), automatically invalidate key B (the product listing cache that includes product A). Powerful but adds operational complexity — the subscriber must be always running.

Decision sequence

Adding a cache layer — decide in this order

Work through this before adding Redis to your architecture.

1
Is the underlying query already fast?
Run EXPLAIN ANALYZE. If the query is slow (> 10ms for a simple lookup), fix it first — add an index, rewrite the query, optimise the schema. Caching a slow query hides the problem temporarily and creates a 2-second spike every time the cache expires. Fix first, then cache if throughput requires it.
slow query + cache = hidden problem, not solved problem
2
Is this data cacheable? (immutable, shared, or tolerates staleness)
Not all data is worth caching. User-specific real-time data (account balance, unread message count) changes frequently and is unique per user — caching adds complexity without much benefit. Cacheable data: product catalog, public blog posts, configuration, feature flags, search results for common queries, user profiles (tolerate 5-minute staleness). Ask: "If I serve 60-second-old data here, does anything break?"
real-time user-specific data rarely benefits from caching
3
Choose cache-aside unless you have a reason not to
Cache-aside is simpler to implement, reason about, and debug than write-through or write-behind. It only caches data that's actually been requested (no wasted memory), and the cache key structure is explicit in the application code. Use write-through only if you can't tolerate cache misses on reads (cache must always be warm). Use write-behind only if write performance is genuinely the bottleneck.
default: cache-aside with TTL
4
Set TTL deliberately — not arbitrarily
TTL should reflect how long stale data is acceptable, not a round number you picked. For a product price: maybe 5 minutes (customers are okay with seeing a price that's up to 5 minutes old). For a user profile photo: maybe 24 hours. For a feature flag: maybe 60 seconds. Write the TTL value down with its justification. "TTL = 3600 because I wasn't sure" is a future incident waiting to happen.
TTL = stale tolerance, not a guess
5
Add jitter to TTL and protect against stampede
TTL jitter prevents simultaneous expiry of all keys. Add random(0, TTL*0.2) to every TTL. For high-traffic keys (homepage product list, trending feed), add stampede protection: mutex lock pattern or stale-while-revalidate. A cache miss on a key that serves 1,000 req/s is 1,000 simultaneous database queries — design for it explicitly.
high-traffic keys need stampede protection, not just a TTL
6
Measure hit rate — don't assume the cache is working
Monitor redis.info('stats'): keyspace_hits / (keyspace_hits + keyspace_misses). For a cache that's actually helping, you want ≥ 80% hit rate. Below 60%, the cache is consuming memory without delivering much value — keys are too short-lived, the data is too unique per request, or the cache is not being populated correctly. Dashboard this metric and set an alert.
hit rate is the only metric that proves the cache is working

Caching checklist

HR
Hit rate monitored

Track keyspace_hits / (hits + misses). Alert below 80%. A low hit rate means the cache isn't helping.

JTR
TTL jitter

Add ±20% random jitter to every TTL. Prevents all keys expiring simultaneously and causing a stampede.

EVT
Eviction policy

Never leave Redis at default noeviction for a cache. Set maxmemory + allkeys-lru. Failing writes in production hurt.

PRV
Auth = private

All authenticated responses have Cache-Control: private, no-store. CDN must never cache user-specific data.

STP
Stampede protection

High-traffic cache keys have mutex lock or stale-while-revalidate. A popular key expiring = N simultaneous DB queries.

NUL
Cache nulls

Cache DB nulls with a short TTL. Prevents stampede for non-existent entities (user fetching their own deleted profile, bots probing IDs).
