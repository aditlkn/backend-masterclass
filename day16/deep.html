Deep Dive 01

Production Redis cache class:
everything in one place

A cache scattered across the codebase — different TTLs, different key formats, different null-handling — is a maintenance nightmare. This cache class centralises the logic: stampede protection, null caching, metrics, typed keys, and jitter all in one place.

Cache class — production-ready with stampede protection + metricsTYPESCRIPT
import { Redis } from 'ioredis';
import { metrics } from './metrics';  // Prometheus/StatsD

const NULL_SENTINEL = '__null__';  // represents a cached null/not-found

export class Cache {
  constructor(private redis: Redis, private prefix = 'v1') {}

  private key(k: string) { return `\${this.prefix}:\${k}`; }

  // Add jitter to avoid synchronized expiry
  private jitteredTtl(baseTtl: number, jitterPct = 0.2): number {
    return Math.floor(baseTtl * (1 + jitterPct * Math.random()));
  }

  // Core get-or-set with stampede protection + null caching
  async getOrSet<T>(
    k: string,
    fetch: () => Promise<T | null>,
    opts: { ttl: number; nullTtl?: number; stampede?: boolean } = { ttl: 300 }
  ): Promise<T | null> {
    const cacheKey = this.key(k);
    const start = Date.now();

    // 1. Check cache
    const cached = await this.redis.get(cacheKey);
    if (cached !== null) {
      metrics.increment('cache.hit', { key: k });
      metrics.timing('cache.latency', Date.now() - start, { result: 'hit' });
      return cached === NULL_SENTINEL ? null : JSON.parse(cached);
    }

    metrics.increment('cache.miss', { key: k });

    // 2. Stampede protection (optional, for high-traffic keys)
    if (opts.stampede) {
      const lockKey = this.key(`lock:\${k}`);
      const lockVal = Math.random().toString(36);
      const acquired = await this.redis.set(lockKey, lockVal, 'NX', 'EX', 10);
      if (!acquired) {
        // Wait for the lock holder to populate cache
        await new Promise(r => setTimeout(r, 50));
        const retried = await this.redis.get(cacheKey);
        if (retried) return retried === NULL_SENTINEL ? null : JSON.parse(retried);
      }
    }

    // 3. Fetch from source
    try {
      const value = await fetch();
      metrics.timing('cache.fetch_duration', Date.now() - start);

      if (value === null || value === undefined) {
        // Cache the null — prevents repeated DB hits for non-existent entities
        const nullTtl = opts.nullTtl ?? 60;
        await this.redis.setex(cacheKey, nullTtl, NULL_SENTINEL);
        return null;
      }

      const ttl = this.jitteredTtl(opts.ttl);
      await this.redis.setex(cacheKey, ttl, JSON.stringify(value));
      return value;
    } finally {
      // Release stampede lock if we held it
      if (opts.stampede) {
        await this.redis.eval(
          `if redis.call('get',KEYS[1])==ARGV[1] then return redis.call('del',KEYS[1]) else return 0 end`,
          1, this.key(`lock:\${k}`), ''  // lockVal lost in catch — harmless (EX=10 auto-releases)
        );
      }
    }
  }

  // Invalidate a specific key or pattern
  async del(k: string): Promise<void> {
    await this.redis.del(this.key(k));
  }

  // Bump cache version to invalidate ALL keys (deploy strategy)
  static withVersion(redis: Redis, version: string) {
    return new Cache(redis, `v\${version}`);
  }
}

// ── Usage ────────────────────────────────────────────────────────
const cache = new Cache(redis);

// Normal cached lookup (5 min TTL + jitter)
const product = await cache.getOrSet(
  `product:\${id}`,
  () => db.findProductById(id),
  { ttl: 300 }
);

// High-traffic key with stampede protection
const homepage = await cache.getOrSet(
  'homepage:featured',
  () => db.getFeaturedProducts(),
  { ttl: 60, stampede: true }
);

// Deploy-time version bump (invalidates all old keys)
const versionedCache = Cache.withVersion(redis, process.env.DEPLOY_VERSION);
// Old keys (v1:*) become unreachable; new keys (v2:*) populate lazily
Verify cache is working correctly in production
✓Hit rate ≥ 80%: redis-cli info stats | grep keyspace → keyspace_hits / (keyspace_hits + keyspace_misses)
✓Null values are cached: make a request for a non-existent entity, check that subsequent requests don't hit the DB (redis-cli GET "v1:product:99999" returns __null__)
✓Jitter is applied: check TTLs of recently set keys — redis-cli TTL "v1:product:123" — should vary across similar keys by ±20%.
✓Eviction policy is set: redis-cli config get maxmemory-policy should return allkeys-lru or allkeys-lfu, not noeviction.
Deep Dive 02

Cloudflare cache rules:
caching at the edge

Cloudflare caches responses at 300+ edge locations worldwide. Correctly configured, a Cloudflare cache rule means your origin server never receives the majority of static and public-content requests. Incorrectly configured, it either fails to cache (costing you performance) or caches authenticated responses (a privacy incident).

Cloudflare cache rules — Terraform configurationTERRAFORM + BASH
# Cloudflare Cache Rules via Terraform (cloudflare_ruleset)
resource "cloudflare_ruleset" "cache_rules" {
  zone_id = var.cloudflare_zone_id
  name    = "Cache rules"
  kind    = "zone"
  phase   = "http_request_cache_settings"

  rules {
    # Rule 1: Static assets — cache for 1 year
    description = "Static assets - immutable"
    expression  = "(http.request.uri.path matches \"^/static/.*\\.(js|css|woff2|png|jpg|webp)$\")"
    action      = "set_cache_settings"
    action_parameters {
      cache = true
      edge_ttl {
        mode    = "override_origin"
        default = 31536000  # 1 year
      }
      browser_ttl {
        mode    = "override_origin"
        default = 31536000
      }
      cache_key {
        ignore_query_strings_order = false
        cache_deception_armor      = true
      }
    }
  }

  rules {
    # Rule 2: Public API responses — cache at edge, not browser
    description = "Public API - edge cache only"
    expression  = "(http.request.uri.path matches \"^/api/v1/(products|categories|search)\")"
    action      = "set_cache_settings"
    action_parameters {
      cache = true
      edge_ttl {
        mode    = "override_origin"
        default = 60  # 60 seconds at edge
      }
      browser_ttl {
        mode    = "bypass"  # browser always fetches fresh from edge
      }
    }
  }

  rules {
    # Rule 3: Bypass cache for authenticated requests
    description = "Bypass - authenticated"
    expression  = "(http.request.headers[\"authorization\"] ne \"\" or http.cookie contains \"session=\")"
    action      = "set_cache_settings"
    action_parameters {
      cache = false  # NEVER cache authenticated responses
    }
  }
}

## ── Purge Cloudflare cache on deploy ──────────────────────────────
# Option A: Purge entire cache (simple, slower recovery)
curl -X POST "https://api.cloudflare.com/client/v4/zones/${ZONE_ID}/purge_cache" \
  -H "Authorization: Bearer ${CF_API_TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{"purge_everything": true}'

# Option B: Purge specific URLs (precise, preferred)
curl -X POST "https://api.cloudflare.com/client/v4/zones/${ZONE_ID}/purge_cache" \
  -H "Authorization: Bearer ${CF_API_TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{
    "files": [
      "https://yourapp.com/api/v1/products",
      "https://yourapp.com/api/v1/categories"
    ]
  }'

# Option C: Purge by cache tag (requires Cloudflare Enterprise)
# Add Cache-Tag header in responses: Cache-Tag: product-123,category-electronics
# Then purge by tag:
curl -X POST ".../purge_cache" -d '{"tags": ["product-123"]}'

## ── Verify Cloudflare is caching (check response headers) ────────
curl -I https://yourapp.com/api/v1/products
# Look for: CF-Cache-Status: HIT (or MISS on first request)
# CF-Cache-Status values:
#   HIT    → served from Cloudflare edge cache
#   MISS   → served from origin, now cached
#   BYPASS → bypassed cache (auth request or rule bypass)
#   EXPIRED → cache existed but expired, fetching from origin
#   DYNAMIC → Cloudflare determined response is dynamic (not cacheable)
Deep Dive 03

Cache warming scripts:
no cold start after deploy

A cold start after a deploy — when all caches are empty and every request hits the database — is avoidable. A pre-deploy warm-up script populates the cache with the most frequently accessed data before traffic switches to the new deployment, with jitter to prevent the midnight-expiry problem.

Cache warm-up script — pre-deploy, jittered, with hit rate monitoringTYPESCRIPT
// scripts/cache-warmup.ts
// Run before traffic cutover in CI/CD pipeline

const WARMUP_CONFIG = [
  // [cacheKey, fetchFn, baseTtl, priority]
  { key: 'homepage:featured',  fetch: () => db.getFeaturedProducts(),  ttl: 300,  priority: 1 },
  { key: 'homepage:categories', fetch: () => db.getCategories(),       ttl: 3600, priority: 1 },
  { key: 'config:features',     fetch: () => db.getFeatureFlags(),     ttl: 300,  priority: 1 },
  { key: 'config:pricing',      fetch: () => db.getPricingRules(),    ttl: 600,  priority: 1 },
];

async function warmTopProducts(limit = 500) {
  // Warm top N products by traffic (from analytics or access logs)
  const topProducts = await analytics.getTopProducts(limit);
  return topProducts.map(id => ({
    key: `product:\${id}`,
    fetch: () => db.getProduct(id),
    ttl: 300,
    priority: 2,
  }));
}

function jitteredTtl(base: number): number {
  // ±20% jitter to prevent simultaneous expiry
  return Math.floor(base * (0.9 + Math.random() * 0.2));
}

async function warmCache() {
  const cacheVersion = process.env.DEPLOY_VERSION;
  const cache = new Cache(redis, `v\${cacheVersion}`);

  const items = [
    ...WARMUP_CONFIG,
    ...await warmTopProducts(500),
  ].sort((a, b) => a.priority - b.priority);  // P1 first

  let success = 0, failed = 0;

  // Batch in groups of 20 (don't overwhelm DB)
  for (let i = 0; i < items.length; i += 20) {
    const batch = items.slice(i, i + 20);
    await Promise.all(
      batch.map(async item => {
        try {
          const value = await item.fetch();
          const ttl = jitteredTtl(item.ttl);
          await redis.setex(`v\${cacheVersion}:\${item.key}`, ttl, JSON.stringify(value));
          success++;
        } catch (err) {
          console.error(`Failed to warm \${item.key}:`, err);
          failed++;
        }
      })
    );
    await sleep(50);  // breathe between batches
  }

  console.log(`Warm-up complete: \${success} succeeded, \${failed} failed`);

  // Verify hit rate within 60s of traffic shift
  await sleep(60000);
  const info = await redis.info('stats');
  const hits = parseInt(info.match(/keyspace_hits:(\d+)/)[1]);
  const misses = parseInt(info.match(/keyspace_misses:(\d+)/)[1]);
  const hitRate = hits / (hits + misses);
  console.log(`Hit rate 60s post-deploy: \${(hitRate*100).toFixed(1)}%`);

  if (hitRate < 0.7) {
    console.error('⚠ Hit rate below 70%. Cache warm-up may have failed.');
    process.exit(1);  // fail the deploy pipeline
  }
}

warmCache().catch(console.error);

Verify warm-up is effective
✓Hit rate ≥ 80% within 60 seconds of deploy: the warm-up script verifies this automatically and fails the pipeline if not met.
✓TTLs are spread: redis-cli --scan --pattern "v2:product:*" | head -20 | xargs -I{} redis-cli TTL {} — values should vary, not all be identical.
✓DB load stays flat during warm-up: check DB query rate during the warm-up phase — should be a steady low rate (20 concurrent queries max), not a spike.
✓Old version keys expire naturally: after the TTL window, old v1:* keys should be gone: redis-cli --scan --pattern "v1:*" | wc -l should approach 0.
