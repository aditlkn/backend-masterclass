Production Incident  ·  Redis Cache  ·  Cache Stampede

Every cache key expired at exactly midnight — 800 simultaneous DB queries

New Year's Eve 00:00:00 IST  ·  Duration: 47 seconds  ·  Severity: P1  ·  Homepage and product listings unavailable

The platform had deployed a cache warm-up script at 11:58 PM to ensure the cache was fully populated for the midnight traffic spike. The script set 2,400 cache keys — all the popular product listings, category pages, and homepage content — with a TTL of exactly 120 seconds and no jitter.

At midnight, traffic doubled as users streamed in for the New Year's sale. At 00:02:00 — exactly 120 seconds after the warm-up script ran — every single cache key expired simultaneously. 800 concurrent requests suddenly had zero cache hits. Every one of them fired a database query. The database, already under 2× load from the traffic spike, received an additional 800 queries in the same second. Response times spiked to 15 seconds. The homepage returned 503s. Users saw errors during the exact moment the promotion was supposed to launch.

Root cause + two-part fix

Root cause: all cache keys populated at the same time with identical TTLs expired simultaneously — the thundering herd. Fix 1: TTL jitter — every key gets base_ttl + random(0, jitter) so keys expire spread over a window. Fix 2: stale-while-revalidate — when a key expires, serve the stale value immediately while asynchronously refreshing from the database. Users never see a miss.

The fixes — jitter + stale-while-revalidateTYPESCRIPT
❌ Incident: all keys same TTL, mass expiry at once
// Warm-up script — sets all keys with identical TTL
for (const key of popularKeys) {
  await redis.setex(key, 120, data);  // ← all expire at T+120s
}

✅ Fix 1: TTL jitter — spread expiry over a window
function withJitter(baseTtl: number, jitterPct = 0.2): number {
  const jitter = baseTtl * jitterPct * Math.random();
  return Math.floor(baseTtl + jitter);  // baseTtl to baseTtl × 1.2
}

for (const key of popularKeys) {
  const ttl = withJitter(120);  // 120-144s, random per key
  await redis.setex(key, ttl, data);  // keys expire spread over 24s window
}

✅ Fix 2: stale-while-revalidate — serve stale, refresh async
async function getWithSWR(key: string, fetch: () => Promise, ttl: number): Promise {
  const [value, expiresAt] = await redis.hmget(`swr:${key}`, 'value', 'expires_at');

  if (value) {
    const isStale = parseFloat(expiresAt) < Date.now() / 1000;
    if (isStale) {
      // Serve stale immediately, refresh in background
      setImmediate(async () => {
        const fresh = await fetch();
        await redis.hset(`swr:${key}`, 'value', JSON.stringify(fresh),
          'expires_at', Date.now() / 1000 + ttl);
        await redis.expire(`swr:${key}`, ttl * 2);  // keep key in Redis
      });
    }
    return JSON.parse(value);  // ← always returns immediately (stale or fresh)
  }
  // True miss (first time, key never set)
  const fresh = await fetch();
  await redis.hset(`swr:${key}`, 'value', JSON.stringify(fresh), 'expires_at', Date.now() / 1000 + ttl);
  await redis.expire(`swr:${key}`, ttl * 2);
  return fresh;
}
What this incident teaches
▶Never set the same TTL for batch-populated cache keys. A cache warm-up that sets all keys with identical TTL creates a time bomb. The keys will all expire together, causing a stampede at a predictable moment.
▶TTL jitter is mandatory for any batch cache population. Add ±20% random jitter to every TTL. 2,400 keys expiring over a 24-second window instead of all at once reduces peak DB load by ~20×.
▶Stale-while-revalidate eliminates the stampede entirely for most use cases. Users get an immediate response (stale is fine for product listings), and the DB is hit by exactly 1 background refresh per key — not by every concurrent request.
▶Pre-planned traffic spikes are the worst time for cache misses. Black Friday, New Year, product launches — these are the moments when traffic peaks AND when cache expiry was most likely set up the same time. Use longer TTLs before known spike events and always add jitter.
