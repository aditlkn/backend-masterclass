Deep Dive 01

Resilience4j:
complete Java/Kotlin configuration

Resilience4j is the standard resilience library for Java and Kotlin, replacing the deprecated Netflix Hystrix. It's modular — each pattern is a separate module. Configuration goes in application.yml for Spring Boot apps or programmatically for other frameworks.

Resilience4j — complete Spring Boot configurationYAML + KOTLIN
# application.yml
resilience4j:
  circuitbreaker:
    instances:
      paymentService:
        slidingWindowType: COUNT_BASED   # or TIME_BASED
        slidingWindowSize: 10            # last 10 calls
        minimumNumberOfCalls: 5          # don't open on first few calls
        failureRateThreshold: 50         # open at 50% failure rate
        slowCallRateThreshold: 50        # also open if 50% of calls are slow
        slowCallDurationThreshold: 2000ms  # "slow" = > 2 seconds
        waitDurationInOpenState: 30s     # stay open 30s before trying half-open
        permittedNumberOfCallsInHalfOpenState: 3
        automaticTransitionFromOpenToHalfOpenEnabled: true
        recordExceptions:
          - java.io.IOException
          - java.util.concurrent.TimeoutException
          - feign.FeignException$InternalServerError
        ignoreExceptions:
          - com.example.BusinessException  # don't count business errors as failures

      inventoryService:
        slidingWindowSize: 20
        minimumNumberOfCalls: 10
        failureRateThreshold: 60
        waitDurationInOpenState: 10s     # internal service recovers faster

  retry:
    instances:
      paymentService:
        maxAttempts: 3
        waitDuration: 500ms
        enableExponentialBackoff: true
        exponentialBackoffMultiplier: 2
        retryExceptions:
          - java.io.IOException
          - java.util.concurrent.TimeoutException
        ignoreExceptions:
          - com.example.PaymentDeclinedException  # don't retry declined cards!

  bulkhead:
    instances:
      paymentService:
        maxConcurrentCalls: 15
        maxWaitDuration: 0ms    # fail immediately if full (no queuing)

  timelimiter:  # Resilience4j's timeout
    instances:
      paymentService:
        timeoutDuration: 3s
        cancelRunningFuture: true

## ── Kotlin usage with Spring annotations ─────────────────────────
# Order of decorator application (outer → inner):
# @Bulkhead → @CircuitBreaker → @Retry → @TimeLimiter → actual call

@Service
class PaymentServiceClient {

  @Bulkhead(name = "paymentService", fallbackMethod = "bulkheadFallback")
  @CircuitBreaker(name = "paymentService", fallbackMethod = "circuitFallback")
  @Retry(name = "paymentService", fallbackMethod = "retryFallback")
  @TimeLimiter(name = "paymentService")
  fun charge(orderId: String, amount: Long): CompletableFuture<PaymentResult> {
    return CompletableFuture.supplyAsync {
      paymentGateway.charge(orderId, amount)  // actual HTTP call
    }
  }

  fun bulkheadFallback(orderId: String, amount: Long, ex: BulkheadFullException): CompletableFuture<PaymentResult> {
    logger.warn("Payment bulkhead full for order {}", orderId)
    return CompletableFuture.completedFuture(PaymentResult.pending("Service at capacity"))
  }

  fun circuitFallback(orderId: String, amount: Long, ex: Exception): CompletableFuture<PaymentResult> {
    logger.warn("Payment circuit open for order {}", orderId)
    return CompletableFuture.completedFuture(PaymentResult.pending("Service temporarily unavailable"))
  }

  fun retryFallback(orderId: String, amount: Long, ex: Exception): CompletableFuture<PaymentResult> {
    logger.error("Payment exhausted retries for order {}: {}", orderId, ex.message)
    // Queue for async retry
    retryQueue.add(RetryJob(orderId, amount))
    return CompletableFuture.completedFuture(PaymentResult.queued())
  }
}

Deep Dive 02

cockatiel:
TypeScript resilience with policy composition

cockatiel is the production-grade TypeScript resilience library (used in VS Code's extension host, among others). Its key feature is explicit policy composition — you define each policy separately and compose them in order. This makes the composition visible and testable, unlike callback-based approaches.

cockatiel — circuit breaker, retry, timeout, full compositionTYPESCRIPT
import {
  CircuitBreakerPolicy, RetryPolicy, TimeoutPolicy,
  BulkheadPolicy, FallbackPolicy,
  ConsecutiveBreaker, SamplingBreaker,
  ExponentialBackoff, handleAll, handleWhen,
  wrap, timeout, retry, circuitBreaker, bulkhead, fallback
} from 'cockatiel';

// ── 1. Define individual policies ──────────────────────────────
// Timeout: 3 seconds
const paymentTimeout = timeout(3000, TimeoutStrategy.Cooperative);

// Retry: exponential backoff, jitter, max 3 attempts
const paymentRetry = retry(
  handleWhen(err => !(err instanceof PaymentDeclinedException)),
  {
    maxAttempts: 3,
    backoff: new ExponentialBackoff({
      initialDelay: 500,  // ms
      maxDelay: 5000,    // ms cap
      exponent: 2,
    })
  }
);

// Circuit breaker: sampling (count-based over time window)
const paymentCB = circuitBreaker(
  handleAll(),  // handle all errors
  {
    halfOpenAfter: 30 * 1000,  // 30s before probing
    breaker: new SamplingBreaker({
      threshold: 0.5,        // 50% failure rate triggers open
      duration:  10 * 1000,  // over 10 second window
      minimumRps: 2,         // need at least 2 req/s before opening
    }),
  }
);

// Bulkhead: max 15 concurrent calls
const paymentBulkhead = bulkhead(15);

// Fallback: return pending result when circuit open
const paymentFallback = fallback(
  handleAll(),
  async (ctx, signal) => ({
    status: 'pending',
    message: 'Payment processing temporarily unavailable',
    retryAfter: 300,
  })
);

// ── 2. Compose in correct order: outer → inner ─────────────────
// Outermost executes first on entry, last on exit
// Order: Fallback → Bulkhead → Circuit → Retry → Timeout → Call
const paymentPolicy = wrap(
  paymentFallback,   // catches ALL errors from inner policies
  paymentBulkhead,   // limits concurrency before attempting call
  paymentCB,         // fast-fails if circuit open
  paymentRetry,      // retries failures (inside circuit)
  paymentTimeout,    // innermost: times out the actual call
);

// ── 3. Use the policy ─────────────────────────────────────────
async function chargePayment(orderId: string, amount: number) {
  return paymentPolicy.execute(
    (ctx) => paymentGateway.charge(orderId, amount),
    AbortSignal.timeout(5000)  // overall deadline
  );
}

// ── 4. Monitor events ──────────────────────────────────────────
paymentCB.onStateChange(state => {
  logger.warn({ state }, 'Payment circuit state changed');
  metrics.gauge('payment_cb_state', { CLOSED:0, OPEN:1, HALF_OPEN:2 }[state] ?? -1);
  if (state === 'OPEN') alerting.notify('payment_circuit_open');
});

paymentRetry.onGiveUp(({ error, attempts }) => {
  logger.error({ error, attempts }, 'Payment retries exhausted');
  metrics.increment('payment_retry_exhausted');
});

paymentBulkhead.onReject(() => {
  metrics.increment('payment_bulkhead_rejected');
});
Deep Dive 03

Composition order:
why it matters and what goes wrong

The order in which you compose resilience patterns is not arbitrary. Each pattern wraps the next. The outermost pattern sees everything that happens inside. Getting the order wrong causes subtle bugs where retries don't count toward the circuit breaker, or timeouts fire before retries run.

Composition order — correct vs wrong, with failure modesTYPESCRIPT
// ── CORRECT order: Fallback → Bulkhead → CB → Retry → Timeout ──
const correct = wrap(fallback, bulkhead, circuitBreaker, retry, timeout);

// Why this order works:
//
// Timeout (innermost): bounds the single call attempt. If it times out,
//   throws TimeoutError → Retry catches it and tries again.
//
// Retry: wraps Timeout. Each retry gets its own fresh timeout.
//   Retry INSIDE circuit breaker: each attempt (including retries) is
//   recorded as a failure/success by the circuit breaker.
//
// Circuit breaker: wraps Retry. Sees final outcome of each request
//   (after retries). If retry succeeds on attempt 3, CB records success.
//   If retry exhausts, CB records failure → may open circuit.
//
// Bulkhead: wraps CB. Limits concurrency BEFORE hitting circuit breaker.
//   If bulkhead full: reject immediately, CB not involved.
//
// Fallback (outermost): catches everything. BulkheadFull, CircuitOpen,
//   RetryExhausted, Timeout — all handled by one fallback.

// ── WRONG: Retry OUTSIDE Circuit Breaker ──────────────────────
const wrong1 = wrap(retry, circuitBreaker, timeout);
// Problem: CB opens after threshold failures.
// Retry wraps CB → retry catches CircuitOpenError and retries!
// Each retry hits the CB which is open → another CircuitOpenError.
// Result: max 3 retries of a circuit that's already open.
// CB is supposed to STOP retries. Here retries happen anyway.

// ── WRONG: Timeout OUTSIDE Retry ──────────────────────────────
const wrong2 = wrap(timeout, retry, actualCall);
// Timeout(3s) wraps Retry(3 attempts).
// After 3s, timeout fires — even if retry is mid-attempt.
// User gets 1 timeout after 3s instead of 3 × 1s attempts.
// If you want: total timeout of 10s with 3 retries of 3s each
// → Need the outer deadline (AbortSignal) for total + inner timeout per attempt.

// ── WRONG: Fallback INSIDE Circuit Breaker ─────────────────────
const wrong3 = wrap(circuitBreaker, fallback, timeout);
// CB wraps Fallback.
// When fallback runs (returning cached data), CB sees a "success".
// Fallback successes count toward CB's success rate → CB may close
// even though the actual downstream service is still failing.
// CB never actually knows the service is down because fallback masks it.

// ── CORRECT: per-attempt timeout vs overall deadline ───────────
const perAttemptTimeout = timeout(1000);      // 1s per attempt
const retryPolicy = retry(handleAll(), { maxAttempts: 3 });

const policy = wrap(retryPolicy, perAttemptTimeout);

// Execute with overall 4s deadline (enough for 3 × 1s attempts + backoff)
await policy.execute(
  (ctx) => service.call(),
  AbortSignal.timeout(4000)  // overall deadline passed via AbortSignal
);
Verify composition is correct
✓Retries don't trigger on open circuit: open the circuit manually, then make a request — it should fast-fail immediately without retry. No retry attempts should appear in logs.
✓Each retry attempt registers with circuit breaker: make 3 failing requests (each retry exhausts) — the CB failure count should increase by 3, not by 1.
✓Fallback doesn't count as CB success: trigger fallback by opening circuit, verify CB state stays OPEN. Fallback responses should not cause the circuit to close.
✓State change alerts fire: with monitoring set up, manually open the circuit — an alert should appear in your observability stack within 30 seconds.
