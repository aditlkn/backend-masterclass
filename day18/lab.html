Production Incident  ·  Cascade Failure  ·  No Circuit Breaker

Payment service slows — checkout, inventory, and notifications all go down

Friday 19:30 IST (peak traffic)  ·  Duration: 22 minutes  ·  Severity: P0  ·  Complete checkout outage

The checkout service called three services sequentially: inventory (reserve stock), payment (charge card), notifications (send confirmation). The payment gateway — a third-party provider — started experiencing latency issues, responding in 8–12 seconds instead of the usual 200ms. No timeout was configured on the payment call.

At 30 requests per second, each checkout tied up a thread for 10 seconds waiting for payment. The 100-thread pool filled in 3 seconds. New requests queued. The queue filled. Health checks started failing. The load balancer removed checkout service pods one by one. But inventory service was still healthy — inventory checks from the product listing pages started failing too, because the inventory service still had successful health checks but was now receiving checkout requests that could never complete. Within 4 minutes, all three services were effectively down.

Friday 19:30 IST

Friday 19:52 IST
Payment gateway recovers. Checkout service restarted. 22 minutes of complete checkout outage. ~4,000 lost orders during peak traffic.
Three-part fix

1: Timeout on payment call: 3 seconds max. Thread held for 3s, not 12s. Thread pool survives. 2: Circuit breaker around payment: after 5 timeouts, circuit opens. Fast-fails remaining requests. Users see "payment unavailable" immediately instead of waiting. 3: Bulkhead: payment calls get their own 15-thread pool. Even with payment fully hung, inventory and notifications are unaffected — different pools.

The fix — timeout + circuit breaker + bulkheadTYPESCRIPT
❌ Incident: no protection on payment call
async function checkout(orderId: string, amount: number) {
  await inventoryService.reserve(orderId);
  await paymentService.charge(orderId, amount);  // no timeout! hangs for 10s
  await notificationService.send(orderId);
}

✅ Fix: timeout + circuit breaker + bulkhead, in composition order
const paymentCB = new CircuitBreaker({
  failureThreshold: 0.5,  // open at 50% failure rate
  minRequests: 5,
  windowMs: 10_000,
  openDurationMs: 30_000,
  fallback: async (orderId) => ({
    status: 'pending',
    message: 'Payment processing delayed. You will be notified.',
    cartSaved: true,
  }),
});

const paymentBulkhead = new Bulkhead(15);  // max 15 concurrent payment calls

async function checkout(orderId: string, amount: number) {
  // Inventory: simple timeout (fast internal service)
  await Promise.race([
    inventoryService.reserve(orderId),
    timeout(500, 'inventory timeout'),
  ]);

  // Payment: bulkhead → circuit breaker → timeout → call
  const result = await paymentBulkhead.execute(() =>
    paymentCB.execute(
      () => Promise.race([
        paymentService.charge(orderId, amount),
        timeout(3000, 'payment timeout'),  // 3s max
      ]),
      orderId  // passed to fallback
    )
  );

  // Notifications: fire-and-forget, non-critical, don't await
  notificationService.send(orderId).catch(err =>
    logger.warn({ err }, 'Notification failed (non-critical)')
  );

  return result;
}
What this incident teaches
▶No timeout = guaranteed cascade. The entire system went down because of one missing line: a timeout on the payment call. Every external call needs a timeout. This is the highest-leverage change you can make to resilience.
▶Notifications should never block user-facing responses. The notification service is non-critical — fire and forget. If it fails, log the error but don't fail the checkout. The pattern: service.send().catch(log) not await service.send().
▶Third-party services need bulkheads. The payment gateway is the highest-risk dependency — it's external, you don't control it, and when it fails it fails slowly. Give it an isolated pool. When the pool is full, fail immediately — don't queue.
▶Define the fallback first. The right fallback for a payment failure is: save the cart, queue for retry, notify the user. This is a product decision. Define it before writing the circuit breaker — otherwise when the circuit opens, the user sees a generic 500 error.
