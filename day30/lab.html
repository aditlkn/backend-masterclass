Performance Regression · GET /events · 120ms → 4,200ms

The event listing endpoint went from 120ms to 4 seconds after last week's deploy.

Production · GET /events · P99 latency · introduced in commit a3f91b2

Last Tuesday's deploy included a new feature: each event in the listing now shows the organiser's profile picture and bio. This required fetching organiser details alongside each event. The deploy went out. Nobody noticed immediately because the development environment only has 12 events. Production has 2,400 active events, paginated 50 per page.

P99 latency on GET /events is now 4,200ms, up from 120ms before the deploy. Users are seeing loading spinners. The on-call engineer has been paged.

Step 1: Check the database

-- Run pg_stat_statements during a load test of GET /events SELECT LEFT(query, 80) AS query_preview, calls, ROUND(mean_exec_time) AS mean_ms, ROUND(total_exec_time) AS total_ms FROM pg_stat_statements WHERE query ILIKE '%organiser%' OR query ILIKE '%event%' ORDER BY calls DESC LIMIT 10; query_preview | calls | mean_ms | total_ms ----------------------------------------------------|--------|---------|---------- SELECT * FROM organisers WHERE id = $1 | 48,000 | 3 | 144,000 SELECT * FROM events ORDER BY starts_at LIMIT 50 | 960 | 12 | 11,520 SELECT * FROM venues WHERE id = $1 | 46,200 | 2 | 92,400 -- Reading this: -- GET /events is called 960 times in this window. -- For each call: 50 events -> 50 organiser queries + 48.125 venue queries -- (some events share venues, so slightly under 50 venue queries per call) -- Total: 960 calls -> 48,000 organiser queries + 46,200 venue queries -- These should be 1 query each (JOIN), not 50 queries each

Step 3: After the fix, what should the numbers look like?

Verification
After replacing both N+1 loops with a JOIN, what should pg_stat_statements show?
What this incident teaches
▶N+1 is invisible in development — catastrophic in production. 12 events in dev means 13 queries at 3ms each = 39ms total. Feels fine. 50 events in prod means 51 queries at 3ms each = 153ms per category, 306ms total — plus the existing venue N+1 for another 150ms. Before the new feature, 120ms. After, 456ms just from queries, plus the pre-existing venue N+1 makes it worse.
▶pg_stat_statements fingerprints N+1 immediately. The signature is unmistakable: a parameterised query with calls >> page_size. 48,000 calls to SELECT * FROM organisers WHERE id = $1 when you only made 960 requests means 50 organiser queries per request. One number tells you everything about the problem.
▶Caching a broken query is wrong. Caching the response would hide the N+1 for cached users, but the cold-cache path (new events, cache expires, cache-busting deploy) still hits 101 queries. Fix the query, then optionally cache the result. Never cache a slow query as a substitute for fixing it.
▶Fix both N+1 patterns, not just the one that caused the incident. The venue N+1 was already there before this incident — it just was not the bottleneck. The correct response to finding multiple N+1 patterns is to fix all of them in one go. Leaving a known N+1 in place because it is not the biggest problem means the next deploy could make it the biggest problem.
