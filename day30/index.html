Section 01

The doctor's diagnosis — treatment without diagnosis is malpractice

A patient arrives with fatigue and shortness of breath. A bad doctor prescribes iron supplements because fatigue often means anaemia. A good doctor takes vitals, runs blood work, does an ECG, looks at the imaging. The fatigue might be anaemia. It might be a heart condition. It might be thyroid. The treatment is completely different for each. Treating anaemia in a patient with a heart condition does nothing — and might cause harm by delaying the correct diagnosis.

Performance work is identical. A slow API endpoint might be slow because of N+1 database queries. It might be slow because of garbage collection pauses. It might be slow because of a synchronous file read blocking the event loop. It might be slow because the network round-trip dominates and the server is actually fast. Adding an index helps N+1. It does nothing for GC pauses. It actively makes things worse if the index is on a write-heavy table where the query was already fast.

The doctor has specific tools for specific diagnostics. Vitals give the baseline — is the heart rate elevated, is the blood pressure low? Blood work isolates systems — is the iron low, is the thyroid hormone off? Imaging shows structure — is there an obstruction? Each tool narrows the differential. Performance profiling works the same way: latency percentiles show you which requests are slow, flame graphs show you where CPU time is spent, heap snapshots show you what is consuming memory, event loop metrics show you whether the server is blocked.

The doctor treats the diagnosis, not the symptom. Treating fatigue with caffeine makes the patient feel alert while the underlying condition progresses. Adding more servers to a system with N+1 queries makes N+1 happen faster — each server makes more queries. The only correct treatment for N+1 is a JOIN. The only correct treatment for a GC pause is reducing allocation rate or tuning heap size. You cannot cache your way out of a CPU bottleneck. You cannot scale your way out of an algorithmic problem.

🔬
Premature optimisation vs appropriate optimisation

Knuth's maxim — "premature optimisation is the root of all evil" — is often misquoted as justification for never thinking about performance. The full quote is: "We should forget about small efficiencies, say about 97% of the time: premature optimisation is the root of all evil. Yet we should not pass up our opportunities in that critical 3%." The key word is premature — optimisation before measurement. Once you have measured and identified the bottleneck, optimisation is not premature. It is the job.

Section 02

What to measure before profiling

Before reaching for a profiler, establish the baseline. A profiler tells you where time is spent. The baseline tells you how much time is spent — and whether there is actually a problem worth profiling.

The four signals that tell you where to look

P99 latency by endpoint: Which specific endpoints are slow? P99 (the 99th percentile) shows the slow tail that affects 1 in 100 requests. P50 (median) shows the typical case. If P99 is 2 seconds but P50 is 80ms, the problem is intermittent — not a steady-state bottleneck. If both are high, the slow path is hit on every request.

CPU utilisation pattern: Is CPU consistently high (CPU-bound computation or hot loop)? Or does CPU spike periodically (GC pause, timer callback, scheduled job)? Steady high CPU and periodic spikes have completely different root causes.

Memory trend: Is memory steady (healthy) or growing linearly (leak)? Does it saw-tooth (allocate rapidly then GC drops it back down — high allocation rate)? Each pattern points to a different problem.

Database query counts and latency: How many queries does each request generate? What is the P99 query latency? A single slow query and N+1 queries both show as high total DB time — but the fixes are completely different.

Establishing the baseline — quick diagnostic commandsBASH
# 1. Load test to generate data
npx autocannon -c 50 -d 30 http://localhost:3000/bookings/list
# Reports: requests/sec, latency P50/P99, throughput

# 2. Check event loop delay (is Node.js blocked?)
# In code: measure lag between scheduled callbacks
# npm install --save-dev @nicolo-ribaudo/eventloop-lag

# 3. Database query analysis (PostgreSQL)
SELECT query, calls, mean_exec_time, total_exec_time
FROM pg_stat_statements
ORDER BY total_exec_time DESC
LIMIT 20;
# Shows top queries by total time: identifies slow queries AND N+1

# 4. Memory snapshot
# In your app: expose a debug endpoint
# GET /debug/heap-stats -> returns process.memoryUsage()

# 5. Check if you even have a performance problem
# Is P99 latency above your SLO? If not, do not profile yet.
# Profiling adds overhead. Do not profile production continuously.
Section 03

Flame graphs — reading a CPU profile in 60 seconds

A flame graph visualises where CPU time is spent in your application. The x-axis is time (wider = more time). The y-axis is the call stack (higher = deeper into the call chain). Each rectangle is a function. The colour is usually random and meaningless — it is the width that matters.

How to read one: Look for wide rectangles near the top of the flame. A function that is wide but has no children wider than itself is a leaf node — it is where CPU is actually being consumed. A wide parent with narrow children means the parent calls many small functions — the bottleneck is the call pattern, not any single function. A flat-topped flame (most time in one function at the same depth) indicates a hot loop.

What clinic.js flame adds

clinic flame colours Node.js-specific code differently: your application code (coloured), Node.js core (blue), V8 internals (green), third-party modules (yellow). This immediately separates "your code is slow" from "the library you are using is slow" from "V8 is spending time on GC." The diagnosis branch you take depends entirely on which category dominates the flame.

clinic.js — profile your Node.js serviceBASH
# Install clinic.js
npm install -g clinic

# Step 1: Generate a CPU flame graph
clinic flame -- node server.js
# In another terminal, run your load test:
npx autocannon -c 50 -d 30 http://localhost:3000/bookings/list
# Ctrl+C to stop the process. Opens HTML flame graph in browser.

# Step 2: Check for event loop blocking (Doctor)
clinic doctor -- node server.js
# Runs the same way. Produces a health report with:
# - Event loop delay timeline
# - CPU usage
# - Memory usage
# - Handles/requests counts
# Highlights anomalies automatically

# Step 3: Check for memory leaks (Heapprofiler)
clinic heapprofiler -- node server.js
# Samples heap allocations over time
# Shows which code paths allocate the most memory
# Useful for identifying allocation-heavy hot paths
⚠️
Profile under realistic load, not in development

A flame graph generated from a single request in development will look nothing like production. You need the full mix of concurrent requests, database pool pressure, and realistic data sizes to see the actual bottleneck. Run your load test while profiling. Profile against a staging environment with production-like data. A flame graph from a cold, idle server is a performance profile of server startup — not of your hot path.

Section 04

N+1 queries — the performance bug hiding in plain sight

The N+1 problem is when fetching a list of N items requires N additional queries to fetch related data for each item, instead of one query to fetch all the related data at once. A list of 100 events that each have a venue means 1 query to get events + 100 queries to get venues = 101 queries. A list of 1,000 events means 1,001 queries.

N+1 is invisible in development because N is small — you test with 5 events, and 6 queries feel fine. It is catastrophic in production where N is 1,000 and 1,001 queries take 5 seconds. The PostgreSQL pg_stat_statements view will show you: one query executed 1,000 times in rapid succession is the N+1 signature.

N+1 — vulnerable vs fixedTYPESCRIPT
// VULNERABLE — N+1: 1 query + N venue queries
async function getEventListings(): Promise<Event[]> {
  const events = await db.query('SELECT * FROM events LIMIT 100');
  for (const event of events.rows) {
    // This runs 100 times — one query per event
    const venue = await db.query(
      'SELECT * FROM venues WHERE id = $1',
      [event.venueId]
    );
    event.venue = venue.rows[0];
  }
  return events.rows;
}

// FIXED option 1 — JOIN: 1 query total
async function getEventListings(): Promise<Event[]> {
  const result = await db.query(`
    SELECT e.*, v.name as venue_name, v.city as venue_city, v.capacity
    FROM events e
    JOIN venues v ON v.id = e.venue_id
    ORDER BY e.starts_at
    LIMIT 100
  `);
  return result.rows;
}

// FIXED option 2 — dataloader pattern: 2 queries total (batch)
async function getEventListings(): Promise<Event[]> {
  const events = await db.query('SELECT * FROM events LIMIT 100');

  // Collect all venue IDs, deduplicate
  const venueIds = [...new Set(events.rows.map(e => e.venueId))];

  // ONE query to fetch all venues needed
  const venues = await db.query(
    'SELECT * FROM venues WHERE id = ANY($1)',
    [venueIds]
  );
  const venueMap = Object.fromEntries(venues.rows.map(v => [v.id, v]));

  // Attach venues without any further queries
  return events.rows.map(e => ({ ...e, venue: venueMap[e.venueId] }));
}

// DETECT N+1 in development — log query counts
// In your DB middleware, count queries per request:
app.use((req, res, next) => {
  req.queryCount = 0;
  res.on('finish', () => {
    if (req.queryCount > 10) {
      logger.warn({ path: req.path, queryCount: req.queryCount }, 'high_query_count');
    }
  });
  next();
});
✅
pg_stat_statements is your N+1 detector in production

Enable pg_stat_statements in PostgreSQL (it is installed by default on most managed databases). Run SELECT query, calls, mean_exec_time FROM pg_stat_statements ORDER BY calls DESC LIMIT 20. A query with 50,000 calls in one hour that takes 2ms each is N+1: that is 50,000 × 2ms = 100 seconds of query time per hour for what should be one batched query. The calls column is your N+1 fingerprint.

Section 05

GC pressure — when memory management becomes a latency problem

Node.js uses V8's garbage collector to reclaim memory. When GC runs, it pauses your application — for anywhere from a few milliseconds to hundreds of milliseconds depending on heap size and allocation rate. Most Node.js engineers never think about this until they see mysterious P99 latency spikes that appear without any corresponding slow query or CPU spike.

How V8's GC works

V8 uses a generational garbage collector. The heap is divided into the young generation (new space: objects recently allocated, typically 1-8MB) and the old generation (old space: objects that survived multiple GC cycles). Minor GC (scavenge) runs frequently, collecting the young generation quickly (~1-5ms). Major GC (mark-and-sweep) runs less frequently but can pause for 10-200ms on large heaps.

The problem: if your application allocates objects faster than minor GC can collect them, objects graduate to the old generation. A large old generation means major GC takes longer, which means longer pauses. The fix is not always "tune the GC" — it is often "allocate less."

Common allocation patterns that cause GC pressure

  • Creating objects in hot loops: arr.map(x => ({ id: x.id, name: x.name })) on a 10,000-element array creates 10,000 objects per call. If this runs 100 times per second, that is 1 million objects per second entering the young generation.
  • String concatenation in loops: let s = ""; for (...) s += item; creates a new string on every iteration. Use Array.join() instead.
  • JSON.parse / JSON.stringify on large objects: Each call allocates a full copy of the data. If a large object is serialised on every request, the allocation rate is proportional to traffic.
  • Closures capturing large scopes: A closure that captures a large array or object keeps it alive until the closure is collected, even if the closure itself is long-lived.
Reducing allocation pressure — before and afterTYPESCRIPT
// HIGH ALLOCATION: creates new objects on every request
app.get('/events', async (req, res) => {
  const events = await db.query('SELECT * FROM events');
  // Creates a new object for each event on every request
  const result = events.rows.map(e => ({
    id: e.id,
    name: e.name,
    startsAt: e.starts_at.toISOString(),
    venue: { id: e.venue_id, name: e.venue_name }
  }));
  res.json(result);
});

// LOWER ALLOCATION: reuse object shape where possible
// Option 1: SELECT only what you need (fewer bytes allocated)
app.get('/events', async (req, res) => {
  const events = await db.query(`
    SELECT id, name, starts_at, venue_id, venue_name
    FROM events_with_venue   -- a view or materialised view
    ORDER BY starts_at LIMIT 50
  `);
  // Return rows directly — no intermediate object creation
  res.json(events.rows);
});

// Option 2: pre-allocate result array to avoid resizing
const result = new Array(events.rows.length);  // pre-allocate
for (let i = 0; i < events.rows.length; i++) {
  result[i] = transformEvent(events.rows[i]);   // reuse same shape
}

// GC tuning flags (when allocation rate is unavoidably high)
// node --max-old-space-size=4096 server.js  (4GB old gen)
// node --max-semi-space-size=128 server.js  (128MB young gen)
// Larger young gen = less frequent minor GC (but larger pauses per cycle)

Section 06

Event loop delay — when Node.js stops being concurrent

Node.js processes one thing at a time on the event loop. Async I/O (database queries, HTTP calls, file reads) yields to the event loop while waiting — other requests can be processed in parallel. But synchronous code blocks the entire event loop. A synchronous operation that takes 500ms means every other request waits 500ms before it can be processed.

What causes event loop blocking

  • Synchronous file system operations: fs.readFileSync, crypto.createHash on large inputs — these block the event loop for their entire duration.
  • Expensive JSON operations: JSON.stringify and JSON.parse on large objects are synchronous and can take 50-200ms for megabyte payloads.
  • Dense CPU computation: A Fibonacci calculation, a sorting algorithm on a large array, regex evaluation on untrusted input — any tight loop that does not yield.
  • Synchronous crypto: bcrypt.hashSync with high work factor (10-12) is intentionally slow (100-300ms) to make brute force expensive. Run it in a worker thread.

Detecting event loop delay

Measure the gap between a scheduled callback and when it actually runs. A healthy event loop fires callbacks within 1-2ms of when they were scheduled. An event loop with blocking code fires them 50-500ms late. clinic doctor visualises this. In production, track it with Prometheus: the nodejs_eventloop_lag_seconds metric from prom-client.

Event loop delay — measure and fixTYPESCRIPT
// Measure event loop lag in production
import { monitorEventLoopDelay } from 'perf_hooks';

const h = monitorEventLoopDelay({ resolution: 20 });
h.enable();

setInterval(() => {
  const lagMs = h.mean / 1e6;  // nanoseconds -> milliseconds
  prometheusGauge.set(lagMs);
  if (lagMs > 100) {
    logger.warn({ lagMs }, 'event_loop_delay_high');
  }
  h.reset();
}, 5000);

// BLOCKING: password hashing on the event loop
app.post('/auth/register', async (req, res) => {
  const hash = bcrypt.hashSync(req.body.password, 12);  // blocks ~200ms
  await db.createUser({ email: req.body.email, hash });
  res.json({ created: true });
});

// FIXED: run in worker thread
import { Worker, isMainThread, parentPort, workerData } from 'worker_threads';

// hash-worker.js
if (!isMainThread) {
  const hash = bcrypt.hashSync(workerData.password, 12);
  parentPort!.postMessage(hash);
}

// In your handler — bcrypt runs in a thread, event loop is free
async function hashPassword(password: string): Promise {
  return new Promise((resolve, reject) => {
    const worker = new Worker('./hash-worker.js', { workerData: { password } });
    worker.on('message', resolve);
    worker.on('error', reject);
  });
}

app.post('/auth/register', async (req, res) => {
  const hash = await hashPassword(req.body.password);  // non-blocking
  await db.createUser({ email: req.body.email, hash });
  res.json({ created: true });
});

Section 08

Production reality

Never profile in production — use continuous profiling

Running clinic flame in production adds 20-40% CPU overhead and generates gigabytes of profile data. Instead, use continuous profiling tools that sample at low overhead: Pyroscope (open source, integrates with Grafana), Datadog Continuous Profiler, or Google Cloud Profiler. These run at <1% overhead continuously, letting you see performance over time without a specific profiling session. The flame graph from a continuous profiler looks the same as a clinic flame — just collected differently.

Caching is not a performance fix — it is a performance multiplier

Caching a slow query makes the cached result fast but the uncached path still slow. The first user after a cache miss, or any user during cache warm-up, still hits the slow path. Caching is valuable after you have fixed the underlying performance problem. A query that takes 2ms is worth caching because it reduces database load. A query that takes 2 seconds has a bug — fix the bug, then optionally cache the result.

Indexing strategy — based on query patterns, not guesses

An index on the right column makes a query 1000x faster. An index on the wrong column adds write overhead with no read benefit. The right column is the one in your WHERE and ORDER BY clauses for your slowest queries. Run EXPLAIN ANALYZE on every slow query. A sequential scan on a large table with a WHERE clause always needs an index. A sequential scan on a small table (under 1,000 rows) probably does not — the planner is right.

Section 09

The things nobody tells you

The bottleneck moves when you fix it

You profile, find the N+1 query, fix it. P99 drops from 2 seconds to 800ms. You profile again. Now the flame graph shows the serialisation bottleneck that was previously invisible behind the query time. Fix that. P99 drops to 400ms. Profile again. Now GC pressure is the top entry. Each fix reveals the next constraint. This is not a sign that your profiling was wrong — it is how performance work actually proceeds. The bottleneck shifts every time you remove the current one.

Memory leaks look like performance problems

A slow memory leak increases heap size over hours. As heap grows, major GC runs take longer. P99 latency climbs slowly — not dramatically, just 10ms per hour. After 24 hours, the service is slow enough to notice. The symptom is a performance problem (high latency). The cause is a memory leak. If you profile at the point of noticing the problem, you see high GC time — but the real fix is finding and stopping the leak, not tuning GC. Always check memory trends before profiling CPU.

setTimeout(fn, 0) does not mean zero delay

Developers sometimes use setImmediate or process.nextTick to defer work and "unblock the event loop." This helps briefly — it yields to other callbacks in the queue. But if you defer heavy work and immediately re-queue it, you still block the event loop for the full duration of the work, just in smaller slices. The only way to truly avoid blocking is to use worker threads for CPU-intensive work.

EXPLAIN ANALYZE costs money

EXPLAIN ANALYZE actually executes the query to produce timing information. On a large table, running EXPLAIN ANALYZE SELECT * FROM events ORDER BY starts_at without a LIMIT will do a full sequential scan — which might take 30 seconds and lock resources. Always add LIMIT when analyzing, or use EXPLAIN (without ANALYZE) to see the query plan without executing it. Run EXPLAIN ANALYZE on a replica, not on primary.

Decision sequence

Diagnosing a performance problem

In order. Do not skip to profiling before establishing the baseline.

1
Confirm there is a problem worth fixing
What is the P99 latency for the slow endpoint? What is your SLO? If P99 is 400ms and SLO is 500ms, there is no problem. If P99 is 2,000ms and SLO is 500ms, the problem is real. Measure with a load test. Profile symptoms that affect real users, not worst-case lab scenarios.
profiling has overhead — do not profile if there is no problem
2
Check database query counts first
Run pg_stat_statements during a load test. Look for queries with very high calls count — this is N+1. Look for queries with high mean_exec_time without indexes — this needs an index. N+1 and missing indexes account for the majority of API performance problems. Fix either of these before any other profiling.
N+1 or missing index = fix first, profile later
3
Check memory trend
Is memory growing over time (leak) or saw-toothing (high allocation rate)? A leak means CPU profiling will show GC time as the symptom — fix the leak first. High allocation rate means reducing object creation. Steady memory with high GC time means the heap is too small for the working set — tune --max-old-space-size.
GC symptoms often come from allocation, not from GC settings
4
Run clinic doctor under load
clinic doctor -- node server.js while running a load test. Look at the event loop delay timeline — spikes indicate blocking code. Look at CPU — is it pegged (CPU-bound) or idle (I/O-bound)? Doctor gives you the category of the problem before you spend time on a detailed flame graph.
doctor gives category, flame gives location
5
Run clinic flame for CPU bottleneck
If doctor shows high CPU, clinic flame -- node server.js under the same load test. Find the widest rectangles near the top of your application code. That is your hot path. Look up what function that is. Determine if it is unavoidable or if the work can be reduced, cached, or moved to a worker thread.
widest rectangle near top = the bottleneck
6
Fix, measure, repeat
Make one change at a time. Measure after each change with the same load test. If P99 improved: quantify the improvement and decide whether to continue. If P99 did not improve: revert the change. Never deploy multiple performance changes simultaneously — you cannot attribute the improvement (or regression) to any specific change.
one change at a time — you cannot measure combined changes
