Production Incident  ·  Sharding  ·  Shard Key Design

Sharded by user_id — then a single enterprise customer brought down their entire shard

Wednesday 14:30 IST  ·  Duration: 4h 15m  ·  Severity: P0  ·  12,000 users affected

A B2B SaaS platform launched sharding 18 months into its life, when its primary was showing signs of write saturation. The engineering team designed the system carefully — they chose user_id as the shard key, built a consistent hash router, and validated the distribution looked even across their 4 shards. They spent three months building it. They were proud of it.

Six months after launch, they signed their first enterprise customer: a company with 12,000 employees, all of whom would use the platform daily. All 12,000 users had user_ids in a sequential range (provisioned as a batch). All 12,000 landed on Shard 2. On the first day of the enterprise rollout, Shard 2 immediately reached capacity. The 12,000 enterprise users couldn't work. Neither could the 3,000 existing users who happened to be on Shard 2.

18 months post-launch
Platform shards by user_id across 4 shards using consistent hash. Distribution looks even at ~25% per shard. No tenant_id concept in the shard key.
Six months later
Enterprise customer signs: 12,000 users, user_ids 850001-862000. All land on Shard 2 (hash distribution clusters sequential IDs). Shard 2 now has 4× normal load.
Wednesday 14:30 IST
Enterprise rollout goes live. All 12,000 users log in. Shard 2 CPU: 97%. Login latency: 8 seconds. Existing users on Shard 2 also affected.
14:45 IST
Team tries to add replicas to Shard 2. Helps reads, not writes. The problem is writes. Enterprise users create sessions, log events, generate notifications — all writes to Shard 2.
18:45 IST
Emergency: manual data migration of 4,000 enterprise users to Shard 0 and Shard 3. Double-write period of 90 minutes. Shard 2 recovers. Resolution: 4h 15m downtime for 15,000 users.
One week later
Architecture changed to directory sharding: tenant_id → shard_id lookup in Redis. Large tenants can be pinned to dedicated shards. Enterprise customers never auto-assigned — always manually placed.
Root cause

Sharding by user_id without a tenant concept means each enterprise customer's users are distributed by their individual IDs — and sequential user provisioning clusters them on one or two shards. The correct approach for B2B SaaS: shard by tenant_id with a directory lookup. All of a tenant's users are logically grouped, the directory assigns tenants to shards manually, and large customers get dedicated shards before they're onboarded.

❌ Incident: shard by user_id — no tenant awareness
function getShardForUser(userId: number): Pool {
  return shards[murmurhash(userId) % shards.length];
  // Enterprise user IDs cluster → shard overload
}

✅ Fixed: directory sharding — tenant controls which shard
-- Redis directory: tenant_id → shard_id
-- Large tenants manually assigned to dedicated shards
async function getShardForUser(userId: number, tenantId: number): Promise<Pool> {
  // Look up tenant's shard from Redis (cached, ~0.3ms)
  const shardId = await redis.get(`tenant:${tenantId}:shard`);
  if (!shardId) throw new Error(`No shard for tenant ${tenantId}`);
  return shards[parseInt(shardId)];
}

What this incident teaches
▶B2B SaaS must shard by tenant_id, not user_id. Users belong to tenants. All activity for a tenant must be predictably routable. A large tenant activating all their users simultaneously must not affect other tenants.
▶Shard assignment for large customers must be manual and deliberate. Never auto-assign an enterprise customer to a shard. Before onboarding, check shard utilization, assign the new tenant to the least loaded shard, and for very large customers allocate a dedicated shard cluster.
▶Directory sharding enables tenant migration without resharding. When a shard fills up, update the Redis mapping (tenant → new shard) and migrate data. The application immediately routes to the new shard once Redis is updated. No infrastructure change needed.
▶Test your shard key with the largest expected customer before launch. Before signing an enterprise deal, simulate what happens if they have 10,000 users. Run the simulation against your sharding architecture. The bugs here are architectural — expensive to fix after the fact.
