Your shard key is wrong.
You won't know it until you can't change it.

Sharding distributes data across multiple database instances. The shard key — the column you partition by — determines whether your shards are balanced or whether one shard holds 80% of your traffic while the others sit idle. Choosing it wrong is one of the most expensive mistakes in backend engineering.

Section 01

When sharding is actually the right answer

Sharding is often reached for prematurely. It introduces fundamental operational complexity — cross-shard queries, distributed transactions, resharding pain — that you want to avoid as long as possible. Most applications never need it. The engineers who implement sharding too early spend months managing complexity that a read replica and better indexes would have handled.

Exhaust these first

Before sharding, verify you've genuinely exhausted:

  • Read replicas: If your bottleneck is read throughput, replicas scale reads horizontally without any of sharding's complexity.
  • Better indexes: A missing index on a 500GB table causes more pain than the table size. Confirm with EXPLAIN ANALYZE that you're not doing sequential scans.
  • Connection pooling: High connection count is often mistaken for database capacity limits. PgBouncer can solve this.
  • Partitioning: PostgreSQL native partitioning keeps data on a single server but distributes it across logical tables. Range partitioning by date enables partition pruning — queries for this week's data never touch last year's partitions. This can provide 10-100× speedup with no application changes.
  • Vertical scaling: A larger instance with more RAM and faster NVMe SSDs is often cheaper and simpler than sharding. AWS RDS goes up to 256 vCPUs and 12TB storage. If your data fits on one machine, sharding is premature.

The actual sharding signals

Write throughput exceeds what a single primary can handle — replicas don't help with writes; only sharding distributes write load. Data volume exceeds what a single machine can economically store — when you're looking at 10+ TB and the cost of vertical scaling exceeds horizontal. Single-table hot spots — a specific partition of your data receives so much traffic that even a very large instance struggles to keep up.

⚠️
Sharding adds permanent complexity

Once you shard, every query involving data from multiple shards becomes a cross-shard query that must be assembled in the application layer or by a query router. Foreign keys across shards don't exist. Transactions spanning shards require 2PC or sagas. Resharding (adding shards or changing the key) is a major operational event. These costs are permanent — you don't get to un-shard once you've built around it.

Section 02

PostgreSQL partitioning: the step before sharding

PostgreSQL native partitioning splits a single logical table into multiple physical sub-tables (partitions), all on the same server. The table looks and behaves like one table to queries — but the query planner knows to only access the relevant partitions. This enables partition pruning: a query for "orders this week" only touches this week's partition, even if the orders table contains 5 years of data.

Partition types

Range partitioning: Partitions defined by a column range — typically time (monthly or quarterly partitions). Best for time-series data where queries almost always include a time filter. Old partitions can be detached and archived or dropped without touching other partitions.

List partitioning: Partitions defined by explicit values — regions, countries, status types. Good for data that naturally clusters by a categorical value and where queries filter by that value.

Hash partitioning: Partitions defined by a hash of a column, distributing rows evenly. No single "large" partition emerges. Good for distributing write load evenly without natural range boundaries.

The partition pruning guarantee

For partition pruning to work, your query's WHERE clause must include the partition key column. SELECT * FROM orders WHERE created_at > '2024-01-01' on a monthly-partitioned table touches only 2024 partitions. But SELECT * FROM orders WHERE user_id = 123 on the same table touches every partition — no pruning possible without the partition key in the WHERE clause.

PostgreSQL range partitioning — monthly orders tableSQL
-- Create the partitioned parent table
CREATE TABLE orders (
  id           BIGSERIAL,
  user_id      BIGINT NOT NULL,
  status       TEXT NOT NULL,
  amount_paise BIGINT NOT NULL,
  created_at   TIMESTAMPTZ NOT NULL DEFAULT now()
) PARTITION BY RANGE (created_at);

-- Create monthly partitions (can be automated with pg_partman)
CREATE TABLE orders_2024_01 PARTITION OF orders
  FOR VALUES FROM ('2024-01-01') TO ('2024-02-01');

CREATE TABLE orders_2024_02 PARTITION OF orders
  FOR VALUES FROM ('2024-02-01') TO ('2024-03-01');

-- Each partition can have its own indexes
CREATE INDEX ON orders_2024_01 (user_id, created_at DESC);
CREATE INDEX ON orders_2024_02 (user_id, created_at DESC);
-- Partitioned indexes (apply to all partitions at once):
CREATE INDEX ON orders (user_id, created_at DESC);

-- Verify partition pruning is happening
EXPLAIN SELECT * FROM orders WHERE created_at >= '2024-01-01' AND created_at < '2024-02-01';
-- Should show: Append → Seq Scan on orders_2024_01 (only one partition!)
-- NOT: Append → Seq Scan on orders_2024_01, orders_2024_02, ... (all partitions)

-- Archive old partition (no DELETE, no rewrite — instant)
ALTER TABLE orders DETACH PARTITION orders_2022_01;
-- orders_2022_01 now exists as a standalone table, can be moved to cold storage

-- Drop old partition (instant, regardless of row count)
DROP TABLE orders_2022_01;
-- vs DELETE FROM orders WHERE created_at < '2022-02-01' → takes hours on large table

-- Automate with pg_partman extension
SELECT partman.create_parent(
  p_parent_table   := 'public.orders',
  p_control        := 'created_at',
  p_type           := 'range',
  p_interval       := 'monthly',
  p_premake        := 4    -- pre-create 4 future partitions
);

Section 03

Shard key design: the most consequential decision

The shard key determines which shard stores which rows. It cannot be changed after the fact without resharding the entire dataset — an operation that can take days and carries significant risk. Getting it wrong means living with an imbalanced, poorly-performing distributed system indefinitely.

Properties of a good shard key

  • High cardinality: Enough distinct values to distribute rows across many shards. Sharding by a boolean column is catastrophic — only 2 possible values, so you'll always have 2 shards max, and they'll likely be unbalanced.
  • Uniform distribution: Values that spread rows evenly across shards. Sharding by user_id works if users are equally active. Sharding by city fails if Mumbai has 10× the traffic of all other cities combined.
  • Query locality: The most common queries should hit a single shard. If your most frequent query is "all orders for user X," sharding by user_id means all of user X's orders are on one shard — one network call. Sharding by order_id instead means user X's orders are spread across all shards — every query is a cross-shard scatter-gather.
  • Avoid sequential keys: Using an auto-incrementing ID as a shard key means all new rows route to the same shard (the highest range). You'll have one perpetually hot shard and many empty ones. Use consistent hashing on the ID instead.

Common shard key patterns

user_id: Best for multi-user applications where most queries are per-user. All of a user's data (orders, notifications, sessions) co-locates on the same shard. Cross-user analytics become expensive.

tenant_id: Best for B2B SaaS. All of a tenant's data lives together. Large tenants can overwhelm their shard (the "whale tenant" problem) — mitigated with tenant-to-shard mapping tables that allow large tenants to get their own dedicated shard.

geographic region: Best when data sovereignty or latency requirements dictate where data lives. EU user data must stay in EU. Cross-region queries are explicitly forbidden.

hash(primary_key): Uniform distribution but no query locality. Every per-entity query hits all shards. Only works if your queries are truly random and you don't need per-entity aggregations.

Section 04

Sharding strategies: range, hash, and directory

Range sharding

Rows are distributed based on a value range of the shard key. Shard 1 gets user_id 1-1,000,000; shard 2 gets 1,000,001-2,000,000. Simple to understand and implement. The problem: if newer users are more active (they are), the highest-range shard always receives the most traffic. Sequential inserts always land on the same shard.

Works well for: time-series data where old data is cold and new data is hot but you can plan for it. Archive old shards as they go cold.

Hash sharding (consistent hashing)

A hash function is applied to the shard key, and the hash value determines the shard. shard = hash(user_id) % num_shards. Distribution is uniform by construction — no hot shards caused by skewed values. The downside: range queries become cross-shard. WHERE user_id BETWEEN 1000 AND 2000 must query all shards because the hash function destroys ordering.

Consistent hashing is the production-grade version: instead of hash % N, keys are placed on a virtual ring, and shards own segments of the ring. Adding a new shard only redistributes a fraction of keys (not everything), making it practical to add capacity without full resharding.

Directory (lookup) sharding

A central lookup table maps each shard key value to a shard. When a new user signs up, the directory assigns them to a shard. The directory can use any logic — it can deliberately assign whale tenants to dedicated shards, balance load by shard utilisation, or move tenants between shards by updating the directory.

This is the most flexible approach and the most common in mature B2B SaaS products. The cost: the directory itself becomes a critical dependency — it must be highly available, fast (usually cached in Redis), and consistent.

Section 05

The hotspot problem: when one shard gets all the traffic

A hotspot occurs when one shard receives disproportionately more reads or writes than others. The shard is overwhelmed while others sit idle. Hotspots defeat the entire purpose of sharding — you've added distributed system complexity but haven't distributed the load.

Causes of hotspots

Viral content or events: A specific item, user, or event suddenly becomes popular. All reads for that item route to one shard.

Skewed shard keys: Using a categorical key with uneven distribution (city, country, category) where one value has far more data than others.

Sequential inserts: All new rows go to the highest-range shard. Common with auto-increment IDs and time-based range sharding without proper planning.

Whale tenants: In B2B SaaS, one large customer can have 10× the data and traffic of all others combined. Sharding by tenant_id puts the whale on one shard.

Mitigation strategies

Random suffix (write hotspot): For counters or high-write keys, append a random suffix to create multiple virtual keys. user:123:views → user:123:views:1, user:123:views:2, ..., user:123:views:10. Write to a random suffix; read by summing all suffixes.

Dedicated shard for whales: In directory sharding, assign large tenants to a dedicated shard (or multiple shards). The directory maps tenant_id=bigcorp → shard_cluster_5 which has extra capacity.

Time-based rotation: For time-series hotspots, ensure new partitions (recent data) aren't on a single shard. Rotate new partition assignments across shards.

Caching: A hotspot shard suffering from read pressure can be mitigated with a cache layer (Redis). Writes still hit the shard, but reads are served from cache.

Section 06

Cross-shard queries: the distributed system tax

When a query needs data from multiple shards, the application (or a query router) must fan out the query to all relevant shards, wait for all responses, and assemble the final result. This is called a scatter-gather query. It works — but it's slower, more complex, and introduces new failure modes.

Scatter-gather mechanics

For a query like "find all orders with status=pending across all users" on a user-sharded system: send the query to all N shards in parallel (fan-out), collect N result sets, merge and sort them in memory, return the final result. The latency is max(shard_latency) + merge_time, not sum(shard_latency) since they run in parallel. But a slow shard (GC pause, lock contention) makes the entire query slow.

What becomes impossible or very expensive

  • Cross-shard JOINs: Joining tables from different shards requires fetching data from multiple shards and joining in application memory. With large result sets this is impractical.
  • Cross-shard transactions: No single-shard ACID. Need 2PC or sagas, with all their complexity and failure modes.
  • Global uniqueness constraints: UNIQUE on an email column only enforces uniqueness within a shard. Across shards: application must check all shards or use a global lookup table.
  • Global aggregates: COUNT(*) across all shards requires querying all shards and summing. ORDER BY with LIMIT requires fetching top-N from each shard and re-sorting.
  • Foreign keys: Cannot reference a row in a different shard. Referential integrity is application responsibility.
Cross-shard scatter-gather in application codeTYPESCRIPT
class ShardRouter {
  private shards: Pool[] = [shard0, shard1, shard2, shard3];

  // Single-shard query (fast — hits one shard by key)
  async getUserOrders(userId: number): Promise<Order[]> {
    const shard = this.shards[userId % this.shards.length];
    return shard.query('SELECT * FROM orders WHERE user_id = $1', [userId]);
  }

  // Cross-shard scatter-gather (slow — hits all shards)
  async getPendingOrders(): Promise<Order[]> {
    // Fan out to all shards in parallel
    const results = await Promise.all(
      this.shards.map(shard =>
        shard.query('SELECT * FROM orders WHERE status = $1 ORDER BY created_at DESC LIMIT 100', ['pending'])
      )
    );

    // Merge and re-sort in memory
    const allOrders = results.flatMap(r => r.rows);
    allOrders.sort((a, b) => b.created_at - a.created_at);
    return allOrders.slice(0, 100); // global top-100
    // Note: had to fetch top-100 from EACH shard to guarantee global top-100
    // With 4 shards: 400 rows fetched, 300 discarded — wasteful at scale
  }

  // Global COUNT — must sum across all shards
  async getTotalOrderCount(): Promise<number> {
    const counts = await Promise.all(
      this.shards.map(async shard => {
        const r = await shard.query('SELECT count(*) AS n FROM orders');
        return parseInt(r.rows[0].n);
      })
    );
    return counts.reduce((a, b) => a + b, 0);
  }
}

Section 07

Citus: distributed PostgreSQL without leaving Postgres

Citus is a PostgreSQL extension (acquired by Microsoft, now open source) that turns a PostgreSQL cluster into a distributed database. You define distribution columns for your tables, and Citus automatically handles routing queries to the correct shards, parallelising across nodes, and assembling results.

How Citus works

Citus adds a coordinator node and one or more worker nodes. The coordinator stores the shard metadata — which shards contain which row ranges, which workers hold which shards. Application queries go to the coordinator, which rewrites them into distributed queries, fans them out to workers, and assembles the result.

Table types: distributed tables are sharded across workers by a distribution column. Reference tables are replicated to every worker (for small lookup tables used in JOINs). Local tables live only on the coordinator (for metadata, shard maps, etc.).

Citus setup — distributed table from regular PostgreSQLSQL
-- On coordinator: install and enable Citus
CREATE EXTENSION citus;

-- Register worker nodes
SELECT citus_add_node('worker-1.internal', 5432);
SELECT citus_add_node('worker-2.internal', 5432);
SELECT citus_add_node('worker-3.internal', 5432);

-- Create table (looks exactly like regular PostgreSQL)
CREATE TABLE orders (
  id         BIGSERIAL PRIMARY KEY,
  user_id    BIGINT NOT NULL,
  status     TEXT,
  amount     BIGINT,
  created_at TIMESTAMPTZ DEFAULT now()
);

-- Distribute it! Citus creates shards across workers.
SELECT create_distributed_table('orders', 'user_id');
-- user_id is the distribution column
-- Default: 32 shards per table, distributed across workers
-- You can query the table as if it's a single table

-- Small lookup tables: replicate to every worker for efficient JOINs
CREATE TABLE products (id BIGINT PRIMARY KEY, name TEXT, price BIGINT);
SELECT create_reference_table('products');
-- Now: SELECT o.*, p.name FROM orders o JOIN products p ON o.product_id = p.id
-- Works efficiently because products is on every worker

-- Check shard placement
SELECT * FROM citus_shards ORDER BY shard_id;

-- Co-location: tables sharded by the same column
-- store on same physical shard = JOINs don't need cross-shard calls
CREATE TABLE order_items (
  order_id BIGINT, user_id BIGINT, product_id BIGINT, quantity INT
);
SELECT create_distributed_table('order_items', 'user_id', colocate_with := 'orders');
-- Now: SELECT o.*, i.* FROM orders o JOIN order_items i ON o.id = i.order_id AND o.user_id = i.user_id
-- Executes on a single shard (both tables have same user_id on same worker)

-- Monitor shard sizes (detect imbalance)
SELECT shardid,
  pg_size_pretty(shard_size) AS size
FROM citus_shard_sizes()
ORDER BY shard_size DESC;

Section 08

Production reality

Resharding is a multi-week project

Adding shards (or changing the shard count) requires moving data between shards. With consistent hashing, adding one shard moves 1/N of the data. With range or hash sharding, resharding may require moving almost all data. This must happen online (the database is serving production traffic) with zero data loss. Tools: Citus automates this. Manual: double-write to old and new shard while backfilling, then cut over — risky and slow.

The data gravity problem

Once a tenant's data is large (multi-GB), moving it between shards is slow and risky. Tenant migration tooling must be tested before you need it. At Shopify scale, moving a large merchant from one pod to another is a weekend-long event with a dedicated runbook. Build tenant migration tooling early, even if you don't use it for months.

Shard-local transactions only

Any operation that must be atomic should be designed to operate within a single shard. If your checkout process involves inventory, payment, and order creation, and these are on different shards, you have a distributed transaction problem. Redesign: either co-locate these entities (same user_id distribution key means all data for user X is on shard X), or use sagas to handle cross-shard operations.

pg_partman for partition automation

Manually creating monthly partitions is error-prone — someone forgets to create next month's partition and inserts fail with "no partition for value." pg_partman automates this: it pre-creates future partitions and archives or drops old ones based on retention policy. Run it in maintenance mode for large tables to avoid locking.

Section 09

The things nobody tells you

Partition detach/attach is instant; partition DROP is also instant

Dropping a regular table with 100 million rows takes minutes (must free pages, update free space maps). Dropping a partition of a partitioned table is instant — the files are just deleted. This is one of the biggest operational wins of range partitioning for time-series data: instead of a slow DELETE with replication lag, just DROP the old partition. Takes milliseconds.

enable_partition_pruning must be on

PostgreSQL's partition pruning is controlled by enable_partition_pruning (default: on). If this is accidentally turned off, all queries scan all partitions. Verify: SHOW enable_partition_pruning. Also: partition pruning happens at plan time for static values but at execution time for prepared statements with runtime parameters — confirm with EXPLAIN ANALYZE, not just EXPLAIN.

Citus reference tables have replication lag

Reference tables are replicated synchronously during writes. A write to a reference table on the coordinator blocks until all workers confirm they've applied it. If you have many workers and a network hiccup, this can cause write latency spikes. Don't use reference tables for high-write tables — only truly read-heavy, rarely-changed lookup data.

Foreign keys work within a shard but not across shards in Citus

Citus supports foreign keys between distributed tables if they're co-located (same distribution column). order_items.user_id can reference orders.user_id if both are distributed by user_id. But orders.product_id cannot be a FK to a distributed products table — make products a reference table instead, then FKs work.

Decision sequence

You need to scale data storage — choose in this order

Work through this before committing to sharding. The earlier exit points save weeks of work.

1
Identify the actual bottleneck — not the symptom
"The database is slow" is a symptom. Use EXPLAIN ANALYZE, pg_stat_statements, and pg_stat_activity to identify exactly what's slow and why. Is it missing indexes? Lock contention? Connection exhaustion? Slow queries? These have targeted fixes. Sharding doesn't fix a missing index.
slow query log + pg_stat_statements first. always.
2
Try partitioning — same server, dramatic improvement for time-series
If your large table is time-series (orders, events, logs), range partition by month or week. Queries with date filters get 10-100× faster through partition pruning. Old partitions can be dropped instantly. No application changes required for most queries. This alone may solve your problem with a single CREATE TABLE ... PARTITION BY RANGE.
partitioning = sharding benefit without sharding complexity
3
Add read replicas before distributing writes
If reads are the bottleneck: add replicas and route read traffic appropriately (Day 10 covers this). If write throughput is fine but reads are slow, sharding is the wrong tool. Read replicas are operationally simpler and don't require any application changes beyond connection routing.
read bottleneck = replicas, not sharding
4
If you must shard: choose the key before everything else
The shard key is permanent. Before writing a single line of sharding infrastructure, answer: what column do most queries filter by? Is that column high-cardinality? Is it uniformly distributed? Will there be whale values? A weekend's thought on this is worth months of future operational pain. Write down the top-5 query patterns and verify they all work efficiently with the chosen key.
wrong shard key = can't be fixed without full resharding
5
Use Citus if you want to stay in PostgreSQL
Citus handles shard routing, rebalancing, and cross-shard query execution automatically. Your application still talks to a single PostgreSQL endpoint. Co-locate related tables by the same distribution key to keep JOINs local. Use reference tables for small lookup tables. This is the pragmatic choice for 95% of teams that reach for sharding.
Citus = sharding with PostgreSQL semantics
6
Build tenant migration tooling before you need it
In a sharded system, moving data between shards is inevitable — a shard fills up, a whale tenant needs their own shard, rebalancing becomes necessary. Build and test the migration tooling before any shard becomes a problem. The pattern: double-write to old and new shard, backfill historical data, verify consistency, cut over reads, stop writing to old shard. Test this with a non-critical small tenant first.
no migration tooling = shards fill up and you're stuck

Section 11

Sharding & partitioning checklist

PRT
Partition first

Try partitioning before sharding. Range partitioning + partition pruning + pg_partman solves most time-series scale problems.

KEY
Shard key is forever

Validate with top-5 query patterns. High cardinality. Uniform distribution. No whale values. Write it down before coding.

COL
Co-location

Related tables (orders + order_items) must share the same shard key so JOINs stay local to one shard.

PRU
Prune verification

EXPLAIN ANALYZE all key queries. Verify partition pruning is happening. Partitions scanned should be minimal.

MIG
Migration tooling

Build and test tenant/shard migration before any shard fills up. Double-write pattern + verification + cutover runbook.

HOT
Hotspot detection

Monitor shard sizes and QPS distribution weekly. Alert when one shard is 2× the average. Act before it becomes a crisis.
