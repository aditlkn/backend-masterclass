Deep Dive 01

pg_partman:
automated partition lifecycle

pg_partman automates the three things that make partitioning painful in production: creating future partitions before they're needed, maintaining indexes on new partitions, and dropping or archiving old partitions past their retention period. Without it, you manually create partitions monthly and eventually forget — and PostgreSQL returns "no partition for value" errors on insert.

pg_partman — complete setup and configurationSQL + BASH
-- ── Install pg_partman ─────────────────────────────────────────
-- On Ubuntu/Debian:
-- apt install postgresql-16-partman
-- On RDS: enable in parameter group: shared_preload_libraries = 'pg_partman_bgw'

CREATE SCHEMA partman;
CREATE EXTENSION pg_partman SCHEMA partman;

-- ── Create the parent partitioned table ─────────────────────────
CREATE TABLE events (
  id         BIGSERIAL,
  tenant_id  BIGINT NOT NULL,
  event_type TEXT NOT NULL,
  payload    JSONB,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
) PARTITION BY RANGE (created_at);

-- ── Hand off to pg_partman ──────────────────────────────────────
SELECT partman.create_parent(
  p_parent_table   := 'public.events',
  p_control        := 'created_at',     -- partition key column
  p_type           := 'range',           -- range or list
  p_interval       := 'monthly',         -- or: weekly, daily, hourly, '7 days', '4 hours'
  p_premake        := 4,                 -- pre-create 4 future partitions
  p_start_partition := '2024-01-01',     -- start of partition history
  p_automatic_maintenance := 'on'        -- enable background maintenance
);

-- ── Create indexes on all partitions (past and future) ──────────
-- pg_partman 5.x: use template tables for future partition indexes
CREATE TABLE public.events_template (LIKE public.events);
CREATE INDEX ON public.events_template (tenant_id, created_at DESC);
CREATE INDEX ON public.events_template (event_type, created_at DESC);
-- New partitions inherit template's indexes automatically

-- ── Retention policy (auto-drop old partitions) ─────────────────
UPDATE partman.part_config
SET retention = '12 months',           -- drop partitions older than 12 months
    retention_keep_table = false,       -- actually drop (not just detach)
    retention_keep_index = false        -- drop indexes with the partition
WHERE parent_table = 'public.events';

-- OR: detach rather than drop (keep data accessible as standalone table)
UPDATE partman.part_config
SET retention = '12 months',
    retention_keep_table = true         -- detach and rename, don't drop
WHERE parent_table = 'public.events';

-- ── Schedule maintenance ─────────────────────────────────────────
-- Option A: pg_partman background worker (add to postgresql.conf)
-- shared_preload_libraries = 'pg_partman_bgw'
-- pg_partman_bgw.interval = 3600   -- run every hour
-- pg_partman_bgw.role = 'partman'

-- Option B: pg_cron job (if pg_cron is available)
SELECT cron.schedule('partition-maintenance', '0 * * * *',
  'SELECT partman.run_maintenance_proc()');

-- Option C: external cron job
-- */30 * * * * psql -c "SELECT partman.run_maintenance_proc()" >> /var/log/partman.log 2>&1

-- ── Verify state ─────────────────────────────────────────────────
SELECT parent_table, partition_interval, retention, premake, last_partition
FROM partman.part_config;

-- List all partitions and their sizes
SELECT
  child.relname AS partition,
  pg_size_pretty(pg_relation_size(child.oid)) AS size,
  pg_get_expr(child.relpartbound, child.oid) AS bounds
FROM pg_inherits
JOIN pg_class parent ON pg_inherits.inhparent = parent.oid
JOIN pg_class child  ON pg_inherits.inhrelid  = child.oid
WHERE parent.relname = 'events'
ORDER BY child.relname;

-- Run maintenance manually (useful for testing retention/premake)
SELECT partman.run_maintenance('public.events', p_analyze := true);
Verify pg_partman is running correctly
✓Future partitions exist: SELECT relname FROM pg_class WHERE relname LIKE 'events_2024%' ORDER BY relname DESC LIMIT 5 — should show partitions several months ahead.
✓Partition pruning works: EXPLAIN SELECT * FROM events WHERE created_at > now()-interval '7 days' — should show 1-2 partitions, not all of them.
✓Maintenance is running: SELECT last_partition, last_run FROM partman.part_config WHERE parent_table = 'public.events' — last_run should be recent.
✓Old partitions being dropped: check that partitions older than retention period don't exist.

Citus full deployment:
coordinator + workers

Citus turns your PostgreSQL cluster into a distributed database. The coordinator routes queries and stores shard metadata; workers store the actual sharded data. This is the minimal viable Citus deployment for a B2B SaaS application.

Citus deployment — docker-compose + full schema setupYAML + SQL
# docker-compose.yml — development Citus cluster
version: '3.8'
services:
  coordinator:
    image: citusdata/citus:12.1
    ports:
      - "5432:5432"
    environment:
      POSTGRES_PASSWORD: password
      CITUS_HOST: coordinator
    volumes:
      - coordinator_data:/var/lib/postgresql/data

  worker-1:
    image: citusdata/citus:12.1
    environment:
      POSTGRES_PASSWORD: password
    volumes:
      - worker1_data:/var/lib/postgresql/data

  worker-2:
    image: citusdata/citus:12.1
    environment:
      POSTGRES_PASSWORD: password
    volumes:
      - worker2_data:/var/lib/postgresql/data

  worker-3:
    image: citusdata/citus:12.1
    environment:
      POSTGRES_PASSWORD: password

      - worker3_data:/var/lib/postgresql/data

  coordinator_data: {}
  worker1_data: {}
  worker2_data: {}
  worker3_data: {}

-- ── Connect to coordinator and set up cluster ───────────────────
-- psql -h localhost -U postgres

CREATE EXTENSION citus;

-- Register workers
SELECT citus_add_node('worker-1', 5432);
SELECT citus_add_node('worker-2', 5432);
SELECT citus_add_node('worker-3', 5432);

-- Verify cluster
SELECT * FROM citus_get_active_worker_nodes();

-- ── Multi-tenant SaaS schema ─────────────────────────────────────

-- Reference table: small, replicated to every worker
CREATE TABLE plans (
  id       BIGINT PRIMARY KEY,
  name     TEXT NOT NULL,
  max_users INT,
  features JSONB
);
SELECT create_reference_table('plans');
INSERT INTO plans VALUES (1, 'starter', 5, '{"api":true}'), (2, 'pro', 50, '{"api":true,"sso":true}');

-- Distributed table: sharded by tenant_id
CREATE TABLE tenants (
  id         BIGINT PRIMARY KEY,
  name       TEXT NOT NULL,
  plan_id    BIGINT REFERENCES plans(id),
  created_at TIMESTAMPTZ DEFAULT now()
);
SELECT create_distributed_table('tenants', 'id');

-- Co-locate all tables with tenants → JOINs are local to one shard
CREATE TABLE users (
  id         BIGSERIAL,
  tenant_id  BIGINT NOT NULL,
  email      TEXT NOT NULL,
  name       TEXT,
  created_at TIMESTAMPTZ DEFAULT now(),
  PRIMARY KEY (id, tenant_id)           -- partition key must be in PK for Citus
);
SELECT create_distributed_table('users', 'tenant_id', colocate_with := 'tenants');

CREATE TABLE orders (
  id         BIGSERIAL,
  tenant_id  BIGINT NOT NULL,
  user_id    BIGINT NOT NULL,
  amount     BIGINT,
  status     TEXT,
  created_at TIMESTAMPTZ DEFAULT now(),
  PRIMARY KEY (id, tenant_id)
);
SELECT create_distributed_table('orders', 'tenant_id', colocate_with := 'tenants');

-- This JOIN is shard-local (all co-located by tenant_id):
SELECT u.name, count(o.id) AS order_count
FROM users u
JOIN orders o ON u.id = o.user_id AND u.tenant_id = o.tenant_id
WHERE u.tenant_id = 42
GROUP BY u.name
ORDER BY order_count DESC;
-- ↑ Executes entirely on one worker shard — no cross-shard traffic

-- ── Monitoring shard balance ─────────────────────────────────────
SELECT
  nodename,
  count(*) AS shard_count,
  pg_size_pretty(sum(shard_size)) AS total_size
FROM citus_shard_sizes() cs
JOIN pg_dist_placement p ON cs.shardid = p.shardid
JOIN pg_dist_node n ON p.groupid = n.groupid
GROUP BY nodename
ORDER BY total_size DESC;

-- ── Rebalance shards (moves shards between workers online) ───────
SELECT citus_rebalance_start();
SELECT * FROM citus_rebalance_status();  -- monitor progress
-- Wait for STATUS = 'complete' before considering it done

-- ── Schema changes propagate to all shards ───────────────────────
ALTER TABLE orders ADD COLUMN metadata JSONB;
-- Citus applies this to all shards on all workers automatically
-- No need to run on each worker separately
Deep Dive 03

Consistent hashing:
implementation from scratch

Consistent hashing allows adding or removing shards without reassigning most keys. Instead of hash(key) % N, keys and nodes are placed on a virtual ring. Adding a new node only moves the keys between that node and its predecessor — roughly 1/N of all keys.

Consistent hashing — complete TypeScript implementationTYPESCRIPT
import * as crypto from 'crypto';
import { Pool } from 'pg';

class ConsistentHashRing {
  private ring: Map<number, string> = new Map(); // hash → node name
  private sortedHashes: number[] = [];
  private VIRTUAL_NODES = 150; // virtual nodes per physical shard
  // More vnodes = more even distribution but more memory

  addNode(nodeName: string): void {
    for (let i = 0; i < this.VIRTUAL_NODES; i++) {
      const virtualKey = `\${nodeName}:vnode:\${i}`;
      const hash = this.hash(virtualKey);
      this.ring.set(hash, nodeName);
    }
    // Re-sort after adding node
    this.sortedHashes = Array.from(this.ring.keys()).sort((a, b) => a - b);
  }

  removeNode(nodeName: string): void {
    for (const [hash, name] of this.ring) {
      if (name === nodeName) this.ring.delete(hash);
    }
    this.sortedHashes = Array.from(this.ring.keys()).sort((a, b) => a - b);
  }

  getNode(key: string): string {
    if (this.ring.size === 0) throw new Error('Ring is empty');
    const keyHash = this.hash(key);

    // Binary search: find first ring position >= keyHash
    let lo = 0, hi = this.sortedHashes.length - 1;
    while (lo < hi) {
      const mid = Math.floor((lo + hi) / 2);
      if (this.sortedHashes[mid] < keyHash) lo = mid + 1;
      else hi = mid;
    }

    // Wrap around (key > all nodes → use first node)
    const idx = this.sortedHashes[lo] >= keyHash ? lo : 0;
    return this.ring.get(this.sortedHashes[idx])!;
  }

  private hash(key: string): number {
    const buf = crypto.createHash('md5').update(key).digest();
    // Take first 4 bytes as unsigned 32-bit int
    return buf.readUInt32BE(0);
  }

  getDistribution(): Map<string, number> {
    const counts = new Map<string, number>();
    for (const node of this.ring.values()) {
      counts.set(node, (counts.get(node) || 0) + 1);
    }
    return counts;
  }
}

// ── Usage: Database shard router ────────────────────────────────
class ShardRouter {
  private ring = new ConsistentHashRing();
  private pools = new Map<string, Pool>();

  constructor(shards: { name: string; connString: string }[]) {
    for (const shard of shards) {
      this.ring.addNode(shard.name);
      this.pools.set(shard.name, new Pool({ connectionString: shard.connString }));
    }
  }

  getPool(shardKey: string): Pool {
    const nodeName = this.ring.getNode(shardKey);
    return this.pools.get(nodeName)!;
  }

  async addShard(name: string, connString: string): Promise<void> {
    // 1. Add shard to ring — now some keys route here
    this.ring.addNode(name);
    this.pools.set(name, new Pool({ connectionString: connString }));
    // 2. ~1/N of existing keys now route to new shard
    // 3. Must migrate those rows from old shards to new shard
    // 4. This is the "resharding" work: backfill → verify → update ring
  }

  getDistribution() {
    return this.ring.getDistribution();
  }
}

// Demo: adding a 4th shard only moves 25% of keys
const router = new ShardRouter([
  { name: 'shard-0', connString: 'postgresql://shard0...' },
  { name: 'shard-1', connString: 'postgresql://shard1...' },
  { name: 'shard-2', connString: 'postgresql://shard2...' },
]);
// Before: keys distributed across 3 shards (~33% each)
// After addShard('shard-3'):
//   shard-0: ~25%, shard-1: ~25%, shard-2: ~25%, shard-3: ~25%
//   Only ~8% of keys actually MOVED (from each old shard to new)
//   vs hash(key) % N: adding one shard reassigns ~75% of all keys
Verify consistent hashing is working correctly
✓Distribution is even: call router.getDistribution() and verify no shard has more than 1.5× the keys of the smallest shard.
✓Same key always routes to the same shard: router.getPool('user:123') === router.getPool('user:123') must be true across process restarts.
✓Adding a shard moves ~1/N of keys: before adding: note which shard getPool('user:500') returns. After adding: most keys still return the same shard.
✓Virtual node count ≥ 100: fewer vnodes causes uneven distribution. Test with 10 vnodes vs 150 and compare the spread.
