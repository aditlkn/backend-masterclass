LAB
OOMKill Loop  ·  booking-service  ·  Every ~4 hours

Pods are being OOMKilled every four hours. HPA keeps scaling up. It is not helping.

Production · booking-service · namespace: production · 3 replicas, scaling to 8

The on-call engineer gets paged at 02:00. The booking service is restarting repeatedly. Each pod runs for approximately 4 hours before being OOMKilled. The HPA has scaled from 3 to 8 replicas trying to absorb the load — but the pattern continues across all replicas. Error rate is elevated during each restart cycle.

The engineer checks the Grafana dashboard: memory usage is climbing linearly from ~280Mi at pod start to ~512Mi (the limit) over 4 hours, then the pod is killed. The pattern is identical across all 8 pods, offset by their start times.

What this incident teaches
▶Linear memory growth is always a leak, never a limit problem. If memory grows at a constant rate independent of request volume, the limit is not the issue — code is accumulating objects without releasing them. Increasing the limit delays the OOMKill but does not prevent it. Find and fix the leak.
▶HPA cannot fix a resource leak. HPA adds capacity for load. A leak is not a load problem. More pods handling the same requests means more pods each accumulating memory at the same rate. The root cause — the leak — must be fixed in the application.
▶OOMKill has no log output. The pod is terminated immediately by the OS. The last few seconds of logs are typically lost. The only evidence is in kubectl describe pod under Last State: Reason: OOMKilled. If you are missing logs on a crashing service, check for OOMKills before debugging the application.
▶EventEmitter listener leaks are one of the most common Node.js memory leaks. Always use emitter.once() instead of emitter.on() for one-time listeners, or explicitly call emitter.removeListener(). Node.js will emit a warning when a single emitter has more than 10 listeners — this is the built-in leak detector.
