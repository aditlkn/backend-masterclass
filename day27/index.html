Section 01

The shipping port — and why the port master does not care what is in the containers

A shipping port receives cargo containers. Each container has a declared weight and cargo type. The port master (scheduler) looks at the fleet of ships (nodes) sitting in the harbour, checks their remaining capacity, and assigns each container to a ship that can handle it. The port master does not know or care what is inside the container — only how much space it needs and what kind of ship can carry it.

That is Kubernetes scheduling. Your application is the cargo. The container image is the sealed container. The pod is the container on the ship. The node is the ship. The scheduler assigns pods to nodes based on available resources (CPU, memory) and any special requirements (this container needs a GPU ship, or must not share a ship with another container of the same type).

The port has weight limits per berth. A ship rated for 50,000 tonnes cannot take a 60,000-tonne load. If you try, the berth inspector rejects it. Kubernetes resource requests are the declared weight — the scheduler will not place a pod on a node that does not have enough CPU and memory available. Resource limits are the physical limit of the container itself — if a container tries to use more than its limit, Kubernetes throttles or kills it.

Customs inspection before cargo is cleared. When a ship arrives, customs inspects each container. Is it structurally sound? Does it contain what the manifest says? If a container fails inspection, it is offloaded and quarantined. Kubernetes liveness and readiness probes are this inspection. Liveness: is the container still alive and not deadlocked? Readiness: is it ready to receive cargo (traffic)? Fail the readiness probe and traffic is routed away. Fail the liveness probe and the container is replaced.

Calling in more ships when the queue grows. If the port receives more containers than the current fleet can handle, the port authority calls in additional ships from the reserve fleet. When demand drops, reserve ships are sent back to minimise docking fees. That is the Horizontal Pod Autoscaler — adding pods when CPU or custom metrics exceed a threshold, removing them when demand drops.

⚓
Where the port analogy breaks

A shipping port is mostly stateless — once a container is offloaded, the ship moves on. Kubernetes manages both stateless workloads (Deployments) and stateful ones (StatefulSets), where the pod identity and storage must survive restarts. A stateful pod is more like a permanent resident in a specific berth with a reserved locker — it cannot simply be moved to another ship without migrating their belongings. StatefulSets handle this, but they require significantly more care than Deployments.

Section 02

Which Kubernetes object does what?

Kubernetes has many object types but backend engineers primarily work with five. Understanding which layer each operates at — and why they are separate — makes the whole system click.

Pod: The smallest deployable unit. One or more containers that share a network namespace and storage. You almost never create pods directly — you create Deployments that create and manage pods. A pod has an IP address, runs on one node, and is mortal: it will eventually die and not be restarted. The Deployment creates a new pod in its place.

Deployment: Manages a set of identical pod replicas. Declares how many replicas you want, which container image to run, and what resources each pod needs. When a pod dies or a node fails, the Deployment controller creates a replacement. Rolling updates (new version deployed incrementally, old pods removed as new ones become ready) are a Deployment feature.

Service: A stable virtual IP address and DNS name that routes traffic to a set of pods. Pods come and go — their IPs change. The Service IP is stable. Other services call booking-service:3000, and Kubernetes routes that request to one of the healthy booking pods via kube-proxy. Services are how microservices discover and call each other inside a cluster.

ConfigMap and Secret: Configuration and sensitive data injected into pods as environment variables or mounted files. Secrets are base64-encoded (not encrypted at rest by default — use a secrets manager like Vault or AWS Secrets Manager for true encryption). Never hardcode configuration in container images.

Namespace: A virtual cluster within the cluster. Provides isolation between environments (staging, production) or teams. Resource quotas are applied at the namespace level. Service DNS names are scoped to namespaces: booking-service.production.svc.cluster.local.

Section 03

Why are resource requests and limits the most important thing you configure?

Resource requests and limits are the single most impactful configuration in a Kubernetes workload. Get them wrong and you get OOMKills (container killed for exceeding memory limit), CPU throttling (container throttled to its CPU limit, causing latency spikes), or pending pods (scheduler cannot find a node with enough requested resources). Get them right and the scheduler places workloads efficiently and the cluster runs stably.

Requests — what the scheduler uses

The resource request is what the scheduler treats as reserved. A pod requesting 500m CPU and 256Mi memory will only be placed on a node that has at least that much unreserved capacity. Requests do not prevent a pod from using more — they are a scheduling guarantee, not a hard limit. If a node has 2000m CPU and three pods each request 500m, the scheduler will place all three (1500m total requested) even though actual usage may vary.

Limits — what the runtime enforces

The resource limit is what the container runtime enforces. A pod with a 512Mi memory limit that attempts to allocate 600Mi will be OOMKilled. A pod with a 1000m CPU limit that wants to use 2000m will be throttled — it will continue running but at reduced speed. Memory OOMKills are immediate and painful. CPU throttling is silent and causes latency spikes that are hard to diagnose without the right metrics.

The right sizing approach

Set requests based on typical (P50) usage. Set limits at 2-3x the typical usage to handle spikes without killing the pod for normal load variation. Monitor actual usage with kubectl top pods or Prometheus/Grafana for two weeks, then adjust. The Vertical Pod Autoscaler (VPA) can recommend and apply right-sizing automatically based on historical usage.

🔴
No limits = noisy neighbour problem

Pods without CPU limits can consume all available CPU on a node, starving other pods. This is especially dangerous in shared clusters where one team's runaway pod affects another team's production service. Always set resource limits. If you do not know the right value, set a generous limit and monitor actual usage — a pod using 10% of its limit tells you the limit is 10x too high. A pod hitting its limit regularly tells you it needs more.

deployment.yaml — resource configurationYAML
apiVersion: apps/v1
kind: Deployment
metadata:
  name: booking-service
  namespace: production
spec:
  replicas: 3
  selector:
    matchLabels:
      app: booking-service
  template:
    metadata:
      labels:
        app: booking-service
    spec:
      containers:
      - name: booking-service
        image: registry.example.com/booking-service:v2.4.1
        ports:
        - containerPort: 3000
        resources:
          requests:
            cpu: "250m"      # 0.25 vCPU — scheduler guarantee
            memory: "256Mi"  # 256 MiB — scheduler guarantee
          limits:
            cpu: "1000m"     # 1 vCPU — throttled above this
            memory: "512Mi"  # 512 MiB — OOMKilled above this
        env:
        - name: DATABASE_URL
          valueFrom:
            secretKeyRef:
              name: booking-secrets
              key: database-url
        - name: NODE_ENV
          value: production

Section 04

Health probes — the three questions Kubernetes asks your pod

Kubernetes asks three separate health questions. Each has a different consequence if the answer is "no." Getting the semantics right is one of the most practically important things you do when deploying a service.

Liveness — is the container alive?

If the liveness probe fails, Kubernetes kills and restarts the container. Use this for detecting deadlocks and zombie processes — situations where the application is running but not making progress and cannot recover itself. Do not put expensive logic in the liveness probe — if it times out, the pod is killed. A liveness probe should check that the process is responsive, not that all its dependencies are healthy.

Readiness — is the container ready for traffic?

If the readiness probe fails, the pod is removed from the Service's endpoint list — no new traffic is routed to it, but the pod is not killed. Use this for detecting that the application has not finished initialising, or is temporarily overloaded and needs to drain. A pod that is unready but live will recover and be re-added to traffic when the probe starts passing again.

Startup — has the container finished starting?

The startup probe protects slow-starting containers from being killed by the liveness probe before they have finished initialising. Until the startup probe passes, liveness and readiness probes are disabled. Essential for applications that take 30+ seconds to start (JVM warmup, loading large ML models, database migration on startup).

⚠️
The most common probe mistake: checking dependencies in liveness

A liveness probe that checks the database connection will kill and restart the pod when the database is down. This restarts every pod simultaneously, causing a thundering herd that makes the database outage worse. Liveness should only check that the application process itself is alive. Dependency health belongs in readiness: a pod that cannot reach its database should stop receiving new requests (readiness fails), but should not be killed (liveness stays passing).

Probe configuration — all three typesYAML
        livenessProbe:
          httpGet:
            path: /health/live    # returns 200 if process is alive
            port: 3000
          initialDelaySeconds: 10
          periodSeconds: 10
          failureThreshold: 3     # 3 failures = restart
          timeoutSeconds: 5

        readinessProbe:
          httpGet:
            path: /health/ready   # returns 200 if ready for traffic
            port: 3000            # checks DB, cache, dependencies
          initialDelaySeconds: 5
          periodSeconds: 5
          failureThreshold: 2     # 2 failures = remove from Service
          timeoutSeconds: 3

        startupProbe:
          httpGet:
            path: /health/live
            port: 3000
          failureThreshold: 30    # 30 x 10s = 5 min to start
          periodSeconds: 10

Section 05

HPA — how Kubernetes decides to add or remove pods

The Horizontal Pod Autoscaler watches a metric (CPU utilisation, memory, or a custom metric) and adjusts the replica count of a Deployment to keep the metric near a target. When average CPU across all pods exceeds 70%, the HPA adds pods. When it drops well below 70%, the HPA removes pods after a cooldown period.

The control loop

Every 15 seconds, the HPA fetches the current metric value, calculates the desired replica count as ceil(currentReplicas * (currentValue / targetValue)), and tells the Deployment to scale to that count. The ceiling function means it rounds up — it would rather have a slightly over-provisioned cluster than an under-provisioned one. Scale-up is immediate. Scale-down has a 5-minute cooldown to prevent thrashing.

CPU-based HPA — the most common and most misunderstood

CPU-based HPA scales on utilisation relative to the request, not absolute CPU. A pod requesting 250m CPU at 80% utilisation is using 200m. A pod requesting 1000m CPU at 80% is using 800m. The HPA sees the same 80% in both cases. This means: if resource requests are wrong, HPA scaling behaviour is wrong. Over-requested pods appear under-utilised (HPA scales down too aggressively). Under-requested pods appear over-utilised (HPA scales up too eagerly).

Custom metrics HPA — scale on what actually matters

For most backend services, CPU is not the right signal. Request latency, queue depth, or active connections are better proxies for "this service needs more capacity." Custom metrics HPA requires the Prometheus adapter or KEDA (Kubernetes Event-Driven Autoscaling). KEDA can scale on Kafka consumer lag, SQS queue depth, Redis list length, or any Prometheus metric — making it the right tool for event-driven workloads that need to scale based on backlog rather than CPU.

hpa.yaml — CPU and custom metric autoscalingYAML
# CPU-based HPA
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: booking-service-hpa
  namespace: production
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: booking-service
  minReplicas: 3
  maxReplicas: 20
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70   # scale up when avg CPU > 70%
  - type: Resource
    resource:
      name: memory
      target:
        type: Utilization
        averageUtilization: 80
  behavior:
    scaleDown:
      stabilizationWindowSeconds: 300   # 5 min cooldown before scale-down
      policies:
      - type: Pods
        value: 2
        periodSeconds: 60   # remove at most 2 pods per minute
    scaleUp:
      stabilizationWindowSeconds: 0    # scale up immediately
      policies:
      - type: Percent
        value: 100
        periodSeconds: 15   # can double replicas every 15s

---
# KEDA ScaledObject — scale on Kafka consumer lag
apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: email-consumer-scaler
spec:
  scaleTargetRef:
    name: email-consumer
  minReplicaCount: 1
  maxReplicaCount: 10
  triggers:
  - type: kafka
    metadata:
      bootstrapServers: kafka:9092
      consumerGroup: email-service
      topic: booking.confirmed
      lagThreshold: "100"   # 1 pod per 100 events of lag

Section 06

How does the scheduler decide where to put your pod?

The Kubernetes scheduler runs a two-phase process: filtering (which nodes can run this pod?) and scoring (among those nodes, which is best?). Understanding this makes the difference between a cluster that efficiently uses its nodes and one where pods are pending despite available capacity.

Filtering — eliminating impossible nodes

A node is eliminated if it does not have enough unreserved CPU and memory to satisfy the pod's requests, if the pod has node selectors or affinity rules that the node does not satisfy, if there are pod anti-affinity rules that conflict with existing pods on that node, or if the node has taints that the pod does not tolerate. Any pod that cannot be placed on any node after filtering enters Pending state.

Node affinity — controlling where pods land

Node affinity lets you express preferences and requirements for which nodes a pod should run on. Required affinity is like a filter — the pod will not be scheduled on a node that does not match. Preferred affinity is like a scoring boost — the scheduler prefers matching nodes but will use others if necessary. Common use cases: run GPU workloads only on GPU nodes, run production pods only on nodes in a specific availability zone, keep data processing pods close to storage.

Pod anti-affinity — spreading for high availability

Pod anti-affinity ensures replicas of the same service are spread across different nodes (or availability zones). Without anti-affinity, the scheduler might place all three replicas of your booking service on the same node — and a node failure takes down your entire service. With topologySpreadConstraints, Kubernetes ensures replicas are distributed evenly across failure domains.

Section 07

How do services find each other inside a cluster?

Inside a Kubernetes cluster, every Service gets a DNS name: service-name.namespace.svc.cluster.local. Within the same namespace, you can use just service-name. The booking service calls the payment service at payment-service:3000 — Kubernetes DNS resolves this to the Service's ClusterIP, and kube-proxy routes the request to one of the healthy payment pods.

Service types — ClusterIP, NodePort, LoadBalancer

ClusterIP (default): only reachable from inside the cluster. Use for all internal services. NodePort: exposes the service on a port on every node's IP. Useful for debugging; not for production external traffic. LoadBalancer: provisions a cloud load balancer (AWS NLB, GCP Load Balancer) and routes external traffic to the service. Use for services that need to be reachable from outside the cluster.

Ingress — HTTP routing into the cluster

An Ingress is an HTTP(S) routing rule. It sits in front of multiple Services and routes requests based on host or path: api.example.com/bookings goes to the booking service, api.example.com/payments goes to the payment service. Ingress requires an Ingress Controller (nginx-ingress, Traefik, AWS ALB Ingress Controller) that watches Ingress resources and configures the underlying load balancer. TLS termination happens at the Ingress.

Section 08

Production reality

Pod Disruption Budgets — protecting against accidental downtime

During node maintenance, Kubernetes drains nodes by evicting pods. Without a Pod Disruption Budget (PDB), it might evict all three replicas of your service simultaneously, causing an outage. A PDB specifies the minimum number of pods that must remain available during voluntary disruptions. minAvailable: 2 means at most one pod can be evicted at a time. Always set a PDB for any service with SLOs.

Rolling updates and zero-downtime deploys

A rolling update replaces old pods with new ones gradually: spawn a new pod, wait for its readiness probe to pass, then remove an old pod. The key configuration: maxSurge (how many extra pods can exist during the update — default: 25%) and maxUnavailable (how many pods can be unavailable during the update — set to 0 for zero-downtime). With maxUnavailable: 0 and a readiness probe, old pods are never removed until new ones are confirmed ready.

Resource quotas — preventing namespace sprawl

A ResourceQuota applied to a namespace limits total CPU, memory, and object counts that namespace can consume. Without quotas, a team can request unlimited resources and starve other namespaces. Apply quotas per namespace, matched to the team's expected workload with headroom for HPA scale-up.

Section 09

The things nobody tells you

CPU throttling is silent and extremely common

A pod that is CPU-throttled continues to run and passes all health checks, but every operation takes longer. P99 latency spikes, SLOs are missed, and the only evidence is a metric called container_cpu_cfs_throttled_seconds_total that most teams never look at. Check this metric in your Grafana dashboard. If it is non-zero, your CPU limit is too low and your service is being throttled.

The scheduler does not rebalance

Once a pod is scheduled onto a node, Kubernetes does not move it unless the node fails or the pod is deleted. If a new, under-utilised node joins the cluster, existing pods do not migrate to it. The cluster only rebalances when pods are created or recreated. Descheduler is a separate tool that evicts pods from over-utilised nodes to trigger rebalancing — consider running it on a schedule in production clusters.

Secrets are not secret without additional work

Kubernetes Secrets are base64-encoded, not encrypted, by default. Anyone with read access to the namespace can decode them. Enable encryption at rest for Secrets in etcd, use sealed secrets (Bitnami Sealed Secrets), or use an external secrets operator that pulls from AWS Secrets Manager, HashiCorp Vault, or GCP Secret Manager. For production, treat a Kubernetes Secret as only as secure as the access controls on your cluster.

OOMKill leaves no logs

When a container is OOMKilled, the process is terminated immediately — no graceful shutdown, no log flush. The last few seconds of logs may be lost. The only evidence is kubectl describe pod showing OOMKilled in the Last State, and the container_oom_events_total metric. If you are missing logs on a service that crashes, check if it is being OOMKilled before suspecting a logging configuration problem.

Decision sequence

Deploying a service to Kubernetes

Work through this before writing a single manifest.

1
Set resource requests from measured usage, not guesses
Run the service under realistic load in staging. Measure P50 CPU and memory usage with kubectl top pods. Set requests at P50 usage. Set limits at 2-3x P50. Update after two weeks in production with real traffic data. Wrong requests cause cascading problems: pending pods, HPA misbehaviour, noisy neighbour issues.
guessed resource values cause 80% of K8s production incidents
2
Implement all three probe endpoints in your application
GET /health/live: returns 200 if the process is alive. No dependency checks. GET /health/ready: returns 200 if the service can handle traffic — checks database connection, cache reachability. Returns 503 when not ready. GET /health/startup: same as live, used only during startup phase with generous timeout.
dependencies in liveness = cascading restart storms
3
Set maxUnavailable: 0 and configure readiness before HPA
Rolling updates with maxUnavailable: 0 ensure old pods are not removed until new ones pass readiness. This requires a correct readiness probe. Without it, Kubernetes removes old pods before the new ones are actually ready to serve traffic, causing brief outages during every deploy. Validate this in staging by deploying while load testing.
maxUnavailable:0 + readiness = zero-downtime deploys
4
Add a PDB before enabling HPA
A PDB ensures that during node maintenance, at least N pods remain available. Without it, draining a node can take down all replicas simultaneously. Set minAvailable: 2 for most services. The HPA needs to be aware of the PDB — it will not scale below minReplicas, and the PDB prevents eviction below minAvailable.
no PDB = node maintenance causes outages
5
Spread replicas across availability zones
Use topologySpreadConstraints with topologyKey: topology.kubernetes.io/zone to ensure replicas are spread across AZs. Without this, the scheduler may place all replicas in one AZ — a zone failure takes your service down despite having 3 replicas. This is the most common HA configuration mistake in Kubernetes.
all replicas in one AZ = single AZ failure = full outage
6
Monitor CPU throttling, OOMKills, and pending pods
Three metrics that matter most: container_cpu_cfs_throttled_seconds_total (CPU throttling — silent latency killer), container_oom_events_total (OOMKills — silent crash), kube_pod_status_phase{phase="Pending"} (pending pods — scheduler cannot place them). Alert on all three. They indicate resource misconfiguration before it becomes a user-facing incident.
these three metrics tell you about problems before users do

The port master does not ask what is in the container. It asks how much space it needs, whether it passed customs inspection, and how many ships to call in when the queue grows. Kubernetes asks exactly these three questions — about resources, health, and scale — and keeps asking them every 15 seconds for the lifetime of your cluster.

The next time a Kubernetes incident wakes you up — ask: is it a scheduling problem, a health probe problem, or a resource problem? Those three categories cover 90% of what goes wrong.
